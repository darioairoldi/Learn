# Folder-metadata inheritance (`_metadata.yml`)

> **Status: PROPOSED (pending owner review/commit).** This spec realizes [04-folder-metadata-inheritance-update-plan.md](../../../src/docs/90.%20Issues/202606/20260621.01-learninghub-update/04-folder-metadata-inheritance-update-plan.md). Until committed and activated, it is a design contract, **not** active guidance; the live amendments it triggers are listed in § Propagation on activation.

## Overview

A folder places a `_metadata.yml` file whose `identity:` section is **inherited** by descendant articles through a deterministic keyed-merge, and the resolved result is **materialized** into a generated `effective:` block in each descendant. This lets folder-scoped identity (`goal`, `scope`, `boundaries`, `rationales`, `domain`, `additional_domains`) be authored once and reused, without restating it per article and without silent divergence.

It extends the per-file dual-metadata model (see [../90.00-learning-hub/02-dual-yaml-metadata.md](../90.00-learning-hub/02-dual-yaml-metadata.md)) from a single file to a folder tree, reusing two patterns the repo already runs: the `appsettings.json` folder-depth cascade and the validation-caching generated-block.

## Applicability domain and exemptions

The cascade runs ONLY in:

- **Article content trees** (the published `.md` corpus), and
- **`.copilot/context/{domain}/`** folders (for the non-inherited `domain_profile:` section).

It is **never** applied to:

- **PE artifact folders** — `.github/agents`, `.github/prompts`, `.github/instructions`, `.github/templates`, `.github/skills`, `.github/prompt-snippets` — which are governed by their mandated `pe-*` schemas, and
- **vision / use-case documents**, whose frontmatter schema is mandated by `vision-frontmatter.instructions.md` / `use-case-documents.instructions.md`.

## `_metadata.yml` sectioned schema

> Baseline only — **designed to expand**. A top-level `schema_version` gates forward-compatibility; consumers MUST preserve unknown keys. New fields are additive via a `schema_version` bump; renames/removals are breaking and human-only.

**File-level**

| Field | Type | Inherited? | Purpose |
|---|---|---|---|
| `schema_version` | int | n/a | Contract version for forward-compat (REQUIRED). |
| `inherit` | bool | n/a | Whether descendants inherit `identity:` (default `true`). |

**`identity:` — INHERITED** (the inheritable subset of the article stable-identity block; **never** includes per-article `title`, `date`, `description`)

| Field | Type | Merge rule |
|---|---|---|
| `author` | scalar | nearest-wins |
| `categories` | array | inherit-all + keyed ops |
| `domain` | scalar | nearest-wins · the single **primary** domain (classification / domain-coherent batching) |
| `additional_domains` | ordered list `[id, …]` | inherit-all + keyed ops (key = id, position = relevance) · the *secondary* related domains for role/domain-switch; full ranked list = `[domain] + additional_domains` |
| `goal` | scalar | nearest-wins (article refines) |
| `scope` | scalar | nearest-wins |
| `boundaries` | keyed array `[{id, text}]` | inherit-all · `remove` requires `reason` (no silent weakening) |
| `rationales` | keyed array `[{id, text}]` | inherit-all + keyed ops |

> **`domain` vs `additional_domains` vs `domain_profile`.** `domain:` (scalar) is the one *primary* domain used for classification and domain-coherent batching — kept scalar (not promoted to a section) so existing `domain: <id>` frontmatter stays valid. `additional_domains:` (ordered list) holds the *secondary* related domains for role/domain-switch grounding — each domain is stated exactly once (no redundancy with `domain`), and list position encodes descending relevance, so the full ranked list is `[domain] + additional_domains`. `domain_profile:` is neither a label nor a list — it is the *domain's own identity-and-contract* (a domain has an identity, just like an artifact does), defined **once per domain** in that domain's folder and **referenced by id** from any artifact that declares the domain.

**`domain_profile:` — OPTIONAL · context folders · NOT inherited.** The domain's identity & contract; **grounding is one facet**. Its fields split into **grounding facets** (what a pass *acquires* to work in the domain) and **contract facets** (what every artifact in the domain must *respect* and is graded against). Extensible via `schema_version`.

*Grounding facets (acquire):*

| Field | Type | Purpose | Consumed by |
|---|---|---|---|
| `role` | scalar | expert framing / persona for the role/domain switch | `domain-expertise-injection` |
| `authoritative_sources` | `[{url, version_scheme}]` | per-domain Type-B staleness sources | `domain-expertise-injection` + `external-knowledge` |
| `monitored_invariants` | array | domain truths watched for drift | `domain-expertise-injection` |

*Contract facets (respect / graded against):*

| Field | Type | Purpose | Consumed by |
|---|---|---|---|
| `scope` | scalar | what the domain covers / excludes | classification + relevance |
| `boundaries` | keyed array `[{id, text}]` | domain-wide constraints EVERY artifact in the domain MUST respect (in addition to its own) | `runtime-grounding` + `metadata-guarded-changes` (domain-level) |
| `quality_criteria` | keyed array `[{id, text}]` | domain-specific dimensions of "good" applied when grading | the assessment / dimension system |
| `conventions` | array | domain terminology / naming / style | authoring |

> **Composition.** An artifact's **effective contract** = its own identity ⊕ ⋃(`domain_profile` of each declared domain in `[domain] + additional_domains`). Grounding facets are *loaded*; contract facets are *checked against*. Multi-domain artifacts union the profiles (`boundaries`: more-restrictive-wins or escalate, per the keyed-merge rule); `role` is the lead persona, taken from the most-relevant (`domain`) when several apply.

**`monitoring:` — OPTIONAL · subject folders · NOT inherited** (coverage is derived, not authored)

| Field | Type | Purpose |
|---|---|---|
| `product_dependencies` | `[{product, min_version, docs_url, alert_keywords}]` | content-freshness inputs |
| `review_cadence` | enum | revalidation cadence |
| `staleness_signals` | array | events that trigger revalidation |
| `grounds_on_domain` | ref | links a subject to a `domain_profile` domain (the bridge) |

## Deterministic keyed-merge rule

The merge is a pure function over the ancestor chain (root → leaf, the article's own frontmatter is the nearest layer):

- **Scalars** (`author`, `domain`, `goal`, `scope`, …): nearest-wins.
- **Arrays** (`boundaries`, `rationales`, `additional_domains`, `categories`): inherit-all by default; per-element ops `add`, `override` (by key), `remove` (by key, **`reason` required**); dedupe by key.
- **`boundaries`**: removal MUST carry a `reason` — no silent weakening of a safety constraint.
- **Keys**: auto-assigned by the watcher (slug/hash of element text); a dangling `override`/`remove` key (the parent text changed) is **flagged**, never silently dropped.

Because folders form a **tree**, every file has exactly one ancestor chain — there is no diamond/multiple-inheritance conflict, so resolution is a deterministic linear fold.

## Materialized `effective:` block

The resolved result is generated into each descendant article's **bottom (volatile) block** — do-not-edit, maintained by MetadataWatcher, recomputed on any ancestor `_metadata.yml` change (subtree recompute). It is both the effective-vs-declared view and the staleness key.

| Field | Type | Purpose |
|---|---|---|
| `resolved_at` | timestamp | when the cascade was last computed |
| `sources` | array | provenance: each ancestor `_metadata.yml@<hash>` + `self` |
| `identity` | object | the fully merged effective identity values |
| `inputs_hash` | hash | hash of (ancestor chain + declared) — the freshness key |
| `status` | `fresh` \| `stale` | consumed by the runtime staleness check below |

## Runtime metadata-cache staleness verification

Every consumer that reads metadata at execution time — notably PE artifacts under `runtime-grounding` — MUST, before relying on the `effective:` block, recompute `inputs_hash` over the current ancestor chain and compare it to the cached value. On mismatch the block is **stale** → recompute, or **fail-closed/escalate** when recompute is unavailable, rather than act on stale inheritance. This extends `validation-caching` (see [04.01-validation-caching-pattern.md](04.01-validation-caching-pattern.md)) and `metadata-guarded-changes` from per-file to the cascade.

## Safety and backward-compatibility (additive by design)

- **Zero-touch when absent** — no `_metadata.yml` in the ancestor chain ⇒ nothing computed, no `effective:` block written; existing articles are untouched.
- **Nearest-wins preserves today's behavior** — while articles keep their declared frontmatter, effective = declared (identical to now); dropping now-redundant fields is a separate, later, opt-in retrofit.
- **Bounded applicability** — see § Applicability domain and exemptions.
- **Spec-now / build-later** — until MetadataWatcher implements the cascade the mechanism is inert; the watcher ships with a dry-run mode and the zero-touch guard.
- **Quarto-safe by location** — `.copilot/context/` manifests are outside the Quarto render root; content-tree `_metadata.yml` uses Quarto's own directory-metadata mechanism, gated by a reserved-key compatibility check before rollout.

## Folder manifest template

```yaml
# _metadata.yml — folder-level metadata; the identity section cascades to descendant articles.
schema_version: 1
inherit: true
identity:
  author: "Dario Airoldi"
  domain: azure                              # the single primary domain
  additional_domains: [security, networking] # secondary, descending relevance
  scope: "Azure platform services, architecture, and cost."
  boundaries:
    - { id: b-no-secrets, text: "Never include real tenant IDs; redact to <tenant-id>." }
domain_profile:          # OPTIONAL — the domain's identity & contract; only in a .copilot/context/{domain}/ folder; NOT inherited
  # grounding facets (acquire)
  role: "Senior Azure solutions architect."
  authoritative_sources:
    - { url: "https://learn.microsoft.com/azure/", version_scheme: date }
  monitored_invariants:
    - "Managed identity is preferred over connection strings."
  # contract facets (respect / graded against)
  scope: "Azure platform services, architecture, cost, security."
  boundaries:
    - { id: b-no-secrets, text: "Never embed real tenant IDs; redact to <tenant-id>." }
  quality_criteria:
    - { id: q-waf, text: "Recommendations align with the Well-Architected Framework pillars." }
  conventions:
    - "Spell out 'resource group' on first mention, not 'RG'."
monitoring:              # OPTIONAL — only in a content subject folder; NOT inherited
  review_cadence: quarterly
  staleness_signals: [product_release, broken_link]
```
