**Referenced by**: `templates.instructions.md`, `prompt-builder`, `agent-builder`, `context-builder`, `skill-builder`, template creation workflows

## Audience-Aware Design

Templates serve different consumers. The consumer determines the design approach — get this wrong and the template either confuses users or is unparsable by agents.

### Consumer Detection

Before creating a template, identify the primary consumer:

| If the template is used by... | Consumer type | Design priority |
|---|---|---|
| An agent generating a report | **Agent** | Parsability — tables, field markers, minimal prose |
| A user filling in project details | **User** | Readability — descriptions, examples, natural language |
| An agent following a workflow | **Agent** | Imperativeness — decision points, phase gates, ordered steps |
| A user and an agent (e.g., input collected by user, processed by agent) | **Both** | Balance — structured fields with inline descriptions |

### Design Rules by Consumer Type

**Agent-consumed templates** (`output-*`, `guidance-*`, `*-structure`):
- Use tables over prose for structured data
- Use `[placeholder]` markers with unambiguous field names
- Keep descriptions minimal — only when the field name is ambiguous
- Use consistent section ordering that agents can navigate predictively
- Severity/status markers: `✅`, `⚠️`, `❌` for quick parsing

**User-consumed templates** (`input-*`, `pattern-*`):
- Use natural language prompts: "Describe the primary goal..."
- Include examples for every field that isn't self-explanatory
- Use checkboxes `[ ]` for required/optional field tracking
- Group related fields under descriptive subheadings

**Dual-audience templates**:
- Lead each field with a parsable marker: `**Field Name:** [value]`
- Follow with a brief description in parentheses or on the next line
- Use HTML comments for agent-only metadata: `<!-- REQUIRED: min 3 items -->`

## Placeholder Conventions

### Syntax

| Syntax | Meaning | Example |
|---|---|---|
| `[description]` | User-filled or agent-filled content | `[One-sentence purpose statement]` |
| `{{variable}}` | System-filled metadata (timestamps, filenames) | `{{ISO-8601 timestamp}}` |
| `{artifact}` | Naming pattern variable in conventions | `output-{artifact}-{purpose}.template.md` |

### Naming Placeholders

Placeholder text MUST describe the expected content, not the field name:

```markdown
✅ [One-sentence description of what this file provides]
✅ [List of 3-7 tools from tool-composition-guide]
❌ [description]
❌ [tools]
```

### Optional vs Required

Mark optional sections explicitly:

```markdown
## Prerequisites
<!-- OPTIONAL: Include only if the workflow has prerequisites -->
[List prerequisites, or remove this section if none]
```

Required sections use:

```markdown
## Purpose
<!-- REQUIRED: Every template must include this -->
[One-sentence description]
```

## Category Selection

Choose the template category based on its function in the workflow:

```
What does this template define?
├─ The format of a generated report or output?
│  → output-* (e.g., output-prompt-validation-phases.template.md)
├─ Fields to collect from a user or upstream agent?
│  → input-* (e.g., input-article-requirements.template.md)
├─ A step-by-step process for an agent to follow?
│  → guidance-* (e.g., guidance-input-collection.template.md)
├─ The scaffold/skeleton of a new artifact?
│  → *-structure (e.g., promptengineering-context-structure.template.md)
├─ A reusable prose pattern for content creation?
│  → pattern-* (e.g., pattern-key-takeaways.template.md)
└─ None of the above?
   → Reconsider — templates should fit a category
```

### Category-Specific Constraints

| Category | Max lines | Consumer | Must include |
|---|---|---|---|
| `output-*` | 100 | Agent | All fields the downstream consumer needs |
| `input-*` | 50 | User | Field descriptions, examples, validation hints |
| `guidance-*` | 80 | Agent | Decision points, phase gates, ordered steps |
| `*-structure` | 60 | Agent | Required sections, naming patterns, YAML skeleton |
| `pattern-*` | 40 | User | Before/after examples, voice guidance |

## Composition Patterns

### Single-Purpose Templates

Each template MUST serve exactly one purpose. If a template grows beyond its category budget, decompose:

```
❌ output-full-review-report.template.md (200 lines)

✅ output-review-summary.template.md (40 lines)
✅ output-review-findings.template.md (60 lines)
✅ output-review-recommendations.template.md (40 lines)
```

### Consumer Chain Awareness

Templates often participate in multi-step workflows where one template's output feeds another's input. When creating a template, verify the chain:

```
input-requirements.template.md
    ↓ (agent fills in during research phase)
guidance-analysis.template.md
    ↓ (agent produces output using)
output-analysis-report.template.md
    ↓ (downstream consumer reads)
```

**Chain validation rule**: The output template MUST include every field that the downstream consumer expects. Missing fields cause the next phase to hallucinate or fail.

### Template References in Consumers

Templates are referenced via `📖` markers — never embedded inline:

```markdown
✅ **📖 Output format:** `.github/templates/00.00-prompt-engineering/output-prompt-validation-phases.template.md`

❌ (embedding 50 lines of template content inline in the agent file)
```

## Anti-Patterns

| Anti-pattern | Fix |
|---|---|
| **Embedding inline** | Reference via `📖`, load on-demand |
| **Generic placeholders** (`[value]`) | Use descriptive: `[One-sentence purpose]` |
| **Missing consumer comment** | Add `<!-- Used by: ... -->` at top |
| **Over-sized (>100 lines)** | Split into sub-templates |
| **Root-level placement** | Move to area subfolder |
| **Prose in agent templates** | Replace with tables, lists, field markers |
| **Missing chain validation** | Trace consumer chain end-to-end |

---

## Token Impact

Templates load on-demand — cost incurred only when referenced. Split templates >100 lines into sub-templates. If a template has 5+ consumers, every line saved reduces system-wide token consumption.

### Deprecate

| Trigger | Process |
|---|---|
| Template replaced by a better alternative | Add `<!-- DEPRECATED: Use [replacement] instead. Remove after YYYY-MM-DD -->` |
| All consumers switched to a different template | Verify zero `📖` references remain → remove the file |
| Category/naming conventions changed | Rename (preserving content) or replace and deprecate the old name |

**Deprecation rules:**
- 30-day grace period before removal (per `01.06-system-parameters.md`)
- Deprecated templates MUST still pass validation during the grace period
- Consumers referencing a deprecated template get a HIGH-severity finding in healthchecks
