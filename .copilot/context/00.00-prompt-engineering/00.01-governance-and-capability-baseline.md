# Prompt Engineering Governance and Capability Baseline

**Purpose**: North star governance document defining WHAT the PE system must do and WHY. All other tiers validate against this. Also serves as a functional capability map documenting the use cases the system supports, artifact chains that implement each, and verification criteria for capability preservation.

**Authority**: This is Tier 0 — the most stable layer. Changes to this document require explicit user approval. Meta-operations (Tier 5) validate against this, not the other way around.

**Referenced by**: `meta-validator` (capability preservation checks), `meta-designer` (change impact analysis), `meta-prompt-engineering-update` (regression detection)

---

## System Purpose

This PE system provides a structured, multi-agent framework for creating, reviewing, maintaining, and improving GitHub Copilot customization files (prompts, agents, instructions, skills, hooks, templates, snippets, MCP servers). Its patterns are portable to any Copilot-enabled workspace.

### Design goals

1. **Quality through orchestration** — Multi-agent workflows (researcher → builder → validator) produce higher-quality artifacts than single-pass creation
2. **Self-improving** — Meta workflows let the system incorporate new technologies, models, and platform capabilities without manual rewrites
3. **Discoverable** — Every capability has a clear entry point (prompt command or agent mention) documented in the capability baseline below
4. **Consistent** — Shared context files, instruction files, and validation patterns enforce uniform quality across all artifact types
5. **Token-efficient** — Progressive disclosure, tiered context, and caching patterns minimize context window usage

---

## Quality Criteria

The PE system's effectiveness is measured across three dimensions:

| Dimension | Definition | Threshold |
|---|---|---|
| **Effectiveness** |  Follow all applicable rules and pass validation, Artifacts achieve their stated goal — prompts, agents, and skills produce the intended outcome when executed against realistic scenarios | All validator agents report PASS on critical checks, Use case challenges pass (3–5 scenarios per artifact); orchestrated workflows reach their goal without manual correction |
| **Reliability** | Artifacts produce consistent, predictable results across repeated executions — no drift, no skipped steps, no hallucinated procedures. Artifacts are internally coherent — no ambiguities, no contradictions between rules, boundaries, or referenced context. Orchestrated workflows reach the same outcome regardless of model variability | Same input yields structurally equivalent output across runs; no contradictions between artifact sections or cross-referenced files; orchestrator phases don't diverge from their defined scope; no manual correction needed to compensate for drift or ambiguity |
| **Efficiency** | AI handles analysis, judgment, and creative decisions; deterministic logic runs as code (hooks, scripts, tasks). Context files stay within token budgets and don't duplicate information across artifacts | Deterministic operations use hooks/scripts, not AI reasoning; all context files ≤ 2,500 tokens; no duplicate rules across files; workflows don't re-read information already in context |

## Stability Rules

These rules protect the governance layer from uncontrolled changes:

1. **Capability removal** — Removing a use case from the baseline requires explicit user approval. Meta-agents MUST NOT remove capabilities autonomously.
2. **Governance changes** — Modifications to this file's System Purpose, Quality Criteria, or Stability Rules sections require user confirmation.
3. **Principle changes** — Changes to Tier 1 foundation principles that would invalidate existing artifacts require a migration plan.
4. **Baseline updates** — Adding new capabilities or updating artifact chains is encouraged and doesn't require approval (expansion is safe; contraction is not).

## Validation Contract

Meta-operations MUST verify against this document at these checkpoints:

| Checkpoint | Who | What to verify |
|---|---|---|
| Before proposing changes | `meta-designer` | Every capability in the baseline will still be supported after the change |
| After applying changes | `meta-validator` | Every artifact chain still resolves end-to-end |
| During healthcheck | `meta-validator` | No capability has degraded (broken chains, missing files, weakened boundaries) |
| During performance optimization | `meta-optimizer` | Token savings don't remove information required by any capability chain |

---

## How to Use This Baseline

### During updates (meta-designer, meta-optimizer)

Before proposing changes, check: "Will every use case in this baseline still be supported after the change?" If a change removes or weakens a use case, it MUST either:
1. Provide a replacement that supports the same use case more effectively, OR
2. Flag the capability loss to the user for explicit approval

### During validation (meta-validator)

After applying changes, verify: "For each use case, does the artifact chain still resolve?" Specifically:
- Every agent in the chain exists and has non-empty content
- Every handoff target in the chain exists
- Tool alignment is preserved (plan = read-only, agent = read-write)
- The use case can still be invoked by the user via the documented entry point

### After healthcheck (meta-validator, ecosystem audit mode)

Scan this baseline against the actual artifact state. Flag:
- CRITICAL: A use case's artifact chain is broken (missing agent, empty file, broken handoff)
- HIGH: A use case's artifact chain works but has degraded quality (weaker boundaries, missing sections)
- MEDIUM: A use case is supported but via a suboptimal path (e.g., generic agent instead of dedicated one)

## Maintaining This Baseline

### When to Update

- After creating a new orchestrator prompt (add use case to capability map)
- After creating a new skill (add to Category 5)
- After adding a new meta-prompt mode (add to Category 4)
- After deprecating an artifact (remove or update affected use cases)

### Who Updates

- `meta-designer` — proposes baseline updates as part of change specifications
- `meta-optimizer` — updates after applying optimizations
- Manual — when creating new workflows outside the meta-prompt pipeline
