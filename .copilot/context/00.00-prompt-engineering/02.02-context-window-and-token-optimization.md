# Context Window Management and Token Optimization

**Purpose**: Defines context rot, failure modes, information flow patterns, and nine actionable optimization strategies for reducing token consumption in multi-agent workflows.

**Referenced by**: All orchestrator prompts, agent files with handoffs, `.github/instructions/pe-prompts.instructions.md`

---

## Context Rot

<mark>Context rot</mark> is the progressive degradation of model accuracy as the context window fills (📖 see [01.05-glossary.md](01.05-glossary.md) for full definition). Key research finding:

- At the accuracy degradation threshold (📖 see [01.06-system-parameters.md](01.06-system-parameters.md)), even capable models lose most of their ability to follow earlier instructions
- Earlier instructions progressively lose influence as conversation grows
- Every additional token actively degrades the model's ability to follow earlier instructions

**Mitigations:**
- Start new chat sessions frequently to reset the context window
- Clear context between workflow phases rather than carrying forward
- Transfer only essential information during handoffs
- Use right-sized models — smaller models hit thresholds faster
- **Plan-first, then execute in a fresh session:** agree the plan with a strong model, persist it to a file, then start a clean session (often a cheaper model) to execute — this resets accumulated context and avoids rot while preserving the agreed plan

**Note on 1M-token context windows.** Some frontier models (Claude Opus 4.7+, GPT-5.5+) now support 1M-token contexts in supported VS Code surfaces (1.122+). Larger windows raise the absolute degradation point but do NOT eliminate context rot — every token still consumes credits under GitHub Copilot's usage-based billing model. Treat 1M as headroom for irreducible long-context tasks, not as a target.

## Three Failure Modes

| Failure Mode | Symptom | Root Cause |
|---|---|---|
| **Context Loss** | Agent doesn't know about earlier decisions | Handoff didn't transfer critical context |
| **Context Bloat** | Responses become slow, expensive, or confused | Too much irrelevant context accumulated |
| **Context Conflict** | Agent receives contradictory instructions | Multiple sources provide conflicting guidance |

## Five Information Flow Patterns

| Strategy | Token Efficiency | Robustness | Complexity | Best For |
|---|---|---|---|---|
| **Full Context** (`send: true`) | Low | High | Low | Simple 2–3 phase workflows |
| **Progressive Summarization** | Medium | Medium-High | Medium | Multi-phase workflows (3–5) |
| **File-Based Isolation** | High | Medium | Medium | Specialized single-purpose agents |
| **User-Mediated Handoff** | Maximum | Variable | High (user effort) | Maximum control needed |
| **Structured Report Passing** | Medium-High | High | Medium | Complex orchestrations |

### Token Impact Comparison

```
Full Context (send: true):           ~20,500 tokens across all phases
Progressive Summarization:           ~9,800 tokens (52% reduction)
File-Based Isolation:                ~3,400 tokens (83% reduction)
```

### Strategy Selection

| Scenario | Recommended Strategy |
|---|---|
| Simple 2–3 phase workflow | Full context (`send: true`) |
| Multi-phase (3–5 phases) | Progressive summarization |
| Long workflow (5+ phases) | File-based isolation |
| Maximum control needed | User-mediated handoff |
| Complex data contracts | Structured report passing |
| Mixed requirements | Combine strategies per phase |

## Context Window Breakdown (Optimization Targets)

Typical distribution showing where optimization effort SHOULD focus:

| Category | Typical % | Optimization Target |
|---|---|---|
| System context | 5–15% | Instruction file pruning, narrow `applyTo`, agent streamlining |
| User context | 5–20% | Prompt compression, snippet elimination |
| Tool results | 20–60% | **Primary target**: targeted reads, result limits, deterministic tools |
| Conversation history | 10–50% | Progressive summarization, file-based isolation |

**Focus on tool results and conversation history first** — they represent 70%+ of typical context.

### Always-on context carries a recurring cost

`copilot-instructions.md`, `AGENTS.md`, and broadly-scoped instruction files are injected on **every** interaction and re-billed on **every** agent step, so their cost is continuous rather than one-time. Two consequences:

- Keep always-on context to durable, high-value rules ("landmines only"); push optional or task-specific guidance into prompt-snippets (`#file:`) or narrowly-scoped `applyTo` instructions.
- Treat LLM-generated `/init`-style boilerplate with caution — auto-generated repo context often inflates token cost without improving (and can degrade) correctness. Curate it by hand.

📖 Minimization rules: [01.01-context-engineering-principles.md](./01.01-context-engineering-principles.md) (instruction minimization, single source of truth).

## MCP Communication Flow

MCP tools add tokens to the context window via tool results, following the same accumulation patterns as built-in tools. Agent-level MCP config uses `mcp-servers:` YAML syntax.

**Tool-surface tax and audit habit.** Beyond their results, every enabled tool/MCP server injects its schema into context — roughly 100–500 tokens per tool, re-counted on each agent step. Audit the active surface regularly: disable unused MCP servers and extensions, use a focused profile, and prefer a scoped custom agent that declares only the **minimal necessary tool set**. 📖 Per-tool cost analysis and minimal-surface guidance: [01.04-tool-composition-guide.md](./01.04-tool-composition-guide.md).

**📖 MCP design patterns:** [03.04-mcp-server-design-patterns.md](./03.04-mcp-server-design-patterns.md)

## Prompt-Snippets vs Instructions

Snippets add tokens only when explicitly referenced via `#file:`. Instructions always add tokens when `applyTo` matches. Use snippets for optional context; instructions for mandatory standards.

**📖 Full comparison:** [01.02-prompt-assembly-architecture.md](./01.02-prompt-assembly-architecture.md) → Prompt-Snippets

## Architectural Boundaries

VS Code tasks (`.vscode/tasks.json`) are NOT accessible to agents/prompts. Use `run_in_terminal` for equivalent outcomes.

## Nine Token Optimization Strategies

Without optimization, a 5-phase workflow consumes ~14,300 tokens (9.5× single request).

### Strategy Overview

| # | Strategy | Category | Savings | Effort | Best For |
|---|---|---|---|---|---|
| 1 | **Context Reduction** | Input | 30–90% | Low | All workflows |
| 2 | **Provider Prompt Caching** | Input | 50–90% | Low | High-volume, consistent prompts |
| 3 | **Semantic Caching** | Input | 50–80% | High | Repeated queries |
| 4 | **Model Selection** | Processing | 60–80% | Low | Simple tasks, high volume |
| 5 | **Batch Processing** | Processing | 50% cost | Low | Non-urgent, high volume |
| 6 | **Request Consolidation** | Processing | 40–60% | Medium | Multi-step pipelines |
| 7 | **Output Token Reduction** | Output | 30–50% | Low | Verbose outputs |
| 8 | **Deterministic Tools** | Output | 70–100% | Medium | Cache checks, validation, file ops |
| 9 | **Streaming/Parallelization** | Output | Latency only | Low–Medium | User-facing, independent tasks |

### Provider Caching Comparison

**Rule:** Place static instructions first, dynamic context last to maximize cache hits.

**📖 Provider thresholds table (min tokens, cache duration, read discount, write cost):** [01.06-system-parameters.md](./01.06-system-parameters.md) → "Provider Caching" (single source of truth).

**Cache invalidation:** tool definition changes, feature toggles, image changes, any prefix content edit.
**Anthropic break-even:** savings start at the **3rd reuse** (1.25 + 0.1 + 0.1 = 1.45 vs 3.0 without caching).
**Batch + Cache discounts stack:** 50% + 90% = **95% combined savings**.
**Cache stability within a session:** switching the model, the active MCP set, or the agent/profile mid-thread invalidates the cached prefix and forces a full re-read. Keep `{model, active MCP set, active agent}` stable inside a long thread to preserve the cached-input discount; when you must switch lanes, start a fresh chat with a short handoff summary rather than reconfiguring in place (📖 [03.02-model-specific-optimization.md](./03.02-model-specific-optimization.md)).

### Deterministic Tools Decision

Use deterministic tools (not AI) for predictable operations (YAML parsing, file existence checks, timestamp comparison, pattern counting). Use AI for code semantics, creative content, and judgment calls.

### Implementation Priority

1. Context reduction (targeted reads) — 30–50% savings, low effort
2. Provider caching (prompt structure) — 50–90% savings, low effort
3. Model selection (right-size tasks) — 60–80% savings, low effort
4. Deterministic tools (cache checks) — 70%+ for cached ops, medium effort
5. Output reduction (structured output) — 30–50% savings, low effort

**Context rot warning:** At the accuracy degradation threshold (📖 see [01.06-system-parameters.md](01.06-system-parameters.md)), optimization preserves accuracy, not just cost.

## Runtime Token-Discipline Habits

These are authoring and session habits that complement the nine strategies. They are defaults, not absolute rules — clarity still wins on trade-offs.

- **Terse output by default.** Generated tokens typically cost several times more than input tokens, so request code-only responses for code tasks and bullets over paragraphs for explanations unless detail is asked for. Extends Strategy 7 (Output Token Reduction).
- **Right-size the interaction mode.** Use a single-turn *Ask* for simple questions; reserve *Agent* mode for genuinely multi-step tasks — the agent loop adds tool-schema and reasoning overhead on every step.
- **Convert rich files to Markdown before ingestion.** `.docx`, `.pdf`, `.pptx`, `.xlsx`, and full HTML carry layout and markup that inflate tokens and confuse extraction; convert to Markdown first for chat, agent, or RAG input.

### Out of scope for this system

The community guidance that motivated these habits also proposes techniques this system deliberately does **not** adopt:

- **Aggressive prompt compression / "caveman-speak."** Rejected: it conflicts with the boundary *"prioritize clarity over token minimization."* Clarity wins even when compression would save tokens.
- **Language token-efficiency switching.** Not applicable: repo content is English-only.
- **CLI-only tooling (`/chronicle` cost tips, CodeAct) and enterprise FinOps / UBB budgets.** Out of scope for this personal repo's authoring workflow.

## Deterministic Context Tracking

For orchestrations with 5+ phases, track accumulated context deterministically:

1. After each phase: count files read, estimate tokens (`files_read × avg_lines × 6.67`)
2. Track running total across phases
3. **≤10,000 tokens** → continue; **>10,000** → MUST summarize prior phases; **>20,000** → MUST use file-based state + new session
4. For 5+ phase workflows: mandate file-based isolation at midpoint (`.copilot/temp/pe-meta-state/`)
