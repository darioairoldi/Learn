# pe-meta Review Dimension Catalog

## Canonical reference form

The canonical reference form for any dimension outside this catalog is `D#-readable-id` (for example `D6-consistency`, `D12-staleness`, `D26-model-routing`). The numeric prefix is preserved for grep-based tooling and contract continuity; the kebab-case suffix is the value from the `ID` column of the dimension table below.

- Combined form (preferred citation): `D6-consistency`
- Bare numeric form (accepted as input, e.g. for `--dim`): `D6`
- Bare readable form: NOT used (always carry the `D#` prefix)
- Ranges: rewrite as explicit lists (`D6-consistency through D11-actionability`) or as the dimension-group name when the row's intent is the group

Historical changelog entries that describe a past state may retain the bare `D#` form.

*`D17-cross-coherence` for context files = peer-review mode (checks against peer context files loaded by same consumers)

> **`Deterministic?` granularity (vision v15.5 § `dimension-rule-self-containment`).** The `Deterministic?` value above is **dimension-level**. A dimension marked `⚠️ Partial` (e.g. `D5-boundaries`, `D14-craftsmanship`) is *composite* — its sub-checks split into deterministic and reasoning rows, now enumerated per-sub-check with their own `Deterministic?` flag in [05.08-pe-meta-type-checklists.md](05.08-pe-meta-type-checklists.md). A composite dimension's verdict consolidates all its sub-checks: it is `verified` only when every applicable sub-check is evaluated and passes, and `partial` when any declared sub-check went unevaluated (see the [evidence-coverage contract](../../../.github/prompt-snippets/pe-meta-evidence-coverage.md) § graded verdict).

## Dimension groups (shortcuts)

### Which `--dim` group should I pick?

| Your intent | Group | Writes changes (`--mode apply`)? |
|---|---|---|
| Audit everything | `--dim full` (default) | Yes |
| Is the generated artifact **good**? (goal alignment, correctness, craft) | `--dim quality` | Yes |
| Make it **cheaper/faster** to run (low latency + token cost) | `--dim efficiency` | Yes — apply delegates to `@meta-optimizer` |
| Will it **behave the same** every run? | `--dim reliability` | Yes |
| Does it still serve the **long-term vision**? (rare, high-level) | `--dim strategic` | Yes |

> **Coverage guarantee.** The four primary groups — `quality`, `efficiency`, `reliability`, `strategic` — are a **disjoint partition**: every one of the 35 dimensions belongs to exactly one of them, and their union is the complete set. `--dim full` runs that union. The narrower groups below (`structural`, `freshness`, `adherence`, `model`, `context-full`, `context-health`) are convenience **subsets** that overlap the primary four for targeted runs.

> **Artifact-level vs runtime efficiency (scope boundary).** The `--dim efficiency` dimensions and `@meta-optimizer` optimize PE **artifacts** (token budgets, tool surface, references, handoffs, model routing). Runtime and user-session token habits — interaction-mode sizing, converting rich files to Markdown, and within-session cache stability — are **out of the optimizer's mandate** and live as authoring guidance in [02.02-context-window-and-token-optimization.md](02.02-context-window-and-token-optimization.md) → "Runtime Token-Discipline Habits". `D4` and `D20` cover the always-on context and tool-surface costs an artifact can be measured against.

### Primary groups (disjoint partition — together cover all 35)

| Group | Dimensions | Purpose |
|---|---|---|
| `--dim full` | All 35 (`D1-metadata` through `D35-portability-boundary`) | Complete audit (default if no `--dim` specified) — the union of the four primary groups |
| `--dim quality` | `D1-metadata`, `D2-references`, `D5-boundaries`, `D6-consistency`, `D7-non-redundancy`, `D8-prioritization`, `D9-clarity`, `D10-completeness`, `D11-actionability`, `D12-staleness`, `D13-source-verification`, `D14-craftsmanship`, `D16-adherence`, `D17-cross-coherence`, `D18-coverage`, `D19-artifact-structure`, `D22-context-optimization`, `D27-model-adherence` | **Is the generated artifact good?** Goal alignment, correctness, completeness, consistency, freshness, craft, and correct implementation of its dependencies. Includes the six prerequisite guidance-quality properties (`D6`–`D11`) |
| `--dim efficiency` | `D3-token-budget`, `D4-tool-alignment`, `D20-token-chain`, `D21-deterministic-first`, `D23-reference-efficiency`, `D24-handoff-efficiency`, `D25-processing-efficiency`, `D26-model-routing` | **Does it run economically?** Low latency + token efficiency. With `--mode apply`, apply work delegates to `@meta-optimizer` |
| `--dim reliability` | `D28-reproducibility` through `D35-portability-boundary` | **Is the behaviour repeatable?** Reproducibility, regression protection, metadata guards, validation invariants, rollback readiness, boundary actionability, autonomy calibration, portability boundaries (goal-trio reliability pole) |
| `--dim strategic` | `D15-vision-alignment` | **Does it still serve the vision?** The single high-level check that loads the vision document; expensive, run rarely |

### Convenience groups (subsets of the primary four — for targeted runs)

| Group | Dimensions | Use when |
|---|---|---|
| `--dim structural` | `D1-metadata` through `D5-boundaries`, `D14-craftsmanship` | Quick structural pass — cheapest useful review (draws from `quality` + `efficiency`) |
| `--dim freshness` | `D12-staleness`, `D13-source-verification` | Staleness and source verification (subset of `quality`) |
| `--dim adherence` | `D5-boundaries`, `D6-consistency`, `D16-adherence`, `D18-coverage` | Boundaries + consistency + adherence + coverage (subset of `quality`). Canonical name as of v13 |
| `--dim model` | `D26-model-routing`, `D27-model-adherence` | Model routing (`efficiency`) + model-specific adherence (`quality`) |
| `--dim context-full` | `D1-metadata` through `D3-token-budget`, `D6-consistency` through `D15-vision-alignment`, `D17-cross-coherence`, `D19-artifact-structure`, `D22-context-optimization` | Full context file review (16 applicable dims) |
| `--dim context-health` | `D6-consistency` through `D12-staleness`, `D22-context-optimization` | Quick context quality + staleness + organization |

## Dimension-to-handler mapping

| Type | Handler | Why |
|---|---|---|
| Deterministic (`D1-metadata` through `D4-tool-alignment`) | pe-meta-validator with deterministic tools | Parse YAML, count lines, resolve paths |
| Structural (`D5-boundaries`, `D14-craftsmanship`) | pe-meta-validator with standard model | Pattern matching against known structures |
| Semantic (`D6-consistency` through `D11-actionability`, `D16-adherence` through `D19-artifact-structure`) | pe-meta-validator with reasoning model | Requires understanding meaning |
| External (`D12-staleness`, `D13-source-verification`) | pe-meta-researcher with `fetch_webpage` | Requires accessing external sources |
| Cross-artifact (`D17-cross-coherence`, `D20-token-chain`) | pe-meta-validator in dependency-aware mode | Requires reading multiple files |
| Efficiency (`D21-deterministic-first`, `D23-reference-efficiency` through `D25-processing-efficiency`) | pe-meta-validator with standard model | Analyzing reference patterns and process phases |
| Model (`D26-model-routing`, `D27-model-adherence`) | pe-meta-validator with reasoning model | Comparing patterns against model-specific guidance |
| System-level (`D22-context-optimization`) | pe-meta-validator in system mode | Requires reading ALL context files |
| Reliability (`D28-reproducibility` through `D35-portability-boundary`) | pe-meta-validator with reasoning model + deterministic helpers | `D30-metadata-guard` uses deterministic parsers; `D28-reproducibility`/`D29-regression-protection`/`D31-multipass-validation-invariant` through `D35-portability-boundary` require reasoning over process invariants |
| Efficiency apply (`--dim efficiency --mode apply`) | pe-meta-validator (review) → @meta-optimizer (apply) | Review scores the efficiency dims; apply delegates the token/latency optimizations to the optimizer agent |
