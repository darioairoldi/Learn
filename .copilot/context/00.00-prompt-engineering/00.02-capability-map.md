## Capability Preservation Verification

### Quick Verification (run after any PE change)

- **Categories 1-2**: All entry points invocable and each command file resolves on disk (artifact chains are read from each prompt's `handoffs:` on demand — not stored in this map)
- **Category 3**: All standalone entry points invocable, inline validation present
- **Category 4**: All 3 meta-prompt modes functional, feedback loop works
- **Category 5**: Both skills discoverable, resource files exist

### Deep Verification (run before/after a full apply-mode review)

1. Trace each command's `handoffs:` on demand — verify handoff targets exist (chains are not stored in this map)
2. Verify tool alignment — plan=read-only, agent=read+write
3. Check boundary completeness — 3/1/2 minimums
4. Test skill discoverability via description keywords
5. **P0/P1 implementer coverage (coverage-gap guard)** — for every vision `scope.covers` capability tagged **P0** or **P1**, confirm a matching use-case row above exists and its command entry point resolves on disk (a cheap deterministic **existence lint**). A P0/P1 capability with no row or an unresolvable command is a **CRITICAL** gap (a promised capability with no implementer). Deeper handoff-chain resolution is covered separately by tracing each prompt's `handoffs:` on demand (chains are no longer stored here — B0, 2026-06-25). This check is **central by design**: the capability→command mapping lives ONLY here and in the vision — artifacts do NOT carry a per-file `implements:` back-reference (that would create a second source of truth to drift). The `pe-meta-review` Phase 7b check 6 enforces this guard at runtime.

### Regression Detection

When a change removes/modifies an artifact: check which use cases depend on it, verify chains still resolve. If broken, MUST provide replacement OR get user approval.
