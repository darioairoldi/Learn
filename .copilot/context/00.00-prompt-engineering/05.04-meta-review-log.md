# PE Meta-Review Log

**Purpose**: Canonical audit trail for PE system meta-reviews. Records when each meta-workflow mode was executed, what external sources were processed, and what changes resulted. Enables the session-start hook and scheduled review prompts to detect whether a review is overdue or a source has already been analyzed.

**Referenced by**: `pe-staleness-check` hook, `meta-prompt-engineering-scheduled-review` prompt, `meta-prompt-engineering-update` prompt, `meta-validator` agent

---

## How to Use This Log

### Reading (staleness detection)

1. Check the `last_updated` date in the YAML frontmatter — if >30 days old, a meta-review is overdue
2. Find the relevant mode in the review history below — check whether recent external sources have been processed
3. The session-start hook (`pe-staleness-check.json`) automates step 1 on every session

### Writing (after a meta-review)

After completing a meta-review, append a new entry to the appropriate mode section using this format:

```markdown
### YYYY-MM-DD — [mode] [scope]

- **Source**: [what triggered or was analyzed]
- **Dimensions**: [which dimensions were checked]
- **Findings**: [count] issues ([count] CRITICAL, [count] HIGH, [count] MEDIUM, [count] LOW)
- **Changes applied**: [brief list of files modified, or "None (plan only)"]
- **Health score**: [before] → [after] (or "N/A" for first run)
```

After appending, update the `last_updated` field in the YAML frontmatter to today's date.
