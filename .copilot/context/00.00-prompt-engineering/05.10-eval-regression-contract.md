# Eval and regression contract

## Purpose

Specify the fixed, versioned **golden cases** that make the engine's effectiveness claim falsifiable: before a self-update is promoted, the candidate version is scored against the same cases as the incumbent, and a regression on any graded dimension blocks auto-promotion. This closes gap **G6** (no eval harness / regression suite). It is a **contract only** — the harness build is parked.

## Referenced by

- **PE vision** — the `progressive-learning` principle (the gate that decides whether v(N+1) is promoted)
- **`05.05-practical-effectiveness-log.md`** — human-reported outcomes; this contract is the machine-graded counterpart
- **`05.07-pe-meta-dimension-catalog.md`** — the canonical scoring axes every eval case must reference
- **`05.09-self-update-config-and-state.md`** — the state namespace where golden cases and run results are versioned

## Eval case schema

Spec-now / build-later; consumers MUST preserve unknown keys (forward-compat via `case_version`).

| Field | Type | Purpose |
|---|---|---|
| `case_version` | int | Contract version for forward-compat (REQUIRED). |
| `artifact_type` | enum | What is being graded (`prompt` / `agent` / `instruction` / `context` / `template` / …). |
| `dimension` | ref | The graded axis — MUST be an id from `05.07-pe-meta-dimension-catalog.md`. |
| `input` | object | The fixed golden input (task + context fixture). |
| `expected` | object | Golden output OR a checkable criteria set (exact, contains, rubric-score ≥ threshold). |
| `baseline_score` | number | The incumbent's recorded score — the value a candidate must meet or beat. |

## Coverage expectations

- Every **artifact type** the engine maintains SHOULD have ≥1 golden case per **dimension** it is graded on (the catalog defines which dimensions apply per type).
- Cases are **fixed**: changing a golden input/output is a deliberate, logged baseline reset — never an incidental edit during a self-update.

## Regression-gate semantics

1. On a candidate self-update, run all applicable cases for the affected artifact(s) at v(N) and v(N+1).
2. **Promote-eligible** only if no case regresses below its `baseline_score` on any graded dimension.
3. Write the run (scores + decision) to the `outcome-log` in the state namespace; `progressive-learning` consumes it.
4. A regression ⇒ `propose`/`escalate` (never silent auto-promote), surfacing the failing case to the human.

## Relationship to the effectiveness log

`05.05-practical-effectiveness-log.md` records *what humans observed after the fact*; this contract records *what fixed golden cases measure before promotion*. Both feed `progressive-learning`: the log catches gaps the goldens miss, the goldens prevent regressions the log would only notice later.

## Status

**Spec-now / build-later.** This file defines the contract; the golden dataset, runner, and scoring rubrics are parked (see the plan's § Park lot). Until then, effectiveness rests on structural validation + the human-reported log only.
