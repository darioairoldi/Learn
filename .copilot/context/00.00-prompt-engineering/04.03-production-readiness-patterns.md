# Production Readiness Patterns

**Purpose**: Defines the production-readiness requirements that every prompt and agent file MUST implement to ensure reliable, professional behavior in real-world usage. Based on Mario Fontana's ["6 VITAL Rules for Production-Ready Copilot Agents"](https://www.linkedin.com/learning/mastering-ai-agents-the-prompt-engineering-masterclass).

**Referenced by**: `pe-prompts.instructions.md`, `pe-agents.instructions.md`, `prompt-create-update`, `agent-create-update`, `prompt-builder`, `agent-builder`, `prompt-validator`, `agent-validator`

## The 6 Production-Readiness Requirements

### 1. Response management (data gaps handling)

**Rule** (REQUIRED):
- Every prompt and agent MUST define professional responses when information is missing. The model MUST NOT hallucinate, guess, or proceed with invented data.

**Rationale**: Unhandled data gaps are the primary cause of hallucination — explicit response templates force the model to acknowledge gaps instead of inventing answers.

**Required response template:**

```markdown
## Response Management

### When information is missing
I couldn't find [specific] in [locations searched].
I did find [partial context] in [location].
Recommendation: [escalation path or alternative]
```

**Required scenarios** (customize per artifact type):
- Missing context/information
- Ambiguous requirements (present options, don't guess)
- Tool failures (fallback behavior)

**Type-specific scenarios:**

| Artifact type | Additional scenarios |
|---|---|
| Orchestration prompts | Agent not found, phase validation failure, handoff ambiguity |
| Implementation prompts | Pattern not found, file access failure, convention unclear |
| Analysis prompts | No matches found, contradictory sources, insufficient data |
| Validation prompts | Missing metadata, ambiguous rules, caching issues |

### 2. Error recovery workflows

**Rule** (REQUIRED):
- Every prompt and agent MUST define fallback behavior for critical tools. The model MUST NOT proceed with invented data or half-implemented changes.

**Rationale**: Tool failures are common in multi-step workflows — without explicit fallbacks, failures cascade silently into incorrect outputs.

**Required pattern:**

```markdown
### When tool failures occur

- `semantic_search` returns nothing → Try grep_search, then escalate
- `read_file` fails → Verify path, report error with context
- `create_file` fails → Check permissions, suggest manual creation
```

**Golden rule:** NEVER proceed with invented data or half-implemented changes.

### 3. Embedded test scenarios

**Rule** (REQUIRED):
- Every prompt and agent MUST include test scenarios to validate behavioral reliability.

**Rationale**: Test scenarios catch role confusion, boundary violations, and hallucination patterns before the artifact is used in production.

**Minimum counts:**
- Prompts: 3–5 test scenarios
- Agents: 3 test scenarios (less complex role)

**Required test types:**

| Test type | Purpose | Pass criteria |
|---|---|---|
| **Happy path** | Expected success scenario | Produces correct output |
| **Ambiguous input** | Vague/conflicting requirements | Asks clarifying questions (doesn't guess) |
| **Missing context (plausible trap)** | Incomplete data that looks plausible | Detects gap, reports what's missing, doesn't hallucinate |
| **Out of scope** | Request beyond capability boundaries | Professional refusal with redirect |

**Test scenario template:**

```markdown
### Test N: [scenario name]
**Input:** [description]
**Expected:** [behavior]
**Pass criteria:** [specific outcomes]
```

### 4. Token budget compliance

**Rule** (REQUIRED):
- Every prompt and agent MUST respect type-specific token limits to prevent context rot.

**Rationale**: Exceeding token budgets degrades model accuracy (context rot) and increases cost — budgets are the primary defense against both.

**Prompt token budgets:**

| Prompt type | Max tokens |
|---|---|
| Simple validation | 500 |
| Multi-step workflow | 1,500 |
| Multi-agent orchestrator | 2,500 |

**Agent token budgets:**

| Agent type | Max lines | Max tokens (est.) |
|---|---|---|
| Builder/Researcher | 150 | ~2,000 |
| Validator | 200 | ~2,500 |
| Meta agent | 250 | ~3,000 |

**Optimization techniques:**
- Reference context files instead of embedding principles
- Use imperative language (no filler)
- Place critical instructions in first 30% of the prompt
- Factor large prompts into multiple smaller, focused prompts
- Structure with **static content first, dynamic last** to enable provider prompt caching

**📖 Detailed strategies:** [02.02-context-window-and-token-optimization.md](02.02-context-window-and-token-optimization.md)

### 5. Context rot prevention

**Rule** (REQUIRED):
- Long conversations accumulate context that degrades model accuracy. Every multi-phase prompt MUST implement mitigation:

**Rationale**: Accuracy drops from 88% to 30% at 32K tokens (Liu et al. 2023) — proactive mitigation is cheaper than debugging degraded outputs.

- Start new chat sessions frequently to reset the context window
- Transfer only essential information during handoffs
- Use right-sized models — smaller models hit context thresholds faster
- Progressive summarization for multi-phase workflows (3–5 phases)
- File-based isolation for long workflows (5+ phases)

**📖 Detailed patterns:** [02.02-context-window-and-token-optimization.md](02.02-context-window-and-token-optimization.md)

### 6. Template externalization

**Rule** (REQUIRED):
- Any inline content block exceeding 10 lines MUST be externalized to a template file. This prevents token bloat and enables reuse across consumers.

**Rationale**: Inline blocks >10 lines consume tokens on every load — externalizing them saves tokens for all consumers except the one that actually needs the content.

**📖 Complete guidance:** [01.01-context-engineering-principles.md](01.01-context-engineering-principles.md) → Principle 8

**📖 Template design patterns:** [03.07-template-authoring-patterns.md](03.07-template-authoring-patterns.md)

## Validation Checklist

Validators MUST verify these requirements against every prompt and agent file:

| # | Requirement | Check | Severity |
|---|---|---|---|
| 1 | Response management | Section present with data gap scenarios | HIGH |
| 2 | Error recovery | Fallback defined for critical tools | HIGH |
| 3 | Embedded tests | Minimum count met (3–5 prompts, 3 agents) | HIGH |
| 4 | Token budget | Within type-specific limit | CRITICAL |
| 5 | Context rot prevention | Multi-phase prompts use summarization or isolation | MEDIUM |
| 6 | Template externalization | No inline blocks >10 lines | MEDIUM |
