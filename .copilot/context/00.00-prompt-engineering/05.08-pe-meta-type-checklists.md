# pe-meta Per-Type Validation Checklists

## How to use

The pe-meta-validator loads this file and dispatches to the appropriate section based on the target artifact's file path:
- `.copilot/context/` → Context file checklist
- `.agent.md` → Agent checklist
- `.prompt.md` → Prompt checklist
- `.instructions.md` → Instruction file checklist
- `SKILL.md` → Skill checklist
- `.template.md` → Template checklist
- `.json` in `hooks/` → Hook checklist
- `prompt-snippets/` → Snippet checklist

> **`Deterministic?` column + sub-check granularity (per `dimension-rule-self-containment`, vision v15.5).** In every table below, each row is one **sub-check**, and the `Deterministic?` column records whether that sub-check is mechanically computable (✅ Yes), needs reasoning (❌ No), or is a composite/partial judgment (⚠️ Partial). A dimension's verdict consolidates *all* its rows — it is `verified` only when every applicable row is evaluated and passes; an unevaluated row makes the dimension `partial` (see [evidence-coverage contract](../../../.github/prompt-snippets/pe-meta-evidence-coverage.md) § graded verdict). A composite dimension (e.g. `D5`, `D14`) splits across deterministic and reasoning rows, so its dimension-level `⚠️ Partial` in [05.07](05.07-pe-meta-dimension-catalog.md) is decomposed here.

> **Metadata field placement (per [00.03-metadata-contracts](00.03-metadata-contracts.md)).** Where a checklist row references `version:` or `last_updated:`, evaluate those fields against the artifact's **bottom `{type}_metadata` HTML-comment block** — NOT the top frontmatter. PE artifacts carry change-prone tracking fields (`version`, `last_updated`, `created`, optional `changelog:`) only in the bottom block; finding them in top frontmatter is itself a defect. **Exception:** vision documents (`06.00-idea/**/*vision*.md`) and use-case documents (`**/*usecases/**`) keep `version`/`last_updated` in top frontmatter because their amendment protocols read it there.

| Check | Standard bar | Exemplary bar (PE-for-PE) | Deterministic? | Dimension |
|---|---|---|---|---|
| `description:` present | Required | Required | ✅ Yes | D1-metadata |
| `applyTo:` glob pattern | Required | Required | ✅ Yes | D1-metadata |
| `version:` SemVer | Recommended | Required | ✅ Yes | D1-metadata |
| `goal:` single sentence | Recommended | Required | ✅ Yes | D1-metadata |
| `rationales:` list | Optional | Required (≥1) | ✅ Yes | D1-metadata |
| All `📖` refs resolve | Required | Required | ✅ Yes | D2-references |
| Token count ≤1,500 | Required | Required | ✅ Yes | D3-token-budget |
| Rules are testable/mechanical only (instruction-minimization) | Required | Required | ⚠️ Partial | D14-craftsmanship |
| No behavioral/strategic rules (instruction-minimization) | Required | Required | ⚠️ Partial | D14-craftsmanship |
| Severity index present *(see scope note)* | Recommended | Required | ✅ Yes | D14-craftsmanship |

> **Severity-index scope note.** The severity index is a *validator-facing projection of the global `validation-rules` matrix* onto the artifact type an instruction file governs (its entries reference global codes `[C#]`/`[H#]`/`[M#]`). It therefore applies **only to instruction files that govern PE artifacts validated by that matrix** (prompts, agents, context files, skills, hooks, prompt-snippets, templates, instruction files). For **document-governance instruction files** — those whose `applyTo` targets user-content documents that no PE validator runs the matrix against (e.g. `use-case-documents`, `vision-amendment`, `vision-frontmatter`) — the severity index is **N/A**: their rules are single-file-class document-shape rules with no representation in the global matrix, and adding one would either project 2–3 barely-applicable codes or manufacture new severity codes. Treat such files as **standard bar** and do not flag a missing severity index.
