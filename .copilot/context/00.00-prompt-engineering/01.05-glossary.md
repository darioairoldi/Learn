# PE System Glossary

**Purpose**: Single source of truth for term definitions used across all PE artifacts. Context files, agents, and prompts MUST reference this file instead of defining terms inline.

**Referenced by**: All context files, instruction files, agents, prompts, skills

---

## Core Concepts

| Term | Definition |
|---|---|
| **Context window** | The total text (system prompt + user messages + tool results + conversation history) visible to the model during a single request. Fixed-size per model. |
| **Context rot** | Progressive degradation of model accuracy as the context window fills. Earlier instructions lose influence as token count grows. See `01.06-system-parameters.md` for the measured accuracy-drop threshold. |
| **Context loss** | Failure mode where an agent doesn't know about earlier decisions because the handoff didn't transfer critical information. |
| **Context bloat** | Failure mode where too much irrelevant context accumulates, making responses slow, expensive, or confused. |
| **Context conflict** | Failure mode where an agent receives contradictory instructions from multiple sources. |
| **Token** | The atomic unit of text processing for LLMs. Roughly 0.75 words or 4 characters in English. |

## Artifact Types

| Term | Definition |
|---|---|
| **Prompt file** | A `.prompt.md` file that provides task-specific instructions, injected into the USER prompt layer. Triggered by `/command`. Single execution scope. |
| **Agent file** | An `.agent.md` file that defines a persona with default tools and behavior, injected into the SYSTEM prompt layer. Triggered by `@mention`. Session-wide scope. |
| **Instruction file** | An `.instructions.md` file with an `applyTo` glob pattern, auto-injected into the SYSTEM layer when matching files are active. Always-on within scope. |
| **Skill file** | A `SKILL.md` file that packages domain knowledge with progressive disclosure (discovery → instructions → resources). AI-discovered via description keywords. |
| **Context file** | A `.md` file in `.copilot/context/` that provides shared reference knowledge, discovered via semantic search. On-demand loading. |
| **Prompt-snippet** | A `.md` file in `.github/prompt-snippets/` manually included via `#file:` reference. Adds tokens only when explicitly referenced. |
| **Template file** | A `.template.md` file providing structured output formats consumed by agents or users. |
| **Hook** | A `.json` configuration in `.github/hooks/` that triggers deterministic automation at agent lifecycle events. |
| **MCP server** | A Model Context Protocol server providing tools, resources, and prompts to the model via a standardized transport. |

## Agent Roles

| Term | Definition |
|---|---|
| **Researcher** | A `plan`-mode agent that analyzes requirements, validates scope, and produces research findings. Read-only tools. |
| **Builder** | An `agent`-mode agent that creates or modifies files based on research findings or change specs. Read+write tools. |
| **Validator** | A `plan`-mode agent that certifies artifacts against quality criteria (effectiveness, reliability, efficiency). Read-only tools. |
| **Orchestrator** | A prompt that coordinates a multi-agent workflow by invoking researchers, builders, and validators in sequence. |

## Agent Modes

| Term | Definition |
|---|---|
| **Plan mode** | Agent runs with read-only tools. Used for researchers and validators. Cannot modify files. |
| **Agent mode** | Agent runs with read+write tools. Used for builders. Can create, edit, and delete files. |

## Handoff Concepts

| Term | Definition |
|---|---|
| **Handoff** | Transfer of control and context from one agent to another in an orchestrated workflow. |
| **Send mode** (`send: true`) | Handoff strategy where the entire conversation history transfers to the next agent. High token cost, high robustness. |
| **Label mode** (`send: false`) | Handoff strategy where nothing transfers automatically — the next agent starts fresh. Requires explicit context passing. |
| **Reliability checksum** | A pre-handoff validation pattern that ensures five critical elements survive the transfer: goal preservation, scope boundaries, tool requirements, critical constraints, and success criteria. |
| **Progressive summarization** | Information flow strategy where each phase summarizes its output before passing to the next, reducing token accumulation. |
| **File-based isolation** | Information flow strategy where agents write results to files instead of passing context directly, minimizing token transfer. |

## Quality Dimensions

| Term | Definition |
|---|---|
| **Effectiveness** | Artifacts achieve their stated goal — produce the intended outcome when executed against realistic scenarios. |
| **Reliability** | Artifacts produce consistent, predictable results with no drift, no ambiguities, and no contradictions across repeated executions. |
| **Efficiency** | AI handles analysis and judgment; deterministic logic runs as code. Context stays within token budgets with no duplication. |

## Validation Concepts

| Term | Definition |
|---|---|
| **Use case challenge** | A validation technique where 3–5 realistic scenarios test whether an artifact achieves its goal under varied conditions. |
| **Boundary actionability** | Whether each rule in an artifact's Always/Ask/Never boundaries is unambiguously testable by AI. |
| **Tool alignment** | The requirement that plan-mode agents have read-only tools and agent-mode agents have read+write tools. |
| **Validation caching** | A policy where validation results are cached and reused if content hasn't changed within a defined period. See `01.06-system-parameters.md` for the cache duration. |

## Tier Architecture

| Term | Definition |
|---|---|
| **Tier 0 (Governance)** | North star layer defining system purpose, capability requirements, and quality criteria. Most stable — changes require user approval. |
| **Tier 1 (Foundations)** | Core principles, architecture, file type decisions, and tool composition. High impact — changes affect all other tiers. |
| **Tier 2 (Multi-Agent)** | Orchestration patterns — handoffs, context management, design patterns. |
| **Tier 3 (Specialized)** | Platform-specific features — skills, models, hooks, MCP, Spaces, SDK, templates. |
| **Tier 4 (Conventions)** | Validation, caching, production readiness, and runtime validation patterns. |
| **Tier 5 (Meta-Ops)** | System self-improvement — dependency map, lifecycle management, workflow entry points. |

## Progressive Disclosure

| Term | Definition |
|---|---|
| **Level 1 (Discovery)** | The SKILL.md `description` field — always visible to AI for routing decisions. Minimal token cost. |
| **Level 2 (Instructions)** | The SKILL.md body content — loaded when AI decides the skill is relevant. Moderate token cost. |
| **Level 3 (Resources)** | Supporting files referenced from SKILL.md — loaded on-demand during execution. Variable token cost. |
