# PE-for-PE Strategic Review Criteria

**Purpose**: Strategic review guidance for PE artifacts that serve the PE system itself. Supplements structural rules (from instruction files) with vision alignment, quality bar, and self-update readiness checks.

**Referenced by**: `pe-meta-review` prompt, `pe-meta-design` prompt, `pe-meta-create-update` prompt, `pe-meta-validator` (ecosystem audit)

---

## When to use this file

Use when creating, reviewing, or updating artifacts in:
- `.github/agents/00.0x-pe-*` (PE agents)
- `.github/prompts/00.0x-pe-*` (PE prompts)
- `.copilot/context/00.00-prompt-engineering/` (PE context files)
- `.github/instructions/pe-*` (PE instruction files)
- `.github/skills/pe-*` (PE skills)
- `.github/templates/00.00-prompt-engineering/` (PE templates)

Do NOT use for domain artifacts (article-writing, documentation, etc.) — those need domain expertise activation, not vision alignment.

## Vision alignment checklist

For each PE artifact, verify alignment with applicable vision rationales. Not all rationales apply to all types.

| Rationale | Agents | Prompts | Context | Instructions | Skills | Templates |
|---|---|---|---|---|---|---|
| **metadata-driven** | goal, scope, boundaries, rationales in YAML | goal, scope in YAML | goal, scope, boundaries, rationales in YAML | applyTo, description | description in YAML | template_metadata |
| **chain-alignment** | Category refs (L1.5) for context file references | Category refs for context file references | Inter-file refs stay file-specific (peers) | Category refs for context pointers | Category refs if loading context | N/A |
| **structural-separation** | Rule-bearing boundary sections should use Rule/Rationale/Example | N/A (prompts orchestrate, don't define rules) | MUST use for rule-bearing sections | MUST use for rule-bearing sections | Should use for checklist items | N/A |
| **role-declaration** | description + goal clearly state what and why | description + goal | Purpose section + goal | description + applyTo scope | description | Purpose clear from filename |
| **decomposition** | ≤7 tools, handoffs for scope overflow | Phase isolation, summarization protocol | ≤2,500 tokens, split if exceeded | ≤1,500 tokens | Progressive disclosure (3 levels) | N/A |
| **self-correction** | Handoff to validator after build | Validation phase in workflow | Reviewed by context-validator | Reviewed by instruction-validator | Reviewed by skill-validator | Reviewed by template-validator |
| **model-routing** | Process phases specify model class (reasoning/standard/deterministic) | Phase steps specify model class | N/A | N/A | N/A | N/A |

**How to use**: For a given artifact, read down the column for its type. Each cell describes what compliance looks like. Missing compliance = finding.

## Three review modes

pe-meta supports three review modes, each answering a different question:

| Mode | Invocation | Question | Cost |
|---|---|---|
| **Individual** | `/pe-meta-{type}-review <file>` | "Is this artifact correct?" | O(1) |
| **Dependency-aware** | `/pe-meta-{type}-review <file> --with-deps` | "Does this + its dependency chain produce reliable behavior?" | O(d) |
| **Guidance-first** | `/pe-meta-adherence <guidance-file>` | "Do consumers implement this file's rules?" | O(c) |

For selective dimension targeting, use `--dim <group>`. Dimension catalog: load `pe-dimension-catalog` category from `.copilot/context/00.00-prompt-engineering/`.

## PE-for-PE quality bar

PE artifacts that build, validate, or manage other PE artifacts MUST be **exemplary** — they set the standard that every other artifact follows.

### What "exemplary" means

| Dimension | Standard (domain artifacts) | Exemplary (PE-for-PE) |
|---|---|---|
| **Metadata** | goal + scope recommended | goal + scope + boundaries + rationales REQUIRED |
| **Boundaries** | ≥3 Always / ≥1 Ask First / ≥2 Never | ≥5 Always / ≥2 Ask First / ≥3 Never, with specific line-item detail |
| **References** | File-level OK if descriptive | Category-based (Level 1.5) REQUIRED for context file references |
| **N-1 separation** | Recommended for rule sections | REQUIRED for all rule-bearing sections |
| **Test scenarios** | ≥3 scenarios | ≥3 scenarios covering happy path + error + edge case |
| **Response management** | List common error scenarios | List + specific recovery action for each |
| **Handoff contracts** | Describe what's sent | Table with Include/Exclude/Max tokens per transition |
| **Version tracking** | version + last_updated | version + last_updated + changes list in bottom metadata |

### Why the higher bar

PE-for-PE artifacts are the **reference implementation**. When pe-con-builder creates a new agent, it follows the patterns it sees in existing PE agents. If pe-meta-validator has sloppy boundaries, every validator it produces will copy that sloppiness. The quality bar compounds through the artifact chain.

## Category reference enforcement

### When Level 1.5 (category) references are required

| Reference type | Level required | Example |
|---|---|---|
| Loading context files for validation/research | **Level 1.5** (category) | `Load the \`validation-rules\` files from .copilot/context/00.00-prompt-engineering/` |
| Loading a specific template | **Level 2** (file) — templates are singletons | `output-researcher-report.template.md` |
| Loading the dispatch table | **Level 2** (file) — singleton | `artifact-type-dispatch.template.md` |
| YAML `context_dependencies` | **Level 1** (folder) | `"00.00-prompt-engineering/"` |
| Handoff targets | **Level 2** (file) — must resolve to exact agent | `agent: pe-meta-validator` |
| `applyTo` patterns | **Level 2** (file pattern) — routing mechanism | `'.github/agents/**/*.agent.md'` |

### How to audit

1. Read the artifact's body for references to `.copilot/context/00.00-prompt-engineering/` files
2. For each reference, check: is it using a category ID or a filename?
3. If filename: is it a singleton (template, dispatch table, handoff target)? → Level 2 OK
4. If filename and NOT a singleton: → should be converted to category reference
5. Check 00.00-context-structure-index.md Functional Categories to verify the category ID exists

## N-1 adoption per artifact type

| Artifact type | N-1 applicable? | How to apply | Example |
|---|---|---|---|
| **Context files** | YES — all rule-bearing sections | Each section with MUST/NEVER/ALWAYS gets `**Rule**:` / `**Rationale**:` / `**Example**:` blocks | See `01.01-context-engineering-principles.md` |
| **Instruction files** | YES — all rule-bearing sections | Same pattern; rules are the testable checks | Validation checklist items as `**Rule**:` blocks |
| **Agent bodies** | PARTIAL — boundary sections only | Always Do / Ask First / Never Do sections don't need N-1 (they ARE the rules). Process sections with embedded rules should use N-1 | Process phase with "MUST load dispatch table" → make it a labeled rule |
| **Prompts** | NO — prompts orchestrate, not define rules | Prompts reference rules from context/instruction files. Process steps are workflow, not rules | N/A |
| **Skills** | PARTIAL — checklist items | Checklist items that are enforceable rules should use N-1 | `**Rule**: MUST have goal: field` |
| **Templates** | NO — templates define structure, not rules | Placeholder conventions are structural, not rule-bearing | N/A |
| **Hooks** | NO — JSON config, not prose rules | Hook configs are schema-validated, not prose | N/A |

## Self-update readiness checklist

Each vision rollout phase has preconditions. An artifact is "ready for Phase N" when it meets that phase's requirements.

| Phase | What's autonomous | Readiness criteria |
|---|---|---|
| **Phase 1** | Templates, snippets, hooks | ✅ File exists with valid schema. No prose rules to classify. |
| **Phase 2** | Skills, agent tool/handoff lists | ✅ Phase 1 criteria + structured metadata (goal, scope, version). Diffs are deterministic (YAML field changes). |
| **Phase 3** | Agent bodies, context files, instructions | ✅ Phase 2 criteria + N-1 structural separation in all rule-bearing sections + rationales metadata. Enables deterministic breaking/non-breaking classification. |
| **Phase 4** | Cross-tier refactors, file moves | ✅ Phase 3 criteria + all references use Level 1.5 categories + rename log operational. References survive file reorganization. |

### How to assess an artifact's readiness

1. Identify the artifact's type → determines which phase applies
2. Check the readiness criteria for that phase
3. If criteria not met → the artifact will require human approval for changes that could otherwise be autonomous
4. Report the gap as a finding with severity based on how many criteria are missing

## Context file autonomy routing

Context file changes follow the same risk-calibrated gradient as all artifacts. The foundation-layer position increases **validation depth**, not the autonomy threshold for validated non-breaking changes.

| Change type | Autonomy level | Pre-conditions |
|---|---|---|
| Metadata fixes (version, last_updated) | Autonomous | Deterministic, no content change |
| Reference fixes | Autonomous | Deterministic, no guidance change |
| Wording improvements preserving same guidance | Autonomous with notification | Pre-change guard + construction invariants + `D17-cross-coherence` peer coherence pass |
| Non-redundancy refactoring | Autonomous with notification | Pre-change guard + no rule lost + consumers still covered |
| New coverage within declared scope | Autonomous with notification | Pre-change guard + `D6-consistency` + `D17-cross-coherence` peer coherence pass |
| Rule modification changing behavior | Human approval required | Breaking for consumers |
| Scope expansion/contraction | Human approval required | Changes artifact contract |
| File split/merge/reorganization | Human approval required | Affects category mapping |

**Validation-reduced escalation**: Changes that pass pre-change guard + construction invariants + peer coherence + consumer test scenarios can be reclassified from escalation to autonomous.
