# PE Metadata Contracts

**Purpose**: Canonical schema for PE artifact metadata. Every PE artifact MUST carry these fields to enable deterministic change classification, drift detection, scope preservation, and runtime self-constraint enforcement.

**Referenced by**: All builder agents, all validator agents, all create-update prompts, `pe-meta-validator`, `pe-meta-optimizer`

---

## Required fields per artifact type

All fields go in **top-level YAML frontmatter** alongside existing fields.

### All artifact types (universal)

| Field | Required | Format | Purpose |
|---|---|---|---|
| `goal:` | ✅ MUST | Single sentence | The one outcome this artifact exists to achieve |
| `scope:` | ✅ MUST | `{covers: [...], excludes: [...]}` | Topics covered and explicitly excluded |
| `boundaries:` | ✅ MUST | List of strings | Constraints the artifact operates within (MUST/NEVER statements) |
| `rationales:` | SHOULD | List of strings | Design constraints that explain why key decisions must be preserved |
| `version:` | ✅ MUST | SemVer (`"1.0.0"`) | Patch = non-breaking fix, minor = additive, major = breaking |

### Type-specific required fields (in addition to universal)

| Artifact type | Additional required fields | Notes |
|---|---|---|
| **Context** | `title:`, `description:`, `last_updated:`, `domain:` | Already present in most files |
| **Instruction** | `description:`, `applyTo:`, `last_updated:`, `domain:` | `applyTo` is the glob pattern trigger |
| **Agent** | `description:`, `agent:`, `tools:`, `capabilities:`, `domain:` | `agent: plan\|agent` is the mode |
| **Prompt** | `name:`, `description:`, `agent:`, `tools:`, `domain:` | `name:` is the slash-command name |
| **Skill** | `name:`, `description:`, `domain:` | `description:` drives AI discovery |
| **Template** | `description:`, `domain:` | Minimal — templates are consumed, not invoked |
| **Hook** | N/A (JSON files) | Hooks use JSON schema, not YAML |

### `domain:` field semantics

The `domain:` field identifies the **one semantic domain the artifact belongs to**. It is a single scalar string (e.g. `domain: "prompt-engineering"`, `domain: "article-writing"`, `domain: "learning-hub"`).

- **Single-valued by design.** List form (`domain: [a, b]`), plural variant (`domains:`), and primary/secondary pairs are NOT supported. An artifact that *feels* like it belongs to multiple domains usually has dependencies that span domains — captured at the **dependency-closure level**, not in the artifact's own metadata.
- **Cross-domain dependencies are first-class.** When a single-domain artifact's `--deps full` closure spans additional domains, the orchestrator emits `bundle=cross-domain-deps` and runs one review with **per-dependency-domain specialized analysis lenses** (see vision v15 § Domain detection § Seed footprint vs dependency footprint). The artifact stays single-domain in its metadata; the cross-domain nature emerges from the closure walk — NOT from a list-valued field.
- **Reserved value.** `domain: "unknown"` is reserved as the Tier-3 fallback in the metadata-first 3-tier resolution algorithm. Artifacts MUST NOT declare `unknown` as their semantic domain; doing so collides with the fallback signal.
- **Authoritative for Phase 0b.** The orchestrator's Phase 0b reads this field per-file via Tier-1 of the resolution algorithm; Tier-2 (`pe-domain-map.yaml` heuristic) and Tier-3 (`unknown`) apply only when the field is absent.

> **Future extension (NOT yet implemented — do not use):** A list-valued `additional_domains:` field has been *considered* for genuinely cross-cutting artifacts (e.g. a skill that serves two domains as a peer to each) but is **not supported today**. Until a future phase introduces it, authors MUST declare a single `domain:` and rely on the cross-domain-deps closure (above) for multi-domain coverage. Do not add `additional_domains:` to any artifact.

## Metadata-guarded change protocol

Every artifact change — whether autonomous or human-initiated — follows this two-step protocol.

### Pre-change guard (default: BLOCK)

Before applying any change, compare its effect against declared metadata:

| Check | Condition | Default action |
|---|---|---|
| Contradicts `goal:` | Change alters the artifact's stated purpose | **BLOCK** — requires explicit human override |
| Contradicts `scope.covers:` | Change removes a covered topic | **BLOCK** — potential coverage regression |
| Contradicts `scope.excludes:` | Change adds a topic in the exclusion list | **BLOCK** — scope creep |
| Contradicts `boundaries:` | Change violates a declared constraint | **BLOCK** — boundary violation |
| Contradicts `rationales:` | Change invalidates a design decision | **ESCALATE** — replacement rationale required |

### Post-change reconciliation (MANDATORY)

After every change is applied, update metadata to reflect the new state:

1. **`goal:`** — Still accurate? Update if purpose shifted.
2. **`scope.covers:`** — Added coverage? Add topic. Removed coverage? Flag as regression.
3. **`scope.excludes:`** — Still accurate? Update if exclusions changed.
4. **`boundaries:`** — New constraints? Relaxed constraints? Update.
5. **`rationales:`** — Invalidated? Replace with new rationale text. Added? Include.
6. **`version:`** — Bump: patch (non-breaking), minor (additive), major (breaking).

## Validation rules

Validators MUST check these conditions:

| Rule | Severity | Check |
|---|---|---|
| `goal:` present and non-empty | **CRITICAL** | YAML field exists, is a single sentence |
| `scope.covers:` present and non-empty | **CRITICAL** | YAML field exists, is a non-empty list |
| `scope.excludes:` present | **HIGH** | YAML field exists (empty list acceptable) |
| `boundaries:` present and ≥1 item | **HIGH** | YAML field exists, has at least 1 boundary |
| `rationales:` present | **MEDIUM** | YAML field exists (Phase 1: recommended; Phase 2+: required) |
| `version:` present and valid SemVer | **CRITICAL** | Matches `N.N.N` pattern |
| Metadata reflects content | **HIGH** | `scope.covers` topics match actual content sections |
| Agent three-tier boundaries align with YAML `boundaries:` | **HIGH** | Each YAML boundary has a corresponding Always Do / Never Do entry |
| Prompt workflow includes scope enforcement | **MEDIUM** | Process steps include scope/boundary verification before proceeding |
| No body-level constraint contradicts YAML metadata | **HIGH** | Body instructions don't override or weaken YAML boundaries |

## Runtime grounding protocol

The metadata-guarded change protocol (above) protects artifacts during **modification**. The runtime grounding protocol protects artifacts during **execution** — ensuring the executing model enforces its own declared constraints.

### What runtime grounding means

| YAML field | Runtime enforcement |
|---|---|
| `goal:` | Model verifies each request contributes to the declared goal before proceeding |
| `scope.covers:` | Model confirms the request falls within covered topics |
| `scope.excludes:` | Model rejects requests matching excluded topics with redirection |
| `boundaries:` | Model enforces all MUST/NEVER constraints throughout execution |

### Per-type grounding patterns

| Artifact type | Grounding mechanism | Notes |
|---|---|---|
| **Agent** | Three-tier boundaries section (Always Do / Ask First / Never Do) reflects YAML boundaries | Ensure three-tier content aligns with YAML `boundaries:` |
| **Prompt** | Workflow includes scope enforcement step: "Verify request falls within declared scope before proceeding" | Lightweight — one process step, not a full section |
| **Context** | N/A — reference material, not independently executed | No grounding needed |
| **Instruction** | N/A — injected into agent/prompt context, not independently executed | No grounding needed |
| **Skill** | N/A — discovery metadata, grounding handled by consuming agent | No grounding needed |
| **Template** | N/A — output scaffold, not independently executed | No grounding needed |

### Grounding alignment rule

For agents and prompts: body-level enforcement content MUST be consistent with YAML metadata. If `boundaries:` declares "MUST verify handoff targets exist" then the body workflow MUST include a step that performs handoff target verification.

**Violation indicator:** A boundary exists in YAML but has no corresponding enforcement in the body workflow.

**This is NOT duplication.** The body references metadata constraints operationally (e.g., "verify request falls within declared scope") — it doesn't restate the full scope list. YAML remains the single source of truth; the body is the enforcement mechanism.

## Examples

### Context file

```yaml
---
title: "Context engineering principles"
description: "Consolidated best practices for crafting effective instructions, prompts, and agent personas"
version: "2.2.0"
last_updated: "2026-03-19"
domain: "prompt-engineering"
goal: "Provide the single source of truth for context engineering principles across all PE file types"
scope:
  covers:
    - "8 core principles (narrow scope, early commands, imperative language, boundaries, context minimization, tool scoping, uncertainty, templates)"
  excludes:
    - "Tool-specific composition rules (see 01.04-tool-composition-guide.md)"
    - "Model-specific optimization (see 03.02-model-specific-optimization.md)"
boundaries:
  - "MUST NOT duplicate tool rules from 01.04"
  - "MUST stay under 2,500 tokens"
rationales:
  - "Principles are foundational (Tier 1) — changes here cascade to all consumers"
---
```

### Agent file

```yaml
---
description: "Research specialist for prompt requirements"
agent: plan
tools: [semantic_search, grep_search, read_file, file_search, list_dir, fetch_webpage]
version: "1.0.0"
last_updated: "2026-03-20"
domain: "prompt-engineering"
goal: "Produce a research report that enables a builder to create the prompt without ambiguity"
scope:
  covers:
    - "Use case challenge validation"
    - "Pattern discovery from existing prompts"
    - "Requirements clarification"
  excludes:
    - "File creation or modification"
    - "Validation (delegated to validator)"
boundaries:
  - "MUST NOT create or modify files — read-only"
  - "MUST challenge every purpose with ≥3 use cases"
rationales:
  - "Read-only mode prevents research from having side effects"
---
```
