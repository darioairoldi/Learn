# PE Workflow Entry Points

**Purpose**: Decision guide for choosing the right prompt or agent when working with prompt engineering artifacts — differentiating standalone prompts from orchestrator prompts and clarifying when to use each.

**Referenced by**: All PE artifact creators, meta-review prompts, users starting PE workflows

Consumption status: Active workflow guidance (not parked). Primary consumers include [pe-meta-update.prompt.md](../../../.github/prompts/00.09-pe-meta/pe-meta-update.prompt.md), [pe-meta-review.prompt.md](../../../.github/prompts/00.09-pe-meta/pe-meta-review.prompt.md), [pe-meta-design.prompt.md](../../../.github/prompts/00.09-pe-meta/pe-meta-design.prompt.md), and [pe-meta-create-update.prompt.md](../../../.github/prompts/00.09-pe-meta/pe-meta-create-update.prompt.md).

## Quick Decision: What Do You Want to Do?

```
What do you want to do?
│
├─ CREATE a new artifact?
│  ├─ PE-for-PE artifact (pe-* agents, PE context files, pe-* prompts) → /pe-meta-design
│  ├─ Any type, single entry point → Consolidated Prompt (/pe-con-design)
│  ├─ Complex/uncertain requirements → Granular Orchestrator Prompt (Phase A)
│  └─ Clear requirements, known pattern → Standalone Prompt (Phase B)
│
├─ REVIEW/VALIDATE an existing artifact?
│  ├─ PE-for-PE artifact → /pe-meta-review (vision + structural + ecosystem)
│  ├─ Any type, single entry point → /pe-con-review
│  └─ Type-specific review → Validation Prompt (Phase C)
│
├─ UPDATE/FIX an existing artifact?
│  ├─ PE-for-PE artifact → /pe-meta-create-update (pre-change guards + strategic validation)
│  ├─ Any type, single entry point → /pe-con-create-update
│  ├─ From validation report → Agent directly (Phase D)
│  └─ Ad hoc change → Standalone Prompt in update mode (Phase B)
│
├─ Review the PE SYSTEM itself?
│  └─ → Meta-Prompt (Phase E)
│
└─ Not sure which artifact type to create?
   └─ → Read 01.03-file-type-decision-guide.md first
```

## Consolidated Prompts (Recommended Default)

**Use when**: You want a single entry point for any artifact type. Consolidated agents handle all 8 types via dispatch table.

| Workflow | Slash Command | Pipeline | When to Use |
|---|---|---|---|
| Design any artifact | `/pe-con-design <type> <desc>` | pe-con-researcher → pe-con-builder → pe-con-validator | Uncertain requirements, need research + challenge |
| Create/update any artifact | `/pe-con-create-update <type> <path>` | pe-con-builder → pe-con-validator | Clear requirements, skip research |
| Review any artifact | `/pe-con-review <type> <path>` | pe-con-validator (→ pe-con-builder for fixes) | Validate existing artifact |

**Supported types**: agent, prompt, context file, instruction file, skill, template, hook, prompt snippet

**Example:**
```
/pe-con-design agent for reviewing Quarto configuration
/pe-con-create-update hook for PreToolUse deny on .env
/pe-con-review context .copilot/context/00.00-prompt-engineering/02.04-agent-shared-patterns.md
```

## Common Scenarios

| Scenario | Entry Point |
|---|---|
| Any artifact, single entry point | `/pe-con-design <type> <desc>` or `/pe-con-create-update <type> <path>` |
| **PE-for-PE artifact (strategic)** | `/pe-meta-design <type> <desc>` or `/pe-meta-create-update <type> <path>` |
| **Review PE artifact against vision** | `/pe-meta-review <path>` |
| Quick scaffold (built-in, no PE system) | `/create-prompt`, `/create-instruction`, `/create-skill`, `/create-agent`, `/create-hook` |
| Simple prompt with clear requirements | `/pe-gra-prompt-create-update` |
| Complex multi-agent workflow | `/pe-gra-prompt-design` |
| Validate an existing artifact | `/pe-con-review <type> <path>` or `/pe-gra-{type}-review` |
| Fix a specific issue | `@pe-con-builder` or `@pe-gra-{type}-builder` directly |
| Document new knowledge | `/pe-con-create-update context <desc>` or `/pe-gra-context-information-create-update` |
| New domain with uncertain scope | `/pe-con-design context <desc>` or `/pe-gra-context-information-design` |
| System-level audit | `@pe-meta-validator` |
| Weekly maintenance | `/pe-meta-scheduled-review` |
| New VS Code release | `/pe-meta-release-monitor` |

## Built-in VS Code Creation Commands

VS Code provides built-in `/create-*` commands that scaffold customization files with AI assistance. These are **simpler alternatives** to the PE system's orchestrated workflows — useful for quick one-off scaffolds but without research, validation, or quality gates.

| Command | Creates | PE equivalent |
|---|---|---|
| `/create-prompt` | `.prompt.md` file | `/pe-con-create-update prompt` |
| `/create-instruction` | `.instructions.md` file | `/pe-con-create-update instruction` |
| `/create-skill` | `SKILL.md` + folder | `/pe-con-create-update skill` |
| `/create-agent` | `.agent.md` file | `/pe-con-create-update agent` |
| `/create-hook` | Hook config | `/pe-con-create-update hook` |
| `/init` | `copilot-instructions.md` | Manual creation |

**When to use built-in vs PE system**: Use built-in commands for exploratory or quick-and-dirty scaffolds. Use PE system commands when you need research-backed design, validation, and consistency with existing artifacts.

## Entry-point execution guarantees

### Orchestration phase sequencing

When an entry point triggers multi-family validation or updates, preserve this order:

1. Context files
2. Instruction files
3. Agent files
4. Template files
5. Prompt files
6. Hook and snippet files
7. Skill files

This prevents consumer-level checks from running against stale guidance foundations.

### Handoff failure recovery at entry points

If an entry-point workflow handoff fails (missing inputs, unresolved contradictions, or failed downstream gate), apply this fallback sequence:

1. Stop at the failing handoff boundary.
2. Emit explicit missing/conflicting fields.
3. Retry once after clarification or correction.
4. If still failing, escalate with residual blockers and recommended next action.

### Exemplary-bar enforcement timing

Entry points that create or update artifacts MUST enforce the exemplary bar at:

1. Pre-write guard (before changes land), and
2. Post-write validation (before marking workflow complete).

Use [05.02-artifact-lifecycle-management.md](05.02-artifact-lifecycle-management.md) and [05.08-pe-meta-type-checklists.md](05.08-pe-meta-type-checklists.md) as the enforcement anchors.

## Anti-Patterns

- **Orchestrator for simple tasks** — Use standalone prompts for known patterns
- **Standalone for novel workflows** — Use orchestrators for uncertain scope
- **Skipping validation** — Always validate after creation or update
- **Direct file editing** — Use builder agents for tracked, validated changes
