My colleague Vishnu and I are going to walk you through what's happening and what we are doing about it at Microsoft to solve for the problem of prompt caching. To set the stage, let's look at how inference works today without caching. Modern AI agents run on long, static enterprise context, system instructions, tool definitions, retrieval context. And they invoke these prompt contexts across millions of invocations. In most production agents, the static, unchanged portion of the prompt represents more than 60% of inference cost.
Here is why. On every turn, the GPU has to recompute the model attention over the entire prefix, the static system prompt, the tool definitions, and all of those prior turns. Turn two adds even more history on top of that. And then turn three adds even more. By the time you're a few turns in, the GPU is burning expensive cycles, re-dealing the same tokens it has already seen before. That recompute is what drives up Time to First Token, or as we call it, TTFT, and the end-to-end inference latency, and further, what makes long context agents so expensive to operate. Now let's apply prompt caching to the same picture. The idea is simple but very powerful. Instead of recomputing attention on every turn, we cache the KV state of the repeated context token and reuse it for the next turn. On turn 1, the GPU still processes the full prefix once, but as it does it, it also writes the attention state out as a context-cached object. Turn 2, the GPU doesn't redo that work. It just loads the cache state and only computes the attention on the new tokens for that turn.
Turn 3 and every turn after that looks exactly the same. The result is a dramatic reduction in GPU compute per turn. And customers can get inference cost savings up to 90% along with significantly lower time to first token. In fact, this is so impactful that model providers like OpenAI already offer implicit prompt caching APIs out of the box. That's a great starting point.
But for enterprise scenarios, our customers are building by using open source models or their own models, that alone is not enough. And that's exactly what Vishnu and I will talk about next.
