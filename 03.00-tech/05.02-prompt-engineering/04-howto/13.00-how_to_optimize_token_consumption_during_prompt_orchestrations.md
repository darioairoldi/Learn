# How to optimize token consumption during prompt orchestrations

> Every token you spend on overhead is a token you can't spend on actual work. In multi-agent orchestrations, token costs compound at every handoff, tool call, and context injection. This article teaches you how to keep those costs under control.

Up to this point in the orchestration sequence, you've learned how to [design orchestrator prompts](./10.00-how_to_design_orchestrator_prompts.md), [delegate to subagents](./11.00-how_to_design_subagent_orchestrations.md), and [manage information flow](./12.00-how_to_manage_information_flow_during_prompt_orchestrations.md). Each of those articles mentioned token costs — but none provided a systematic framework for minimizing them. This article fills that gap with nine concrete strategies, cost models, and practical patterns you can apply immediately.

## 🎯 Why token optimization matters

In a simple single-prompt interaction, token costs are straightforward — you send a message, the model responds, done. In multi-agent orchestrations, costs multiply through three mechanisms:

| Cost multiplier | How it works | Impact |
|----------------|-------------|--------|
| **System prompt repetition** | Instruction files, agent bodies, and tool schemas are re-sent with *every* request | Linear growth per exchange |
| **Context window accumulation** | Conversation history, tool results, and file contents pile up | Accelerating growth |
| **Handoff overhead** | Each agent transition carries context from the previous phase | Compounding cost |

A four-phase orchestration (research → plan → implement → review) with ten tool calls per phase can easily consume 80K–120K tokens — approaching the context window limit of GPT-4o (128K) in a single workflow run.

### The cost equation

```
Total tokens = Σ (system_prompt + conversation_history + tool_results + user_message)
               for each request in the session
```

The system prompt is the **constant tax** — paid on every request. Conversation history is the **growing tax** — paid and growing with each exchange. Tool results are the **variable tax** — unpredictable spikes from file reads, search results, and command output.

Your optimization goal: minimize all three taxes without sacrificing the quality of results.

## 📊 Understanding token costs in orchestrations

Before optimizing, understand where tokens go. Here's a typical breakdown for a multi-agent workflow:

| Component | Typical size | Frequency | Total cost pattern |
|-----------|-------------|-----------|-------------------|
| **Built-in system prompt** (layers 1–4) | 3K–5K tokens | Every request | Fixed overhead |
| **Custom instructions** (layer 5) | 500–2K tokens | Every matching request | Fixed per session |
| **Agent body** (layer 6) | 300–1.5K tokens | Every request while active | Fixed per agent |
| **Tool schemas** | 200 tokens × number of tools | Every request | Fixed per session |
| **Prompt file body** | 200–800 tokens | Once per invocation | One-time |
| **Tool call results** | 500–2K tokens per call | Per tool invocation | Variable, accumulating |
| **Conversation history** | Grows with each exchange | Every request | Accumulating |

### Token budget benchmarks

| Orchestration complexity | Typical total | Budget guidance |
|-------------------------|---------------|----------------|
| Simple (2 phases, 5 tool calls) | 15K–30K tokens | Any model |
| Medium (3–4 phases, 15 tool calls) | 40K–80K tokens | 128K+ context models |
| Complex (5+ phases, 30+ tool calls) | 80K–200K tokens | 200K+ context models; use subagents |

## 📐 Strategy 1: Right-size your prompts

The simplest optimization: write less. Every word in your prompt files, instruction files, and agent bodies is a token you pay on every request.

### Prompt file sizing

| Size | Lines | Tokens | Assessment |
|------|-------|--------|-----------|
| Lean | <80 | <500 | Ideal for focused tasks |
| Standard | 80–200 | 500–1,200 | Good for complex workflows |
| Heavy | 200–400 | 1,200–2,400 | Acceptable if template-externalized |
| Bloated | >400 | >2,400 | Split or externalize |

### Instruction file sizing

Instruction files are more expensive than prompt files because they're injected on *every matching request*, not just when invoked:

| Size | Tokens | Assessment |
|------|--------|-----------|
| Concise | <300 | Ideal |
| Acceptable | 300–600 | Review for trimming |
| Expensive | >600 | Split using `applyTo` patterns |

### Practical techniques

**Remove examples the model doesn't need.** If you're writing instructions for an experienced model like Claude Opus 4.6, you likely don't need three examples of every pattern. Start with zero-shot; add examples only if output quality suffers.

**Use references instead of inline content.** Instead of embedding a 200-line template in your prompt, reference it:

```markdown
<!-- ❌ Expensive: 200 lines inline -->
Here is the template to follow:
[... 200 lines of template ...]

<!-- ✅ Cheap: 1 line reference, loaded on demand -->
Use the template at `.github/templates/article.template.md`
```

**Leverage skills for resource-heavy workflows.** Skills load resources progressively — the SKILL.md body loads only when the description matches, and resource files load only when explicitly referenced. This is far cheaper than embedding everything in a prompt file.

## 📝 Strategy 2: Progressive summarization

When passing context between phases in an orchestration, don't pass raw data — pass summaries. Each phase should produce a condensed output that the next phase can work with.

### The summarization pattern

```
Phase 1: Research (reads 20 files, 50K tokens of raw data)
    │
    ▼ Summary: 500 tokens
Phase 2: Plan (reads summary + 5 targeted files)
    │
    ▼ Summary: 400 tokens
Phase 3: Implement (reads plan + specific code sections)
    │
    ▼ Summary: 300 tokens
Phase 4: Review (reads implementation summary + test results)
```

Without summarization, Phase 4 would inherit 50K+ tokens of accumulated context. With summarization, it gets ~1,200 tokens of focused summaries.

### Implementing summarization in handoffs

Use the handoff `prompt` field to carry only the summary:

```yaml
handoffs:
  - label: "Implement Plan"
    agent: implementer
    send: false
    prompt: |
      **Research findings summary:**
      - Key dependencies: React 18, TypeScript 5.3
      - Migration risk: HIGH (3 shared services affected)
      - Recommended approach: adapter pattern
      
      **Implementation plan:**
      1. Create adapter interface in src/adapters/
      2. Migrate auth module (src/auth/)
      3. Update shared services
      
      Execute this plan. The full research is in research-output.md if needed.
```

### Budget guidance

| Content type | Raw size | Target summary | Compression |
|-------------|----------|----------------|-------------|
| File analysis (per file) | 500–2K tokens | 50–100 tokens | 10–20× |
| Research phase output | 5K–20K tokens | 300–500 tokens | 15–40× |
| Implementation report | 3K–10K tokens | 200–400 tokens | 10–25× |

## 🔀 Strategy 3: Subagent isolation

Each subagent gets a <mark>fresh context window</mark> — it doesn't inherit the parent's conversation history, accumulated tool results, or context bloat. This isolation is the single most effective strategy for managing token costs in complex orchestrations.

### Why isolation helps

```
Without subagents (monolithic):
Request 1: 10K tokens (system + first message)
Request 5: 35K tokens (system + 4 exchanges of history)
Request 10: 70K tokens (system + 9 exchanges)
↳ Total: ~250K tokens for 10 exchanges

With subagents (isolated):
Main agent: 10K tokens (stays lean)
Subagent A: 15K tokens (fresh context, focused task)
Subagent B: 12K tokens (fresh context, different task)
Subagent C: 18K tokens (fresh context, complex task)
↳ Total: ~55K tokens for equivalent work
```

The savings come from two sources: each subagent starts fresh (no inherited history), and the main agent receives only a summary (not the full transcript).

### When to use subagents vs. inline work

| Factor | Inline (same context) | Subagent (isolated) |
|--------|----------------------|---------------------|
| Task requires prior context | ✅ Free access to history | ❌ Must pass context explicitly |
| Task is independent | ❌ Adds to growing context | ✅ Clean slate, no bloat |
| Result needed verbatim | ✅ Available directly | ❌ Only summary returns |
| Token budget tight | ❌ Context keeps growing | ✅ Isolated, predictable cost |

**Rule of thumb:** If a subtask doesn't need the last five exchanges of conversation to do its work, it should be a subagent.

## 📁 Strategy 4: File-based context offloading

Instead of passing large content through the context window, write intermediate results to files and let the next phase read only what it needs.

### The offloading pattern

```
Phase 1 (Research):
  ├─ Reads files, analyzes codebase
  ├─ Writes: .copilot/scratch/research-report.md (full details)
  └─ Passes to Phase 2: "Research complete. Full report at 
     .copilot/scratch/research-report.md. Key findings: [3 bullets]"

Phase 2 (Planning):
  ├─ Reads 3 key bullets from handoff (50 tokens)
  ├─ Reads targeted sections of research-report.md (200 tokens)
  └─ Never loads the 20 source files Phase 1 analyzed
```

### Token savings

| Approach | Tokens in context | Files in context |
|----------|------------------|-----------------|
| Full context pass | 20K–50K | All source files |
| File-based offload | 2K–5K | Only the summary file |
| **Savings** | **75–90%** | **Selective reads** |

### Implementation

Instruct your orchestrator to offload results:

```markdown
## Phase completion protocol

After each phase, BEFORE any handoff:
1. Write detailed results to `.copilot/scratch/{phase-name}-output.md`
2. Produce a summary (max 300 tokens) with:
   - Key decisions (bullet list)
   - File paths created or modified
   - Blockers or questions for the next phase
3. Pass ONLY the summary to the next phase
```

## 🗜️ Strategy 5: The `/compact` command

VS Code 1.110 introduced the <mark>`/compact` command</mark> — an explicit way to compress conversation history when the context window is getting full.

### How `/compact` works

When you type `/compact` in Copilot Chat:

1. VS Code sends the full conversation history to the model with a summarization instruction
2. The model produces a condensed summary of the conversation so far
3. The condensed summary replaces the full history in the context window
4. Subsequent requests use the summary instead of the raw history

### What survives compaction

| Content | Survives? | Notes |
|---------|-----------|-------|
| System prompt (instructions, agent) | ✅ | Re-injected fresh each request |
| Copilot Memory | ✅ | Injected independently of history |
| Recent messages | ✅ | Last 1–2 exchanges preserved |
| Earlier conversation | Summarized | Compressed into a summary |
| Tool call details | Summarized | Results condensed |

### When to use `/compact`

- When you notice the model "forgetting" earlier instructions (context rot)
- After a phase of heavy tool usage that generated large results
- When switching focus within the same session
- Proactively, every 8–10 exchanges in agent mode

### Limitations

- The summary may lose nuance from early exchanges
- Tool call specifics (exact file contents, search results) are condensed
- Not available in all execution contexts (check your VS Code version)

## 💾 Strategy 6: Prompt caching

<mark>Prompt caching</mark> is a provider-level optimization that avoids reprocessing static content across requests. When the beginning of your prompt matches a cached prefix, the provider charges reduced rates for the cached portion.

### How it works

```
Request 1:                          Request 2:
┌─────────────────────────┐         ┌─────────────────────────┐
│ System prompt (5K)      │ ──────► │ System prompt (5K) CACHED│
│ Instructions (1K)       │ ──────► │ Instructions (1K) CACHED │
│ Agent body (800)        │ ──────► │ Agent body (800) CACHED  │
├─────────────────────────┤         ├─────────────────────────┤
│ User message (200) NEW  │         │ User message (300) NEW   │
└─────────────────────────┘         └─────────────────────────┘
                                    
Total: 7K tokens                    Cached: 6.8K | New: 300
```

### Provider-specific cache economics

| Provider | Cache write cost | Cache read cost | Break-even |
|----------|-----------------|-----------------|------------|
| **OpenAI** | 1× (no surcharge) | 0.5× | 2nd use |
| **Anthropic** | 1.25× | 0.1× | 3rd use |
| **Google** | 1× (no surcharge) | 0.25× | 2nd use |

### Optimization: static content first

Caching works on prefixes — the static portion at the *beginning* of the prompt. To maximize cache hits, place stable content before dynamic content:

```markdown
<!-- ✅ Good: static first (cacheable) -->
## Instructions (static — 1,500 tokens, cached after first request)
[Detailed rules that don't change between requests]

## Current task (dynamic — changes each request)
${input}
```

```markdown
<!-- ❌ Bad: dynamic first (breaks caching) -->
## Current task: ${input}

## Instructions (static — never cached because dynamic content precedes it)
[Same rules, but now re-processed every request]
```

For detailed implementation patterns and pitfalls, see [Appendix: Token optimization patterns](./13.01-appendix_token_optimization_patterns.md).

## 🎯 Strategy 7: Prompt-directed attention

Instead of including all possible context and hoping the model finds what's relevant, explicitly direct the model's attention to specific locations.

### The attention pattern

```markdown
<!-- ❌ Vague: model reads everything, most of it irrelevant -->
Review the codebase for security issues.

<!-- ✅ Directed: model reads only what matters -->
Review these specific files for authentication vulnerabilities:
1. `src/auth/login.ts` (lines 45–120) — token validation logic
2. `src/middleware/auth.ts` (lines 10–50) — middleware chain
3. `src/utils/crypto.ts` (lines 1–30) — hashing functions

Focus on: SQL injection, token expiration, and password hashing.
Ignore: UI components, test files, and configuration.
```

### Token impact

| Approach | Files read | Tokens consumed | Accuracy |
|----------|-----------|----------------|----------|
| Undirected search | 15–30 files | 20K–50K | Medium (noise dilutes signal) |
| Directed attention | 3–5 files | 3K–8K | High (focused on relevant code) |

The directed approach uses 75% fewer tokens *and* produces better results because the model isn't distracted by irrelevant content.

## 🔧 Strategy 8: Tool call efficiency

Every tool call adds tokens to the context window — both the request and the result. Inefficient tool use is one of the largest sources of token waste.

### Common wasteful patterns

| Pattern | Waste | Fix |
|---------|-------|-----|
| Reading entire files | 500–2K tokens per file | Use line ranges: `startLine`/`endLine` |
| Broad `grep_search` | 50+ matches returned | Add specific patterns, use `maxResults` |
| Sequential small reads | Multiple round-trips | Read larger ranges in fewer calls |
| Redundant searches | Same query, different wording | Check previous results before searching |

### Efficient tool patterns

**Read targeted sections, not full files:**

```markdown
<!-- ❌ Expensive: reads entire 500-line file -->
Read the auth module.

<!-- ✅ Efficient: reads only the relevant 30 lines -->
Read src/auth/login.ts lines 45-75 to understand the token validation logic.
```

**Batch operations where possible:**

```markdown
<!-- ❌ 5 separate tool calls, 5 context entries -->
Replace line 10 in file A.
Replace line 20 in file A.
Replace line 30 in file A.

<!-- ✅ 1 tool call, 1 context entry -->
Use multi_replace_string_in_file to make all three changes at once.
```

### Token cost reference

| Tool | Typical result size | Optimization |
|------|-------------------|-------------|
| `read_file` (full) | 500–2K tokens | Use line ranges |
| `read_file` (targeted) | 100–300 tokens | Specify startLine/endLine |
| `grep_search` | 200–1K tokens | Use specific patterns, limit results |
| `semantic_search` | 500–2K tokens | Use precise queries |
| `run_in_terminal` | 100–5K tokens | Pipe through `Select-Object -First` |

## 📋 Strategy 9: Template externalization

When prompts reference templates, examples, or checklists, store them as separate files rather than embedding them inline. The model loads external files only when needed, while inline content is always present.

### The externalization pattern

```markdown
<!-- ❌ 800 tokens always present in every request -->
## Output Format
### Phase 1 Output
| Column | Description |
| ...800 tokens of template... |

### Phase 2 Output
| Column | Description |
| ...600 tokens of template... |
```

```markdown
<!-- ✅ 50 tokens in prompt; templates loaded on demand -->
## Output Format

📖 Use templates from `.github/templates/workflow/`:
- Phase 1: `phase-1-output.template.md`
- Phase 2: `phase-2-output.template.md`
```

### Savings estimation

| Content type | Inline cost | Externalized cost | Savings |
|-------------|------------|-------------------|---------|
| Output templates | 500–1,500 tokens/prompt | 50 tokens (reference) | 90–97% |
| Example blocks | 300–800 tokens | 30 tokens (reference) | 90–96% |
| Checklists | 200–500 tokens | 20 tokens (reference) | 90–96% |

The trade-off: the model makes an additional `read_file` call to load the template (~200–500 tokens for the result). But this happens once per workflow, while inline content is paid on *every request* in the session.

## ⚖️ Strategy selection guide

Not every strategy applies to every situation. Use this decision guide:

```
What's consuming the most tokens?
│
├─ System prompt (instructions + agent body too large)
│   └─ → Strategy 1 (right-size) + Strategy 9 (externalize)
│
├─ Conversation history (long sessions)
│   └─ → Strategy 5 (/compact) + Strategy 3 (subagents)
│
├─ Tool call results (many file reads, searches)
│   └─ → Strategy 8 (tool efficiency) + Strategy 7 (directed attention)
│
├─ Handoff overhead (multi-phase workflows)
│   └─ → Strategy 2 (progressive summarization) + Strategy 4 (file offloading)
│
└─ Repeated static content (same prefix every request)
    └─ → Strategy 6 (prompt caching)
```

### Strategy impact summary

| Strategy | Effort | Token savings | Best for |
|----------|--------|-------------|----------|
| 1. Right-size prompts | Low | 10–30% | All projects |
| 2. Progressive summarization | Medium | 40–80% per handoff | Multi-phase workflows |
| 3. Subagent isolation | Medium | 50–80% total | Complex orchestrations |
| 4. File-based offloading | Low | 60–90% per handoff | Data-heavy workflows |
| 5. `/compact` command | Zero | 30–50% of history | Long interactive sessions |
| 6. Prompt caching | Low | 50–90% of static prefix | Repeated workflow execution |
| 7. Directed attention | Low | 50–75% of search costs | Code review, analysis |
| 8. Tool call efficiency | Low | 30–60% of tool costs | Tool-heavy workflows |
| 9. Template externalization | Medium | 80–97% of template costs | Template-heavy prompts |

## 🎯 Conclusion

Token optimization in multi-agent workflows isn't about penny-pinching — it's about keeping your orchestrations within context window limits and maintaining model accuracy. Context rot degrades output quality long before you hit hard token limits.

The nine strategies work at different levels: prompt authoring (strategies 1, 7, 9), workflow design (strategies 2, 3, 4), runtime management (strategies 5, 8), and provider-level (strategy 6). Start with the low-effort, high-impact strategies (right-sizing, directed attention, tool efficiency) and add complexity only when your workflows demand it.
