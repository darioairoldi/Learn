## Context 🎯

Azure Key Vault and Storage Account share the same **3–24 character limit**. Storage accounts are forced into concatenated naming (no hyphens allowed), which raises the question: should Key Vault adopt the same hyphen-free pattern to maximize character headroom?

---

## Constraint comparison 📊

| Property | Key Vault | Storage Account |
|---|---|---|
| **Max length** | **3–24** | **3–24** |
| **Hyphens allowed** | **Yes** | **No** |
| **Valid chars** | Alphanumerics + hyphens | Lowercase letters + numbers only |
| **Must start with** | Letter | Lowercase letter or number |
| **Must end with** | Letter or number | Letter or number |
| **Consecutive hyphens** | **Not allowed** | N/A |
| **Scope** | Global | Global |

The critical difference: **Key Vault allows hyphens, storage accounts don't.** Storage is forced into concatenated naming; Key Vault has a choice.

**Sources:**

- **[Azure resource naming rules — Microsoft.KeyVault](https://learn.microsoft.com/en-us/azure/azure-resource-manager/management/resource-name-rules#microsoftkeyvault)** 📘 [Official]
- **[Azure resource naming rules — Microsoft.Storage](https://learn.microsoft.com/en-us/azure/azure-resource-manager/management/resource-name-rules#microsoftstorage)** 📘 [Official]

## Character budget analysis 🏗️

Using the standard pattern `{workload}-{env}-kv-{region}-{instance}`:

| Example | With hyphens | Length | Without hyphens | Length | Savings |
|---|---|---|---|---|---|
| samples / testmc / itn | `samples-testmc-kv-itn-01` | **24** (at limit) | `samplestestmckvitn01` | 20 | 4 chars |
| samples / prod / eus | `samples-prod-kv-eus-01` | **22** | `samplesprodkveus01` | 18 | 4 chars |
| ecommerce / testmc / itn | `ecommerce-testmc-kv-itn-01` | **27** (exceeds!) | `ecommercetestmckvitn01` | 22 | 5 chars |
| hrportal / dev / weu | `hrportal-dev-kv-weu-01` | **23** | `hrportaldevkvweu01` | 18 | 5 chars |

Hyphens consume **4 separator characters**. With longer workload or environment names, you **will** hit the 24-char wall.

---

## Options compared ⚙️

### Option A: keep hyphens, shorten components when needed

```text
samples-testmc-kv-itn-01    ← 24 chars (just fits)
ecom-testmc-kv-itn-01       ← 21 chars (shortened workload)
hr-dev-kv-weu-01            ← 16 chars
```

**Pros:**

- Microsoft CAF standard — all official examples use hyphens for Key Vault
- Readable: segments are visually separated
- Auditing-friendly: you can parse the name at a glance

**Cons:**

- Tight budget — requires abbreviating workload/env names for longer combinations
- Inconsistent workload abbreviations across resources (full name on App Service, shortened on KV)

### Option B: drop hyphens (storage-account style)

```text
samplestestmckvitn01        ← 20 chars
ecommercetestmckvitn01      ← 22 chars
hrdevkvweu01                ← 12 chars
```

**Pros:**

- Maximum character headroom
- Consistent with storage account and container registry naming

**Cons:**

- **Poor readability** — where does `testmc` end and `kv` begin?
- **Not what Microsoft recommends** for Key Vault
- Creates a second concatenation pattern to remember with no Azure-enforced reason
- Key Vault is a security resource — name clarity matters for auditing and incident response

### Option C: hybrid — drop instance number, keep hyphens

```text
samples-testmc-kv-itn       ← 21 chars
ecommerce-tmc-kv-itn        ← 20 chars
```

**Pros:**

- Saves 3 chars while keeping full readability
- Works when you only have one Key Vault per workload/env/region

**Cons:**

- Inconsistency if you later need instance `02`
- Breaks the standard pattern (every other resource has `-01`)

## What's commonly used 💡

Microsoft CAF, Azure Landing Zones, and the official [Azure Naming Tool](https://github.com/microsoft/CloudAdoptionFramework/tree/master/ready/AzNamingTool) all use **hyphens for Key Vault**. The no-hyphen pattern is only applied to resources that **prohibit** hyphens:

| Pattern | Resources |
|---|---|
| **Hyphens** (standard) | Resource Groups, App Services, Key Vaults, Functions, SQL, Cosmos DB, Service Bus, etc. |
| **No hyphens** (forced) | Storage Accounts, Container Registries, Data Lake Storage |

No widely adopted convention drops hyphens from Key Vault names by choice.

---

## Recommendation ✅

**Keep hyphens for Key Vault (Option A).** When names exceed 24 chars, shorten the workload or environment code — don't drop hyphens.

**Rationale:**

1. **Microsoft CAF alignment** — official examples consistently show `kv` with hyphens
2. **Security auditability** — Key Vault is a security-critical resource; readable names matter for incident response, access reviews, and compliance audits
3. **Cognitive overhead** — dropping hyphens creates a second naming pattern to remember for no Azure-enforced reason
4. **Proven fix** — the standard remedy for long names is shorter component codes (`ecom` instead of `ecommerce`), not abandoning separators

**Practical guidance for tight names:**

| Situation | Resolution |
|---|---|
| Workload name too long | Abbreviate: `ecommerce` → `ecom`, `hrportal` → `hr` |
| Environment code too long | Shorten: `testpersonal` → `tpers`, `testmc` stays (6 chars is fine) |
| Still too long | Drop instance number if only one instance exists |
| Still too long | Use 2-letter region code: `itn` → `in` (only as last resort) |
