## B — New content: "Boundaries" as quality property

### B.1 — Add "Boundaries" row to the quality properties table (Slide 3.2)

**Location**: Slide 3.2 table

**Rationale**:  
The deck already defines boundaries in two places:
- Slide 3.3: example artifact with a `## Boundaries` section (documenting intent)
- Slide 4.2: the ✅/⚠️/🚫 three-tier enforcement pattern (robustness)

But "boundaries" never appears in the quality properties table. This is a gap: an artifact without explicit scope and limits is incomplete by definition. Boundaries belong in quality because they're a precondition for correct behavior — not just a robustness technique.

**Add this row to the table** (insert after "Actionability"):

| Property | Definition | How to check |
|---|---|---|
| **Boundaries** | Artifact explicitly declares what it will do, what it won't do, and what requires escalation | Does the artifact have Always/Ask-first/Never-do declarations? Is out-of-scope behavior defined? |

**After adding**: Update Slide 3.1 property count from 7 → 8 (see A.1).

### B.2 — Update the explanatory text below the table (Slide 3.2)

**Current text** ends with:
```
**These are preconditions, not nice-to-haves**: Ambiguous guidance → ambiguous output. Missing exception handling → silent failures or unbounded token consumption when the LLM retries indefinitely.
```

**Proposed addition** (append after that paragraph):
```
**Why boundaries is separate from completeness**: Completeness covers whether rules exist for every scenario the artifact handles. Boundaries covers whether the artifact declares what it *won't* handle. Without explicit boundaries, the model defaults to attempting everything — including tasks it's not designed for, causing scope creep, unsafe actions, or silent wrong output. See Slide 3.3 for the intent + boundaries template and Slide 4.2 for the enforcement pattern.
```

## C — Layout: group properties by dimension

### C.1 — Restructure Slide 3.2 table into three groups

**Current structure**: flat table of seven rows (eight after B.1)

**Problem**: All properties look equal weight with no conceptual grouping. Harder to remember and apply.

**Proposed structure**: three groups with visual separators

The eight properties divide cleanly into three dimensions:

**Group 1 — Rule precision** (applies to individual rules):
- Clarity
- Actionability

**Group 2 — System coherence** (applies across the rule set):
- Non-redundancy
- Non-contradiction
- Prioritization

**Group 3 — Coverage and scope** (applies to what the artifact handles):
- Completeness
- Exception handling
- Boundaries *(new from B.1)*

**Proposed table structure**:

```markdown
| Property | Dimension | Definition | How to check |
|---|---|---|---|
| **Clarity** | Rule precision | Rule understood identically by humans and LLMs | Two independent LLM passes produce same interpretation |
| **Actionability** | Rule precision | Translatable to boolean check | Can each rule be evaluated pass/fail? |
| **Non-redundancy** | System coherence | Each rule in exactly one location | Scan for duplicate text across context files |
| **Non-contradiction** | System coherence | Rules across files don't conflict | Coherence check across the artifact set |
| **Prioritization** | System coherence | Conflict precedence is explicit | When rules could conflict, is the priority declared? |
| **Completeness** | Coverage & scope | No gaps where guidance is needed for normal operation | Scan agent behaviors that lack backing rules |
| **Exception handling** | Coverage & scope | Artifact defines behavior for failures, ambiguity, missing input, and out-of-scope requests | Check: does every workflow define what happens when things go wrong? |
| **Boundaries** | Coverage & scope | Artifact explicitly declares what it will do, what it won't do, and what requires escalation | Does the artifact have Always/Ask-first/Never-do declarations? Is out-of-scope behavior defined? |
```

**Alternative** (if the Dimension column feels redundant): use table section breaks with bold header rows instead of a column:

```markdown
**Rule precision — applies to individual rules**

| Property | Definition | How to check |
|---|---|---|
| **Clarity** | ... | ... |
| **Actionability** | ... | ... |

**System coherence — applies across the rule set**

| Property | Definition | How to check |
|---|---|---|
| **Non-redundancy** | ... | ... |
| **Non-contradiction** | ... | ... |
| **Prioritization** | ... | ... |

**Coverage & scope — applies to what the artifact handles**

| Property | Definition | How to check |
|---|---|---|
| **Completeness** | ... | ... |
| **Exception handling** | ... | ... |
| **Boundaries** | ... | ... |
```

**Recommendation**: use the Dimension column approach (single table, easier for slides/projection). The multi-table alternative is better for a published article.
