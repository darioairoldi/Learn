# Slides Content Update Plan: Remove pe-* Infrastructure Assumptions

> The deck should teach prompt engineering and cost control to any audience. Remove self-update infrastructure concepts that only make sense if you know the pe-* system.

---

## Analysis: What belongs vs. what doesn't

### The deck's stated goal

"Prompt Engineering & Azure OpenAI Cost Control — How to bring GitHub Copilot to the next step"

**Target audience**: Developers and teams using GitHub Copilot who want to write better prompts and control Azure costs. No assumption of a self-updating PE infrastructure.

### Content that belongs (general PE knowledge)

- How GHCP assembles prompts (Section 1) ✅
- Artifact types — what they are, when to use them (Section 2) ✅
- How to write good prompts, agents, instructions (Sections 3-4, partially) ✅
- Token cost anatomy and reduction strategies (Section 5, mostly) ✅
- Model-specific optimization (Section 6) ✅
- Tools and MCP ecosystem (Section 7) ✅

### Content that assumes pe-* infrastructure (remove or generalize)

| Location | pe-* content | Issue |
|---|---|---|
| **Slide 2.5** | "N-1 structural separation", "Metadata contract: goal, scope, boundaries, rationales" | pe-* convention, not general GHCP knowledge |
| **Slide 3.1** | "Adherence", "quality propagation through dependencies" | pe-* vision concepts |
| **Slide 3.3** | Entire slide: metadata contracts with goal/scope/boundaries/rationales YAML | pe-* self-update infrastructure — not relevant without automated drift detection |
| **Slide 3.4** | Entire slide: N-1 structural separation, "deterministic change classification" | pe-* concept for autonomous maintenance — irrelevant to general audience |
| **Slide 5.4** | Progressive assessment cost gradient | pe-* self-update concept (research → screening → deep pass) |
| **Slide 5.5** | Modular invocation for cost control | pe-* self-update invocation model |
| **Slide 5.7** | References to R-L3, R-P5 rationale IDs | pe-* vision rationale IDs |
| **Slide 7.1** | `pe-prompt-engineering-validation, pe-artifact-coherence-check` | pe-* specific skills |
| **Section 8** | Entire section (4 slides): self-maintaining PE systems | pe-* self-update vision |
| **Summary** | Points 3, 4, 6 reference pe-* concepts | "metadata contracts, N-1 separation", "progressive assessment", "self-update systems" |

---

## Proposed changes

### C1 (✅ Done): Remove Section 8 entirely — self-maintaining PE systems

Slides 8.1-8.4 cover the pe-* self-update architecture (detect → assess → propose → execute, progressive assessment, quality by construction, artifact-type routing). This is the pe-* vision, not general PE knowledge.

**Replacement**: A single closing slide about "Keeping artifacts fresh" with general advice (version your artifacts, review when platforms update, track what's changed) — no self-update infrastructure assumed.

- [x] Apply

---

### C2 (✅ Done): Replace Slides 3.3 and 3.4 with general best practices

**Slide 3.3 (metadata contracts)**: Replace with "Documenting artifact intent" — general advice about writing clear goal statements, scope descriptions, and constraints in comments or frontmatter, without the pe-* metadata contract schema.

**Slide 3.4 (N-1 structural separation)**: Replace with "Structuring rules for clarity" — general advice about separating rules from explanations and examples in context files, without the pe-* "deterministic change classification" framing.

- [x] Apply

---

### C3 (✅ Done): Simplify Slide 3.1 — quality equation

Remove "Adherence" and "quality propagation through dependencies" (pe-* concepts). Keep the core insight: guidance quality matters more than artifact structure.

- [x] Apply

---

### C4 (✅ Done): Replace Slides 5.4 and 5.5 with general cost strategies

**Slide 5.4 (progressive assessment)**: Replace with "Review what changed, not everything" — general advice about focusing review effort on artifacts affected by platform changes, without the pe-* research/screening/deep-pass model.

**Slide 5.5 (modular invocation)**: Replace with "Scope your reviews" — general advice about reviewing one artifact type at a time and checking dependencies only when issues are found, without the pe-* invocation table.

- [x] Apply

---

### C5 (✅ Done): Clean up pe-* references in other slides

- **Slide 2.5**: Remove "N-1 structural separation" and "Metadata contract" from best practices. Keep token budget and single-source-of-truth advice.
- **Slide 5.7**: Remove R-L3, R-P5 references. Use plain language instead.
- **Slide 7.1**: Replace pe-* skill names with generic examples (e.g., "custom validation skills, domain-specific review skills").
- **Summary slide**: Rewrite points 3, 4, 6 without pe-* terminology.

- [x] Apply
