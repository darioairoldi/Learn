# Deck Goal and Structure Summary

## Goal

Teach developers and teams how GitHub Copilot's prompt assembly works under the hood, how to write effective prompt engineering artifacts for each type in the customization stack, and how to control Azure OpenAI costs through systematic prompt engineering practices.

**Target audience**: Developers and teams using GitHub Copilot who want to move beyond basic chat into structured, production-grade prompt engineering — with cost awareness.

**No assumptions**: The deck assumes no knowledge of any specific PE infrastructure, self-update systems, or internal tooling. All content is based on general GitHub Copilot customization capabilities and publicly documented features.

---

## Structure (8 sections, ~40 slides)

### 1. The prompt assembly machine (6 slides)

**What the audience learns**: How VS Code assembles the actual request sent to the model — the multi-layered system prompt, the user prompt, where each artifact type lands, and what you control vs. what's automatic.

| Slide | Topic |
|---|---|
| 1.1 | What happens when you type a message — the full assembly diagram |
| 1.2 | System prompt vs user prompt — why the distinction matters for instruction strength |
| 1.3 | Where each artifact type lands — the definitive mapping table |
| 1.4 | Context window growth and context rot (Liu et al. 2023: 88% → 30% accuracy) |
| 1.5 | Issues you must manage — context rot, collisions, layer confusion, token bloat, stale guidance, tool noise |
| 1.6 | The cost equation — how input tokens translate to Azure OpenAI billing |

**Key insight**: You're not sending your message to the model — you're contributing a fraction of a multi-layered request. Understanding the assembly is the foundation for everything else.

---

### 2. The artifact toolkit (8 slides)

**What the audience learns**: Every artifact type in the Copilot customization stack — what it does, when to use it, its limitations, and its cost impact.

| Slide | Topic |
|---|---|
| 2.1 | Decision tree — which artifact type for which need |
| 2.2 | Instruction files — auto-injectors with collision risk (limitation: no precedence, high collision risk → keep minimal and safe) |
| 2.3 | Prompt files — explicit workflows with full control |
| 2.4 | Agents — persistent personas with curated tools and orchestration patterns |
| 2.5 | Context files — shared knowledge base with multiple loading paths (semantic search, explicit reference, artifact declarations) |
| 2.6 | Skills, hooks, templates, prompt snippets — secondary artifact types and their cost profiles |
| 2.7 | MCP servers — the extensible tool ecosystem and its cost implications |
| 2.8 | copilot-instructions.md — the final authority (injected last, highest priority) |

**Key insight**: Each artifact type occupies a specific slot in the assembly. Wrong type = wrong behavior. Cost varies dramatically — instruction files cost tokens on every request; hooks cost zero.

---

### 3. Quality strategies (4-5 slides)

**What the audience learns**: How to make artifacts produce reliable, correct output — focusing on guidance quality as the foundation.

| Slide | Topic |
|---|---|
| 3.1 | The quality equation — guidance quality matters more than artifact structure |
| 3.2 | Six measurable quality properties — clarity, non-redundancy, non-contradiction, completeness, prioritization, actionability |
| 3.3 | Documenting artifact intent — clear goals, scope descriptions, and constraints (general best practice, not infrastructure-specific) |
| 3.4 | Structuring rules for clarity — separating rules from explanations and examples for readability and maintainability |
| 3.5 | Multi-pass validation — independent review before committing changes |

**Key insight**: Ambiguous guidance → ambiguous output, regardless of how good the prompt structure is. Quality starts with the rules, not the formatting.

---

### 4. Robustness strategies (5 slides)

**What the audience learns**: How to handle failures, ambiguity, and edge cases in production artifacts.

| Slide | Topic |
|---|---|
| 4.1 | Response management patterns — missing context, ambiguity, out of scope, tool failure, errors |
| 4.2 | Boundary enforcement — always-do / ask-first / never-do categories |
| 4.3 | Error recovery workflows — retry → escalate → abort with iteration limits |
| 4.4 | Handoff contracts for multi-agent workflows — what to include/exclude at each transition |
| 4.5 | Embedded test scenarios — behavioral examples the LLM uses as training signal |

**Key insight**: Production artifacts must handle the unhappy path. Silent guessing on missing input produces confident-looking wrong output.

---

### 5. Efficiency & cost strategies (7 slides)

**What the audience learns**: How to control Azure OpenAI costs through systematic token management, model selection, and review efficiency.

| Slide | Topic |
|---|---|
| 5.1 | Token cost anatomy — Azure pricing, cost drivers, fixed vs variable costs |
| 5.2 | Summarization protocols — compress between phases for 13x context cost reduction |
| 5.3 | Token budget patterns — per-artifact budgets (context ≤2,500; instructions ≤500; agent ≤3,000) |
| 5.4 | Review what changed, not everything — focus effort on artifacts affected by actual changes |
| 5.5 | Scope your reviews — review one artifact type at a time, check dependencies only when needed |
| 5.6 | Model selection for cost optimization — match model capability to task complexity |
| 5.7 | Azure OpenAI cost control checklist — actionable audit items |

**Key insight**: System prompt and tool schemas are fixed costs on every request. A 50% reduction in system prompt size = 50% reduction in input cost for every interaction.

---

### 6. Model-specific optimization (5 slides)

**What the audience learns**: How different models respond to the same prompt differently, and how to optimize per model family.

| Slide | Topic |
|---|---|
| 6.1 | The compiler analogy — same source code, different executables |
| 6.2 | GPT models — explicit instruction style, Markdown structure, few-shot examples |
| 6.3 | Claude models — XML tags, dense context, caution with contradictions |
| 6.4 | Gemini models — structure-first, multimodal, 1M+ context window |
| 6.5 | Reasoning models (o3/o4-mini) — high-level goals, let them think, 4-10x cost premium |

**Key insight**: "A good generic prompt doesn't exist" — there's only a good prompt for that specific model. Every model/version change requires re-validation.

---

### 7. Tools & ecosystem (4 slides)

**What the audience learns**: What tools are available, how MCP works, and when to use hooks vs skills.

| Slide | Topic |
|---|---|
| 7.1 | The tool landscape — built-in, MCP, skills, extensions |
| 7.2 | MCP server patterns — copilot-hosted, stdio, SSE |
| 7.3 | Agent hooks — deterministic automation at zero LLM cost |
| 7.4 | Skills — auto-discovered capabilities with description matching |

**Key insight**: Curate tools per agent. More tools ≠ better — every tool schema costs tokens in the system prompt.

---

### 8. Keeping it fresh (1 slide)

**What the audience learns**: Artifacts decay — platforms change monthly, rules become stale. Plan for maintenance.

| Slide | Topic |
|---|---|
| 8.1 | The drift problem and practical maintenance advice — version artifacts, review on platform updates, track what's changed |

**Key insight**: Drift is the default. Without periodic review, your artifacts silently degrade — they pass structural checks but produce progressively worse output.

## Narrative arc

```
Understanding (1)          → What's happening under the hood
  ↓
Tooling (2)                → What you can control
  ↓
Quality (3) + Robustness (4) → How to do it well
  ↓
Cost (5)                   → How to do it efficiently
  ↓
Models (6) + Tools (7)     → Platform-specific optimization
  ↓
Maintenance (8)            → How to keep it working
```

The deck moves from **understanding** (how the system works) through **practice** (how to use it well and cheaply) to **sustainability** (how to keep it working over time).
