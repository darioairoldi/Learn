## 🎯 Problem this solves

The research vision v1 has a strong pipeline design (Discover → Validate → Reason → Synthesize → Publish) and a strong hallucination resistance architecture (7 layers). What it lacks is the **structural resilience** that prevents the research system *itself* from degrading over time.

PE v7 solves this problem for PE artifacts through four interlocking mechanisms. Research v1 has none of them.

### Gap analysis: PE v7 vs. research v1

| Mechanism | PE v7 | Research v1 | Consequence of the gap |
|---|---|---|---|
| **Metadata contracts** | Every artifact carries structured YAML (`goal`, `scope`, `boundaries`, `rationales`, `version`); pre-change guards block contradictions; post-change reconciliation keeps metadata fresh | Articles mention metadata but no contract schema exists for research artifacts (claim sets, reasoning reports, domain configs) | Research infrastructure can drift from its declared purpose with no automated detection |
| **Chain alignment levels** | Three-level hierarchy: Level 1 (folder, stale-proof), Level 1.5 (category via structure index, precise + stale-proof), Level 2 (file, fragile) | All cross-references use direct file paths (Level 2 only) | Every file rename breaks consumers — already proven: both PE vision references in research v1 are stale |
| **Structural separation for change classification** | N-1 convention: Rule/Rationale/Example labels enable deterministic breaking/non-breaking diff classification | No equivalent — confidence levels exist for claims but pipeline phases lack labeled content blocks | Every change to a research artifact requires full human review — the autonomy gradient collapses to binary |
| **Refactoring protection** | Pre-change guards detect scope regression AND category regression via `required_categories` in structure indexes | No refactoring protection | Restructuring domain configs or claim schemas during multi-domain scaling silently loses coverage |

### The concrete evidence

The research vision references the PE vision twice:

- `../self-updating-prompt-engineering/06.000-vision.v6.md` (in the comparison table)
- `../self-updating-prompt-engineering/06.000-vision.v6.md` (in the References section)

The PE vision was renamed to `20260428.01-vision.v7.md`. Both references are now broken — demonstrating exactly the fragility that Level 1.5 references prevent.

## 🏗️ Artifact architecture: what is a research artifact?

Before defining metadata contracts, we must answer a foundational question: **what are research artifacts, and how do they relate to article-writing artifacts?**

### The existing article-writing pipeline

The Learning Hub already has a complete document production system:

| Layer | What exists | Agents |
|---|---|---|
| **Research** | `documentation-researcher` — quality analysis, gap detection, source verification | Read-only |
| **Build** | `documentation-builder` — article creation following Diátaxis, style, metadata standards | Read-write |
| **Validate** | `documentation-validator` — 7-dimension quality assurance | Read-only |
| **Workflow** | 7-phase creation workflow (planning → research → drafting → validation → review → publication → maintenance) | — |
| **Standards** | Style guide (`01-style-guide.md`), validation criteria (`02-validation-criteria.md`), creation rules (`03-article-creation-rules.md`), templates | Context files |

### Where research overlaps — and where it doesn't

The research vision's 5 phases map against article-writing like this:

| Research phase | Article-writing equivalent | Overlap |
|---|---|---|
| **Phase 1: Discover** (source search, knowledge mapping) | Phase 2: Research (source discovery, reference collection) | Partial — article-writing discovers sources but not autonomously/at scale |
| **Phase 2: Validate** (triangulation, confidence grading) | None — article-writing verifies claims but doesn't formally triangulate | **New capability** |
| **Phase 3: Reason** (deduction, comparison, trend analysis) | None — article-writing doesn't do structured reasoning | **New capability** |
| **Phase 4: Synthesize** (article writing) | Phase 3-4: Drafting + Validation | **Full overlap** |
| **Phase 5: Review & publish** | Phase 5-7: Review + Publication + Maintenance | **Full overlap** |

Research Phases 1–3 are **pre-writing capabilities** that produce intermediate artifacts with no article-writing equivalent. Phases 4–5 are the existing article-writing pipeline.

### Three architecture options

| Option | Description | Verdict |
|---|---|---|
| **A: Article-writing manages everything** | Research pipeline produces intermediate markdown files treated as articles; article-writing agents handle all creation/validation | **Rejected** — article-writing doesn't understand research-specific concerns (triangulation, confidence grading, reasoning chains); `documentation-validator` checks readability and Diátaxis type, which are meaningless for a claim set; intermediate artifacts don't fit article templates |
| **B: Research artifacts replace article-writing** | Everything is research-specific — new agents, new metadata, new validation for all outputs including articles | **Rejected** — duplicates the entire writing pipeline; when product documentation arrives, it needs yet another set; 3× maintenance burden; article-writing improvements don't propagate |
| **C: Layered architecture** | Article-writing remains the **base layer** for all document production; research adds a **pre-writing layer** producing research-specific intermediate artifacts that feed into article-writing as structured input | **Recommended** — no duplication; article-writing improvements propagate; each pre-writing layer defines only what's unique; product documentation slots in naturally |

### The layered architecture

```
                    ┌──────────────────────────────────┐
                    │      ARTICLE-WRITING LAYER        │  ← base: creates all documents
                    │  (builder, validator, 7-phase)     │
                    │  Standards: style, Diátaxis, YAML  │
                    ├──────────────────────────────────┤
                    │       PRE-WRITING INPUTS           │
                    └──────┬────────────┬───────────────┘
                           │            │
              ┌────────────┴──┐    ┌────┴──────────────┐
              │ RESEARCH      │    │ PRODUCT DOC        │  ← future
              │ PRE-WRITING   │    │ PRE-WRITING        │
              │               │    │                    │
              │ Discover      │    │ Detect changes     │
              │ Validate      │    │ Diff APIs          │
              │ Reason        │    │ Assess impact      │
              │               │    │                    │
              │ Output:       │    │ Output:            │
              │ Research      │    │ Change assessment  │
              │ report        │    │ report             │
              └───────────────┘    └────────────────────┘
```

### What is and isn't a research artifact

| Artifact | Research artifact? | Why |
|---|---|---|
| **Domain config** | ✅ Yes | Research-specific: source lists, cadence, confidence thresholds — no article-writing equivalent |
| **Knowledge map** | ✅ Yes | Research-specific: inventory of existing coverage + gap analysis — produced by Phase 1 |
| **Validated claim set** | ✅ Yes | Research-specific: triangulated claims with confidence levels — produced by Phase 2 |
| **Reasoning report** | ✅ Yes | Research-specific: deductive conclusions with premise chains — produced by Phase 3 |
| **Research report** | ✅ Yes (handoff artifact) | Structured input to article-writing: synthesizes findings from Phases 1–3 into actionable writing spec |
| **Draft article** | ❌ No — article-writing artifact | Created by `documentation-builder` using article-writing standards; research extends it with `research_metadata` |
| **Published article** | ❌ No — article-writing artifact | Validated by `documentation-validator` across 7 quality dimensions |

### The handoff contract

The **research report** is the bridge between the research layer and the article-writing layer. It's the structured output of research Phases 1–3, formatted as actionable input for `documentation-builder`:

```yaml
# Research report metadata
type: research-report
cycle_id: "2026-04-28-pe-weekly"
domain: "prompt-engineering"
target_action: "new-article" | "update-article" | "series-extension"
target_path: "03.00-tech/05.02-prompt-engineering/"

# Content: validated claims + reasoning + writing spec
findings:
  - claim: "..."
    confidence: established | supported | emerging
    sources: [...]
    reasoning: "..." # if deductive
    
article_spec:
  suggested_title: "..."
  diataxis_type: explanation | how-to | tutorial | reference
  target_audience: intermediate
  key_topics: [...]
  relationship_to_existing: [...]
```

The `documentation-builder` receives this report and produces an article following its normal 7-phase workflow — with the research layer having done the heavy lifting of Phases 1–2 (planning + research) with rigorous triangulation.

### Why this matters for product documentation

Self-updating product documentation faces the same structural question. Its pre-writing layer would detect product changes (release notes, API diffs, changelog entries), assess impact on existing docs, and produce a **change assessment report** — another handoff artifact for `documentation-builder`.

Both pre-writing layers share:
- Source monitoring (research: knowledge sources; product docs: release/API sources)
- Validation (research: triangulation; product docs: changelog verification)
- Impact assessment (research: knowledge gaps; product docs: doc staleness)
- Structured output to article-writing

Both reuse article-writing's writing, validation, and publication capabilities without duplication.

## 💡 Core proposal

Apply **Option C: Layered architecture with adapted inheritance** — research adds a pre-writing layer that produces research-specific intermediate artifacts, while article-writing remains the base layer for document production. Governance patterns are inherited from PE v7 and adapted to research artifact types.

This combines two independent decisions:
1. **Artifact architecture** (from the analysis above): layered — research pre-writing + article-writing base
2. **Governance adaptation** (from PE v7): adapted inheritance — same metadata contracts and chain alignment, different vocabulary

### Architecture options evaluated

| Option | Description | Verdict |
|---|---|---|
| **A: Direct port** | Copy PE v7 mechanisms verbatim (rules/rationales/examples labels, breaking/non-breaking vocabulary, PE-specific metadata fields) | Rejected — research artifacts (claims, sources, reasoning reports) aren't PE artifacts (agents, prompts, instructions); forcing PE vocabulary creates artificial mapping |
| **B: Research-native** | Design metadata, references, and classification from scratch for research | Rejected — duplicates governance patterns already proven in PE; divergence makes shared infrastructure harder; requires re-deriving principles that already exist |
| **C: Adapted inheritance** | Inherit PE governance patterns; adapt vocabulary and schemas to research artifact types (claims/evidence/analysis instead of rules/rationales/examples; confidence-based classification instead of breaking/non-breaking) | **Recommended** — leverages proven patterns; stays compatible with shared governance; fits research artifact types naturally |

### What the combined strategy means concretely

| PE v7 mechanism | Research adaptation |
|---|---|
| Metadata contracts with `goal`/`scope`/`boundaries`/`rationales`/`version` | Same fields, applied to research-specific artifact types: domain config, validated claim set, reasoning report, research report — NOT to articles (those use article-writing metadata) |
| Three-level chain alignment (folder / category / file) | Same levels, with `RESEARCH-INDEX.md` as the structure index and research-specific category IDs (`source-discovery`, `claim-validation`, etc.) |
| N-1 structural separation (Rule / Rationale / Example) | Research equivalent: Claims (factual assertions) / Evidence chains (source links, triangulation) / Analytical conclusions (deductions) |
| Breaking/non-breaking classification | Confidence-based classification: claim-touching changes are breaking candidates; evidence-only changes are non-breaking |
| Refactoring protection (scope + category regression) | Same guards applied to research artifacts and `RESEARCH-INDEX.md` |
| Pre-change guard + post-change reconciliation | Identical protocol applied to research artifacts |
| Article metadata | Article-writing YAML unchanged; extended with optional `research_metadata` block for traceability back to research layer |

## 🔄 How PE v7 mechanisms map to research

### Metadata contracts: same schema, research-specific artifact types

PE v7 defines metadata contracts for agents, prompts, instructions, context files, skills, and templates. Research needs contracts for its own **pre-writing layer** artifact types — not for articles (those remain article-writing artifacts):

| Research artifact type | Role | Metadata fields |
|---|---|---|
| **Domain config** | Declares what to research, where, and how often | `goal`, `scope.covers/excludes`, `boundaries`, `rationales`, `version`, `sources[]`, `cadence`, `min_confidence` |
| **Knowledge map** | Inventory of existing coverage + gap analysis (Phase 1 output) | `cycle_id`, `domain`, `article_count`, `gap_count`, `coverage_summary`, `content_hash`, `version` |
| **Validated claim set** | Triangulated claims with confidence (Phase 2 output) | `cycle_id`, `domain`, `claim_count`, `confidence_distribution{}`, `source_count`, `content_hash`, `version` |
| **Reasoning report** | Deductive conclusions with premise chains (Phase 3 output) | `goal`, `scope`, `premises[]` (each linking to claim IDs), `confidence`, `version` |
| **Research report** | Handoff artifact: structured input to article-writing pipeline | `cycle_id`, `domain`, `target_action`, `target_path`, `findings_count`, `article_spec`, `version` |

**Articles are NOT research artifacts.** They're article-writing artifacts created by `documentation-builder` and validated by `documentation-validator`. When an article originates from research, it carries an optional `research_metadata` extension in its YAML for traceability:

```yaml
# In article YAML — extends article-writing metadata, doesn't replace it
research_metadata:
  cycle_id: "2026-04-28-pe-weekly"
  source_claims: ["claim-001", "claim-003", "claim-007"]
  confidence_summary: {established: 5, supported: 3, emerging: 1}
```

This extension lets `documentation-validator` optionally verify that factual claims trace back to validated sources — without requiring any changes to the validator's core 7-dimension assessment.

Pre-change guards and post-change reconciliation apply to research artifacts: before changing a domain config's scope, verify the change doesn't contradict the declared `goal` and `boundaries`. After every change, reconcile metadata to reflect the new state.

### Chain alignment: same levels, research-specific index

| Level | PE v7 application | Research application |
|---|---|---|
| **Level 1 (folder)** | Reference by PE artifact folder (e.g., `.copilot/context/00.00-prompt-engineering/`) | Reference by domain folder (e.g., `03.00-tech/05.02-prompt-engineering/`) |
| **Level 1.5 (category)** | Category ID in STRUCTURE-README.md (e.g., `validation-rules`, `token-optimization`) | Category ID in RESEARCH-INDEX.md (e.g., `source-discovery`, `claim-validation`, `reasoning-pipeline`) |
| **Level 2 (file)** | Reference by specific filename | Reference by specific filename (only for singletons: vision doc, root config) |

Default preference: Level 1.5 for cross-artifact references within and across the research pipeline.

### Structural separation: claims / evidence / analysis

PE v7's N-1 convention labels content blocks as Rule, Rationale, or Example. The research equivalent labels content by its epistemological status:

| PE v7 label | Research equivalent | Classification impact |
|---|---|---|
| **Rule** (enforceable statement — MUST/NEVER) | **Claim** (factual assertion with confidence level) | Diff touching claims → breaking candidate |
| **Rationale** (why the rule exists) | **Evidence chain** (source links, triangulation data, confidence justification) | Diff touching evidence only → non-breaking |
| **Example** (illustrative, non-canonical) | **Analytical conclusion** (deduction, trend, cross-domain connection — explicitly labeled speculative) | Diff touching analysis only → non-breaking unless analysis is cited as premise elsewhere |

This separation enables the same deterministic classification that PE v7 achieves: parse labeled blocks, check which block type changed, route accordingly.

## 📋 Proposed changes to the vision

### Change R1: Add layered artifact architecture and metadata contracts (HIGH)

**Location:** New subsection under Design principles, after "Shared governance from PE vision"

**What to add:**

```markdown
### Layered artifact architecture

Research operates as a **pre-writing layer** that feeds into the existing article-writing pipeline. This separation means:

- **Research artifacts** are the intermediate outputs of Phases 1–3 (discover, validate, reason) — they carry research-specific metadata and are governed by research-specific rules
- **Articles** are article-writing artifacts — created by `documentation-builder`, validated by `documentation-validator`, following established article-writing standards
- **The research report** is the handoff artifact that bridges the two layers — structured research output formatted as actionable input for article-writing

This architecture reuses article-writing's document production capabilities without duplication and extends naturally to future pre-writing layers (e.g., self-updating product documentation).

### Metadata-driven research artifacts

*Inherits: PE vision R-S1-metadata-driven, R-S4-role-declaration*

The research system's ability to detect when its own infrastructure drifts, skip redundant validation, and evaluate artifact effectiveness depends on machine-readable metadata — the same principle that makes PE self-update reliable.

Every research artifact should declare:

- **`goal:`** — single sentence: the one outcome the artifact exists to achieve
- **`scope: {covers: [...], excludes: [...]}`** — what the artifact covers and explicitly excludes
- **`boundaries:`** — constraints the artifact operates within
- **`rationales:`** — design constraints that explain why key decisions must be preserved across updates
- **`version:`** — SemVer for individual artifact versioning

Research-specific artifact types carry additional fields:

| Artifact type | Additional fields | Purpose |
|---|---|---|
| **Domain config** | `sources[]`, `cadence`, `min_confidence` | What to research, where, how often, and how conservatively |
| **Knowledge map** | `cycle_id`, `domain`, `article_count`, `gap_count`, `coverage_summary` | Inventory of what's covered and where gaps are |
| **Validated claim set** | `cycle_id`, `domain`, `claim_count`, `confidence_distribution{}`, `source_count`, `content_hash` | Traceability and freshness of the claim inventory |
| **Reasoning report** | `premises[]` (linking to claim IDs), `confidence` | Auditability of deductive chains |
| **Research report** | `cycle_id`, `domain`, `target_action`, `target_path`, `findings_count`, `article_spec` | Handoff contract to article-writing layer |

**Articles are article-writing artifacts, not research artifacts.** When an article originates from research, it carries an optional `research_metadata` extension for traceability back to the research layer — but it's created, validated, and maintained by the article-writing pipeline.

**Pre-change guard (default: block).** Before changing a research artifact, compare the change against `goal`, `scope`, `boundaries`, and `rationales`. Contradictions are blocked by default — proceeding requires explicit human override.

**Post-change reconciliation (mandatory).** After every change, review and update metadata to reflect the new state. This includes bumping `version` per SemVer, updating `scope.covers` if coverage changed, and replacing invalidated `rationales` with new ones.

**Why this matters.** Without metadata contracts, a domain config's scope can silently expand or contract during editing; a reasoning report's premises can become stale when source claims are updated; a research report can diverge from its source findings during revision. Metadata contracts make these drifts detectable and preventable.
```

**Classification:** Non-breaking (additive — new principle, no existing content removed)
**Confidence:** High — direct adaptation of proven PE v7 pattern; layered architecture uses existing article-writing infrastructure
**Impact:** HIGH — foundational change that enables all subsequent changes (R3, R4 depend on metadata)

### Change R2: Add chain alignment levels for research cross-references (HIGH)

**Location:** New subsection under Design principles, after the new "Metadata-driven research artifacts"

**What to add:**

```markdown
### Resilient cross-references

*Inherits: PE vision R-S5-chain-alignment*

Research artifacts reference each other across pipeline phases and across domains. These references must survive the renames, splits, and reorganizations that naturally occur as the research system scales.

Three reference levels are available, in order of preference:

| Level | Mechanism | When to use |
|---|---|---|
| **Level 1 (folder)** | Reference by domain folder path (e.g., `03.00-tech/05.02-prompt-engineering/`) | When the entire domain's article inventory is relevant |
| **Level 1.5 (category)** | Reference by functional category ID in `RESEARCH-INDEX.md` (e.g., `source-discovery`, `claim-validation`) | **Default** for cross-artifact references within the research pipeline |
| **Level 2 (file)** | Reference by specific filename | Only for singletons (vision doc, root config) where the name is descriptive and stable |

**Default to Level 1.5.** Category IDs are stable semantic identifiers that describe the capability needed, not the file that provides it. When files are renamed or reorganized, only the structure index's category mappings are updated — consumer artifacts don't change.

**Structure index.** Each research domain (or the root research folder) maintains a `RESEARCH-INDEX.md` with a Functional Categories section mapping category IDs to current files. Category IDs use kebab-case and describe capability, not location.

**Why this matters.** Research v1 uses direct file references (Level 2) everywhere. The PE vision was renamed from `06.000-vision.v6.md` to `20260428.01-vision.v7.md`, and both references in research v1 are now stale — demonstrating the fragility that Level 1.5 references prevent. As research scales to multiple domains with many claim sets, reasoning reports, and articles, fragile file references compound into a maintenance burden.
```

**Classification:** Non-breaking (additive — new principle)
**Confidence:** High — proven in PE v7; concrete evidence of the problem exists in this document
**Impact:** HIGH — enables R4 and R5

### Change R3: Add confidence-based change classification (MEDIUM)

**Location:** Extend "The trustworthiness gradient" section in The goal

**What to add after the existing trustworthiness gradient table:**

```markdown
### Change classification for research artifacts

The trustworthiness gradient determines autonomy for *publication* decisions. A parallel classification determines autonomy for *artifact maintenance* decisions — changes to the research system's own infrastructure.

Research artifacts separate three content types by epistemological status:

| Content type | What it is | Research analog of | Change impact |
|---|---|---|---|
| **Claims** | Factual assertions with confidence levels | PE Rules | Diff touching claims → breaking candidate → full validation |
| **Evidence chains** | Source links, triangulation data, confidence justifications | PE Rationales | Diff touching evidence only → non-breaking → standard autonomy routing |
| **Analytical conclusions** | Deductions, trends, cross-domain connections (explicitly labeled) | PE Examples | Diff touching analysis only → non-breaking, unless cited as premise elsewhere |

This separation enables deterministic change classification:

1. Identify which labeled blocks the diff touches
2. If any **Claim** block is modified → breaking candidate → human review or full validation pipeline
3. If only **Evidence chain** or **Analytical conclusion** blocks are modified → non-breaking → standard autonomy routing
4. If the content lacks labels → fall back to LLM classification (lower confidence, higher cost)

| Change type | Classification | Autonomy routing |
|---|---|---|
| Adding new claims at ≥ `supported` confidence | Non-breaking additive | Autonomous with notification |
| Updating existing claims with higher confidence | Non-breaking improvement | Autonomous with notification |
| Downgrading claim confidence | Potentially breaking | Human review |
| Removing claims from a validated set | Breaking (coverage loss) | Human review |
| Changing domain scope or boundaries | Breaking | Human review |
| Updating sources without changing claims | Non-breaking | Autonomous |
| Updating evidence chains (better triangulation) | Non-breaking | Autonomous |
| Modifying analytical conclusions | Non-breaking (unless cited as premise) | Autonomous with notification |

**Without this classification**, every research artifact change requires full human review — collapsing the autonomy gradient to binary (trivial vs. everything-else).
```

**Classification:** Non-breaking (additive — extends existing section)
**Confidence:** High — structural equivalent of PE v7's R-P4
**Impact:** MEDIUM — makes the autonomy gradient practical for research infrastructure maintenance

### Change R4: Add refactoring protection for research artifacts (MEDIUM)

**Location:** Extend the "Shared governance from PE vision" section, or add to the new "Metadata-driven research artifacts" principle (from R1)

**What to add:**

```markdown
### Refactoring protection

*Inherits: PE vision metadata-guarded changes → refactoring protection*

As the research system scales to multiple domains, research artifacts will be restructured — domain configs split, claim set schemas evolved, reasoning report formats refined. Without guards, restructuring silently loses coverage.

Two refactoring guards apply:

1. **Scope regression guard** — when restructuring domain configs or splitting claim sets, verify that all topics in the original `scope.covers` are still covered by the resulting artifacts. Missing coverage is flagged as a regression.
2. **Category regression guard** — when restructuring research pipeline artifacts, verify that every `required_categories` entry in `RESEARCH-INDEX.md` still maps to at least one existing file. Missing category coverage is flagged as a regression.

Both guards use the pre-change / post-change protocol from the metadata-driven principle:

| Guard | When | Default action |
|---|---|---|
| Scope regression detected | Before apply | **BLOCK** — explicit human approval required |
| Category regression detected | Before apply | **BLOCK** — explicit human approval required |
| Post-change reconciliation | After apply | **MANDATORY** — update `RESEARCH-INDEX.md` mappings, verify all categories resolve |
```

**Classification:** Non-breaking (additive)
**Confidence:** High — directly adapts PE v7 refactoring protection
**Impact:** MEDIUM — prevents silent capability loss during scaling

## ⚠️ Risk assessment

### Risks

| Risk | Severity | Mitigation |
|---|---|---|
| **Over-engineering before the pipeline exists** | MEDIUM | These mechanisms are design-time contracts, not runtime infrastructure. They cost nothing until research artifacts are created — at which point they prevent the same silent degradation the vision warns about. The PE system proved that adding contracts retroactively is harder than designing them in. |
| **Vocabulary confusion between PE and research** | LOW | The mapping is explicit and documented (R3). Claims/Evidence/Analysis parallels Rules/Rationales/Examples — the analogy is structural, not lexical. |
| **RESEARCH-INDEX.md becomes critical infrastructure** | MEDIUM | Same risk as STRUCTURE-README.md in PE — mitigated by declaring it as high-severity for staleness checks. The index is simple (a markdown table) and deterministically verifiable. |
| **Category proliferation** | LOW | Initial category set (7 entries) maps directly to research pipeline phases + governance. Growth is bounded by the pipeline's architecture. |
| **Metadata overhead for lightweight artifacts** | LOW | Knowledge maps, validated claim sets, and reasoning reports are transient per-cycle artifacts. Full metadata is required only for persistent artifacts (domain configs, research reports). Transient artifacts carry minimal metadata (`cycle_id`, `domain`, `content_hash`, `version`). |
| **Divergence from PE governance patterns** | LOW | Option C (adapted inheritance) was chosen specifically to prevent divergence. Every research mechanism maps to a named PE v7 mechanism. Changes to the shared governance core propagate to both systems. |
| **Article-writing coupling** | LOW | The layered architecture creates a dependency on article-writing's handoff contract. This is a feature, not a bug — the contract is explicit and enforced. If article-writing evolves, the research report schema evolves to match. |
| **Product documentation forward compatibility** | LOW | The layered architecture is designed to accommodate future pre-writing layers. The risk is that product documentation's needs differ enough to require architectural changes. Mitigation: the handoff contract pattern (structured report → article-writing) is generic enough for any pre-writing source. |

### What this doesn't change

- The research pipeline architecture (5 phases) remains unchanged
- The hallucination resistance architecture (7 layers) remains unchanged
- The trustworthiness gradient for publication decisions remains unchanged
- The inherited governance patterns from PE remain unchanged (extended, not replaced)
- Domain configuration model remains unchanged
- All existing content in the vision is preserved
- The article-writing pipeline (agents, workflows, standards) remains unchanged — research feeds into it, not replaces it

R1 (metadata contracts) ─┬─► R3 (change classification — needs metadata to classify)

## ✅ Validation criteria

The vision update succeeds when:

1. **The layered artifact architecture is explicit** — research pre-writing layer + article-writing base layer, with a clear handoff contract (research report → documentation-builder)
2. **Research artifact types are defined with clear boundaries** — domain config, knowledge map, validated claim set, reasoning report, research report are research artifacts; articles are NOT research artifacts
3. **Five research artifact types have defined metadata schemas** — each with `goal`/`scope`/`boundaries`/`rationales`/`version` plus type-specific fields
4. **Three reference levels are explicitly described** with clear preference ordering and research-specific examples
5. **A `RESEARCH-INDEX.md` structure** is defined with functional categories mapping to research pipeline phases (not article-writing phases)
6. **Claims/Evidence/Analysis separation** is defined as the research equivalent of Rules/Rationales/Examples, with deterministic classification rules
7. **Refactoring protection** explicitly includes both scope regression and category regression guards
8. **All PE vision cross-references** point to the current filename (`20260428.01-vision.v7.md`)
9. **YAML frontmatter** `scope.covers` and `description` accurately reflect the layered architecture and new mechanisms
10. **No existing vision content is removed or contradicted** — all changes are additive
11. **Every research mechanism maps to a named PE v7 mechanism** — the adaptation is traceable
12. **The architecture explicitly supports future pre-writing layers** (product documentation) without requiring article-writing changes
