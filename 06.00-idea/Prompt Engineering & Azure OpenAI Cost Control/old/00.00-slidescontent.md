## 1. The prompt assembly machine

### Slide 1.1 — What happens when you type a message

```
You type: "Fix the auth bug"

VS Code intercepts → assembles a MUCH larger request:

┌──────────────────────────────────────────────┐
│              SYSTEM PROMPT                    │
│  ┌────────────────────────────────────────┐  │
│  │ 1. Core identity & global rules        │  │
│  │    "You are an expert AI assistant..."  │  │
│  │ 2. Instruction files (.instructions.md)│  │
│  │    Auto-injected by file pattern match │  │
│  │ 3. Agent persona (if @agent selected)  │  │
│  │    Overrides identity layer            │  │
│  │ 4. Skill descriptions (auto-matched)   │  │
│  │ 5. Tool schemas (MCP + built-in)       │  │
│  │ 6. copilot-instructions.md (LAST)      │  │
│  └────────────────────────────────────────┘  │
├──────────────────────────────────────────────┤
│              USER PROMPT                      │
│  ┌────────────────────────────────────────┐  │
│  │ Your message + conversation history    │  │
│  │ + referenced files (#file:)            │  │
│  │ + workspace structure                  │  │
│  │ + prompt file content (/command)       │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘

→ Sent to model → Response → Optional tool calls → Final answer
```

**Key insight**: You're not sending your message to the model. You're contributing to a multi-layered request where your message is a fraction of the total.

---

### Slide 1.2 — System prompt vs user prompt: why it matters

| Dimension | System prompt | User prompt |
|---|---|---|
| **Model treats as** | Built-in rules and identity | User-provided context |
| **Instruction strength** | Highest — model's own constraints | Lower — user requests |
| **Persistence** | Entire session (agents) or every request (instructions) | Single request |
| **Wrong slot = wrong behavior** | If your rules land here, they're rigid | If your rules land here, they're suggestions |

**The #1 mistake**: Putting behavioral rules in prompt files (user prompt) instead of instruction/agent files (system prompt). The model follows them less reliably because they're treated as user context, not identity rules.

---

### Slide 1.3 — Where each artifact type lands

| Artifact | Lands in | Activation | Lifetime | You control? |
|---|---|---|---|---|
| **Instructions** (`.instructions.md`) | System prompt | Auto (file pattern) | Every matching request | Pattern only |
| **Prompt files** (`.prompt.md`) | User prompt | `/command` | Single invocation | Full |
| **Agents** (`.agent.md`) | System prompt (identity) | Agent picker / `@` | Entire session | Full |
| **Skills** (`SKILL.md`) | System prompt | Auto (description match) | Single request | Description |
| **Hooks** (`.github/hooks/`) | Outside prompt | Lifecycle event | Deterministic | Full |
| **MCP servers** | Tool schemas in system | Runtime tool calls | Session | Config |
| **Context files** (`.copilot/context/`) | Semantic search | Auto (relevance) | Per request | Content |
| **copilot-instructions.md** | System prompt (LAST) | Always | Every request | Full |

### Slide 1.4 — The context window: growth and decay

```
Request 1:  [system ~3K] [user ~1K]                    = ~4K tokens  ✅ accurate
Request 5:  [system ~3K] [history ~8K] [user ~2K]      = ~13K tokens ✅ good
Request 15: [system ~3K] [history ~25K] [user ~3K]      = ~31K tokens ⚠️ degrading
Request 30: [system ~3K] [history ~50K] [user ~5K]      = ~58K tokens ❌ context rot
```

**Context rot** (Liu et al. 2023): Accuracy drops from 88% → 30% at 32K tokens.

The model "forgets" instructions in the middle of the context. Early system prompt rules and recent user messages are retained; everything in between degrades.

---

### Slide 1.5 — Issues you must manage

| Issue | What happens | Mitigation |
|---|---|---|
| **Context rot** | Long conversations degrade accuracy | Decompose into phases; summarize between handoffs |
| **Instruction collision** | Multiple instruction files match → contradictory rules | Minimize instruction files to testable rules only |
| **Layer confusion** | Rules in wrong artifact type → wrong treatment | Match content to the right layer (see Slide 1.3) |
| **Token bloat** | Too much context loaded → expensive + inaccurate | Token budgets per artifact; progressive loading |
| **Stale guidance** | Rules become obsolete silently | Self-update system; version tracking |
| **Tool noise** | Too many tool schemas → model confused | Curate tool lists per agent; use tool groups |

---

### Slide 1.6 — The cost equation

```
Cost = input_tokens × input_price + output_tokens × output_price

Where input_tokens =
  system_prompt (fixed per session)
  + conversation_history (grows per turn)
  + user_context (files, workspace)
  + tool_schemas (per active tool)
  
Every unnecessary token in the system prompt
is paid on EVERY request in the session.
```

**Azure OpenAI billing**: Per 1K tokens. A bloated system prompt with 10K tokens of instructions costs you 10K input tokens × every request × every user × every day.

A 50% reduction in system prompt size = 50% reduction in input cost for every request.

## 2. The artifact toolkit

### Slide 2.1 — The artifact type decision tree

```
Need to add guidance to Copilot?
│
├─ Should it apply automatically to matching files?
│  └─ YES → Instruction file (.instructions.md)
│          Rules: testable/mechanical only, applyTo pattern
│
├─ Should the user invoke it explicitly?
│  └─ YES → Prompt file (.prompt.md)
│          Full workflow, tool selection, model choice
│
├─ Should it define a persistent persona with tools?
│  └─ YES → Agent file (.agent.md)
│          Identity, tools, handoffs, boundaries
│
├─ Should it be auto-discovered by description?
│  └─ YES → Skill (SKILL.md)
│          Description-matched, self-contained
│
├─ Should it run deterministically on events?
│  └─ YES → Hook (.github/hooks/*.json)
│          No LLM involved, lifecycle automation
│
├─ Should it provide shared reference knowledge?
│  └─ YES → Context file (.copilot/context/)
│          Semantic search, guidance/rules
│
├─ Should it expose external tools?
│  └─ YES → MCP server (mcp.json)
│          Tool schemas, runtime calls
│
└─ Should it be the final authority on repo conventions?
   └─ YES → copilot-instructions.md
            Injected LAST, highest authority
```

---

### Slide 2.2 — Instruction files: the auto-injectors

**Role**: Apply rules automatically to every request matching a file pattern

**Best practices**:
- ✅ Only testable, mechanical rules (boolean pass/fail)
- ✅ `applyTo` patterns that match precisely
- ✅ Sentence-case, imperative language
- ❌ Behavioral/strategic guidance (use context files)
- ❌ Overlapping patterns across instruction files

**Limitation**: No precedence mechanism. High risk for collision: Instruction files auto-inject by pattern match — you can't predict which other instruction files will be present in your repository (from extensions, PE systems, team conventions). When two files match the same request, the model receives both with no signal about priority. Contradictions are silent. 

**Consequence**: Instruction files should be minimal and limited to "safe" rules — testable, mechanical checks (boolean pass/fail) that either agree with any other instruction file, measurably contradict (detectable), or add non-overlapping checks. Behavioral or strategic guidance belongs in context files or agent bodies where consumers explicitly choose what to load.

**Cost impact**: Injected on every matching request. Keep them minimal.

```yaml
---
applyTo: '*.md'
description: "Markdown formatting rules"
---
# Rules
- MUST use sentence-case headings
- MUST include alt text for images
```

---

### Slide 2.3 — Prompt files: the explicit workflows

**Role**: User-invoked workflows with full control over model, tools, and handoffs

**Best practices**:
- ✅ Clear goal statement and boundaries
- ✅ Response management (missing context, ambiguity, errors)
- ✅ Embedded test scenarios
- ✅ Summarization protocols for context rot prevention
- ❌ Rules that should apply automatically (use instructions)

**Key feature**: Can specify `model:`, `tools:`, and `handoffs:` in YAML frontmatter.

**Cost impact**: Loaded only when invoked. Larger prompts are acceptable — they're one-time cost.

---

### Slide 2.4 — Agents: the persistent personas

**Role**: Define identity, tools, and behavior for an entire session

**Best practices**:
- ✅ Clear persona with expertise boundaries
- ✅ Curated tool list (only what this agent needs)
- ✅ Handoff declarations for multi-agent orchestration
- ✅ `customInstructions:` for agent-specific rules
- ❌ Generic "do everything" agents (they lack focus)

**Orchestration patterns**:
- **Solo agent**: handles everything within its expertise
- **Orchestrator → specialists**: delegates to focused agents
- **Chain**: sequential processing through specialized agents

---

### Slide 2.5 — Context files: the shared knowledge base

**Role**: Provide guidance, patterns, and rules that any artifact can reference

**Best practices**:
- ✅ Single source of truth — each concept in one file only
- ✅ ≤2,500 tokens per file (split if larger)
- ✅ N-1 structural separation (Rule / Rationale / Example blocks)
- ✅ Metadata contract: `goal:`, `scope:`, `boundaries:`, `rationales:`
- ❌ Duplicating content from other context files

**Cost impact**: Context files are loaded through multiple paths: (1) **semantic search** — Copilot indexes `.copilot/context/` and auto-retrieves relevant files based on your request, (2) **explicit `#file:` references** — user or artifact directly points to a file, (3) **`📖` references in agent/prompt bodies** — artifacts declare which context files they depend on. Well-scoped context files = fewer tokens loaded per request regardless of loading path. Oversized or vaguely-scoped files risk being loaded when only partially relevant — paying full token cost for partial value.

---

### Slide 2.6 — Skills, hooks, and templates

| Artifact | Role | Key feature | Cost |
|---|---|---|---|
| **Skills** | Auto-discovered capabilities | Description matching — no explicit invocation needed | Schema in system prompt when matched |
| **Hooks** | Deterministic lifecycle automation | No LLM — runs scripts/commands on events | Zero LLM cost |
| **Templates** | Reusable output formats | Loaded on-demand via `📖` references | Only when referenced |
| **Prompt snippets** | Reusable prompt fragments | Included via `#file:` references | Only when included |

**Hooks are the only zero-cost artifact**: they don't consume tokens because they don't involve the LLM at all. Use them for metadata updates, file validation, and automation.

---

### Slide 2.7 — MCP servers: the tool ecosystem

**Role**: Expose external capabilities as tools the model can call

**How it works**:
1. MCP server exposes tool schemas (name, description, parameters)
2. Schemas are injected into the system prompt
3. Model decides when to call tools based on the task
4. Tool execution happens outside the LLM — results fed back

**Cost considerations**:
- Tool schemas cost input tokens on EVERY request (they're in the system prompt)
- More tools = more tokens = higher cost + potential model confusion
- **Curate tool lists per agent** — don't expose everything to everyone

**Available ecosystems**: Azure MCP, GitHub MCP, PostgreSQL MCP, Playwright MCP, custom servers

---

### Slide 2.8 — copilot-instructions.md: the final authority

**Role**: Repository-wide rules injected LAST into every system prompt

**Why "last" matters**: In prompt assembly, later content has higher effective priority. `copilot-instructions.md` is the final layer — it can override or reinforce earlier instructions.

**Best practices**:
- ✅ Keep it short — it's loaded on every single request
- ✅ Repo-specific conventions only (not general guidance)
- ✅ Reference other files for detail instead of inlining
- ❌ Long guidance that duplicates context files

## 3. Quality strategies

### Slide 3.1 — The quality equation for PE artifacts

```
Artifact quality = Guidance quality × Artifact craftsmanship × Adherence
```

- **Guidance quality**: Are the rules the artifact follows clear, complete, and non-contradictory? (6 measurable properties)
- **Artifact craftsmanship**: Is the artifact well-structured with proper metadata, boundaries, and N-1 separation?
- **Adherence**: Does the artifact actually implement the rules it claims to follow?

**Quality propagates through dependencies**: A prompt can't be better than the agent it invokes, which can't be better than the context it loads. Fix quality bottom-up.

---

### Slide 3.2 — Six measurable guidance quality properties

| Property | Definition | How to check |
|---|---|---|
| **Clarity** | Rule understood identically by humans and LLMs | Two independent LLM passes produce same interpretation |
| **Non-redundancy** | Each rule in exactly one location | Scan for duplicate text across context files |
| **Non-contradiction** | Rules across files don't conflict | Coherence check across the artifact set |
| **Completeness** | No gaps where guidance is needed | Scan agent behaviors that lack backing rules |
| **Prioritization** | Conflict precedence is explicit | Priority matrix coverage check |
| **Actionability** | Translatable to boolean check | Can each rule be evaluated pass/fail? |

**These are preconditions, not nice-to-haves**: Ambiguous guidance → ambiguous output, regardless of how good the prompt structure is.

---

### Slide 3.3 — Metadata contracts for every artifact

Every artifact should declare:

```yaml
goal: "Single sentence: what this artifact achieves"
scope:
  covers: ["topic 1", "topic 2"]
  excludes: ["out of scope topic"]
boundaries:
  - "Max 2,500 tokens per context file"
  - "Single source of truth for X"
rationales:
  - "Why key decision X was made"
version: "1.0.0"
```

**Why this matters**:
- Pre-change guard: blocks changes that contradict declared goal/scope
- Post-change reconciliation: metadata updated after every change
- Machine-readable classification: breaking vs non-breaking without LLM judgment

---

### Slide 3.4 — N-1 structural separation

```markdown
### Rule name

**Rule** (REQUIRED):
- MUST/NEVER/ALWAYS...

**Rationale** (OPTIONAL):
- Why this rule exists

**Example** (OPTIONAL):
- Concrete illustration
```

**Why this structure**:
- A diff touching a **Rule** block → breaking change → full validation
- A diff touching only **Rationale/Example** → non-breaking → lightweight review
- Without labels → every diff requires LLM judgment (expensive + unreliable)

**Result**: Deterministic change classification at near-zero cost.

---

### Slide 3.5 — Multi-pass validation

```
Draft artifact
  → Pass 1: Build/update (standard model)
  → Pass 2: Independent validation (same or different model)
  → Agreement? → Commit
  → Disagreement? → Investigate, iterate, or escalate
```

**Rule**: No change committed without at least one independent validation pass. Critical changes get multiple passes.

**Why**: A single LLM pass can introduce errors that look correct. A second pass — by a different model or at minimum a different invocation — catches inconsistencies before they propagate.

## 4. Robustness strategies

### Slide 4.1 — Response management patterns

Every production-ready artifact should handle:

```
1. Missing context → Tell user what's needed, offer alternatives
2. Ambiguous request → Clarify before proceeding (max 2 rounds)
3. Out of scope → Redirect to the right artifact
4. Tool failure → Retry once, then alternative approach
5. Error → Diagnose, don't just retry
```

**Anti-pattern**: Silently guessing when information is missing. This produces confident-looking wrong output.

---

### Slide 4.2 — Boundary enforcement

```yaml
boundaries:
  - "NEVER modify files outside the target folder"
  - "NEVER skip validation to save tokens"
  - "MUST escalate when confidence is below threshold"
```

**Critical boundaries always**: No-approval-needed vs Ask-first vs Never-do

| Category | Examples | Enforcement |
|---|---|---|
| ✅ Always do | Use sentence-case, validate references | Implicit — every invocation |
| ⚠️ Ask first | Change article type, restructure | Require user confirmation |
| 🚫 Never do | Skip validation, combine formatting | Hard block |

---

### Slide 4.3 — Error recovery workflows

```
Step fails
  → Retry (1x with diagnostic adjustment)
  → Escalate (present partial results + options)
  → Abort (2 retries failed, report what was accomplished)
```

**Iteration limits**: Research max 2, Build→Validate max 3, Total specialist invocations max 5.

**Why limits matter**: Without them, a failing step can loop indefinitely, consuming unbounded tokens with no progress.

---

### Slide 4.4 — Handoff contracts for multi-agent workflows

| Transition | Include | Exclude | Max tokens |
|---|---|---|---|
| Orchestrator → Specialist | Goal restatement, structured summary | Raw conversation, prior analysis | ~1,500 |
| Specialist → Validator | File path + "validate this" | Builder's reasoning | ~200 |
| Validator → Builder (fix) | Issue list (severity + fix instruction) | Scores, passing checks | ~500 |

**Rule**: Every handoff summarizes prior phases. Never pass accumulated raw context — it causes context rot in the receiving agent.

---

### Slide 4.5 — Embedded test scenarios

Every production artifact should include:

```markdown
## Embedded test scenarios

| # | Scenario | Expected behavior |
|---|---|---|
| 1 | Happy path | Full workflow completes, output validated |
| 2 | Missing input | Asks for clarification, doesn't guess |
| 3 | Conflicting request | Detects conflict, presents options |
```

**Why embedded**: The LLM reads them as part of the prompt and uses them as behavioral examples. They're not just documentation — they're training signal.

## 5. Efficiency & cost strategies

### Slide 5.1 — Token cost anatomy

```
Azure OpenAI pricing (illustrative):
  GPT-4o:        $2.50 / 1M input,  $10.00 / 1M output
  GPT-4.1:       $2.00 / 1M input,   $8.00 / 1M output
  o3:           $10.00 / 1M input,  $40.00 / 1M output
  Claude Opus:   $15.00 / 1M input,  $75.00 / 1M output
  Claude Sonnet:  $3.00 / 1M input,  $15.00 / 1M output

Cost driver breakdown for a typical PE workflow:
  System prompt (fixed):     ~3,000 tokens  × every request
  Context files loaded:      ~2,000 tokens  × when relevant
  Conversation history:      grows 500-2K   × per turn
  Tool schemas:              ~1,000 tokens  × per active tool
  User message + files:      varies         × per request
```

**The insight**: System prompt and tool schemas are fixed costs per request. Reducing them saves on EVERY interaction.

---

### Slide 5.2 — Summarization protocols

```
After each phase, summarize to compressed format:

| After phase | Summarize to | Max tokens | Discard |
|---|---|---|---|
| Research | Topic, sources, gap list | ≤500 | Raw file reads, search results |
| Build | File path + section count | ≤200 | Builder's reasoning, drafts |
| Validate | Pass/fail + issue list | ≤500 | Full analysis prose |

Trigger: Before EVERY handoff, estimate context.
If >8K tokens: MUST summarize all prior phases.
```

**Cost impact**: Without summarization, a 5-phase workflow accumulates ~40K tokens of context. With summarization: ~3K. That's 13x cost reduction on context alone.

---

### Slide 5.3 — Token budget patterns

| Content type | Budget | Rationale |
|---|---|---|
| Context file | ≤2,500 tokens | Keeps individual files focused; split if larger |
| Instruction file | ≤500 tokens | Auto-injected on every match — keep minimal |
| Template | ≤100 lines | On-demand loading — concise output format |
| Agent system prompt | ≤3,000 tokens | Fixed cost per session — every token counts |
| Handoff payload | ≤1,500 tokens | Prevents context rot in receiving agent |

**Rule**: Every token in a persistent artifact (instructions, agent identity) is paid on every request. Budget ruthlessly.

---

### Slide 5.4 — Progressive assessment: the cost gradient

```
When reviewing artifacts, don't apply full analysis to everything:

Level 0: Cache fresh + no external changes  → SKIP         Cost: 0
Level 1: Cache stale, structure unchanged    → Tier 1 check Cost: near-zero
Level 2: External changes exist              → Relevance    Cost: ~800 tokens
Level 3: Screening fails or explicit target  → Deep pass    Cost: full

Most artifacts on most invocations stay at Level 0-1.
Full analysis only where evidence warrants it.
```

**Azure cost impact**: A PE system with 60 artifacts — full review costs ~120K tokens ($0.30-$1.80 depending on model). With progressive assessment: ~5K tokens for a typical scoped review ($0.01-$0.08).

---

### Slide 5.5 — Modular invocation for cost control

```
"Review my agents"     → Deep: agents     Screen: context, instructions
"Update context files"  → Deep: context    Screen: nothing (base layer)
"Review prompts"       → Deep: prompts    Screen: agents, context, instructions

Each invocation costs proportional to what actually needs reviewing,
not the size of the entire artifact set.
```

| Invocation | Deep pass | Screening | Estimated cost |
|---|---|---|---|
| Context files | Context | None | ~15K tokens |
| Agents | Agents | Context + instructions | ~20K tokens |
| Full system | Everything | None | ~120K tokens |

---

### Slide 5.6 — Model selection for cost optimization

| Task | Recommended model | Why | Cost ratio |
|---|---|---|---|
| Research & analysis | Reasoning model (o3) | Needs deep thinking | 1x (baseline) |
| Implementation | Standard model (GPT-4o) | Execution, not analysis | 0.25x |
| Deterministic checks | Small model or scripting | YAML parse, hash compare | 0.01x |
| Screening checks | Lightweight LLM (~800 tokens) | Scope comparison only | 0.05x |

**Rule**: Match model to task. Don't use a reasoning model for file formatting. Don't use a small model for architectural analysis.

---

### Slide 5.7 — Azure OpenAI cost control checklist

- [ ] **System prompt audit**: Remove redundant instructions, deduplicate rules
- [ ] **Token budgets**: Set per-artifact budgets and enforce during review
- [ ] **Model routing**: Use appropriate model per task type (R-L3)
- [ ] **Progressive assessment**: Screen before deep-diving (R-P5)
- [ ] **Summarization protocols**: Compress between phases
- [ ] **Curated tool lists**: Only necessary tools per agent
- [ ] **Context file optimization**: ≤2,500 tokens, focused scope
- [ ] **Validation caching**: Skip recently-validated unchanged artifacts
- [ ] **Instruction minimization**: Keep auto-injected files minimal
- [ ] **Monitor & track**: Token consumption per trigger type, per review cycle

## 6. Model-specific optimization

### Slide 6.1 — The compiler analogy

```
Same prompt + different model = different result

Think of each model as a different compiler:
  GPT-4o/4.1:  Explicit instruction follower — tell it exactly what to do
  Claude:       Context-sensitive reasoner — give it rich context + clear structure
  Gemini:       Structure-oriented — Markdown/XML formatting matters
  Reasoning:    Goal-oriented — minimal prompting, let it think
```

**The rule**: Every time you change model or version → re-read the official guide → re-validate existing prompts.

---

### Slide 6.2 — GPT models: explicit instruction style

```yaml
model: gpt-4.1  # or gpt-4o, gpt-5
```

- ✅ Explicit, step-by-step instructions
- ✅ Few-shot examples for formatting
- ✅ `developer` message role (GPT-5+) for system-level instructions
- ✅ Markdown structure for readability
- ❌ Vague or implicit expectations
- ❌ Overly long system prompts without structure

**GPT-specific tip**: Use `#` headings and numbered lists — GPT models parse hierarchical Markdown well.

---

### Slide 6.3 — Claude models: context and clarity

```yaml
model: claude-opus-4.6  # or claude-sonnet-4
```

- ✅ XML tags for structured sections (`<context>`, `<instructions>`)
- ✅ Dense system prompts with detailed instructions
- ✅ "Think step by step" when analysis is needed
- ✅ Long-context capability — can handle large reference documents
- ❌ Contradictory instructions (Claude is cautious — may refuse)
- ❌ "Simply" or "just" (Claude takes instructions literally)

**Claude-specific tip**: Put examples inside `<example>` tags. Claude treats tagged content as distinct from instructions.

---

### Slide 6.4 — Gemini models: structure-first

```yaml
model: gemini-2.0-flash  # or gemini-3
```

- ✅ Clear Markdown/XML structure
- ✅ Few-shot examples for output formatting
- ✅ Leverage the 1M+ context window for reference docs
- ✅ Multimodal input (images, diagrams)
- ❌ Unstructured prose instructions
- ❌ Assuming behavior from GPT/Claude transfers

---

### Slide 6.5 — Reasoning models: let them think

```yaml
model: o3  # or o4-mini
```

- ✅ High-level goals, not step-by-step instructions
- ✅ Constraints and success criteria
- ✅ Complex multi-step problems
- ❌ Chain-of-thought prompting (they do it internally)
- ❌ Excessive hand-holding (degrades performance)
- ❌ Simple tasks (overkill — use standard models)

**Cost warning**: Reasoning models are 4-10x more expensive. Use them only for tasks that genuinely require multi-step reasoning.

## 7. Tools & ecosystem

### Slide 7.1 — The tool landscape

```
Built-in tools (VS Code):
  read_file, grep_search, semantic_search, run_in_terminal,
  replace_string_in_file, create_file, list_dir, file_search...

MCP servers (extensible):
  Azure (50+ tools), GitHub, PostgreSQL, Playwright,
  Microsoft Learn, Foundry, custom servers...

Agent skills (auto-discovered):
  pe-prompt-engineering-validation, pe-artifact-coherence-check,
  azure-*, article-review, microsoft-foundry...

Extension tools:
  GitKraken, GitHub PRs, database tools...
```

**Tool selection principle**: Curate per agent. More tools ≠ better. Every tool schema costs tokens in the system prompt.

---

### Slide 7.2 — MCP server patterns

```json
// .vscode/mcp.json
{
  "servers": {
    "azure": {
      "type": "copilot",
      "scopes": ["azure"]
    },
    "custom-api": {
      "command": "node",
      "args": ["./mcp-server.js"],
      "env": { "API_KEY": "${input:apiKey}" }
    }
  }
}
```

**Patterns**:
- **Copilot-hosted**: Azure, GitHub, Microsoft Learn — zero setup
- **Stdio-based**: Custom servers — run locally, full control
- **SSE-based**: Remote servers — shared infrastructure

---

### Slide 7.3 — Agent hooks: deterministic automation

```json
{
  "hooks": {
    "postSave": [{
      "event": "onFileSaved",
      "pattern": "**/*.md",
      "command": "node validate-metadata.js ${file}"
    }]
  }
}
```

**Zero LLM cost**: Hooks run scripts, not models. Use them for:
- Metadata validation after file saves
- Reference integrity checks
- Timestamp updates
- Build triggers

---

### Slide 7.4 — Skills: auto-discovered capabilities

```markdown
<!-- SKILL.md -->
```

## 8. Self-maintaining PE systems

### Slide 8.1 — The drift problem

```
Day 1:    Artifacts are fresh, accurate, consistent     ✅
Month 3:  VS Code 1.110 ships — new YAML fields         ⚠️
Month 6:  3 context files have contradictory rules       ❌
Month 9:  Half the agents reference deprecated features  ❌❌

Silent degradation: artifacts pass all structural checks
but produce progressively worse output.
```

**Without systematic maintenance**: Quality decays through platform drift, internal drift, and ecosystem evolution — exactly the forces that affect any software system.

---

### Slide 8.2 — The self-update architecture

```
Detect → Assess → Propose → Execute

Detect:  Platform releases, ecosystem events, internal changes
Assess:  Research → Informed screening → Deep pass (progressive depth)
Propose: Scoped correction plans with evidence
Execute: Validated changes with rollback guarantee

Modular: "Review agents" only deep-reviews agents,
         screens context/instructions cheaply.
```

**Cost-bounded**: Each invocation applies the minimum assessment the evidence requires. Most artifacts on most cycles → skip or lightweight screen.

---

### Slide 8.3 — Quality by construction, not just validation

```
When creating/updating guidance artifacts:

Construction invariants (checked DURING the change):
  ✅ Non-redundancy: rule already exists elsewhere? Reference it.
  ✅ Non-contradiction: conflicts with existing rules? Resolve first.
  ✅ Actionability: can it be evaluated pass/fail? If not, rewrite.

Assessment cadence:
  Every change:      cheap invariants (non-redundancy, non-contradiction)
  Scheduled reviews: clarity test, completeness scan, prioritization audit
  Phase transitions: all 6 properties at full depth
```

**The principle**: A clarity issue caught during construction costs one rewrite. The same issue caught during validation costs a finding + fix cycle + re-validation.

---

### Slide 8.4 — Artifact-type-aware quality routing

| Artifact type | Role | Quality path |
|---|---|---|
| **Context files** | Define guidance | Quality-by-construction: 6 properties as invariants |
| **Instruction files** | Auto-inject rules | Minimization check + non-contradiction |
| **Agents** | Consume guidance | Adherence check against loaded guidance |
| **Prompts** | Orchestrate agents | Adherence + workflow correctness |
| **Templates/hooks** | Shape output | Structural correctness |

**Bottom-up**: Fix context files first → agents improve without changes → prompts improve without changes. Quality propagates through the dependency chain.

## Key takeaways

### Slide — Summary

1. **Understand the assembly**: Know where your content lands in the prompt — system vs user prompt determines how the model treats it
2. **Right artifact for the job**: Each type has a specific role — wrong type = wrong behavior
3. **Quality is systematic**: 6 measurable properties, metadata contracts, N-1 separation, multi-pass validation
4. **Cost is controllable**: Token budgets, summarization, progressive assessment, model routing, instruction minimization
5. **Models are compilers**: Same prompt ≠ same result. Optimize per model, re-validate on model changes
6. **Maintenance is architecture**: Self-update systems keep artifacts fresh at bounded cost — drift is the default
