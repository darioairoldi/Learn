## 🎯 The problem

> **Scope note.** This vision governs the PE artifacts that control article writing — instructions, context files, agents, prompts, skills, templates, and workflows. It doesn't govern the articles themselves (that's the article-writing pipeline's job) or the research that feeds them (that's the research self-update system's job). It governs the *system that governs writing*.

The Learning Hub's article-writing infrastructure is a layered PE system:

```
article-writing.instructions.md          ← Tier 0: auto-loaded voice, mechanics, formatting
documentation.instructions.md            ← Tier 0: base structure, reference classification
    ↓
01-style-guide.md                        ← Tier 1: quantitative targets, replacement tables
02-validation-criteria.md                ← Tier 1: 7 validation dimensions, freshness scoring
03-article-creation-rules.md             ← Tier 1: Diátaxis patterns, required elements
    ↓
01-article-creation-workflow.md          ← Tier 2: 7-phase creation sequence
02-review-workflow.md                    ← Tier 2: review process
03-series-planning-workflow.md           ← Tier 2: multi-article planning
    ↓
documentation-builder.agent.md           ← Tier 3: article creation agent
documentation-researcher.agent.md        ← Tier 3: quality analysis agent
documentation-validator.agent.md         ← Tier 3: 7-dimension validation agent
    ↓
article-design-and-create.prompt.md      ← Tier 4: creation prompt
article-review-*.prompt.md               ← Tier 4: review prompts
    ↓
article-review skill                     ← Tier 5: validation checklists
article-template.md, howto-template.md   ← Tier 5: document templates
```

This architecture works when it's fresh. It decays through three forces — the same three that affect any PE system, but with domain-specific manifestations.

### Writing-standards drift

Writing standards evolve through multiple independent sources:

| Source | Change velocity | Examples |
|---|---|---|
| **Microsoft Writing Style Guide** | Quarterly | New terminology guidance, updated UI verbs, revised accessibility language |
| **Diátaxis framework** | Annual | Refined type definitions, updated structure patterns, new examples |
| **WCAG accessibility** | Multi-year | WCAG 3.0 evolving new scoring model, new success criteria |
| **Google Developer Style Guide** | Quarterly | Alternative best practices, different recommendations for the same scenarios |
| **VS Code / Copilot tooling** | Monthly | New markdown features, new Copilot capabilities for writing, new tool behaviors |

When the MWSG updates its guidance on UI element terminology and `article-writing.instructions.md` still says "select" but MWSG now recommends "choose" for certain contexts — the system produces articles with guidance that's behind the standard it claims to follow. This drift is silent: nothing fails, everything looks correct, but the authority behind the rules has moved.

### Internal consistency drift

The article-writing system has **15+ artifacts** with cross-dependencies. A rule change in one artifact must propagate to every artifact that references, enforces, or builds on it.

Concrete failure modes:

- **Style guide ↔ validation criteria mismatch.** The style guide (`01-style-guide.md`) defines Flesch 50–70 as the target range. The validation criteria (`02-validation-criteria.md`) defines pass/fail thresholds. If one is updated and the other isn't, articles pass validation but violate the style guide — or vice versa.
- **Instruction ↔ context file contradiction.** `article-writing.instructions.md` defines a rule (always-loaded). `03-article-creation-rules.md` extends it (loaded on-demand). If the extension contradicts the base rule, the LLM receives conflicting guidance depending on which files are in context.
- **Agent ↔ workflow desynchronization.** The creation workflow defines 7 phases. The builder agent implements them. If the workflow adds a phase but the agent body isn't updated, the agent skips it.
- **Prompt ↔ template drift.** A creation prompt references a template filename. If the template is renamed or restructured, the prompt silently fails to locate it.

These inconsistencies are invisible during normal use because no single artifact shows the problem — it only appears when two artifacts interact.

### Quality signal disconnection

The article-writing system produces articles. Those articles are reviewed — by `documentation-validator`, by human reviewers, by readers. Each review generates signals about rule effectiveness: which rules produce good articles, which rules cause friction, which rules are routinely violated because they're unclear or impractical.

Currently, **none of this signal flows back into the rules.** A validation pattern that consistently produces false positives (flagging correct content) stays in the criteria. A writing rule that every article violates (because it's impractical) stays in the instruction file. The system can't learn from its own outputs.

### Why this is different from PE self-update and research self-update

| Dimension | PE self-update | Research self-update | Article-writing self-update |
|---|---|---|---|
| **What it maintains** | PE infrastructure (all domains) | Knowledge articles | Article-writing PE artifacts |
| **Primary drift force** | Platform changes (VS Code, Copilot) | Knowledge evolution | Writing standards evolution |
| **Core detection signal** | Release notes, changelogs | New publications, discoveries | Standards updates, quality outcomes |
| **Domain expertise needed** | PE framework knowledge | Subject-domain knowledge | Writing quality expertise |
| **Quality metric** | Artifact structural health | Article factual accuracy | Article readability, clarity, consistency |
| **Output** | Updated PE artifacts | New/updated articles | Updated writing standards/rules |

The three systems are **siblings, not parent-child**: they share governance infrastructure from the PE v7 engine but have distinct pipelines, detection sources, and quality metrics. They can trigger each other: a PE platform change (new Copilot writing tool) triggers article-writing detection; a research finding (new accessibility standard) triggers article-writing assessment; an article-writing rule change (updated procedure formatting) propagates to all active articles via research maintenance.

## 💡 The goal

Build a self-update system that maintains article-writing PE artifacts at peak **reliability** (consistent, predictable outputs from every writing rule and workflow), **effectiveness** (rules that produce high-quality articles meeting their declared standards), and **efficiency** (minimal token consumption, no redundant rules, no unused artifacts) — with the maximum degree of autonomy that assessed risk allows.

### The article-writing self-update cycle

The system performs a continuous cycle:

```
    ┌──────────────────────────────────────────────────┐
    │                                                  │
    ▼                                                  │
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────┴───┐
│  DETECT  │───▶│  ASSESS  │───▶│ PROPOSE  │───▶│ EXECUTE  │
│          │    │          │    │          │    │          │
│ Standards│    │ Impact   │    │ Scoped   │    │ Apply +  │
│ sources  │    │ analysis │    │ changes  │    │ verify   │
│ Quality  │    │ Metadata │    │ Evidence │    │ Learn    │
│ signals  │    │ guards   │    │ based    │    │ from     │
│ Internal │    │ Chain    │    │ Human or │    │ outcomes │
│ events   │    │ checks   │    │ auto     │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

### The autonomy gradient for writing artifacts

Autonomy is calibrated to **artifact tier × change type**:

| Autonomy level | Scope | Examples |
|---|---|---|
| **Autonomous** | Low tier, non-breaking | Template formatting fix, prompt reference update, skill checklist addition |
| **Autonomous with notification** | Low-medium impact, validated | Replacement table update from confirmed MWSG change, workflow step clarification |
| **Human approval required** | High impact or medium confidence | Changing a voice principle, modifying validation thresholds, restructuring a context file |
| **Human-only** | Architectural or strategic | Adding new Diátaxis types, changing the validation dimension model, vision changes |

### Breaking vs. non-breaking for writing artifacts

The PE v7 classification applies directly because article-writing artifacts ARE PE artifacts:

| Change category | Goal/scope/boundary impact | Impact level |
|---|---|---|
| Fix typo in replacement table | Preserved | Low |
| Update example to match current MWSG | Preserved | Low |
| Add coverage for a new UI pattern | Scope expands within boundaries | Low-medium |
| Tighten readability threshold | Boundary change | Medium-high |
| Relax a MUST rule to SHOULD | Boundary weakened | High |
| Change a voice principle | Goal affected | High |

### Key definitions

| Term | Definition |
|---|---|
| **Writing artifact** | A PE artifact in the article-writing domain — instructions, context files, agents, prompts, skills, templates, or workflows that govern article creation, validation, and publication |
| **Standards source** | An authoritative external reference for writing quality: MWSG, Diátaxis, WCAG, Google Developer Style Guide |
| **Quality signal** | Evidence from article outcomes about rule effectiveness — validation results, human corrections, reader feedback |
| **Internal consistency** | The property that all writing artifacts agree on the same rules — no contradictions, no gaps, no redundancies |
| **Rule effectiveness** | Whether a writing rule achieves its stated purpose when applied to articles |

## 🔄 Options analysis

Three architectural approaches for maintaining article-writing PE artifacts:

### Option A: PE self-update manages everything

The generic PE self-update engine handles article-writing artifacts as part of its general artifact maintenance. No writing-domain-specific logic.

**Strengths:**
- Zero additional infrastructure
- Catches structural issues (dangling references, stale metadata, broken chains)
- Leverages all PE v7 governance patterns automatically

**Weaknesses:**
- Can't monitor writing-specific standards sources (MWSG, Diátaxis, WCAG)
- Can't assess writing-rule effectiveness from article quality outcomes
- Treats article-writing artifacts the same as any PE artifact — misses domain-specific drift
- Internal consistency across writing artifacts isn't assessed as a coherent system

**Verdict:** Necessary but insufficient. PE self-update catches structural problems but misses domain-specific ones. It's the engine layer, not the full solution.

### Option B: Standalone article-writing self-update

A completely independent system for article-writing artifacts with its own governance, metadata schemas, infrastructure, and detection pipeline.

**Strengths:**
- Full domain specialization for writing-specific concerns
- Complete control over detection, assessment, and execution
- No dependency on PE self-update engine's capabilities or limitations

**Weaknesses:**
- Duplicates the entire governance infrastructure (autonomy gradient, metadata contracts, multi-pass validation, chain alignment)
- Diverges from PE engine over time, losing proven patterns
- When PE engine improves, article-writing doesn't benefit
- 2× maintenance burden for governance infrastructure
- Requires building from scratch what already exists

**Verdict:** Rejected — the duplication cost exceeds the specialization benefit. Writing artifacts are PE artifacts; they don't need a separate engine.

### Option C: Domain-specialized layer on PE self-update engine (recommended)

PE self-update provides the engine and shared governance. Article-writing adds a **domain-specific layer** that understands writing standards, quality signals, and the writing artifact graph. The domain layer plugs into the engine's Detect → Assess → Propose → Execute cycle with writing-specific implementations.

**Strengths:**
- No governance duplication — inherits proven patterns from PE v7
- PE engine improvements propagate automatically
- Writing-domain expertise where it matters (detection sources, quality assessment, rule effectiveness)
- Clean separation: engine handles mechanics, domain layer handles judgment
- Same architecture as research self-update's pre-writing layer — consistent sibling pattern
- Forward-compatible with other domain layers (DevOps documentation, API documentation)

**Weaknesses:**
- Depends on PE engine capabilities and maturity
- Domain layer must stay compatible with engine's governance contracts
- Adds complexity to the engine's consumer model

**Verdict:** Recommended. This approach maximizes leverage of proven patterns while adding the domain-specific intelligence that generic PE maintenance can't provide.

## ⚙️ The rationale

### Inherited rationales from PE v7

The following PE v7 rationales apply directly to article-writing artifacts (not repeated in full — see [PE vision](../self-updating-prompt-engineering/20260428.01-vision.v7.md)):

| ID | Rationale | Article-writing application |
|---|---|---|
| **R-L1-llm-reasoning** | LLM reasoning and judgment | Evaluate whether a writing-standards update warrants a rule change |
| **R-L2-self-correction** | Multi-pass validation | No writing rule change is committed without independent validation |
| **R-L4-external-knowledge** | External knowledge access | Monitor MWSG, Diátaxis, WCAG, Google Style Guide for updates |
| **R-P1-deterministic-processing** | Deterministic where possible | Cross-reference checking, metadata validation, consistency audits |
| **R-P2-decomposition** | Context window control | Writing system review decomposed by artifact tier |
| **R-P4-structural-separation** | N-1 for change classification | Writing rules use Rule/Rationale/Example labels for deterministic classification |
| **R-S1-metadata-driven** | Metadata contracts | Every writing artifact carries goal/scope/boundaries/rationales/version |
| **R-S2-dependency-graph** | Scoped impact analysis | A change to the style guide triggers checks on validation criteria, agents, and prompts that reference it |
| **R-S5-chain-alignment** | Resilient cross-references | Writing artifacts reference by folder (Level 1) or category (Level 1.5), not filename (Level 2) |
| **R-S6-tier-blast-radius** | Tier-based risk | Tier 0 instruction changes get highest scrutiny; Tier 5 template changes get lowest |
| **R-S8-instruction-minimization** | Collision prevention | Writing instruction files carry only testable/mechanical rules |
| **R-G1-autonomy-gradient** | Risk-calibrated autonomy | Autonomy per change, not per cycle |
| **R-G3-progressive-learning** | Outcome-driven improvement | Track which rule changes improved article quality |

### Article-writing-specific rationales

| ID | Rationale | Design implication |
|---|---|---|
| **R-W1-standards-triangulation** | **Writing-standards changes require source verification.** A single blog post claiming "MWSG now recommends X" isn't sufficient. Standards changes must be verified against the authoritative source before updating rules. This is the article-writing equivalent of research's R-R1 source triangulation — applied to standards, not subject claims. | Every proposed rule change cites the specific standards source, edition/date, and relevant section. Changes from unofficial sources are flagged as unverified. |
| **R-W2-internal-consistency** | **The writing system must be internally consistent.** A rule in the style guide, a threshold in the validation criteria, and a check in the agent body should all agree. Inconsistency between writing artifacts is a quality defect — it produces contradictory guidance that the LLM resolves unpredictably. | Every rule change triggers a consistency check across all artifacts that reference or enforce the affected rule. Inconsistency is HIGH severity. |
| **R-W3-quality-feedback** | **Article outcomes are evidence for rule effectiveness.** Validation results, human corrections, and review patterns contain signal about which rules work and which don't. Rules that consistently produce false positives, or rules that articles consistently violate, are candidates for review. | Quality signals are logged and periodically analyzed. Patterns (>N% false positive rate, >M% violation rate) trigger assessment cycles for the affected rules. |
| **R-W4-conservative-relaxation** | **Tightening rules is safer than relaxing them.** Adding a new check or raising a threshold has bounded downside (articles that were passing may now fail — detectable). Removing a check or lowering a threshold has unbounded downside (articles that should fail now pass — silent quality regression). | Rule tightening can be autonomous with notification. Rule relaxation requires human approval and evidence of false positives or impracticality. |
| **R-W5-writing-artifact-coherence** | **The writing system is a coherent whole, not independent artifacts.** Style guide, validation criteria, creation rules, workflows, agents, prompts, and templates form a dependency graph where changes ripple across tiers. Assessment must treat the system as a graph, not a collection. | Impact analysis traverses the writing artifact graph. A change to a Tier 1 context file triggers assessment of all Tier 2+ artifacts that reference it. |
| **R-W6-rule-traceability** | **Every writing rule should be traceable to its authority.** "Use sentence-style capitalization" exists because MWSG says so. "Use contractions" exists because MWSG warm-and-relaxed principle says so. When the authority changes, the rule should change. When a rule has no traceable authority, it's a convention that should be documented as such. | Writing rules are classified as: standards-derived (traceable to MWSG/Diátaxis/WCAG), evidence-derived (from quality outcomes), or convention (team decision). Classification determines the assessment approach. |

### Conflict resolution

When article-writing-specific rationales conflict with inherited PE rationales:

| Conflict | Resolution |
|---|---|
| R-W1-standards-triangulation vs R-L4-external-knowledge | R-W1 wins — external knowledge discovers changes, but writing-standards triangulation confirms them before action. Discovery is fast; confirmation is required. |
| R-W4-conservative-relaxation vs R-G1-autonomy-gradient | R-W4 constrains R-G1 — rule relaxation is NEVER autonomous regardless of what the gradient would otherwise allow. Tightening follows the normal gradient. |
| R-W3-quality-feedback vs R-W1-standards-triangulation | Independent signals — quality feedback justifies rule changes based on outcomes; standards triangulation justifies rule changes based on authority. Either signal can trigger assessment; both inform the proposal. |

## 🏗️ The vision

A self-updating article-writing system **continuously monitors writing-standards sources and article quality outcomes**, **detects when writing artifacts have drifted from their authoritative standards or become internally inconsistent**, **proposes evidence-based corrections**, and **executes validated corrections autonomously when risk is low enough**.

### The article-writing artifact graph

The writing system forms a tiered dependency graph. Understanding this graph is essential for impact analysis:

| Tier | Artifacts | Dependency direction | Blast radius |
|---|---|---|---|
| **Tier 0** | `article-writing.instructions.md`, `documentation.instructions.md` | Foundation — everything depends on these | CRITICAL — changes here affect every article and every downstream artifact |
| **Tier 1** | `01-style-guide.md`, `02-validation-criteria.md`, `03-article-creation-rules.md` | Extend Tier 0; consumed by Tier 2+ | HIGH — changes here affect workflows, agents, and prompts |
| **Tier 2** | `01-article-creation-workflow.md`, `02-review-workflow.md`, `03-series-planning-workflow.md` | Consume Tier 0–1; guide Tier 3 | MEDIUM — changes here affect agent behavior and prompt sequences |
| **Tier 3** | `documentation-builder.agent.md`, `documentation-researcher.agent.md`, `documentation-validator.agent.md` | Consume Tier 0–2; invoked by Tier 4 | MEDIUM — changes here affect how agents process prompts |
| **Tier 4** | `article-design-and-create.prompt.md`, `article-review-*.prompt.md` | Consume Tier 0–3; use Tier 5 | LOW-MEDIUM — changes here affect specific workflows |
| **Tier 5** | `article-review` skill, templates | Consumed by Tier 3–4 | LOW — leaf artifacts with minimal downstream impact |

**Dependency rule.** Changes propagate upward through the tier stack: a Tier 1 change may affect Tiers 2–5; a Tier 5 change affects nothing else. The PE v7 principle R-S6-tier-blast-radius applies directly.

### Detect

*Inherits: PE engine Detect cycle. Adds: writing-specific trigger sources.*

Five trigger sources initiate assessment:

| Signal | Source | What it indicates |
|---|---|---|
| **Standards update** | MWSG, Diátaxis, WCAG, Google Style Guide | Writing guidance has evolved — check whether affected rules need updating |
| **Platform event** | VS Code/Copilot changes | New writing tools or changed tool behaviors — check whether workflows and agents need updating |
| **Internal event** | Writing artifact modified (file watcher) | A writing artifact changed — check downstream consistency |
| **Quality signal** | Validation results, human corrections | Pattern detected in article outcomes — investigate affected rules |
| **Scheduled** | Time-based interval | Periodic full-system consistency check |

**Standards monitoring.** The system watches four primary standards sources:

| Source | URL | Check frequency | What to look for |
|---|---|---|---|
| **MWSG** | `learn.microsoft.com/en-us/style-guide/` | Monthly | Terminology changes, new guidance, deprecated recommendations |
| **Diátaxis** | `diataxis.fr` | Quarterly | Framework refinements, type definitions, structural patterns |
| **WCAG** | `w3.org/WAI/standards-guidelines/wcag/` | Quarterly | New success criteria, scoring model changes |
| **Google Style** | `developers.google.com/style` | Monthly | Alternative practices worth evaluating for adoption or contrast |

### Assess

*Inherits: PE engine Assess cycle. Adds: writing-specific assessment dimensions.*

When detection triggers assessment, the system evaluates along four dimensions:

**1. Standards alignment.** For each writing rule affected by a standards update: is the current rule still consistent with the authoritative source? If not, what's the gap — is it a terminology change, a principle shift, or a new recommendation?

*Relies on: R-W1-standards-triangulation, R-W6-rule-traceability*

**2. Internal consistency.** For each writing artifact affected by a change: do all artifacts that reference or enforce the affected rule still agree? Consistency checking traverses the artifact graph from the changed artifact through all its dependents.

*Relies on: R-W2-internal-consistency, R-W5-writing-artifact-coherence*

**3. Quality effectiveness.** For rules flagged by quality signals: what's the false-positive rate? What's the violation rate? Is the rule too strict (many false positives), too lenient (many misses), or unclear (inconsistent application)?

*Relies on: R-W3-quality-feedback*

**4. Structural health.** Standard PE assessment: metadata completeness, chain alignment, reference resolution, N-1 structural separation.

*Relies on: PE engine assessment (R-S1, R-S2, R-S5, R-P4)*

**Finding severity classification.** Follows PE v7 severity model, with writing-specific examples:

| Severity | Criteria | Writing-specific examples |
|---|---|---|
| **CRITICAL** | Blocks artifact operation or self-update infrastructure | Broken reference in instruction file, missing required metadata, self-update infrastructure staleness |
| **HIGH** | Breaks the artifact's declared contract | Rule contradicts its declared standards source, instruction ↔ context file contradiction, validation threshold contradicts style guide target |
| **MEDIUM** | Affects quality without breaking contract | Outdated example, missing coverage for new writing scenario, suboptimal readability target |
| **LOW** | Cosmetic or optimization | Wording improvement, token-saving reformulation, example modernization |

### Propose

*Inherits: PE engine Propose cycle. Adds: writing-specific evidence requirements.*

Every proposed change to a writing artifact must cite:

1. **What triggered it** — standards update, quality signal, consistency finding, or scheduled check
2. **What evidence supports it** — specific MWSG section, validation result pattern, cross-artifact inconsistency
3. **What the impact is** — which artifacts are affected, what tier blast radius
4. **What the risk is** — breaking vs. non-breaking classification with confidence

**Decomposition-first.** Large standards updates (e.g., MWSG quarterly revision) are decomposed into individual rule-level changes. Each change is independently classifiable and independently executable.

**Rule evolution types.** Proposals fall into four categories:

| Type | What changes | Autonomy routing |
|---|---|---|
| **Update** | Existing rule updated to match current standard | Depends on breaking/non-breaking + tier |
| **Add** | New rule for previously uncovered scenario | Autonomous with notification (additive) |
| **Deprecate** | Rule marked as outdated (not removed) | Human approval (even though non-breaking — prevents premature removal) |
| **Remove** | Rule removed from the system | Human approval (always — R-W4 conservative relaxation) |

### Execute

*Inherits: PE engine Execute cycle. Adds: writing-specific validation steps.*

Standard PE v7 execution applies: decomposed steps, pre-change metadata guard, multi-pass validation, post-change reconciliation, tier-scoped rollback.

**Writing-specific validation additions:**

| Validation step | What it checks | Mechanism |
|---|---|---|
| **Consistency sweep** | After applying a change to artifact X, verify all artifacts that reference X still agree | Traverse artifact graph from X; compare rule formulations |
| **Standards citation check** | Every modified rule still cites a valid standards source | Deterministic: URL exists, section matches |
| **Example coherence** | Updated rules have examples that demonstrate the updated behavior, not the old behavior | LLM pass: do examples match rules? |
| **Workflow continuity** | Workflow phases still reference existing artifacts in the correct order | Deterministic: file existence, sequence check |

### Chain alignment for writing artifacts

*Inherits: PE v7 R-S5-chain-alignment*

Writing artifacts reference each other extensively. These references must survive the renames, splits, and reorganizations that occur as writing standards evolve.

Three reference levels apply (same hierarchy as PE v7 and research):

| Level | Mechanism | When to use | Writing examples |
|---|---|---|---|
| **Level 1 (folder)** | Reference by folder path | When the entire folder's content is relevant | "See `.copilot/context/01.00-article-writing/`" |
| **Level 1.5 (category)** | Reference by functional category ID in structure index | **Default** for cross-artifact references | "See `readability-targets` category", "See `validation-dimensions` category" |
| **Level 2 (file)** | Reference by specific filename | Only for singletons or when precision is load-bearing | "See `article-writing.instructions.md`" (always-loaded, singleton) |

**Default to Level 1.5.** When agents, prompts, or workflows reference context file content, they should reference the capability needed (e.g., `readability-targets`) rather than the specific file (`01-style-guide.md`). This way, if the style guide is split into two files or renumbered, only the structure index update is needed — not every consumer.

**Structure index.** The existing STRUCTURE-README.md pattern (from PE context files) applies to the article-writing context folder. Category IDs describe writing capabilities:

| Category ID | Description | Current files |
|---|---|---|
| `readability-targets` | Quantitative readability metrics, audience calibration | `01-style-guide.md` |
| `replacement-tables` | Wordy→crisp, UI verbs, phrasal verbs, bias-free terms | `01-style-guide.md` |
| `validation-dimensions` | 7 validation dimensions, pass/fail thresholds, freshness scoring | `02-validation-criteria.md` |
| `diataxis-patterns` | Article type structures, required elements per type | `03-article-creation-rules.md` |
| `creation-rules` | Content design principles, quality checklists, technical content rules | `03-article-creation-rules.md` |
| `creation-workflow` | 7-phase article creation sequence | `workflows/01-article-creation-workflow.md` |
| `review-workflow` | Article review process and checklist | `workflows/02-review-workflow.md` |
| `series-workflow` | Multi-article series planning | `workflows/03-series-planning-workflow.md` |

### Metadata contracts for writing artifacts

*Inherits: PE v7 R-S1-metadata-driven*

Every writing artifact already carries structured YAML frontmatter with `goal`, `scope`, `boundaries`, `rationales`, and `version`. The self-update system uses these declarations for:

**Pre-change guard.** Before modifying a writing artifact, verify the change doesn't contradict the artifact's declared goal, scope, boundaries, or rationales. Contradictions are blocked.

**Post-change reconciliation.** After every change, update metadata to reflect the new state — scope expanded or narrowed, version bumped, rationales updated if invalidated.

**Example: how metadata prevents drift.** The style guide declares:

```yaml
boundaries:
  - "MUST NOT duplicate rules already in article-writing.instructions.md — provide quantitative complements only"
```

If a proposed change adds a voice-principle rule to the style guide, the pre-change guard blocks it — the boundary explicitly forbids rule duplication. The change is rerouted to `article-writing.instructions.md` where voice principles belong.

### Structural separation for writing rules

*Inherits: PE v7 R-P4-structural-separation (N-1)*

Writing artifacts are PE artifacts. The N-1 convention applies directly — rule-bearing sections label content as Rule, Rationale, or Example:

```markdown
### Sentence-style capitalization

**Rule:**
- MUST use sentence-style capitalization for all headings
- MUST capitalize only the first word and proper nouns

**Rationale:**
- MWSG consistently recommends sentence case
- Title Case creates ambiguity with proper nouns

**Example:**
- ✅ "Getting started with Azure"
- ❌ "Getting Started With Azure"
```

This separation enables deterministic change classification:

- Diff touches a **Rule** block → breaking candidate → full validation pipeline
- Diff touches only **Rationale** or **Example** blocks → non-breaking → standard autonomy routing

Without N-1, every writing-rule diff requires LLM judgment to classify — collapsing the autonomy gradient to binary.

### Quality feedback loop

*Relies on: R-W3-quality-feedback, R-G3-progressive-learning*

The quality feedback loop connects article outcomes back to writing rules:

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ Writing rules │─────▶│ Articles     │─────▶│ Validation   │
│ (artifacts)  │      │ (outputs)    │      │ (outcomes)   │
└──────────────┘      └──────────────┘      └──────┬───────┘
       ▲                                           │
       │                                           │
       └───────── Quality signal analysis ◀────────┘
```

**Signal types:**

| Signal | Source | What it indicates |
|---|---|---|
| **Consistent false positive** | `documentation-validator` flags content that human reviewers approve | Rule is too strict or poorly calibrated |
| **Consistent violation** | Articles routinely violate a rule | Rule is unclear, impractical, or conflicting with another rule |
| **Correction pattern** | Human reviewers consistently make the same correction | Missing rule or rule that doesn't match current best practice |
| **Quality improvement** | Articles score higher after a rule change | Rule change was effective (positive signal for progressive learning) |

**Phase gating.** Quality feedback requires sufficient volume to produce meaningful signal. The feedback loop activates only when outcome data accumulates:

- **Phase 1 (current):** Manual observation — humans notice patterns and trigger assessment
- **Phase 2:** Structured logging — validation results are logged with rule-level granularity
- **Phase 3:** Pattern detection — automated analysis identifies rules with high false-positive or high violation rates
- **Phase 4:** Autonomous assessment — quality patterns trigger assessment cycles without human initiation

## 📋 Design principles

### Standards-grounded evolution

*Relies on: R-W1-standards-triangulation, R-W6-rule-traceability*

Writing rules evolve because their authoritative sources evolve. A rule change must cite its source — a specific MWSG section, a WCAG criterion, a Diátaxis clarification. Rules without traceable authority are conventions; conventions are maintained differently from standards-derived rules (conventions need team consensus; standards-derived rules need source verification).

### Consistency as the primary internal quality signal

*Relies on: R-W2-internal-consistency, R-W5-writing-artifact-coherence*

When the system checks its own health, the first question is: "do all writing artifacts agree?" A style guide target that contradicts a validation threshold is a defect. An agent body that references a deprecated rule is a defect. An instruction file that says MUST while a context file says SHOULD for the same behavior is a defect. Internal consistency is cheaper to check than external standards alignment and catches the most common drift mode.

### Conservative relaxation, liberal tightening

*Relies on: R-W4-conservative-relaxation*

Tightening a writing rule (adding a check, raising a threshold) produces visible failure — articles that were passing now fail. This is detectable and reversible. Relaxing a writing rule (removing a check, lowering a threshold) produces invisible regression — articles that should fail now pass. This is undetectable until quality degrades.

**Consequence:** Rule tightening follows the normal autonomy gradient. Rule relaxation ALWAYS requires human approval with evidence of false positives or impracticality.

### Shared governance from PE v7

The following PE v7 design principles apply directly (not repeated in full — see [PE vision](../self-updating-prompt-engineering/20260428.01-vision.v7.md)):

- **Staleness avoidance first** — writing self-update infrastructure is validated before it acts on writing artifacts
- **Human governance, autonomous execution** — humans set writing standards direction; the system executes maintenance
- **Minimal disruption** — quiet when healthy, specific when not
- **Scoped over exhaustive** — tier + dependency analysis, not full-system audit every cycle
- **Evidence-based** — every proposed change cites trigger, evidence, and justification
- **Graceful degradation** — if a standards source is unavailable, flag uncertainty rather than guess
- **Deterministic where possible** — cross-reference checks, metadata validation, consistency audits are deterministic
- **Portable by design** — engine/integration separation
- **Discovery over prescription** — discover writing artifact structure from what exists
- **Metadata-driven** — structured metadata enables pre-change guards and post-change reconciliation
- **Metadata-guarded changes** — block contradictions before apply; reconcile metadata after apply
- **Iteration budget** — bounded autonomous changes per cycle
- **Loop stability** — convergence toward a stable writing-standards state

## 🔍 What "self-updating article writing" means (and what it doesn't)

**It means:**

- Monitoring writing-standards sources for updates that affect writing artifacts
- Detecting internal inconsistencies across the writing artifact graph
- Analyzing article quality outcomes for evidence about rule effectiveness
- Proposing evidence-based corrections to writing artifacts (instruction files, context files, agents, prompts, workflows, skills, templates)
- Executing low-risk corrections autonomously with notification
- Escalating high-risk or uncertain changes to a human with full evidence
- Tracking outcomes to tune autonomy thresholds and rule effectiveness
- Maintaining cross-artifact consistency after every change

**It doesn't mean:**

- Changing writing-standards direction without human authorization
- Relaxing rules without evidence of false positives or impracticality
- Creating entirely new writing frameworks or artifact types from scratch
- Writing or reviewing articles (that's the article-writing pipeline's job)
- Discovering subject knowledge for articles (that's the research system's job)
- Replacing human judgment for voice, tone, or editorial direction
- Guaranteeing zero drift (some lag between standards change and detection is expected)
- Modifying the vision, architectural principles, or risk thresholds autonomously

## ⚠️ Risks and tensions

### Writing-standards interpretation

Standards like MWSG and WCAG don't always give binary guidance. "Write in a warm, relaxed tone" requires interpretation. When a standard updates ambiguous guidance, the system can detect the change but may not reliably determine whether existing rules already comply. *Mitigation: ambiguous standards changes always escalate to human review. Only clear, testable guidance changes are candidates for autonomous processing.*

### Quality signal noise

Article outcomes are influenced by many factors beyond writing rules — topic difficulty, author expertise, time pressure. Attributing quality outcomes to specific rules requires careful analysis to avoid spurious correlations. *Mitigation: quality signal thresholds require high confidence (>N% pattern consistency) before triggering assessment. Phase gating delays automated quality analysis until sufficient volume accumulates.*

### Consistency vs. evolution tension

Perfect internal consistency can resist necessary change — a rule that should evolve is blocked because changing it introduces temporary inconsistency with dependent artifacts. *Mitigation: decomposition-first — large changes are broken into steps that propagate through tiers. Temporary inconsistency within a decomposed sequence is accepted; inconsistency at cycle completion is not.*

### Instruction file collision risk

Article-writing instruction files auto-inject for all `.md` files. If the PE self-update engine is also installed with its own instruction files, overlapping `applyTo` patterns cause silent collisions. *Mitigation: instruction minimization (R-S8) — writing instruction files carry only testable/mechanical rules. Behavioral guidance stays in context files and agent bodies.*

### Over-engineering the feedback loop

Building infrastructure for quality signal analysis before there's sufficient volume produces empty scaffolding. *Mitigation: phase-gated activation — manual observation first, structured logging second, automated analysis only when volume justifies it.*

### Standards source availability

MWSG, Diátaxis, and WCAG are web resources that can be temporarily unavailable, restructured, or paywalled. *Mitigation: graceful degradation — if a standards source is unavailable, the cycle continues with time-based fallback checks. Missing source access is logged, not escalated unless persistent.*

### False confidence from process rigor

A multi-layer assessment process can create false confidence: "it passed consistency, standards, and structural checks — it must be right." Process rigor reduces error probability but doesn't eliminate it. The effectiveness hard limit from PE v7 applies: validation can check structural health, but only actual use reveals whether rules produce good articles. *Mitigation: progressive learning tracks actual outcomes. Confidence is always explicit.*

### Complexity creep

The writing artifact graph has 15+ artifacts across 6 tiers. Adding self-update logic for every cross-reference and consistency dimension creates maintenance burden on the maintenance system. *Mitigation: start with the most common drift modes (Tier 0–1 inconsistency, standards misalignment) and expand only when evidence shows additional coverage is needed.*

## ✅ Success criteria

The self-updating article-writing system succeeds when:

1. **Internal consistency stays high.** Cross-artifact contradictions are detected within the same cycle they're introduced — not discovered weeks later during article review. *Metric: number of cross-artifact inconsistencies found per cycle, trending toward zero.*

2. **Standards alignment stays current.** Writing rules reflect the current edition of their authoritative sources within one detection cycle of the change. *Metric: time from standards update to writing-artifact update.*

3. **Article quality improves measurably.** Articles produced under maintained writing rules score higher on validation dimensions than articles produced under stale rules. *Metric: average validation scores over time.*

4. **False positive rate stays low.** Writing rules that generate false positives are identified and corrected. *Metric: ratio of validation findings confirmed by human review vs. overridden.*

5. **Manual consistency audits become redundant.** Full cross-artifact reviews shift from discovery to confirmation. *Metric: count of HIGH/CRITICAL findings in periodic audits.*

6. **Rule changes are traceable.** Every writing-rule modification links to a standards source, quality signal, or consistency finding. *Metric: percentage of rule changes with documented evidence.*

7. **The system stays simple enough to maintain.** Self-update machinery for writing artifacts is understandable and debuggable. *Metric: percentage of meta-infrastructure changes requiring human escalation.*

8. **Autonomous execution success rate exceeds threshold.** Autonomous writing-artifact changes pass post-implementation verification at >95%. *Metric: success rate of autonomous changes over rolling 30 days.*
