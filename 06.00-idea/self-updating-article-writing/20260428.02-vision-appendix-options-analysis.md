# Vision appendix: options analysis for article maintenance architecture

> **This is an appendix to the [article-writing self-update vision](20260428.01-vision.v1.md).** It evaluates three architectural approaches for maintaining published articles. The vision adopts Option B (event-driven maintenance with autonomous execution).

---

## Three architectural approaches

### Option A: Scheduled batch reviews

A periodic (weekly/monthly) process runs validation across all articles, produces a report, and queues fixes.

**Strengths:**
- Simple implementation — a scheduled prompt that runs validation
- Catches degradation before it becomes critical
- Report-based — human reviews findings before fixes are applied

**Weaknesses:**
- Batch processing is expensive — validates everything, including articles that haven't changed
- No event-driven response — a broken link waits until the next batch run
- All-or-nothing — can't prioritize P0 issues over P4 enhancements
- Batch reports can be overwhelming — 50 findings across 30 articles is hard to triage

**Verdict:** Better than manual, but inefficient. Wastes tokens on healthy articles and misses urgent issues between batch runs. A useful fallback for scheduled reviews but not the primary strategy.

### Option B: Event-driven maintenance with autonomous execution (recommended)

The system continuously monitors articles through freshness scoring and external signals. When degradation is detected, it triggers scoped assessment, proposes targeted fixes, and executes low-risk fixes autonomously while escalating high-risk changes.

**Strengths:**
- Responds to degradation when it happens, not when the next batch runs
- Efficient — only assesses articles with detected signals, not the entire inventory
- SLA-aware — P0 issues (security, accuracy) trigger faster response than P4 (style)
- Autonomy gradient — broken link fixes are autonomous; factual rewrites escalate
- Quality feedback loop — article outcomes signal PE artifact issues to the PE engine

**Weaknesses:**
- Requires orchestration infrastructure (trigger wiring, signal processing)
- Autonomous execution carries risk — incorrect fixes degrade articles
- Depends on reliable freshness scoring and signal detection

**Verdict:** Recommended. This approach combines the efficiency of targeted responses with the safety of the autonomy gradient.

### Option C: Full article rewrite pipeline

When an article's freshness drops below a threshold, regenerate it from scratch using builder capabilities with current sources.

**Strengths:**
- Guarantees current content
- Avoids incremental drift — each version is generated fresh

**Weaknesses:**
- Extremely expensive — full article generation costs far more than targeted fixes
- Loses editorial voice and human-contributed nuance
- Author attribution becomes meaningless
- Doesn't preserve deliberate editorial choices
- Overkill for minor degradation (broken link → full rewrite?)

**Verdict:** Rejected. Regeneration should be a last resort for articles with freshness score <25, not the default maintenance strategy.
