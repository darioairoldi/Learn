# Article maintenance — canonical invocation surface

> Appendix to [20260428.01-vision.v1.md](20260428.01-vision.v1.md). It specifies the invocation surface the `predictable-invocation-surface` principle requires. This is a contract, not an implementation — the orchestrator that executes it is parked (see the companion plan's park lot).

## Why a small surface

A wide invocation surface (many flags, per-collection defaults, preset aliases) forces the operator to remember which option applies where, and a parameter-less request with no contract gets silently narrowed to the cheapest pass. The surface answers with **four canonical parameters**, a **default-full** contract, and a **resolved-scope echo** on every run.

## The four parameters

| Parameter | Selects | Values (default in **bold**) |
|---|---|---|
| `--scope` | which articles | a single article · a series · a folder · a subject · a tag set · **all** |
| `--dim` | which quality dimensions | verified · reliable · readable · understandable · up-to-date · the collection dimensions · **all applicable** |
| `--deps` | how far to traverse | none · direct · full — follow cross-references, collection links, and cited sources · **the command's natural depth** |
| `--start` / `--end` | research window | a date (`YYYY-MM-DD`, `now`, `-90d`) OR a source-version token resolved to a timestamp · **unbounded for a full run; the trigger delta for a trigger-fired run** |

`--scope` and `--deps` compose orthogonally (target set ∪ dependency traversal); each included article is assessed with the dimensions applicable to it.

## Default-full and resolved-scope echo

- A **manual, parameter-less** invocation is a **full review**: every in-scope article, every applicable dimension, every monitored source. Parameters only *narrow* the full default — the system never opts into a narrower scope on the operator's behalf.
- A **trigger-fired** invocation (freshness drop, broken link, source change, scheduled review) is **incremental**: the trigger is the delta signal.
- Every run echoes its **resolved scope** as the first line of its report — `articles × dimensions × window` — so what ran can never silently diverge from what was asked.

## Collection scope

`--scope` accepts a collection (series, folder, subject, or tag). A collection run applies the collection-structural gate (orphan, scope overlap, navigation chain) before per-article work, per the *Series before articles* (collection) principle.

## What this appendix is (and isn't)

This surface adds no special flags: the target collection and the time window are *resolved* attributes, not parameters. The appendix defines the **contract** only — the dimension vocabulary and the freshness/SLA formula come from the vision, and building the orchestrator that executes this surface is out of scope here and tracked separately.
