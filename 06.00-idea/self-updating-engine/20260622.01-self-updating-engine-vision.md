## 🎯 The problem

> **Scope note.** This vision defines the **portable engine** — the machinery shared by every self-updating system. It is **domain-agnostic**: each domain (prompt engineering, article writing, research, and future domains) instantiates the engine by supplying a configuration — its identity, quality model, sources, and thresholds — and never re-implements the machinery. This document does **not** define any one domain's quality rules, nor a concrete implementation.

Several kinds of managed artifacts — prompt-engineering artifacts, published articles, research outputs, and more — **degrade through the same forces and need the same maintenance machinery**. Each silently drifts (a declared scope no longer matches the content), each goes stale against an external reality, and each carries cross-references that rot. The response in every case is the same shape: detect the drift, assess it at a cost proportional to the evidence, propose a scoped correction, and execute it autonomously when the risk is low enough.

Today that machinery lives **inside one domain** (the prompt-engineering system) and is *borrowed* by the others. Three problems follow:

1. **Coupling.** A domain that borrows the machinery is tied to the domain that owns it, even though their subject matter is unrelated.
2. **Re-derivation.** Every new self-updating domain re-discovers the autonomy gradient, the risk model, and the metadata guards instead of inheriting them.
3. **Drift between copies.** The same rule, expressed slightly differently in each domain, diverges over time.

The shared challenges are real in every domain: the **cost** of assessing many units against many external sources; the **usability** of invoking maintenance predictably; and the **autonomy** question of how much the system may change without a human. These are engine concerns, not domain concerns.

**A sustainable answer is a single engine** that owns the shared machinery and exposes a clean seam: the engine ships the verbs (detect, assess, classify risk, guard, roll back, learn); each domain supplies the nouns (what an artifact is, what makes it stale, what "good" means).

## 💡 The goal

Build an engine that **maintains any set of managed artifacts at peak reliability, effectiveness, and efficiency — with the maximum autonomy that assessed risk allows** — by continuously detecting drift, assessing it proportionally, proposing scoped corrections, and executing validated ones. Each domain instantiates the engine with a configuration; nothing domain-specific lives in the engine.

> **Priority for implementations.** When subsections conflict, resolve toward *metadata-guarded changes within risk-calibrated autonomy bounds*, and never expand autonomous-execution scope to reduce assessment cost.

### Key definitions

| Term | Definition |
|---|---|
| **Managed artifact** | Any unit the engine maintains — a PE artifact, a published article, a research output, or a future artifact type |
| **Domain** | A family of managed artifacts with a shared identity, quality model, and sources (e.g. prompt engineering, article writing) |
| **Domain config** | The object a domain supplies to instantiate the engine: identity (`domain_profile`), quality model, sources, freshness formula, autonomy thresholds |
| **Autonomy gradient** | The risk-calibrated scale from autonomous to human-only |
| **Change classification** | Breaking vs non-breaking, assessed across consistency, efficiency, behavioral compatibility, and effectiveness |
| **Processing unit (PU)** | One `(artifact × applicable-dimension)` pair — the unit of coverage state |

### The maintenance loop

```
    ┌──────────────────────────────────────────────────────┐
    │                                                      │
    ▼                                                      │
┌───────────┐    ┌───────────┐    ┌───────────┐    ┌───────┴───┐
│  DETECT   │───▶│  ASSESS   │───▶│  PROPOSE  │───▶│  EXECUTE  │
│           │    │           │    │           │    │           │
│ triggers  │    │ cost-     │    │ scoped    │    │ verified  │
│ + sources │    │ stratified│    │ change +  │    │ change +  │
│ + state   │    │ + graded  │    │ autonomy  │    │ rollback  │
│           │    │ verdict   │    │ routing   │    │ ready     │
└───────────┘    └───────────┘    └───────────┘    └───────────┘
```

### The autonomy gradient

Autonomy is a gradient calibrated to **assessed impact × confidence**:

| Autonomy level | Scope | Examples (any domain) |
|---|---|---|
| **Autonomous** | Low impact, high confidence | Deterministic fixes, metadata-timestamp refresh |
| **Autonomous with notification** | Low-medium impact, validated | Verified factual/version update, redundancy removal after validation |
| **Human approval required** | High impact or medium confidence | Rule/behavior changes, scope expansion, external-artifact adoption |
| **Human-only** | Architectural or strategic | Vision changes, principle changes, threshold changes |

The gradient is **configurable per domain** — a domain calibrates the thresholds to its own risk (for example, a reader-facing domain tightens content thresholds; an infrastructure domain weights blast radius). The *shape* is the engine's; the *thresholds* are the domain's.

## ⚙️ The rationale

> **Terminology note.** These are **engine-level design rationales** — the architectural reasons the engine is built this way. Each is referenced by a readable kebab-case identifier so a principle's `relies_on` is legible without cross-referencing. They are distinct from per-artifact `rationales:` metadata.

| ID | Rationale | Design implication |
|---|---|---|
| **llm-reasoning** | Models can analyze artifacts, compare against specifications, and make judgment calls. | Delegate analysis and judgment to models; reserve humans for above-threshold decisions. |
| **deterministic-processing** | Parsing, existence checks, timestamp and hash comparison need no judgment. | Use deterministic tools for predictable operations; reserve model reasoning for the rest. |
| **metadata-driven** | Machine-readable `goal`/`scope`/`boundaries`/`rationales` turn implicit intent into checkable state. | Every artifact declares its contract; the engine guards and grounds on it. |
| **dependency-graph** | A change's blast radius is the set of artifacts that depend on it. | Scope validation to what a change actually affects; weight risk by coupling. |
| **decomposition** | Model accuracy degrades as context grows; large tasks must be split. | Decompose maintenance into focused, individually-verifiable steps. |
| **self-correction** | A single pass introduces errors; an independent pass catches them. | No change commits without at least one independent validation pass. |
| **progressive-depth** | Full-depth assessment of everything does not scale. | Three phases — research (once) → screening (per unit) → deep (escalated only). |
| **validation-caching** | A unit validated against the current world and unchanged need not be re-checked. | Skip provably-current units; never cache the engine's own meta-infrastructure. |
| **autonomy-gradient** | Autonomy is not binary; it scales to risk. | Classify every change on the gradient; make the gradient configurable. |
| **trust-calibrated-adoption** | External sources and artifacts deserve unequal trust. | Weight evidence and adoption by source trust. |
| **progressive-learning-rationale** | Outcomes reveal which changes helped and which regressed. | Log every autonomous action and its outcome; tune thresholds from the log. |
| **portable-packaging** | Engine logic delivered through standard mechanisms imposes no framework. | Package for MCP / SDK / extension; discover conventions rather than prescribe them. |
| **external-knowledge** | Models can consume a domain's authoritative documentation directly. | Monitor each domain's declared sources; ground assessment in the domain's profile. |
| **structured-contracts** | Probabilistic output feeding deterministic processing needs a schema bridge. | Inter-phase communication uses defined schemas; the invocation surface is canonical. |

> **Conflict precedence.** Safety takes precedence over efficiency; reliability over autonomy. `metadata-guarded-changes` and `staleness-avoidance-first` win over any cost or depth optimization.

## 🏗️ The vision

A self-updating engine **continuously monitors managed artifacts for drift, assesses drift at a cost proportional to the evidence, proposes scoped corrections, and executes validated ones autonomously when risk is low** — for any domain, governed by that domain's configuration.

### The engine/integration seam

The engine is split from its per-consumer configuration along one contract:

| Side | Owns |
|---|---|
| **Portable engine** (ships once) | The Detect → Assess → Propose → Execute loop; the autonomy gradient; change-risk classification; metadata guards + runtime grounding; cost-stratified assessment; processing state; the graded verdict; the canonical invocation surface; rollback; the outcome log |
| **Per-consumer config** (supplied per domain) | Artifact paths; the **domain profile** (role + authoritative sources); the quality model (dimensions + cost tiers); the freshness/health formula; autonomy thresholds; detection sources |

A domain instantiates the engine by supplying *only* the config side. Nothing domain-specific lives in the engine; nothing engine-generic is re-implemented in a domain.

### The domain-config contract

Every domain supplies, at minimum:

| Config element | Purpose |
|---|---|
| **`domain_profile`** | The domain's identity-and-contract: role, authoritative sources, monitored invariants — the grounding for `domain-expertise-injection` |
| **Quality model** | The domain's assessable dimensions, each with a cost tier and trigger condition |
| **Degradation forces** | What makes this domain's artifacts drift or go stale |
| **Health formula** | How freshness/health is scored and what triggers assessment |
| **Autonomy thresholds** | Where this domain sets the gradient's cut points |

### Assessment: cost-stratified and state-aware

Assessment runs in three phases — **research** (once per invocation: what changed in the world), **screening** (per unit: deterministic checks + lightweight relevance), and **deep** (only escalated units). State is tracked on two axes: a **source ledger** (what each source last published) and a **`(unit × dimension)` coverage record** (what each dimension was assessed against). A processing unit that has never been assessed is always in the work set; a unit is skipped only when provably current. Each dimension carries a **graded verdict** — `verified` / `pass-weak` / `partial` / `fail` — computed from its evaluated/declared sub-check ratio, so a dimension can never go green on its easiest check.

### Invocation: predictable by contract

Maintenance is invoked through a **small canonical surface** that selects scope, dimensions, dependency traversal, and a time window. A parameter-less invocation is **default-full**; every run **echoes its resolved scope** first, so what executed can never silently diverge from what was asked. The exact parameter names are an integration detail; the *contract* (minimal, default-full, echoed) is the engine's.

### Delivery vehicles

The same engine ships through different interfaces — an **MCP server** (primary), an **SDK component**, a **VS Code extension**, or a **skill pack**. The vehicle changes the interface, not the engine logic.

## 📋 Design principles

> **Priority for implementations.** When principles conflict, the P0 invariants win; every other principle operates within the boundary they establish.

### The engine executes only within human governance

**Priority: P0** · `human-governed-autonomous-execution`

*Relies on: autonomy-gradient, self-correction*

Humans own the vision, the principles, and the risk thresholds. The engine never changes its own purpose or governing rules without explicit human direction; within those rules it executes improvements autonomously, provided it can validate that risk is low enough.

### Autonomy is risk-calibrated

**Priority: P0** · `risk-calibrated-autonomy`

*Relies on: autonomy-gradient, dependency-graph*

Every change is classified by assessed impact × confidence and routed on the gradient. A command's default mode is a workflow convenience, never a permission ceiling — the only legitimate gate is assessed risk.

### Changes are guarded by metadata

**Priority: P0** · `metadata-guarded-changes`

*Relies on: metadata-driven, self-correction*

Before a change applies, it is checked against the artifact's declared `goal`/`scope`/`boundaries` and **blocked by default** if it contradicts them. After a change applies, the artifact's metadata is **reconciled** to its new state. At execution time, the same metadata is enforced with precedence over body content (runtime grounding), including a staleness check on any inherited metadata.

### Staleness avoidance comes first

**Priority: P0** · `staleness-avoidance-first`

*Relies on: metadata-driven, validation-caching*

The engine's own infrastructure is validated on every cycle and never cached. Suspect meta-state fails closed — a missed review beats a wrong change made on stale self-knowledge.

### Portable by design

**Priority: P0** · `portable-by-design`

*Relies on: portable-packaging, external-knowledge*

The engine holds reusable logic; the integration connects it to a domain's artifacts and sources. The engine discovers conventions rather than imposing structure, and never hard-codes a domain, source, or path.

### Decisions are evidence-based

**Priority: P1** · `evidence-based-decisions`

*Relies on: structured-contracts, llm-reasoning*

Every proposal cites its trigger, its assessment, and the evidence behind each dimension's graded verdict. A dimension whose evidence was not gathered reports *not assessed* — never a low-confidence pass.

### Assessment cost is proportional

**Priority: P1** · `cost-proportional-assessment`

*Relies on: progressive-depth, validation-caching*

Tier 0 (deterministic) runs first and cheaply; deeper tiers run only for the dimensions the trigger demands. The engine never applies more assessment than the evidence requires.

### Deterministic where possible

**Priority: P1** · `deterministic-where-possible`

*Relies on: deterministic-processing, structured-contracts*

Predictable operations — parsing, reference resolution, hashing, timestamp comparison — use deterministic tools; model reasoning is reserved for judgment.

### Decompose and earn autonomy per step

**Priority: P1** · `decompose-and-earn-autonomy`

*Relies on: decomposition, autonomy-gradient*

Large changes decompose into the smallest individually-verifiable steps; each step earns its own autonomy, so one escalation never blocks an independent batch.

### Reversible by default

**Priority: P1** · `reversible-by-default`

*Relies on: self-correction, deterministic-processing*

Before any change, the engine records a snapshot; rollback is one deterministic operation. Reversibility is what makes autonomy grantable at all.

### Predictable invocation surface

**Priority: P1** · `predictable-invocation-surface`

*Relies on: structured-contracts*

A small canonical surface governs every run, defaults to a full review, and echoes its resolved scope first.

### Domain-expertise injection

**Priority: P1** · `domain-expertise-injection`

*Relies on: external-knowledge, metadata-driven*

Before assessing a domain's artifact, the engine adopts that domain's declared role and authoritative sources (its `domain_profile`); when no profile exists it degrades transparently and records the gap, never assuming generic equivalence.

### Progressive learning

**Priority: P1** · `progressive-learning`

*Relies on: progressive-learning-rationale*

Every autonomous action and its outcome — including human rejections and their stated reasons — is logged, and the log tunes the thresholds over time.

### Loop stability

**Priority: P2** · `loop-stability`

*Relies on: self-correction, progressive-learning-rationale*

The loop converges toward a fixed point. Increasing change volume without new triggers (divergence) or alternating changes (oscillation) freezes the affected scope and escalates.

### Minimal disruption

**Priority: P2** · `minimal-disruption`

*Relies on: autonomy-gradient*

The engine is quiet when artifacts are healthy and specific when they are not; notifications are actionable, not informational.

## 🔍 What "the self-updating engine" means (and what it doesn't)

**It means:**

- Defining the **shared machinery** — the loop, the autonomy gradient, the risk classifier, the metadata guards, the state model, the graded verdict, the invocation contract — once, for every domain.
- Exposing a **clean seam** so a domain instantiates the engine with configuration only.
- Grounding assessment in a domain's expertise through its declared profile.
- Executing validated, reversible changes autonomously within human-set thresholds.

**It doesn't mean:**

- Defining any one domain's **quality model** or **degradation forces** — those are domain config.
- Prescribing a concrete **implementation** — the engine is a contract, realized by an MCP server, SDK, or extension.
- Changing its own **vision or principles** autonomously — those are human-only.
- Forcing a **migration** — when and whether existing domains move onto this engine is a separate, owner-made decision.

## ⚠️ Risks and tensions

### Over-abstraction

An engine pitched too abstractly fits every domain and serves none. *Mitigation: the engine is extracted from a mature, working system, not designed in a vacuum; every engine concept already has at least one real domain instantiation.*

### Premature coupling of domains to the engine

Pinning domains to an immature engine spreads churn. *Mitigation: domains pin an engine version as a building-block dependency; an engine change is an explicit, reviewable bump per consumer — and migration is deferred until the engine is validated.*

### Configuration that smuggles logic

A domain config that encodes engine logic defeats the seam. *Mitigation: the config contract is closed — identity, quality model, sources, thresholds — and anything beyond it is a signal the seam is wrong, not the config.*

### Loop instability across domains

A shared loop that diverges in one domain could erode trust in all. *Mitigation: `loop-stability` is a P2 invariant; divergence/oscillation freezes the affected scope only and escalates.*

### Governance erosion

A shared engine could become a place where domain-specific exceptions accumulate. *Mitigation: vision and principles are human-only; exceptions belong in domain config, not the engine.*

---

## ✅ Success criteria

The self-updating engine succeeds when:

1. **One machinery, many domains.** At least two domains instantiate the engine with configuration only, sharing the loop, gradient, and guards. *Metric: count of domains consuming the engine with no machinery re-implementation.*
2. **No cross-domain references.** No domain vision references another domain — only the engine. *Metric: cross-domain reference count (target: zero).*
3. **The seam holds.** Every domain's footprint is identity + quality model + sources + thresholds, with no engine logic. *Metric: domain-config audits show no machinery leakage.*
4. **Autonomy is earned, not assumed.** Autonomous changes carry a snapshot and a graded verdict; none is committed above its risk tier. *Metric: zero above-tier autonomous commits (rolling 90 days).*
5. **Learning moves thresholds.** The outcome log produces threshold adjustments over time. *Metric: count of evidence-backed threshold changes per quarter.*
6. **Versioning is explicit.** Each domain pins an engine version; engine bumps are reviewed per consumer. *Metric: every consumer declares a pinned engine version.*
