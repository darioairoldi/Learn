# Vision update plan: elevate structural separation (N-1)

## 🎯 The gap

Structural separation (N-1) — the convention that rule-bearing sections separate **rules**, **rationales**, and **examples** into distinct, labeled blocks — appears in the v4 vision in seven places:

| Location | How N-1 appears | Problem |
|---|---|---|
| YAML `scope.covers` | Listed as a bullet: "Rules/rationales/examples structural separation (N-1)" | Label only — no reader knows what this means until they reach the design principle |
| Rollout phases table (Phase 3) | Precondition: "rule/rationale/example separation (N-1) in place" | Positioned as a Phase 3 concern — suggests it's a late-stage, nice-to-have prerequisite |
| Design principle: "Rules, rationales, and examples are separable" | Full description with convention template | Buried as one of ~15 peer-level principles — no signal that it's more foundational than the others |
| Execute section (static validation) | References "breaking/non-breaking classification" that depends on N-1 | Dependency is implicit — "classification" is mentioned but N-1 isn't named |
| Breaking vs. non-breaking table (The goal) | Explains the classification framework | Never states that N-1 is what makes this classification deterministic rather than LLM-dependent |
| Metadata-driven principle | Explains that deterministic comparison needs structured declarations | Doesn't connect to N-1 as the structural counterpart that makes content (not just metadata) classifiable |
| Metadata-guarded changes principle | Describes pre-change and post-change guards | Doesn't explain that N-1 is what enables guards to operate on content blocks rather than entire files |

**The core problem:** A reader can go through the entire vision and understand *that* N-1 exists without ever understanding *why it's load-bearing*. The causal chain — N-1 enables deterministic classification, which enables earned autonomy, which enables the vision's goal — is never stated.

---

## 💡 Why N-1 is critical to ease of use

The vision promises a system that "executes validated corrections autonomously when risk is low enough." For that promise to be practical, the system must answer one question cheaply and reliably: **is this change breaking or non-breaking?**

Without N-1, answering that question requires an LLM to read the full content of the changed section, understand the semantic intent of each sentence, and judge whether any sentence constitutes a rule, a rationale, or an example — and then determine whether the diff affects a rule (breaking) or a rationale/example (non-breaking). This is expensive, slow, and unreliable. It means:

- Every diff escalates to human review (the system can't confidently classify)
- The autonomy gradient collapses to a binary: autonomous for trivial structural changes, human-required for everything else
- Token cost per assessment cycle grows linearly with artifact content size
- The "earned autonomy per step" promise becomes theoretical

With N-1, answering that question is deterministic:

1. Parse the changed section's labeled blocks
2. If the diff touches a `**Rule**:` block → breaking candidate → route through full validation
3. If the diff touches only `**Rationale**:` or `**Example**:` blocks → non-breaking → eligible for autonomous execution

This is the **single mechanism** that bridges the gap between the vision's design (metadata-driven, deterministic-where-possible) and its operational promise (earned autonomy, minimal disruption). Without it, the design principles are internally consistent but practically unachievable.

### The causal chain

```
N-1 structural separation
  → deterministic rule/rationale/example identification
    → deterministic breaking/non-breaking classification
      → confident risk assessment per step
        → earned autonomy per step (not per change-set)
          → the vision's core promise: autonomous maintenance at bounded risk
```

Every link in this chain depends on the one below it. N-1 is the foundation.

## 📋 Proposed changes (priority-ordered)

### Change 1 (HIGH): Add N-1 to "The rationale" section as a processing strategy

**Where:** The rationale → Processing strategies table (after R-P3)

**What:** Add a new rationale entry:

> | **R-P4-structural-separation** | **Structural separation for deterministic classification.** When rule-bearing sections separate rules, rationales, and examples into distinct labeled blocks, breaking/non-breaking classification becomes deterministic: a diff touching a Rule block is a breaking candidate; a diff touching only Rationale or Example blocks is non-breaking. Without this separation, every diff requires LLM judgment to classify — lower confidence, higher cost, lower autonomy eligibility. | Require the N-1 structural convention for all rule-bearing artifacts before they're eligible for autonomous semantic edits. Classification logic checks block labels first, content second. |

**Why this is the highest priority change:** The rationale section is where readers learn *why* design decisions exist. Without an explicit rationale entry, N-1 appears to be a formatting preference rather than a load-bearing architectural choice.

**Impact on existing content:**
- Process-to-rationale mapping needs a new row: "Analyze" and "Validate" steps add R-P4
- Conflict resolution table: R-P4 vs R-P2 — R-P4 is orthogonal to decomposition (operates at section level, not artifact level), no conflict

---

### Change 2 (HIGH): Rewrite breaking/non-breaking section to show the N-1 dependency

**Where:** The goal → Breaking vs. non-breaking classification

**What:** After the existing table and the multi-dimensional impact paragraph, add a subsection:

> ### How classification works in practice
>
> The breaking/non-breaking distinction is only useful if it can be assessed cheaply and confidently. The mechanism that enables this is **structural separation (R-P4-structural-separation)**: the N-1 convention where rule-bearing sections label their content as Rule, Rationale, or Example.
>
> With N-1 in place, classification follows a deterministic path:
>
> 1. Identify which labeled blocks the diff touches
> 2. If any `Rule` block is modified → breaking candidate → full validation pipeline
> 3. If only `Rationale` or `Example` blocks are modified → non-breaking → standard autonomy routing
> 4. If the section lacks N-1 labels → fall back to LLM classification (lower confidence, higher cost)
>
> Without N-1, step 1 is impossible and every diff requires LLM judgment — collapsing the autonomy gradient to a binary: trivial structural changes (autonomous) vs. everything else (human-required).

**Why this matters:** The breaking/non-breaking classification is the vision's decision mechanism. Showing that N-1 is what makes it practical — not theoretical — connects the design principle to the operational promise.

---

### Change 3 (MEDIUM): Reposition the design principle from peer-level to foundational

**Where:** Design principles section

**What:** Move "Rules, rationales, and examples are separable" from its current position (principle #2, after "Staleness avoidance first") to immediately after "Staleness avoidance first" but with a **structural elevation**:

Add a brief introductory paragraph before the design principles list:

> Two principles are **foundational** — they enable the mechanisms that all other principles depend on:
>
> 1. **Staleness avoidance first** — ensures the system's own infrastructure is trustworthy before it acts on managed artifacts.
> 2. **Rules, rationales, and examples are separable** — enables deterministic classification that makes the autonomy gradient practical rather than theoretical.
>
> The remaining principles operate within the foundation these two establish.

Then keep the existing principle content but add one sentence at the end:

> This separation is the structural counterpart of the metadata-driven principle — metadata makes *artifact-level* classification deterministic; N-1 makes *content-level* classification deterministic. Together, they enable the earned-autonomy-per-step model.

**Why:** Readers currently encounter ~15 design principles with no signal that two of them are load-bearing for the entire architecture. The elevation doesn't remove content — it adds navigation.

---

### Change 4 (MEDIUM): Add N-1 to the Assess layer description

**Where:** The vision → Assess

**What:** In the Assess description, after "Chain alignment checks (R-S5-chain-alignment) run here," add:

> For artifacts with N-1 structural separation, diff classification is deterministic: parse labeled blocks, check which block type changed, route accordingly. For artifacts without N-1, classification falls back to LLM judgment — flagging a capability gap and recommending N-1 adoption.

**Why:** Assess is where classification actually happens. Showing that N-1 is the mechanism used during assessment connects the design principle to the operational layer.

---

### Change 5 (MEDIUM): Reframe Phase 3 precondition language

**Where:** Rollout phases table

**What:** Change the Phase 3 "Why" column from:

> These encode rules in prose. Without N-1, every diff escalates. Without structured metadata, scope preservation can't be mechanically checked.

To:

> These encode rules in prose. N-1 structural separation (R-P4) is the mechanism that enables deterministic diff classification — without it, every content diff requires LLM judgment and escalates by default. Structured metadata (R-S1) enables mechanical scope preservation. Together they unlock autonomous execution for rule-bearing artifacts.

**Why:** The current phrasing states the consequence ("every diff escalates") but doesn't explain the mechanism. The rewrite connects Phase 3's precondition to the rationale, making the rollout logic self-explanatory.

Together, they form the two-layer classification mechanism the autonomy gradient depends on.

**Why:** Connects the two complementary mechanisms that readers currently encounter as unrelated principles.
