# Vision: future improvements and deferred infrastructure

> Items deferred from the current improvement cycle. Each has a **trigger condition** — the signal that moves it from backlog to active work.

---

## 🏗️ Deferred self-update infrastructure

### Structured outcome logging (JSONL)

**What**: `outcome-log.jsonl` — append-only structured log of all autonomous changes with schema: `{change_id, artifact_path, artifact_tier, pre_hash, post_hash, pre_version, post_version, classification, confidence, autonomy_level, trigger_type, trigger_source, validations, outcome}`

**Why deferred**: Currently nobody writes to it (no autonomous changes exist) and nobody reads it (agents read markdown). Building empty JSONL infrastructure that no agent consumes is speculative.

**Trigger condition**: When the meta-prompt healthcheck cycle produces >20 findings per run, OR when autonomous changes are enabled (Phase 2+ of the rollout), structured logging justifies its cost. The existing `05.04-meta-review-log.md` handles current volume.

**Vision reference**: R-G3-progressive-learning

---

### Structured rename log (JSONL)

**What**: `rename-log.jsonl` — tracks file/rule renames with `{old, new, date, reason}` entries for automated reference rewrite.

**Why deferred**: Git already tracks renames. The Phase 0 rename (62 files) was done successfully with `grep_search` + `replace_string_in_file` — no rename log was needed. JSONL adds value only when renames happen frequently enough that manual reference updates are a bottleneck.

**Trigger condition**: When file renames happen more than once per month, or when chain alignment (R-S5) needs automated reference rewriting.

**Vision reference**: R-S5-chain-alignment

---

### Self-update config file

**What**: `pe-self-update.config.json` — declares artifact paths, autonomy thresholds, iteration budget, monitored sources, state namespacing.

**Why deferred**: No agent or hook reads a JSON config. All behavior is configured via YAML frontmatter in agents/prompts and context files in `.copilot/context/`. A config file adds value only when the self-update engine is extracted as a portable, installable component.

**Trigger condition**: When the PE system is ported to a second repository, or when the engine is packaged as an MCP server / VS Code extension (Phase 0 of the rollout).

**Vision reference**: R-S7-portable-packaging, per-repo integration model

---

### State folder and snapshots

**What**: `.copilot/state/<engine_id>/<domain_id>/<instance_id>/` with outcome log, rename log, snapshots, triggers.

**Why deferred**: Git provides rollback (`git stash`, `git checkout -- file`). Snapshots add value only when autonomous changes can't be rolled back via git (e.g., MCP server running outside the git repo).

**Trigger condition**: When autonomous changes are enabled AND the engine runs outside git (portable deployment).

**Vision reference**: Execute layer, reversibility guarantee

## 🤖 Agent architecture consolidation

### Option D: Hybrid consolidated agents

**What**: Replace 24 type-specific agents with 3 consolidated agents (`pe-researcher`, `pe-builder`, `pe-validator`) in `.github/agents/00.01-prompt-engineering/`. Uses self-documenting naming: shorter name = broader scope (`pe-builder` handles all types; `pe-context-builder` handles one type). See [05.000-prompt-engineering-implementation.plan.md](05.000-prompt-engineering-implementation.plan.md).

**Why deferred**: Phase 0 (metadata + namespace) had to complete first. The side-by-side comparison (Step 3 of the implementation plan) requires both architectures to coexist.

**Trigger condition**: Phase 0 complete (✅). Ready to proceed with Step 0 (guidance audit) → Step 1 (build consolidated agents) → Step 3 (comparison).

**Current status**: Plan written, not yet started.

## 📦 Delivery vehicles

### MCP server packaging

**What**: Package the self-update engine as an MCP server exposing tools callable from chat, prompts, and agents.

**Why deferred**: The engine logic is currently embedded in agent/prompt markdown files, not in executable code. Extraction requires either: (1) reimplementing the logic in a runtime (Node.js, Python), or (2) wrapping the prompt/agent invocation pattern as MCP tools.

**Trigger condition**: When the PE system is validated in one repo and ready for portable deployment.

**Vision reference**: R-S7-portable-packaging, Phase 0 of the rollout

---

### VS Code extension

**What**: IDE commands for healthcheck, staleness check, metadata validation. UI indicators for artifact freshness.

**Why deferred**: Requires extension development (TypeScript), testing, packaging. High effort, low urgency while the prompt-based workflow works.

**Trigger condition**: When healthcheck/staleness workflows are stable enough to automate and the manual prompt invocation becomes a friction point.

---

## 📊 Progressive learning and convergence

### Convergence detection

**What**: Detect when successive self-update cycles produce increasing proposed changes without new triggers (divergence signal) or when the same artifact oscillates between states (oscillation signal). 

**Why deferred**: Requires sufficient volume of autonomous actions to produce meaningful signal. Currently all changes are manual.

**Trigger condition**: When >50 autonomous actions are logged in the outcome log, enabling statistical analysis.

**Vision reference**: Loop stability principle

---

### Autonomy threshold tuning

**What**: Adjust autonomy thresholds based on outcome data — loosen for high-success-rate change types, tighten for frequently-rejected types.

**Why deferred**: Same as convergence — requires outcome volume.

**Trigger condition**: When >100 logged outcomes with human accept/reject decisions.

**Vision reference**: R-G3-progressive-learning

---

## 🌐 Cross-domain scaling

### Beyond article-writing

**What**: Apply PE infrastructure to additional domains: `02.00-events`, `03.00-tech`, etc. Each domain gets context files, agents, prompts — either domain-specific triads or (if Option D wins) context files only.

**Why deferred**: Phase 8 of the improvement plan covers `01.00-article-writing` first. Additional domains are incremental once the pattern is validated.

**Trigger condition**: When Phase 8 (article-writing enablement) is complete and validated.

---

### Ecosystem adoption framework

**What**: Formalize the trust registry and adopt/incorporate/ignore decision framework for evaluating external PE artifacts (Microsoft official skills, community agent packs, etc.).

**Why deferred**: No external PE artifacts currently overlap with owned artifacts. The framework adds value when the ecosystem matures.

**Trigger condition**: When Microsoft or trusted third parties publish PE artifact packs that overlap with owned artifacts.

**Vision reference**: R-G2-trust-calibrated-adoption
