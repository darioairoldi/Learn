# Prompt engineering implementation plan

> **Goal**: Deliver a reliable, effective, and efficient PE artifact system that can create and maintain artifacts for any domain (starting with `00.00-prompt-engineering` itself and `01.00-article-writing`), aligned with the self-updating vision in [04.000-vision.v4.md](04.000-vision.v4.md).

---

## 🎯 What this plan covers

The [improvement plan](04.000-prompt-engineering-improvement.plan.md) identified gaps against the vision. This plan addresses the **prerequisite question**: what's the right artifacts architecture to implement before investing in metadata, N-1 separation, and self-update infrastructure?

The core decision is the **agent architecture** — the most impactful structural choice, because it determines:
- How many handoffs lose context in every workflow
- How much agent content is duplicated across files
- How many files need updating when PE rules change
- How effective the system is at cross-domain artifact generation

## 📊 Architecture analysis: evidence from the current system

### Quantified overlap across the 9 researcher agents

I compared `prompt-researcher`, `agent-researcher`, and `skill-researcher` line-by-line. The pattern holds across all 9:

| Section | Identical across all 9? | What differs |
|---|---|---|
| YAML frontmatter structure | ✅ Same shape | `description`, `handoffs.agent` target, `capabilities`, `goal` wording |
| Mode and tools | ✅ Identical | All are `plan` mode, all use `semantic_search, grep_search, read_file, file_search, list_dir, fetch_webpage` |
| Role definition paragraph | ⚠️ ~80% same | Substituted artifact type name; same sentence structures |
| Expertise list | ⚠️ ~70% same | 4 of 6 items are shared ("Pattern Discovery", "Best Practice Research", "Context Gathering", "Scope Definition"); 2 are type-specific |
| CRITICAL BOUNDARIES ✅ Always Do | ⚠️ ~85% same | Same shared `📖` references (02.04, 02.05); 1-2 type-specific rules added |
| CRITICAL BOUNDARIES ⚠️ Ask First | ✅ ~90% same | Nearly identical; "scope too broad" → "suggest narrowing" |
| CRITICAL BOUNDARIES 🚫 Never Do | ✅ ~95% same | "NEVER create files", "NEVER skip research", "NEVER make assumptions" |
| Phase 0: Handoff Validation | ✅ Identical | Same table, same logic |
| Phase 1: Requirements | ⚠️ ~75% same | Complexity table identical; challenge method differs ("use case" vs "role" vs "progressive disclosure") |
| Phase 2: Pattern Discovery | ⚠️ ~80% same | Same search approach; different target instruction file |

**Bottom line**: ~80% of each agent file is shared PE expertise. ~20% is artifact-type-specific dispatch (which instruction file to read, which template to apply, which type-specific rules to check).

### Same pattern for builders

`prompt-builder` vs `agent-builder`:
- Same tools: `read_file, grep_search, file_search, create_file, replace_string_in_file, get_errors`
- Same Phase 0 handoff validation (identical)
- Same breaking-change detection workflow
- Same `📖` references to shared patterns
- Differ only in: which instruction file to read, which template to apply, tool-count enforcement (agents have 3-7 limit)

### Validator already consolidated

`prompt-validator` currently validates **both prompts and agents** — its description says "prompt and agent file validation". This proves a single validator can handle multiple artifact types effectively.

### Handoff cost analysis

**Current architecture**: creating one artifact requires 3 handoffs minimum:
```
Orchestrator prompt → Researcher → Builder → Validator
                    (handoff 1)  (handoff 2) (handoff 3)
```

Each handoff with `send: true` accumulates ~6,000+ tokens. By Phase 3 (validator), the context window carries ~18,000+ tokens of accumulated history from prior phases, much of it no longer needed.

With `send: false`, context is lost and must be re-injected, causing either:
- Missing information → lower quality
- Manual re-injection → more tokens spent explaining what the previous agent already knew

**Consolidation benefit**: A single agent doing research → build → validate keeps all context in one window without handoff overhead. The 80% shared knowledge isn't duplicated — it's loaded once.

## 🏗️ Architecture options

### Option A: Current (28+4 = 32 agents)

9 artifact-type triads × 3 roles + 4 meta-agents.

| Dimension | Assessment |
|---|---|
| Specialization depth | High per-type, but ~80% is shared |
| Handoff cost | 3 handoffs per artifact creation |
| Maintenance cost | 27 files to update when shared PE rules change |
| Context efficiency | Low — shared knowledge duplicated 9× per role |
| Cross-domain scalability | Poor — adding a domain means creating 9 new agents |
| Debuggability | Moderate — each agent is simple, but the chain is complex |

### Option B: Consolidated (3+4 = 7 agents)

3 role-based agents (pe1-researcher, pe1-builder, pe1-validator) + 4 meta-agents.

| Dimension | Assessment |
|---|---|
| Specialization depth | Moderate — uses artifact-type dispatch within each agent |
| Handoff cost | Same 3 handoffs, but each agent is more capable |
| Maintenance cost | 3 files to update when shared PE rules change |
| Context efficiency | High — shared knowledge loaded once |
| Cross-domain scalability | Good — adding a domain is a context file addition, not 9 new agents |
| Debuggability | Moderate — agents are larger, but fewer moving parts |
| Risk | Agent files may grow too large (>200 lines) |

### Option C: Minimal (1+3 = 4 agents)

1 unified `pe-engineer` agent (handles research, build, validate as phases) + 3 meta-agents (merge meta-researcher + meta-designer).

| Dimension | Assessment |
|---|---|
| Specialization depth | Low per-phase, high PE expertise |
| Handoff cost | **Zero** — no handoffs within artifact workflows |
| Maintenance cost | 1 file for artifact work |
| Context efficiency | Highest — everything in one context window |
| Cross-domain scalability | Best — one agent, all domains |
| Debuggability | Harder — large file, phases harder to isolate |
| Risk | Agent file becomes a monolith; mode conflict (plan vs agent within phases) |

### Option D: Hybrid — role-based with domain overlays (recommended)

3 role-based agents + 4 meta-agents + domain-specific context (not agents).

```
pe1-researcher  ← loads domain context per request
pe1-builder     ← loads domain templates + instructions per request
pe1-validator   ← loads domain rules per request

meta-researcher → meta-designer → meta-optimizer → meta-validator
```

Each agent carries **full PE expertise** and **dispatches to artifact-type-specific rules** by loading the right instruction file and context dynamically. Adding a new domain (e.g., `02.00-events`) means creating context files and templates — not new agents.

| Dimension | Assessment |
|---|---|
| Specialization depth | High PE expertise; artifact-type specificity via dynamic context loading |
| Handoff cost | 3 handoffs, but each agent carries richer context |
| Maintenance cost | 3 artifact agents + 4 meta agents = 7 files |
| Context efficiency | High — shared knowledge loaded once per agent |
| Cross-domain scalability | **Best** — new domains = context files, not agents |
| Debuggability | Good — clear role separation, fewer files |
| Risk | Agent files are larger (~150-200 lines vs ~100 current); requires good artifact-type dispatch logic |

---

## ✅ Architecture decision: Option D (hybrid role-based with domain overlays)

**Rationale:**

1. **80% shared knowledge** across current agents makes per-type triads wasteful — consolidation eliminates 20 redundant agent files
2. **Handoff quality** improves because each agent carries more complete PE knowledge
3. **Cross-domain scaling** is the killer advantage — creating `01.00-article-writing` artifacts doesn't require 9 new agents; the same 3 agents load article-writing context/instructions/templates
4. **Maintenance scales linearly** — PE rule changes touch 3 agent files, not 27
5. **The validator precedent** proves it works — `prompt-validator` already validates both prompts and agents successfully

**Mode conflict resolution**: `pe1-researcher` uses `plan` mode (read-only). `pe1-builder` uses `agent` mode (read-write). `pe1-validator` uses `plan` mode (read-only). Clean separation maintained.

### Comparison strategy: folder separation

Instead of choosing upfront, **build both and compare with evidence**. The folder-separation approach makes this cheap and risk-free:

| Location | Content | Architecture | Prefix |
|---|---|---|---|
| `.github/agents/00.00-prompt-engineering/` | Current 28 agents (require `pe-` rename) | Option A: Full granular | `pe-` (REQUIRED) |
| `.github/agents/00.01-prompt-engineering/` | 3 new consolidated agents | Option D: Hybrid | `pe1-` |
| `.github/prompts/00.00-prompt-engineering/` | Current 23 prompts (require `pe-` rename) | Option A entry points | `pe-` (REQUIRED) |
| `.github/prompts/00.01-prompt-engineering/` | 3 new consolidated prompts | Option D entry points | `pe1-` |
| `.github/instructions/` | Shared — 9 files need `pe-` prefix rename | Both architectures | `pe-` (REQUIRED) |
| `.github/skills/` | 2 PE skills need `pe-` folder prefix rename | Both architectures | `pe-` on folder (REQUIRED) |
| `.github/templates/00.00-prompt-engineering/` | PE templates (already namespaced by subfolder) | Both architectures | Subfolder = namespace |
| `.copilot/context/00.00-prompt-engineering/` | Shared context (already namespaced by subfolder) | Both architectures | Subfolder = namespace |
| `.github/hooks/` | Already `pe-` prefixed | Both architectures | ✅ Already portable |

**Portability rule**: ALL PE artifacts MUST use a namespace prefix — filenames (`pe-` or `pe1-`) for flat folders, subfolders for hierarchical locations. No generic names without prefixes. See [improvement plan](04.000-prompt-engineering-improvement.plan.md) § Naming conventions for the full rename table.

**What stays shared (zero duplication)**:
- All context files — both agent sets reference the same PE knowledge (subfolder-namespaced)
- All `pe-` instruction files — `applyTo` globs match both `00.00` and `00.01` agent/prompt folders automatically
- PE templates in `00.00-prompt-engineering/` subfolder — already namespaced
- Hooks — already `pe-` prefixed

**Comparison protocol**:
1. Pick 3 test tasks of increasing complexity (hook, agent, prompt)
2. Run each through Option A (existing agents), then Option D (consolidated agents)
3. Measure per run: validation pass rate (effectiveness), context gaps (reliability), token consumption (efficiency), correct dispatch (accuracy)
4. Evidence-based decision: winner stays, loser folder gets deleted

**Risk is near zero**: new agents coexist alongside old ones with different names and folders. No existing files are modified.

## 📋 Implementation plan

### Step 0: Strengthen the guidance layer first

Before building new agents, ensure the PE guidance (context files + instruction files) they depend on is solid. **This is the highest-leverage work** — better guidance makes any agent architecture more effective.

#### 0.1: Audit and improve PE instruction files for artifact-type coverage

**Problem**: Instruction files exist for each artifact type (`pe-prompts.instructions.md`, `pe-agents.instructions.md`, `pe-context-files.instructions.md`, `pe-skills.instructions.md`, `pe-templates.instructions.md`, `pe-hooks.instructions.md`, `pe-prompt-snippets.instructions.md`) but their rule coverage and structure varies in quality.

**Actions**:
1. Read each instruction file and assess: Are all creation/update/validation rules present? Are severity codes consistent? Do `📖` references resolve?
2. For each file, ensure it covers:
   - Required YAML fields for the artifact type
   - Required sections/structure
   - Naming and location conventions
   - Validation checklist
   - What to reference from context files (not duplicate)
3. Add `goal:` and `scope:` metadata to every instruction file YAML

**Method**: Run `/instruction-file-review` on each of the 7 PE-relevant instruction files. Apply fixes via `/instruction-file-create-update`.

**Deliverable**: 7 instruction files with complete, consistent, severity-indexed rules

#### 0.2: Audit and improve core context files for artifact-type guidance

**Problem**: Context files provide excellent general PE principles but don't have a clear "per-artifact-type quick reference" that an agent can load to know all rules for creating a specific artifact type.

**Actions**:
1. Verify `01.03-file-type-decision-guide.md` has complete, actionable guidance for each artifact type (when to use, structure, naming)
2. Verify `01.07-critical-rules-priority-matrix.md` covers all artifact types with severity codes
3. If gaps exist in per-type guidance, add them to the existing context files (no new files needed if they fit within token budgets)

**Method**: Read the files, identify gaps, update via `/context-information-create-update`

**Deliverable**: Context files with complete per-artifact-type guidance

#### 0.3: Create artifact-type dispatch reference

**What**: Create a new context file `.copilot/context/00.00-prompt-engineering/01.08-artifact-type-dispatch.md` that serves as the consolidated agents' lookup table.

**Content**: For each artifact type (context, instruction, agent, prompt, skill, template, hook, snippet):
- Which instruction file to load
- Which template(s) to use
- Which type-specific validation rules apply
- Which context files are most relevant
- Output structure expectations

**Why**: This is the key enabler for consolidated agents — one file that tells `pe1-builder` "you're building an agent, here's what to load and check."

**Deliverable**: 1 context file ≤2,500 tokens

#### 0.4: Template deduplication

**Problem**: Templates exist at both `.github/templates/` root and `.github/templates/00.00-prompt-engineering/` with duplicates (e.g., `prompt-template.md` vs `prompt.template.md`).

**Actions**:
1. Inventory all templates in both locations
2. Identify duplicates by comparing content
3. Choose canonical location for each template (prefer the `00.00-prompt-engineering/` subfolder for PE-specific templates)
4. Update all references to point to canonical versions
5. Remove duplicates (with user approval)

**Deliverable**: Single canonical location for each template; no duplicates

#### 0.5: Namespace prefix rename for ALL PE artifacts

**Problem**: Current PE artifacts use generic names without a `pe-` prefix across agents, prompts, instructions, and skills. These names clash when the PE system is ported to another repository. The vision mandates artifact namespacing for portability.

**Actions**:
1. **Agents** (28 files): `prompt-researcher.agent.md` → `pe-prompt-researcher.agent.md`, etc.
2. **Prompts** (23 files): `prompt-design.prompt.md` → `pe-prompt-design.prompt.md`, etc.
3. **Instructions** (9 files): `pe-agents.instructions.md` → `pe-agents.instructions.md`, etc. (keep `article-writing.instructions.md` and `documentation.instructions.md` as-is — repo-specific)
4. **Skills** (2 folders): `pe-prompt-engineering-validation/` → `pe-prompt-engineering-validation/`, `pe-artifact-coherence-check/` → `pe-artifact-coherence-check/` (keep `article-review/` as-is — domain-specific)
5. Update all internal references: `handoffs.agent` targets, `agents:` arrays, `📖` links, YAML `name:` fields, skill name references in agent files
6. Update `05.01-artifact-dependency-map.md` and `05.03-pe-workflow-entry-points.md` with new filenames
7. Instruction file `applyTo` patterns need NO changes — globs like `.github/agents/**/*.agent.md` already match renamed files

**Method**: Terminal `mv` for file/folder renames. `grep_search` + `replace_string_in_file` for reference updates.

**Scope**: 28 agents + 23 prompts + 9 instructions + 2 skill folders = **62 renames** + reference updates

**Note**: This is a prerequisite for the comparison (Step 3) — both Option A (`pe-*`) and Option D (`pe1-*`) must be namespaced to be portable.

**Deliverable**: All PE artifacts in `00.00` have `pe-` prefix; no generic names remain

- Handoff to: `pe1-builder` (send: true)

```markdown
# PE Researcher
## Your Expertise (shared PE knowledge)
## Artifact-Type Dispatch (load per request)
## CRITICAL BOUNDARIES (shared)
## Process
  Phase 0: Handoff Validation (shared)
  Phase 0.5: Artifact Type Detection + Context Loading
  Phase 1: Requirements + Use Case Challenge (shared pattern, type-specific rules loaded)
  Phase 2: Pattern Discovery (shared pattern, type-specific search targets)
  Phase 3: Research Report (shared format)

**Estimated size**: ~150 lines (vs 9 × ~100 lines = 900 lines currently)

#### 1.2: Create `pe1-builder.agent.md`

**Location**: `.github/agents/00.01-prompt-engineering/pe1-builder.agent.md`

**Design principles**:
- `agent: agent` mode (read-write)
- Tools: `read_file, grep_search, file_search, create_file, replace_string_in_file, get_errors`

  Phase 0: Handoff Validation (shared)

#### 1.3: Create `pe1-validator.agent.md`

- Load `pe-artifact-coherence-check` skill for cross-artifact consistency

## Your Expertise (shared validation knowledge)
## Artifact-Type Dispatch (load per request)
## CRITICAL BOUNDARIES (shared validation rules)

  Phase 0: Determine Artifact Type + Load Rules

  Phase 3: Content Validation (boundaries, rules consistency, references)

```

**Estimated size**: ~140 lines

## ⚠️ Risks and mitigations

| Risk | Mitigation |
|---|---|
| Consolidated agents become too large (>200 lines) | Artifact-type dispatch to external context file keeps agent body lean. Type-specific rules stay in instruction files, not inline. |
| Loss of specialization depth for complex artifact types | The specialization depth was 80% shared anyway. The 20% that's type-specific moves to instruction files and the dispatch context file — still available, just not duplicated. |
| Consolidated prompts lose argument clarity | Each consolidated prompt uses `argument-hint` to guide type selection. The dispatch is deterministic: user specifies artifact type as first argument. |
| Transition period with two systems | Keep deprecated files for 30 days. Old agents still work during transition. New agents are additive, not destructive. |
| Meta-agents reference old agents | Update meta-agent handoff targets in Step 2.4 before deprecating old agents. |
