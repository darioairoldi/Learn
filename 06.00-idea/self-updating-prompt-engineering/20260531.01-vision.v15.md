## The problem 🎯

> **Scope note.** This vision is designed for prompt engineering systems built on GitHub Copilot's customization framework (instructions, agents, prompts, context files, skills, templates, hooks) in VS Code. The system is architected for portability: its core engine is repository-agnostic and installable via MCP servers, GHCP SDK components, or VS Code extensions. The principles and rationales transfer to any repository with PE artifacts.

Prompt engineering artifacts form a layered system where every layer depends on the ones below it. A context file defines a rule. An instruction file references that rule. An agent enforces it. A prompt orchestrates the agent. A template shapes the output.

This architecture works well when it's fresh. But it decays through three forces.

**Platform drift** is the primary cause. VS Code ships monthly. GitHub Copilot evolves weekly. New YAML fields appear, tool behaviors change, features move from preview to GA or get deprecated. A context file written for VS Code 1.100 may be actively wrong by 1.110.

**Internal drift** is the secondary cause. As you add artifacts, you introduce subtle inconsistencies: a rule tightened in one context file but not propagated to the agents that enforce it, a template referenced by a prompt that was renamed, a token budget defined in two places with different numbers. Beyond reference integrity, internal drift includes **guidance quality degradation**: rules that were clear when first written become ambiguous as the system grows, complete guidance that develops gaps as new artifact types are added, and priorities that become unclear as the rule set expands.

**Ecosystem evolution** is the third force. Microsoft ships official agent skills. OpenAI, Anthropic, Google, and Meta publish best practices and tooling. Community specialists produce high-quality, narrowly scoped artifacts. External work may overlap with, improve upon, or entirely replace something you've built. The system needs a way to evaluate what's available, decide whether to adopt or learn from it, and integrate either choice without breaking internal consistency.

All three forces share the same outcome: **the system silently degrades**. Silent degradation takes two forms that require different responses.

**Structural staleness** — dangling references, stale validation timestamps, broken chain alignment, mismatched metadata. Deterministic to detect. Fail-closed is the right response.

**Capability/logic staleness (Type B)** — the artifacts still parse, still reference-resolve, still chain-align, but *the logic they encode is obsolete*. A new model family is released and existing model-specific guidance is outdated. A feature moves from preview to GA and the "preview limitations" note remains. Type B staleness passes all integrity checks yet produces progressively worse outputs. This is the harder variant — and the primary motivating concern. Manual audits catch it eventually, but "eventually" is the failure mode.

**The cost challenge.** Detecting drift — especially Type B — requires comparing artifacts against external reality. For a system with dozens of artifacts across multiple types, full-depth assessment of every artifact on every review cycle is prohibitively expensive. The cost grows with the number of artifacts × the number of external change sources × the depth of the dependency chain. A sustainable self-update system must bound this cost without sacrificing detection quality — assessing deeply where it matters and lightly where it doesn't, guided by what's actually changed in the world.

**The usability and flexibility challenge.** Even with the right cost discipline, a self-update system is only useful if invoking it is **easy to learn**, **flexible enough to express every reasonable analysis-and-processing shape**, and **predictable in what it actually executes**. Three failure modes recur when those properties are absent:

1. **Cognitive overhead.** A wide invocation surface (many flags, family-specific defaults, preset aliases) forces users to remember which option applies to which command. The result is under-scoped or over-scoped invocations and frequent rework.
2. **Silent re-interpretation.** When a parameter-less invocation does not have an explicit contract, the orchestrator silently picks a scope on the user's behalf — typically the cheapest path. The user cannot tell whether the system did what was asked or quietly substituted a narrower job.
3. **No path from intent to execution.** Free-form requests like *"review my agents"* or *"run a structural pass on the last 7 days"* have no canonical resolution. Each run interprets them differently, so the same intent produces different outputs.

The cost of these failures is twofold: invocations that miss the user's real intent waste tokens on the wrong scope, and invocations whose resolved meaning is invisible to the caller erode trust in the system's outputs.

A sustainable self-update system must therefore satisfy three properties, in priority order:

1. **Predictability (highest priority).** The resolved invocation MUST be visible to the caller on every run — logged as the first line of every run report — so what was actually executed can never silently diverge from what the user asked for. Without predictability, neither the user nor the system can audit outcomes.
2. **Ease of use.** The invocation surface MUST be small enough that a user can recall it without reference, and natural-language input MUST be accepted as a first-class entry point that resolves to canonical options before execution. Ease of use is what determines whether the system is reachable at all.
3. **Flexibility.** Every reasonable analysis-and-processing shape — narrow per-file edits, broad ecosystem audits, time-bounded deltas, dimension-specific passes, command-family combinations — MUST be expressible in the same canonical vocabulary, with no shape requiring a special flag. Flexibility serves real workflows; it MUST NOT be paid for by inflating the surface.

These three properties are interdependent: predictability without ease of use produces a system nobody invokes; ease of use without flexibility produces a system that handles only the easy cases; flexibility without predictability produces a system whose outputs cannot be trusted. The strategy that satisfies all three is documented in § Default-full invocation contract (under The goal) and operationalized in § Command families and option model.

## The goal 💡

Build a self-update system that maintains prompt engineering artifacts at peak reliability, effectiveness, and efficiency — with the maximum degree of autonomy that assessed risk allows — by continuously detecting drift, evaluating improvement opportunities, and executing validated corrections.

> **Priority for implementations.** When subsections in this section conflict, resolve toward source-grounded staleness resolution within risk-calibrated autonomy bounds, and never sacrifice the design/review parity contract to optimize cost or execution depth.

### Priority: staleness resolution through source-grounded autonomy

Resolving staleness is not a secondary optimization. It is a top-priority objective for the PE system.

The system should:

1. Investigate authoritative sources autonomously as a default behavior.
2. Investigate user-provided sources autonomously when they are supplied.
3. Validate source trust, relevance, and claim consistency before integration.
4. Integrate validated findings autonomously when risk is below threshold.
5. Escalate to human approval for high-impact, low-confidence, or contradictory evidence.

This priority is how reliability, effectiveness, and efficiency are preserved over time under Type B staleness pressure.

### Two dimensions

**The self-update process itself** must be reliable (detecting the same drift consistently when run twice), effective (catching real drift and proposing correct fixes), and efficient (assessing deeply where changes warrant it and lightly where they don't — bounded by a progressive cost gradient from zero-cost cache hits through lightweight screening to full-depth review, never applying more assessment than the evidence requires).

**The artifacts the process maintains** must reliably, effectively, and efficiently serve their defined purpose — producing predictable outputs, achieving their stated objectives, consuming minimal tokens, attention, and effort.

### Key definitions

| Term | Definition |
|---|---|
| **Reliable** | Produces consistent, predictable outcomes across invocations |
| **Effective** | Achieves its stated purpose |
| **Efficient** | Achieves its purpose with minimal resource consumption |
| **Autonomous** | Operates without human intervention for decisions at or below a validated risk threshold |
| **Risk-calibrated** | Autonomy level is determined by assessed impact × confidence |

### Prerequisites

The self-update workflow requires two things that can't be substituted:

1. **A clear vision with explicit rationale.** If the system knows *what* it's trying to achieve and *why*, it has a reference point for evaluating any change — internal or external, platform-driven or ecosystem-driven. Without a well-defined vision, there's nothing to validate against.
2. **Access to the world.** If the system can observe what's happening — platform releases, ecosystem developments, its own internal state — it has the raw material to reason about. Without awareness of change, there's nothing to respond to.
3. **Modular, type-aware invocation.** The system can be invoked for specific artifact types — "review my agents", "update context files" — not just "review everything." Each invocation assesses the target type deeply while screening its dependencies lightly, escalating to deep assessment only when screening detects issues. This keeps the cost of any single invocation proportional to the work that actually needs doing, not the size of the entire artifact set.

### The autonomy gradient

The degree of autonomy isn't binary — it's a gradient calibrated to assessed risk:

| Autonomy level | Scope | Examples |
|---|---|---|
| **Autonomous** | Low impact, high confidence | Reference updates, stale timestamp fixes, formatting corrections |
| **Autonomous with notification** | Low-medium impact, validated | Redundancy removal after validation passes, adding new metadata fields |
| **Human approval required** | High impact or medium confidence | Restructuring artifacts, changing rules, adopting external artifacts |
| **Human-only** | Architectural or strategic | Vision changes, principle modifications, risk threshold adjustments |

**Low-risk autonomy rule:** Changes classified as non-breaking (goal, scope, and boundaries preserved) with high confidence (machine-verifiable or multi-pass validated) MUST proceed autonomously regardless of which command family triggered them. A command's default mode (`plan` vs `apply`) is a UX convenience for the user's workflow preference — it MUST NOT be interpreted as a permission ceiling that overrides risk classification. The only legitimate gate is assessed risk, never command identity.

### Change classification

The autonomy level for any change is determined by **assessed impact × confidence**. A **non-breaking** change preserves the artifact's declared goal, scope, and boundaries. A **breaking** change alters them. Impact is multi-dimensional: consistency, efficiency, behavioral compatibility, and effectiveness are each assessed independently — the highest-risk dimension governs.

Confidence increases when classification is machine-verifiable: metadata comparison (metadata-driven), structural separation labels (structural-separation), and historical success rates (progressive-learning). Without metadata and N-1 labels, all classification falls to LLM judgment — the most expensive and least confident path.

**How classification works in practice (short form).** Structural separation labels (Rule / Rationale / Example blocks per structural-separation) make most diffs deterministically classifiable: a diff touching only Example blocks is non-breaking; a diff touching Rule blocks gates on goal/scope/boundary preservation. Without those labels, every diff falls back to LLM judgment, collapsing the autonomy gradient to *trivial structural changes (autonomous)* vs. *everything else (human-required)*. Full block-label mechanics, the deterministic-vs-LLM decision table, and the multi-dimensional impact treatment: see The rationale § Change classification mechanics.

### Design and review parity contract

Design and review share one destination: the same quality objective and the same scope intent, filtered only by artifact applicability.

Process differences are role-specific, not quality-specific:

1. **Design** performs requirements synthesis and construction to reach the quality target.
2. **Review** performs analysis, verification, and improvement to keep artifacts at that same target.
3. **Governance and cost controls** can change execution depth or approval posture, but they MUST NOT lower quality ambition for either role.

### Guidance quality as prerequisite

The self-update system's ability to maintain artifacts "at peak reliability, effectiveness, and efficiency" is bounded by the quality of the guidance those artifacts are built from. Six measurable properties determine guidance quality:

| Property | Definition | Measurement |
|---|---|---|
| **Clarity** | A rule can be understood unambiguously by both humans and LLMs | LLM agreement test: two independent passes produce the same interpretation |
| **Non-redundancy** | Each rule is defined in exactly one canonical location | Automated: scan for duplicate rule text across context files |
| **Non-contradiction** | Rules across files do not conflict | Automated + LLM: coherence-check Workflow 2 |
| **Completeness** | No gaps exist where guidance is needed but absent | LLM-assisted: scan agent behaviors that lack backing rules |
| **Prioritization** | When rules could conflict, precedence is explicit | Automated: check critical-rules-priority-matrix coverage |
| **Actionability** | An agent can act on the rule without additional interpretation — AND can enforce its own metadata as runtime constraints | LLM test: can the rule be translated into a boolean check? Can the model verify "am I within scope?" |

Guidance quality is a **precondition for autonomy**, not a parallel concern. Before increasing autonomy phase (Phase 2→3→4), the guidance that the system enforces should meet minimum quality thresholds on all six properties. Autonomous application of ambiguous or contradictory guidance produces unreliable results regardless of how accurate the classification machinery is.

## The rationale (system design justifications) ⚙️

> **Priority for implementations.** When rationales pull in opposite directions, safety takes precedence over efficiency and reliability takes precedence over autonomy. The full conflict-resolution table below codifies this for each known pair; new rationale conflicts must follow the same precedence direction.

> **Terminology note.** This section describes **system-level design rationales** — the architectural reasons WHY the self-update system is built the way it is (e.g., **llm-reasoning**, **deterministic-processing**, **metadata-driven**). Every rationale is referenced by a readable kebab-case identifier so the reader can grasp what it stands for without cross-referencing. These system-level rationales are distinct from **artifact `rationales:` metadata** — the per-artifact YAML field that captures design decisions specific to each individual artifact. See the Metadata-driven principle for artifact-level rationales.

Maintaining a PE system against drift is familiar engineering work: read release notes, scan documentation, review external work, assess relevance, apply changes aligned with your vision. LLMs can perform every step of this process. The self-update workflow has nine steps — each within reach of current LLM capabilities, often augmented by deterministic tooling.

| Step | What happens | LLM role |
|---|---|---|
| **Search** | Gather change digest: monitor platforms, ecosystem, model updates, tooling, vision changes, outcome history, and internal modifications since last review | Browse documentation, release notes, changelogs; analyze outcome log; compare file hashes — producing a structured change digest that informs all subsequent assessment |
| **Analyze** | Screen dependencies against change digest; deep-assess target artifacts | Tier 1: deterministic structural checks (references, hashes, metadata). Tier 2: relevance screening against change digest (~800 tokens). Deep pass: full quality assessment for targets and escalated dependencies |
| **Reason (organizational)** | Assess whether the trigger implies structural changes to artifact organization | Evaluate role clarity, dependency structure, layer correctness, coverage gaps, redundancy |
| **Reason (per-artifact)** | Assess whether the trigger implies content changes to specific artifacts | Evaluate quality, adherence, drift for individual artifacts within the validated structure |
| **Design** | Propose what the implementation should look like | Restructure artifacts, rethink relationships |
| **Plan** | Sequence changes for minimal disruption | Break design into concrete steps, prioritize by effectiveness per effort |
| **Validate** | Confirm proposed changes don't regress existing behavior | Compare before and after, check coherence |
| **Implement** | Execute the plan | Apply changes through tool use |
| **Verify** | Confirm actual changes match the plan's intent | Run coherence checks, validate outcomes |
| **Iterate** | Cycle back if verification reveals gaps | Refine design, adjust plan, escalate |

### LLM capabilities

| ID | Rationale | Design implication |
|---|---|---|
| **llm-reasoning** | **LLM reasoning and judgment.** LLMs can analyze artifacts, compare implementations against specifications, identify inconsistencies, and make judgment calls about whether a change improves adherence to the vision. | Delegate analysis, comparison, and decision-making to LLMs. Reserve human oversight for decisions above the risk threshold. |
| **self-correction** | **Self-correction through multi-pass validation.** A single LLM pass can introduce errors. A second pass — by the same or different model — detects and corrects those errors before commit. | No change is committed without at least one independent validation pass. Critical changes get multiple passes. |
| **model-specialization** | **Model specialization.** Reasoning models excel at analysis and planning. Standard models excel at execution. Small models suffice for deterministic-like tasks. | Assign models by task type: research/analysis → reasoning models; implementation → standard models. See model-routing for the operational application. |
| **external-knowledge** | **External knowledge access.** LLMs can consume platform documentation, release notes, changelogs, and community repositories — giving the system awareness of changes without requiring a human to read and summarize them. | Monitor defined sources (VS Code release notes, Copilot documentation, trusted ecosystem projects). |

### Processing strategies

| ID | Rationale | Design implication |
|---|---|---|
| **deterministic-processing** | **Deterministic processing for predictable operations.** Parsing YAML, checking file existence, comparing timestamps, computing content hashes don't need LLM judgment. Deterministic tools are cheaper and more reliable. | Use deterministic tools for all predictable operations. Reserve LLM processing for tasks requiring reasoning or natural language understanding. |
| **decomposition** | **Decomposition for context window control.** LLM accuracy degrades as context grows (context rot — accuracy drops from 88% to 30% at 32K tokens). Self-update tasks spanning dozens of artifacts must be decomposed into focused units. | Self-update workflows decompose into phases. Each phase has a token budget. Handoffs carry only essential state. |
| **structured-contracts** | **Structured contracts between phases.** When LLM output feeds a downstream phase, structured formats (JSON, YAML, tables) bridge probabilistic LLM output with deterministic downstream processing. | Inter-phase communication uses defined schemas. LLM output is parsed deterministically into structured records. |
| **structural-separation** | **Structural separation for deterministic classification.** Labeled blocks (Rule/Rationale/Example) make breaking/non-breaking classification deterministic — without them, every diff requires LLM judgment. See *Rules, rationales, and examples are separable* principle. | Classification logic checks block labels first, content second. |
| **progressive-depth** | **Progressive validation depth for cost-bounded assessment.** Full-depth validation of all dependencies doesn't scale. Three phases (research → screening → deep) bound cost while preserving quality chain guarantees. See Assess § Assessment depth for the three-phase model and cost gradient. | Apply three phases: research (once) → screening (per dependency) → deep pass (targets + escalated). Maximum cascade depth: 2 levels. |
| **model-routing** | **Model routing for cost-optimal quality.** Different assessment steps have different computational requirements. Routing reasoning tasks to reasoning models, execution tasks to standard models, and predictable operations to deterministic tools optimizes both cost AND quality — each step gets the best-suited processing without over-provisioning. | Assign models by task type within each assessment step. Research/analysis → reasoning models. Implementation/formatting → standard models. YAML parsing, reference checking, hash comparison → deterministic tools. Model routing is a dimension of review quality (see Assess § Selective review dimensions) — adherence to routing recommendations is verifiable. |
| **capability-applicability** | **Capability-based option applicability.** Commands accept options based on implemented capabilities, not family membership. When an unsupported option is provided, the system MUST: (1) reject deterministically, (2) name the missing capability, (3) suggest the canonical command that provides it. This enables consistent UX across command families while allowing capability-gated extension. | Each command declares its supported option classes. The option applicability matrix is the single source of truth for which options each command accepts. Unsupported options produce deterministic corrective guidance, not silent ignoring or ambiguous errors. |
| **default-full-investigation** | **Manual invocations default to full investigation — across every `/pe-meta-*` command.** A parameter-less manual invocation of any `/pe-meta-*` command (review, update, create, design, audit, scheduled) is contracted as a full review across every in-scope artifact, every monitored source, deep per-artifact tier (organizational-pass-first). The contract is command-family-agnostic: what differs per command is mutation posture, not investigation breadth or depth. The cost of a missed Type B finding exceeds the cost of an over-investigation; therefore the system MUST NOT silently narrow scope on the user's behalf. Parameters are *subtractive* — users opt into narrower scopes; the system never opts in for them. Breadth (`full`/`incremental`/`bounded-delta`) is a resolved attribute, not an input flag: it is derived from `--start`/`--end` plus caller-type and logged on the first line of every run report. | The orchestrator MUST treat absence of slicers (`--scope`, `--source`, `--dim`, `--start`/`--end`) as "do everything," never as "do the minimum." Trigger-fired invocations resolve to incremental because the trigger itself is the delta signal; manual invocations resolve to full because the user did not signal a narrow scope. The resolved breadth and the resolved invocation string MUST be echoed back to the caller before Phase 1 runs and emitted as the first line of the Phase 8 report. |
| **minimal-consistent-option-surface** | **Minimal canonical parameter surface.** The orchestrator exposes exactly seven canonical parameters: `--mode`, `--scope`, `--source`, `--dim`, `--start`/`--end`, `--deps`, `--skip`. Every additional flag risks overlap with an existing parameter, inconsistent semantics across phases, or silent re-interpretation downstream. The long tail of natural-language scoping intent (subjects, concerns, consumer chains) is handled by the conversational pre-parser (Phase 0a), which resolves free-form input to canonical options before strict parsing. Phases 1–8 only ever consume canonical options. | New parameters MUST be justified against minimal-consistent-option-surface and MUST NOT replicate semantics already covered. The value-shape `--scope` parser (artifact-type token OR set of folders/files) absorbs `--area` and `--artifact`; consumer chains are expressed as `--scope <agent-or-prompt-file>` + `--deps full`; subject/concern keywords are resolved via Phase 0a to a comma-separated `--scope` enumeration. Deprecated-flag aliases MAY be accepted during a single migration window with a deterministic deprecation notice. |
| **domain-coherent-batching** | **Domain-coherent batching.** `default-full-investigation` + `minimal-consistent-option-surface` together can resolve a parameter-less invocation to a scope that spans multiple semantic domains. Domain heterogeneity inside one review undermines approval-gate granularity (reviewers see unrelated changelists), reviewer domain expertise (rarely covers all in-scope domains), and rollback scope (a single bundle discards work across unrelated domains). The orchestrator MUST surface the domain footprint and propose a per-domain split before Phase 1 runs whenever ≥ 2 distinct domain-ids appear. Semantic domain is a content property declared in artifact metadata (`domain:` YAML field), NOT a path property — a metadata-first 3-tier resolution algorithm (declared metadata → optional per-repo `pe-domain-map.yaml` heuristic → `unknown`) keeps the model repo-portable. **Seed-vs-deps carve-out:** when the seed is single-domain and `--deps` adds further domains, a single bundle is correct because the consumer artifact needs ALL its declared dependencies to be reviewable; this case is emitted as `bundle=cross-domain-deps` with per-dependency-domain specialized analysis lenses (not split). | Phase 0b (Domain coherence check) runs after Phase 0a and before Phase 1; it is **not skippable**. Multi-domain seeds produce a hard-gated split proposal in `--mode apply` and an advisory proposal in `--mode plan`. Cross-domain-deps invocations run as one review with per-dep-domain audit lenses in Phase 2–4 and per-dep-domain report sectioning in Phase 8. The resolved `bundle=` marker (`single-domain` | `multi-domain-gated` | `accepted-bundle` | `split-N` | `cross-domain-deps`) and per-file `domain-source=` markers (`declared-metadata` | `path-heuristic` | `unknown`) are emitted on the first line of every run report (criterion #12). See § Domain detection — metadata-first resolution. |

### Change classification mechanics

The core impact dimension is whether a change is breaking or non-breaking. A **non-breaking** change preserves the artifact's declared goal, scope, and boundaries while improving its reliability, effectiveness, or efficiency. A **breaking** change alters what the artifact is for, what it covers, or what it constrains.

| Change category | Goal/scope/boundary impact | Impact level |
|---|---|---|
| Content fix, better wording, example update | Preserved | Low |
| Deeper coverage within declared scope | Preserved | Low-medium |
| Scope extension (new topics) | Scope expands | Medium-high |
| Scope narrowing (removes coverage) | Scope contracts | High |
| Goal or boundary change | Goal or boundaries changed | High |

**Impact is multi-dimensional.** Goal/scope/boundary preservation describes the structural dimension only. A change can be non-breaking structurally while having impact on other dimensions:

- **Consistency** — does it contradict or reprioritize existing rules?
- **Efficiency** — does it increase token consumption beyond declared budgets?
- **Behavioral compatibility** — does it break consumers that parse the artifact's structure (YAML fields, section headings, metadata format)?
- **Effectiveness** — does it degrade output quality? (Only measurable through actual use — see effectiveness hard limit in Risks.)

Each dimension is assessed independently. The autonomy level for a change is determined by the *highest-risk* dimension. When assessing impact, consider all applicable dimensions — but don't mandate a formal multi-dimensional evaluation for every change. The cost of assessment must be proportionate to the change's scope.

Behavioral compatibility has two cost layers. Schema-level checks (YAML structure, section headings, metadata field names) are deterministic and near-zero cost. Consumer-behavioral checks (does restructured content change LLM behavior?) require extra LLM passes and belong in Phase 2–3. Only the schema-level layer is applied now.

**Confidence** increases when the determination is machine-verifiable: deterministic comparison of goal/scope/boundary metadata (metadata-driven), multi-pass LLM validation (self-correction), deterministic structural checks (deterministic-processing), and historical success rate for the change type (progressive-learning). Without structured metadata declarations, "is this change non-breaking?" requires LLM judgment and is therefore lower-confidence.

**How classification works in practice.** The breaking/non-breaking distinction is only useful if it can be assessed cheaply and confidently. The mechanism that enables this is **structural separation (structural-separation)**: the N-1 convention where rule-bearing sections label their content as Rule, Rationale, or Example.

With N-1 in place, classification follows a deterministic path — see *Rules, rationales, and examples are separable* principle for the block-label mechanism. Without N-1, every diff requires LLM judgment — collapsing the autonomy gradient to a binary: trivial structural changes (autonomous) vs. everything else (human-required).

### Conflict resolution among rationales

When rationales pull in opposite directions, the meta-principle is: **safety takes precedence over efficiency; reliability takes precedence over autonomy.** Specific resolution rules for known conflict pairs:

| Conflict | Resolution |
|---|---|
| llm-reasoning vs deterministic-processing | deterministic-processing wins when both can perform the operation. LLM only when judgment is required. |
| validation-caching vs external-knowledge | external-knowledge wins when there's evidence of platform change. Re-validate even if cached. |
| self-correction vs decomposition | self-correction wins up to the budget, then escalate. Never skip validation to save tokens. |
| autonomy-gradient vs self-correction | self-correction is a precondition for autonomy-gradient. The gradient classifies only after validation has run. |
| trust-calibrated-adoption vs autonomy-gradient | autonomy-gradient governs the final decision. Trust reduces assessment cost, not the autonomy threshold. |
| model-routing vs deterministic-processing | deterministic-processing wins when the operation is fully deterministic. model-routing governs model selection only for operations that require LLM processing. |
| Process optimization vs design/review parity | Design/review parity wins on quality target and scope intent. Process optimization may reduce cost or change execution posture, but MUST NOT define a lower quality objective for either role. |
| guidance-quality vs autonomy-gradient | guidance-quality is a precondition for autonomy-gradient. The gradient classifies only after guidance quality is confirmed for the affected rules. |
| Terminology overload (former robustness alias) | "Robustness" is a reliability property in standard SE usage. The legacy v12 group named `--dim robustness` (which bundled the consumer-correctness cluster `D5-boundaries` / `D6-consistency` / `D16-adherence` / `D18-coverage`) was renamed to `--dim adherence` in v13 so the name matches what those dimensions actually check. The name "robustness" is freed; **the parser does not accept `--dim robustness` as a synonym for any group** (retired per Issue 20260525.03). Callers must pick explicitly between `--dim adherence` (consumer-correctness: `D5-boundaries`, `D6-consistency`, `D16-adherence`, `D18-coverage`) and `--dim reliability` (system-level process reliability: `D28-reproducibility` through `D35-portability-boundary`). Unknown values, including `robustness`, are rejected via CF-05 with the full canonical enumeration. |

Most rationale pairs are orthogonal and never compete. The conflict set is small and bounded.

## The vision 🏗️

> **Priority for implementations.** When the vision's behavioral promises trade off against each other, resolve toward *detect-and-propose with risk-calibrated execution* — never expand autonomous-execution scope to reduce detection or proposal cost.

A self-updating prompt engineering system **continuously monitors its own health**, **detects when artifacts may have drifted from reality**, **proposes corrections before the drift causes failures**, and **executes validated corrections autonomously when risk is low enough**.

It isn't a system that rewrites itself without oversight. It's a system that behaves like a responsible maintainer: it watches for change signals, assesses impact, runs targeted checks, and acts on findings — autonomously for low-risk changes, with human approval for high-risk ones.

The architecture separates **engine** from **integration**. The engine contains detection, assessment, proposal, validation, and execution logic. The integration is the thin per-repo layer that points the engine at local artifacts and optional runtime hooks. This separation keeps the logic portable while preserving repository-specific control.

The system operates **modularly and progressively**. It can be invoked for a specific artifact type — reviewing agents without full-depth analysis of every context file they depend on. Assessment proceeds in three phases: research gathers what's changed in the world (once per invocation), informed screening checks dependencies cheaply (deterministic + lightweight relevance), and deep pass applies full quality requirements only to targets and dependencies that screening flags. This progressive model keeps each invocation cost-bounded while preserving detection quality — the system never applies more assessment than the evidence requires.

The vision has four operational layers.

### Per-repo integration model

The integration footprint stays intentionally small.

| File/component | Purpose | Required? |
|---|---|---|
| `pe-self-update.config.json` | Declares artifact paths, autonomy thresholds, iteration budget, monitored sources, and state namespacing | Yes |
| Optional `.github/copilot-instructions.md` addition | Makes self-update capability visible in conversational workflows | No |
| Optional `.github/agents/pe-self-update.agent.md` | Provides `@mention` wrapper over engine tools | No |
| Optional namespaced state folder | Stores repo-local logs and snapshots at the configured `state.path` | No |

Default target is **1 required file + optional wrappers**. The engine can also keep state outside the repo when configured.

Required state config keys in `pe-self-update.config.json`:

- `state.namespace` — unique namespace for engine/domain instance
- `state.path` — resolved state directory path (repo-local or external)

Guardrail: no engine may use an unscoped shared state directory by default.

### Delivery vehicles

The same engine can be delivered through different interfaces.

| Vehicle | How it works | Best for |
|---|---|---|
| **MCP server** (primary) | Exposes engine capabilities as tools callable from chat, prompts, and agents | Composable automation and direct tool control |
| **GHCP SDK component** | Runs as a hosted service with managed execution | Always-on and centrally managed workflows |
| **VS Code extension** | Adds IDE commands, UI signals, and task integration | IDE-native experience |
| **Skill pack** | Encodes lighter-weight behavior in skill form | Quick adoption with minimal runtime setup |

The delivery vehicle changes the interface, not the core engine logic.

### Three operating levels

Assessment and proposal can run at three levels, in strict order for broad triggers:

**Organizational level** — The artifact set as a system is the reference point. Evaluates role clarity, dependency structure, layer correctness, and coverage/redundancy. Runs first on broad triggers because per-artifact improvements within a suboptimal organization are wasted effort. See Assess § Assessment ordering for the full checklist. For narrow triggers, skipped unless the change implies organizational impact.

**Architecture level** — The full vision is the reference point. The system evaluates whether a major external change (a new platform capability, a fundamentally better approach from a trusted source) warrants structural rethinking.

**Quality dimension level** — Individual measurable properties serve as the reference. These operate at two scales:

**Per-artifact quality**: non-contradiction (within the file), consistency (with declared metadata), reliability (produces predictable output), effectiveness (achieves stated purpose), efficiency (within token budget).

**System-level quality**: non-contradiction (across files), non-redundancy (each concept in one canonical location), completeness (no guidance gaps for active artifact behaviors), prioritization (conflict resolution is explicit), layer correctness (right content in right artifact type — testable rules in instructions, behavioral guidance in context files, type-specific logic in agent bodies), and token-chain efficiency (total loading cost for a consumer's guidance set stays within context window bounds).

Per-artifact quality is assessed on every validation. System-level quality is assessed during ecosystem audits (`/pe-meta-update --mode plan --skip research`) and scheduled reviews.

Some quality dimensions are independent (non-redundancy and non-contradiction can be assessed in isolation). Others are coupled (reliability and effectiveness are coupled with efficiency — a change improving one may degrade the other). Independent dimensions run in parallel; coupled dimensions run together.

### Detect

*Relies on: metadata-driven, external-knowledge, validation-caching*

The system knows when to look. Five trigger sources initiate scoped health checks:

| Signal | Source | What it indicates |
|---|---|---|
| **Platform event** | VS Code/Copilot release notes, changelog feeds | New YAML fields, new features, deprecations, GA transitions |
| **Model event** | Provider release notes, model catalogs | Capability shifts — re-evaluate model-specific guidance |
| **Ecosystem event** | Trusted sources (Microsoft, OpenAI, Anthropic, Google, Meta, recognized specialists) | Potential adoption or incorporation candidate |
| **Internal event** | File system watcher / git hook (existing `pe-file-modification-tracker`) | Scope the dependent check |
| **Scoped invocation** | User or automation requests review of a specific artifact type | "Review my agents", "Update context files", "Validate prompts" |
| **Scheduled** | Time-based interval | Catch anything that event triggers missed |

Trigger wiring is integration-dependent. Existing hooks (`pe-healthcheck`, `pe-staleness-check`, `pe-file-modification-tracker`) remain a valid deterministic host, but MCP/SDK/extension schedulers are equally valid trigger hosts in portable deployments. Validation caching (validation-caching) skips artifacts validated recently against the current platform version — but never meta-infrastructure components (see Staleness avoidance principle). Missing or broken trigger wiring is a CRITICAL finding.

**Manual vs trigger-fired invocations resolve to different breadth contracts** (see default-full-investigation):

- A **manual** invocation of any `/pe-meta-*` command (issued by a human with no scoping parameters) is contracted as **full**: every monitored source, every in-scope artifact, deep tier, organizational pass enabled. The user did not signal a narrow scope, so the system MUST NOT silently pick one. The contract applies uniformly across all command families; only mutation posture differs per command.
- A **trigger-fired** invocation (Scheduled / Internal event / Platform event / Model event / Ecosystem event) is contracted as **incremental**: the trigger itself is the delta signal, so research-phase output is a bounded change digest scoped to deltas since `last_review_timestamp` for that source.
- A **bounded-delta** invocation — any caller that passes `--start` / `--end` — is contracted to a window. The bound may be a date OR a source-version token resolved to a timestamp via the source's `version_scheme` (see [§ Processing-state model](#processing-state-model)). Inside the window, recorded `pass` coverage is OVERRIDDEN and re-assessed — this is the re-baseline / distrust-recovery path, used to re-run a window whose prior processing is no longer trusted even though it was already covered.
- The resolved breadth (`full` | `incremental` | `bounded-delta`) and the resolved canonical invocation string are echoed back to the caller before Phase 1 runs and emitted as the first line of the Phase 8 report (§ Success criteria, criterion #12).

**Phase 1 precondition (domain-coherent-batching).** Phase 1 (Research) MUST NOT begin until Phase 0b (Domain coherence check) has confirmed the resolved scope is single-domain OR the user has explicitly accepted a multi-domain bundle (`bundle=accept`) OR the seed is single-domain and the multi-domain spread comes from `--deps` traversal (`bundle=cross-domain-deps`). Domain footprint is computed deterministically from each in-scope file's declared `domain:` metadata via the 3-tier resolution algorithm; the per-file `domain-source=` provenance and the resolved `bundle=` marker are emitted on the first line of the Phase 8 report. See § Command families and option model → § Domain detection — metadata-first resolution.

### Assess

*Relies on: llm-reasoning, dependency-graph, tier-blast-radius, trust-calibrated-adoption, guidance-quality, quality-propagation*

The system knows what to check.

**Assessment ordering.** For broad triggers (platform releases, ecosystem changes, scheduled reviews), assessment follows a two-pass sequence:

**Organizational pass.** Before evaluating individual artifact quality, evaluate the artifact set as a system:
- **Role clarity**: Are artifact roles well-defined and non-overlapping? Does each artifact have a single, distinct responsibility?
- **Dependency structure**: Is the dependency graph clean? No circular dependencies, minimal coupling, clear hierarchical layers?
- **Layer correctness**: Is content in the right artifact type? Testable rules in instruction files, behavioral guidance in context files, type-specific logic in agent bodies?
- **Coverage and redundancy**: Are there gaps where guidance is needed but absent? Are there redundancies where the same concept is defined in multiple places?

Organizational findings are addressed before the per-artifact pass begins — optimizing individual artifacts within a suboptimal organization is wasted effort. Organizational changes are HIGH impact by definition (restructuring) and route through the standard autonomy gradient.

**Per-artifact pass.** With organizational structure validated or accepted, assess individual artifact quality following quality dependency chains (quality-propagation): start with the lowest-tier dependencies and work upward, so each artifact is assessed against already-validated dependencies. Per-artifact assessment uses the existing mechanisms: tier + dependency analysis for risk, N-1 for diff classification, metadata for scope preservation.

**Artifact-type-aware routing within the per-artifact pass.** Per-artifact assessment follows quality-propagation's bottom-up ordering, but different artifact types carry different quality requirements reflecting their role in the system:

| Artifact type | Role | Update-path quality requirements |
|---|---|---|
| **Context files** | Define guidance (rules, patterns, constraints) | Quality-by-construction: 6 properties as construction invariants (see Guidance quality by construction principle) |
| **Instruction files** | Auto-inject testable rules | Minimization check (instruction-minimization) + non-contradiction with context layer |
| **Agent bodies** | Consume guidance + implement logic | Adherence check against loaded guidance + behavioral correctness |
| **Prompts** | Orchestrate agents | Adherence check + workflow correctness |
| **Templates, snippets, hooks** | Shape output / trigger wiring | Structural correctness (schema, references) |

When a broad trigger fires (platform release, ecosystem change):
1. Assess guidance artifacts first — apply construction invariants if updates are needed
2. Then assess consumer artifacts — check adherence against the *already-validated* guidance
3. Each artifact type follows its own quality requirements — no cross-cutting classification needed

**Explicit phase sequence for system-wide assessment.** The bottom-up ordering above is operationalized as a strict phase sequence:

| Phase | Artifact type | Assessment | Why this order |
|---|---|---|---|
| **Phase A** | Context files | All applicable dimensions + construction invariants + peer cross-coherence | Foundation layer — sets the quality ceiling for everything above |
| **Phase B** | Instruction files | Structural + minimization (instruction-minimization) + non-contradiction with context layer | Must be validated against already-validated context files |
| **Phase C** | Agent bodies | All applicable dimensions + adherence (`D16-adherence`) against Phase A/B results | Consumes guidance — adherence only meaningful against validated guidance |
| **Phase D** | Prompts | All applicable dimensions + adherence (`D16-adherence`) against Phase A/B/C results | Orchestrates agents — depends on validated agents and guidance |
| **Phase E** | Templates, snippets, hooks | Structural dimensions (`D1-metadata` through `D3-token-budget`, `D14-craftsmanship`) | Leaf artifacts — minimal dependencies |
| **Phase F** | Skills | Structural + strategic dimensions | May reference context files and agents |

If Phase A finds issues in context files, those MUST be resolved (or accepted as known issues with documented rationale) before Phase C/D assess consumer adherence. Adherence against unvalidated guidance produces unreliable results.

When a narrow trigger fires (single artifact change):
- If the changed artifact is a guidance artifact → apply construction invariants to the change
- If the changed artifact is a consumer → check adherence against current guidance; don't trigger guidance quality investigation
- If the change alters a dependency relationship → re-assess the affected quality chain (quality-propagation)

This routing is a natural extension of quality-propagation's bottom-up ordering — guidance artifacts are the lowest tier in every quality chain, so they're assessed first. Consumer artifacts inherit a quality ceiling from their guidance, so they're assessed second with adherence checks instead of construction invariants.

**Assessment depth: research → informed screening → deep pass.** Assessment operates in three phases. The research phase runs once per invocation. Screening runs per-dependency using research output. Deep pass runs on targets and escalated dependencies.

**Research-phase output contract by invocation shape.** What the research phase produces is determined by the resolved breadth (per default-full-investigation), not by an explicit flag:

| Invocation shape | Trigger | Source coverage | Research output | Tier-2 screening |
|---|---|---|---|---|
| **Full** | Manual, no scoping params | All monitored sources | Current-state **snapshot** (~500 tokens) of each source | Runs against every artifact in `--scope` (default `all`) |
| **Incremental** | Trigger-fired (Scheduled / Internal / Platform / Model / Ecosystem) | Per-source deltas since `last_review_timestamp` | Bounded **change digest** (~500 tokens) of deltas only | Runs against every artifact whose source dependencies appear in the digest |
| **Bounded-delta** | Any caller with `--start` / `--end` (date or source-version) | Per-source changes inside the resolved window; recorded `pass` coverage is overridden inside the window (re-baseline) | Bounded **window digest** (~500 tokens) | Runs against every artifact whose source dependencies appear in the digest |

In all three shapes, downstream phases (Tier-2 screening, deep pass, organizational pass) consume the same digest/snapshot structure — the *content* of the digest changes, the *contract* does not.

**Research phase** (once per invocation) — Gathers what's changed since the last review: platform releases, ecosystem events, model updates, tooling/infrastructure changes, vision/governance changes, outcome history (progressive-learning), and internal modifications. Produces a structured **change digest** (~500 tokens) that all subsequent screening checks reference. Cost: one research operation per invocation, amortized across all screened artifacts. When no external changes are detected, the digest is empty and Tier 2 screening is skipped entirely.

`last_review_timestamp` records when the research phase last ran. It is updated in the state folder (`<state.path>/triggers/last_review_timestamp`) at the end of every successful research phase — not at invocation start, but after the digest is successfully produced. When absent or older than the configurable maximum staleness window (default: 90 days), research gathers changes from the lookback window instead.

**Informed screening** (per dependency) — Two tiers within the same pass:

*Tier 1 — Structural screening (deterministic)*: metadata currency, reference integrity, content hash stability, structural completeness (YAML fields, N-1 labels), and validation cache freshness (validation-caching). Catches internal changes. If Tier 1 passes and the change digest is empty → screening passes.

*Tier 2 — Relevance screening (context-aware, runs only when change digest is non-empty)*: compares the artifact's declared scope and authoritative sources against the change digest. If the artifact's scope intersects with a changed platform area, overlaps with a new ecosystem artifact, or references capabilities affected by a model update → escalate to deep pass. Cost: lightweight LLM relevance judgment (~800 tokens per artifact). Catches semantic staleness (Type B) — the primary motivating concern.

**Deep pass** (per target + escalated dependencies) — Full quality requirements from the artifact-type routing table: construction invariants for guidance artifacts, adherence checks for consumers, minimization for instructions. Informed by the change digest so the reviewer assesses against specific external changes rather than doing open-ended staleness detection.

**Escalation**: When screening fails on a dependency (Tier 1 structural failure or Tier 2 relevance match), that dependency receives a deep pass before the target's review proceeds. Escalation is scoped to the specific failing dependency. Maximum cascade depth: 2 levels. Second-level escalation is reported as a finding rather than automatically pursued.

**Cost gradient**:

| Condition | Action | Cost |
|---|---|---|
| Cache fresh, no external changes | Skip (validation-caching) | Zero |
| Cache stale, Tier 1 passes, no external changes | Pass at Tier 1 | Near-zero |
| Cache stale, Tier 1 passes, external changes exist | Tier 2 relevance screening | Low (~800 tokens) |
| Screening fails (Tier 1 or Tier 2) | Deep pass | Full |

The research phase extends the Detect layer's trigger sources into the Assess layer — triggers identify *that* something changed; research identifies *what* changed. Screening extends validation-caching from a binary skip/deep model to a four-level cost gradient. Tier 2 directly addresses Type B staleness — the vision's primary motivating concern — by checking whether the world has changed in ways that affect the artifact's scope, even when the artifact's content is unchanged.

#### Processing-state model

Incremental and bounded-delta breadth both compute a **work set** of processing units rather than advancing a single scalar watermark. This subsection defines the state stores, the staleness rule, the three watermark origins, and the value-shape resolution of `--start` / `--end`. The governing invariants are the `processing-state-is-multidimensional` and `coverage-completeness-guarantee` principles.

**Two state stores.** Processing state lives on two independent axes:

| Store | Location | Per-record fields | Role |
|---|---|---|---|
| **Source ledger** (input axis) | `<state.path>/triggers/<source-id>.json` | `source_id`, `version_scheme`, `last_seen_version`, `last_seen_timestamp`, `last_digest_hash` | Rebuildable cache of where each monitored source last stood |
| **Artifact coverage** (processing axis) | per-artifact metadata, one cell per applicable dimension | `source_versions: {<source-id>: <version\|timestamp>}`, `depth: research\|screening\|deep`, `status: pass\|fail\|partial\|never`, `last_run` | Durable single source of truth for what each PU was assessed against |

The ledger can be reconstructed from the sources themselves; the coverage record cannot, so the coverage record is the canonical store (`single-source-of-truth`) and the ledger is a derived index.

**Processing unit (PU).** A PU is one `(artifact × applicable-dimension)` pair. A PU is **stale** when any of:

- `status == never` (never covered — ALWAYS in the work set, per `coverage-completeness-guarantee`), OR
- `status != pass` (a previous pass did not clear it), OR
- a source dependency's ledger `last_seen_version` / `last_seen_timestamp` is newer than the PU's recorded `source_versions[dep]`.

**Three watermark origins.** All three breadths process PUs whose source input is newer than a watermark; they differ only in where the watermark comes from and whether recorded coverage is trusted:

| Breadth | Watermark | Recorded coverage | Use |
|---|---|---|---|
| **Full** (manual, no params) | Beginning of time | Ignored — every PU reprocessed | Re-baseline everything; the cost gradient (validation-caching + Tier-1 screening) keeps already-current PUs near-zero cost |
| **Incremental** (trigger-fired) | Each PU's recorded `source_versions` | Trusted — work set is stale PUs only | Routine delta processing |
| **Bounded-delta** (`--start` / `--end`) | Explicit `--start` bound | Overridden inside the window — every PU whose source input falls in `[--start, --end]` is reprocessed even when `status == pass` | Re-baseline a distrusted window without re-doing the whole corpus |

The bounded-delta override is the **distrust-recovery path**: when prior processing of a window is no longer trusted (e.g. an earlier run is believed unreliable), re-run it with `--start` at the suspect bound — for example `--start 1.099 --source <id>` — and the recorded `pass` cells inside the window are re-assessed rather than skipped.

**Value-shape `--start` / `--end` — date OR source-version.** `--start` / `--end` accept two value shapes, resolved deterministically (mirroring the two-shape `--scope` parser):

1. A **date** shape — ISO-8601 (`YYYY-MM-DD`), `now`, or a relative offset (`-7d`, `-90d`).
2. A **source-version** shape — a token matching the `version_scheme` of the scoped source (e.g. `1.099` for a `semver` source, a model-version string for a `model-version` source).

Phase 0a resolves a version-shaped bound to a publish timestamp via the source's `version_scheme`, so phases 1–8 only ever consume a resolved time window. `version_scheme` therefore plays a dual role: it is the ledger's version recorder AND the parser's version-token recognizer. Two guard rails apply:

- A version-shaped bound with a non-singleton `--source` is REJECTED — *"a version window requires a single `--source`."*
- A version-shaped bound against a source whose `version_scheme` is `none` is REJECTED — *"this source has no version scheme; use a date window."*

Resolving a version token to a timestamp adds **no new parameter**: `--start` / `--end` widen to a value-shape exactly as `--scope` already carries an artifact-type token OR a path-set. `minimal-canonical-surface` (P0) is preserved and breadth stays a derived attribute.

**Carve-outs.** The `validation-caching` meta-infrastructure carve-out still holds — meta components are never cached and are reprocessed in every breadth (see Staleness avoidance first). Per `active-dimensions-follow-evidence`, a PU whose evidence pipeline was removed by `--skip` is dropped and reported as *not assessed*, never marked current — so a narrowed pass cannot manufacture false coverage.

**Modular invocation.** The self-update pipeline can be invoked for a specific artifact type rather than the full artifact set. Scoped invocation determines which artifacts receive the deep pass and which receive screening:

| Invocation scope | Deep pass target | Screening dependencies | Cascade on failure |
|---|---|---|---|
| Context files | Context files | None (base layer) | N/A |
| Instruction files | Instruction files | Context files | → deep context review |
| Agents | Agent files | Context + instruction files | → deep instruction/context |
| Prompts | Prompt files | Agents + context + instructions | → deep agent (may cascade) |
| Skills | Skill files | Context + instructions | → deep instruction/context |
| Templates, hooks | Templates, hooks | None (leaf artifacts) | N/A |
| Full system | All types | N/A (all deep) | N/A |

Scoped invocation operates within the existing Assess pipeline — it's a trigger type that constrains which artifacts are targets and which are screened. The organizational pass is skipped for scoped invocations unless the invocation implies organizational impact.

**Cross-invocation caching**: Screening results are cached within the invocation. If context files are screened during instruction review and then needed again during agent review (in a cascade), the cached screening result is reused.

**Cascade depth limit**: Maximum 2 levels. If a screening-triggered deep pass itself triggers further screening escalation, the second-level failure is reported as a finding for human review rather than automatically pursued. This bounds the cost of any single invocation.

For narrow triggers (single file change, specific internal event), the organizational pass is skipped unless the change implies organizational impact (e.g., adding a new artifact type, splitting an existing artifact, creating a new dependency relationship).

Risk classification uses two inputs in order:

1. **Tier (tier-blast-radius) — first sort key.** O(1) lookup from the artifact's filename prefix. Tier 0 → always escalate. Tier 5 → presumptively autonomous. Tiers 1–4 → route by breaking/non-breaking classification (see The goal). Tier is a **heuristic proxy** for dependency depth — it approximates blast radius from naming convention rather than measuring actual topology. This is acceptable at Phase 1 (manual invocation) where humans validate every decision. At Phase 2+, tier should be cross-validated against the actual dependency graph — a tier mismatch (filename says Tier 3 but graph shows Tier 1 reachability) is a finding.
2. **Dependent analysis (dependency-graph) — secondary.** Within a tier, dependency characteristics determine blast radius scope. At Phase 1, raw dependent count (grep-based) provides a coarse signal — more connections mean wider blast radius. At Phase 2+, dependencies are classified by coupling type:

   | Coupling type | Weight | Detection | Example |
   |---|---|---|---|
   | **Behavioral** | High | Agent/prompt loads and enforces rules from this artifact | Agent's `📖` reference + rule citation in boundaries |
   | **Structural** | Medium | Consumer parses specific sections or fields | Template expects specific headings; `applyTo` matches |
   | **Informational** | Low | Consumer mentions for context but doesn't depend on content | "See X for details" in References section |
   | **Cosmetic** | None | Consumer mentions in comments or version history | "Based on findings from X" |

   Weighted coupling replaces raw count for blast radius estimation. A single behavioral dependent carries more risk than five informational mentions. Transitive dependencies (A→B→C) are discovered through graph traversal of the dependency map, with impact attenuating by depth (direct = full weight, one-hop = half, two-hop = quarter).

The `pe-prompt-engineering-validation` and `pe-artifact-coherence-check` skills are the validation pattern library. Chain alignment checks (chain-alignment) run here: every `enforces_rulesets` path resolves to an existing file; every specific `enforces_rules` ID is non-dangling (consulting the rename log). Category-based references (Level 1.5) are validated by checking that the referenced category ID exists in the target folder's structure index and maps to at least one existing file.

For artifacts with N-1 structural separation, diff classification is deterministic: parse labeled blocks, check which block type changed, route accordingly. For artifacts without N-1, classification falls back to LLM judgment — flagging a capability gap and recommending N-1 adoption.

For instruction files, assessment includes an instruction minimization check (instruction-minimization): every rule is classified as testable/mechanical (appropriate) or behavioral/strategic (violation — must be moved to context file or agent body). Minimization violations are HIGH severity findings because they create silent collision risk when the PE engine is installed in repositories with their own instruction files.

For ecosystem changes, assessment includes a trust-and-fit evaluation (trust-calibrated-adoption): does the external artifact come from a trustworthy source? Does it overlap with or improve upon an owned artifact? The answer determines the response strategy.

**Finding severity classification.** Each finding is classified by its impact on the artifact's declared purpose:

| Severity | Criteria | Examples |
|---|---|---|
| **CRITICAL** | Blocks artifact operation or self-update infrastructure integrity | Dangling reference in meta-infrastructure, broken hook wiring, missing required metadata, required category with no mapped files in structure index |
| **HIGH** | Breaks the artifact's declared contract (goal, scope, or boundaries) | Goal drift, scope violation, boundary breach, consumer-breaking schema change, instruction file contains behavioral/strategic rules (minimization violation), category-based reference resolves to non-existent category ID |
| **MEDIUM** | Affects a quality dimension without breaking the contract | Consistency issue, efficiency regression, coverage gap within declared scope |
| **LOW** | Cosmetic or optimization opportunity | Wording improvement, example update, token savings within budget |

Severity determines processing priority and autonomy routing. CRITICAL findings abort the cycle and escalate. HIGH findings require human approval. MEDIUM and LOW findings are routed through the standard autonomy gradient.

**Guidance adherence verification.** Beyond checking that references resolve (structural integrity), assessment includes spot-checking that artifacts **implement** the rules they reference. For each `📖` or category-based reference in an artifact, the Assess layer can verify:

1. **Rule presence**: Does the artifact's behavior or checklist include checks for each rule in the referenced guidance?
2. **Rule accuracy**: Does the artifact's implementation of the rule match the canonical definition?
3. **Rule completeness**: Does the artifact cover all rules in the referenced guidance, or only a subset?

Adherence verification is **expensive** (requires reading both the artifact and its referenced guidance) and is therefore scoped: full adherence checks run during scheduled reviews and ecosystem audits, not on every change. For individual changes, adherence is checked only for the specific guidance affected by the change.

**Example of adherence gap**: H12 defines "cross-reference integrity" as checking `📖` references, markdown links, and slash-command references. A validator that only checks `📖` references and markdown links has a reference to H12 (structural integrity ✅) but doesn't fully implement it (adherence gap ❌). This gap allowed 174 broken slash-command references to persist undetected.

**Quality chain assessment.** Assessment traces quality chains per the *Quality flows through dependencies* principle. The key operational detail: quality chain assessment uses dependency-graph's coupling types but inverts the direction — dependency-graph propagates *impact* upward; quality-propagation propagates *quality constraints* downward. The same dependency graph serves both analyses.

**Three review modes.** Assessment can operate in three modes, each answering a different question at a different scope and cost:

| Mode | Direction | Question it answers | Cost model |
|---|---|---|---|
| **Individual** | Artifact → self | "Is this artifact structurally and strategically correct?" | O(1) artifact + fixed context loading |
| **Dependency-aware** | Artifact → dependencies → back to artifact | "Does this artifact + its dependency chain produce reliable behavior?" | O(d) where d = dependency count |
| **Guidance-first** | Guidance → all consumers | "Do artifacts that load this guidance actually implement it?" | O(c) where c = consumer count |

No single mode catches everything. Each covers a distinct failure class:

| Scenario | Individual | Dependency-aware | Guidance-first |
|---|---|---|---|
| Agent has malformed YAML | ✅ | ✅ | ❌ |
| Agent loads context files that contradict each other | ❌ | ✅ | ❌ |
| Guidance defines 3 ref types but validator only checks 2 | ❌ | ✅ (reviewing validator) | ✅ (reviewing guidance's consumers) |
| Two context files define conflicting token budgets | ❌ | ✅ (if artifact loads both) | ✅ (reviewing either file) |
| 5 of 8 validators miss a rule from the priority matrix | ❌ | ❌ (only checks one) | ✅ (reviews all consumers) |
| Agent declares tools it never uses in process phases | ✅ | ✅ | ❌ |

**Why three modes, not one combined review?** Token cost. A dependency-aware review loads the target (~250 lines) + context files (3-5 × 200-500 lines) + instruction files (~150 lines). That's 2,000-4,000 tokens of source material. Adding guidance-first checks (enumerate all consumers of each context file) multiplies by the consumer count. This violates decomposition. Different triggers also imply different modes: individual review for "validate this file," dependency-aware for "assess this subsystem's coherence," guidance-first for "audit this rule's implementation."

Review modes extend the modular invocation model: modular invocation determines *which artifacts* are targets and screened; review modes determine *what kind of assessment* runs during the deep pass. The screening mechanism is unchanged — review modes affect the deep pass only.

**Connection to quality-propagation.** The three review modes are the operational realization of quality-propagation:
- Individual assessment verifies per-artifact quality
- Dependency-aware mode implements quality-propagation's bottom-up direction: assesses dependency quality first, then the target, ensuring the target's quality ceiling is known
- Guidance-first mode implements quality-propagation in reverse: traces quality from guidance to consumers, producing an adherence matrix showing which consumers implement which rules
- Together, the three modes cover quality-propagation's full quality propagation path — bottom-up quality constraints AND top-down adherence verification

**Selective review dimensions.** Within each mode, assessment can target specific quality dimensions per invocation rather than applying every check. Each dimension is a named, independently invocable quality concern:

| Dimension group | What it covers | Example dimensions |
|---|---|---|
| **Structural** | Per-artifact structural correctness | YAML validity, reference integrity, metadata completeness, token budget compliance |
| **Strategic** | Vision alignment and design quality | Goal alignment, scope coherence, boundary enforcement, rationale currency |
| **Adherence** | Guidance implementation verification | Rule presence in consumers, rule accuracy, rule completeness |
| **Quality** | Guidance quality properties (all 6 prerequisite properties) | Clarity, non-redundancy, non-contradiction, completeness, **prioritization**, actionability |
| **Efficiency** | Resource optimization | Reference efficiency, handoff efficiency, processing efficiency, model routing adherence |
| **System** | Cross-artifact system properties | Cross-dependency coherence, layer correctness, coverage/redundancy analysis |

**Type-scoped dimension groups.** Not all dimensions apply to all artifact types. To avoid wasting processing on non-applicable dimensions, type-scoped groups select only the dimensions relevant to the target type:

| Group | Applicable to | Dimensions | Use when |
|---|---|---|---|
| `--dim context-full` | Context files | `D1-metadata` through `D3-token-budget`, `D6-consistency` through `D15-vision-alignment`, `D17-cross-coherence`, `D19-artifact-structure`, `D22-context-optimization` (16 dims) | Full context file review — every applicable Tier-1 dimension |
| `--dim context-health` | Context files | `D6-consistency` through `D12-staleness`, `D22-context-optimization` (8 dims) | Quick quality + staleness + organization check |
| `--dim reliability` | Agents, prompts (subset on context, instructions, skills, templates, hooks, snippets per matrix below) | `D28-reproducibility` through `D35-portability-boundary` (8 dims) | System-reliability validation: reproducibility, regression protection, metadata guards, validation invariants, rollback readiness, boundary actionability, autonomy calibration, portability boundaries (goal-trio reliability pole) |

The `--dim context-full` group is the correct way to run a Tier-1 review of a context file (structural + strategic + quality + freshness + context-organization). Using `--dim full` (all 35) on a context file wastes processing on 16 non-applicable dimensions (`D4-tool-alignment`, `D5-boundaries`, `D16-adherence`, `D18-coverage`, `D20-token-chain`, `D21-deterministic-first`, `D23-reference-efficiency` through `D28-reproducibility`, `D31-multipass-validation-invariant` through `D34-autonomy-calibration`). For reliability-pole assessment of context files, invoke `--dim reliability` separately (covers `D29-regression-protection`, `D30-metadata-guard`, `D35-portability-boundary` on context files).

The matrix above enumerates all 35 dimensions in catalog v1.4.0 grouped into two conceptual tiers: **Tier 1 — quality/strategic** (`D1-metadata` through `D27-model-adherence`, the original review surface) and **Tier 2 — reliability** (`D28-reproducibility` through `D35-portability-boundary`, the goal-trio reliability pole). The `--dim reliability` group is the canonical invocation for Tier 2; the other 11 groups partition Tier 1 by concern.

Dimensions operate within the progressive depth model: lightweight dimensions (structural) cost near-zero; heavyweight dimensions (adherence, quality) cost proportionally more. The cost of any invocation is bounded by the dimensions selected, not the full dimension catalog.

**Cost gradient for dimensions:**
- Zero cost: cached results for unchanged artifacts (validation-caching)
- Near-zero: deterministic structural dimensions (YAML parsing, reference resolution, hash comparison)
- Low: single-artifact strategic dimensions (goal/scope check against metadata)
- Medium: dependency-aware dimensions (load and compare guidance chains)
- High: guidance-first adherence (enumerate consumers, check rule implementation)

Selective dimensions implement progressive-depth at the per-artifact level — progressive-depth controls assessment depth across the artifact set (research → screening → deep); selective dimensions control assessment breadth within the deep pass. Together they bound cost in two orthogonal dimensions.

**Model routing within dimensions.** Different dimensions have different computational requirements. Applying model-routing to dimensions:
- Structural dimensions → deterministic tools (YAML parsers, reference checkers, hash functions)
- Strategic dimensions → reasoning models (vision alignment requires judgment)
- Adherence dimensions → standard models for rule extraction, reasoning models for gap analysis
- Efficiency dimensions → deterministic tools for measurement, standard models for recommendations

Routing is a verifiable property: each dimension's processing should use the appropriate model tier. Misrouting (using a reasoning model for YAML parsing, or a deterministic tool for strategic analysis) is a finding.

**Cross-dependency coherence.** When an artifact loads multiple guidance files, assessment checks that the rules from those files don't contradict each other. Cross-dependency coherence extends adherence verification (which checks artifact → guidance) with guidance → guidance checks within the consumer's loaded set. This catches conflicts at review time rather than execution time.

**Cross-dependency coherence also applies to context files (peer review).** When the target artifact IS a context file, cross-coherence checks whether the rules in THIS context file contradict rules in peer context files that consumers load alongside it. The mechanism:
1. Identify which consumers load this context file (from `context_dependencies:`, `📖` references, category references)
2. For each consumer, identify what OTHER context files it loads
3. Check for contradictions between rules in THIS file and rules in those PEER files

This is the reverse direction of consumer cross-coherence (which asks "do MY loaded deps contradict each other?"). For context files, it asks "does MY content contradict content in files that get loaded alongside me?" This enables bottom-up detection of cross-context contradictions — detecting them from the guidance layer rather than waiting for a consumer review to reveal them.

**Guidance optimization findings.** Beyond individual artifact findings, assessment can produce system-level optimization recommendations:

| Finding type | What it detects | Example |
|---|---|---|
| **Rule fragmentation** | Same logical rule defined piecewise across multiple files | Tool count limit in three separate files — should be in one canonical location |
| **Priority gap** | Rules that could conflict but no explicit precedence in priority matrix | Two context files define boundary minimums with different thresholds — which takes precedence? |
| **Layer misplacement** | Content in the wrong artifact type | Behavioral guidance in an instruction file (minimization violation); testable rule in a context file |
| **Consumer burden** | Artifact loads more guidance than it needs | Agent references 6 context files but only uses rules from 2; the other 4 add token cost without behavioral impact |

Guidance optimization findings are MEDIUM severity — they don't break individual artifacts but degrade system-level efficiency and clarity. They are produced during dependency-aware and guidance-first reviews where the full guidance chain is visible, not during individual reviews.

### Propose

*Relies on: self-correction, structured-contracts, autonomy-gradient*

The system knows what to suggest. When assessment reveals drift, the system generates a scoped correction plan using structured contracts (structured-contracts). When assessment reveals a valuable ecosystem development, the system proposes:

- **Adopt.** When the external artifact has a well-defined scope, credible roadmap, and comes from a trustworthy source. The system proposes relying on it directly — replacing or deferring to the external artifact.
- **Incorporate.** When the external work contains valuable ideas but doesn't meet the adoption bar. The system proposes updating owned artifacts to absorb the useful parts.

Each proposed change carries an assessed risk level (autonomy-gradient), which determines whether it proceeds autonomously or escalates to a human.

**Decomposition-first.** When a trigger fires involving a large change, the first action is to produce a decomposition plan: the smallest sequence of individually-verifiable steps. Steps are ordered bottom-up by tier — first changes land in high-numbered tiers (lowest blast radius), then propagate downward only after validation. This converts "large change → human approval marathon" into "large change → many small autonomous steps + a few human escalations at natural decision points."

**Risk-ordered execution.** When decomposition produces steps with different risk profiles, execute in risk order: non-regressive steps first (autonomous), potentially-regressive steps next (escalated as needed), optimization steps last (separate cycle). A step's execution is never blocked by a higher-risk step it doesn't depend on. This prevents a single human-approval escalation from blocking the entire decomposed sequence.

### Execute

*Relies on: self-correction, deterministic-processing, decomposition, autonomy-gradient*

The system knows how to act. Implementation is decomposed into focused units (decomposition) to avoid context rot. Before committing any change, at least one independent validation pass runs (self-correction).

Autonomy is earned per step, not per change-set. Each step applies risk-reduction techniques:

| Technique | What it does |
|---|---|
| **Static validation** | Schema validity, reference resolution (including category ID resolution via structure indexes), boundary preservation, token budgets, goal/scope preservation, N-1 block-level classification (deterministic when labels present; LLM fallback when absent). Also includes named check types: **consistency check** (no contradictions introduced, no unintended priority shifts), **efficiency check** (token delta within budget), and **behavioral compatibility check** (schema-level: YAML structure, section headings, and metadata field names unchanged or backward-compatible — deterministic and near-zero cost) |
| **Runtime validation** | Execute in a sandbox; compare outputs to pre-change baseline on a test set |
| **Modular integration** | Add capability as opt-in/optional first; default unchanged until observation confirms value |
| **Redundant processing** | Two independent LLM passes (self-correction), or LLM + deterministic cross-check — agreement required before commit |
| **Reversibility guarantee** | Pre-change tier-scoped snapshot; rollback is one deterministic operation |

**Consistency refinement.** Priority shifts that serve the artifact's declared goal are non-breaking; shifts that oppose the goal or violate boundaries are breaking. This prevents the consistency check from over-blocking goal-aligned changes.

A step that passes enough risk-reduction techniques to drop below the autonomy threshold proceeds without human intervention. A step that doesn't escalates — only the individual step, not the whole change-set.

**Artifact-type requirements in Execute.** The risk-reduction techniques above apply to all changes. Additionally:

- **Guidance artifact changes**: Construction invariants from the Guidance quality by construction principle run as part of the change. When a rule is modified, all artifacts with behavioral coupling (dependency-graph) to that rule are flagged for re-assessment — their quality ceiling may have shifted.
- **Consumer artifact changes**: Adherence to loaded guidance is verified. If adherence gaps are found, they're fixed as part of the change — not deferred to a separate cycle.
- **N-1 prerequisite**: A guidance change to a rule-bearing section without N-1 labels is automatically classified as HIGH risk. N-1 adoption is recommended as a prerequisite for autonomous guidance changes.

**Rollback.** Before Execute: the system records the current group snapshot (version format: `YYYYMMDD.NN`, scoped to the affected tier). During Execute: individual artifacts receive SemVer bumps — patch = non-breaking fix; minor = additive; major = breaking. After Execute: the system validates the tier and creates a new snapshot. If regression is detected: the system reverts the tier slice automatically. Snapshot files live at `<state.path>/snapshots/<change_id>/`.

### Rollout phases

Autonomous execution is not a single switch — readiness is heterogeneous across artifact types. The rollout is phased.

| Phase | Autonomous execution enabled for | Precondition (new) | Why |
|---|---|---|---|
| **Phase 0 (new)** | Engine extraction and packaging: detect + assess in installable form | Engine/integration separation complete, configuration schema defined | Proves portability outside the Learning Hub before higher-autonomy execution |
| **Phase 1 (current)** | Templates, prompt snippets, hooks JSON | None — validators already exist | No rules-with-rationales to classify; purpose obvious from filename/schema |
| **Phase 2** | Skills content, agent tool/handoff lists | Outcome log (C-3) in place + guidance non-redundancy, non-contradiction, and prioritization verified for affected rules | Structured metadata only — diffs are trivially deterministic. Reversibility was the missing piece. |
| **Phase 3** | Agent bodies, context files, instruction files | Semantic metadata (metadata-driven) + rule/rationale/example separation (N-1) in place + all 6 guidance quality properties above threshold for affected rules | These encode rules in prose. N-1 structural separation (structural-separation) is the mechanism that enables deterministic diff classification — without it, every content diff requires LLM judgment and escalates by default. Structured metadata (metadata-driven) enables mechanical scope preservation. Together they unlock autonomous execution for rule-bearing artifacts. |
| **Phase 4** | Cross-tier refactors, rule renames, file moves | Chain alignment (chain-alignment) + rename log operational | Changes ripple across the artifact graph. Resilient references are load-bearing. |

Preconditions are cumulative — Phase 3 also requires Phase 2's outcome log. *Current phase (mutable status):* **Phase 0** — portability extraction. Phases 1–4 each have a named precondition; the vision doesn't claim autonomy it hasn't yet earned.

Together, the four layers create a **continuous quality signal** rather than periodic snapshots. You don't wait for a manual audit to discover that half your agents reference a deprecated tool behavior.

### Validation infrastructure

The vision is enforced by four `*.instructions.md` files that auto-load on relevant file edits. Together they make scope discipline, principle impact, and plan lifecycle mechanically auditable rather than reviewer-dependent.

| File | `applyTo` | Role |
|---|---|---|
| `.github/instructions/vision-frontmatter.instructions.md` | `06.00-idea/**/*vision*.md` | Defines the `principles:` block schema and P0/P1/P2 priority semantics declared in vision frontmatter |

## Design principles 📋

> **Priority for implementations.** When design principles conflict with each other, the two foundational principles below (staleness avoidance first; rules/rationales/examples are separable) win — every other principle operates within the boundary they establish.

Two principles are **foundational** — they enable the mechanisms that all other principles depend on:

1. **Staleness avoidance first** — ensures the system's own infrastructure is trustworthy before it acts on managed artifacts.
2. **Rules, rationales, and examples are separable** — enables deterministic classification that makes the autonomy gradient practical rather than theoretical.

The remaining principles operate within the foundation these two establish.

### Staleness avoidance first — automatic, event-driven, small-step

**Priority: P0**

*Relies on: external-knowledge, metadata-driven, autonomy-gradient, self-correction, decomposition, progressive-learning*

Stale managed artifacts degrade specific capabilities. Stale self-update infrastructure silently degrades every subsequent decision and the system's ability to detect its own degradation. The self-update infrastructure MUST be maintained at higher freshness than the artifacts it manages.

**Rules:**

1. **Meta-infrastructure has the shortest validation cadence.** Tier 0/governance artifacts, all meta-agents, all hooks, and all skills used by the update loop are validated on every self-update cycle. validation-caching does NOT apply to these components — never skip, never cache.
2. **Event-driven triggers.** Every defined external signal (model releases, platform releases, ecosystem publications, internal edits) wires to a scoped pipeline entry point. Missing trigger wiring is a CRITICAL finding.
3. **Decompose-first.** When a trigger fires, the first action is a decomposition plan: the smallest sequence of individually-verifiable steps. The plan itself is validated before execution begins.
4. **Earn autonomy per step.** Each step is classified using the best-available risk-reduction techniques. Steps below the threshold execute; steps above escalate — only the individual step, not the whole integration.
5. **Infrastructure integrity is a hard gate.** Before any managed-artifact work begins, run deterministic integrity checks against the self-update infrastructure. Any failure aborts the cycle and escalates.
6. **No silent fallback.** Suspect meta-state never proceeds under degraded operation. "Missed a review opportunity" beats "applied a wrong change based on stale self-knowledge."
7. **Dangling references in self-update infrastructure are CRITICAL** (vs. HIGH in managed artifacts). Self-update infrastructure refers to the hooks, meta-agents, outcome log, rename log, and snapshot files.

*Subsumes: L-2 (amplified risk for self-updates). Extends: B-3 (integrity pre-step as hard gate). Governs: validation-caching carve-out for meta components.*

### Rules, rationales, and examples are separable

**Priority: P1**

*Relies on: role-declaration, deterministic-processing*

Most context and instruction files mix three content types in the same section:

- **Rules** (REQUIRED) — enforceable statements (MUST/NEVER/ALWAYS). A diff here is breaking for consumers.
- **Rationales** (OPTIONAL, recommended unless the reason is trivially self-evident) — *why* the rule exists. A diff here is non-breaking.
- **Examples** (OPTIONAL — include only when the rule is non-obvious or misapplication is a known failure mode) — illustrative prose or code. A diff here is non-breaking unless the example is referenced as canonical.

Every rule-bearing section uses this structural convention:

```markdown
### [Rule name]

**Rule** (REQUIRED):
- MUST/NEVER/ALWAYS ...

**Rationale** (OPTIONAL):
- ...

**Example** (OPTIONAL):
- ...
```

Without this separation, every diff requires LLM judgment to classify (breaking vs. non-breaking). With it, classification is deterministic and cheap.

**Economy matters.** Optional blocks consume tokens on every consumer load. Every added rationale or example should pay for its token cost by reducing misapplication risk or eliminating LLM judgment on diffs.

*Applies to: context files, instruction files, agent bodies. Connects to: breaking/non-breaking classification in The goal; outcome log `classification` field.*

This separation is the structural counterpart of the metadata-driven principle — metadata makes *artifact-level* classification deterministic; N-1 makes *content-level* classification deterministic. Together, they enable the earned-autonomy-per-step model.

### Human governance, autonomous execution

**Priority: P1**

*Relies on: self-correction, autonomy-gradient*

Humans define the vision, the goals, and the governing rules. The system never changes its own purpose or principles without explicit human direction.

Implementation is a different matter. Once the vision and rules are established, the system executes improvements autonomously — provided it can validate that risk is low enough. Autonomy level follows the gradient defined in The goal (canonical table is there; this section cross-references it).

The human's role shifts to: defining and refining the vision; setting risk thresholds; reviewing escalations when the system can't confidently validate; and auditing outcomes periodically.

### Minimal disruption

**Priority: P2**

*Relies on: autonomy-gradient*

A self-updating system that constantly demands attention is worse than manual audits. The system should be quiet when everything is healthy and specific when something isn't. Notifications should be actionable, not informational.

### Scoped over exhaustive

**Priority: P2**

*Relies on: dependency-graph, tier-blast-radius, metadata-driven*

A full system audit is expensive. Most changes affect a small subset of artifacts. The system identifies the blast radius of a change — using tier (tier-blast-radius) as the first sort key and dependency count (dependency-graph) as secondary — and checks only what's affected.

### Evidence-based

**Priority: P1**

*Relies on: llm-reasoning, structured-contracts*

Every proposed change must cite what triggered it, what assessment found, and why the proposed correction is appropriate. "File X is outdated" isn't enough. "File X references Copilot Spaces as 'public preview' but the April 2026 release notes announce GA availability" is.

### Trust-based adoption

**Priority: P1**

*Relies on: trust-calibrated-adoption*

Not all external work deserves equal trust:

- **Trustworthy sources** (Microsoft, OpenAI, Anthropic, Google, Meta, recognized domain specialists) — adopt directly when scope and quality align.
- **Valuable but unvetted sources** — incorporate ideas into owned artifacts after evaluation and adaptation.
- **Unknown sources** — don't trigger adoption or incorporation proposals autonomously. When valuable ideas come from unvetted sources, extract the core concept into fully-owned content rather than creating a dependency.

Trust is earned through consistency: clear scope, maintained roadmap, demonstrated expertise.

### Graceful degradation

**Priority: P2**

*Relies on: validation-caching, deterministic-processing*

If the system can't determine whether an artifact is affected by a change, it flags uncertainty rather than guessing. If a trigger source becomes unavailable, the system falls back to time-based checks. The worst outcome should be "missed a review opportunity," never "applied an incorrect change."

For self-update infrastructure specifically: suspect meta-state fails closed — no degraded operation, direct escalation.

### Deterministic where possible

**Priority: P1**

*Relies on: deterministic-processing, structured-contracts*

When an operation has a predictable outcome — parsing YAML, checking file existence, comparing timestamps, counting dependencies, computing content hashes — use deterministic tools instead of LLM processing. This principle applies across all four vision layers.

### Portable by design

**Priority: P0**

*Relies on: deterministic-processing, metadata-driven, portable-packaging*

Separate **engine** from **integration**. The engine holds reusable logic; integration connects that logic to repository-local artifacts and optional execution hosts. The engine discovers repository conventions and avoids imposing Learning Hub-specific structure. Integration should remain thin, incremental, and optional in depth.

**Artifact namespacing (REQUIRED).** All PE artifacts (agents, prompts, skills) MUST use a namespace prefix (e.g., `pe-`) that distinguishes them from domain-specific artifacts. Without namespacing, generic names like `pe-prompt-researcher` or `pe-agent-builder` clash when the PE system is ported to a repository that has its own agents with those names. Namespacing is a prerequisite for portability — it ensures PE artifacts can coexist with any domain's artifacts without naming conflicts.

**Instruction minimization (REQUIRED for portability).** See *Instruction minimization* principle (instruction-minimization) — required to prevent silent collisions when PE instruction files auto-inject alongside a target repository's own instruction files.

**Category-based references (RECOMMENDED for cross-artifact references).** When artifacts reference specific context files, they should use functional category IDs defined in the folder's structure index rather than filenames. Category IDs are stable semantic identifiers that describe the capability needed (e.g., `validation-rules`, `token-optimization`) and map to specific files in the structure index. This decouples consumer artifacts from the physical file layout — renames, renumbering, and file splits only require updating the structure index, not every consumer.

### Discovery over prescription

**Priority: P2**

*Relies on: metadata-driven, deterministic-processing, portable-packaging*

Discover PE structure from what already exists. Parse file types, frontmatter fields, references, and conventions. When rich metadata exists, perform precise impact analysis. When metadata is incomplete, run in structural-only mode and report capability-level uncertainty explicitly, with concrete recommendations to improve metadata quality.

### Quality flows through dependencies

**Priority: P1**

*Relies on: dependency-graph, quality-propagation*

An artifact's achievable quality is bounded by the quality of the artifacts it depends on. A perfectly written prompt that loads ambiguous context will produce ambiguous output. A well-structured agent that enforces unclear rules will enforce them inconsistently.

Quality improvement is most effective bottom-up: fix the context file first, then the agent that loads it, then the prompt that invokes the agent. Top-down improvement hits a ceiling set by the weakest dependency.

When a dependency improves, reassess its dependents — they may now achieve higher quality without any changes of their own. When a dependency degrades, all dependents are potentially affected — regardless of their individual quality.

**Operational realization through review modes.** This principle is implemented through the three review modes in the Assess layer: individual assessment checks per-artifact quality, dependency-aware mode traces quality bottom-up through dependency chains, and guidance-first mode traces adherence top-down from guidance to consumers. Together the three modes cover the full quality propagation path that this principle requires.

### Guidance quality by construction

**Priority: P1**

*Relies on: guidance-quality, quality-propagation, self-correction*

Guidance artifacts (context files, and guidance-bearing sections of instruction files and agent bodies) set the quality ceiling for every artifact that depends on them. Their update path carries the 6 guidance quality properties as **construction constraints** — satisfied during creation/update, not only verified after the fact.

**Construction invariants** — the 6 guidance quality properties (see Goal § Guidance quality as prerequisite for definitions) enforced during creation/update: rewrite-before-commit (clarity, actionability), reference-canonical (non-redundancy), resolve-before-commit (non-contradiction), flag-gaps (completeness), declare-priority (prioritization).

**Construction invariant → dimension mapping.** Each construction invariant maps to a review dimension, enabling the create/update workflow to report which invariant was enforced and which dimension it satisfies:

| Construction invariant | Dimension | Verification method |
|---|---|---|
| rewrite-before-commit | `D9-clarity` + `D11-actionability` | LLM agreement test + boolean-check translation |
| reference-canonical | `D7-non-redundancy` | Grep for duplicate rule text across context files |
| resolve-before-commit | `D6-consistency` | Check against direct dependencies |
| resolve-peer-coherence | `D17-cross-coherence` (peer mode) | Check against peer context files loaded by same consumers |
| flag-gaps | `D10-completeness` | Scan for uncovered agent/prompt behaviors |
| declare-priority | `D8-prioritization` | Check priority matrix coverage |

The 4th invariant (resolve-peer-coherence) extends resolve-before-commit to include peer context files — checking that new or modified rules don't contradict rules in files that consumers load alongside this one.

**Assessment cadence** — the 6 properties are assessed at three tiers of cost:
- **On every guidance change** (cheap): non-redundancy scan, non-contradiction check against direct dependencies, actionability self-test. These are the construction invariants.
- **On scheduled reviews** (medium): clarity agreement test (dual-pass), completeness gap scan across the artifact set, prioritization matrix coverage.
- **At phase transitions** (full): all 6 properties at full depth, qualifying the artifact set for the next autonomy phase.

**Why construction, not just validation**: A clarity issue caught during construction costs one rewrite; the same issue caught during validation costs a finding + a fix cycle + a re-validation. For the system's own guidance (meta-infrastructure), construction constraints are especially critical — a quality issue in meta-guidance degrades every subsequent decision.

Consumer artifacts (prompts, agents that consume guidance) don't carry construction invariants. They carry **adherence requirements** — verified against already-validated guidance. This separation keeps guidance update paths focused and consumer update paths lightweight.

**Context file change routing.** Context files carry the lowest tier prefix (Tier 00) — the highest blast radius per tier-blast-radius. Combined with their position as the foundation layer (quality-propagation), context file changes require **deeper validation** than other artifact types. However, the autonomy level follows the same risk-calibrated gradient as all artifacts: non-breaking changes that pass validation proceed autonomously; breaking changes escalate. The foundation-layer position increases validation depth, not the autonomy threshold for validated non-breaking changes.

**Non-breaking classification for context files.** A context file change is non-breaking when it preserves the artifact's declared goal, scope, and boundaries AND the new content is consistent with existing guidance and priorities. The pre-change guard (§ Metadata-guarded changes) is the primary mechanism — if the guard passes, the change doesn't contradict the artifact's declared contract. Additional validation through construction invariants and peer cross-coherence (`D17-cross-coherence`) further reduces risk. When both pass, the breaking risk is LOW regardless of tier.

**Decomposition enables autonomy.** Large context file changes that would be HIGH risk as a single change can be decomposed into smaller independently-verifiable steps, each of which may classify as non-breaking. This is a direct application of the Decompose-first principle — the system should split changes to maximize the autonomous portion.

| Change type | Autonomy level | Pre-conditions for autonomous execution |
|---|---|---|
| Metadata fixes (version, last_updated) | Autonomous | Deterministic, no content change |
| Reference fixes (broken links, stale paths) | Autonomous | Deterministic, no guidance change |
| Wording improvements within scope that preserve the same guidance and priorities | Autonomous with notification | Pre-change guard passes + construction invariants pass + peer coherence (`D17-cross-coherence`) passes |
| Non-redundancy refactoring (consolidating duplicate rules to canonical location) | Autonomous with notification | Pre-change guard passes + no rule lost (`D7-non-redundancy` improves) + all consumers still have access to the rule |
| Adding new coverage within declared scope (e.g., new model patterns when scope covers model optimization) | Autonomous with notification | Pre-change guard passes + new content doesn't contradict existing rules (`D6-consistency`) + peer coherence passes |
| Rule modification that changes behavioral guidance | Human approval required | Breaking for consumers — even if metadata preserved, behavioral impact requires human judgment |
| Scope expansion (new topics not in scope.covers) | Human approval required | Changes what consumers can rely on |
| Scope contraction (removing coverage) | Human approval required | May break consumer adherence |
| File split, merge, or reorganization | Human approval required | Affects category mapping, consumer references |

**Validation-reduced escalation.** When a change initially classifies as potentially breaking, successful validation can reduce the autonomy threshold. If the pre-change guard passes, construction invariants pass, peer coherence passes, AND the change can be validated against consumer test scenarios (embedded test cases in agents/prompts that load this guidance), the breaking risk is reduced. A change that passes all four validation layers can be reclassified from "Human approval required" to "Autonomous with notification" — the validation evidence substitutes for human judgment on the breaking question.

After ANY context file change that modifies rules (not just metadata), the system SHOULD flag affected consumers for re-assessment. This is the operational realization of quality-propagation: when the foundation changes, the quality ceiling shifts for everything above. This consumer flagging is the foundation-layer safeguard — it operates regardless of whether the context file change itself was autonomous or human-approved.

### Instruction minimization

**Priority: P2**

*Relies on: portable-packaging, deterministic-processing*

Instruction files auto-inject via `applyTo` patterns with no precedence mechanism. When multiple PE systems coexist, overlapping patterns cause silent rule collisions. To prevent this:

**Rule:** Instruction files MUST contain only testable, mechanical rules — rules evaluable with a boolean pass/fail without LLM judgment. Behavioral, strategic, and interpretation-dependent rules MUST be placed in context files or agent bodies where consumers explicitly choose what to load.

**Rationale:** Testable rules either agree (identical checks), contradict on a measurable dimension (resolvable by choosing the stricter value), or are additive (non-overlapping checks). Behavioral rules are inherently irreconcilable when doubled — "warm tone" plus "professional tone" has no deterministic resolution.

**Example:** ✅ "YAML MUST have `goal:` field" (testable). ❌ "Write in a warm, conversational tone" (behavioral — belongs in context file).

### Metadata-driven

**Priority: P1** *(maps to `metadata-first-content-properties` in the principles list)*

*Relies on: metadata-driven, role-declaration*

The system's ability to scope its work, skip redundant checks, and evaluate artifact effectiveness depends on machine-readable metadata.

Metadata serves **two audiences**:

1. **Validation tooling** (pe-meta): deterministic change classification, drift detection, scope preservation — consumed during modification workflows (pre-change guard, post-change reconciliation).
2. **Executing model** (runtime): self-constraint enforcement — the model grounds its behavior on declared goal, scope, and boundaries during execution. This is consumed during execution workflows (the model running the prompt or agent).

Both audiences consume the same YAML fields. The difference is timing: validation happens before/after changes; runtime grounding happens during execution. Without runtime grounding, metadata is passive documentation that only pe-meta tooling reads — the executing model can silently drift outside its declared scope.

Every artifact should declare:

- **`goal:`** — single sentence: the one outcome the artifact exists to achieve
- **`scope: {covers: [...], excludes: [...]}`** — what topics the artifact covers and explicitly excludes
- **`boundaries:`** — constraints the artifact operates within (max tokens, single-source-of-truth for, etc.)
- **`rationales:`** — lightweight list of **artifact-specific** design constraints that explain why key decisions in THIS artifact must be preserved across updates. NOT the same as the vision's system-level rationales (e.g., **llm-reasoning**, **deterministic-processing**) — those justify the architecture; these justify individual artifact choices
- **`version:`** — SemVer for individual artifact versioning (patch = non-breaking fix, minor = additive, major = breaking)
- **`required_categories:`** (structure indexes only) — list of functional category IDs that MUST remain mapped to at least one existing file. Refactoring operations that would leave a required category unmapped are blocked by the pre-change guard. This extends chain-alignment to make category contracts enforceable.

Placement: top YAML for context/instruction files; `agent_metadata:` for agents; `template_metadata:` for templates.

`rationales:` is **artifact-level constraint metadata**: it preserves decision memory for THAT specific artifact, not the system architecture. Without it, an artifact can remain in-scope while silently losing important safeguards.

**How artifact `rationales:` are used:**

1. **Decision memory** — When someone modifies this artifact later, the rationales explain WHY specific choices were made. Without them, a reasonable-looking change can silently undo a deliberate decision.
2. **Pre-change guard input** — The metadata-guarded change protocol checks: "does this proposed change invalidate any declared rationale?" If yes, a replacement rationale is required — you can't delete a design constraint without explaining why it's no longer valid.
3. **Scope-preserving safety net** — A change that preserves `goal:` and `scope:` but violates a `rationale:` is routed above baseline autonomy. This catches harmful changes that pass structural checks but undo load-bearing decisions.

Phase gating for `rationales:`:

- Phase 1: recommended, optional
- Phase 2+: required for artifacts eligible for autonomous semantic edits
- Phase 3+: optional upgrade to structured rationale objects where scale justifies it

Routing rule: if a change preserves `goal/scope` but violates a declared rationale, route above baseline autonomy (notification or escalation). If a rationale is replaced, the same change must include replacement rationale text.

Metadata makes artifact-level classification deterministic; N-1 structural separation makes content-level classification deterministic. Together they form the two-layer mechanism the autonomy gradient depends on. See The rationale § Change classification mechanics for the three classification tiers and confidence model.

### Metadata-guarded changes

**Priority: P1**

*Relies on: metadata-driven, role-declaration, self-correction*

Every artifact change — whether autonomous or human-initiated — is validated against the artifact's declared metadata **before** and **after** the change.

**Pre-change guard (default: block).** Before applying a change, compare its effect against the artifact's `goal`, `scope`, `boundaries`, and `rationales`. If the change contradicts any of these declarations, it is **blocked by default**. The change may proceed only with explicit human override.

**Post-change reconciliation (mandatory).** After every change is applied, the artifact's metadata MUST be reviewed and updated to accurately reflect the new state. This includes:
- `goal:` — still accurate? If the change expanded or shifted purpose, update.
- `scope.covers:` — still complete? If the change added coverage, add the topic. If it removed coverage, flag as potential regression.
- `scope.excludes:` — still accurate? Exclusions may need updating.
- `boundaries:` — still enforced? New constraints may apply; old ones may be relaxed.
- `rationales:` — still valid? If the change invalidated a rationale, it MUST be replaced with the new rationale.
- `version:` — bump per SemVer (patch/minor/major based on breaking classification).

**Why this matters.** Without pre-change guards, drift accumulates silently — an artifact's content diverges from its declared purpose while the metadata still claims the old purpose. Without post-change reconciliation, metadata becomes stale — a secondary form of drift that degrades the system's ability to classify future changes. Together, these two guards create a **closed feedback loop** between content and metadata that prevents both content drift and metadata staleness.

**Refactoring protection.** During refactoring (restructuring, splitting, merging artifacts), pre-change guards detect two types of coverage regression:

1. **Scope regression** — if a refactored artifact covers less than its predecessor's declared scope, the gap is flagged.
2. **Category regression** — if a refactored artifact is listed in a structure index's functional categories, the pre-change guard verifies that every affected category will still map to at least one valid file after the change. Missing category coverage is flagged as a regression.

Both types prevent silent capability loss — scope regression protects individual artifact coverage; category regression protects the stable reference contracts that consumers depend on.

| Guard | When | Default action | Override |
|---|---|---|---|
| Pre-change: contradicts goal | Before apply | **BLOCK** | Explicit human approval |
| Pre-change: contradicts scope | Before apply | **BLOCK** | Explicit human approval |
| Pre-change: contradicts boundary | Before apply | **BLOCK** | Explicit human approval |
| Pre-change: contradicts rationale | Before apply | **ESCALATE** (notification or approval) | Replacement rationale required |
| Post-change: metadata reconciliation | After apply | **MANDATORY** — always runs | Cannot be skipped |

**Audit trail.** All autonomous changes produce a structured log entry at `<state.path>/outcome-log.jsonl`. Every entry records: change ID, artifact path, artifact tier, pre/post content hashes, pre/post SemVer versions, breaking/non-breaking classification, confidence score, autonomy level, trigger type and source, validations passed/failed, and outcome (updated after observation window). A companion rename log at `<state.path>/rename-log.jsonl` records every rule/file rename with `old → new` mapping for automated reference rewrite.

Human rejections, including the stated reason, feed back into the outcome log as negative signal for threshold tuning (progressive-learning).

### Runtime grounding

**Priority: P1**

*Relies on: metadata-driven, role-declaration, deterministic-processing*

YAML metadata is not passive documentation — it's an enforceable execution contract. While *Metadata-guarded changes* protects artifacts during modification, **runtime grounding** protects artifacts during execution.

**The problem it solves.** When a model executes a prompt or agent, it receives the full file (YAML + body). The model CAN read the YAML, but without explicit instruction to enforce those fields, it may drift outside its declared scope — silently processing requests it should reject, ignoring boundaries it should enforce. This is a distinct failure mode from modification drift: the artifact doesn't change, but the model's behavior diverges from the artifact's intent.

**The requirement.** Every artifact that is independently executed (agents, prompts) MUST include a grounding mechanism in its body that bridges declarative YAML metadata to imperative runtime behavior:

| Artifact type | Grounding mechanism | Notes |
|---|---|---|
| **Agent** | Three-tier boundaries (Always Do / Ask First / Never Do) MUST reflect all YAML `boundaries:` entries | Existing pattern — ensure alignment |
| **Prompt** | Workflow MUST include scope enforcement: verify request falls within `scope.covers` before proceeding; reject requests matching `scope.excludes` | Lightweight — one process step |
| **Context/Instruction/Skill/Template** | N/A — not independently executed | No grounding needed |

**Alignment rule.** Body-level enforcement content MUST be consistent with YAML metadata. A boundary that exists in YAML but has no corresponding enforcement in the body workflow is a **runtime grounding violation** — the metadata claims a constraint that the executing model has no instruction to enforce.

**This does NOT mean duplicating metadata in the body.** The body references metadata constraints operationally (e.g., "verify request falls within declared scope") — it doesn't restate the full scope list. YAML remains the single source of truth; the body is the enforcement mechanism.

**Infrastructure folder layout (default repo-local mode).** The self-update operational state uses conflict-safe namespacing:

```
.copilot/state/<engine_id>/<domain_id>/<instance_id>/
  outcome-log.jsonl        # Append-only change event log
  rename-log.jsonl         # Append-only rename map for chain alignment
  snapshots/<change_id>/   # Pre-change content snapshots for rollback
  triggers/                # Trigger definitions and last-fired timestamps
```

Minimum uniqueness requirement: `<engine_id>` + `<domain_id>`.

Backward compatibility: legacy `.copilot/self-update/` may be read during migration, but new writes target the namespaced `state.path` only.

Portable deployments can keep this state in engine-managed storage outside the repository and project only audit artifacts into the repo when needed.

### Iteration budget

**Priority: P2**

*Relies on: decomposition, autonomy-gradient, human-governance-autonomous-execution*

Each self-update cycle has a configurable maximum number of autonomous changes (default: 10). The budget exists to bound both cost and blast radius per cycle — a runaway loop cannot consume unbounded tokens or make unbounded changes before a human reviews the outcomes — and to preserve `human-governance-autonomous-execution` (P1) by guaranteeing a hand-off artifact when the budget is exhausted mid-run. The budget applies to autonomous changes only; human-approved changes do not count against it.

**Overflow contract.**

1. **Purpose.** The budget is an end-to-end guard across every `/pe-meta-*` command family that performs writes (per `command-family-agnostic`). It is enforced uniformly regardless of how the run was initiated (manual, trigger-fired, scheduled).
2. **Overflow trigger.** Overflow occurs when Phase 4 (Execution) has applied the maximum allowed autonomous changes AND at least one validated finding remains unapplied. Findings still under investigation (Phases 1–3) and findings rejected during validation do NOT count toward overflow.
3. **Spillover plan emission.** On overflow, the orchestrator MUST emit a **spillover plan file** at `<run-folder>/<NN>-<kebab-name>-spillover.plan.md` (same path-resolution algorithm as § Plan-mode output contract). The spillover plan conforms to `plan-execution.instructions.md` and inherits status `draft`. Emission is NOT skippable.
4. **Content.** The spillover plan MUST cover every remaining-but-unapplied validated finding with one goal-table row per finding carrying `scope tag`, `principle impact`, `downstream landing`, and `original-run=<plan-file-or-run-id>` linkage so the next cycle can resume without re-investigating.
5. **Cross-family applicability.** Every command family that supports `--mode apply` MUST honor the overflow contract; commands that do not write (assessment-only families) are exempt because they cannot trigger overflow.
6. **First-line marker.** The Phase 8 `Resolved invocation:` first-line log MUST emit `spillover=<absolute-or-workspace-relative-path>` when overflow occurred, or `spillover=none` otherwise. Linter rejects reports that omit the marker on overflow runs (criterion #12 marker extension).

### Loop stability

**Priority: P2** *(scope.covers entry; not lifted into the principles list)*

*Relies on: self-correction, autonomy-gradient, progressive-learning*

The self-update loop's goal is **convergence toward a fixed point**: a state where, absent new external triggers, the system proposes no further changes. Every cycle should bring the artifact set closer to this fixed point, not farther from it.

**Convergence rule.** If successive cycles produce increasing numbers of proposed changes without new triggers, the loop is diverging. Divergence is a signal to freeze autonomous execution and escalate for human review.

**Oscillation rule.** If the same artifact is changed and then reverted (or changed in alternating directions) across consecutive cycles, the loop is oscillating. Oscillation is a signal to freeze the affected artifact and escalate — the system lacks sufficient information to make a stable decision.

**Phase note.** The specific mechanisms for detecting convergence and oscillation (change-volume slopes, per-artifact history tracking, trajectory analysis) are Phase 2–3 concerns. At Phase 1 (manual invocation), these rules serve as design intent — they inform how the loop is built, even before automated detection exists.

### Execution depth preserves quality objective

**Priority: P2**

*Relies on: capability-applicability, autonomy-gradient*

Process optimization (pipeline skipping via `--skip`, dimension scoping via `--dim`, target narrowing via `--scope`) can change **execution depth** — how much processing occurs — but MUST NOT change the **quality objective** or **scope intent** of the invocation.

**Rule:** A `--skip` flag or a narrowed `--dim` selection reduces processing effort while targeting the same quality standard for the dimensions that remain in scope. If a process change would lower quality expectations or narrow scope beyond what the user specified, it must be expressed as explicit `--dim` or `--scope` options — never hidden inside a convenience abstraction.

**Example:** Invoking `/pe-meta-update --mode apply --dim optimize --skip research,structure,consistency` skips three pipeline stages (execution depth reduction) but still applies the full quality standard to all efficiency-relevant dimensions across the selected scope. It does NOT lower the quality bar — `--dim optimize` narrows WHICH dimensions are assessed; `--skip` narrows HOW MUCH processing runs. The two controls compose; neither weakens the quality target.

### Scope and depth compose orthogonally

**Priority: P2**

*Relies on: capability-applicability, dependency-graph*

The `--scope` and `--deps` options address two independent concerns:

- **`--scope`** = "what artifacts to target" — a filter that selects the initial set of review targets by artifact type or file path.
- **`--deps`** = "how deep to traverse" — a traversal depth control that determines whether dependencies of the target set are included in assessment.

These compose orthogonally: `--scope context --deps direct` means "target context files, and include their direct dependents in assessment." Neither option implies or overrides the other. Both default independently when omitted (scope defaults to all applicable types; deps defaults to the command's natural traversal depth).

**Composition rule — additive union, not intersection.** The final assessment set is `--scope selection ∪ --deps traversal`. Dependents pulled in by `--deps` are NOT constrained by the `--scope` type filter: `--scope context --deps direct` assesses every context file PLUS every artifact that directly depends on those context files, even when those dependents are instructions, prompts, agents, or other non-context artifact types. Each included artifact is then assessed with the dimensions applicable to its own type, per the dimension applicability matrix — so an instruction file dragged in by `--deps direct` is reviewed against instruction-applicable dimensions, not against context-applicable ones. The `--scope` filter governs the SEED set; the `--deps` traversal grows that set outward across the dependency graph regardless of type.

**Why orthogonality matters:** Without separation, scope and depth conflate into complex combined options (e.g., `--mode dep-aware-full` which mixed "assess dependencies" with "full traversal depth"). Orthogonal composition enables fine-grained control through combination rather than proliferation of compound modes.

### Active dimensions follow assessment capability

**Priority: P1**

*Relies on: capability-applicability, deterministic-processing*

The active assessment dimension set on any invocation is the intersection of three filters:

1. **Type-applicability** — dimensions valid for the artifact type, per the dimension applicability matrix in [Quality validation coverage](#quality-validation-coverage). Dimensions outside the matrix for a given type are skipped, not reported as N/A.
2. **`--dim` selection** — dimensions explicitly chosen by the caller (or all type-applicable dimensions when `--dim` is omitted).
3. **Evidence availability** — dimensions whose evidence inputs were actually gathered, i.e., not removed by `--skip` of the pipeline stage that produces them.

**Rule.** A dimension that loses its evidence input MUST be dropped from the active set with a logged note, never assessed against missing inputs and reported as low-confidence.

**Example.** `--skip research --dim freshness` removes the change digest and source-staleness probes, so `D12-staleness` and `D13-source-verification` drop out — they are NOT reported as failing; they are reported as not assessed because their evidence pipeline was skipped. The orchestrator either fails closed (when the requested `--dim` set and the available evidence don't intersect at all) or proceeds with the reduced set (when at least one requested dimension remains assessable), and always logs which dimensions were dropped and why.

**Why this matters.** Without this rule, future `--skip` combinations silently produce low-confidence findings against missing inputs — the report would show "freshness: low confidence" instead of "freshness: not assessed, evidence pipeline skipped." The former pollutes the assessment trail and erodes trust in the system's scoring; the latter is a clean operational note. This complements capability-applicability (which gates which OPTIONS a command accepts) by gating which DIMENSIONS an option combination can produce evidence for. The single existing fail-closed guard on `--skip research` + missing cached digest is one instance of this general rule.

### Processing state is multidimensional

**Priority: P1**

*Relies on: metadata-driven, deterministic-processing*

Incremental re-processing must know what has already been assessed and what is still stale. A single per-source scalar watermark — e.g. "context files last reviewed at platform `1.099`" — cannot answer that: after a partial run, some artifact-dimension pairs are current and others are not, and a scalar collapses the distinction away.

**Rule.** Processing state is recorded along two independent axes — an **input axis** (per-source ledger, advanced when a source publishes) and a **processing axis** (per-artifact, per-dimension coverage, advanced only when a unit is actually assessed) — and MUST NOT be collapsed to a single per-source scalar. The per-artifact coverage record is the durable single source of truth (`single-source-of-truth`); the ledger is a rebuildable cache. The operational schema, the staleness rule, and the three watermark origins are defined in [§ Processing-state model](#processing-state-model).

### Coverage completeness guarantee

**Priority: P1**

*Relies on: validation-caching, deterministic-processing*

A narrowed pass (incremental or bounded-delta) MUST never silently and permanently drop a unit from coverage.

**Rule.** A processing unit (PU = one `artifact × applicable-dimension` pair) whose `status == never` is ALWAYS included in the work set, regardless of breadth — a never-covered unit can never be "skipped as current." A PU is skipped only when it is provably current against the active watermark: `status == pass` AND its recorded `source_versions` is not older than the watermark. `validation-caching` supplies the skip mechanism; the staleness comparison is deterministic. This complements `active-dimensions-follow-evidence`: a PU whose evidence pipeline was removed by `--skip` is reported as *not assessed*, never marked current — so a narrowed pass cannot manufacture false coverage. The three watermark origins (full, incremental, bounded-delta) and the bounded-delta re-baseline override are defined in [§ Processing-state model](#processing-state-model).

## Command families and option model 🎛️

> **Priority for implementations.** When the command surface trades off against caller convenience, the **explicit option discipline** rule wins — a parameter MUST appear only when it has an effect, and parameter-less manual invocations resolve to *full* per the default-full contract. Slicing must be requested; it is never inferred.

*This section defines the canonical command families, their shared option model, and the normalization principles that govern option applicability across the PE-meta command set.*

**Seven canonical parameters** form the entire input surface of every `/pe-meta-*` command: `--mode`, `--scope`, `--source`, `--dim`, `--start` / `--end`, `--deps`, and `--skip`. The set is intentionally minimized (minimal-consistent-option-surface) so semantics stay consistent across phases and command families. Free-form scoping intent (subjects, concerns, consumer chains) is resolved into these seven parameters by the conversational pre-parser (Phase 0a) before strict parsing begins; phases 1–8 only ever consume these seven. Breadth (`full` | `incremental` | `bounded-delta`) is a *derived attribute*, not a flag — it is computed from `--start` / `--end` plus caller type and logged on the first line of every run report.

### Default-full invocation contract

A parameter-less manual invocation of **any `/pe-meta-*` command** is a **deliberate ask for a full review**, not a default-to-cheap. The contract applies uniformly across every command family — review, update, create, design, audit, scheduled — so no command-specific optimization can silently invert it.

**Rule.**

- A **manual command-line invocation of any `/pe-meta-*` command with no scoping parameters** is a full review for that command's mutation posture: every in-scope artifact, every monitored source, deep per-artifact tier, organizational-pass-first (Phase 1.5 runs). What changes per command is the *mutation envelope* (read-only review vs. write-permitted update vs. create-new), not the breadth or depth of investigation.
- A **trigger-fired invocation** (scheduled, git-hook, file-watcher) is an incremental review: deltas since `last_review_timestamp` per source, screened via change digest.
- **User-supplied parameters narrow the full default. They do not widen a minimal default.** Strategies are *subtractive*; the system never opts into a narrower scope on the user's behalf.
- **Breadth is a resolved attribute, not a flag.** `full` / `incremental` / `bounded-delta` is derived from `--start` / `--end` plus caller-type (interactive vs trigger-fired) and logged on the first line of every run report.

**Why.** The 2026-05-27 staleness review (issue 20260525.03) revealed that, in the absence of an explicit contract, manual invocations were silently re-interpreted as incremental — the cheapest path on the next-token gradient. The vision did not state what parameter-less meant, so the orchestrator chose for the user. Stating the contract closes that loophole. See default-full-investigation and minimal-consistent-option-surface.

**What “full” means operationally.**

| Attribute | Full (manual, no params) | Incremental (trigger-fired) | Bounded-delta (any caller with `--start`/`--end`) |
|---|---|---|---|
| Source coverage | All monitored sources | Per-source `last_review_timestamp` deltas | Window between `--start` and `--end` (date or source-version) |
| Artifact coverage | Every in-scope artifact in `--scope` (default `all`) | Same | Same |
| Per-artifact tier | Deep | Tier 2 relevance screening; deep on flagged matches only | Tier 2 against window digest; deep on flagged matches only |
| Organizational pass (Phase 1.5) | Runs | Skipped unless trigger implies organizational impact | Skipped unless `--scope` broader than single file |
| Research-phase output | Current-state snapshot | ~500-token change digest | ~500-token bounded-delta digest |

### Command families

The PE-meta system organizes commands into families based on their mutation posture and operational purpose. Families are the headline framing of the PE-meta surface: every command belongs to exactly one family, the family fixes its mutation posture (read-only assessment vs write-permitted apply vs create-new), and every option, gate, and mechanism described later in this section operates uniformly across families per `command-family-agnostic` (P0).

| Family | Mutation posture | Purpose | Example commands |
|---|---|---|---|
| **Creation** | Write (new artifact) | Research from scratch, create artifacts at best quality (may reuse existing as foundations) | `pe-meta-create-*` |
| **Review** | Assessment-first (write via `--mode apply`) | Understand existing artifacts, improve structure and quality to reach the same target as Creation | `pe-meta-review` |
| **Guidance-first** | Assessment-first (write via `--mode apply`) | Trace quality from guidance to consumers, improve adherence | `pe-meta-adherence` |
| **Scheduled** | Autonomous low-risk apply | Periodic detection + autonomous fix for low-risk findings | `pe-meta-scheduled-review` |
| **Update** | Write (modify existing) | Apply assessed improvements to existing artifacts; reconcile artifacts with external platform changes when scoped to an external `--source` | `pe-meta-update` |

**Create/Review parity principle:** Create and Review commands achieve the same quality goal. They differ in starting point and process:

- **Create** commands focus on research and investigation from scratch, producing artifacts at best quality (possibly reusing existing artifacts as foundations)
- **Review** commands focus on understanding existing artifacts and improving their structure and quality to reach the same target

The confirmation gate is **risk-proportional, not command-proportional**. A Review command that identifies a low-risk improvement applies it autonomously (or with lightweight notification), exactly as an Update command would. The command family determines the *default posture* (assess-first vs. write-first), not the *permission ceiling*.

### Option taxonomy

**Seven canonical parameters** form the entire input surface of the PE-meta command set. The set is intentionally minimized (minimal-consistent-option-surface) so semantics remain consistent across phases and command families. Free-form scoping intent (subjects, concerns, consumer chains) is resolved into these seven parameters by the conversational pre-parser (Phase 0a) before strict parsing begins; phases 1–8 only ever consume these seven.

| Parameter | Purpose | Value space | Default when omitted |
|---|---|---|---|
| **`--mode plan\|apply`** | Execution depth — assess only, or assess and implement | `plan` (findings + actionable plan file emitted at the canonical plan path; see § Plan-mode output contract), `apply` (findings + changes; spillover plan file emitted when the iteration budget is exhausted, see § Iteration budget) | `apply` for families where `--mode` is applicable. `--mode plan` explicitly requests assessment-only output AND emits an actionable plan file. |
| **`--scope <token\|paths>`** | Target scoping — *which* artifacts to assess or modify | **Two value shapes only:** (1) an **artifact-type token** (`context` \| `instructions` \| `agents` \| `prompts` \| `skills` \| `hooks` \| `snippets` \| `templates`); (2) a **set of target folders/files** (paths, comma-separated, workspace-relative). | `all` — every in-scope artifact across every artifact type. Resolves to *full* breadth when combined with parameter-less invocation. |
| **`--source <url\|feed-id>`** | Source scoping — *which* monitored sources to research | A URL, a feed identifier from the configured source list, or `all` | `all` — every monitored source in the configured feed list |
| **`--dim <group\|D#>`** | Dimension scoping — *which* quality dimensions to assess or enforce | `full`, `structural`, `quality`, `strategic`, `freshness`, `efficiency`, `adherence`, `reliability`, `context-full`, `context-health`, `model`, `optimize`, or specific dimension IDs from `D1-metadata` to `D35-portability-boundary` (bare `D1`…`D35` also accepted for backward compatibility) | `full` (all dimensions applicable to the target artifact type) |
| **`--start <date\|version>` / `--end <date\|version>`** | Time-window scoping — *when* to look | **Two value shapes:** (1) a **date** — ISO-8601 (`YYYY-MM-DD`), `now`, or a relative offset (`-7d`, `-90d`); (2) a **source-version token** resolved to a publish timestamp via the scoped source's `version_scheme` (e.g. `1.099`). Either bound may be omitted (open window). | Manual invocation: full snapshot, no window. Trigger-fired invocation: `last_review_timestamp` → `now`. Explicit bound → `bounded-delta` (re-baseline; see § Processing-state model). |
| **`--deps none\|direct\|full\|<N>`** | Dependency traversal depth | `none` (target only), `direct` (one hop), `full` (transitive), or numeric depth `<N>` | Command-specific natural depth |
| **`--skip <stage>[,...]`** | Pipeline-stage skipping — omit specific processing phases | See § Pipeline phases and `--skip` mapping for the canonical phase list and per-phase skip semantics | None skipped |

**Value-shape `--scope` parser — two shapes only.** The `--scope` parser distinguishes its two value shapes deterministically:

1. A bare token that exactly matches one of the eight artifact-type names (`context`, `instructions`, `agents`, `prompts`, `skills`, `hooks`, `snippets`, `templates`) is treated as an **artifact-type token**.
2. Anything else — a single path, a comma-separated list of paths, a glob pattern, a workspace-relative file reference — is treated as a **set of target folders/files**.

This collapses the v13 `--area` (folder scope) and `--artifact` (specific artifact) flags into a single parameter. Consumer-chain scoping (formerly `--consumer <agent-or-prompt>`) is expressed as `--scope <agent-or-prompt-file> --deps full`. Subject/concern keywords (formerly `--subject`, `--concern`) are resolved by Phase 0a into a comma-separated `--scope` enumeration.

**Value-shape `--start` / `--end` parser — date OR source-version.** `--start` / `--end` carry the same two-shape discipline as `--scope`: a bound is either an ISO-8601 date / `now` / relative offset, OR a source-version token resolved to a publish timestamp via the scoped source's `version_scheme`. Phase 0a performs the version→timestamp resolution so phases 1–8 consume only a resolved time window; breadth stays a derived attribute and no new parameter is introduced (`minimal-canonical-surface`). Two guard rails apply: a version-shaped bound REQUIRES a singleton `--source` (rejected otherwise — *"a version window requires a single `--source`"*), and a source whose `version_scheme` is `none` rejects version tokens (*"use a date window"*). The full state model and re-baseline semantics are specified in [§ Processing-state model](#processing-state-model).

**Option resolution rules.** The orchestrator MUST apply these eight rules in order before Phase 1 runs:

1. **Phase 0a first.** Free-form input is normalized into the seven canonical parameters by the conversational pre-parser. Phases 1–8 never see raw user text.
2. **Echo before execute.** The resolved canonical invocation string is echoed back to the caller before any work begins and is emitted as the first line of the Phase 8 report (criterion #12).
3. **Subtractive only.** Absent slicers (`--scope`, `--source`, `--dim`, `--start`/`--end`) mean *do everything*, never *do the minimum* (default-full-investigation).
4. **Breadth is derived, not declared.** `full` / `incremental` / `bounded-delta` is resolved from `--start` / `--end` + caller-type, not from any input flag.
5. **Capability gating.** Unsupported parameter × family combinations are rejected deterministically per capability-applicability — the parser names the missing capability and suggests the canonical command that provides it.
6. **Deprecated-flag acceptance window.** Deprecated v13 flags MAY be accepted during a single migration window, but every accepted deprecated flag MUST emit a one-line deprecation notice and a canonical-equivalent suggestion (see § Deprecated flags migration).
7. **Phase 0b after resolution.** Once parameters are resolved, the orchestrator MUST compute the domain footprint of the resolved scope (Phase 0b — Domain coherence check, see § Domain detection — metadata-first resolution) and apply domain-coherent-batching before Phase 1 begins. Multi-domain seeds produce a hard-gated split proposal in `--mode apply` and an advisory proposal in `--mode plan`. The carve-out applies when the seed is single-domain and `--deps` adds further domains: that case resolves to `bundle=cross-domain-deps` (one review, per-dep-domain specialized lenses) and does NOT require a split.
8. **Bundle marker required.** The resolved `bundle=` marker (`single-domain` | `multi-domain-gated` | `accepted-bundle` | `split-N` | `cross-domain-deps`), the per-invocation `scope-source=` (`token` | `path-set` | `positional` | `default-all`), and the per-file `domain-source=` provenance (`declared-metadata` | `path-heuristic` | `unknown`) MUST appear in the first-line `Resolved invocation:` block of the Phase 8 report. Linter rejects entries that omit any of these markers (criterion #12).

#### Option applicability matrix

Not every parameter applies to every command family. Applicability is governed by the capability-based applicability principle (capability-applicability):

| Parameter | Creation | Review | Guidance-first | Scheduled | Update |
|---|---|---|---|---|---|
| `--mode` | ❌ | ✅ | ✅ | ❌ | ✅ |
| `--scope` | ✅ | ✅ | ✅ | ✅ | ✅ |
| `--source` | ✅ | ✅ | ✅ | ✅ | ✅ |
| `--dim` | ❌ | ✅ | ✅ | ✅ | ✅ |
| `--start` / `--end` | ✅ | ✅ | ✅ | ✅ | ✅ |
| `--deps` | ❌ | ✅ | ✅ | ✅ | ✅ |
| `--skip` | ❌ | ✅ | ✅ | ✅ | ✅ |

**Key design decisions:**

- **`--mode` is NOT applicable to Scheduled**: Scheduled review uses `--deps` for traversal depth control and an internal rotation mechanism for mode cycling. See § Mode elimination from scheduled review.
- **`--mode` IS applicable to Update**: Update is write-permitted, but `--mode plan` lets the caller preview the proposed changes as an actionable plan file before committing. The default when omitted is `apply` (see Default-full semantics by parameter); `--mode plan` is the opt-in preview.
- **External platform reconciliation is an Update invocation**: To reconcile artifacts against an external platform change, invoke `/pe-meta-update --source <url>` (optionally with `--start`/`--end` to bound the source window). There is no separate release-diff family — the Update family with an external `--source` covers it.
- **`--start` / `--end` applies to every family**: The window may scope research (Update / Review / Scheduled), external source-diff selection (Update with an external `--source`), or seed-corpus selection (Creation, Guidance-first). A bound may be a date or a source-version token; a version-shaped bound REQUIRES a singleton `--source` (see § Option taxonomy → value-shape parser).

#### Default-full semantics by parameter

When a parameter is omitted from a manual invocation, the orchestrator resolves it to its **full** behavior — never to a silent minimum. This is the per-parameter restatement of `default-full-investigation`: every absent slicer means *do everything in scope for this parameter*. The table below names the resolved value the parser MUST substitute for each omitted parameter, and the effective behavior that resolution produces.

| Omitted parameter | Resolved value | Effective behavior under default-full |
|---|---|---|
| `--mode` | `apply` (where applicable) | Findings + changes. Assessment-only requires explicit `--mode plan`. Not applicable to Creation, Scheduled (see applicability matrix); applicable to Update with `apply` default. |
| `--scope` | `all` | Every in-scope artifact across every artifact-type root (the eight canonical roots). `scope-source=default-all` MUST appear in the Phase 8 resolved-invocation block. |
| `--source` | `all` | Every monitored source in the configured feed list. |
| `--dim` | `full` | Every dimension applicable to the target artifact type — narrowed only by capability gating (`Active dimensions follow assessment capability`). |
| `--start` / `--end` | Caller-derived | Manual caller: full snapshot, no window. Trigger-fired caller: `last_review_timestamp` → `now`. Explicit `--start`/`--end` (date or source-version) overrides to `bounded-delta` and re-baselines recorded coverage inside the window. |
| `--deps` | Command-specific natural depth | Review / Update / Scheduled / Guidance-first: `direct`. Creation: not applicable. |
| `--skip` | None | Every applicable phase runs end-to-end. Phases 0, 0a, 0b, 8 are never skippable regardless. |

**Why default-full, not default-minimal.** The orchestrator defaults to maximum breadth and lets the caller *subtract* via explicit slicers. The reverse default — narrow-by-default, opt-in to breadth — would silently miss findings whose cost (Type B false confidence) exceeds the cost of a wider sweep. The subtractive contract is enforced as rule #3 of *Option resolution rules* above and as success criterion #12 of the Phase 8 report.

Forward references: the per-phase resolution of `--mode plan` vs `--mode apply` is detailed in *Pipeline phases and `--skip` mapping* below; the principle anchors are `default-full-investigation`, `minimal-consistent-option-surface`, and `Active dimensions follow assessment capability` (Goal section).

### Plan-mode output contract

**Priority: P2**

*Relies on: command-family-agnostic, human-governance-autonomous-execution, default-full-invocation*

`--mode plan` is contracted as more than "findings only" — every plan-mode invocation MUST emit an **actionable plan file** on disk so the human reviewer has a single artifact to review, promote to `actionable`, and execute. The contract is uniform across every `/pe-meta-*` command family that supports `--mode` (per `command-family-agnostic`).

**Required clauses.**

1. **Plan file MUST be written.** Every `--mode plan` invocation writes one Markdown plan file conforming to `plan-execution.instructions.md` (clear goal, actionability-gate-ready items, park lot section, lifecycle status `draft`). Plan emission is NOT skippable in `--mode plan`; `--skip plan-emission` is REJECTED with CF-05.
2. **Canonical plan path.** `<run-folder>/<NN>-<kebab-name>.plan.md` where `<run-folder>` is the conversation's current working folder (or `.copilot/temp/pe-meta-state/plans/YYYYMMDD-HHMMSS/` when no obvious folder applies); `<NN>` is the next available two-digit prefix in that folder; `<kebab-name>` is derived from the resolved invocation (e.g. `pe-meta-update-context-freshness`).
3. **Coverage promise.** Plan content MUST cover every finding produced by Phases 1–4 with one goal-table row per finding carrying `scope tag`, `principle impact`, and `downstream landing` (per `vision-amendment.instructions.md`).
4. **First-line marker.** The `Resolved invocation:` first-line log MUST include `plan-file=<absolute-or-workspace-relative-path>`; the Phase 8 echo MUST include the same marker. Linter rejects `--mode plan` reports that omit the marker (criterion #12).

**Why on-disk, not chat-only.** A chat-only findings report disappears with the conversation. A plan file landed in the run folder is reviewable later, version-controlled, and promotable to `actionable` through the same gate that governs human-authored plans — closing the human-handoff loop that `human-governance-autonomous-execution` (P1) requires.

### Mechanisms that protect batch quality

The contracts above (default-full, command families, option taxonomy, plan-mode output) establish what every invocation MUST produce. The subsections below specify the **implementation mechanisms** that protect the quality of those outputs when a parameter-less manual invocation resolves to a multi-artifact or multi-domain batch — they are quality-protection guards, not primary concepts.

#### Domain-coherent batching

The `default-full-investigation` and `minimal-consistent-option-surface` rationales together produce an unintended consequence: a parameter-less manual `/pe-meta-update --mode apply --scope context` correctly resolves to full breadth + full type scope, but then would process semantically independent domains (e.g. `prompt-engineering`, `article-writing`, `learning-hub`) as one atomic bundle. Domain-coherent batching is the quality-protection mechanism that intercepts this case: **multi-domain scope is legitimate; silently treating it as one atomic batch is not** (`domain-coherent-batching`).

**Rule.**

- The orchestrator MUST compute the **domain footprint** of the resolved scope by reading the declared `domain:` metadata field from each in-scope file's YAML frontmatter (with optional per-repo path-slug heuristic fallback, and `unknown` as final fallback — see § Domain detection).
- When ≥ 2 distinct domain-ids appear in the footprint, the orchestrator MUST present a numbered split proposal showing the canonical per-domain invocations.
- In `--mode apply`, the split proposal is a **hard gate**: Phase 1 MUST NOT run until the user selects a split OR explicitly accepts the bundle with `bundle=accept`.
- In `--mode plan`, the split proposal is **advisory**: Phase 1 proceeds, but the proposal is emitted as a next-step list.
- The split proposal MUST be canonical commands — same seven-parameter surface — so the user can copy/paste any single line into a fresh session.

**Why splitting is the default recommendation.** Domain-coherent bundles produce: (1) smaller approval surfaces; (2) domain-specialist reviewability (reviewer expertise rarely covers all in-scope domains); (3) granular rollback (a single-bundle rollback discards work across unrelated domains); (4) non-cross-contaminated findings (rules from one domain do not silently leak into another's review).

**Domain is a content property, not a path property.** Path slugs like `00.09-pe-meta/` or `00.00-prompt-engineering/` denote *housekeeping/infrastructure* groupings — a prompt under `.github/prompts/00.09-pe-meta/` may target ANY semantic domain because pe-meta prompts are themselves customization tooling, not domain content. The 3-tier resolution algorithm reads `domain:` from each file's frontmatter (authoritative); a per-repo path-slug heuristic is an *optional* Tier 2 fallback; `unknown` is the final Tier 3 fallback.

**Repo-portability.** The vision specifies WHAT to read (the `domain:` field) and the priority order; it does NOT mandate any folder-naming convention. A repository that does not use `<NN>.<NN>-<slug>/` subfolders drives Phase 0b correctly as long as every PE artifact declares `domain:` in its frontmatter. See § Domain detection — metadata-first resolution for the full algorithm, the per-invocation-type scope-extraction matrix, the CF-05 artifact-type/path consistency check, and the seed-footprint vs dependency-footprint distinction.

**Seed-footprint vs dependency-footprint.** When the seed scope is single-domain and `--deps` traversal adds further domains, the orchestrator emits `bundle=cross-domain-deps` and runs **one** review against the union with per-dependency-domain specialized analysis lenses in Phase 2–4 — splitting in this case would produce incomplete reviews because a consumer artifact needs ALL its declared dependencies to be reviewable. See § Domain detection → § Seed footprint vs dependency footprint.

### Pipeline phases and `--skip` mapping

The PE-meta orchestrator runs an explicit **eight-phase pipeline**. The `--skip` parameter accepts one or more phase identifiers from the table below. Phase 0 (Inventory) and Phase 0a (Conversational pre-parser) are NEVER skippable — they are prerequisites for deterministic behavior and the safe execution of every subsequent phase.

| Phase | Identifier | Purpose | `--skip` semantics |
|---|---|---|---|
| **0** | `inventory` | Discover and enumerate every artifact in scope, build the dependency graph, hash inputs for caching | NOT skippable |
| **0a** | `pre-parser` | Normalize free-form user input into the seven canonical parameters; echo the resolved invocation string | NOT skippable |
| **0b** | `domain-coherence` | Compute per-file domain footprint via the metadata-first 3-tier resolution algorithm (declared `domain:` frontmatter → optional per-repo `pe-domain-map.yaml` heuristic → `unknown`); produce per-domain split proposal when ≥ 2 distinct domain-ids appear in the seed footprint; emit `bundle=cross-domain-deps` carve-out when seed=1 domain and `--deps` adds further domains | NOT skippable; `--skip domain-coherence` is REJECTED with CF-05 |
| **1** | `research` | Gather change digest / snapshot from monitored sources per `--source` and the resolved breadth | `--skip research` — use cached digest only; fail closed if no cached digest is fresh |
| **1.5** | `organizational` | Organizational pass: role clarity, dependency structure, layer correctness, coverage/redundancy (runs only when breadth=full AND `--scope` is broader than a single file) | `--skip organizational` — bypass the system-level pass; per-artifact pass still runs |
| **2** | `structure` | Per-artifact structural checks: YAML frontmatter, required sections, reference integrity | `--skip structure` — omit structural-dimension checks |
| **3** | `consistency` | Per-artifact non-contradiction, cross-coherence, peer review, adherence checks | `--skip consistency` — omit consistency-dimension checks |
| **4** | `content` | Content / freshness / effectiveness / strategic dimensions; deep tier on flagged artifacts | `--skip content` — omit content-dimension checks |
| **5** | `approval` | Confirmation gate (autonomous vs propose-and-wait) governed by risk classification | `--skip approval` — NOT permitted for mutation-class findings; ignored otherwise |
| **6** | `apply` | Apply approved changes to artifacts; checkpoint each change for rollback | `--skip apply` — equivalent to `--mode plan` for this phase |
| **7** | `regression` | Post-apply regression sweep (second-pass validation invariant) | `--skip regression` — NOT permitted when phase 6 ran; report-only otherwise |
| **8** | `report` | Emit the Phase 8 report: first line = resolved invocation; followed by findings, applied changes, residual escalations | NOT skippable |

**Seven per-phase skip semantic rules.** The orchestrator MUST enforce these rules deterministically when resolving `--skip`:

1. **Phase 0, 0a, 8 are never skippable.** Any `--skip` value targeting them is rejected with a deterministic error.
2. **`--skip research` requires a fresh cached digest.** When the cache is stale or absent, the orchestrator MUST fail closed and refuse to proceed — it never silently degrades to a stale review.
3. **`--skip organizational` is implied** when breadth resolves to `incremental` or when `--scope` resolves to a single file. Explicit `--skip organizational` on a `full` invocation is honored but emits a notice.
4. **`--skip approval` is forbidden for mutation-class findings.** Approval gating is the contract between the orchestrator and the user; bypassing it for mutations breaks the autonomy-gradient invariant (autonomy-gradient).
5. **`--skip apply` is equivalent to `--mode plan`** — the orchestrator accepts both and reports the equivalence in the echoed invocation string.
6. **`--skip regression` is forbidden when Phase 6 ran.** Multi-pass validation (validation-caching, tier-blast-radius) requires regression after every apply; the orchestrator rejects this combination with a deterministic error.
7. **`--skip domain-coherence` is forbidden (domain-coherent-batching).** Phase 0b is never skippable. `--skip domain-coherence` (or `--skip 0b`) MUST be rejected with CF-05; the only way past the domain-coherence gate is (a) a single-domain footprint (auto-pass), (b) explicit `bundle=accept` on the invocation, (c) accepting a proposed per-domain split, or (d) the `bundle=cross-domain-deps` carve-out when seed=1 domain and `--deps` adds further domains.

Reference: the same eight-phase pipeline appears in the use-cases catalog (`05-reliability/`) keyed off resolved-invocation signatures.

### Conversational pre-parser (Phase 0a)

**Purpose.** Free-form user input (`"check my agents"`, `"look at the prompts that read context for adherence"`, `"run a fast structural pass on the last week"`) is normalized into the seven canonical parameters before strict parsing begins. The pre-parser is an LLM-driven phase — the only phase where natural language is the input. Phases 1–8 only ever consume the resolved canonical invocation.

**Inputs and outputs.**

| Input | Output |
|---|---|
| Raw user input (CLI args, chat message, or trigger payload) | Resolved canonical invocation string `--mode … --scope … --source … --dim … [--start … --end …] --deps … --skip …` |
| Active workspace context (artifact inventory, configured sources) | Confidence score per resolved parameter |
| Caller-type (interactive human vs trigger-fired) | Resolved breadth (`full` \| `incremental` \| `bounded-delta`) per default-full-investigation |

**Six resolution rules.** The pre-parser MUST apply these rules in order:

1. **Default to full when ambiguous.** Per default-full-investigation, absent slicers resolve to *do everything*. The pre-parser MUST NOT silently narrow scope to save tokens.
2. **Echo, then execute.** The resolved canonical invocation MUST be presented to the caller before Phase 1 runs. For interactive callers this is an explicit echo; for trigger-fired callers it is logged as the first line of the Phase 8 report (criterion #12).
3. **Subject / concern / consumer keywords resolve to `--scope`.** Subject and concern keywords (e.g., `"adherence rules"`, `"naming conventions"`) resolve to a comma-separated `--scope` enumeration of concrete artifact paths. Consumer chains (e.g., `"prompts that read the context-structure-index"`) resolve to `--scope <consumer> --deps full`.
4. **Window phrases resolve to `--start` / `--end`.** Phrases like `"the last week"`, `"since the v1.107 release"`, `"between June 1 and now"` resolve to `--start` / `--end` values. Ambiguity raises a clarification prompt; the pre-parser never invents a window.
5. **Single confirmation round.** The pre-parser MAY ask exactly one clarification question per invocation when confidence is below threshold. After the answer (or a timeout), it commits to the canonical resolution and proceeds.
6. **No silent invention.** When a parameter cannot be resolved from input + context, the pre-parser MUST surface the gap rather than invent a value. The only auto-defaulted parameter is the seven defaults defined in the option taxonomy.

**Worked examples.**

| Free-form input | Resolved canonical invocation |
|---|---|
| (parameter-less `/pe-meta-update`) | `--mode apply --scope all --source all --dim full --deps direct --skip <none>` (breadth = full) |
| `"check my agents"` | `--mode apply --scope agents --source all --dim full --deps direct --skip <none>` (breadth = full) |
| `"run a structural pass on the last 7 days"` | `--mode plan --scope all --source all --dim structural --start -7d --end now --deps direct --skip <none>` (breadth = bounded-delta) |
| `"audit adherence of the agents that read pe-common.instructions.md"` | `--mode plan --scope pe-common.instructions.md --source all --dim adherence --deps full --skip <none>` (breadth = full) |
| `"reconcile against the latest VS Code release notes"` | `--mode apply --scope all --source vscode-release-notes --dim freshness --deps direct --skip <none>` (breadth = incremental) |

### Per-artifact prompt invocation matrix

**Purpose.** Once `--scope` resolves to one or more artifact types (or to specific artifacts whose types are inferred), the orchestrator dispatches each artifact through a per-type, per-mode prompt. The matrix is the single source of truth for `(--scope artifact-type, --dim) → pe-meta-{type}-{review\|create-update\|design}` dispatch.

**Prompt naming convention.**

- `pe-meta-{type}-review` — reviews an existing artifact (assess + propose changes; risk-proportional apply)
- `pe-meta-{type}-create-update` — creates a new artifact or makes an additive update to an existing one
- `pe-meta-{type}-design` — designs the structure/shape of a new artifact category or a major refactor

where `{type}` is one of the eight artifact-type tokens (`context`, `instructions`, `agents`, `prompts`, `skills`, `hooks`, `snippets`, `templates`).

**Per-artifact dispatch matrix.** Given a resolved `(--scope artifact-type, --dim)`, the orchestrator selects the prompt as follows:

| Artifact type | Default review prompt | Triggered when |

| `instructions` | `pe-meta-instructions-review` | `--dim` ⊆ `{full, structural, quality, adherence, freshness}` |
| `agents` | `pe-meta-agents-review` | `--dim` ⊆ `{full, structural, quality, adherence, efficiency, model, reliability}` |

| `skills` | `pe-meta-skills-review` | `--dim` ⊆ `{full, structural, strategic, freshness}` |

| `templates` | `pe-meta-templates-review` | `--dim` ⊆ `{full, structural}` |

**Create-update vs design dispatch.**

| Caller intent | Resolved dispatch |
|---|---|
| Resolved `--scope` references a non-existent path (create flow) | `pe-meta-{type}-create-update` for the inferred type |

### Domain detection — metadata-first resolution

**Purpose.** Phase 0b (Domain coherence check) computes the **domain footprint** of the resolved scope and gates Phase 1 per domain-coherent-batching. Domain is a **content property declared in artifact metadata**, NOT a path property. This subsection specifies the artifact-type root / semantic-domain distinction, the 3-tier metadata-first resolution algorithm, the per-invocation-type scope-extraction matrix, the CF-05 artifact-type/path consistency check, the seed-footprint vs dependency-footprint distinction (with the `bundle=cross-domain-deps` specialized-lens branch), and cross-repo portability.

#### Artifact-type root vs. semantic domain

**Two distinct concepts; conflating them produces silent misclassification.**

| Concept | Source | Decidable from path alone? | What it drives |
|---|---|---|---|
| **Artifact-type root** | The structural prefix of the file path matched against the 8 canonical roots listed below | **Yes** — deterministic | Scope expansion when `--scope <token>` is used; CF-05 prompt-name/path consistency check |
| **Semantic domain** | The `domain:` field declared in the artifact's YAML frontmatter | **No** — declared, not derived | Phase 0b footprint and bundle gating |

Path slugs like `00.09-pe-meta/`, `00.00-prompt-engineering/`, `pe-granular/` denote *housekeeping/infrastructure* groupings. A prompt under `.github/prompts/00.09-pe-meta/` may target ANY semantic domain because pe-meta prompts are PE customization tooling — their CONTENT determines the domain. A pe-meta prompt that designs agents for the `article-writing` domain may legitimately live under `.github/prompts/00.09-pe-meta/` and declare `domain: article-writing` in its frontmatter.

#### Eight canonical artifact-type roots (for scope expansion and CF-05)

These roots are used for (a) expanding `--scope <token>` to a file set and (b) the CF-05 prompt-name/path consistency check. They are NOT a semantic-domain partitioning.

| # | Artifact-type root | Scope-expansion glob |
|---|---|---|
| 1 | `.copilot/context/` | `.copilot/context/**/*.md` |
| 2 | `.github/prompts/` | `.github/prompts/**/*.prompt.md` |
| 3 | `.github/instructions/` | `.github/instructions/*.instructions.md` |
| 4 | `.github/agents/` | `.github/agents/**/*.agent.md` |
| 5 | `.github/templates/` | `.github/templates/**/*.md` |
| 6 | `.github/skills/` | `.github/skills/*/SKILL.md` |
| 7 | `.github/hooks/` | `.github/hooks/*` |
| 8 | `.github/prompt-snippets/` | `.github/prompt-snippets/**/*.md` |

#### Domain resolution algorithm — metadata-first (3 tiers, deterministic)

Applied **per file** to every artifact in the resolved scope (and every file pulled in by `--deps`).

1. **Tier 1 — declared metadata (authoritative).** Open the artifact's YAML frontmatter and read the `domain:` field. When present, the field value is the domain-id. Record `domain-source=declared-metadata`. Stop.
2. **Tier 2 — optional per-repo path-slug heuristic (fallback).** When `domain:` is absent AND the repository root contains `pe-domain-map.yaml` (or equivalent config registered with the orchestrator), apply path-prefix → domain-id mappings from that map. Record `domain-source=path-heuristic`. The presence of any `domain-source=path-heuristic` assignment in the footprint MUST be flagged in the Phase 8 report ("N files lacked declared `domain:` and were classified by heuristic; consider adding `domain:` to their frontmatter"). Stop.
3. **Tier 3 — `unknown` (final fallback).** When neither tier resolves, assign domain-id `unknown` and emit a per-file warning. `unknown` is a real domain-id for footprint purposes (3 files all `unknown` = single-domain footprint; 1 `unknown` + 1 `prompt-engineering` = 2-domain footprint, gate fires).

**Worked examples.**

| Artifact path | `domain:` frontmatter | `pe-domain-map.yaml`? | Resolved domain-id | `domain-source` |
|---|---|---|---|---|
| `.copilot/context/00.00-prompt-engineering/01.07-critical-rules-priority-matrix.md` | `prompt-engineering` | n/a | `prompt-engineering` | `declared-metadata` |
| `.github/prompts/00.09-pe-meta/pe-meta-agent-design.prompt.md` (declared domain matches housekeeping slug) | `prompt-engineering` | n/a | `prompt-engineering` | `declared-metadata` |
| `.github/prompts/00.09-pe-meta/pe-meta-agent-design.prompt.md` (designs an agent for article-writing) | `article-writing` | n/a | `article-writing` | `declared-metadata` |
| `.github/instructions/legacy-instruction.instructions.md` (older file, no `domain:` declared) | absent | yes; map says `.github/instructions/legacy-* → legacy` | `legacy` | `path-heuristic` (flagged in report) |
| `.github/instructions/legacy-instruction.instructions.md` (same file, different repo without the map) | absent | no | `unknown` | `unknown` (per-file warning) |
| `articles/foo.md` (hypothetical other-repo layout) | `articles` | yes; map says `articles/ → articles` | `articles` | `path-heuristic` |

**Repo-portability property.** The algorithm makes no assumption about subfolder naming. A repo that doesn't use `<NN>.<NN>-<slug>/` and doesn't ship `pe-domain-map.yaml` still works correctly — every file declaring `domain:` in its frontmatter resolves deterministically at Tier 1; files without the declaration fall to `unknown` (recoverable by adding the field).

#### Per-invocation-type scope-extraction matrix

Scope extraction (producing the FILE SET for Phase 0b) is **orthogonal** to domain resolution (producing the DOMAIN-ID per file). Every row below feeds the same 3-tier metadata-first algorithm above.

| Invocation shape | Example | `scope-source` marker | How files are resolved |
|---|---|---|---|
| **Artifact-type token** | `/pe-meta-update --scope context` | `token` | Expand token to all files under the matching artifact-type root (per the 8-root scope-expansion table) |
| **Path-set (single)** | `/pe-meta-update --scope .copilot/context/00.00-prompt-engineering/` | `path-set` | Glob the single path |
| **Path-set (multi)** | `/pe-meta-update --scope .copilot/context/00.00-prompt-engineering/,.copilot/context/01.00-article-writing/` | `path-set` | Glob each comma-separated path; union the file set |
| **Positional `<file-path>`** | `/pe-meta-context-review '.copilot/context/00.00-prompt-engineering/01.07-critical-rules-priority-matrix.md' --deps full` | `positional` | The positional path IS the file (or with `--deps`, the file + its dependency set) |
| **No scope (default-all)** | `/pe-meta-update --mode plan` | `default-all` | Per `default-full-investigation`, expand to every file under every artifact-type root the invoked command supports |

**Phase 0b applies identically across all five rows.** After the file set is resolved, each file's `domain:` frontmatter (or fallback) is read; the resulting domain-id set is the footprint; the gate behavior is decided by footprint size and `--mode`. Per-artifact prompts typically use the **positional** row when invoked directly; a single-file positional invocation produces a one-file file set whose footprint is whatever that single file declares (Tier 1) or `unknown` (Tier 3). With `--deps full`, the traversal set MAY be multi-domain because consumers/dependencies declare their own `domain:` independently.

#### Artifact-type/path consistency check (CF-05, Phase 0a precondition)

**This check operates on artifact-type ROOT (deterministic from path), NOT on semantic domain.**

Per-artifact prompts have a name-encoded artifact-type expectation:

| Prompt-name prefix | Expected artifact-type root |
|---|---|
| `pe-meta-context-*` | `.copilot/context/` |
| `pe-meta-prompt-*` | `.github/prompts/` |
| `pe-meta-instruction-*` | `.github/instructions/` |
| `pe-meta-agent-*` | `.github/agents/` |
| `pe-meta-template-*` | `.github/templates/` |
| `pe-meta-skill-*` | `.github/skills/` |
| `pe-meta-hook-*` | `.github/hooks/` |
| `pe-meta-snippet-*` | `.github/prompt-snippets/` |

**Rule.** When the positional path's artifact-type root does NOT match the expected root for the invoked prompt's prefix, Phase 0a MUST reject the invocation with CF-05 BEFORE Phase 0b runs. The error message MUST identify both the supplied artifact-type root and the expected root, and SHOULD suggest the canonically-correct prompt name.

**Worked rejection example.** `/pe-meta-context-review '.github/prompts/00.09-pe-meta/pe-meta-agent-design.prompt.md' --deps full` is REJECTED with CF-05: supplied path is under `.github/prompts/`, but `pe-meta-context-review` expects `.copilot/context/`. Canonical replacement: `/pe-meta-prompt-review '.github/prompts/00.09-pe-meta/pe-meta-agent-design.prompt.md' --deps full`.

**Orchestrator-level prompts** (`pe-meta-update`, `pe-meta-review`, `pe-meta-create-update`, `pe-meta-design`, `pe-meta-scheduled-review`, `pe-meta-adherence`) are artifact-type-agnostic and skip this check.

#### Single-domain shortcut

When every file in the resolved file set resolves to the SAME domain-id (regardless of which tier produced the resolution, regardless of invocation shape, regardless of whether the domain is `unknown`), Phase 0b emits `bundle=single-domain` in the log and Phase 1 proceeds without any prompt.

#### Seed footprint vs dependency footprint — the specialized-lens branch

Phase 0b computes the domain footprint **separately for the seed scope and for the dependency closure**, then combines them through the following decision matrix:

| Seed footprint | Deps footprint (additional domains beyond seed) | Bundle disposition | Phase 1 behavior |
|---|---|---|---|
| 1 domain | 0 (same single domain, or `--deps off`) | `bundle=single-domain` | Proceed without prompt |
| ≥ 2 domains | any | `bundle=multi-domain-gated` | Propose per-domain split (`domain-coherent-batching`) |
| 1 domain | ≥ 1 additional domain (from `--deps direct`/`full`) | `bundle=cross-domain-deps` | Proceed as ONE review against the union, with per-dependency-domain specialized analysis lenses |

**Definitions.**

- **Seed footprint** = set of distinct `domain:` values across files explicitly scoped by the invocation BEFORE any `--deps` traversal runs:
  - positional invocation → seed = the positional path(s) (single file or single token-expansion set)
  - `--scope <artifact-type-token>` → seed = all files under that artifact-type root (e.g. `--scope context` → all `.copilot/context/**`)
  - `--scope <path-set>` → seed = the listed paths
  - no `--scope` → seed = all artifacts under all eight canonical artifact-type roots
- **Dependency footprint** = set of distinct `domain:` values across files added by `--deps direct` (one hop) or `--deps full` (transitive closure) traversal from the seed.
- **Additional domain** = a domain that appears in the dependency footprint but NOT in the seed footprint.

**Why `bundle=cross-domain-deps` is not a split.** A consumer artifact NEEDS all its declared dependencies to be reviewable. Splitting a single-seed cross-domain-deps invocation by dep-domain would produce **incomplete reviews**: either each split reviews the seed against a subset of its context (the seed cannot be evaluated correctly because rules it inherits are missing), or each split drops the seed (the user's actual review request is silently ignored). Neither outcome is acceptable. The right behavior is to keep the union together and apply **per-dependency-domain specialized analysis lenses** during Phase 2–4 audits.

**What "specialized analysis lenses" means concretely.** Phase 2–4 audits run ONCE on the seed, but the rule-adherence and context-comparison checks within each audit are evaluated **per dependency-domain**, using domain-specific expectations:

- When comparing the seed against `article-writing` dependencies → exercise Microsoft Writing Style Guide voice rules, Diátaxis type validation, accessibility (alt text, inclusive language), readability targets, emoji-H2 rule.
- When comparing the seed against `prompt-engineering` dependencies → exercise the PE rationale set (the `deterministic-processing` through `domain-coherent-batching` family), boundary-actionability redteam pattern, three-layer rule architecture, tool-restriction-as-strict-allowlist, dimension scope contract.
- When comparing the seed against `learning-hub` dependencies → exercise dual-metadata system (top YAML / bottom HTML comment), reference-classification emoji markers, kebab-case naming.

The Phase 8 report MUST section findings **per dependency-domain** (one subsection per distinct dep-domain) so reviewers can see which lens produced which findings and trace each finding back to the rules that triggered it.

**Worked example — `bundle=cross-domain-deps`.**

`/pe-meta-prompt-review '.github/prompts/01.00-article-writing/article-review-for-consistency-gaps-and-extensions.prompt.md' --deps full`

1. **Seed footprint computation.** The positional path is one file. Open it, read `domain:` → `article-writing`. **Seed footprint = { article-writing }**.
2. **Deps full closure.** Walk the prompt's references, file links, and `#file:` snippet inclusions. Closure yields:
   - 6 files under `.copilot/context/01.00-article-writing/` → all declare `domain: article-writing`
   - 4 files under `.copilot/context/00.00-prompt-engineering/` → all declare `domain: prompt-engineering`
   - 2 instruction files (`article-writing.instructions.md`, `documentation.instructions.md`) → declare `domain: article-writing`
3. **Deps footprint = { article-writing, prompt-engineering }**. Additional domains beyond seed = `{ prompt-engineering }` (one).
4. **Decision matrix lookup.** seed = 1, deps adds ≥ 1 → `bundle=cross-domain-deps`. Phase 1 proceeds against the union without prompting for a split.
5. **Phase 8 report sections findings per dep-domain.** One subsection "Findings vs `article-writing` context" applying MWSG/Diátaxis lenses; another "Findings vs `prompt-engineering` context" applying R-P* and boundary-actionability lenses.

**Contrast with `bundle=multi-domain-gated`.** The same closure under `--scope context` (no positional seed) would produce **seed footprint = { article-writing, prompt-engineering, … }** because the user explicitly asked for the whole context-files artifact-type root. There `domain-coherent-batching` fires — splitting is appropriate because each split is independently meaningful (an article-writing review of the article-writing context subset; a PE review of the PE context subset).

**Why `domain-coherent-batching` still applies.** `domain-coherent-batching` governs the seed-multi-domain case, not the seed-single-domain case. The new `bundle=cross-domain-deps` branch is NOT a bypass of `domain-coherent-batching` — it's a different case that `domain-coherent-batching` was never the right tool for. `domain-coherent-batching`'s rationale (per-domain reviewer attention, scoped rollback boundaries, single-domain context windows) all assume the user's intent spans multiple domains. When the user named ONE artifact, the cross-domain deps are infrastructure the seed requires, not separate review targets.

#### `--deps full` and the metadata-first payoff

This is the case where path-based domain extraction silently misclassifies entire runs and the metadata-first model pays off.

**Worked example.** `/pe-meta-update '.github/prompts/00.09-pe-meta/pe-meta-agent-design.prompt.md' --dim context-quality-lifecycle --deps full`:

1. **Phase 0a.** Orchestrator-level prompt (`pe-meta-update`), so CF-05 prompt-name/path check is skipped. `--dim context-quality-lifecycle` parsed and validated against the dimension-group enumeration.
2. **Phase 0b scope extraction.** Positional path → file set = { seed } ∪ deps-full-closure(seed). Suppose closure yields 12 files spanning `.github/prompts/00.09-pe-meta/`, `.copilot/context/00.00-prompt-engineering/`, `.copilot/context/01.00-article-writing/`, `.github/skills/pe-context-file-creation/`.
3. **Phase 0b domain resolution (per file, metadata-first).** Each of the 12 files is opened and its `domain:` field is read:
   - 8 files declare `domain: prompt-engineering`
   - 3 files declare `domain: article-writing`
   - 1 file has no `domain:` field and the repo has no `pe-domain-map.yaml` → `unknown`
4. **Footprint = 3 domains** (`prompt-engineering`, `article-writing`, `unknown`). Gate fires per `domain-coherent-batching`.
5. **Phase 0b output.** Numbered split proposal showing 3 canonical per-domain invocations. `bundle=multi-domain-gated` recorded on the log. User selects split 1 OR `bundle=accept`.

**Why path-walk fails this case.** A path-walk algorithm would read the seed file's path (`.github/prompts/00.09-pe-meta/`), derive `pe-meta` as the domain, and either (a) silently classify the entire 12-file traversal as `pe-meta` (false-negative; gate doesn't fire on a genuinely multi-domain bundle) or (b) classify each file by its own path-walk and produce a footprint based on housekeeping slugs (`pe-meta`, `prompt-engineering`, `article-writing`, `pe-context-file-creation`) that doesn't match what the FILES actually claim to be about.

**Note on `--dim` orthogonality.** `--dim context-quality-lifecycle` filters which dimensions Phase 2–4 audit — it does NOT declare or override the semantic domain of any artifact. Domain footprint is computed entirely from per-file `domain:` metadata.

#### Cross-repo portability

The vision specifies WHAT to read (the `domain:` field) and HOW to fall back (optional path-slug map, then `unknown`). It does NOT mandate any folder-naming convention.

**Optional config schema** (`pe-domain-map.yaml` at repo root):

```yaml
# Per-repo path-slug heuristic for files lacking declared `domain:` metadata.
# Used at Tier 2 of the domain resolution algorithm. Optional.
mappings:
  - path-prefix: .copilot/context/00.00-prompt-engineering/
    domain: prompt-engineering
  - path-prefix: .copilot/context/01.00-article-writing/
    domain: article-writing
  - path-prefix: .github/prompts/00.09-pe-meta/
    domain: prompt-engineering    # Note: housekeeping slug ≠ semantic domain
```

**Migration guidance for repos adopting v15:** the recommended path is to declare `domain:` in every PE artifact's frontmatter (so Tier 1 always resolves). Shipping `pe-domain-map.yaml` is a transitional convenience for legacy/unannotated files only; it is NOT required for the algorithm to work.

#### Mode elimination from scheduled review

The scheduled review command historically used a `--mode` parameter with values `individual|dep-aware-full|dep-aware-shallow|guidance-first`. This created semantic overloading — `--mode` meant "execution depth" in other commands but "traversal depth + rotation behavior" in scheduled review.

**Resolution:** Scheduled review's modes map directly to `--deps` values plus an internal rotation mechanism:

| Legacy mode | Replacement | Behavior |
|---|---|---|
| `--mode individual` | `--deps none` | Assess each target in isolation |
| `--mode dep-aware-full` | `--deps full` | Include transitive dependencies |
| `--mode dep-aware-shallow` | `--deps direct` | Include direct dependencies only |
| `--mode guidance-first` | Internal rotation | Auto-delegates to `/pe-meta-adherence` |

**Auto-rotation cycle:** When `--deps` is omitted, scheduled review automatically cycles through depth levels:

```
--deps none → --deps full → --deps direct → adherence-delegation → repeat
```

This cycle ensures comprehensive coverage across invocations without requiring the user to manually specify traversal depth each time. The adherence-delegation step invokes the guidance-first family (`/pe-meta-adherence`) to cover the reverse-direction quality trace.

### Explicit option discipline

Commands accept only the seven canonical parameters (`--mode`, `--scope`, `--source`, `--dim`, `--start`/`--end`, `--deps`, `--skip`). Convenience presets are not provided: every invocation MUST state its execution depth (`--mode`), pipeline stages (`--skip`), and target dimensions/scope explicitly. The default values defined in the [option taxonomy](#option-taxonomy) apply when an option is omitted.

**Rationale.** A self-documenting command line is more valuable than a memorable shortcut. Named presets force the reader to look up an expansion table to predict behavior; explicit options encode the behavior on the command line itself. With only seven orthogonal parameters, the taxonomy is small enough that explicit invocation is the better default (minimal-consistent-option-surface).

**Worked equivalences** (for migration from earlier vision drafts that defined `healthcheck` / `performancecheck` / `fullcheck` preset aliases):

| Legacy preset | Canonical replacement |
|---|---|
| `healthcheck` | `--mode plan --skip research` |
| `performancecheck` | `--mode apply --dim optimize --skip research,structure,consistency` |
| `fullcheck` | `--mode apply` (or omit `--mode` entirely — `apply` is the default) |

**Composition rule.** Explicit user flags always win. When a `--skip` value is supplied on top of one of the legacy expansions during migration, lists are merged with deduplication (e.g., legacy `healthcheck --skip research` resolves to `--mode plan --skip research` with `research` appearing once).

**UX safety net.** Natural-language intent words (`review`, `audit`, `optimize`, `dry-run`) MAY be resolved by the orchestrator prompt into canonical option combinations, but the resolved combination MUST be echoed back to the user as a canonical invocation string before execution. No hidden expansions, no named presets in the parser's accepted vocabulary.

## What "self-updating" means (and what it doesn't) 🔍

**Maintenance scope.** Self-update maintains and evolves *existing* artifacts — updating, restructuring, merging, splitting, or deprecating them. Creation of entirely new artifact categories (those that don't replace or extend an existing one) is out of scope and flagged as a gap for human action.

**It means:**

- Watching for platform, internal, and ecosystem signals that artifacts may need attention
- Mapping signals to specific artifacts via tier + dependency analysis
- Running targeted validation checks on affected artifacts
- Evaluating external artifacts for adoption or incorporation based on trust and fit
- Generating human-readable correction proposals with evidence
- Executing low-risk changes autonomously with post-execution notification
- Escalating high-risk or uncertain changes to a human with full context
- Tracking outcomes to tune autonomy thresholds

**It doesn't mean:**

- Changing the system's vision, goals, or governing rules without human direction
- Proceeding when risk assessment indicates high uncertainty
- Skipping validation to save time or tokens
- Generating entirely new artifact categories from scratch
- Replacing human judgment for architectural or strategic decisions
- Automatically adopting every external artifact that overlaps with an owned one
- Guaranteeing zero drift (some lag between change and detection is expected and acceptable)
- Executing changes that haven't passed at least one independent validation pass

## Risks and tensions ⚠️

> **Priority for implementations.** When risks force a trade-off, never trade away reliability or safety to recover efficiency or autonomy. Mitigations that lower the autonomy ceiling are preferred over mitigations that loosen validation or skip multi-pass checks.

### False confidence (includes effectiveness hard limit)

A passing health check doesn't mean artifacts are optimal — just internally consistent and reference-sound. More critically: validation can check structural integrity and internal consistency, but **it cannot verify that a prompt still produces good output after a change**. Effectiveness is only measurable through actual use. This is a permanent limitation. Autonomous operation can maintain reliability and efficiency; effectiveness can only be validated indirectly or through human judgment. *Mitigation: self-correction (multi-pass) + progressive-learning (outcome tracking) catch regressions that post-deployment behavior reveals. Residual: silent effectiveness drift remains a hard boundary.*

### Self-update infrastructure staleness (amplified risk)

A bug in a regular context file affects the prompts that reference it. A bug in the system's own meta-guidance affects every decision the system makes — including its ability to detect the bug. The system CAN update its operational components through the standard pipeline, but these updates carry amplified blast radius and receive elevated risk classification. Updates to the vision, meta-principles, and risk threshold boundaries are human-only. *Mitigation: Staleness avoidance first principle — hard gate before every cycle, no caching on meta-components, CRITICAL findings for dangling references in self-update infrastructure.*

### Over-correction

A system tuned to detect every minor inconsistency generates noise. A typo in a version history entry isn't worth a correction proposal. Severity thresholds limit findings to those affecting artifact reliability or correctness.

### Trigger completeness

The system can only react to changes it knows about. Missing triggers — in monitored platform channels, ecosystem sources, or internal events — let drift accumulate until a time-based check catches it. Trigger coverage is a perpetual gap to manage, not a problem to solve once.

### Adoption judgment

Deciding whether to adopt or incorporate external work requires judgment the system can only partially automate. Trust signals are heuristic, not definitive. The adopt-vs-incorporate decision remains a human call.

### Cost of vigilance

Every health check consumes tokens. Tier-based priority cadence bounds the cost: Tier 0/1 artifacts are reviewed every cycle; lower tiers are checked only on relevant stale triggers. The dependency graph (dependency-graph) minimizes redundant checks; caching (validation-caching) skips recently-validated artifacts; meta-components are always checked.

### Concurrency (future concern)

Concurrent modification — two self-update cycles running simultaneously, or a human editing an artifact while the system proposes changes — requires file-level locking or sequential processing. Acceptable at current manual-invocation scale; must be addressed before event-driven automation.

### Complexity ceiling

A self-updating system that becomes too complex to understand is self-defeating. Simplicity in the update mechanism is a design goal, not a nice-to-have.

### Deterministic rigidity

Deterministic parsers break when formats change (new YAML field, new metadata structure). When a deterministic check encounters unexpected structure, the system falls back to LLM processing and logs the fallback for investigation. Accumulated fallback logs signal that the deterministic check needs updating.

### Instruction file collision

Instruction files auto-inject by glob pattern with no precedence mechanism. Overlapping `applyTo` patterns cause silent rule collisions. *Mitigation: Instruction minimization principle (instruction-minimization). Residual: testable rules can still contradict on measurable dimensions, but these contradictions are detectable and resolvable.*

### Insufficient outcome data

Progressive learning (progressive-learning) requires sufficient volume of autonomous actions and outcomes to produce meaningful signal. Start with conservative thresholds and only loosen as outcome data accumulates.

## Success criteria ✅

> **Priority for implementations.** Outcome criteria (1–9) are the measurable results the system must produce. Process invariants (10–12) are non-negotiable runtime conditions — when an outcome criterion can only be met by breaking an invariant, the invariant wins.

The self-updating prompt engineering system succeeds when:

### Outcome criteria

1. **Time to detect drift drops from weeks to days.** *Query: `outcome-log.jsonl` entries where `trigger ≠ scheduled` have `timestamp` within N days of the triggering event.*

2. **Manual audit frequency decreases.** Full system reviews become confirmation checks rather than discovery sessions. *Query: count of `autonomy_level: autonomous | notify` entries over the period.*

3. **Correction proposals are actionable.** When the system surfaces a finding, it's specific enough to approve or reject without additional investigation. *Query: human-approved entries in the log without additional follow-up actions recorded.*

4. **False positive rate stays low.** *Query: ratio of `outcome: success` to `outcome: rolled_back` for `autonomy_level: autonomous` entries over rolling 30 days.*

5. **The system stays simple enough to maintain.** Update machinery is understandable, debuggable, and modifiable. *Query: percentage of meta-infrastructure changes that required human escalation.*

6. **Artifact quality improves measurably.** Review scores trend upward. CRITICAL and HIGH findings per audit cycle decrease. *Query: per-cycle average `validations_failed` count over time.*

7. **Self-update process cost stays bounded.** Token consumption per review cycle doesn't grow unbounded as the artifact set grows. Scoped invocations (per-type review) cost significantly less than full-system reviews. *Query: estimated token cost per `trigger_source` type, tracked over releases. Ratio of screening-only completions to screening-triggered-deep-pass escalations, tracked per invocation type.*

8. **Autonomous execution success rate exceeds threshold.** The percentage of autonomous changes that pass post-implementation verification stays above 95%. When it drops, the system tightens autonomy thresholds. *Query: `outcome: success` / total `autonomy_level: autonomous` entries over rolling 30 days.*

9. **Guidance quality improves measurably.** The six guidance quality properties (clarity, non-redundancy, non-contradiction, completeness, prioritization, actionability) score above threshold across all active context files. *Query: per-context-file scores on each property, tracked across review cycles.*

### Process invariants

10. **Design/review parity stays explicit and stable.** Goal, rationale, and process sections consistently state one shared quality objective and one shared scope intent for design and review. *Query: deterministic parity check confirms required parity statements are present and no section claims lower quality targets for either role.*

11. **Reliability coverage is explicit.** Every reliability anchor in `scope.covers` and in the rationales (loop stability, multi-pass validation invariant, rollback readiness, metadata-guarded protocol enforcement, fail-closed boundaries, boundary actionability, autonomy calibration, conflict-resolution precedence, portability boundary, mode-vs-risk decoupling) is addressed by at least one use case in `05-reliability/`. *Query: deterministic coverage check confirms each anchor maps to ≥1 UC ID in `usecase-index.json` with `group: "05-reliability"`.*

12. **Resolved invocation is logged on every run as the first line of the report.** Every invocation of any `/pe-meta-*` command (not just `/pe-meta-update`), regardless of caller type, emits the resolved canonical invocation string (the seven canonical parameters with their resolved values) and the resolved breadth (`full` | `incremental` | `bounded-delta`) as the first line of the Phase 8 report. No exceptions. This makes silent re-interpretation of parameter-less invocations impossible to hide (default-full-investigation). **v15 marker extensions.** The first-line `Resolved invocation:` block ALSO emits (a) `scope-source=` with values `token` | `path-set` | `positional` | `default-all` recording which row of the per-invocation-type scope-extraction matrix produced the file set, and (b) `bundle=` with values `single-domain` | `multi-domain-gated` | `accepted-bundle` | `split-N` | `cross-domain-deps` recording the Phase 0b disposition. **v15.1 marker extensions.** The first-line block ALSO emits (c) `plan-file=<path>` whenever the resolved `--mode` is `plan`, recording the canonical plan-file path written under § Plan-mode output contract; and (d) `spillover=<path>|none` recording whether the run hit the iteration-budget overflow (§ Iteration budget). The Phase 8 footprint table additionally emits a `domain-source=` column per file with values `declared-metadata` | `path-heuristic` | `unknown`, so reviewers can audit which files were resolved at each of the three resolution tiers (domain-coherent-batching, § Domain detection — metadata-first resolution). *Query: every entry in `outcome-log.jsonl` has a non-empty `resolved_invocation` field, a `resolved_breadth` value drawn from the three legal values, a `scope_source` value drawn from the four legal values, a `bundle` value drawn from the five legal values, `plan_file` is non-empty when `mode == "plan"`, `spillover` is either a path or `none`, and every file in the footprint table carries a `domain_source` value drawn from the three legal values; deterministic linter rejects entries that violate any of these.*

### Parity acceptance checks for future vision updates

Use these deterministic checks in future vision updates:

1. **P-1 shared quality objective present** — Goal section and rationale text include explicit parity wording for design and review.
2. **P-2 shared scope intent present** — Scope/goal language states that applicability filtering is by artifact type, not by role quality target.
3. **P-3 process-role separation present** — Design is described as synthesis/construction and review as analysis/verification/improvement.
4. **P-4 conflict guardrail present** — Conflict-resolution section states that process optimization cannot downgrade parity on quality objective or scope intent.
