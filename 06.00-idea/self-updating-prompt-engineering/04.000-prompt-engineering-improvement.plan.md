# Prompt engineering improvement plan

> **Goal**: Bring the existing PE infrastructure into alignment with [04.000-vision.v4.md](04.000-vision.v4.md) so that every artifact type can be reliably created, reviewed, updated, and self-maintained — and so that the same infrastructure can produce and maintain artifacts for any domain (starting with `01.00-article-writing`).
>
> **Related**: [05.000-prompt-engineering-implementation.plan.md](05.000-prompt-engineering-implementation.plan.md) — addresses agent architecture (Option A vs Option D comparison via folder separation)

---

## 🎯 Vision summary and ultimate goal

The vision describes a **self-updating PE system** that:

1. **Continuously detects drift** (platform, internal, ecosystem)
2. **Assesses impact** using tier + dependency analysis
3. **Proposes corrections** with evidence and risk classification
4. **Executes validated corrections** autonomously for low-risk changes, with human approval for high-risk ones
5. **Learns from outcomes** to tune autonomy thresholds over time

The **ultimate goal** is a PE system that behaves like a responsible maintainer: it watches, assesses, acts, and improves — with calibrated autonomy. The system never silently degrades because it detects staleness before it causes failures.

### Instruction file strategy

Instruction files use `applyTo` glob patterns — their location doesn't affect injection, only the glob pattern does. Two viable approaches:

- **Prefix (chosen)**: `pe-agents.instructions.md` → `pe-agents.instructions.md` — simpler, no folder restructuring
- **Subfolder**: `.github/instructions/00.00-prompt-engineering/` — cleaner but adds a folder level

`pe-` prefixed instructions apply to ALL matching files via their `applyTo` globs — this is correct behavior. PE rules (`pe-agents.instructions.md`) should govern all agent files (both `00.00` and `00.01` architectures). No duplication between `pe-` and `pe1-` — the same instruction files serve both.

### Phase 0: Foundation — metadata contracts and structural separation

**Goal**: Establish the metadata and structural prerequisites that all subsequent phases depend on.

#### Action 0.1 (✅ Done): Define and document the canonical metadata schema

- **What**: Create a new context file `.copilot/context/00.00-prompt-engineering/00.03-metadata-contracts.md` defining the required YAML fields for every artifact type
- **Fields to standardize**: `goal:` (single sentence), `scope: {covers: [...], excludes: [...]}`, `boundaries:` (list), `rationales:` (list of design constraints), `version:` (SemVer)
- **Include**: placement rules (top YAML for context/instructions; `agent_metadata:` for agents; `template_metadata:` for templates), examples for each artifact type, validation rules
- **Deliverable**: Context file ≤2,500 tokens
- **Vision reference**: R-S1-metadata-driven, R-S4-role-declaration

#### Action 0.2 (✅ Done): Apply metadata contracts to governance and foundation context files (Tier 0–1)

- **Method**: Run `/context-information-create-update` for each file with the metadata additions
- **Validation**: `context-validator` confirms schema compliance
- **Estimated effort**: 9 file updates

#### Action 0.3 (✅ Done): Apply metadata contracts to Tier 2–5 context files

#### Action 0.4 (✅ Done): Apply metadata contracts to instruction files

## ⚠️ Risk mitigation

| Risk | Mitigation |
|---|---|
| Metadata additions increase token consumption | Keep metadata fields minimal (5 fields, ~50 tokens per artifact). Monitor total token impact after Phase 0 |
| N-1 refactoring breaks existing context file consumers | Refactor one file at a time, validate all consumers after each change. Start with lowest-blast-radius files (Tier 5 → Tier 0) |
| Agent consolidation removes specialization | Action 4.4 is assessment only. Any consolidation requires separate human-approved plan |
| Template deduplication breaks references | Audit all consumers before removing any template (Action 6.5). Update references first |
| Structured logs add repo clutter | State folder is gitignore-able if preferred. Portable deployments can keep state external |
