## 🎯 Purpose of this document

The [v2 vision](02.000-vision.v2.md) defines a complete self-updating PE system — detection, assessment, proposal, execution — with a rich governance model, risk calibration, and rollout phases. It works. But it's designed for one repository: the Learning Hub.

This document answers one question:

**How does the vision change when the goal isn't just "maintain my PE artifacts" but "give anyone a self-updating PE system they can install in their own repository with minimal setup"?**

The answer introduces a new design constraint — **portability** — and traces its implications through the vision's four operational layers, its infrastructure assumptions, and its delivery model.

---

## 💡 The core insight

The v2 vision conflates two things that should be separate:

1. **The engine** — the logic that detects drift, assesses impact, proposes corrections, and executes validated changes. This is repository-agnostic. It reasons about PE artifacts regardless of which repo they live in.
2. **The integration** — the infrastructure the engine needs in a specific repository to operate: trigger wiring, state files, metadata conventions, agent definitions, hook configurations.

Today, both are woven together. The engine's logic lives in meta-agents that live in `.github/agents/`. The state lives in `.copilot/self-update/`. The triggers live in `.github/hooks/`. The validation skills live in `.github/skills/`. A developer who wants this system must adopt the entire Learning Hub PE structure.

**The portability principle:** separate the engine from the integration. Package the engine as something installable — an MCP server, a GHCP SDK component, a skill pack, or a VS Code extension. Keep the per-repo integration as thin as possible — ideally a single configuration file and a handful of optional GHCP customization artifacts.

This isn't a minor addition. It's an architectural constraint that reshapes how the vision's infrastructure, state management, and delivery model work.

## 📊 Gap analysis: what v2 assumes vs. what portability requires

### Infrastructure assumptions in v2

The v2 vision assumes significant per-repo infrastructure:

| Component | v2 location | Files/complexity | Portability problem |
|---|---|---|---|
| Meta-agents (27 files) | `.github/agents/00.00-prompt-engineering/` | 27 agent definitions | Developer must adopt the full meta-agent fleet |
| Context files (30+ files) | `.copilot/context/00.00-prompt-engineering/` | 30+ context documents | Developer must absorb ~30K tokens of PE context |
| Instruction files (12 files) | `.github/instructions/` | 12 instruction files | Developer's instruction layer gets crowded |
| Skills (3 PE-specific) | `.github/skills/` | 3 skill folders | Manageable, but still per-repo |
| Hooks (3+ JSON) | `.github/hooks/` | 3+ hook configs + scripts | Developer must adopt hook infrastructure |
| State files | `.copilot/self-update/` | outcome-log, rename-log, snapshots, triggers | New folder with append-only files |
| Templates | `.github/templates/` | Multiple template files | Developer must absorb template conventions |

**Total footprint:** ~80+ files across 7 directories. That's not "install and use" — that's "adopt a framework."

### What portability requires

| Concern | Requirement |
|---|---|
| **Minimal footprint** | The per-repo integration should be < 5 files. Everything else lives in the engine. |
| **No structural imposition** | The system should work with the developer's existing PE structure, not demand a specific one. It discovers artifact types, locations, and conventions — it doesn't prescribe them. |
| **Opt-in depth** | A developer can start with "scan my artifacts and tell me what's stale" and progressively enable assessment, proposal, and execution. |
| **State externalization** | Outcome logs, rename logs, and snapshots can live outside the repo (in the engine's state) or in a single, well-scoped repo location. |
| **Platform leverage** | The system runs on capabilities that already exist: GHCP chat, GHCP agents, MCP tool calls, VS Code tasks, `fetch` for web content. No custom runtime. |

## 🏗️ Proposed architectural change: engine/integration separation

### The engine (portable, installable)

The engine contains the self-update logic from all four vision layers (Detect, Assess, Propose, Execute). It exposes its capabilities through a standard interface — MCP tools, GHCP agent skills, or SDK functions — rather than as in-repo agent files.

**What moves into the engine:**

| Capability | v2 location | Engine form |
|---|---|---|
| Drift detection logic | Meta-agents + hooks | MCP tools or GHCP SDK functions: `detect_platform_drift`, `detect_internal_drift`, `detect_ecosystem_changes` |
| Impact assessment | Meta-agents + skills | MCP tools: `assess_impact`, `classify_risk`, `compute_blast_radius` |
| Proposal generation | Meta-agents | MCP tools: `propose_corrections`, `decompose_changes` |
| Validation | Skills (`prompt-engineering-validation`, `artifact-coherence-check`) | MCP tools or bundled skill definitions: `validate_coherence`, `validate_artifacts` |
| Execution | Meta-agents + deterministic scripts | MCP tools: `execute_change`, `rollback_change` |
| State management | `.copilot/self-update/` files | Engine-managed state (local DB, JSON files in engine directory, or configurable repo path) |

**What the engine needs from the repo (discovery, not prescription):**

| Input | How the engine gets it |
|---|---|
| Where PE artifacts live | Configuration file or workspace scan (glob patterns) |
| What artifact types exist | File extension + YAML frontmatter detection |
| Artifact metadata (goal, scope, boundaries) | Parse from existing YAML frontmatter where present; work without it where absent |
| Dependency relationships | Parse from `#file:` references, `applyTo` patterns, `enforces_rulesets` declarations, import paths |
| Platform version | VS Code API or `code --version` |
| GHCP capabilities | Feature detection at runtime |

### The integration (per-repo, minimal)

The per-repo integration is what a developer installs to connect the engine to their repository.

**Thin integration model (< 5 files):**

| File | Purpose | Required? |
|---|---|---|
| `.github/copilot-instructions.md` addition | One paragraph telling GHCP that self-update tools are available | Optional — engine works via direct MCP tool calls too |
| `pe-self-update.config.json` (or YAML) | Artifact locations, glob patterns, autonomy thresholds, iteration budget, monitored sources | Required — but sensible defaults for everything |
| `.github/agents/pe-self-update.agent.md` | A single agent definition that wraps the engine's MCP tools into a conversational interface | Optional — for developers who prefer `@pe-self-update` over direct tool calls |
| `.copilot/self-update/` | State directory (outcome log, snapshots) — only created when execution is enabled | Optional — detect/assess/propose work without it |

**That's 1 required file and 3 optional files.** Compare to v2's ~80+ files.

### Discovery over prescription

The engine doesn't require artifacts to follow the Learning Hub's conventions. It discovers:

- **Artifact types** by scanning file extensions, YAML frontmatter (`description:`, `applyTo:`, `tools:`), and conventional paths (`.github/instructions/`, `.github/agents/`, `.github/prompts/`, `.copilot/context/`)
- **Metadata** by parsing whatever's present — `goal:`, `scope:`, `boundaries:`, `version:` — and operating in degraded-but-functional mode when metadata is absent
- **Dependencies** by parsing `#file:` includes, `applyTo` globs, folder references, and cross-file citations
- **Conventions** by inferring from patterns (naming conventions, folder structure, metadata consistency)

When metadata is missing, the engine can still detect structural staleness (dangling references, broken paths). It can't assess capability staleness without `goal:`/`scope:` — and it says so, recommending metadata addition as a first step.

## 🔬 Robustness review

Each proposed change is tested against three questions:

1. **Does it maintain the v2 vision's existing strengths?** (governance model, risk calibration, multi-dimensional impact, loop stability)
2. **Does it introduce infrastructure that doesn't exist yet?**
3. **Is it proportional to the problem?**

### What this plan preserves from v2

Everything. The engine/integration separation is an architectural *refactoring*, not a rewrite. Every v2 concept survives:

| v2 concept | Status in v3 |
|---|---|
| Four operational layers (Detect, Assess, Propose, Execute) | Preserved — now exposed as engine capabilities |
| Autonomy gradient | Preserved — thresholds configurable per-repo |
| Multi-dimensional impact classification | Preserved |
| Finding severity (CRITICAL/HIGH/MEDIUM/LOW) | Preserved |
| Risk-ordered execution | Preserved |
| Loop stability (convergence, oscillation detection) | Preserved |
| Iteration budget | Preserved — configurable in `pe-self-update.config.json` |
| Rollout phases | Preserved — phases now also describe engine capability tiers |
| Validation caching | Preserved — state managed by engine |
| Breaking/non-breaking classification | Preserved |

### What changes

| Area | v2 | v3 |
|---|---|---|
| **Deployment model** | In-repo artifacts only | Engine (external) + integration (in-repo, minimal) |
| **State management** | `.copilot/self-update/` assumed | Engine-managed, configurable location |
| **Artifact conventions** | Learning Hub structure prescribed | Discovered from whatever structure exists |
| **Entry point** | Meta-agent `@mention` | MCP tool call, `@agent` mention, GHCP SDK function, or VS Code command |
| **Scalability** | One repo | Any repo with PE artifacts |

### Risk assessment of this change

| Risk | Severity | Mitigation |
|---|---|---|
| **Engine becomes a black box** | MEDIUM | Engine exposes structured output (JSON findings, proposals) — the developer sees exactly what it found and why |
| **Discovery heuristics fail on unusual repo structures** | LOW | Configuration file overrides discovery; explicit glob patterns always work |
| **State externalization complicates rollback** | MEDIUM | State format is identical to v2 (JSONL logs, file snapshots); only the *location* changes |
| **MCP server adds operational complexity** | LOW | MCP servers are a standard pattern in GHCP; the developer already has MCP infrastructure if they use Copilot |
| **Engine versioning creates drift between engine and repo expectations** | MEDIUM | Engine version declared in config; engine validates compatibility on startup |

## ✅ Recommended changes to the vision

Seven changes, organized by theme. Each traces back to a specific v2 section and explains what's added or modified.

### Theme 1: portability as a design principle

**Where it goes:** New design principle after "Deterministic where possible" in the Design principles section.

| # | Change | What it adds |
|---|---|---|
| 1 | Add a **"Portable by design"** principle: the self-update system separates engine (logic, validation, state management) from integration (per-repo configuration, optional agent wrappers). The engine is repository-agnostic. The integration is minimal. | Establishes portability as a first-class design constraint, not an afterthought. |

**Proposed text:**

> ### Portable by design
>
> *Relies on: R-P1-deterministic-processing, R-S1-metadata-driven*
>
> The self-update system separates **engine** from **integration**. The engine contains all detection, assessment, proposal, and execution logic. The integration is the minimal per-repo configuration that connects the engine to a specific repository's artifacts. The engine is repository-agnostic — it discovers artifact types, locations, and conventions rather than prescribing them. The integration is small (< 5 files), opt-in in depth, and doesn't impose structural conventions on the developer's PE setup.

### Theme 2: delivery vehicles

**Where it goes:** New section after "Rollout phases" in The vision, or as a subsection within it.

| # | Change | What it adds |
|---|---|---|
| 2 | Add a **"Delivery vehicles"** section describing how the engine can be packaged: MCP server (primary), GHCP SDK component, VS Code extension, or skill pack. Each vehicle uses existing platform capabilities — no custom runtime. | Answers "how does someone actually install this?" |

**Proposed text:**

> ### Delivery vehicles
>
> The engine can be packaged through multiple delivery vehicles, each leveraging existing platform capabilities:
>
> | Vehicle | How it works | Best for |
> |---|---|---|
> | **MCP server** (primary) | Exposes engine capabilities as MCP tools. Developer adds the server to their MCP configuration. Tools are available in GHCP chat and agents. | Developers who want tool-level control and composability |
> | **GHCP SDK app** | Engine runs as a hosted service. Developer connects via GHCP SDK integration. | Teams who want a managed, always-on service |
> | **VS Code extension** | Engine bundled as a VS Code extension with commands, status bar indicators, and task integration. | Developers who prefer IDE-native experience |
> | **Skill pack** | Engine logic packaged as a set of GHCP skills. Lighter than MCP but less composable. | Quick adoption with minimal infrastructure |
>
> All vehicles share the same core engine. The delivery vehicle determines the *interface*, not the *logic*. A developer can switch vehicles without changing their PE artifacts or configuration.
>
> The MCP server is the primary vehicle because it's the most composable: MCP tools can be called from agents, prompts, chat, and automation — covering all GHCP interaction surfaces. The GHCP SDK vehicle is the path to always-on operation (scheduled triggers, event-driven updates).

### Theme 3: integration model

**Where it goes:** Modifies the infrastructure references throughout The vision (Detect, Assess, Propose, Execute layers).

| # | Change | What it adds |
|---|---|---|
| 3 | Add a **"Per-repo integration"** subsection defining the minimal footprint: 1 required configuration file + 3 optional files. Replaces the assumption that meta-agents, hooks, and state files are always present. | Defines what "install" means for a new repository. |
| 4 | Add a **"Discovery over prescription"** principle: the engine discovers artifact types, locations, dependencies, and conventions from whatever structure exists. It operates in degraded-but-functional mode when metadata is absent. | Removes the dependency on Learning Hub conventions. |

**Proposed text for principle:**

> ### Discovery over prescription
>
> *Relies on: R-S1-metadata-driven, R-P1-deterministic-processing*
>
> The engine discovers a repository's PE structure rather than prescribing one. It scans for artifact types by file extension and YAML frontmatter, maps dependencies from `#file:` references and `applyTo` patterns, and infers conventions from naming patterns and folder structure. When structured metadata (`goal:`, `scope:`, `boundaries:`) is present, the engine uses it for precise impact classification. When it's absent, the engine operates in structural-only mode — detecting dangling references, broken paths, and format inconsistencies — and recommends metadata addition as an improvement.

### Theme 4: new rationale

**Where it goes:** New rationale in the System infrastructure section.

| # | Change | What it adds |
|---|---|---|
| 5 | Add rationale **R-S7-portable-packaging**: the self-update system's logic should be deliverable through standard platform mechanisms (MCP servers, GHCP SDK, VS Code extensions) without requiring the target repository to adopt a specific PE framework or folder structure. | Grounds the portability principle in a named, referenceable rationale. |

**Proposed rationale:**

> | **R-S7-portable-packaging** | **Portable packaging through platform mechanisms.** The engine's logic must be deliverable through standard platform mechanisms — MCP servers, GHCP SDK components, VS Code extensions — without requiring the target repository to adopt a specific PE framework or folder structure. The per-repo integration is configuration, not adoption. | Package the engine for standard delivery vehicles. Define a minimal configuration contract. Discover rather than prescribe artifact structure. |

### Theme 5: rollout phase revision

**Where it goes:** Modifies the existing "Rollout phases" table.

| # | Change | What it adds |
|---|---|---|
| 6 | Add a **Phase 0** to rollout phases: "Engine extraction and packaging." This phase separates engine logic from Learning Hub-specific integration, defines the configuration contract, and produces the first installable MCP server. | Acknowledges that portability itself is a deliverable, not just a principle. |

**Proposed addition to the rollout table:**

> | **Phase 0 (new)** | Engine extraction: MCP server with detect + assess tools | Engine/integration separation complete, configuration schema defined | Proves the architecture works outside the Learning Hub before adding execution autonomy |

### Theme 6: scope note revision

**Where it goes:** Modifies the scope note at the top of "The problem" section.

| # | Change | What it adds |
|---|---|---|
| 7 | Expand the scope note to acknowledge that the system is designed to be portable across repositories, not just applicable to the Learning Hub. | Sets the right expectation from the first paragraph. |

**Proposed revision:**

> **Scope note.** This vision is designed for prompt engineering systems built on GitHub Copilot's customization framework (instructions, agents, prompts, context files, skills, templates, hooks) in VS Code. The system is architected for portability — its core engine is repository-agnostic and installable via MCP servers, GHCP SDK, or VS Code extensions. The principles and rationales transfer to any repository with PE artifacts.

## ⚠️ Changes that are premature or out of scope

| Idea | Why it's premature |
|---|---|
| **Define the full MCP tool API** | The tool surface should emerge from building Phase 0, not from upfront design. The vision states the *principle*; the implementation defines the *interface*. |
| **Build a marketplace or distribution system** | Distribution mechanics (npm, VS Code marketplace, GitHub releases) are implementation details for after Phase 0 proves the architecture. |
| **Multi-repo orchestration** | Running the engine across multiple repositories simultaneously is a Phase 3+ concern. Phase 0–1 is single-repo. |
| **Authentication and access control for hosted engine** | Only relevant for the GHCP SDK vehicle. Deferred until that vehicle is built. |
| **Automatic PE structure scaffolding** | The engine should discover, not prescribe. A "scaffold my PE structure" tool is a separate product. |

## 🚀 Delivery vehicle options

A deeper analysis of each vehicle to inform Phase 0 design decisions.

### MCP server (recommended primary vehicle)

**Why primary:** MCP is the standard tool protocol for GHCP. An MCP server is discoverable by any GHCP agent, callable from any prompt, and composable with other MCP servers. It's the most flexible delivery vehicle.

**What it looks like:**

```
pe-self-update-mcp/
  package.json           # or Cargo.toml, go.mod, pyproject.toml
  src/
    server.ts            # MCP server entry point
    tools/
      detect.ts          # detect_platform_drift, detect_internal_drift
      assess.ts          # assess_impact, classify_risk
      propose.ts         # propose_corrections, decompose_changes
      validate.ts        # validate_coherence, validate_artifacts
      execute.ts         # execute_change, rollback_change
    discovery/
      scanner.ts         # Artifact type and location discovery
      metadata.ts        # YAML frontmatter parsing
      dependencies.ts    # Reference and dependency mapping
    state/
      outcome-log.ts     # Outcome logging (local or repo)
      snapshots.ts       # Pre-change snapshots
```

**Developer installation:**

```json
// .vscode/mcp.json (or settings.json)
{
  "servers": {
    "pe-self-update": {
      "command": "npx",
      "args": ["pe-self-update-mcp"],
      "env": {}
    }
  }
}
```

Plus a configuration file in the repo:

```json
// pe-self-update.config.json
{
  "artifact_paths": {
    "context": ".copilot/context/",
    "instructions": ".github/instructions/",
    "agents": ".github/agents/",
    "prompts": ".github/prompts/",
    "skills": ".github/skills/",
    "templates": ".github/templates/",
    "hooks": ".github/hooks/"
  },
  "autonomy": {
    "max_autonomous_changes_per_cycle": 10,
    "autonomous_severity_threshold": "LOW",
    "notify_severity_threshold": "MEDIUM"
  },
  "monitored_sources": [
    "https://code.visualstudio.com/updates",
    "https://docs.github.com/en/copilot/customizing-copilot"
  ]
}
```

### GHCP skill pack (lightweight alternative)

**Why alternative:** Skills are simpler than MCP servers — they're Markdown files with structured instructions. A skill pack can provide the engine's reasoning patterns without requiring a running server process.

**Trade-off:** Skills are loaded into the LLM's context window. The engine's full logic may exceed practical token budgets for a single skill. Works well for detect + assess; less well for execute.

**What it looks like:**

```
pe-self-update-skills/
  detect-drift/SKILL.md
  assess-impact/SKILL.md
  propose-corrections/SKILL.md
  validate-artifacts/SKILL.md
```

Developer copies skills into `.github/skills/` or references them from a shared location.

### VS Code extension (IDE-native experience)

**Why option:** A VS Code extension can provide status bar indicators ("3 stale artifacts"), command palette commands ("PE: Run health check"), and task integration — a more integrated experience than MCP tools in chat.

**Trade-off:** Distribution through VS Code marketplace. More complex to build and maintain. Limited to VS Code (no GitHub.com, no CLI).
