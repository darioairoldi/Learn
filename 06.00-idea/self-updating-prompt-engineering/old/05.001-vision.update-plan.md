## 🏗️ Proposed: instruction minimization principle (for v6)

### The problem

Instruction files auto-inject via `applyTo` glob patterns. VS Code provides **no precedence mechanism** — when two instruction files match the same path, both inject, and conflicting rules collide silently. The LLM receives contradictory directives with no signal about which takes priority.

This is a **structural vulnerability** that the vision's portability promise (R-S7-portable-packaging) exposes:

**Scenario:** Repository uses two PE systems — a granular architecture (type-specific agents like `pe-context-builder`) and a consolidated architecture (broad agents like `pe-builder`, per the improvement plan's Option D). If both architectures produce instruction files with overlapping `applyTo` patterns:
- `pe-agents.instructions.md` — `applyTo: '.github/agents/**/*.agent.md'`
- `pe-agents-strict.instructions.md` — `applyTo: '.github/agents/**/*.agent.md'`

Both match ALL agent files. If the first says "tool count 3–7" and the second says "tool count 2–5", the agent receives both rules simultaneously. The LLM cannot determine which is authoritative. The result: unpredictable behavior that varies by model, context window position, and prompt structure.

**This problem scales with adoption.** Every repository that installs the PE engine gets a set of instruction files. If the repository already has its own instruction files for the same file types, collisions are guaranteed.

### Why other artifact types don't have this problem

| Artifact type | Injection mechanism | Conflict risk |
|---|---|---|
| **Instruction files** | Auto-inject by glob pattern — **no opt-in, no precedence** | 🔴 HIGH — silent collision |
| **Context files** | Loaded by semantic search or explicit `📖` reference — **consumer chooses** | 🟢 LOW — explicit selection |
| **Agent bodies** | Rules embedded in agent — **scoped to that agent's invocations** | 🟢 LOW — no cross-agent injection |
| **Prompt bodies** | Rules in prompt workflow — **scoped to that prompt's execution** | 🟢 LOW — no cross-prompt injection |
| **Skills** | Loaded by AI matching or explicit read — **consumer chooses** | 🟢 LOW — explicit selection |

The difference is **opt-in vs. auto-inject**. Context files, agent bodies, and skills are consumed by choice. Instruction files are consumed by pattern match — whether the consumer wants them or not.

### The causal chain

```
Multiple PE systems or repo-local instruction files coexist
  → instruction files with overlapping applyTo patterns
    → silent rule collisions (no precedence mechanism)
      → LLM receives contradictory directives
        → unpredictable behavior per invocation
          → undermines deterministic classification (Tier 1+2 become unreliable)
            → autonomy gradient collapses (can't trust rule application)
```

### The solution: instruction minimization

**Principle:** Instruction files SHOULD contain only **testable, mechanical rules** — rules that can be evaluated with a boolean pass/fail without LLM judgment. Behavioral, strategic, and interpretation-dependent rules belong in context files or agent bodies where consumers explicitly choose what to load.

**Rule classification:**

| Rule type | Definition | Example | Belongs in |
|---|---|---|---|
| **Testable/mechanical** | Boolean pass/fail, no judgment needed | "YAML MUST have `goal:` field", "Token budget ≤1,500", "File MUST be in `.github/instructions/` root" | ✅ Instruction file |
| **Behavioral/strategic** | Requires interpretation or judgment | "Write in a warm, conversational tone", "Use appropriate level of detail", "Follow Diátaxis patterns" | ❌ Context file or agent body |
| **Governance/meta** | System-level enforcement rules | "applyTo MUST NOT overlap", "Cascade validation: if context newer, flag stale" | ✅ Instruction file |

**Why this works:** When instruction files contain only testable rules, collisions are either:
1. **Identical** (both say "YAML must have `goal:` field") → no conflict
2. **Contradictory on a measurable dimension** (one says "≤1,500 tokens", other says "≤2,000 tokens") → detectable and resolvable by choosing the stricter rule
3. **Non-overlapping** (different mechanical checks) → additive, not conflicting

Behavioral rules, by contrast, are inherently ambiguous when doubled — "warm tone" from one system plus "professional tone" from another creates irreconcilable contradiction.

### Proposed vision changes (priority-ordered)

#### Change 1 (HIGH): Add instruction minimization as a design principle

**Where:** Design principles section, after "Portable by design"

**What:** Add a new principle:

> ### Instruction minimization
>
> *Relies on: R-S7-portable-packaging, R-P1-deterministic-processing*
>
> Instruction files auto-inject via `applyTo` patterns with no precedence mechanism. When multiple PE systems coexist, overlapping patterns cause silent rule collisions. To prevent this:
>
> **Rule:** Instruction files MUST contain only testable, mechanical rules — rules evaluable with a boolean pass/fail without LLM judgment. Behavioral, strategic, and interpretation-dependent rules MUST be placed in context files or agent bodies where consumers explicitly choose what to load.
>
> **Rationale:** Testable rules either agree (identical checks), contradict on a measurable dimension (resolvable by choosing the stricter value), or are additive (non-overlapping checks). Behavioral rules are inherently irreconcilable when doubled — "warm tone" plus "professional tone" has no deterministic resolution.
>
> **Example:** ✅ "YAML MUST have `goal:` field" (testable). ❌ "Write in a warm, conversational tone" (behavioral — belongs in context file).

**Why this is the highest priority:** Without this principle, the vision's portability promise (R-S7) creates a conflict vector that undermines the deterministic classification the entire autonomy gradient depends on.

### Integration notes

- **No content removed.** All changes are additions.
- **New rationale ID R-S8** follows the existing numbering (R-S1 through R-S7).
- **Instruction minimization is orthogonal to N-1.** N-1 addresses content-level classification within artifacts. Instruction minimization addresses cross-artifact collision at the injection layer. Both are prerequisites for reliable autonomous execution.
- **Existing instruction files may need refactoring.** The `article-writing.instructions.md` file contains significant behavioral content (Microsoft Voice Principles, Diátaxis framework, emoji H2 prefixes) that should migrate to context files per this principle. This refactoring is a separate improvement plan action, not a vision change.
