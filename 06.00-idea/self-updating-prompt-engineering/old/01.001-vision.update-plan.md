## 🔍 Scope and method

This plan targets [01.000-vision.v1.md](01.000-vision.v1.md). It's triggered by a single high-value observation: the current breaking/non-breaking classification model is **one-dimensional**, and this limits the autonomy the system can earn for additive changes.

The plan analyzes the gap, defines the new concepts it introduces, identifies which sections of the vision are affected, and produces a prioritized set of changes.

It does **not** re-review findings already addressed in [00.002-vision-update-plan2.md](00.002-vision-update-plan2.md). It focuses exclusively on the staged integration strategy and its implications.

## 🎯 Core insight

### The observation

The current impact classification table measures a single dimension: **goal/scope/boundary preservation**. A change categorized as "Preserved" (non-breaking structurally) receives a single impact rating and is routed to a single autonomy level.

But a **single change can have heterogeneous impact across multiple quality dimensions**. Consider: "deeper coverage within declared scope."

- **Structural dimension** (goal/scope/boundary): Preserved → Low
- **Consistency dimension** (non-contradiction with existing rules): May be breaking if the new coverage changes existing priorities, introduces new rules/workflows, or contradicts existing guidance
- **Efficiency dimension** (token consumption, processing overhead): May be breaking if the new content increases the token cost of every artifact that loads this one, risking token budget violations or context rot

Under the current model, this change receives a single "Low-medium" rating that merges all three dimensions. The result is either:

- **Over-blocking**: A safe addition (consistency and efficiency are fine) is escalated to human approval because the merged rating is "Low-medium" with low confidence.
- **Under-blocking**: An addition that changes rule priorities or increases token budgets is approved autonomously because the merged structural rating is "Preserved."

### The proposed strategy

**Stage 1 — Non-regression gate**: Before integrating any additive change, run targeted validation on the dimensions that can produce silent regression:

1. **Consistency check** — Does the new content contradict existing rules? Does it change priorities of existing rules? Does it introduce workflows that conflict with existing workflows?
2. **Efficiency check** — Does the token delta stay within the artifact's declared token budget? Does the processing overhead remain within pipeline tolerance?

If Stage 1 passes → the change is **proven non-regressive** and can be integrated autonomously. The integration doesn't break existing artifacts or workflows.

**Stage 2 — Optimization phase**: After Stage 1 integration, schedule a separate step to identify whether the new information reveals redesign opportunities:

1. Are existing rules now improvable using the new coverage?
2. Are existing workflows now restructurable for better efficiency or effectiveness?
3. Does the new information supersede or make redundant existing guidance?

Stage 2 operates at a higher autonomy level (restructuring, rule modification) and is scheduled as a distinct work item — not blocked by Stage 1.

### Why this unlocks higher autonomy

Under the current model, a change with mixed-dimension impact waits for the highest-risk dimension to be cleared before any integration proceeds.

Under staged integration, dimensions are processed independently:

| Dimension | Stage 1 gate | Autonomy if gate passes |
|---|---|---|
| Structural (goal/scope/boundary) | Metadata comparison | Autonomous |
| Consistency (non-contradiction) | Rule-set comparison | Autonomous with notification |
| Efficiency (token delta) | Token count deterministic check | Autonomous |
| Effectiveness (output quality) | Not a Stage 1 gate — see note | Deferred to Stage 2 or observation |

> **Note on effectiveness**: Effectiveness can't be validated deterministically before integration (it's only measurable through actual use — the effectiveness hard limit, already in the vision). It's not a Stage 1 gate. Stage 2 captures effectiveness improvement opportunities through the optimization phase; post-integration observation captures regression signals through the outcome log.

The result: **more changes proceed autonomously in Stage 1** without relaxing the non-regression requirement. Stage 2 captures the full optimization value, scheduled separately at appropriate autonomy level.

## 📊 Gap analysis

### G1: The impact classification is one-dimensional (HIGH)

**Current state.** The breaking/non-breaking classification table maps change category → impact level using a single dimension: goal/scope/boundary impact.

```
| Deeper coverage within declared scope | Preserved | Low-medium |
```

**Gap.** "Preserved" refers only to the structural dimension. The same change can be breaking on consistency (contradictions, priority shifts) or efficiency (token budget) while structural is Preserved.

**Evidence.** A context file addition that introduces a new rule (deeper coverage, declared scope) can simultaneously:
- Preserve the file's stated goal and scope → structural: non-breaking
- Change priorities of existing rules by proximity effects → consistency: potentially breaking
- Add 200 tokens to every agent that loads the file → efficiency: breaking if the agent's token budget is 500 and the file now consumes 600

**Impact.** The merged "Low-medium" rating merges three independent risk signals. This produces classification errors in both directions: safe changes over-escalated, risky changes under-escalated.

**Proposed change.** After the classification table in "Breaking vs. non-breaking classification," add:

> **Impact is multi-dimensional.** Goal/scope/boundary preservation describes the structural dimension only. A change can be non-breaking structurally while having impact on **consistency** (does it contradict or reprioritize existing rules?) and **efficiency** (does it increase token consumption beyond declared budgets?). Each dimension is assessed independently. The autonomy level for a change is determined by the *highest-risk* dimension — unless staged integration applies (see Execute → Staged integration pattern).

### G2: No non-regression gate concept (HIGH)

**Current state.** The Execute layer has a risk-reduction techniques table that includes "Static validation" (schema validity, reference resolution, boundary preservation) and "Redundant processing" (LLM passes). There's no named concept for the non-regression precondition specifically for additive changes.

**Gap.** The validation described by "Static validation" focuses on structural integrity. It doesn't include:
- Consistency check: do the new rules/workflows contradict existing ones?
- Priority-shift check: do new additions implicitly demote existing rules?
- Token delta check: does the added content stay within the artifact's declared token budget?

**Proposed change.** Add "Non-regression gate" to the risk-reduction techniques table in Execute:

| Technique | What it does |
|---|---|
| **Non-regression gate** | For additive changes: run consistency check (no contradictions introduced, no priority shifts), efficiency check (token delta within budget), and structural check. Must pass before autonomous integration of any content addition. |

Add a note: Non-regression gate failure converts a non-breaking structural classification to a human-approval routing — the change is additive but not provably non-regressive.

---

### G3: Staged integration pattern not formalized (HIGH)

**Current state.** The "Decomposition-first" principle and the Execute layer reference decomposing large changes into steps. This applies to change-sets spanning many artifacts. There's no formalized pattern for decomposing a *single change* that has heterogeneous impact across quality dimensions.

**Gap.** A single additive change to one artifact (e.g., adding a new sub-section to a context file) can have:
- Low-impact structural dimension (Stage 1 eligible)
- Medium-impact consistency dimension (Stage 1 gate — validate before proceeding)
- Deferred optimization dimension (Stage 2, scheduled separately)

This is not a "large change" requiring decompose-first for token management. It's a normal-sized change requiring staged integration for risk management.

**Proposed change.** Add a "Staged integration pattern" subsection to Execute (after the risk-reduction table):

> **Staged integration for heterogeneous-impact changes.** When a change is non-breaking on the structural dimension but has potential impact on consistency or efficiency, integrate in two stages rather than one.
>
> **Stage 1 — Non-regression integration**: Apply the content change. Run the non-regression gate. If it passes, commit autonomously with notification. The integration is proven non-regressive — existing artifacts and workflows are unaffected.
>
> **Stage 2 — Optimization phase**: Schedule a separate work item (triggered by Stage 1 completion) to identify whether the integrated information reveals redesign opportunities for existing rules or workflows. Stage 2 operates at its own autonomy level, determined by the scope of restructuring it proposes.
>
> Stage 2 is never a blocker for Stage 1. The value of adding new guidance doesn't have to wait for all optimization opportunities to be identified.

## 🔄 Stability and convergence analysis

### Framing

The self-update loop is an iterative process: Detect → Assess → Propose → Execute, repeated per trigger and per cycle. For the system to be feasible, two properties must hold at the **loop level**, not just at the per-change level:

- **Stability** — the loop doesn't oscillate, doesn't amplify errors, and returns to a good state after disturbances.
- **Convergence** — successive iterations reduce the gap between the actual artifact state and the vision's target state. Eventually, absent new external triggers, the system reaches a fixed point where no further changes are proposed.

These are emergent properties of the whole loop. Per-change safeguards (R-L2-self-correction validation, rollback, R-G1-autonomy-gradient escalation) are necessary but not sufficient. A system with perfect per-change validation can still diverge if cycle N+1 generates more changes than cycle N, or the same changes are applied and reverted repeatedly.

### What the vision already provides

The vision has building blocks for stability, but they're framed as per-change safeguards rather than loop-level guarantees:

| Mechanism | Stability contribution | Gap |
|---|---|---|
| R-L2-self-correction (multi-pass validation) | Prevents bad individual changes from being committed | Doesn't prevent valid-but-contradictory changes across cycles |
| Tier-scoped rollback + snapshots | Undoes bad changes after detection | Doesn't prevent re-application of the same change after rollback |
| R-G3-progressive-learning (threshold tuning) | Tightens autonomy when success rate drops | Is itself a feedback loop that could over-correct or under-correct |
| Human-only gate on vision/principles | Provides an immutable reference point for convergence | Only protects the reference point, not the convergence process toward it |
| Success criterion #8 (95% success rate) | Triggers threshold tightening on regression | Reactive, not preventive; no notion of convergence direction |
| Staleness avoidance hard gate | Ensures meta-infrastructure is fresh | Doesn't address whether the loop's behavior is converging |

### What's missing: loop-level gaps

#### G7: No convergence criterion or fixed-point definition (HIGH)

**Current state.** The vision defines success criteria for the system's outputs (drift detection time, correction quality, false positive rate) but not for the loop's convergence behavior. There's no definition of what "converged" means — no way to distinguish "the system is making progress toward the vision" from "the system is churning."

**Gap.** Without a convergence criterion, the system can't know when it's done (for a given vision state) or whether it's moving in the right direction. The rollout phases define what gets autonomy when, but not how iterations within each phase converge.

**Evidence.** The success criteria are all measured over rolling time windows (30 days, per-cycle) — they measure ongoing health, not convergence toward a target state. There's no metric for "distance from vision" or "change volume trend."

**Proposed change.** ~~(Original: change-volume monotonic decrease — superseded by G40's distance-based model.)~~ Add a "Convergent iteration" design principle and a convergence criterion to the success criteria:

> **Convergence criterion (distance-based — G40).** The system is converging when the per-artifact distance between content and declared metadata (goal, scope, boundaries, rationales, implements) decreases across consecutive cycles on all measured dimensions. A cycle that increases distance on any artifact's measured dimension, absent new external triggers, is a **convergence concern** — the system runs a priority assessment for the specific artifact and dimension, not a system-wide pause.
>
> **Fixed-point definition.** The system has converged (for the current vision state and external signal set) when all artifacts report zero distance on all measured dimensions (excluding effectiveness, which requires observation data). New external triggers restart the convergence process.
>
> **Measurement dimensions:** Goal alignment (LLM), scope coverage (semi-deterministic), boundary compliance (deterministic), rationale integrity (semi-deterministic), implementation fidelity (LLM). Caching via three-hash key (content hash + metadata hash + vision hash) — reuses R-S3-validation-caching.

#### G8: No oscillation detection (HIGH)

**Current state.** The outcome log records changes and their outcomes, but nothing analyzes the log for oscillation patterns — the same artifact being changed and reverted (or changed in contradictory directions) across consecutive cycles.

**Gap.** Oscillation is the most common instability pattern in feedback systems. Without detection, the system can loop indefinitely: detect drift → apply fix → regression detected → rollback → detect the original drift again → re-apply fix → loop.

**Evidence.** The rollback mechanism (tier-scoped snapshots) handles the "undo" case, but doesn't prevent re-triggering. R-G3-progressive-learning's outcome log has the data needed for oscillation detection, but the vision doesn't specify this analysis.

**Proposed change.** ~~(Original: outcome-log apply/rollback pattern matching — superseded by G43's distance-vector approach.)~~ Add an oscillation detection mechanism to the convergence model:

> **Oscillation detection (distance-based — G43).** For each artifact on each measured dimension, the system tracks distance at cycle boundaries. Oscillation is detected when the sign of the distance delta (Δ = d(t+1) − d(t)) alternates for k or more consecutive cycles (configurable, default: k=3 — requires at least three sign changes to distinguish oscillation from noise).
>
> When oscillation is detected: the artifact and dimension are flagged as **oscillating**. The system freezes autonomous changes for the oscillating dimension, logs the distance history, and escalates to human review. The human receives the distance vector (showing the pattern), the dimension involved, and the content changes that produced the oscillation.
>
> **Stall detection (G45).** If |Δ| < ε (configurable threshold) for m consecutive cycles, the dimension is **stalled** — the system is running but making no progress. Stall triggers priority assessment: is the artifact near convergence (expected stall at distance ≈ 0), or does the stall indicate conflicting constraints or missing metadata?
>
> The outcome log (R-G3-progressive-learning) retains its role for audit trail, progressive learning, and effectiveness signal — but oscillation *detection* is based on distance measurements, not log parsing.

#### G9: No iteration budget or stopping criterion (MEDIUM)

**Current state.** A single self-update cycle has no explicit bound on how many changes it can propose or execute. The cost-of-vigilance risk mentions token consumption but doesn't set a per-cycle cap.

**Gap.** Without a budget, a single cycle triggered by a major platform release could propose dozens of changes, each consuming tokens for validation, each potentially triggering dependent checks. The cost and risk scale linearly (or worse) with change count.

**Proposed change.** Add an iteration budget to the Execute layer:

> **Iteration budget.** Each self-update cycle has a maximum number of autonomous changes (configurable, default: 10). Changes beyond the budget are queued for the next cycle, prioritized by tier and severity. The budget prevents unbounded token consumption and limits blast radius per cycle.

#### G10: No cascade damping (MEDIUM)

**Current state.** The dependency graph (R-S2-dependency-graph) maps how changes propagate across tiers. The vision doesn't distinguish between externally-triggered changes and changes triggered by other changes (cascades).

**Gap.** A change at tier 1 could trigger validation checks at tiers 2–5, which could each propose their own changes, which trigger further checks. Without damping, change volume grows with each cascade level. This is an amplification risk.

**Proposed change.** Add a cascade damping rule:

> **Cascade damping.** Changes triggered by other changes (not by external signals) run at one autonomy level higher than the triggering change's classification would normally require. A cascade of depth > 2 (change triggers change triggers change) is a **complexity signal** — the system logs a warning and defers further cascade execution to the next cycle.

#### G11: Bootstrapping from existing artifacts not addressed (MEDIUM)

**Current state.** The rollout phases define what artifact types get autonomous execution and when, based on preconditions (validators, outcome log, semantic metadata, chain alignment). They're structured as a progression of capabilities to be built.

**Gap.** The vision doesn't acknowledge that autonomous execution can start from existing artifacts. Many building blocks already exist — agents, MCP servers, validation tools, prompts. The implementation path isn't "build everything then start" — it's "start from what exists and iterate toward the vision." The rollout phases describe the destination but not the journey.

**Evidence.** Phase 1 says "None — validators already exist" as a precondition, implicitly acknowledging existing infrastructure. But the bootstrapping process (how the self-update loop itself converges from an initial state to its target operating state) isn't formalized.

**Proposed change.** Add a bootstrapping note to the rollout phases section:

> **Bootstrapping strategy.** The self-update system doesn't start from zero. It starts from the current artifact set and existing tooling (agents, MCP servers, validators, prompts). Each rollout phase adds autonomous capabilities incrementally. The convergence criterion (G7) applies from the first iteration: each cycle should leave the system closer to the vision than the previous one. The rollout phases define what autonomy the system earns; the convergence criterion defines whether it's making progress.

#### G12: Progressive learning threshold drift (LOW)

**Current state.** R-G3-progressive-learning tunes autonomy thresholds based on the outcome log. The vision says "start with conservative thresholds and only loosen as outcome data accumulates" (Risks: Insufficient outcome data).

**Gap.** Threshold tuning is itself a feedback loop. If early autonomous changes are disproportionately easy (Phase 1 covers templates and hooks), the success rate will be artificially high, potentially loosening thresholds before the system encounters harder changes (Phase 3 context files, Phase 4 cross-tier refactors).

**Proposed change.** Add a threshold stability clause to R-G3-progressive-learning:

> **Threshold stability.** Threshold adjustments require a minimum outcome volume per rollout phase, not just globally. Phase 1 outcomes don't count toward Phase 3 threshold loosening. Each phase maintains its own success rate history. Threshold loosening requires evidence from the phase it applies to.

### Proposed new rationale: R-G5-convergent-iteration

| ID | Rationale | Design implication |
|---|---|---|
| **R-G5-convergent-iteration** | **Convergent iteration discipline.** The self-update loop is a feedback system. Like all feedback systems, it must be stable (no oscillation, no amplification) and convergent (each cycle moves closer to the vision's target state). Per-change safeguards (validation, rollback, escalation) are necessary but not sufficient — the loop's emergent behavior must also be governed. | Monitor change volume per cycle. Detect oscillation via outcome log analysis. Apply cascade damping to internally-triggered changes. Define a convergence criterion (decreasing change volume) and a fixed-point (zero proposed changes). Budget iterations to bound cost and blast radius. Escalate divergence signals to human review. |

### Relationship to existing rationales

| Existing rationale | Relationship to convergent iteration |
|---|---|
| R-G3-progressive-learning | R-G5-convergent-iteration governs the loop that R-G3-progressive-learning tunes. G12's threshold stability clause ensures R-G3 doesn't destabilize the loop. |
| R-L2-self-correction | Per-change validation is a component of loop stability — necessary but not sufficient. R-G5-convergent-iteration adds the loop-level properties that R-L2-self-correction can't provide alone. |
| R-S2-dependency-graph | The dependency graph enables cascade analysis. R-G5-convergent-iteration adds cascade damping as a governance rule on top of the structural information R-S2-dependency-graph provides. |
| R-G1-autonomy-gradient | The autonomy gradient operates per-change. R-G5-convergent-iteration adds a per-cycle dimension: when the loop diverges, autonomy tightens system-wide, not just per-change. |
| R-P2-decomposition | Decomposition bounds per-step complexity. R-G5-convergent-iteration bounds per-cycle complexity via iteration budgets. |

## � Trust-based adoption analysis

### Framing

The Trust-based adoption design principle defines three trust tiers and routes ecosystem work accordingly. The Propose section defines two response strategies (adopt, incorporate). R-G2-trust-calibrated-adoption grounds both in the rationale framework.

The concern: the current rule for unknown sources — "don't trigger adoption or incorporation proposals autonomously" — creates a dead end where valuable ideas are silently lost. This analysis examines what's missing and proposes a strategy that captures value from unknown sources without creating maintenance risk.

### What the vision currently provides

| Trust tier | Current response | Dependency created |
|---|---|---|
| **Trustworthy** | Adopt (replace owned artifact) | Full — owned artifact deferred to external |
| **Valuable but unvetted** | Incorporate (absorb ideas) | Soft — ideas absorbed with attribution |
| **Unknown** | No autonomous processing | None — but no value captured either |

The Propose section's two strategies both assume some degree of trust:

- **Adopt** requires "well-defined scope, credible roadmap, trustworthy source"
- **Incorporate** requires "valuable ideas but doesn't meet the adoption bar"

Neither strategy addresses what to do when a source has zero trust history but contains genuinely valuable insight.

### What's missing: trust-tier gaps

#### G13: Unknown-source value loss (HIGH)

**Current state.** The Trust-based adoption principle says: "Unknown sources — don't trigger adoption or incorporation proposals autonomously." This means the system doesn't analyze unknown-source material at all during autonomous operation.

**Gap.** Unknown sources can contain original ideas not yet explored by the established community. Some of the most valuable innovations come from individuals and small projects without brand recognition. The current rule optimizes for safety (no bad integrations) but sacrifices discovery (no good ideas captured either).

**The real risk isn't the ideas — it's the dependency.** Adopt creates a full dependency (if the external artifact disappears, you lose the capability). Incorporate creates a soft dependency (the understanding of what was incorporated traces back to a source that may vanish). For unknown sources, both strategies carry unacceptable maintenance risk because there's no track record to predict continued availability.

**What's needed is a third response strategy that creates zero dependency.**

**Proposed change.** Add an **Extract** strategy to the Propose section and update the Trust-based adoption principle:

> **Extract.** When analysis identifies a valuable idea from an unknown or low-trust source, the system extracts the core insight and encodes it as fully owned content — validated against the artifact's goal and boundaries. No reference to the external source is maintained. No dependency is created. The idea is internalized, not adopted or incorporated.
>
> Extract differs from incorporate in a critical way: incorporate absorbs ideas "after evaluation and adaptation" but implicitly traces back to the source. Extract severs that connection entirely. The owned artifact stands on its own — if the external source vanishes, nothing is affected because nothing references it.

Rewrite the Trust-based adoption principle:

> ### Trust-based adoption
>
> *Relies on: R-G2-trust-calibrated-adoption*
>
> Not all external work deserves equal trust. Trust determines the **response strategy** — not whether the work is analyzed.
>
> **Analysis prioritization by trust tier:**
>
> - **Trustworthy sources** — analyzed first. Platform and ecosystem triggers from these sources have the shortest assessment latency.
> - **Valuable but unvetted sources** — analyzed at normal cadence. Ecosystem triggers from these sources follow the standard assessment queue.
> - **Unknown sources** — analyzed when capacity allows, or on explicit human request. Lower analysis priority, but never skipped entirely for signals that overlap with owned artifact scope.
>
> **Response strategy by trust tier:**
>
> - **Trustworthy sources** (Microsoft, OpenAI, Anthropic, Google, Meta, recognized domain specialists) — **adopt** directly when scope and quality align. The system proposes replacing or deferring to the external artifact. Adoption decisions always surface to humans.
> - **Valuable but unvetted sources** — **incorporate** ideas into owned artifacts after evaluation and adaptation. Incorporation can proceed autonomously if validated (R-L2-self-correction).
> - **Unknown sources** — **extract** valuable ideas and encode as fully owned content. Direct adoption is not available. Direct incorporation is discouraged (creates a soft dependency on an unproven source). Extraction requires elevated validation rigor: the idea must be validated against the artifact's goal and boundaries with at least two independent passes (R-L2-self-correction), and the resulting owned content must pass the non-regression gate (G2) before integration. Extraction always requires human approval.
>
> **Trust graduation:** Unknown sources that repeatedly produce ideas the system successfully extracts are candidates for trust promotion. The outcome log tracks extraction events by source. When a source accumulates N successful extractions (configurable, default: 3) without regressions, the system proposes promoting it to "valuable but unvetted." Trust promotion is a human-approval decision.
>
> Trust is earned through consistency: clear scope, maintained roadmap, demonstrated expertise — or, for unknown sources, through a track record of extracted value.

#### G14: Missing "Extract" response strategy in Propose (HIGH)

**Current state.** The Propose section defines two strategies: adopt and incorporate. Both assume some trust in the source and both create dependencies (full or soft).

**Gap.** There's no strategy for capturing value without creating dependency. When an unknown source contains a valuable approach, the only options today are "adopt" (unacceptable — full dependency on unproven source), "incorporate" (risky — soft dependency), or "ignore" (value lost).

**Proposed change.** Add a third strategy to the Propose section:

> - **Extract.** When the external work contains a valuable idea but the source doesn't meet the incorporation bar (unknown trust, no maintenance track record, uncertain longevity). The system proposes extracting the core insight and encoding it as fully owned content — validated against the artifact's goal and boundaries with elevated rigor. No dependency on the external source is created. The owned artifact is self-standing.
>
> Extract requires human approval. The system surfaces: the source, the identified idea, the proposed owned encoding, and the validation results. The human decides whether the extraction is accurate and whether the owned encoding captures the value without distortion.

Update the Propose section's trust-response mapping:

| Source trust | Strategy | Dependency | Approval |
|---|---|---|---|
| Trustworthy | Adopt | Full | Human approval |
| Valuable but unvetted | Incorporate | Soft | Autonomous if validated |
| Unknown | Extract | None | Human approval |

#### G15: No trust graduation mechanism (MEDIUM)

**Current state.** Trust tiers are static. The principle says "trust is earned through consistency: clear scope, maintained roadmap, demonstrated expertise." But there's no mechanism for the system to track or propose trust changes over time.

**Gap.** A source that starts as "unknown" but repeatedly produces ideas the system successfully extracts should be promotable. Without graduation, the system permanently treats every unknown source with maximum friction, even after repeated positive outcomes.

**Proposed change.** Add trust graduation to R-G2-trust-calibrated-adoption's design implication:

> Track extraction and incorporation events per source in the outcome log. When a source accumulates sufficient successful events without regressions, propose trust promotion (unknown → valuable but unvetted → trustworthy). Trust promotion is always a human-approval decision. Trust demotion (regression or source degradation) follows the same path in reverse.

#### G16: No analysis prioritization by trust tier (LOW)

**Current state.** All ecosystem triggers are processed equally regardless of source trust. A Microsoft documentation update and an unknown blog post get the same analysis priority.

**Gap.** Analysis capacity is bounded (token budget, cycle time). Without prioritization, the system may spend analysis capacity on low-trust sources while high-trust signals queue. This is inefficient but not dangerous (no bad changes result — just delayed good ones).

**Proposed change.** Add analysis prioritization to the Detect or Assess layer:

> **Analysis prioritization.** Ecosystem triggers are prioritized by source trust tier. Trustworthy sources are assessed first. Unknown sources are assessed when remaining capacity allows within the cycle's token budget, or when a human explicitly requests analysis. No source is permanently excluded from analysis.

### Relationship to existing mechanisms

| Mechanism | Relationship to trust-tier changes |
|---|---|
| R-G2-trust-calibrated-adoption | Extended with "Extract" strategy and trust graduation. The core rationale (calibrate response to trust) is unchanged — the calibration now has a third setting. |
| R-G1-autonomy-gradient | Extract is always human-approval (consistent with the gradient: unknown source + idea extraction = high uncertainty). |
| R-L2-self-correction | Extract requires elevated validation — at least two independent passes, not just one. This is a stricter application of R-L2, not a new mechanism. |
| R-G3-progressive-learning | Extraction outcomes feed the outcome log like any other change. Trust graduation uses the same log for evidence. |
| Staged integration (G3) | Extract is compatible with staged integration. Stage 1 = validate the extraction is non-regressive. Stage 2 = identify optimization opportunities from the extracted insight. |
| Convergent iteration (G7) | Extractions are changes like any other — counted in the convergence criterion, subject to the iteration budget. |

## 📐 Metadata rationale analysis

### Framing

The Metadata-driven design principle currently specifies four metadata attributes: `goal:`, `scope:`, `boundaries:`, and `version:`. These declare the artifact's **contract** — what it's for, what it covers, what constrains it, and how it's versioned. The self-update system uses this contract to make deterministic breaking/non-breaking classifications.

The concern: when the system extends or re-validates an artifact, it can check whether goal, scope, and boundaries are preserved — but it can't check whether the **design decisions that shaped the artifact's content** are respected. An extension that preserves the contract but violates a non-obvious design constraint passes all current checks yet produces a regression.

This analysis examines whether adding `rationales:` to artifact metadata strengthens the self-update loop's stability and convergence, and how to manage the relationship between metadata-level and in-body declarations.

### What the vision currently provides

| Metadata attribute | What it enables | What it can't detect |
|---|---|---|
| `goal:` | "Did this change alter the artifact's purpose?" | "Did this change undermine a design decision that serves the purpose?" |
| `scope: {covers, excludes}` | "Did this change add or remove topics?" | "Did this change violate an ordering or priority constraint within covered topics?" |
| `boundaries:` | "Did this change exceed token limits or single-source-of-truth declarations?" | "Did this change weaken a constraint that exists for a non-obvious reason?" |
| `version:` | "Is this patch, minor, or major?" | "Should this change be classified higher because it violates an undeclared constraint?" |

The in-body content contains rationales (per the N-1 separability principle), but they're human-readable prose. The self-update system can't consume them deterministically — it needs LLM judgment to extract and evaluate them, which lowers confidence and reduces autonomy eligibility.

### The three layers (contract, rationales, content)

The analysis reveals three distinct layers, each with a different consumer, granularity, and update cadence:

| Layer | What it captures | Format | Consumer | Update trigger |
|---|---|---|---|---|
| **Contract metadata** (`goal`, `scope`, `boundaries`, `version`) | The artifact's external promise — what it delivers and within what constraints | Machine-readable YAML | Self-update system (deterministic comparison) | When the artifact's promise changes (breaking) |
| **Rationale metadata** (`rationales:`) | The artifact's design constraints — why specific content choices were made | Machine-readable YAML (ID + constraint + impact) | Self-update system (constraint validation during extension/re-validation) | When a design constraint is added, invalidated, or discovered to be missing |
| **In-body content** (rules, rationales, examples) | The full human-readable articulation | Prose (Markdown) | Humans and LLMs consuming the artifact at runtime | When content improves, extends, or adapts |

**These layers are NOT redundant.** They serve different consumers at different granularities:

- **Contract metadata** answers: "Is this change breaking?" (deterministic, O(1))
- **Rationale metadata** answers: "Does this change respect the design constraints?" (deterministic, O(n) where n = rationale count)
- **In-body rationales** answer: "Why does this rule exist?" (human-readable, consumed at runtime by LLMs and authors)

### What rationale metadata looks like

```yaml
goal: "Validate PE artifact quality across 7 dimensions"
scope:
  covers: [structure, references, coherence, readability, ...]
  excludes: [deployment, runtime behavior]
boundaries:
  max_tokens: 800
  single_source_of_truth_for: validation-criteria
version: "1.3.0"
rationales:
  - id: "ordering-dependency"
    constraint: "Structural validation must complete before reference or coherence checks"
    impact_if_violated: "False positives in reference/coherence checks"
  - id: "token-budget-priority"
    constraint: "Rules are mandatory; rationales and examples are load-bearing only when misapplication risk is high"
    impact_if_violated: "Consumer context windows overflow, accuracy degrades"
    relies_on: "R-P2-decomposition"
```

Each rationale entry is minimal: an ID, a constraint statement (one sentence), and the impact if violated. Optional: `relies_on` linking to the vision-level rationale that grounds the constraint.

**The `reason:` field is deliberately omitted from metadata.** The full "why" explanation lives in the in-body rationale (per N-1 separation). Metadata captures *what* must be preserved and *what breaks if it isn't* — the minimum the self-update system needs for deterministic constraint checking. The full explanation is for humans.

### Metadata vs. in-body: abstraction levels and update criteria

#### Abstraction level

| Level | What it captures | Example |
|---|---|---|
| **Metadata rationale** (artifact-level) | Design constraints that govern the artifact as a whole — decisions that affect multiple rules or the artifact's overall behavior | "Structural validation before reference checks" — governs 3+ rules' execution order |
| **In-body rationale** (per-rule) | Why a specific rule exists — the justification for an individual enforceable statement | "Why the max H3 heading depth rule exists" — local to one rule |

The test: **if violating the constraint could cause regression across multiple rules or consumers, it's artifact-level (metadata). If it only affects one rule's application, it's per-rule (in-body only).**

#### Update criteria

| Attribute | Update trigger | Impact classification | Approval |
|---|---|---|---|
| **In-body rationale** | Rule changes, better explanation, new evidence | Non-breaking (per N-1) | Autonomous |
| **Metadata rationale added** | New design constraint discovered during extension or review | Medium — expands the constraint surface future changes must respect | Autonomous with notification |
| **Metadata rationale removed** | Constraint found to be unnecessary or invalidated by platform change | Medium-high — loosens the constraint surface, may enable changes that were previously blocked | Human approval |
| **Metadata rationale modified** | Constraint refined based on outcome data or vision change | Medium — changes what the self-update system checks | Human approval |
| **Contract metadata changed** | Goal, scope, or boundary change | High (breaking) | Human approval |

**Key asymmetry:** Adding a rationale (tightening constraints) is lower-risk than removing one (loosening constraints). This mirrors the convergence insight — the constraint surface should grow monotonically during normal operation, with removal only via deliberate human decision.

### How rationale metadata strengthens stability and convergence

#### Direct contributions to convergence (G7)

Each cycle that adds rationale metadata to an artifact makes future cycles **more constrained** — fewer changes can pass the rationale check without detection. This is a natural convergence mechanism:

1. **Cycle 1:** Artifact has goal/scope/boundaries only. Many changes pass as "non-breaking."
2. **Cycle 5:** Artifact has goal/scope/boundaries + 3 rationale constraints. Fewer changes pass — extensions must respect ordering, priority, and token budget constraints.
3. **Cycle N:** Constraint surface stabilizes. Only genuinely non-violating changes proceed autonomously.

The constraint surface acts as an **attractor** — the system converges toward a state where the rationale metadata accurately captures all artifact-level design decisions. New constraints are discovered through review or regression; once captured, they prevent recurrence.

#### Direct contributions to oscillation prevention (G8)

Without rationale metadata: change is applied → regression detected → rollback → same change proposed again (the system doesn't know *why* it regressed).

With rationale metadata: the regression is traced to a specific violated constraint → the constraint is captured in metadata → future proposals are checked against it before execution. The oscillation is broken at the source.

#### Direct contributions to cascade damping (G10)

Rationale metadata bounds what an extension can do *before execution*. A proposed cascade that would violate a downstream artifact's declared constraints is caught at assessment time, not after propagation.

### When to populate rationale metadata

Rationale metadata doesn't need to exist from day one. The population strategy follows the rollout phases:

| Phase | Rationale metadata status |
|---|---|
| **Phase 1** (templates, snippets, hooks) | Not needed — these artifacts have trivial design constraints (schema-driven). |
| **Phase 2** (skills, agent tool lists) | Optional — add when a regression reveals an undeclared constraint. |
| **Phase 3** (agents, context files, instructions) | Recommended — these encode rules in prose; artifact-level constraints are common and non-obvious. |
| **Phase 4** (cross-tier refactors) | Required — cross-tier changes need to know what constraints exist at each level. |

**Discovery mechanism:** When a self-update cycle detects a regression that passed all contract checks (goal/scope/boundaries preserved, but output degraded), the post-mortem should identify the violated constraint and propose adding it as rationale metadata. This is an explicit learning loop: regressions that aren't caught by the contract surface trigger rationale metadata expansion.

### Gap findings

#### G17: No rationale metadata in artifact declarations (HIGH)

**Current state.** The Metadata-driven principle specifies `goal:`, `scope:`, `boundaries:`, and `version:`. The self-update system can check contract preservation but not design constraint preservation.

**Gap.** When extending an artifact, changes that preserve the contract but violate undeclared design constraints pass all deterministic checks. This forces either (a) LLM-based deep analysis for every extension (expensive, lower confidence) or (b) accepting the risk of constraint-violating changes passing (regressions caught only post-commit via the outcome log).

**Proposed change.** Add `rationales:` to the metadata specification in the Metadata-driven principle:

> Every artifact should declare:
>
> - **`goal:`** — single sentence: the one outcome the artifact exists to achieve
> - **`scope: {covers: [...], excludes: [...]}`** — what topics the artifact covers and explicitly excludes
> - **`boundaries:`** — constraints the artifact operates within (max tokens, single-source-of-truth for, etc.)
> - **`version:`** — SemVer for individual artifact versioning (patch = non-breaking fix, minor = additive, major = breaking)
> - **`rationales:`** — artifact-level design constraints the self-update system must respect during extension or re-validation. Each entry: `id` (slug), `constraint` (one-sentence statement of what must be preserved), `impact_if_violated` (what breaks). Optional: `relies_on` (vision-level rationale ID).
>
> **Contract vs. rationale metadata.** Contract attributes (`goal`, `scope`, `boundaries`) declare the artifact's external promise. Rationale attributes declare the artifact's internal design constraints. Both are machine-readable. A change that preserves the contract but violates a rationale is **not non-breaking** — it requires at minimum "autonomous with notification" classification and elevated validation.

#### G18: No rationale-aware breaking/non-breaking classification (MEDIUM)

**Current state.** The breaking/non-breaking classification table (in The goal) evaluates impact based on goal/scope/boundary preservation only. Rationale violations aren't mentioned.

**Gap.** With rationale metadata in place, the classification should account for it. A change that preserves the contract but violates a declared rationale is higher-impact than the current model recognizes.

**Proposed change.** Add a row to the breaking/non-breaking classification table:

| Change category | Goal/scope/boundary impact | Rationale impact | Impact level |
|---|---|---|---|
| Content fix, better wording, example update | Preserved | None | Low |
| Deeper coverage within declared scope | Preserved | None | Low-medium |
| Deeper coverage that violates a declared rationale | Preserved | Constraint violated | Medium-high |
| Scope extension (new topics) | Scope expands | May add constraints | Medium-high |
| Scope narrowing (removes coverage) | Scope contracts | May remove constraints | High |
| Goal or boundary change | Goal or boundaries changed | Constraints may be invalidated | High |

#### G19: No rationale metadata population strategy (MEDIUM)

**Current state.** No guidance on when to add rationale metadata — it would be a new concept.

**Gap.** Without a population strategy, rationale metadata is either added exhaustively up front (expensive, speculative) or never added (defeating its purpose).

**Proposed change.** Add a population strategy to the Metadata-driven principle:

> **Rationale metadata population.** Rationale metadata is populated incrementally, not exhaustively:
>
> 1. **Discovery through regression.** When a self-update cycle detects a regression that passed all contract checks, the post-mortem identifies the violated constraint and proposes adding it as rationale metadata.
> 2. **Discovery through review.** Human review of artifacts may identify design constraints that should be declared. These are added with "autonomous with notification" approval.
> 3. **Phase-appropriate expectations.** Phase 1–2 artifacts rarely need rationale metadata (schema-driven). Phase 3–4 artifacts should accumulate rationale metadata as constraints are discovered. Rationale metadata is never required for autonomy — its absence means lower confidence, not blocked execution.
>
> **Monotonic growth.** Under normal operation, the rationale metadata set grows (new constraints discovered) but rarely shrinks (constraints removed only by human decision when invalidated). This monotonic growth is a convergence mechanism — each added constraint makes future cycles more predictable.

#### G20: No relationship between rationale metadata and N-1 separation (LOW)

**Current state.** The N-1 principle (rules/rationales/examples separable) defines in-body rationales as non-breaking prose that explains why rules exist. The proposed rationale metadata is a different concept — artifact-level design constraints.

**Gap.** Without explicit disambiguation, authors may confuse in-body rationales (per-rule, human-readable, non-breaking) with metadata rationales (artifact-level, machine-readable, medium-impact). The two serve different purposes and follow different update rules.

**Proposed change.** Add a disambiguation note to the N-1 principle:

> **Disambiguation: in-body rationales vs. rationale metadata.** In-body rationales explain *why a specific rule exists* — they're human-readable, per-rule, and non-breaking to change. Rationale metadata declares *artifact-level design constraints* — they're machine-readable, govern multiple rules, and medium-impact to change. A change to an in-body rationale doesn't affect the metadata constraint surface. A change to rationale metadata affects what the self-update system checks during future modifications.

### Relationship to existing mechanisms

| Mechanism | Relationship to rationale metadata |
|---|---|
| R-S1-metadata-driven | Extended with `rationales:` attribute. The core principle (machine-readable metadata enables automation) is unchanged — rationale metadata is a new instance of it. |
| R-S4-role-declaration | Rationale metadata extends role declaration: not just "what the artifact does" and "why it exists" but also "what constraints shaped its content." |
| N-1 (rules/rationales/examples separable) | Rationale metadata complements N-1 — N-1 separates content types for diff classification; rationale metadata captures artifact-level constraints for extension validation. Different layers, different purposes. |
| Breaking/non-breaking classification | Extended with a rationale dimension. A change preserving the contract but violating a rationale is no longer classified as "Low." |
| Convergent iteration (G7/R-G5) | Rationale metadata is a convergence mechanism — monotonic constraint growth makes each cycle more predictable. |
| Oscillation detection (G8) | Rationale metadata breaks oscillation at the source — regressions are traced to specific constraints, captured in metadata, and checked pre-execution on future proposals. |
| Staged integration (G3/G4) | Stage 1's non-regression gate should check rationale constraints in addition to goal/scope/boundary preservation. |
| Outcome log | Regression events that trace to undeclared constraints trigger rationale metadata discovery — closing the learning loop. |

### Stability and convergence contribution summary

| Convergence property | Contribution of rationale metadata |
|---|---|
| **Monotonic progress** | Constraint surface grows over time → fewer valid autonomous changes → natural convergence toward fixed point |
| **Oscillation prevention** | Regressions traced to specific constraints → captured in metadata → blocked pre-execution on recurrence |
| **Cascade damping** | Downstream artifacts' declared constraints bound what upstream changes can propagate |
| **Convergence criterion** | Decreasing change volume + growing constraint surface = dual convergence signal |
| **Fixed-point definition** | The system has converged when: zero proposed changes AND the constraint surface hasn't grown in N cycles |

## 📦 Audit trail storage analysis

### Framing

The vision specifies two append-only JSONL files as the audit trail backbone:

- **`outcome-log.jsonl`** — every autonomous change (change ID, artifact path, tier, pre/post hashes, SemVer versions, classification, confidence, autonomy level, trigger, validations, outcome)
- **`rename-log.jsonl`** — every rule/file rename (`old → new` mapping)

Both are specified as single files at fixed paths under `.copilot/self-update/`. This analysis examines whether this design scales, identifies the consumers and their access patterns, and proposes a partitioned storage model that preserves append-only semantics while bounding query cost.

### How the audit trail is generated

| Producer | What it writes | When |
|---|---|---|
| **Execute layer** | Outcome log entry (one per autonomous change) | After every change commit (fields: change ID through validations). Outcome field updated after observation window. |
| **Human rejection** | Outcome log entry with `outcome: rejected` + rejection reason | When a human rejects an escalated proposal |
| **Rollback** | Outcome log entry with `outcome: rolled_back` | When post-execution validation detects regression |
| **Chain alignment (R-S5)** | Rename log entry (`old → new`) | When a rule ID or file is renamed |

Both logs are append-only: entries are never modified in place. The outcome log has one exception — the `outcome` field is updated after the observation window closes, but this is a targeted field update, not a full rewrite.

### How the audit trail is consumed (6 distinct consumers)

| Consumer | What it queries | Access pattern | Recency bias |
|---|---|---|---|
| **Oscillation detection (G8)** | Entries for a specific artifact over last N cycles | Filter by `artifact_path`, scan recent entries | Strong — last 3–5 cycles only |
| **R-G3 threshold tuning** | Success/failure rates over rolling 30 days | Aggregate by `autonomy_level` + `outcome` | Strong — rolling 30-day window |
| **Trust graduation (G15)** | Extraction events by source over time | Filter by `trigger_source`, count successes | Moderate — months to accumulate evidence |
| **Convergence criterion (G7)** | Change count per cycle | Count by `cycle_id` | Strong — recent cycles vs. prior cycles |
| **Success criteria (8 metrics)** | Various aggregates (detection latency, success rate, cost per trigger type) | Full-log statistics over rolling windows | Moderate — 30-day to 90-day windows |
| **Rename log lookup (Assess)** | `old → new` mapping for a specific ID | Lookup by `old_id` | Weak — any rename in history may be needed |

**Key observation:** 5 of 6 consumers need only recent data (last 30–90 days). Only the rename log lookup needs unbounded historical access. Trust graduation needs moderate history (months, not years).

### The scaling problem

**Single-file JSONL doesn't scale for multiple reasons:**

1. **Linear scan cost.** Every query must parse from line 1. At ~500 bytes/entry and ~50 entries/week, year 1 produces ~1.3 MB (~2,600 entries). Manageable. But Phase 3–4 (more artifact types eligible for autonomous processing) could produce 200+ entries/week. By year 3: ~30,000+ entries, ~15 MB. Every oscillation check, every threshold query scans the entire file.

2. **Observation window updates.** The outcome field update requires locating a specific entry in an append-only file. With a single file, this means scanning backward or maintaining an index — neither is free.

3. **Git bloat.** The file is append-only but git tracks it as a changed file every cycle. A single JSONL file that grows continuously produces large diffs. Partitioned files keep diffs small (only the current day's file changes).

4. **Rotation complexity.** Removing old data from a single JSONL file requires rewriting the entire file. Partitioned files allow folder-level deletion.

5. **Concurrent access.** If multiple self-update steps run in parallel (decomposition produces independent steps), a single file becomes a contention point. Partitioned files allow concurrent writes to different files.

### Proposed storage model: monthly folders + daily files

Inspired by the test folder pattern shown in the analysis request, the partitioned model organizes entries by time:

```
.copilot/self-update/
  outcome-log/
    current.index.jsonl     # Rolling index: last 90 days of entry summaries (ID, path, outcome, timestamp)
    202604/
      21.001.jsonl          # Day 21, sequence 001 — full entry
      21.002.jsonl          # Day 21, sequence 002 — full entry
      21.003.jsonl          # Day 21, sequence 003 — outcome update for 21.001
      28.001.jsonl          # Day 28, sequence 001
    202605/
      03.001.jsonl
      ...
    archive/
      202601-summary.jsonl  # Monthly aggregate: counts, rates, per-artifact summaries
      202602-summary.jsonl
      202603-summary.jsonl
  rename-log/
    current.jsonl           # Active rename mappings (never rotated — all renames are permanent)
  snapshots/
    <change_id>/            # Unchanged — per-change snapshot folders
  triggers/                 # Unchanged — trigger definitions
```

**File naming convention:** `DD.NNN.jsonl` where `DD` = zero-padded day of month, `NNN` = zero-padded sequence number within the day. Each file contains exactly one JSONL entry (the full change record). This enables:

- **Chronological sorting** by filename (lexicographic order = chronological order)
- **O(1) access** to a specific entry by its change ID (the change ID encodes the date + sequence)
- **Folder-level queries** — "last 30 days" means: current month folder + previous month folder
- **Folder-level rotation** — archive old months by replacing the folder with a summary file

### Three-tier lifecycle: hot, cold, archive

| Tier | Contents | Retention | Query support |
|---|---|---|---|
| **Hot** (current + previous month) | Individual entry files | Always available | Full per-entry queries — oscillation detection, threshold tuning, observation window updates |
| **Cold** (3–12 months old) | Individual entry files | Available, rotation candidate | Per-entry queries for trust graduation and trend analysis |
| **Archive** (>12 months) | Monthly summary files (aggregated statistics) | Permanent (small) | Aggregate queries only — long-term trends, system health history |

**Rotation process** (deterministic, runs at month boundary):

1. Compact months older than 12 months into `archive/YYYYMM-summary.jsonl`
2. Rebuild `current.index.jsonl` from hot + cold tier entries
3. Delete compacted month folders

**The rolling index** (`current.index.jsonl`) is a lightweight summary of the last 90 days. Each line contains: change ID, artifact path, outcome, timestamp, autonomy level. This enables fast aggregate queries without scanning individual entry files. It's rebuilt deterministically from the entry files — never the source of truth, always derivable.

### Rename log: different strategy

The rename log has fundamentally different access patterns — any rename in history may be needed (a consumer may reference an ID renamed months ago). This means:

- **No rotation.** All rename mappings are permanent.
- **Single file is acceptable** — rename events are rare (estimated: <10/month). A single `current.jsonl` file with all mappings stays small indefinitely.
- **If it grows unexpectedly:** split into `active.jsonl` (mappings still referenced by at least one consumer) and `historical.jsonl` (mappings where all consumers have been updated). But this optimization isn't needed until the rename count exceeds ~500.

### Gap findings

#### G21: Single-file audit trail doesn't scale (HIGH)

**Current state.** The outcome log is a single append-only JSONL file at `.copilot/self-update/outcome-log.jsonl`. Every consumer scans the entire file. No rotation, no partitioning, no indexing.

**Gap.** As the system matures (Phase 2–4), entry volume grows with the number of artifact types eligible for autonomous processing. Linear scan cost, git bloat, observation window update complexity, and rotation difficulty all increase with file size. The rename log is less affected (low volume) but the outcome log will become a performance bottleneck.

**Proposed change.** Replace the single-file outcome log with a partitioned storage model:

> **Outcome log storage.** The outcome log uses a partitioned storage model: monthly folders containing individual daily entry files. File naming: `DD.NNN.jsonl` (day + sequence). A rolling index (`current.index.jsonl`) provides fast aggregate queries over the last 90 days without scanning individual files.
>
> Three-tier lifecycle: **hot** (current + previous month, full per-entry access), **cold** (3–12 months, per-entry access for trust graduation and trend analysis), **archive** (>12 months, compacted to monthly summary files). Rotation runs deterministically at month boundaries.
>
> The rename log remains a single file (`rename-log/current.jsonl`) — rename events are infrequent and all mappings are permanent.

Update the infrastructure folder layout accordingly.

#### G22: No observation window update mechanism (MEDIUM)

**Current state.** The vision says the `outcome` field is "updated after observation window" but doesn't specify how. In a single JSONL file, this requires locating and modifying a specific line — incompatible with append-only semantics.

**Gap.** Without a defined mechanism, observation window updates either break append-only guarantees (in-place modification) or require workarounds (appending an "update" entry that references the original). The partitioned model resolves this naturally — an outcome update is a new file in the same month folder, referencing the original change ID.

**Proposed change.** Add observation window update mechanism:

> **Observation window updates.** When the observation window for a change closes, the system writes a new entry file in the same month folder with `type: outcome_update`, referencing the original `change_id`. The rolling index reflects the latest outcome. Consumers resolve the final outcome by checking for update entries — or by reading the rolling index, which always reflects the most recent state.

#### G23: No change ID encoding convention (LOW)

**Current state.** The vision mentions "change ID" as a field in outcome log entries but doesn't define its format.

**Gap.** The partitioned storage model requires change IDs that encode temporal information (for file naming and O(1) lookup). Without a convention, the mapping between change IDs and file paths is arbitrary.

**Proposed change.** Add change ID convention:

> **Change ID format:** `YYYYMMDD.NNN` — date + zero-padded sequence number. The change ID doubles as the entry filename (`DD.NNN.jsonl` within the `YYYYMM/` folder). This enables O(1) lookup by change ID and chronological sorting by filename.

### Relationship to existing mechanisms

| Mechanism | Relationship to partitioned storage |
|---|---|
| R-G3-progressive-learning | Threshold tuning reads the rolling index for 30-day aggregate statistics instead of scanning the full log. Faster, bounded cost. |
| Oscillation detection (G8) | Scans current + previous month folders for target artifact entries. Bounded to ~60 days of files instead of the entire history. |
| Trust graduation (G15) | Reads cold-tier entries (3–12 months) by `trigger_source`. May scan multiple month folders but skip archive summaries. |
| Success criteria | Most metrics read the rolling index. Long-term trends use archive summaries. No full-log scans needed. |
| Convergent iteration (G7) | Change volume per cycle reads the rolling index (cycle ID + count). |
| Staleness avoidance principle | The rolling index and month folders are self-update infrastructure — validated every cycle per the meta-infrastructure rule. Missing or corrupted index is rebuilt from entry files (deterministic recovery). |
| Snapshots | Unchanged — snapshot folders continue to use `<change_id>/` naming. The change ID convention (G23) makes the mapping from outcome log entry to snapshot folder deterministic. |

This analysis audits each risk's defense coverage across all three quality dimensions, identifies where goal/scope/boundaries/rationales could strengthen defenses, and proposes gaps where mitigations are either absent or mentioned-but-undefined.

| **Insufficient outcome data** | Adequate — with G11 (bootstrapping) + G12 (threshold stability) | Adequate — conservative-first stated | Adequate | ✅ Addressed by pending gaps |

### Cross-cutting insight: metadata as under-leveraged defense

The metadata-driven principle (R-S1-metadata-driven) with extensions from G17 (rationale metadata) provides goal, scope, boundaries, and rationale constraints per artifact. This structured metadata is the system's most powerful self-defense mechanism, yet **no risk in the vision cites it as a mitigation**. The Risks section references R-L2, R-G3, R-S3, R-S2, and tier-based cadence — but never R-S1.

Each risk benefits from structured metadata as a defense:

| Risk | How metadata defends |
|---|---|
| **False confidence** | Goal/scope comparison detects structural effectiveness changes even when output quality can't be measured. If a change alters the goal or shrinks the scope, it's flagged regardless of validation pass. |
| **Over-correction** | Findings that don't affect goal, scope, or boundaries are automatically lower severity. Metadata provides the severity anchor that the risk's mitigation mentions but doesn't define. |
| **Complexity ceiling** | The constraint surface (rationale count + dependency edges + metadata field count) is a measurable complexity proxy. Metadata-driven artifacts make complexity auditable. |
| **Deterministic rigidity** | Missing or malformed metadata structure signals parser failure. If the deterministic check expects `goal:` and `scope:` but finds neither, it knows the format changed. Metadata provides the canary. |
| **Trigger completeness** | Artifacts with complete metadata are self-describing — their declared scope tells the Detect layer what external changes are relevant, narrowing the trigger gap. |

**Recommendation:** Each risk's mitigation should explicitly connect to R-S1-metadata-driven where applicable. This doesn't add new mechanisms — it documents existing defense coverage that the Risks section currently omits.

### Gap analysis

#### G24: No effectiveness signal beyond rejection (HIGH)

**Source:** False confidence risk — "Residual: silent effectiveness drift remains a hard boundary."

**Problem:** The outcome log records `outcome: success | rolled_back | rejected`, but there's no way to record "this change was structurally correct but made the artifact less effective." The only negative signal path is rejection (human blocks a proposal) or rollback (post-execution validation detects regression). Neither captures **gradual effectiveness degradation** — the scenario where each individual change passes validation but cumulative drift reduces artifact quality.

**What's missing:**
- No effectiveness annotation in outcome log entries. A human who notices degraded output has no structured way to feed that signal back.
- R-G3-progressive-learning learns from success/failure rates, but can't learn from effectiveness observations because the signal doesn't exist.
- The success criteria define metric #6 ("artifact quality improves over time — measured by feedback from downstream consumers") but the vision specifies no mechanism to collect downstream consumer feedback.

**Proposed change:** Add an optional `effectiveness_signal` field to the outcome log schema. Values: `effective` (confirmed by human observation), `degraded` (human observed reduced quality), `unknown` (default — no observation). R-G3 uses this field: entries with `outcome: success` but `effectiveness_signal: degraded` contribute to threshold tightening, not loosening. Success criteria #6 is measurable via the ratio of `effective` to `degraded` signals over time.

**Connects to:** R-G3-progressive-learning (needs the signal), R-S1-metadata-driven (goal/scope comparison provides partial proxy), success criteria #6 (measurability).

#### G25: Finding severity classification undefined (HIGH)

**Source:** Over-correction risk — "Severity thresholds limit findings to those affecting reliability or correctness."

**Problem:** The mitigation cites severity thresholds that are **mentioned but never defined**. Without a severity scale, every finding competes equally for attention. The system can't distinguish "broken reference in meta-infrastructure" (must fix immediately) from "slightly suboptimal phrasing" (may never need fixing). This directly feeds over-correction: without severity, the system treats cosmetic findings with the same urgency as operational ones.

**What's missing:**
- No severity classification table defining CRITICAL/HIGH/MEDIUM/LOW.
- No mapping from severity to response expectation (e.g., CRITICAL = blocks next cycle, LOW = backlog).
- No connection between severity and metadata impact. The metadata-driven principle provides the natural anchor: changes affecting goal = HIGH, changes affecting scope = MEDIUM, changes affecting neither = LOW.

**Proposed change:** Add a finding severity classification to the Assess layer:

| Severity | Criterion | Response |
|---|---|---|
| **CRITICAL** | Affects meta-infrastructure integrity or blocks operation | Must resolve before next cycle |
| **HIGH** | Affects artifact goal or breaks declared contract | Resolve within current cycle |
| **MEDIUM** | Affects artifact scope or degrades a quality dimension | Schedule for next cycle |
| **LOW** | Cosmetic, hygiene, or optimization opportunity | Backlog — resolve when adjacent changes touch the same artifact |

This connects severity to the metadata-driven principle: goal impact → HIGH, scope impact → MEDIUM, boundary impact → varies by boundary type, no metadata impact → LOW.

**Connects to:** R-S1-metadata-driven (provides severity anchor), autonomy gradient (severity feeds autonomy routing), G18 (rationale-aware classification).

#### G26: No complexity governor or simplification trigger (MEDIUM)

**Source:** Complexity ceiling risk — "Simplicity in the update mechanism is a design goal."

**Problem:** The complexity ceiling is acknowledged as a risk and simplicity is stated as a design goal, but neither has a metric, a threshold, or a corrective mechanism. The system can grow in complexity indefinitely — each analysis round adds rationales, mechanisms, and gaps (the plan is at G23 after five rounds). There's no governor that detects when the system is approaching its complexity ceiling and triggers simplification before adding more mechanism.

**What's missing:**
- No complexity proxy metric. What constitutes "too complex"? Candidates: total rationale count, active infrastructure component count, number of inter-artifact dependency edges, metadata field count per artifact.
- No complexity ceiling value or threshold.
- No simplification trigger — a mechanism that, when the complexity proxy exceeds its ceiling, requires a simplification review before any new mechanism can be added.

**Proposed change:** Add a complexity governor to the design principles:

- **Complexity proxy:** Total constraint surface = rationale count + dependency graph edge count + metadata fields per artifact (average). This reuses the convergent iteration insight: the constraint surface is already tracked for convergence; use it for complexity bounding too.
- **Ceiling:** Define a maximum constraint surface size (concrete value deferred to implementation). When the proxy exceeds the ceiling, the system enters **simplification mode**: new mechanisms require demonstrating that they consolidate or replace existing mechanisms rather than adding to the total.
- **Measurement frequency:** Complexity proxy evaluated once per cycle as part of meta-infrastructure health.

**Connects to:** R-G5-convergent-iteration (constraint surface is already tracked), metadata-driven (complexity is measurable through metadata), design principle "Simplicity is a design goal" (operationalizes it).

#### G27: No deterministic fallback protocol (MEDIUM)

**Source:** Deterministic rigidity risk — "Must detect when a check produces incorrect results and fall back to LLM processing."

**Problem:** The risk correctly identifies that deterministic parsers break when formats change. The mitigation says "detect and fall back" but specifies neither the detection mechanism nor the fallback protocol. Without a defined mechanism, the system can't distinguish between "the deterministic check found nothing" (correct result — the artifact is clean) and "the deterministic check couldn't parse the artifact" (incorrect result — format changed).

**What's missing:**
- No detection mechanism. How does the system know a deterministic check produced incorrect results vs. correct-but-empty results?
- No switching protocol. What happens when the system detects a deterministic failure? Which LLM model is used, with what prompt, at what cost?
- No fallback logging. The outcome log doesn't record deterministic-vs-LLM processing mode, so R-G3 can't learn from fallback patterns.

**Proposed change:** Add a deterministic fallback protocol to the Execute layer or as a design principle:

1. **Detection:** Every deterministic check must produce a confidence signal alongside its result. Confidence = `high` when the expected structure was found and parsed. Confidence = `low` when the parser encountered unexpected structure (missing expected fields, unknown format, parse errors). A `low` confidence result triggers LLM fallback.
2. **Metadata canary:** If a deterministic check expects metadata fields (goal, scope, boundaries from R-S1), missing metadata is a format-change signal, not a clean-artifact signal. This leverages metadata as a structural canary.
3. **Fallback logging:** Add an optional `processing_mode` field to outcome log entries: `deterministic`, `llm_fallback`, `llm_primary`. R-G3 can then track fallback frequency by artifact type — rising fallback rates signal format drift requiring parser updates.

**Connects to:** R-P1-deterministic-processing (extends it with failure handling), R-L1-llm-reasoning (the fallback target), R-S1-metadata-driven (provides the canary), graceful degradation principle (operationalizes it).

#### G28: Risks section doesn't cite metadata-driven defenses (LOW)

**Source:** Cross-cutting insight — none of the 9 risks cite R-S1-metadata-driven as a mitigation.

**Problem:** The Risks section cites R-L2, R-G3, R-S3, R-S2, severity thresholds, and tier-based cadence as mitigations. R-S1-metadata-driven is never cited, even though structured metadata provides defense coverage for at least 5 of the 9 risks (as shown in the metadata defense matrix above). This isn't a missing mechanism — the defense exists. It's a documentation gap that makes the risk section appear less defended than it actually is, and makes it harder to reason about defense completeness.

**Proposed change:** Update each applicable risk's mitigation prose to cite R-S1-metadata-driven alongside existing mitigations. Specifically:
- **False confidence:** Add "R-S1-metadata-driven provides partial proxy — goal/scope drift detection."
- **Over-correction:** Add "R-S1-metadata-driven anchors severity classification (G25) — metadata impact determines finding priority."
- **Complexity ceiling:** Add "R-S1-metadata-driven enables measurement — constraint surface is a complexity proxy (G26)."
- **Deterministic rigidity:** Add "R-S1-metadata-driven provides structural canary — missing expected metadata signals format change (G27)."

**Connects to:** All risks with ⚠️ verdicts in the defense matrix. Makes the existing defense posture explicit.

### How goal, scope, boundaries, and rationales strengthen defense

The user's core question: "Is there enough self-defense in place?" The answer: **reliability is well-defended, efficiency is adequately defended, effectiveness is the systematic weak spot.** The metadata-driven principle is the most under-leveraged defense — citing it explicitly in the Risks section connects existing mechanisms rather than adding new ones.

**Goal** defends against false confidence and over-correction. When each artifact declares its goal, changes that don't affect the goal are inherently lower risk. Changes that DO affect the goal are inherently higher risk, regardless of structural validation results.

**Scope** defends against trigger completeness and over-correction. Declared scope tells the Detect layer what external changes are relevant (narrowing the trigger gap) and provides severity discrimination (scope-affecting vs. scope-neutral findings).

**Boundaries** defend against complexity ceiling and deterministic rigidity. Declared boundaries are constraints the system can enforce deterministically. Boundary count is a complexity proxy. Missing boundaries signal format change.

**Rationales** (from G17) defend against all five ⚠️ risks. Rationale constraints are the most specific defense: they encode WHY an artifact is structured as it is. A change that violates a declared rationale triggers escalation even if it passes all other checks. The rationale constraint surface is the convergence metric (from G7), the complexity proxy (G26 above), and the severity anchor (G25 above). Rationales are the keystone of the self-defense system — they transform implicit design decisions into explicit, enforceable constraints.

## 📏 Dimension-specific impact classification analysis

### Framing

G1 identifies that the vision's breaking/non-breaking classification uses a single dimension (goal/scope/boundary preservation). The Core Insight section proposes three additional dimensions (consistency, efficiency, effectiveness) and the staged integration model processes them independently. But the dimension set was derived from a single example — "deeper coverage within declared scope" — not from a systematic analysis of what dimensions are relevant for a PE self-update architecture.

This analysis answers: **What is the complete, minimal, orthogonal set of impact dimensions for classifying changes to prompt engineering artifacts?** It then designs dimension-specific assessment mechanisms and maps each dimension to reliability, effectiveness, and efficiency outcomes.

### Dimension derivation method

The dimension set is derived from three sources:

1. **The vision's own quality dimensions.** The "Two operating levels" section lists: *non-contradiction, non-redundancy, consistency, prioritization, coverage, reliability, effectiveness, efficiency*. These are quality properties the system monitors — some of these are impact dimensions, others are composite outcomes.

2. **The vision's key definitions.** The goal section defines three quality outcomes: *reliable* (consistent, predictable), *effective* (achieves stated purpose), *efficient* (minimal resource consumption). These are output properties, not impact dimensions — but each impact dimension should map to at least one of these.

3. **The artifact lifecycle.** PE artifacts have a lifecycle: they're authored → loaded into context → processed by the model → produce output. A change can have impact at each lifecycle stage. Structural changes affect authoring-time validation. Consistency changes affect model-time reasoning. Efficiency changes affect load-time budgets. Effectiveness changes affect output-time quality.

### The seven impact dimensions

The analysis identifies seven dimensions. The first four were already in the Core Insight; three are new. Each dimension is orthogonal — a change can be breaking on one and non-breaking on all others.

#### 1. Structural (goal/scope/boundary preservation)

**What it measures:** Does the change alter what the artifact is *for*, what it *covers*, or what it *constrains*?

**Assessment mechanism:** Deterministic comparison of declared goal, scope, and boundary metadata (R-S1-metadata-driven). If metadata exists: deterministic, high confidence. Without metadata: LLM judgment, lower confidence.

**Breaking signal:** Goal changed, scope expanded/contracted, boundary added/removed/modified.

**Non-breaking signal:** Goal preserved, scope preserved, boundaries preserved — the change operates within the existing contract.

**Maps to:** Reliability (contract stability), Effectiveness (purpose preservation).

**Stage eligibility:** Stage 1 — always assessed first. A structural break overrides all other dimensions.

**Already in vision:** Yes — this is the current single-dimension classification.

#### 2. Consistency (non-contradiction with existing rules)

**What it measures:** Does the change introduce rules, workflows, or guidance that contradicts, overrides, or reprioritizes existing content — within the same artifact or across dependent artifacts?

**Assessment mechanism:** Primarily LLM-based (R-L1-llm-reasoning). Rule-set comparison requires understanding semantic relationships between rules. Can be partially deterministic when artifacts use N-1 separation (rules are extractable as discrete units). Priority shifts require understanding context-dependent ordering effects — fundamentally requires judgment.

**Breaking signal:** New rule contradicts existing rule. New workflow overrides existing workflow. Addition of high-priority content demotes previously prominent rules (proximity/recency bias in LLM context windows). Guidance conflicts with rationale constraints (G17).

**Non-breaking signal:** New content is additive, no existing rules affected, no priority shifts.

**Maps to:** Reliability (predictable rule application), Effectiveness (no degraded reasoning from contradictions).

**Stage eligibility:** Stage 1 gate — must pass before autonomous integration. Failure escalates to human review.

**Already in Core Insight:** Yes, as "consistency dimension."

#### 3. Efficiency (token consumption and processing cost)

**What it measures:** Does the change increase the token cost of loading, processing, or validating the artifact or its dependents beyond declared budgets?

**Assessment mechanism:** Deterministic. Token count delta is computable. Declared token budgets (from `boundaries:` metadata) provide the threshold. Dependent impact is computable via R-S2-dependency-graph — sum the delta across all consumers.

**Breaking signal:** Token count exceeds artifact's declared budget. Cumulative token cost across consumer chain exceeds any consumer's budget. Processing time exceeds pipeline tolerance (if declared).

**Non-breaking signal:** Token delta within budget, no consumer budget violated.

**Maps to:** Efficiency (directly), Reliability (token budget violations cause context rot → degraded outputs).

**Stage eligibility:** Stage 1 gate — deterministic, cheap, always assessed.

**Already in Core Insight:** Yes, as "efficiency dimension."

#### 4. Effectiveness (output quality impact)

**What it measures:** Does the change improve, degrade, or leave unchanged the quality of outputs produced by artifacts that consume the changed artifact?

**Assessment mechanism:** **Not deterministically assessable before integration** — this is the effectiveness hard limit acknowledged in the vision. Can only be measured through: (a) post-integration observation via outcome log, (b) human feedback via effectiveness signal (G24), (c) proxy indicators (rule specificity, example quality, coverage completeness).

**Breaking signal:** Observable output quality degradation after integration (detected via outcome log or human report).

**Non-breaking signal:** No degradation observed during observation window.

**Maps to:** Effectiveness (directly).

**Stage eligibility:** **Not a Stage 1 gate.** Assessed in Stage 2 (optimization) and through post-integration observation. This is the dimension that justifies the staged model — you can't wait for effectiveness evidence before integrating structurally non-breaking content.

**Already in Core Insight:** Yes, explicitly noted as deferred.

#### 5. Referential integrity (cross-artifact reference validity)

**What it measures:** Does the change break, create, or invalidate references between artifacts? This includes: rule ID references (`enforces_rules`), file path references (`enforces_rulesets`), domain citations, metadata cross-references, and chain alignment (R-S5-chain-alignment).

**Assessment mechanism:** Deterministic. Reference resolution is a graph operation — check every outgoing reference resolves to an existing target. The rename log (R-S5) provides the `old → new` mapping for moved references. Existing vision mechanisms (chain alignment checks in Assess, dangling reference detection) already cover this partially.

**Breaking signal:** Reference points to non-existent target. Rename not recorded in rename log. New reference creates a circular dependency. Reference target's contract has changed (structural break at the target).

**Non-breaking signal:** All references resolve. No new references introduced. Renamed references have rename-log entries.

**Maps to:** Reliability (broken references cause runtime failures), Efficiency (dangling references waste validation cycles).

**Stage eligibility:** Stage 1 gate — deterministic, cheap, always assessed.

**Why this is a separate dimension:** The Core Insight subsumes referential integrity under "structural." But referential integrity is distinct from structural preservation: a change can preserve its own goal/scope/boundaries (structural: non-breaking) while breaking a reference to another artifact (referential: breaking). The vision already treats dangling references as CRITICAL in self-update infrastructure. Separating this dimension makes the CRITICAL classification mechanically derivable rather than requiring case-by-case judgment.

#### 6. Behavioral compatibility (consumer processing impact)

**What it measures:** Does the change alter how consuming artifacts or tools process the changed artifact? This is distinct from consistency (which measures rule-level contradiction) — behavioral compatibility measures whether the artifact's *structure* or *interface* remains processable by its consumers.

**Assessment mechanism:** Partially deterministic. Schema changes (new YAML fields, changed metadata structure, new section headings) are detectable by structural comparison. Behavioral impact on LLM consumers (does the restructured content produce different model behavior?) requires LLM judgment or observation.

**Breaking signal:** Schema change that consumers don't handle. Metadata field renamed/removed. Section structure changed in a way that breaks deterministic parsers (R-P1). Format change that triggers deterministic fallback (G27).

**Non-breaking signal:** Schema preserved. Metadata fields preserved or added (backward compatible). Content changes within existing structure.

**Maps to:** Reliability (consumer processing must not break), Efficiency (format changes may force expensive LLM fallback).

**Stage eligibility:** Stage 1 gate for deterministic schema checks. Stage 2 for behavioral impact requiring LLM judgment or observation.

**Why this is a separate dimension:** The deterministic rigidity risk specifically calls out format changes that break parsers. G27 proposes a fallback protocol but doesn't classify format-breaking changes as a distinct impact dimension. Separating behavioral compatibility from structural preservation makes the deterministic fallback trigger mechanically classifiable: a structural-non-breaking change that alters the artifact's schema is behavioral-breaking and routes differently than a pure content change.

#### 7. Dependency propagation (downstream blast radius)

**What it measures:** How far does the change's impact propagate through the dependency graph? A change to a tier-0 artifact can affect every artifact in the system. A change to a tier-5 leaf artifact affects nothing else.

**Assessment mechanism:** Deterministic. R-S2-dependency-graph + R-S6-tier-blast-radius provide the propagation path and scope. Tier is already the first sort key in risk classification.

**Breaking signal:** High fan-out — the change affects many dependents. Tier is low (0–2) — blast radius is wide. Cascade depth exceeds damping threshold (G10).

**Non-breaking signal:** Low fan-out — few or no dependents. Tier is high (4–5) — blast radius is narrow.

**Maps to:** Reliability (wide propagation amplifies any per-change error), Efficiency (wide blast radius increases validation cost).

**Stage eligibility:** Not a gate — a **routing modifier**. Dependency propagation doesn't determine whether a change is breaking; it determines the *cost of being wrong*. A non-breaking change with high propagation gets more validation passes than the same change with low propagation.

**Why this is a separate dimension:** The vision uses tier as a sort key for risk classification but merges it into the single-dimensional model. Separating dependency propagation as its own dimension allows the classification model to distinguish "low-impact wide-propagation" (safe but expensive to validate) from "high-impact narrow-propagation" (dangerous but contained). These need different autonomy routing.

### Dimension orthogonality and completeness

**Orthogonality check:** Each dimension measures a distinct property. A change can be:
- Structural: non-breaking + Consistency: breaking (adds contradictory rule within scope)
- Efficiency: breaking + Referential: non-breaking (increases tokens, no references change)
- Behavioral: breaking + Structural: non-breaking (new YAML field changes parser behavior)
- Dependency propagation: high + every other dimension: non-breaking (safe change at tier 0)

No dimension is derivable from the others — all seven carry independent information.

**Completeness check against vision quality properties:**
- *Non-contradiction* → Consistency (dim 2)
- *Non-redundancy* → Efficiency (dim 3) + Effectiveness (dim 4) — redundancy wastes tokens and dilutes output quality
- *Consistency* → Consistency (dim 2) + Behavioral (dim 6) — both must be consistent
- *Prioritization* → Consistency (dim 2) — priority shifts are consistency breaks
- *Coverage* → Structural (dim 1) — coverage is a scope property
- *Reliability* → Structural (dim 1) + Referential (dim 5) + Behavioral (dim 6) + Dependency (dim 7)
- *Effectiveness* → Effectiveness (dim 4)
- *Efficiency* → Efficiency (dim 3)

All eight vision quality properties map to at least one dimension. No quality property is unmapped.

### Dimension classification matrix

This matrix defines how each dimension produces a per-dimension impact rating. The overall change rating is determined by the composition rules below.

| Dimension | Assessment method | Deterministic? | Stage gate? | Breaking signal | Non-breaking signal |
|---|---|---|---|---|---|
| 1. Structural | Metadata comparison (R-S1) | Yes (with metadata) | Stage 1 | Goal/scope/boundary changed | Preserved |
| 2. Consistency | Rule-set comparison (R-L1 + N-1) | Partial (N-1 helps) | Stage 1 | Contradiction, priority shift, rationale violation | No contradictions |
| 3. Efficiency | Token count delta | Yes | Stage 1 | Budget exceeded | Within budget |
| 4. Effectiveness | Observation + human signal (G24) | No | Not a gate | Degraded output quality | No degradation observed |
| 5. Referential integrity | Graph resolution (R-S2, R-S5) | Yes | Stage 1 | Dangling/circular reference | All references resolve |
| 6. Behavioral compatibility | Schema comparison + LLM | Partial | Stage 1 (schema) / Stage 2 (behavioral) | Consumer processing breaks | Schema preserved |
| 7. Dependency propagation | Tier + fan-out (R-S6, R-S2) | Yes | Routing modifier | High fan-out / low tier | Low fan-out / high tier |

### Dimension composition rules

Individual dimension ratings must be composed into a single autonomy routing decision. Three rules govern composition:

**Rule 1 — Highest-risk gate wins.** If any Stage 1 gate dimension rates as "breaking," the overall classification is "breaking" and routes to the corresponding autonomy level. Staged integration doesn't apply when a gate dimension breaks — only when gate dimensions pass but non-gate dimensions have potential impact.

**Rule 2 — Propagation scales validation, not classification.** Dependency propagation (dim 7) doesn't change the breaking/non-breaking classification. It scales the *validation effort*: more propagation → more validation passes → higher confidence required. A non-breaking change at tier 0 remains non-breaking but requires more evidence than the same change at tier 5.

**Rule 3 — Effectiveness is always deferred.** Effectiveness (dim 4) is never a pre-integration gate. It feeds the outcome log, R-G3 threshold tuning, and Stage 2 optimization scheduling. This is the effectiveness hard limit — the system acknowledges it structurally rather than ignoring it.

**Composition example:**

A change adds a new rule to a context file (tier 1, 12 dependents):
- Structural: Non-breaking (goal/scope preserved) ✅
- Consistency: Non-breaking (no contradictions, N-1 check passes) ✅
- Efficiency: Non-breaking (+80 tokens, budget is 500, current is 380) ✅
- Referential: Non-breaking (no new references) ✅
- Behavioral: Non-breaking (no schema change) ✅
- Dependency propagation: Medium (tier 1, 12 dependents) → requires 2 validation passes
- Effectiveness: Unknown → deferred to observation

**Result:** All Stage 1 gates pass. Propagation requests additional validation. Autonomy level: **autonomous with notification** (one level above what zero-propagation would warrant, due to the 12-dependent fan-out).

#### G30: No per-dimension severity scale (MEDIUM)

**Current state.** G25 (from the risks analysis) proposes a finding severity classification (CRITICAL/HIGH/MEDIUM/LOW) anchored to metadata impact. **Proposed change.** Define a per-dimension severity scale:

| Severity | Criterion | Example |

| **None** | No impact on this dimension | Token count unchanged (efficiency), no references changed (referential) |
| **Low** | Impact detectable but within tolerance | +50 tokens on a 500 budget (efficiency), minor priority shift (consistency) |

## 🔬 Consistency gate refinement analysis

### Framing

G2's non-regression gate includes a consistency check defined as "no contradictions introduced, no priority shifts." The v1.6 dimension analysis (dim 2) reinforces this with a breaking signal: "Addition of high-priority content demotes previously prominent rules (proximity/recency bias in LLM context windows)."

The observation: priority shifts aren't inherently harmful. A new rule that makes the artifact better at its declared purpose SHOULD shift priorities toward itself — that's an improvement, not a regression. The non-regression gate should distinguish between shifts that serve the artifact's purpose and shifts that undermine it.

This analysis refines the consistency dimension's priority-shift signal, examines the impact on reliability, effectiveness, and efficiency, and identifies the metadata incentive this creates.

### The over-blocking problem

The current consistency check uses a **negative filter**: "no priority shifts." This is a conservative heuristic — it treats all priority changes as potential regressions. The assumption is: if existing rules were prioritized correctly before the change, any shift is suspect.

This assumption fails for additive changes that improve the artifact:

**Example 1 — Goal-aligned shift.** A context file with goal "provide security guidance for Azure Functions" has 10 rules. A new rule about managed identity best practices is added. The new rule, by proximity in the context window, demotes the least security-relevant existing rule (a formatting convention that was included for completeness). The priority shift is TOWARD the artifact's goal (security), not away from it. The current gate blocks this as "priority shift detected."

**Example 2 — Rationale-aligned shift.** An instruction file has a declared rationale: "R-01: Prioritize actionable guidance over theoretical background." New content adds a concrete implementation pattern, shifting priority away from an existing theory section. The shift serves the declared rationale. The current gate blocks it.

**Example 3 — Goal-opposed shift.** A context file has a boundary: "single source of truth for authentication patterns." New content about authorization patterns shifts priority away from authentication. This crosses the declared boundary. The gate SHOULD block this — and does, correctly.

The pattern: examples 1 and 2 are false positives (over-blocked). Example 3 is a true positive (correctly blocked). The negative filter catches all three identically. An alignment filter catches only example 3.

### Quantifying the over-blocking impact

Over-blocking affects the self-update system's effectiveness and efficiency:

**Effectiveness impact.** Every false-positive gate block delays a beneficial change. If the change requires human approval (because the gate routes it upward), the delay is hours to days. If the system is operating in Phase 1–2 (conservative autonomy), most priority-shift detections escalate to human review. Goal-aligned improvements queue behind the same approval process as actual regressions.

**Efficiency impact.** Every false-positive block consumes: (a) the cost of the LLM consistency check that detected the "issue," (b) the cost of generating a human-review request, (c) the human attention cost of reviewing a non-issue, (d) the cost of reprocessing after approval. For a system processing dozens of changes per cycle, false positives compound.

**Reliability impact.** Paradoxically, over-blocking can REDUCE reliability. If the gate produces many false positives, human reviewers develop alarm fatigue — they start approving gate failures without deep review, which means the gate loses its protective value for ACTUAL regressions. The alarm fatigue effect is well-documented in alert systems: when 67% of alerts are false positives, the remaining 33% (true regressions) receive less attention.

### The refinement: alignment-based priority assessment

Replace the negative filter ("no priority shifts") with an alignment filter ("priority shifts must align with artifact purpose").

The alignment filter uses the artifact's declared metadata as the reference:

- **Goal** — does the shift move the artifact toward or away from its declared purpose?
- **Rationale constraints** (G17) — does the shift violate a declared design constraint?
- **Boundaries** — does the shift push the artifact outside its declared scope or constraints?

**Decision tree for priority-shift assessment:**

```
1. Does the change shift priorities of existing rules/content?
   ├─ No → PASS (no shift, no issue)
   └─ Yes ↓
2. Does the shift violate a declared rationale constraint?
   ├─ Yes → FAIL (breaking — rationale violation, per G17)
   └─ No / no rationale metadata ↓
3. Does the shift push the artifact outside its declared boundaries?
   ├─ Yes → FAIL (breaking — boundary violation)
   └─ No ↓
4. Does the shift move the artifact toward its declared goal?
   ├─ Yes → PASS (improvement — goal-aligned shift)
   └─ No / Unknown ↓
5. Does the shift move the artifact away from its declared goal?
   ├─ Yes → FAIL (breaking — goal-opposed shift)
   └─ No (neutral) ↓
6. Shift is neutral to goal → PASS with notification
   (observation window for effectiveness assessment)
```

**Assessment mechanism per step:**

| Step | Deterministic? | With metadata | Without metadata |
|---|---|---|---|
| 1. Shift detection | Partial | LLM comparison + N-1 extraction | LLM comparison |
| 2. Rationale check | Yes (with metadata) | Compare shift against declared constraints | Skip (no constraints to check) |
| 3. Boundary check | Yes (with metadata) | Compare shift against declared boundaries | LLM judgment (lower confidence) |
| 4. Goal alignment | Partial | Compare shift direction against declared goal | LLM judgment (lower confidence) |
| 5. Goal opposition | Partial | Compare shift direction against declared goal | LLM judgment (lower confidence) |
| 6. Neutral classification | N/A | Default if not aligned or opposed | Conservative: treat as goal-neutral with notification |

**Key insight:** Steps 2–5 are partially deterministic WHEN metadata exists. Without metadata, steps 2–5 require LLM judgment with lower confidence. This creates a **natural metadata incentive** — artifacts with richer metadata get higher autonomous throughput because more priority shifts can be classified without LLM judgment.

### Impact on the consistency dimension (dim 2)

The v1.6 dimension analysis defines dim 2's breaking signal as:

> "New rule contradicts existing rule. New workflow overrides existing workflow. Addition of high-priority content demotes previously prominent rules (proximity/recency bias in LLM context windows). Guidance conflicts with rationale constraints (G17)."

The refinement decomposes this into three sub-checks with distinct severity profiles:

| Sub-check | Signal | Assessment | Deterministic? |
|---|---|---|---|
| **Contradiction** | New rule directly contradicts existing rule | LLM rule-set comparison, partially deterministic with N-1 | Partial |
| **Priority alignment** | Priority shift: check alignment with goal/rationale/boundaries | Decision tree above; partially deterministic WITH metadata | Partial (metadata-gated) |
| **Workflow conflict** | New workflow conflicts with existing workflow | LLM workflow comparison | Primarily LLM |

The key change: the breaking signal for priority shifts is no longer "any shift detected" but "shift opposed to declared purpose." The non-breaking signal expands from "no priority shifts" to "no shifts, or shifts aligned with declared purpose."

**Updated dim 2 signals:**

- **Breaking signal:** Direct contradiction. Priority shift opposed to goal, violating rationale constraint, or crossing boundary. Workflow conflict.
- **Non-breaking signal:** No contradictions. No priority shifts, OR priority shifts aligned with declared goal and within rationale constraints and boundaries. No workflow conflicts.

### Impact on reliability, effectiveness, and efficiency

| Quality | Current model (negative filter) | Refined model (alignment filter) | Net change |
|---|---|---|---|
| **Reliability** | High — blocks all shifts (no false negatives, many false positives) | High — blocks goal-opposed / constraint-violating shifts (no false negatives, fewer false positives) | **Maintained** — same false negative rate |
| **Effectiveness** | Reduced — goal-aligned improvements blocked | Improved — goal-aligned improvements pass autonomously | **Better** |
| **Efficiency** | Reduced — false positive review cost | Improved — fewer escalations, less human review | **Better** |

The refinement is a **strict improvement**: same reliability, better effectiveness, better efficiency. There's no tradeoff — the negative filter was unconditionally over-blocking.

### The metadata incentive effect

The refinement creates a virtuous cycle:

1. **Without metadata:** All priority shifts require LLM judgment → conservative classification → more escalations → lower autonomous throughput.
2. **With goal metadata:** Goal-aligned shifts can be classified partially deterministically → fewer escalations → higher autonomous throughput.
3. **With goal + rationale metadata:** Rationale-violating shifts are caught deterministically + goal-aligned shifts pass → best autonomous throughput.
4. **Incentive:** Artifacts that declare metadata get better autonomous throughput → system naturally prioritizes metadata population → more metadata → more deterministic checks → convergence.

This connects directly to G17's rationale metadata population strategy: the "discovery through regression" mechanism (G19) is accelerated because the refined gate produces MORE actionable regressions (fewer false positives means each gate failure is more likely to be a real constraint violation → more likely to trigger rationale metadata discovery).

### Interaction with the seven-dimension model

The consistency gate refinement affects dim 2 specifically but interacts with the broader dimension model:

| Dimension | Interaction with consistency gate refinement |
|---|---|
| **1. Structural** | Unchanged. Structural checks are independent of priority-shift assessment. |
| **2. Consistency** | Directly refined. Breaking signal decomposed into three sub-checks. Non-breaking signal expanded. |
| **3. Efficiency** | Unchanged. Token budget checks are independent. |
| **4. Effectiveness** | Indirectly improved. Goal-aligned shifts that pass the gate are likely effectiveness improvements — the observation window (dim 4) can confirm this. |
| **5. Referential integrity** | Unchanged. Reference resolution is independent. |
| **6. Behavioral compatibility** | Unchanged. Schema checks are independent. |
| **7. Dependency propagation** | Unchanged as a routing modifier. But the reduced false-positive rate means propagation-scaled validation is applied to fewer changes (less wasted validation). |

## 🔧 Staged integration refinement analysis

### Three questions about the current model

G3 proposes a two-stage model: Stage 1 (non-regression gate) and Stage 2 (optimization phase). Three questions challenge whether this model is complete:

1. **Why only two stages?** The current model splits on a safety/value axis — Stage 1 asks "does the change break anything?" and Stage 2 asks "can existing content improve?" But nothing between them asks "does the change actually work well in context?"

2. **Per-dimension or holistic validation?** The v1.6 dimension model (G29–G32) defines per-dimension assessment with composition rules. But G3's Stage 1 description doesn't specify whether the gate runs per-dimension independently or as a single holistic check.

3. **Capability endorsement?** Should there be a stage that verifies new capabilities actually deliver their stated purpose? This relates to the effectiveness hard limit (dim 4 can't be validated deterministically before integration) but targets a different gap: post-integration quality confirmation.

### Analysis: from two stages to three

**The two-stage model's blind spot.** Stage 1 proves absence of harm (non-regression). Stage 2 finds optimization opportunities. Neither actively verifies that the integrated artifact is coherent, complete, and well-integrated. A change can pass all five gate checks (structural, consistency, efficiency, referential, behavioral-schema) while introducing subtle coherence problems at the artifact level:

- A new rule that doesn't contradict existing rules (passes consistency) but creates conceptual redundancy
- A new section that preserves the artifact's goal (passes structural) but disrupts the narrative flow
- An additive change that leaves all references intact (passes referential) but exposes content gaps in adjacent sections

These aren't regressions — the artifact isn't worse on any individual dimension. They're coherence issues — the artifact as a whole is less effective than the sum of its parts suggests. The per-dimension gate can't catch them because coherence is an emergent property.

**The proposed three-stage model:**

| Stage | Question | Assessment style | Scope | When | Failure response |
|---|---|---|---|---|---|
| **Stage 1 — Non-regression gate** | Is the change safe? | Per-dimension, fail-fast | Per-change | Pre-integration | Block / escalate |
| **Stage 2 — Coherence verification** | Is the integration good? | Holistic, LLM-based | Per-artifact | Post-integration | Flag for review |
| **Stage 3 — Optimization** | Can the artifact be better? | Artifact-wide, LLM-based | Per-artifact / cross-artifact | Post-observation | New change proposals |

Between Stage 2 and Stage 3: the **observation window** (not a stage — a monitoring period).

**Stage 1 — Non-regression gate** (unchanged in purpose, refined in execution):

- Validates per-dimension independently (from v1.6 Rule 1: highest-risk gate wins)
- Five gate dimensions: structural, consistency, efficiency, referential integrity, behavioral compatibility (schema)
- Decision: Pass all → integrate. Fail any → escalate per the autonomy gradient.
- What it doesn't answer: "Is the change correct? Complete? Well-positioned in context?"

**Stage 2 — Post-integration coherence verification** (NEW):

- Purpose: Actively verify that the integrated artifact maintains or improves quality as a whole
- Four checks:
  - **Goal alignment**: Does the artifact, with the new content, serve its declared goal at least as well as before?
  - **Content coherence**: Is the new content well-positioned? No redundancy introduced? No gaps exposed?
  - **Consumer readiness**: Do known consumers still produce expected results? (From the vision's runtime validation technique)
  - **Completeness**: Does the change deliver what it claimed? (Checked against the change proposal's stated purpose)
- Decision: Pass → confirm integration, enter observation window. Fail → flag for review (not automatic rollback — coherence issues typically need targeted fixes, not full reversion).
- Autonomy: Autonomous with notification — the change is already integrated; Stage 2 confirms or flags.

**Observation window** (timed, post-Stage-2):

- Purpose: Monitor for effectiveness regressions not caught by pre/post checks
- Duration: Configurable per tier — higher tier (lower blast radius) gets shorter windows; lower tier (higher blast radius) gets longer windows
- Signals: Outcome log entries (G24's effectiveness signal), human feedback, consumer output quality
- Decision: No regression signals within window → close window, enable Stage 3. Regression signals → escalate.
- Connects directly to G22 (observation window update mechanism) and G24 (effectiveness signal).

**Stage 3 — Optimization** (renamed from "Stage 2" in the current model, deferred post-observation):

- Purpose: Identify redesign opportunities enabled by the integrated content
- Trigger: Observation window closes with no regression signals, OR human explicitly schedules
- Each optimization is a new change entering its own Stage 1 → Stage 2 → observation → Stage 3 cycle
- Unchanged in purpose from the current model's Stage 2 — repositioned after observation confirmation

**Why three stages, not two.** The two-stage model assumes that passing non-regression checks implies correct integration. This is analogous to assuming that passing unit tests implies feature correctness — a well-known gap in software quality. Stage 2 fills this gap: it's the integration test to Stage 1's unit test. The cost is one additional LLM-based validation pass per change. The benefit: coherence issues are caught immediately rather than discovered through slow observation or manual review.

**Why not four or more stages.** Additional stages (e.g., separate "correctness" and "completeness" checks) would fragment the coherence assessment without adding signal quality. Coherence is inherently holistic — splitting it into sub-stages would force artificial decomposition of an emergent property. Three stages + observation is the minimal model that covers safety (Stage 1), quality (Stage 2), effectiveness (observation), and improvement (Stage 3).

### Analysis: per-dimension vs. holistic validation

**Stage 1 is per-dimension.** This is already established by v1.6's composition rules (Rule 1: highest-risk gate wins). Each of the five gate dimensions is assessed independently, and the overall gate decision is the composite of per-dimension results.

**Stage 2 is holistic.** Coherence is an emergent property of the integrated artifact. A per-dimension coherence check would decompose an inherently holistic property into artificial sub-checks. Stage 2 evaluates the artifact as a whole — its goal alignment, narrative flow, content organization, and consumer compatibility after the change is integrated.

**Clean separation principle:** Per-dimension for safety (Stage 1), holistic for quality (Stage 2). The safety gate needs per-dimension precision to route different failure types to different responses. The quality check needs holistic assessment to catch emergent issues that no individual dimension detects.

### Analysis: cost-ordered execution within Stage 1

The five gate checks have vastly different costs:

| Check | Method | Approximate cost |
|---|---|---|
| Structural (metadata comparison) | Deterministic | O(1) — field comparison |
| Efficiency (token count delta) | Deterministic | O(1) — numeric comparison |
| Referential integrity (graph resolution) | Deterministic | O(N) — reference count |
| Consistency (with metadata — decision tree) | Semi-deterministic | Medium — partial LLM |
| Behavioral compatibility (schema diff) | Semi-deterministic | Medium — schema comparison |
| Consistency (without metadata) | LLM-based | High — full LLM judgment |
| Behavioral compatibility (behavioral) | LLM-based | High — output comparison |

If a deterministic check fails (e.g., the change violates a structural boundary), there's no need to run the expensive LLM-based consistency check. Running checks in cost order — deterministic first, LLM last — saves tokens on every change that fails early.

**Proposed cost-ordered execution tiers within Stage 1:**

| Tier | Checks | Method | Behavior |
|---|---|---|---|
| **Tier A — Deterministic** | Structural, efficiency, referential | Binary, milliseconds | Fail → STOP, escalate. No LLM cost. |
| **Tier B — Semi-deterministic** | Consistency (with metadata), behavioral (schema) | Partial LLM, decision trees | Fail → STOP, escalate. Partial LLM saved. |
| **Tier C — LLM-required** | Consistency (no metadata), behavioral (LLM) | Full LLM judgment | Only reached if Tiers A and B pass. |

This is an execution optimization within Stage 1, not a separate stage. The gate decision is the same — all checks must pass. Only the execution order changes.

**Connection to G34 (metadata incentive amplification).** Cost ordering adds a second incentive dimension to G34's confidence gradient. With cost ordering: metadata-rich artifacts resolve consistency in Tier B (cheaper), saving Tier C's full-LLM cost. Without cost ordering: metadata still improves confidence but doesn't save cost. The combined incentive: metadata improves both autonomy routing (G34) AND validation cost (G36).

### Analysis: the capability endorsement question

The user asked whether there should be a stage to "endorse capabilities introduced by the change." The answer is: **yes, and it's Stage 2** (post-integration coherence verification).

Stage 2's completeness check ("Does the change deliver what it claimed?") is the endorsement mechanism. It verifies the change against its own stated purpose — not against the effectiveness hard limit (which requires observation), but against the change proposal's intent.

**What endorsement IS:**

- Verification that new content is correctly positioned in the artifact
- Confirmation that the change is complete relative to its stated scope
- Consumer-facing validation that known test cases still pass
- A coherence check that the artifact serves its goal at least as well as before

**What endorsement ISN'T:**

- A guarantee of real-world effectiveness (requires observation window)
- A human sign-off (would negate autonomy gains)
- A full regression test suite (too expensive per change)

**The effectiveness hard limit is unchanged.** Stage 2 doesn't claim to measure effectiveness. It verifies coherence and completeness — necessary conditions for effectiveness, but not sufficient. The observation window still provides the effectiveness signal over time. Stage 2 raises the minimum quality floor; observation validates the ceiling.

### R/E/E impact of the three-stage model

| Quality dimension | Impact |
|---|---|
| **Reliability** | Maintained. Stage 1 gate is unchanged. Stage 2 adds a second defense layer (catching coherence issues that pass per-dimension checks). |
| **Effectiveness** | Improved. Stage 2 actively verifies that the change delivers its claimed value. Without Stage 2, non-regressive but incoherent changes accumulate silently. |
| **Efficiency** | Net positive. Stage 2 adds one LLM pass per change (cost increase). Cost-ordered execution (G36) saves tokens on early failures (cost decrease). Stage 3's observation window precondition reduces wasted optimization cycles on unstable changes (cost decrease). |

A change that passes non-regression but introduces coherence problems (redundancy, flow disruption, content gaps) is integrated silently. | Stage 2 — Coherence verification | Verify artifact quality post-change | Holistic, LLM-based | Post-integration |

Running them in arbitrary order wastes tokens when a cheap deterministic check would have caught the failure. | Tier | Checks | Method | Behavior |

## 📋 YAML frontmatter metadata analysis

### The question about G6

G6 proposes updating the `description:` YAML field to mention dimension-aware impact classification and staged integration. Three questions challenge whether this is the right approach:

1. **Why update `description:` at all?** The `description:` field is a free-text summary for human readers and rendering engines (Quarto). The self-update system doesn't consume it — no gate, no check, no decision depends on it. Updating it is cosmetic maintenance, not functional improvement.

2. **Is there a more machine-useful attribute?** The system already proposes `rationales:` (G17) as artifact-level design constraints. But there's no attribute that declares which *vision-level rationales* the artifact operationalizes. A context file that implements R-S1-metadata-driven and R-P1-deterministic-processing doesn't declare this anywhere machine-readable.

3. **What metadata would actually help the self-update system?** The system needs to answer: "Which artifacts are affected when rationale R-G4 changes?" Today this requires full-text search or LLM judgment. A machine-readable declaration would make this deterministic.

### Analysis: two kinds of metadata that are conflated

The current `description:` field mixes two fundamentally different concerns:

| Concern | Consumer | Format needed | Update trigger |
|---|---|---|---|
| **Human-readable summary** | Authors, search indexes, rendering engines | Free text, 120–160 chars | When the artifact's purpose changes significantly |
| **Machine-readable rationale coverage** | Self-update system's Detect and Assess layers | Structured YAML, enumerated IDs | When the artifact starts or stops operationalizing a vision rationale |

The `description:` field serves the first concern. Nothing serves the second. G6 proposes updating the wrong field for the wrong reason — it makes the human summary more accurate but adds zero machine-actionable information.

### Analysis: what `implements:` would enable

A new `implements:` metadata attribute would declare which vision-level rationales the artifact operationalizes:

```yaml
goal: "Validate PE artifact quality across 7 dimensions"
scope:
  covers: [structure, references, coherence, readability]
  excludes: [deployment, runtime behavior]
boundaries:
  max_tokens: 800
  single_source_of_truth_for: validation-criteria
version: "1.3.0"
implements:
  - R-S1-metadata-driven
  - R-P1-deterministic-processing
  - R-L2-self-correction
rationales:
  - id: "ordering-dependency"
    constraint: "Structural validation must complete before reference checks"
    impact_if_violated: "False positives in reference checks"
```

**`implements:` vs. `rationales:` — complementary, not redundant:**

| Attribute | Declares | Direction | Granularity | Consumer question |
|---|---|---|---|---|
| `implements:` | Which vision rationales this artifact operationalizes | Artifact → Vision (upward reference) | Per-rationale ID | "Which artifacts are affected by a change to R-S1?" |
| `rationales:` (G17) | Which design constraints govern this artifact's internal structure | Vision → Artifact (downward constraint) | Per-constraint | "Does this change respect the artifact's design decisions?" |

`implements:` is a **dependency declaration** — it tells the system what vision-level concepts this artifact is wired to. `rationales:` is a **constraint declaration** — it tells the system what internal design decisions must be preserved. Both are machine-readable. Both serve the self-update system. Neither replaces the other.

### What `implements:` enables across the four layers

**Detect layer — targeted impact analysis:**

When a vision rationale changes (e.g., R-S1-metadata-driven is refined), the system can deterministically find all artifacts that declare `implements: [R-S1-metadata-driven]` and scope the impact analysis to exactly those artifacts. Without `implements:`, the system must scan all artifacts with LLM judgment to determine which ones are affected. This is the difference between O(k) targeted analysis (k = implementors) and O(N) full scan (N = all artifacts).

**Assess layer — impact classification precision:**

A change to an artifact that `implements:` a rationale can be cross-checked against that rationale's current definition. If the change moves the artifact further from the rationale's design implication, that's a consistency signal — even if the artifact's own `rationales:` metadata doesn't catch it. `implements:` provides a second consistency check dimension.

**Propose layer — change propagation scoping:**

When the system proposes changes based on an external trigger (e.g., a platform release changes how metadata works), `implements:` tells it which artifacts need updating and which are unaffected. This is the same targeted propagation that `rationales:` enables at the internal level, but at the vision-rationale level.

**Execute layer — blast radius calculation:**

Dependency propagation (G29, dim 7) uses tier and fan-out to estimate blast radius. `implements:` adds a third input: rationale fan-in. An artifact that implements many rationales is more tightly coupled to the vision — changes to it ripple more broadly. This doesn't change the propagation dimension's formula; it enriches one of its inputs.

### Analysis: `description:` is still useful but not for the self-update system

The `description:` field serves a legitimate purpose: human-readable summaries for authors and rendering engines. It should continue to exist. But the self-update system shouldn't depend on it, because:

1. **It's not machine-parseable** — free text can't be consumed deterministically
2. **It has no contract** — there's no validation for whether a description is accurate or complete
3. **It's high-maintenance, low-signal** — updating it on every concept change creates churn without actionable information

G6's proposal to update `description:` when concepts change is valid as cosmetic maintenance, but it shouldn't be a gap finding. The real gap is the absence of `implements:`.

### Analysis: the complete YAML frontmatter model for self-update

Combining all metadata attributes across the plan's history, the full model is:

| Attribute | Purpose | Consumer | Introduced by |
|---|---|---|---|
| `goal:` | The artifact's purpose | Self-update (structural gate) | Vision v1 |
| `scope: {covers, excludes}` | Topic coverage boundaries | Self-update (structural gate) | Vision v1 |
| `boundaries:` | Operational constraints (tokens, SSOT) | Self-update (efficiency gate) | Vision v1 |
| `version:` | SemVer for change classification | Self-update (change tracking) | Vision v1 |
| `rationales:` | Internal design constraints | Self-update (constraint gate) | G17 |
| `implements:` | Vision rationales this artifact operationalizes | Self-update (impact scoping, propagation) | G38 (NEW) |
| `description:` | Human-readable summary | Authors, rendering engines | Standard YAML |

**The first six attributes are machine-actionable.** The self-update system consumes them for deterministic or semi-deterministic decisions. `description:` is human-facing only — useful but not a self-update input.

### Analysis: metadata cost and the population burden

Adding `implements:` creates a population burden: every artifact needs to declare which rationales it implements. This raises the same concern that G17's rationale metadata raises — when and how does the metadata get populated?

**The population strategy for `implements:` mirrors G19 (incremental, regression-driven):**

| Phase | `implements:` status |
|---|---|
| **Phase 1** (templates, snippets, hooks) | Not needed — these artifacts have narrow scope and few rationale connections |
| **Phase 2** (skills, agent tool lists) | Optional — add when impact analysis reveals a missed connection |
| **Phase 3** (agents, context files, instructions) | Recommended — these are the primary rationale implementors |
| **Phase 4** (cross-tier refactors) | Required — cross-tier changes need to know which rationales are at play |

**Discovery mechanism:** When a vision rationale change triggers impact analysis and the system fails to identify an affected artifact (because it doesn't declare `implements:`), the post-mortem adds the missing declaration. This is the same regression-driven discovery pattern as G19.

**Cost-benefit:** `implements:` is a list of string IDs — much cheaper to maintain than `rationales:` (which require constraint + impact_if_violated). An artifact with 3 implemented rationales adds ~30 bytes of YAML. The self-update system saves O(N) → O(k) on every vision-rationale change analysis.

### R/E/E impact of the metadata refinement

| Quality dimension | Impact |
|---|---|
| **Reliability** | Improved. `implements:` enables targeted impact analysis when vision rationales change — fewer missed artifacts, fewer late-discovered regressions. |
| **Effectiveness** | Improved. Self-update can cross-check artifact changes against their declared rationale implementations — a second consistency signal that catches drifts `rationales:` alone misses. |
| **Efficiency** | Improved. Targeted O(k) analysis replaces O(N) full scan for vision-rationale changes. `description:` updates are reclassified from gap finding to routine maintenance — no longer tracked as a gap. |

### Gap definitions

**G38: No machine-readable declaration of which vision rationales an artifact implements (HIGH)**

The self-update system has no deterministic way to answer "which artifacts are affected when rationale R-X changes?" The `description:` field is free text — not machine-parseable. The `rationales:` metadata (G17) declares internal constraints, not vision-level dependencies. When a vision rationale changes, impact analysis requires either full-text LLM scan of all artifacts (O(N), expensive, lower confidence) or human knowledge of the dependency graph (not autonomous).

**Proposed change.** Add `implements:` to the metadata specification in the Metadata-driven principle:

> - **`implements:`** — list of vision-level rationale IDs that this artifact operationalizes. Used by the self-update system to scope impact analysis when a rationale changes. Each entry is a rationale slug (e.g., `R-S1-metadata-driven`).
>
> **`implements:` vs. `rationales:`.** `implements:` declares upward dependencies (artifact → vision rationale). `rationales:` declares downward constraints (vision rationale → artifact design decisions). A change to a vision rationale triggers re-assessment of all artifacts in its `implements:` set. A change to an artifact triggers constraint checking against its `rationales:` entries. Both are machine-readable. Both serve the self-update system. Neither replaces the other.

Population strategy follows G19's incremental, regression-driven model. Not required in Phase 1–2; recommended in Phase 3; required in Phase 4.

**Target section:** Metadata-driven design principle (adjacent to G17's `rationales:` addition).

**Connects to:** G17 (rationales — complementary attribute), G19 (population strategy — same pattern), G29 dim 7 (dependency propagation — `implements:` enriches fan-in calculation), R-S1-metadata-driven (the principle being extended), R-S2-dependency-graph (implements is a dependency edge).

**G39: G6 conflates human-readable summary with machine-actionable metadata (LOW)**

G6 proposes updating the `description:` YAML field to reflect new concepts. This treats `description:` as a self-update input, but the field is free text consumed only by humans and rendering engines. No gate, check, or routing decision depends on it. Updating `description:` is valid cosmetic maintenance but not a self-update gap.

**Proposed change.** Reclassify G6 from "gap finding" to "routine maintenance note." The `description:` field should be updated when significantly stale, but this is author maintenance, not a self-update system concern. The actual self-update gap (machine-readable rationale coverage) is addressed by G38.

Add a note to the Metadata-driven principle:

> **`description:` is human-facing metadata.** The free-text description field serves authors, search indexes, and rendering engines. It is NOT consumed by the self-update system for any gate, check, or routing decision. Keeping it accurate is good authorship practice but is not a self-update precondition. Machine-actionable metadata lives in `goal:`, `scope:`, `boundaries:`, `version:`, `rationales:`, and `implements:`.

**Target section:** Metadata-driven design principle.

**Connects to:** G6 (the original gap being reclassified), G38 (``implements:`` addresses the real gap G6 was approximating), R-S1-metadata-driven (clarifies what's machine-actionable and what isn't).

## 🔄 Oscillation detection refinement analysis

### Framing

G8 identifies a real problem — feedback-loop oscillation — but proposes a solution anchored to outcome-log parsing. With G40's distance-based convergence model now available, oscillation detection can be grounded in the same measurement framework that governs convergence, divergence, and set-level coherence. This analysis examines G8's structural weaknesses and proposes distance-vector oscillation detection as the primary mechanism.

### What G8 currently proposes (and why it's fragile)

G8's proposed change says: "query the outcome log for the target artifact. If the artifact has been changed and subsequently rolled back (or changed in a contradictory direction) within the last N cycles, classify the change as oscillating."

This approach has four structural weaknesses:

1. **It relies on infrastructure whose schema isn't specified for this use case.** The outcome log (`.copilot/self-update/outcome-log.jsonl`, from R-G3-progressive-learning) is defined for tracking autonomous actions and outcomes. G21–G23 add partitioned storage, observation updates, and change IDs. But none define the specific fields needed to detect "changed and subsequently rolled back" — what constitutes a "rollback" entry? How is it distinguished from a forward correction? The detection mechanism depends on log conventions that don't exist yet.

2. **It only catches explicit rollbacks.** Oscillation has several forms:
   - **Explicit:** Apply change → rollback → re-apply → rollback. G8 catches this.
   - **Implicit:** Apply change A → apply change B that reverses A's effect → apply change C that reverses B's effect. The log shows three forward changes, not apply/rollback pairs. G8 misses this.
   - **Environmental:** External trigger causes drift toward state X → system corrects back → next trigger causes same drift. No rollback occurs at any point. G8 misses this.
   - **Cross-dimension:** Structural improvement degrades consistency → consistency fix degrades structure → repeat. Each change is valid per its own dimension. G8's single-artifact log check misses the cross-dimension pattern.

3. **"Contradictory direction" is undefined.** G8's fallback — detecting changes "in a contradictory direction" — requires semantic comparison of content changes. This is an LLM judgment call, subject to the same reasoning limits that may have caused the oscillation. The detection mechanism has the same failure mode as the problem it's detecting.

4. **"N cycles" lookback is arbitrary.** Too short (N=2) misses slow oscillation where the system alternates over longer periods. Too long (N=20) over-flags artifacts that had a single correction months ago. The window doesn't adapt to artifact volatility or cycle frequency.

### Why G40's distance model subsumes G8's detection

With G40, the system already measures per-artifact, per-dimension distance at cycle boundaries. This distance history is exactly the signal needed for oscillation detection — no additional infrastructure required.

For each artifact on each dimension, the system maintains a distance series: d(t₁), d(t₂), d(t₃), ...

The consecutive deltas tell the full story:

| Pattern | Delta signs | Meaning | Response |
|---|---|---|---|
| Convergence | −, −, −, ... | Distance decreasing | Healthy — continue |
| Divergence | +, +, +, ... | Distance increasing | Convergence concern (G40) — priority assessment |
| Oscillation | +, −, +, −, ... or −, +, −, +, ... | Distance alternating | Oscillation (G43) — freeze dimension, escalate |
| Stall | ≈0, ≈0, ≈0, ... | Distance not changing | Stall (G45) — assess whether expected |

This approach has five structural advantages over G8's outcome-log approach:

1. **Reuses existing measurements.** G40's distance evaluations are already computed at cycle boundaries. Oscillation detection adds only a comparison of consecutive deltas — no new infrastructure, no log parsing, no additional LLM calls.

2. **Catches all oscillation forms.** Explicit rollbacks, implicit reversals, environmental drift, and cross-dimension oscillation all produce the same distance-alternation pattern. The detection mechanism is form-agnostic.

3. **Is deterministic.** Sign-change counting on a distance series is a simple comparison operation. No LLM judgment needed. No ambiguity about what constitutes "contradictory direction."

4. **Is per-dimension.** When oscillation is detected, you know exactly which dimension is oscillating — structural, consistency, efficiency, referential, or behavioral. This enables targeted intervention rather than blanket escalation.

5. **Has a natural sensitivity parameter.** The threshold k (consecutive sign changes before flagging) is a single tunable parameter with clear semantics: k=2 is aggressive (flags after one alternation), k=3 is balanced (requires sustained alternation), k=4+ is conservative. Unlike "N cycles" of log lookback, this parameter directly maps to the phenomenon being detected.

### What the outcome log still provides

The outcome log isn't displaced — it serves different purposes:

| Purpose | Mechanism | Owner |
|---|---|---|
| **Oscillation detection** | Distance-delta sign analysis | G43 (replaces G8's log-parsing approach) |
| **Audit trail** | Partitioned log entries | G21–G23 (unchanged) |
| **Progressive learning** | Success/failure tracking | R-G3-progressive-learning (unchanged) |
| **Effectiveness signal** | Observation window outcomes | G24 (unchanged) |
| **Root-cause investigation** | Change history when escalation occurs | Human reviewer (enhanced by audit trail) |

Oscillation *detection* is based on distance measurements. The log provides *explanation* — why did the system oscillate? — after detection identifies *that* it oscillated.

### Complete convergence monitoring model (post v2.1)

After this analysis, the convergence model covers all trajectory states:

| State | Signal | Detection mechanism | Response |
|---|---|---|---|
| **Converging** | Distance decreasing | G40 — per-artifact distance measurement | Continue — healthy trajectory |
| **Diverging** | Distance increasing | G40 — convergence concern flag | Priority assessment for specific artifact and dimension |
| **Oscillating** | Distance alternating | G43 — delta sign analysis | Freeze dimension, log history, escalate to human |
| **Stalled** | Distance approximately constant | G45 — delta magnitude check | Assess: expected (near zero distance) or blocked? |
| **Set-level diverging** | Aggregate distance or coherence degrading | G41 — set-level convergence | Priority assessment at artifact-set level |
| **Converged** | Zero distance, all dimensions | G40 — fixed-point definition | Done — no further changes unless external trigger |

All detection mechanisms operate on the same underlying data: per-artifact, per-dimension distance vectors computed at cycle boundaries. No separate oscillation-detection infrastructure, no log parsing, no LLM-based "contradictory direction" judgment.

### What's missing: oscillation detection gaps

#### G43: G8's oscillation detection relies on outcome-log parsing instead of distance-vector analysis (HIGH)

G8 correctly identifies oscillation as a critical convergence failure mode. But the proposed detection mechanism — querying the outcome log for apply/rollback patterns — is fragile: it depends on log schema not yet specified for this use, catches only explicit rollbacks, requires undefined "contradictory direction" judgment, and uses an arbitrary lookback window.

G40's distance model provides a strictly superior detection mechanism: oscillation manifests as alternating signs in consecutive distance deltas. This is deterministic, per-dimension, catches all oscillation forms, and reuses existing measurements.

**Proposed change.** Replace G8's outcome-log oscillation detection with distance-vector oscillation detection:

> **Oscillation detection (distance-based).** For each artifact on each measured dimension, the system tracks distance at cycle boundaries. Oscillation is detected when the sign of the distance delta (Δ = d(t+1) − d(t)) alternates for k or more consecutive cycles (configurable, default: k=3 — requires at least three sign changes to distinguish oscillation from noise).
>
> When oscillation is detected on a dimension:
> 1. **Freeze** autonomous changes for the oscillating artifact on the oscillating dimension.
> 2. **Log** the distance history (full delta series) and the content changes that produced the alternation.
> 3. **Escalate** to human review with: the distance vector, the dimension, the artifact, and the changes.
>
> The outcome log retains its role for audit trail, progressive learning, and effectiveness signal — but oscillation *detection* is based on distance measurements, not log parsing.

**Target section:** G8's oscillation detection rule in the Execute layer.

**Connects to:** G8 (the gap being refined), G40 (distance measurements are the input), G41 (set-level oscillation is detectable from set-level distance), G9 (iteration budget bounds cost even if oscillation detection lags), G10 (cascade damping prevents oscillation from propagating through dependencies), R-G5-convergent-iteration (oscillation detection is a design implication of this rationale).

#### G44: R-G5-convergent-iteration still references outcome-log oscillation detection after G42's update (MEDIUM)

G42 updated R-G5-convergent-iteration's design implication from "change volume per cycle" to "per-artifact distance at cycle boundaries." But the oscillation clause still says: "Detect oscillation via distance oscillation patterns AND outcome log analysis." With G43 establishing distance-vector analysis as the primary mechanism, the "AND outcome log analysis" clause is misleading — it implies the outcome log is a co-equal detection mechanism rather than a supplementary investigation tool.

**Proposed change.** Update R-G5-convergent-iteration's oscillation reference in the design implication:

> **From (G42's proposed text):** "Detect oscillation via distance oscillation patterns AND outcome log analysis."
>
> **To:** "Detect oscillation via distance-delta sign analysis (primary, deterministic). Use outcome log for root-cause investigation after oscillation is detected."

**Target section:** R-G5-convergent-iteration's design implication (supersedes the relevant clause in G42's proposed text).

**Connects to:** G42 (the text being further refined), G43 (the mechanism change driving this update), R-G5-convergent-iteration (the rationale being updated), R-G3-progressive-learning (outcome log's retained role).

#### G45: No stall detection in convergence model (MEDIUM)

G40 covers convergence (distance decreasing) and divergence (distance increasing). G43 covers oscillation (distance alternating). But the model has no detection for **stalling**: distance remains approximately constant across multiple cycles. Stalling means the system runs cycles but makes no progress on a dimension — possibly because of conflicting constraints, insufficient metadata, or dimensions that can't improve without human input.

A stalled dimension differs from a converged one: converged has distance ≈ 0 (at the target), stalled has distance > 0 but Δ ≈ 0 (stuck away from the target).

**Proposed change.** Add stall detection to the convergence model:

> **Stall detection.** If an artifact's distance on a measured dimension changes by less than ε (configurable threshold) for m consecutive cycles (configurable, default: m=5), the dimension is **stalled**. Stall response depends on current distance:
>
> - **Near-zero stall** (distance < δ): Expected — the artifact is near convergence. No action needed; the dimension approaches the fixed-point asymptotically.
> - **Non-zero stall** (distance ≥ δ): Unexpected — the system can't make progress. Trigger priority assessment:
>   1. Does the artifact have sufficient metadata for the system to know what to improve?
>   2. Are there conflicting constraints that prevent convergence on this dimension?
>   3. Does the dimension require human creative judgment that the system can't automate?
>   4. If stall persists after assessment, escalate the dimension for human review.

**Target section:** Adjacent to oscillation detection in the convergence model.

**Connects to:** G40 (distance measurements are the input), G43 (oscillation detection covers the alternating case; stall covers the flat case), G34 (metadata-gated confidence — insufficient metadata is a stall root cause), G9 (iteration budget prevents indefinite cycling on a stalled dimension), R-G5-convergent-iteration (stall detection is a design implication of convergence governance).

### YAML frontmatter metadata analysis (v1.9)

**The question.** G6 proposes updating `description:` when concepts change — but `description:` is free text for humans, not a self-update input. Is there a structured metadata attribute that would serve the self-update system better?

**Finding.** The analysis distinguishes two metadata functions that G6 conflates:

1. **Human-readable summary** (`description:`) — consumed by authors and rendering engines. Valid to keep accurate, but not a self-update concern.
2. **Vision-rationale dependency** (`implements:`) — declares which vision-level rationales the artifact operationalizes. New attribute, not previously identified. Enables deterministic O(k) impact analysis when a rationale changes.

**Key insight: `implements:` vs. `rationales:`.** These serve complementary directions. `implements:` is an upward reference (artifact → vision: "I operationalize these rationales"). `rationales:` (G17) is a downward constraint (vision → artifact: "Preserve these design decisions"). Both are machine-readable. Both are needed. Neither replaces the other.

**G6 reclassification.** G6 is reclassified from gap finding to routine maintenance. The real gap is G38 (`implements:`). G39 adds a metadata classification note to the Metadata-driven principle.

**The complete frontmatter model.** Six machine-actionable attributes (`goal`, `scope`, `boundaries`, `version`, `rationales`, `implements`) plus one human-facing attribute (`description`).

### What the YAML frontmatter metadata analysis does NOT alter

- The **three-layer model** (G17: contract, rationale, content) is unchanged. `implements:` is a fourth concern — vision-rationale dependency — but it's contract-adjacent (it's part of the artifact's external declaration, not internal constraint). It can logically sit alongside `goal`/`scope`/`boundaries` in the contract layer, extending it to include upward dependencies.
- The **population strategy** (G19) is unchanged. `implements:` follows the same phase-gated, regression-driven pattern.
- The **breaking/non-breaking classification** (G18) is unchanged. Modifying an artifact's `implements:` list is a scope change (adding or removing rationale dependencies), which is already covered by existing classification rules.

## 📐 Convergence measurement analysis

### The problem with G7's change-volume metric

G7 proposes: "the system is converging when the volume of proposed changes per cycle decreases monotonically." This metric has three structural weaknesses:

1. **It's an indirect proxy.** Change volume measures *activity*, not *progress*. A cycle with many changes could be converging fast (many needed fixes applied). A cycle with zero changes could be a false positive (the system can't detect the remaining drift).

2. **It requires multi-cycle observation.** You need at least two consecutive cycles to detect divergence — the first to establish a baseline, the second to compare. You need many cycles to distinguish a temporary spike (legitimate response to a trigger) from genuine divergence. This delays detection.

3. **It conflates internally-triggered and cascade-triggered changes.** G7 excludes externally-triggered changes, but the boundary between "external trigger" and "internal cascade" is blurry. A platform release triggers one change, which triggers five cascade validations — are those external or internal? The exclusion rule adds complexity without buying precision.

4. **It doesn't help per-artifact.** Change volume is a system-level aggregate. It can't tell you which specific artifact is diverging or why. When the system detects a volume increase and escalates, the human must manually investigate every proposed change to find the cause.

### The user's insight: distance-based convergence

Instead of counting changes, measure how far each artifact's actual content is from its declared metadata contract. If the distance decreases over cycles, the system is converging. If it increases, it's diverging. If it's zero, the artifact has converged.

This inverts the measurement approach:

| Aspect | Change-volume metric (G7 current) | Distance-based metric (proposed) |
|---|---|---|
| **What it measures** | System activity (count of proposals) | Artifact state (gap between actual and declared) |
| **Granularity** | System-wide aggregate | Per-artifact, per-dimension |
| **Detection latency** | Multi-cycle (needs trend comparison) | Single-cycle (distance is absolute, not relative) |
| **Actionability** | "Something is wrong somewhere" | "Artifact X has a growing gap on dimension Y" |
| **Cost** | Low (count changes) | Medium (evaluate distances) |

### What "distance" means operationally

Distance isn't a single number — it's a multi-dimensional assessment that can be measured at different levels of granularity. The key insight: **the same metadata attributes the self-update system already consumes for gate checks can be reused as convergence anchors.**

#### Per-artifact distance (artifact → its own metadata)

Each artifact declares `goal:`, `scope:`, `boundaries:`, `rationales:` (G17), and `implements:` (G38). Distance measures how well the artifact's actual content fulfills these declarations:

| Dimension | Distance signal | Assessment method | Cost |
|---|---|---|---|
| **Goal alignment** | Does the content achieve what `goal:` declares? | LLM-based (semantic comparison) | Medium |
| **Scope coverage** | Does the content cover all `scope.covers` topics? Are `scope.excludes` topics absent? | Semi-deterministic (topic checklist from scope) | Low-Medium |
| **Boundary compliance** | Does the content stay within `boundaries:` constraints? | Deterministic (token count, SSOT check) | Low |
| **Rationale integrity** | Are all `rationales:` constraints preserved in the content? | Semi-deterministic (constraint validation per G17) | Low-Medium |
| **Implementation fidelity** | Does the content operationalize the rationales listed in `implements:`? | LLM-based (rationale → content mapping) | Medium |

**Not every dimension needs LLM evaluation.** Boundary compliance is deterministic. Scope coverage is semi-deterministic (check whether each declared topic has corresponding content). Only goal alignment and implementation fidelity require LLM judgment — and these can be cached (R-S3-validation-caching) and only re-evaluated when content changes.

#### Per-artifact-set distance (artifacts → their vision)

For a coordinated set of artifacts (e.g., all PE artifacts targeting a single vision document), the distance is aggregated from individual artifact distances. The vision document serves as the reference:

| Level | What it measures | How |
|---|---|---|
| **Individual artifact** | Distance from artifact to its own metadata | Per-dimension assessment (table above) |
| **Artifact set** | Distance from all artifacts to the shared vision | Aggregate individual distances + inter-artifact coherence |
| **Inter-artifact coherence** | Do the artifacts work together toward the vision? | Cross-reference consistency, no contradictions, complete coverage |

The artifact-set distance adds a **coherence component** that per-artifact distance can't capture: two artifacts may each be close to their own goals but contradict each other or leave gaps between them.

#### Dimension-specific distance

The user's question specifically asks whether distance can be evaluated "with respect to some specific dimensions." Yes — and this is where the seven-dimension model (G29) connects:

| Dimension (from G29) | Distance interpretation | Example |
|---|---|---|
| **Structural** | Does the artifact's structure match its declared scope/boundaries? | "Scope declares 7 topics; content covers 5" → distance = 2/7 |
| **Consistency** | Does the content contradict its own rationales or other artifacts? | "Rationale R1 says ordering matters; content doesn't enforce order" → distance = constraint violation count |
| **Efficiency** | Is the artifact within its declared token budget? | "Boundary: max 800 tokens; actual: 950" → distance = 150 tokens over |
| **Effectiveness** | Does the artifact produce the outcomes its goal describes? | Requires observation data — not directly measurable from metadata (effectiveness hard limit still applies) |
| **Referential integrity** | Are all references valid? | "3 broken references out of 20" → distance = 3/20 |
| **Behavioral compatibility** | Do consumers process the artifact correctly? | Schema validation + consumer test cases |
| **Dependency propagation** | (Not a distance dimension — it's a routing modifier) | N/A |

**Effectiveness remains the exception.** The effectiveness hard limit (already in the vision) means effectiveness distance can't be measured from metadata alone. It requires observation data from actual use. This is consistent — distance-based convergence improves measurement for six of seven dimensions while correctly acknowledging that effectiveness needs its own signal path (G24).

### How distance-based convergence replaces change-volume

The convergence criterion rewrites from "decreasing change volume" to "decreasing aggregate distance":

**Current G7 criterion (change-volume):**
> The system is converging when change volume per cycle decreases monotonically. A volume increase is a divergence signal.

**Proposed replacement (distance-based):**
> The system is converging when the aggregate distance (across all artifacts and measured dimensions) decreases or remains stable between consecutive assessment cycles. An artifact whose distance increases on any measured dimension is a **convergence concern** — it gets priority in the next cycle's Assess phase.
>
> **Fixed-point definition** (refined): The system has converged when all measured artifacts report zero distance on all measured dimensions (excluding effectiveness, which requires ongoing observation). New external triggers may increase distance, restarting convergence.

### How distance measurement works in practice

#### Measurement timing

Distance isn't measured on every change — that would be expensive. It's measured at **assessment cycle boundaries** (when the Assess layer runs for a batch of artifacts):

1. **Before a cycle starts:** Measure current distance for artifacts in scope
2. **After changes are applied:** Re-measure distance for changed artifacts
3. **Compare:** Distance decreased → convergence. Distance unchanged → stable. Distance increased → divergence concern.

This integrates naturally with the existing Detect → Assess → Propose → Execute flow. The Assess layer already evaluates artifacts; distance measurement adds structured metrics to what Assess already does.

#### Caching and incremental measurement

Not every dimension needs re-evaluation every cycle:

- **Deterministic dimensions** (boundary compliance, referential integrity): Re-evaluate only when content changes. Cache result with content hash.
- **Semi-deterministic dimensions** (scope coverage, rationale integrity): Re-evaluate when content or metadata changes. Cache per (content hash + metadata hash).
- **LLM-evaluated dimensions** (goal alignment, implementation fidelity): Re-evaluate when content, metadata, or vision changes. Cache per (content hash + metadata hash + vision hash). Reuse R-S3-validation-caching.

This means most distance evaluations in stable periods are cache hits — zero additional cost.

#### Aggregation

Individual dimension distances are combined per artifact, then per artifact set:

**Per-artifact score:** Weighted combination of dimension distances. Weighting follows the dimension importance order from G29 (structural > consistency > efficiency > referential > behavioral). Not a single number — a vector of dimension scores.

**Per-set score:** Aggregate of artifact scores + inter-artifact coherence score (from Stage 2 coherence verification, G35).

**The vector representation is important.** A single "distance number" hides critical information. An artifact that's perfect on structure but violating three rationale constraints is very different from one that's structurally drifted but has intact constraints. The vector preserves dimension-specific actionability.

### What distance-based convergence enables that change-volume doesn't

1. **Targeted remediation.** When distance increases on a specific dimension for a specific artifact, the system knows exactly what to fix. No human investigation needed to find the cause.

2. **Per-artifact convergence tracking.** Some artifacts may converge while others diverge. The system can focus resources on diverging artifacts rather than pausing everything (as G7 currently proposes).

3. **Vision-change impact measurement.** When the vision changes, distance instantly increases for affected artifacts (because the reference moved). The system can quantify the work needed: "This vision change increased aggregate distance by X across Y artifacts."

4. **Incremental progress visibility.** Each cycle's distance reduction is visible — the system can report "Artifact X: goal distance reduced from 0.6 to 0.3 this cycle." This gives humans confidence that the system is working even when it's processing many changes.

5. **Natural stopping criterion.** Zero distance = converged. No need to observe trends over multiple cycles to detect the fixed point.

### What distance-based convergence does NOT replace

- **Oscillation detection (G8):** Distance measurement can detect oscillation (distance goes down → up → down → up), but the dedicated oscillation detection rule (checking the outcome log for apply/rollback patterns) is still valuable as a faster, cheaper signal. Distance catches the effect; oscillation detection catches the mechanism.

- **Iteration budget (G9):** Even with good distance measurement, unbounded changes per cycle are still a cost and blast-radius risk. The budget remains.

- **Cascade damping (G10):** Cascade amplification is an execution-time risk, not a measurement problem. Distance measurement happens at cycle boundaries, not during execution.

- **Effectiveness measurement (G24):** Distance-based convergence covers six of seven dimensions. Effectiveness still requires observation data from actual use. G24's effectiveness signal remains the path for that dimension.

### Impact on G7 and R-G5-convergent-iteration

G7's proposed change needs revision. The "Convergent iteration" design principle and R-G5-convergent-iteration rationale are still valid — the loop must converge — but the **convergence criterion** changes from change-volume to distance-based. The fixed-point definition becomes more precise.

R-G5-convergent-iteration's design implication updates:

| Current implication | Revised implication |
|---|---|
| "Monitor change volume per cycle" | "Monitor per-artifact distance across measured dimensions at cycle boundaries" |
| "Escalate divergence signals (volume increase) to human review" | "Escalate convergence concerns (distance increase on any dimension) to priority assessment" |
| "Fixed-point: zero proposed changes" | "Fixed-point: zero distance on all measured dimensions (excluding effectiveness)" |

**The escalation is less severe.** Under change-volume, any volume increase pauses the entire system. Under distance-based, a distance increase on one artifact's one dimension triggers priority assessment for that artifact — not a system-wide pause. This is more proportionate and less disruptive.

### Gap definitions

**G40: G7's convergence criterion uses indirect proxy (change volume) instead of direct measurement (distance from metadata) (HIGH)**

G7's convergence criterion — "change volume per cycle decreases monotonically" — measures system activity, not progress. It requires multi-cycle observation to detect divergence, provides no per-artifact or per-dimension granularity, and conflates productive changes with churn. The divergence signal (volume increase → system pause) is disproportionate.

**Proposed change.** Replace G7's change-volume convergence criterion with distance-based convergence:

> **Convergence criterion (distance-based).** The system is converging when the aggregate distance from each artifact's content to its declared metadata (goal, scope, boundaries, rationales, implements) decreases or remains stable between consecutive assessment cycles. Distance is measured per-dimension per-artifact as a vector, not a single scalar.
>
> An artifact whose distance increases on any measured dimension between cycles is a **convergence concern** — it receives priority in the next cycle's Assess phase. A convergence concern is NOT a system-wide pause (unlike G7's original divergence signal). It's a targeted escalation for the specific artifact and dimension.
>
> Distance dimensions: structural (scope coverage), consistency (rationale integrity), efficiency (boundary compliance), referential integrity (reference validity), behavioral compatibility (consumer processing). Effectiveness is excluded — it requires observation data (G24), not metadata comparison.
>
> **Fixed-point definition (refined).** The system has converged when all measured artifacts report zero distance on all measured dimensions. New external triggers or vision changes increase distance, restarting convergence for affected artifacts.

**Measurement cost:** Deterministic dimensions (boundary, referential) are O(1) with content-hash caching. Semi-deterministic dimensions (scope, rationale) use metadata-hash caching. LLM-evaluated dimensions (goal alignment, implementation fidelity) are cached per (content + metadata + vision) hash triple. Most evaluations in stable periods are cache hits.

**Target section:** G7's convergence criterion in the "Convergent iteration" design principle and success criteria.

**Connects to:** G7 (the criterion being replaced), G17 (rationale metadata provides distance anchors), G38 (`implements:` provides implementation-fidelity distance anchors), G24 (effectiveness dimension remains observation-dependent), G35 (Stage 2 coherence provides inter-artifact distance signal), R-G5-convergent-iteration (rationale's design implications update), R-S1-metadata-driven (metadata is the distance reference), R-S3-validation-caching (distance evaluations are cacheable).

**G41: No artifact-set convergence measurement (distance aggregation + inter-artifact coherence) (MEDIUM)**

Individual artifact distances don't capture whether a coordinated set of artifacts is converging toward a shared vision. Two artifacts may each be close to their own goals but contradict each other, leave coverage gaps, or duplicate content. Set-level convergence requires aggregation plus an inter-artifact coherence signal.

**Proposed change.** Add artifact-set distance measurement to the convergence model:

> **Artifact-set convergence.** For artifact sets that share a vision document (e.g., all PE artifacts implementing the self-update vision), convergence is measured at two levels:
>
> 1. **Aggregate distance:** Sum of individual artifact distance vectors. Tracks whether the set as a whole is moving toward the vision.
> 2. **Inter-artifact coherence:** Whether the artifacts work together without contradictions, gaps, or redundancy. This is the same signal Stage 2 (G35) coherence verification produces — reused as a convergence input.
>
> Set-level convergence requires both: aggregate distance decreasing AND coherence stable or improving. A set where individual distances decrease but coherence degrades is **locally converging but globally diverging** — the artifacts are getting better individually but worse collectively.

**Target section:** Adjacent to the convergence criterion in the "Convergent iteration" design principle.

**Connects to:** G40 (per-artifact distance — the individual input), G35 (Stage 2 coherence — the inter-artifact signal), R-G5-convergent-iteration (extends the rationale to set-level properties).

**G42: R-G5-convergent-iteration's design implications reference change-volume metric (needs update for distance-based) (LOW)**

R-G5-convergent-iteration's design implication says "Monitor change volume per cycle." With G40's distance-based convergence, this updates to "Monitor per-artifact distance across measured dimensions at cycle boundaries." The rationale's purpose (loop-level convergence governance) is unchanged; only the measurement mechanism changes.

>
> **To:** Measure per-artifact distance from content to declared metadata (goal, scope, boundaries, rationales, implements) at cycle boundaries. Define convergence as decreasing distance, fixed-point as zero distance (excluding effectiveness). Budget iterations to bound cost and blast radius. Escalate convergence concerns (distance increase) to priority assessment for specific artifacts.
