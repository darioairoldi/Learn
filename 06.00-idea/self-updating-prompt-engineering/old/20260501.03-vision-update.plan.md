# Vision Update Plan: Progressive Risk Classification

> Extend R-S6 (tier-blast-radius) and the Assess layer to acknowledge the current model's limitations and define a phase-gated upgrade path.

---

## Problem analysis

The vision's current risk classification uses two inputs:

1. **Tier** — O(1) lookup from filename prefix (e.g., `00.00-` → Tier 0)
2. **Dependent count** — `grep_search` count of references

### Three weaknesses

| # | Weakness | Impact | Example |
|---|---|---|---|
| **W1** | Tier is a naming convention, not a measurement | Misclassified file gets wrong autonomy level | A Tier 3 file with 50 transitive dependents treated as low-risk |
| **W2** | All references treated as equal weight | Blast radius over/under-estimated | Informational mention counted same as behavioral enforcement |
| **W3** | Transitive dependencies invisible | Blast radius underestimated by chain depth | Context A → Agent B → Prompt C: change to A doesn't show C as affected |

### Why it matters now

At Phase 1 (manual invocation), these weaknesses are tolerable — humans review every change. At Phase 2+ (autonomous YAML changes), the system makes go/no-go decisions based on blast radius. Wrong blast radius → wrong autonomy level → either blocked safe changes (friction) or auto-applied dangerous ones (risk).

---

## Alternative approaches evaluated

| Approach | How it works | Accuracy | Cost | Phase fit |
|---|---|---|---|---|
| **Current** (tier + count) | Filename prefix + grep count | LOW | O(1) + O(n) | Phase 1 ✅ |
| **A: Weighted graph** | Classify each dependency by coupling strength (enforces=3, structures=2, references=1, mentions=0) | MEDIUM | O(n) with parsing | Phase 2 |
| **B: Change × type matrix** | Pre-defined risk lookup by change-type + artifact-type combination | MEDIUM | O(1) | Phase 1+ |
| **C: Empirical tracking** | Track actual blast radius per change, learn patterns over time | HIGH (over time) | O(1) + logging | Phase 2+ |
| **D: Hybrid 3-level** | Topology (deterministic) + coupling classification (metadata) + semantic analysis (LLM for high-dep) | HIGH | O(n) + O(k) LLM | Phase 2-3 |

### Recommendation: Hybrid with phase gating

| Phase | Risk assessment | Method | New vs. current |
|---|---|---|---|
| **Phase 1** | Tier + count + change × type matrix | Static, cheap | Add matrix overlay |
| **Phase 2** | + actual topology from metadata | Build dependency graph from `📖`, `handoffs:`, `context_dependencies:`, `applyTo` | Replace grep count with graph traversal |
| **Phase 3** | + weighted coupling analysis | Classify each dependency as behavioral/structural/informational | Refine blast radius per change |
| **Phase 4** | + empirical calibration | Outcome data tunes predictions | Self-improving accuracy |

## Proposed vision changes (v7 → v8 candidates)

### Change V1 (HIGH): Extend R-S6 with progressive maturity model

**Location**: Vision § The rationale → System infrastructure → R-S6-tier-blast-radius

**Current text**:
> **R-S6-tier-blast-radius.** Tier encodes how many layers can depend on an artifact. [...] Use tier as the primary sort key in risk classification. Tier 0 → always escalate. Tier 5 → presumptively autonomous. Tiers 1–4 → route by breaking/non-breaking classification and chain coverage. Group snapshots for rollback are tier-scoped.

**Proposed addition** (append to design implication):
> Risk classification matures with the rollout phases. At Phase 1, tier (from filename prefix) and dependent count (from grep) provide a coarse but cheap estimate. At Phase 2, actual dependency topology (from `context_dependencies:`, `handoffs:`, `📖` references, and `applyTo` patterns) replaces the grep count with graph-based reachability. At Phase 3, each dependency is classified by coupling strength: behavioral (enforces rules from), structural (parses structure of), informational (mentions), or cosmetic (documents). The blast radius estimate improves at each phase without requiring the prior phase's method to be discarded — each phase adds a refinement layer.

**Classification**: EXTENDING — preserves current approach as Phase 1 baseline.

Tier is a **heuristic proxy** for dependency depth — it approximates blast radius from naming convention rather than measuring actual topology. This is acceptable at Phase 1 (manual invocation) where humans validate every decision. At Phase 2+, tier should be cross-validated against the actual dependency graph — a tier mismatch (filename says Tier 3 but graph shows Tier 1 reachability) is a finding.

### Change V3 (MEDIUM): Add coupling-aware dependent analysis

**Location**: Vision § Assess layer

**Current text**:
> 2. **Dependent count (R-S2-dependency-graph) — secondary.** Within a tier, more connections mean wider blast radius. High-dependency artifacts get broader validation.

**Proposed replacement**:
> 2. **Dependent analysis (R-S2-dependency-graph) — secondary.** Within a tier, dependency characteristics determine blast radius scope. At Phase 1, raw dependent count (grep-based) provides a coarse signal — more connections mean wider blast radius. At Phase 2+, dependencies are classified by coupling type:
>
> | Coupling type | Weight | Detection | Example |
> |---|---|---|---|
> | **Behavioral** | High | Agent/prompt loads and enforces rules from this artifact | Agent's `📖` reference + rule citation in boundaries |
> | **Structural** | Medium | Consumer parses specific sections or fields | Template expects specific headings; `applyTo` matches |
> | **Informational** | Low | Consumer mentions for context but doesn't depend on content | "See X for details" in References section |
> | **Cosmetic** | None | Consumer mentions in comments or version history | "Based on findings from X" |
>
> Weighted coupling replaces raw count for blast radius estimation. A single behavioral dependent carries more risk than five informational mentions. Transitive dependencies (A→B→C) are discovered through graph traversal of the dependency map, with impact attenuating by depth (direct = full weight, one-hop = half, two-hop = quarter).

**Classification**: EXTENDING — preserves count as Phase 1 baseline, adds structure for Phase 2+.

- [x] Apply to vision

**Status**: ✅ DONE (2026-05-01). Replaced dependent count with coupling-aware analysis table + transitive dependency attenuation.

### ~~Change V4 (DROPPED): Add change × type matrix to Assess layer~~

**Decision**: DROPPED after analysis. The static matrix is resolution-blind (can't distinguish a rule clarification from a rule reversal), requires manual maintenance for new artifact/change types, and provides a coarser, less accurate version of what N-1 block classification + LLM fallback already handle. V1–V3 strengthen actual blast radius measurement; V4 would add a static approximation that's worse than what already exists.

- Topology (V3): coupling-aware blast radius per dependent

### Change V5 (LOW): Update overview article to reflect progressive model

**Location**: Article #30, § Assess subsection

**Current text** (lines 122-124):
> Risk classification uses two inputs: **tier** (O(1) lookup — how many layers can depend on this?) and **dependent count** (how many artifacts actually reference this?). Together they determine blast radius.

**Proposed replacement**:
> Risk classification uses a **progressive model** that matures with the rollout phases. At Phase 1, two coarse inputs provide the baseline: **tier** (which layers can depend on this artifact?) and **dependent count** (how many artifacts reference this?). At Phase 2+, the system upgrades to actual dependency topology with coupling-aware analysis — distinguishing behavioral dependencies (an agent enforcing rules from a context file) from informational mentions (a "see also" in a references section). This progressive refinement improves blast radius accuracy without requiring Phase 3 infrastructure at Phase 1.

**Classification**: Article change — reflects vision extension.
