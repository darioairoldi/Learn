# Plan A (v4): pe-meta Self-Contained Approach

> pe-meta has its OWN prompts for every artifact type. pe-meta agents implement ALL logic internally. pe-meta supports **selective review dimensions** — each run targets specific quality concerns, not a full scan. Cost is bounded by combining targeted scope with model routing. This version integrates the vision change proposals from Plan 01 and the domain-focus rationale from Plan 03.

---

## Design principles

### pe-meta owns PE-for-PE completely

pe-gra and pe-con continue to serve **domain artifacts** (documentation, article-writing, events, etc.). pe-meta serves **PE-for-PE artifacts exclusively** — with its own complete set of prompts and agents.

### Not every run checks everything

Like the documentation manager agents (which can update "just the architecture section" or "just the security posture" without re-scanning the entire repository), pe-meta review prompts support **selective dimensions**. Each invocation targets specific quality concerns:

```
/pe-meta-agent-review pe-meta-validator.agent.md                     ← full review
/pe-meta-agent-review pe-meta-validator.agent.md --dim consistency   ← only check non-contradiction
/pe-meta-agent-review pe-meta-validator.agent.md --dim token-budget  ← only check token limits
/pe-meta-agent-review pe-meta-validator.agent.md --dim staleness     ← only check source freshness
```

This is the key cost containment mechanism: a full review loads all checklists and assesses all dimensions; a targeted review loads only the relevant dimension's checklist and skips everything else.

### Model routing for cost efficiency

Different review dimensions have different computational requirements:

| Dimension type | Model requirement | Cost |
|---|---|---|
| **Deterministic** (link integrity, metadata, token count) | Small model or deterministic tool | Near-zero |
| **Structural** (YAML, boundaries, tool alignment) | Standard model | Low |
| **Semantic** (consistency, clarity, adherence) | Reasoning model | Medium |
| **Strategic** (vision alignment, quality bar, self-update readiness) | Reasoning model | Medium-high |
| **Cross-artifact** (coherence, redundancy, priority gaps) | Reasoning model + multiple file reads | High |

pe-meta prompts specify which model class each dimension requires. When a targeted review selects a deterministic dimension, the system can route to a cheaper model or skip LLM entirely.

This means:
- pe-meta has its own design/create/review prompts for each artifact type
- pe-meta agents embed ALL validation logic for PE artifacts (structural + strategic)
- pe-meta never hands off to pe-gra validators — it handles everything internally
- pe-gra/pe-con are free to evolve for domain needs without affecting pe-meta

#### D23: Reference efficiency (detail)

**Vision rationale**: R-P2 (decomposition), R-S3 (validation caching)

**What D23 checks:**
1. **Reference count** — How many `📖` and category references does the artifact declare? Flag agents/prompts with >8 references (potential over-loading)
2. **Load necessity** — For each reference: is it actually used in a process phase, boundary, or checklist? References in "Always Do" that no process phase mentions are dead weight — loaded into context but never consumed
3. **Load placement** — Are references placed in the specific phase that needs them, or dumped upfront in "Always Do"? Phase-specific placement enables load-on-demand; upfront placement loads everything regardless of the invocation path
4. **Conditional loading** — Does the artifact support skipping references when they're not needed? (e.g., "Load vision document only for `--dim vision-alignment`" vs. "Always load vision document")

**Example findings:**
- "pe-meta-validator has 21 `📖` references. 6 are used only in ecosystem audit mode but loaded on every invocation → move to the audit-mode section"
- "pe-gra-agent-builder references `agent-patterns` files in 'Always Do' but only uses them in Phase 0 handoff validation → move reference to Phase 0"
- "pe-meta-review loads the vision document on every invocation but only uses it for `--dim vision-alignment` → load conditionally"

**Output format:**
```markdown
## D23: Reference efficiency report

| Artifact | Ref count | Used refs | Unused refs | Upfront refs | Phase-specific refs |
|---|---|---|---|---|---|
| pe-meta-validator | 21 | 15 | 6 | 18 | 3 |
| pe-gra-agent-builder | 7 | 7 | 0 | 5 | 2 |

### Optimization opportunities
| Artifact | Reference | Current placement | Recommended | Savings |
|---|---|---|---|---|
| pe-meta-validator | vision document | Always Do | `--dim vision-alignment` only | ~800 tokens/invocation |
```

#### D24: Handoff efficiency (detail)

**Vision rationale**: R-P2 (decomposition), R-P3 (structured contracts)

**What D24 checks:**
1. **Max token enforcement** — Does every handoff contract specify a max token budget? (already partially in handoff data contracts from Round 1, but not verified as an efficiency concern)
2. **Summarization at transitions** — Does the prompt define summarization protocols between phases? (e.g., "Before handoff, compress all prior phases to ≤500 tokens") Without summarization, each phase inherits the full context of all previous phases → context window exhaustion
3. **Structured over raw** — Do handoffs pass structured data (tables, YAML, JSON) or raw prose? Structured data is cheaper to parse and produces more predictable downstream behavior
4. **Information funnel** — Does the token budget decrease along the chain? (researcher 2000 → builder 1500 → validator 1000) A flat or increasing budget suggests insufficient summarization

**Example findings:**
- "pe-gra-agent-design defines 6 phases but no summarization protocol → by Phase 6, estimated context is ~12,000 tokens (exceeds 8,000-token threshold)"
- "Handoff from researcher to builder passes raw research report (2,000 tokens) when a structured spec (~800 tokens) would suffice"
- "Builder→validator handoff sends full build log when only file path + change summary is needed"

**Output format:**
```markdown
## D24: Handoff efficiency report

| Transition | Max tokens | Summarization? | Format | Assessment |
|---|---|---|---|---|
| Researcher → Builder | 2000 | ✅ Template-based | Structured | ✅ Efficient |
| Builder → Validator | 1500 | ❌ Raw handoff | Prose | ⚠️ Could be 200 (path + summary) |
| Phase 3 → Phase 4 | No limit | ❌ No summarization | Full context | ❌ Context bloat risk |
```

#### D25: Processing efficiency (detail)

**Vision rationale**: R-P5 (progressive depth), R-S3 (validation caching), R-L3 (model specialization)

**What D25 checks for prompt artifacts:**
1. **Targeted review support** — Does the prompt support scoped invocation? (e.g., `--dim`, `--with-deps`, type-specific scope) Or does every invocation run the full pipeline? Prompts without scope support force full reviews even for minor changes
2. **Summarization protocol** — Does the prompt define a summarization checkpoint table? (e.g., "After Phase 1, summarize to ≤500 tokens") The vision's 8,000-token threshold triggers mandatory summarization — but does the prompt actually implement it?
3. **Model routing** — Does the prompt specify which model class each phase or dimension needs? Without model routing, every step uses the most expensive model even when a cheaper one would suffice
4. **Change-scoped processing** — Does the prompt support Phase 0.5 (change impact analysis) that skips irrelevant checks? Or does it always run the full checklist regardless of what changed?
5. **Early exit** — Does the prompt define conditions where it can stop early? (e.g., "If D1 metadata check fails with CRITICAL, skip remaining dimensions and report immediately")

**What D25 does NOT check:**
- Does NOT assess individual dimension efficiency (that's each dimension's own concern)
- Does NOT check artifact content quality (that's D6-D11)
- Does NOT modify the prompt (plan mode)

**Example findings:**
- "pe-meta-review has no `--dim` support → every invocation runs all 6 strategic criteria even when user only needs vision alignment"
- "pe-gra-agent-design has no summarization protocol → by Phase 7, estimated accumulated context is ~15,000 tokens"
- "pe-meta-update has no model routing → staleness checks (could use small model) use the same reasoning model as consistency analysis"
- "pe-gra-context-review has Phase 0.5 change scoping → ✅ supports targeted review when invoked post-change"

**Output format:**
```markdown
## D25: Processing efficiency report

| Feature | pe-meta-review | pe-gra-agent-design | pe-meta-update |
|---|---|---|---|
| Targeted scope (`--dim`) | ❌ Not supported | ❌ Not supported | ✅ healthcheck/fullcheck |
| Summarization protocol | ❌ No checkpoint table | ❌ No checkpoint table | ⚠️ Implicit only |
| Model routing | ❌ Single model | ✅ claude-opus-4.6 specified | ❌ Single model |
| Change-scoped (Phase 0.5) | ❌ Always full | ✅ Phase 0.5 present | ✅ Research phase |
| Early exit conditions | ❌ None defined | ❌ None defined | ⚠️ CRITICAL abort only |
```

#### D21: Deterministic-first (detail)

**Vision rationale**: R-P1 (deterministic processing for predictable operations)

**Critical constraint**: Deterministic processing **augments** reasoning — it never replaces it for reduced scope. Whatever deterministic processing doesn't cover, reasoning still handles. D21 proposes splitting steps, not removing LLM coverage.

**What D21 checks for each process phase in an agent/prompt:**
1. Read each process step's description
2. Identify steps currently described as single LLM tasks
3. For each: assess whether part of the work is predictable (YAML parsing, grep, file existence, pattern matching, counting)
4. If yes: propose a **split** — deterministic pre-processing + LLM reasoning on the remainder
5. Report: which steps could benefit from deterministic acceleration, and what the split looks like

**Example findings:**
- "Phase 3 Step 2: 'Check tool alignment' — YAML extraction (`tools:` array, `agent:` mode) is deterministic. Semantic assessment ('is this tool appropriate for this agent's purpose?') still needs reasoning. Recommend: deterministic extraction → feed structured data to reasoning step."
- "Phase 0.5: COSMETIC detection can be partially deterministic (diff contains only whitespace/punctuation → deterministic COSMETIC classification). If diff contains content changes → still requires LLM classification into STRUCTURAL/VOCABULARY/BEHAVIORAL."
- "Phase 1: 'Read and assess the artifact' — file reading is already deterministic via `read_file`. No split needed."

**What D21 does NOT do:**
- Does NOT replace any LLM step with a deterministic-only step (scope must not be reduced)
- Does NOT flag steps where deterministic processing is already used (no false positives on well-designed agents)
- Does NOT propose removing LLM validation passes (R-L2 self-correction is never skippable)

**Output format:**
```markdown
## D21: Deterministic-first optimization opportunities

| Phase | Step | Current | Deterministic portion | Reasoning remainder | Recommendation |
|---|---|---|---|---|---|
| Phase 3 | Tool alignment | Single LLM step | Parse YAML `tools:` + mode | "Is tool X appropriate?" | Split: parse → reason |
| Phase 0.5 | Change classification | LLM for all types | Whitespace-only diff → COSMETIC | Content-changing diffs | Pre-filter COSMETIC |
```

#### D22: Context file optimization (detail)

**Vision rationale**: R-S9 (guidance quality), R-S10 (quality propagation), R-S8 (instruction minimization)

**Scope**: This is a **system-level** dimension — it operates on the context file SET, not individual files. Individual file quality is covered by D6–D11. D22 checks whether the set is optimally organized for its consumers.

**What D22 checks:**
1. **Layer correctness** — Is content in the right artifact type?
   - Testable/mechanical rules in context files → should be in instruction files (R-S8)
   - Behavioral guidance in instruction files → should be in context files
   - Type-specific logic in context files → should be in agent bodies
   - Report: per-rule layer recommendation

2. **Granularity assessment** — Are files at the right size?
   - Count files in the context folder
   - Flag folders with >12 files (too fragmented — high reference overhead)
   - Flag folders with <3 files (too consolidated — context rot risk)
   - Flag individual files >2,500 tokens (budget violation) or <200 tokens (too small — merge candidate)

3. **Category mapping integrity** — Does STRUCTURE-README match reality?
   - For each category: do the mapped files exist?
   - For each file: is it in at least one category?
   - Are there orphan files (exist but no category maps to them)?
   - Are there empty categories (category exists but maps to zero files)?

4. **Consumer load analysis** — What do consumers actually need?
   - For each consumer (agent/prompt): which context files does it load?
   - Which files are loaded by >10 consumers? (high-value — extra quality investment warranted)
   - Which files are loaded by 0 consumers? (orphan — candidate for removal or merge)
   - Which consumers load >5 context files? (heavy chain — token budget pressure)

5. **Topic coherence** — Does each file cover one coherent topic?
   - Files whose sections span unrelated topics → split candidate
   - Multiple files covering overlapping topics → merge candidate
   - Cross-reference: does the file's `scope.covers` match its actual content headings?

6. **Freshness distribution** — Are some files significantly older than others?
   - Compare `last_updated` dates across the set
   - Flag files >6 months older than the set median (potential staleness cluster)

**What D22 does NOT do:**
- Does NOT check individual rule clarity, consistency, or completeness (that's D6-D11)
- Does NOT check adherence (that's D16)
- Does NOT modify files (plan mode)

**Output format:**
```markdown

### Layer correctness
| File | Rule | Current layer | Recommended layer | Reason |
|---|---|---|---|---|
| 01.04 | "plan=read-only" | Context | Instruction | Testable: boolean check |

| Finding | Files | Recommendation |
|---|---|---|
| Folder has 33 files | .copilot/context/00.00-pe/ | Consider merging related files |
| File too small (180 tokens) | 04.01-validation-caching.md | Merge into parent topic |

### Consumer load
| Consumer | Files loaded | Total tokens | Status |
|---|---|---|---|
```

#### D26: Model routing correctness (detail)

**Vision rationale**: R-L3 (model specialization), R-P1 (deterministic processing)

**Why model routing is both efficiency AND quality:**
- **Efficiency**: reasoning model for YAML parsing = ~10× token cost vs. deterministic tool
- **Quality**: standard model for semantic contradiction detection = shallower analysis, more false negatives
- **Reliability**: small model for vision alignment = inconsistent results, missed rationale violations

**What D26 checks:**
1. **Model declaration** — Does the artifact specify `model:` in YAML? If not, it inherits the default (which may be suboptimal)
2. **Task-model alignment** — For each process step/phase, assess whether the described work matches the model class:

   | Task type | Correct model class | Wrong routing signal |
   |---|---|---|
   | YAML parsing, file existence, counting, grep | Deterministic tool / small model | Step says "analyze YAML" but uses reasoning model |
   | Pattern matching, template filling, structural comparison | Standard model | Step says "apply template" but uses reasoning model |
   | Semantic comparison, contradiction detection, clarity assessment | Reasoning model | Step says "check for contradictions" but no reasoning model available |
   | Multi-step analysis, vision alignment, cross-artifact coherence | Reasoning model | Step says "assess vision alignment" but uses standard model |
   | Web research, release notes analysis | Reasoning + fetch_webpage | Step says "analyze release notes" but no web access tool |

3. **Phase-level routing** — For multi-phase prompts: does each phase specify or imply the right model class? A prompt with `model: claude-opus-4.6` for ALL phases is over-provisioning deterministic steps
4. **Handoff model transitions** — When a prompt hands off to an agent, does the agent's declared model match the task's requirements?

**Example findings:**
- "pe-meta-validator specifies `model: none` (plan mode) but Phase 2 strategic validation requires reasoning → model should be specified for strategic phases"
- "pe-gra-agent-design uses `model: claude-opus-4.6` for all 8 phases including Phase 4 (file creation) which is mechanical → could use standard model for build phase"
- "pe-meta-researcher specifies `model: plan` with `fetch_webpage` for staleness checks (correct) but D12/D13 dimensions also need reasoning model for source content comparison (not just HTTP HEAD)"

**Output format:**
```markdown
## D26: Model routing report

| Phase/Step | Task type | Current model | Recommended | Impact |
|---|---|---|---|---|
| Phase 0: Load context | File reading | Reasoning | Deterministic | Token savings: ~200/invocation |
| Phase 2: Strategic checks | Semantic analysis | Reasoning | ✅ Correct | No change needed |
| Phase 4: Build file | Template application | Reasoning | Standard | Token savings: ~500/invocation |
| Phase 0.5: COSMETIC check | Diff analysis | Reasoning | Deterministic pre-filter | Token savings: ~300/invocation |
```

#### D27: Model-specific guidance adherence (detail)

**Vision rationale**: R-L3 (model specialization), R-S9 (guidance quality)

**Why model-specific adherence matters for quality, efficiency, AND reliability:**
- **Quality**: Claude responds better to XML tags and explicit role prompting; GPT responds better to JSON schema and system/user separation. Using the wrong patterns = degraded output quality
- **Efficiency**: Model-aligned prompts produce correct output in fewer tokens (less "try again" reasoning)
- **Reliability**: Model-aligned prompts produce more consistent results across invocations

**Source guidance for this dimension:**
- `.copilot/context/00.00-prompt-engineering/03.02-model-specific-optimization.md` — cross-model patterns and optimization strategies
- `03.00-tech/05.02-prompt-engineering/04-howto/08.02-appendix-anthropic-prompting-guide.md` — Anthropic Claude best practices
- `03.00-tech/05.02-prompt-engineering/04-howto/08.01-appendix-openai-prompting-guide.md` — OpenAI GPT best practices
- `03.00-tech/05.02-prompt-engineering/04-howto/08.03-appendix-google-prompting-guide.md` — Google Gemini best practices

**What D27 checks:**
1. **Model identification** — Read `model:` from YAML → identify the model family (Anthropic / OpenAI / Google / unspecified)
2. **Pattern matching** — Load the model-specific guidance for the identified family. Check the artifact's structure against recommended patterns:

   **For Anthropic Claude:**
   | Pattern | Check | Example |
   |---|---|---|
   | Role prompting | Does the artifact start with a clear role definition? | "You are a **validation specialist**..." |
   | Structured output | Does the artifact use markdown tables/code blocks for structured output? | Output templates with `\`\`\`markdown` blocks |
   | Explicit reasoning | Does the artifact ask for step-by-step reasoning where appropriate? | "Classify each change by..." |
   | Boundary enforcement | Does the artifact use MUST/NEVER language? | ✅ Always Do / 🚫 Never Do sections |
   | Long context handling | For large artifacts: does it use progressive sections? | Relevant checks grouped before less-relevant ones |

   **For OpenAI GPT:**
   | Pattern | Check | Example |
   |---|---|---|
   | System/user separation | Is the artifact structured as system prompt + user prompt? | YAML `agent:` field implies system prompt |
   | JSON mode | For structured output: does the artifact request JSON format? | Response format specification |
   | Few-shot examples | Does the artifact provide examples for complex tasks? | Embedded test scenarios serving as few-shot |

3. **Anti-pattern detection** — Flag patterns that work well on one model but poorly on the target:
   - Claude: avoid "Do not..." phrasing (less effective than "NEVER...")
   - GPT: avoid overly long system prompts without JSON anchoring
   - Both: avoid mixing XML tags with markdown formatting inconsistently

4. **Cross-model portability assessment** — If the artifact might run on different models (no `model:` specified): does it use model-agnostic patterns that work reasonably across families?

**Example findings:**
- "pe-meta-review specifies `model: claude-opus-4.6` (Anthropic). Artifact uses MUST/NEVER boundary language (✅ Claude-aligned). But Phase 4 output template uses prose format where a structured markdown table would better leverage Claude's structured output capabilities"
- "pe-gra-agent-builder has no `model:` field → runs on default. Artifact uses patterns that work across models (✅ portable). However, if targeting Claude specifically, the process phases could benefit from explicit reasoning chains ('First check X, then assess Y, then conclude Z')"
- "pe-meta-update specifies `model: claude-opus-4.6` but uses 'Do not skip validation' in boundaries (anti-pattern for Claude → change to 'NEVER skip validation')"

**Output format:**
```markdown
## D27: Model-specific adherence report

**Target model**: Claude Opus 4.6 (Anthropic)
**Guidance source**: 03.02-model-specific-optimization.md + 08.02-appendic_anthropic_prompting_guide.md

| Pattern | Status | Finding |
|---|---|---|
| Role prompting | ✅ Aligned | Clear role definition in opening paragraph |
| Boundary language | ⚠️ Partial | 2 boundaries use "Do not..." instead of "NEVER..." |
| Structured output | ✅ Aligned | Uses markdown tables for reports |
| Reasoning chains | ❌ Gap | Phase 2 asks for assessment without step-by-step structure |
```

#### Why `--dim efficiency` includes quality dimensions

Efficiency isn't just "how many tokens does this file use?" — it's "does this artifact minimize wasted work at every level?" Several quality dimensions have direct efficiency implications:

| Dimension in efficiency group | Efficiency reason |
|---|---|
| D3 (token-budget) | Oversized artifact → wasted tokens on every load |
| D4 (tool-alignment) | Wrong tools → wasted capability slots; misaligned mode → wasted write-tool context |
| D7 (non-redundancy) | Duplicate rules → double the tokens for the same guidance |
| D9 (clarity) | Ambiguous rules → wasted LLM reasoning to interpret on every invocation |
| D11 (actionability) | Non-actionable rules → tokens consumed but produce no behavioral change |
| D20 (token-chain) | Heavy consumer chain → context window exhaustion |
| D21 (deterministic-first) | LLM doing deterministic work → wasted reasoning tokens |
| D23 (reference-efficiency) | Unnecessary references → loaded but unused context |
| D24 (handoff-efficiency) | Unsummarized handoffs → context bloat between phases |
| D25 (processing-efficiency) | No targeted review support → full scan on every invocation |
| D26 (model-routing) | Wrong model class per step → wasted tokens OR degraded quality |

**The efficiency group answers**: "Is this artifact causing unnecessary work — for itself, its consumers, or the review system that maintains it?"

**Note on D27 (model-adherence)**: D27 is in the `--dim quality` group (not efficiency) because its primary impact is on output quality and reliability. However, model-aligned prompts also produce correct output in fewer tokens — so there's an efficiency benefit. Use `--dim model` to run both D26 and D27 together when optimizing model usage.

## What each type-specific prompt does

### Design prompts (`/pe-meta-{type}-design`)

Orchestrates multi-phase creation:
1. Requirements gathering with use-case challenge
2. Pattern research (reads existing PE artifacts of same type)
3. Structure definition (applies exemplary quality bar from vision)
4. Construction with guidance quality invariants (v10 § Guidance quality by construction)
5. Validation (structural + strategic + PE quality bar)
6. Dependency analysis + optional recursive creation

**Vision v10 coverage**: R-S9 guidance quality (construction invariants for context files), R-S10 quality propagation (bottom-up ordering), R-S4 role declaration, PE exemplary quality bar.

### Create/Update prompts (`/pe-meta-{type}-create-update`)

Direct build without full research:
1. Load artifact + identify dependencies
2. Apply pre-change compatibility gate (v10 § Metadata-guarded changes)
3. Build/update with construction invariants
4. Post-change reconciliation (mandatory)
5. Validate (structural + strategic)

**Vision v10 coverage**: R-S1 metadata-driven, pre-change guard, post-change reconciliation, breaking/non-breaking classification.

### Review prompts (`/pe-meta-{type}-review`)

Two modes per review prompt:

**Individual mode** (default):
1. Structural validation (YAML, tools, boundaries, references, token budget)
2. Strategic validation (vision alignment, PE quality bar, category refs, N-1, self-update readiness)
3. Report with severity-ranked findings

**Dependency-aware mode** (`--with-deps`):
1. All individual checks on the target artifact
2. Identify dependencies (`📖` refs, `context_dependencies:`, `handoffs:`, `applyTo` matches)
3. For each dependency: run individual validation (recursive — the dependency's own review)
4. Cross-dependency coherence (pe-meta exclusive checks):
   - Do loaded context files contradict each other?
   - Does the artifact implement ALL rules from its dependencies?
   - Are priorities explicit where dependencies could conflict?
   - Consumer burden: does the artifact load more guidance than it uses?
5. Adherence verification: does the artifact's behavior match the rules it loads?
6. Combined report: per-artifact + per-dependency + cross-dependency + adherence

**Vision v10 coverage**: R-S9 guidance quality (6 properties on context deps), R-S10 quality propagation (bottom-up assessment), adherence verification, cross-dependency coherence.

|---|---|

### pe-meta-validator (ENHANCED)

Currently delegates structural validation to pe-gra validators. In the self-contained approach, pe-meta-validator handles EVERYTHING:

**Structural checks** (currently in pe-gra validators — now internalized):
- YAML compliance per artifact type
- Tool alignment (plan=read-only, agent=read+write)
- Tool count (3-7)
- Boundary completeness (≥5/2/3 for PE — exemplary bar)
- Reference integrity (📖, markdown links, slash-commands)
- Token budget compliance
- Convention compliance (naming, required sections)

**Strategic checks** (already in pe-meta — unchanged):
- Vision alignment (R-S1 through R-G3 applicability)
- Category reference compliance (Level 1.5)
- N-1 structural separation
- Self-update readiness (Phase 1-4 criteria)
- Ecosystem impact (blast radius)
- Guidance quality (6 properties — for context file reviews)

**Cross-artifact checks** (new — for dependency-aware mode):
- Cross-dependency coherence
- Adherence verification (does artifact implement loaded rules?)
- Priority gap detection
- Consumer burden analysis

**How it handles 8 artifact types without 8 separate agents**:

The validator uses a **dispatch pattern** — it reads the artifact type from the file path/extension and applies the type-specific checklist:

```
if .copilot/context/ → context validation checklist
if .agent.md → agent validation checklist  
if .prompt.md → prompt validation checklist
if .instructions.md → instruction validation checklist
if SKILL.md → skill validation checklist
if .template.md → template validation checklist
if .json in hooks/ → hook validation checklist
if prompt-snippets/ → snippet validation checklist
```

This is the same dispatch pattern pe-con-validator already uses. The difference: pe-meta-validator adds strategic + cross-artifact checks on top.

**Token budget concern**: Embedding all 8 type-specific checklists in one agent body is ~400-500 lines. This exceeds the 300-line guideline but is within the 3,000-token meta-prompt budget (v10 § system parameters). The dispatch pattern keeps each invocation focused — the validator only reads the relevant checklist, not all 8.

## How dependency-aware review works (recursive)

```
User: /pe-meta-agent-review --with-deps pe-meta-validator.agent.md

pe-meta-agent-review orchestrates:

Step 1: Individual validation of target
  → pe-meta-validator: validate pe-meta-validator.agent.md
  ← structural + strategic findings

Step 2: Identify dependencies
  Read 📖 refs, context_dependencies, handoffs
  → discovers: validation-rules category, tool-alignment category,
    pe-agents.instructions.md, pe-prompt-engineering-validation skill

Step 3: Validate each dependency individually (recursive)
  For each dependency:
    → pe-meta-validator: validate THIS dependency
    ← structural + strategic findings for the dependency
    (If dependency has its own deps AND --with-deps is deep,
     recurse — but cap at depth 2)

Step 4: Cross-dependency coherence (pe-meta-validator)
  Load all validated dependencies together:
  - Do they contradict each other?
  - Does the target implement all rules from each dependency?
  - Are priorities explicit where deps could conflict?
  - Is the target loading more guidance than it uses?

Step 5: Combined report
  - Target findings
  - Per-dependency findings  
  - Cross-dependency coherence findings
  - Adherence matrix
```

**The recursion is bounded**: dependency depth capped at 2 levels. Level 3+ dependencies are reported as findings, not automatically pursued.

**Token management**: Each dependency validation is a SEPARATE pe-meta-validator invocation — not all loaded into one pass. This respects R-P2 (decomposition).

```
  - Rules multiple consumers miss → systemic issue
  - Rules no consumer implements → dead guidance
```

## Trade-offs of this approach

### Strengths

| # | Strength |
|---|---|
| 1 | **Complete self-containment** — pe-meta never depends on pe-gra/pe-con for PE artifacts |
| 2 | **PE-for-PE focus** — every check applies the exemplary bar, vision alignment, strategic criteria |
| 3 | **No cross-tier coupling** — pe-gra can evolve freely for domain needs |
| 4 | **Consistent quality standard** — one validator applies the same PE quality bar everywhere |
| 5 | **Simpler mental model** — "pe-meta handles all PE stuff, pe-gra handles all domain stuff" |

### Weaknesses

| # | Weakness | Impact | Mitigation |
|---|---|---|---|
| 1 | **pe-meta-validator is large** — embeds 8 type-specific checklists + strategic checks | MEDIUM | Dispatch pattern: only relevant checklist loaded per invocation |
| 2 | **22 new prompts** — significant creation effort | HIGH | Use pe-gra prompts as templates, adapt for exemplary bar |
| 3 | **Structural rule duplication** — structural checks (YAML, tools, boundaries) exist in BOTH pe-meta-validator AND pe-gra validators | HIGH | Accept as intentional: pe-meta's copies apply exemplary thresholds (5/2/3 vs 3/1/2) |
| 4 | **Drift risk between pe-meta and pe-gra structural checks** — if a structural rule changes, both need updating | MEDIUM | Both read from the SAME context files (01.04, 01.06, etc.) — the rules are defined once, only the threshold differs |
| 5 | **pe-meta-builder is a single builder for 8 types** — could become overloaded | MEDIUM | Dispatch pattern + context files carry type-specific rules |

### The structural duplication concern (#3-4)

This is the key trade-off. pe-meta-validator and pe-gra-agent-validator both need to check "does this agent have 3-7 tools?" The RULE is the same (from 01.04), but the THRESHOLD differs:
- pe-gra: standard bar (3/1/2 boundaries, 3-7 tools)
- pe-meta: exemplary bar (5/2/3 boundaries, 3-7 tools)

**Mitigation**: Both validators read from the SAME context files. The structural rules are defined once in 01.04, 01.06, etc. What differs is the quality bar applied. pe-meta loads the PE quality bar from 05.06 (strategic review criteria) and applies exemplary thresholds. pe-gra applies standard thresholds from the instruction files.

This means a structural rule change (e.g., tool count limit changes from 7 to 8) only needs updating in the context file — both pe-meta and pe-gra validators read from it. The duplication is in the CHECK LOGIC, not the RULES. This is acceptable because:
- The check logic is simple (parse YAML, count tools, compare threshold)
- The rules live in one place (context files)
- Only the thresholds differ (and those are defined in different context files — 01.06 for standard, 05.06 for exemplary)

## Why three review modes are essential (from Plan 01 analysis)

### Scenario coverage matrix

No single review mode catches everything. Each mode covers a distinct failure class:

| Scenario | Individual | Dependency-aware | Guidance-first |
|---|---|---|---|
| Agent has malformed YAML | ✅ | ✅ | ❌ |
| Agent loads context files that contradict each other | ❌ | ✅ | ❌ |
| H12 defines 3 ref types but validator only checks 2 | ❌ | ✅ (reviewing validator) | ✅ (reviewing H12's consumers) |
| Context files A and B define conflicting token budgets | ❌ | ✅ (if artifact loads both) | ✅ (reviewing either file) |
| 5 of 8 validators miss a rule from the priority matrix | ❌ | ❌ (only checks one) | ✅ (reviews all consumers) |
| Agent declares tools it never uses in process phases | ✅ | ✅ | ❌ |

### Why not combine all three into one review?

**Token cost.** A dependency-aware review loads the target (~250 lines) + context files (3-5 × 200-500 lines) + instruction files (~150 lines). That's 2,000-4,000 tokens of source material. Adding guidance-first checks (enumerate all consumers of each context file) would multiply by the consumer count. This violates R-P2.

**Different triggers.** Individual review = "validate this file." Dependency-aware = "assess this subsystem's coherence." Guidance-first = "audit this rule's implementation." Different questions, different entry points.

**Selective dimensions solve this.** Instead of 3 rigid modes, the `--dim` system supports any combination:
- `--dim structural` = lightweight individual review
- `--dim adherence --with-deps` = dependency-aware with adherence focus
- `--dim quality` on a context file = guidance quality without consumer enumeration
- `/pe-meta-adherence` = dedicated guidance-first with adherence matrix

## PE-for-PE domain focus: Strength, not limitation (from Plan B analysis)

Even in the self-contained model, pe-meta's PE-for-PE exclusivity is a strength:

**pe-gra/pe-con are domain-agnostic by design.** They apply structural checks that work for ANY PE artifact — a documentation agent and a PE agent both need valid YAML, 3-7 tools, and 3-tier boundaries. This is correct and valuable for domain artifacts.

**pe-meta adds what domain artifacts don't need:**
- Vision alignment (R-S1 through R-G3) — only PE agents need this
- Exemplary quality bar (5/2/3 vs 3/1/2) — domain artifacts use standard bar
- Self-update readiness (Phase 1-4 criteria) — only relevant for PE system
- Guidance quality construction invariants — only for PE context files

**In the self-contained model**, pe-meta internalizes structural checks BUT applies them at the exemplary bar. This is NOT redundancy — it's a higher standard applied to a higher-stakes domain. A PE-for-PE agent IS the reference implementation; its quality standard MUST exceed what it enforces on domain artifacts.
