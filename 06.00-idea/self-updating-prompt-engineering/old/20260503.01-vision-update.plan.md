# Vision Update Plan: Progressive Assessment Depth and Modular Invocation

> Make self-update invocations cost-effective without losing quality — through research-informed screening and dependency-aware modular entry points.

---

## Deep analysis: What the vision says vs. what's needed

### What the vision covers well

The vision (v8.2) has all the building blocks:
- **Artifact-type-aware routing** (V12) — different types get different quality requirements
- **R-S10 bottom-up ordering** — assess dependencies before dependents
- **R-S3 validation caching** — skip recently-validated artifacts
- **Guidance adherence verification** (V2) — scoped: full checks on scheduled reviews, targeted checks on individual changes

### What's missing: Two gaps

#### Gap 1: Validation is always full-depth — no progressive model

The vision describes **what** to check per artifact type (V12's routing table) but not **how deep** to check when the artifact isn't the primary target. When you request "review my agents," the vision says:
1. Organizational pass (broad triggers only)
2. Per-artifact pass bottom-up: context → instructions → agents

But every artifact in the chain gets its **full** quality requirements applied. Context files get all 6 construction invariants. Instructions get full minimization check + non-contradiction. Only then do agents get their deep review.

This is correct for a broad trigger (platform release) but wasteful for a targeted request ("review my agents"). The user doesn't need full context quality analysis — they need to know if the context is *good enough* that agent review findings are trustworthy.

**What's needed**: A validation depth concept — a way to distinguish "check this artifact deeply because it's the target" from "screen this dependency for obvious issues that would undermine the target's review."

#### Gap 2: No modular invocation model

The vision describes triggers (platform event, internal event, scheduled, etc.) and the Detect → Assess → Propose → Execute pipeline. But it doesn't describe **scoped entry points** — the ability to invoke the pipeline for a specific artifact type while dependency-screening (not deep-validating) related types.

The user wants:
- "Review context files" → deep on context, nothing else
- "Review agents" → deep on agents, screening on context + instructions, escalate if screening finds issues
- "Review prompts" → deep on prompts, screening on agents + context + instructions, escalate if screening finds issues
- "Review everything" → full depth on everything (existing behavior)

This is a natural extension of V12's artifact-type routing — but V12 describes which quality requirements apply per type, not which depth applies based on invocation scope.

## The three-phase assessment model

### Research → Informed screening → Deep pass

The assessment model has three phases that run in sequence. The research phase runs once per invocation (amortized cost). Screening runs per-dependency using research output. Deep pass runs on targets and escalated dependencies.

```
Invocation: "Review agents"
  Phase 1: Research (once per invocation)
    → Gather change digest: platform releases, ecosystem events,
      internal modifications since last review cycle
    → Output: structured change digest (~500 tokens)

  Phase 2: Informed screening on dependencies (per dependency)
    → Tier 1 — Deterministic checks: hash, references, metadata, structure
    → Tier 2 — Relevance check: does the change digest affect this artifact's scope?
    → If both tiers pass → dependency is healthy, proceed
    → If either fails → escalate to deep pass on that dependency

  Phase 3: Deep pass on targets (per target artifact)
    → Full quality review, informed by research output
    → Construction invariants for guidance, adherence for consumers
```

### Phase 1 (✅ Done): Research (runs once per invocation)

The research phase gathers what's changed since the last review — external events, internal modifications, and the system's own learning. This produces a **change digest**: a structured summary of changes relevant to artifact quality.

| Source | What's gathered | Detection method |
|---|---|---|
| **Platform releases** | VS Code / Copilot changes since last review | Release notes fetch (R-L4) |
| **Ecosystem events** | New skills, updated best practices from trusted sources | Monitored source check (R-G2) |
| **Model events** | New model capabilities, deprecations, behavior shifts | Provider changelog check |
| **Tooling/infrastructure** | MCP server updates, new tools available, tool deprecations | Registry/changelog check |
| **Vision/governance changes** | Vision updates, principle additions, new rationale IDs | Internal: vision file hash comparison |
| **Outcome history** | Recurring failures, rolled-back changes, human rejections by artifact type/domain | Outcome log analysis (R-G3) |
| **Internal changes** | Files modified since last review (git diff / file watcher) | Deterministic (R-P1) |

**Why outcome history matters**: If the outcome log shows that agent changes in a specific domain are consistently rolled back, the research digest flags that domain's guidance as a quality suspect — the guidance may be ambiguous or incomplete, causing downstream artifacts to be unreliable. This connects R-G3 (progressive learning) to the research phase, making the self-update system learn from its own mistakes.

**Why vision/governance changes matter**: A vision update is the highest-impact internal event. When the vision adds a new design principle (e.g., "Guidance quality by construction" in v8.2), every artifact should be assessed against the new requirement. Vision changes are detected by hash comparison — deterministic and zero-cost.

**Timestamp reliability and fallback**:

`last_review_timestamp` records when the research phase last ran. This timestamp is updated as part of the research phase completion — not at invocation start, but after the digest is successfully produced and persisted. This ensures the timestamp reflects completed research, not attempted research.

**Update rule**: `last_review_timestamp` is updated in the state folder (`<state.path>/triggers/last_review_timestamp`) at the end of every successful research phase. This is meta-infrastructure state — per the Staleness avoidance first principle, its integrity is checked before any cycle begins.

**Fallback when timestamp is missing or suspect**:
- If `last_review_timestamp` is absent → first-run mode: research gathers changes from a configurable lookback window (default: 90 days). This bounds the initial research scope without missing recent changes.
- If `last_review_timestamp` is present but older than the configurable maximum staleness window (default: 90 days) → treat as absent: apply the lookback window from current date.
- If `last_review_timestamp` is present and within the maximum staleness window → normal mode: research gathers changes since timestamp.

**Skip option**: For explicit user invocations ("review my agents now"), the research phase can be configured to skip external source fetching and use only internal changes + cached digest from the last research phase. This is safe for narrow invocations where the user knows no external changes are relevant — but the vision defaults to running research because Type B staleness is silent.

**Key property**: Research runs once and produces a reusable digest. All subsequent screening checks reference the same digest — no redundant research per artifact.

**Cost**: One research operation per invocation, regardless of how many artifacts are screened. The digest is condensed to essential changes (~500 tokens). External source fetching is skipped when `last_review_timestamp` is recent and no new releases are detected.

### Phase 2 (✅ Done): Informed screening (per dependency)

Screening operates in two tiers within the same pass. An artifact that passes both tiers is considered healthy. An artifact that fails either tier escalates to deep pass.

**Tier 1 — Structural screening (deterministic)**:

| Check | What it verifies | Cost | Escalation trigger |
|---|---|---|---|
| **Metadata currency** | `last_validated` within window, `version` unchanged | Read YAML | Missing metadata or expired validation |
| **Reference integrity** | All `📖` and category references resolve | File existence check | Broken reference |
| **Content stability** | Content hash matches last-validated state | Hash compare | Hash mismatch (content changed) |
| **Structural completeness** | Required YAML fields present, N-1 labels in rule sections | YAML parse | Missing fields or labels |
| **Validation cache** | Deep-validated recently against current platform version (R-S3) | Timestamp check | Cache absent or expired |

If Tier 1 passes and the change digest is empty (no external changes since last review) → screening passes. No Tier 2 needed.

**Tier 2 — Relevance screening (context-aware, runs only when change digest is non-empty)**:

Checks whether external changes in the digest affect this artifact's declared scope. Two sub-strategies depending on change type:

| Change type | Relevance check method | Cost |
|---|---|---|
| **Platform event** | Compare artifact's `scope.covers` topics and `authoritative_sources` against change digest entries. If the artifact's scope intersects with the changed platform area → escalate. | Low — metadata keyword matching + optional lightweight LLM relevance judgment (~300-500 tokens) |
| **Ecosystem event** | Compare artifact's `scope.covers` against new/updated ecosystem artifacts. If scope overlaps → escalate. | Low — same as platform |
| **Model event** | Check if artifact contains model-specific guidance (model names, capability assumptions). If yes and model capabilities changed → escalate. | Low — keyword scan + digest comparison |
| **Internal event** | Already caught by Tier 1 hash check | Zero — no additional work |

**Relevance screening cost**: When Tier 2 runs, it's a lightweight LLM call with small context: artifact metadata (`goal`, `scope`, ~200 tokens) + change digest (~500 tokens) + question "Does any change in this digest affect this artifact's scope?" Total: ~800 tokens per artifact. This is 5-10x cheaper than a deep pass.

**Key property**: Tier 2 only runs when two conditions are both true: (a) the change digest is non-empty AND (b) the artifact passed Tier 1 (structurally sound but potentially semantically stale). Most invocations with no external changes skip Tier 2 entirely.

### Phase 3 (✅ Done): Deep pass (per target + escalated dependencies)

Full quality review using V12's artifact-type routing table. Informed by the research phase's change digest — the reviewer knows what external changes to assess the artifact against.

This is the existing deep validation behavior, unchanged. The only addition: deep pass receives the change digest as input context, so it can assess relevance to specific external changes rather than doing open-ended staleness detection.

### Cost gradient (four levels)

| Condition | Action | Cost per artifact |
|---|---|---|
| Cache fresh, no external changes | Skip entirely (R-S3) | Zero |
| Cache stale, deterministic screening passes, no external changes | Pass at Tier 1 | Near-zero (YAML + hash reads) |
| Cache stale, deterministic screening passes, external changes exist | Tier 2 relevance screening | Low (~800 tokens) |
| Screening fails (Tier 1 or Tier 2) | Deep pass | Full (proportional to artifact size + chain) |

### Escalation triggers (screening → deep)

When either screening tier fails, the dependency needs deep validation:

| Screening failure | Escalation action |
|---|---|
| Metadata missing or incomplete | Deep pass — can't assess quality without metadata |
| Reference doesn't resolve | Deep pass — broken reference may indicate structural issue |
| Content hash changed since last validation | Deep pass — content changed, cached validation is stale |
| N-1 labels absent in rule-bearing sections | Deep pass — can't do deterministic classification without labels |
| Validation cache expired or absent | Deep pass — no recent validation to rely on |
| Relevance screening detects scope overlap with external change | Deep pass — artifact may be semantically stale despite structural health |

**Escalation is scoped**: only the specific dependency that failed screening gets a deep pass. Other dependencies that passed screening are not affected.

**Escalation is hand-off**: the deep pass on the dependency follows the same model — if it's a context file, it gets full construction invariants. The result flows back to the original invocation, which can then proceed with the target's deep pass knowing the dependency is now validated.

### How this composes with existing vision concepts

| Vision concept | How it interacts with the three-phase model |
|---|---|
| **R-S3 validation caching** | Tier 1 uses cached validation directly — if cache is fresh and no external changes, screening passes automatically |
| **R-S10 bottom-up ordering** | Screening follows the same bottom-up order — screen the lowest dependency first, then work up to the target |
| **V12 artifact-type routing** | Deep pass uses V12's routing table. Screening is a lighter layer that runs before the routing table is consulted |
| **Guidance quality by construction** | Construction invariants apply during deep pass only — screening checks whether previous construction is still valid |
| **Guidance adherence verification** | Adherence checks are deep-pass only for consumer artifacts. Screening checks reference integrity but not adherence |
| **R-L4 external knowledge** | Research phase uses R-L4 to gather the change digest. Screening Tier 2 uses the digest for relevance assessment |
| **Type B staleness** | Tier 2 relevance screening directly addresses the vision's primary concern — logic that's structurally sound but semantically obsolete |

## Modular invocation model

### Entry points by artifact type

| Invocation | Target (deep pass) | Dependencies (screening) | Escalation scope |
|---|---|---|---|
| **Review context files** | Context files | None — context is the base layer | N/A |
| **Review instruction files** | Instruction files | Context files (screening) | Failing context file → deep context review |
| **Review agents** | Agent files | Context files + instruction files (screening) | Failing context → deep context; failing instruction → deep instruction |
| **Review prompts** | Prompt files | Agents + context + instructions (screening) | Failing agent → deep agent (which may cascade to context/instruction screening) |
| **Review skills** | Skill files | Context + instructions (screening) | Same as instructions |
| **Review templates/hooks** | Templates, hooks | None — leaf artifacts | N/A |
| **Review everything** | All types | N/A — all get deep pass | N/A (full depth) |

### Cascade behavior

When screening escalates to a deep pass on a dependency, that deep pass may itself trigger screening on *its* dependencies:

```
User requests: "Review agents"
  → Screen context files    → PASS (cache fresh)
  → Screen instruction files → FAIL (content hash changed)
    → Deep pass on instruction files
      → Screen context files → PASS (already screened above, result cached)
      → Deep instruction review completes → findings reported
    → Return to agent review with validated instructions
  → Deep pass on agents → findings reported
```

**Cascade depth limit**: Maximum 2 levels of escalation. If a screening-triggered deep pass itself triggers further escalation, the second-level escalation is reported as a finding rather than automatically pursued. This prevents unbounded cascade chains.

**Cross-invocation cache**: Screening results are cached within the invocation. If context files are screened for the instruction review and then screened again for the agent review, the second screening reuses the first result.

### How this relates to the workflow steps

Modular invocation operates within the Assess layer. The 9-step workflow is unchanged:

- **Detect**: Trigger can be "user requests agent review" (internal event)
- **Assess**: Organizational pass (skipped for narrow invocation) → Per-artifact pass with screening on dependencies and deep on targets
- **Propose/Execute**: Standard pipeline for any findings

The modular entry point is a **trigger type** that scopes the Assess layer — not a separate workflow.

## Alternative strategies considered

The two extreme approaches — always full depth on everything (correct but prohibitively expensive) and skip dependencies entirely (fast but unsafe, violates R-S10) — are both obviously inadequate. The real design space is between them:

### Strategy A: Validation caching as the only depth control

If a dependency was recently validated, skip it entirely (R-S3). Otherwise, full deep pass. This is binary — either skip or full depth — with no intermediate screening layer.

**Problem**: When cache expires, every dependency gets a full deep pass. Cache expiration is time-based, not issue-based — a context file with zero changes since last validation still gets full deep review when the cache expires.

**Verdict**: Too coarse. Cache expiry should trigger screening (cheap), not deep pass (expensive). Deep pass only when screening finds issues.

### Strategy B: Deterministic-only screening (previous V13 draft)

Add a deterministic screening layer between cache and deep pass: hash comparison, reference checks, metadata currency, structural completeness. Catches internal changes cheaply.

**Problem**: Deterministic checks catch structural staleness (Type A) but are blind to semantic staleness (Type B) — the vision's primary motivating concern. A context file whose content hasn't changed but whose guidance is obsolete due to a platform release passes all deterministic checks. The screening model must account for external changes, not just internal ones.

**Verdict**: Correct for structural issues, blind to semantic staleness. Insufficient for the vision's primary concern.

### Strategy C: Research-informed screening with two-tier depth (adopted)

A research phase runs once per invocation and produces a change digest. Screening operates in two tiers: structural (deterministic, catches internal changes) and relevance (lightweight LLM, catches semantic staleness by comparing artifact scope against external changes). Deep pass only for targets and escalated dependencies.

This addresses both staleness types: Tier 1 catches Type A (structural), Tier 2 catches Type B indicators (semantic). Tier 2 only runs when external changes exist — most invocations with no platform/ecosystem events skip it entirely. When Tier 2 does run, it's cheap (~800 tokens per artifact vs. 2K+ for deep pass).

**Verdict**: Adopted. Catches both staleness types, cost-bounded, only adds LLM cost when external changes warrant it.

## Why screening is safe

The safety concern: "what if screening misses a deep issue in a dependency?"

**Answer**: Screening catches *invalidation signals* at two levels — structural and semantic.

**Tier 1 (structural)** catches internal changes that invalidate the last deep validation:

| What Tier 1 catches | Why it's sufficient |
|---|---|
| Content changed since last validation | If content changed, last validation is stale → deep pass will find issues if they exist |
| References broken | Structural failures visible without content analysis |
| Metadata missing | Can't assess quality at any depth → must deep pass |
| N-1 labels absent | Can't do deterministic classification → flag for investigation |
| Cache expired | Time-based safety net → periodic deep revalidation catches drift |

**Tier 2 (semantic)** catches external changes that make existing content obsolete:

| What Tier 2 catches | Why it's sufficient |
|---|---|
| Platform change overlaps artifact scope | Guidance may be obsolete despite structural health |
| Ecosystem change overlaps artifact scope | New best practices or tools may supersede current guidance |
| Model capability shift affects guidance | Model-specific advice may be outdated |

**What screening doesn't catch**: Content-level quality issues (ambiguous rules, incomplete coverage) that existed when the dependency was last deep-validated AND aren't related to any external change. These are caught by:
1. The dependency's own scheduled deep review cycle
2. Phase transition qualification (all 6 properties at full depth)
3. Explicit invocation ("review context files")

Screening doesn't weaken the quality guarantee — it defers deep analysis to the right moment. The two-tier design ensures that both structural staleness (Type A) and semantic staleness indicators (Type B) are caught cheaply, escalating to deep analysis only when warranted.
