# Vision Update Plan: Cross-Artifact Review and Guidance Optimization

> Make v10's guidance quality, adherence verification, and quality propagation concepts actionable through concrete review modes.

---

## The gap: Vision vs. reality

v10 introduces powerful concepts. None are implemented in pe-meta today.

| v10 Concept | Vision section | pe-meta implementation | Status |
|---|---|---|---|
| **6 guidance quality properties** | Goal § Guidance quality as prerequisite | No pe-meta command assesses clarity, completeness, prioritization, or actionability | ❌ Not implemented |
| **Guidance adherence verification** | Assess § Guidance adherence verification | No pe-meta command checks whether consumers implement the rules they reference | ❌ Not implemented |
| **Quality propagation (R-S10)** | Rationale R-S10 + Principle "Quality flows through dependencies" | No pe-meta command follows quality chains bottom-up | ❌ Not implemented |
| **Artifact-type-aware routing** | Assess § Artifact-type-aware routing | pe-meta-review treats all artifact types the same (6 strategic criteria) | ❌ Not implemented |
| **System-level quality dimensions** | Two operating levels § Quality dimension level | pe-meta-update healthcheck does coherence but not layer correctness or token-chain efficiency | ⚠️ Partial |
| **Progressive assessment (research → screening → deep)** | Assess § Assessment depth | pe-meta-review always runs full depth; no screening phase | ❌ Not implemented |

### The root problem

pe-meta-review has **one mode**: review an artifact (or folder of artifacts) against 6 strategic criteria. It never:
1. Loads the artifact's dependencies to verify adherence
2. Assesses guidance quality of the context files the artifact loads
3. Reviews the artifact together with its dependency chain as a coherent unit
4. Checks whether the guidance chain produces consistent, non-contradictory, well-prioritized behavior

The user's observation is precise: **pe-meta should allow reviewing artifacts together with their dependencies and context information** — not just checking each artifact in isolation.

---

## Three review modes pe-meta should support

### Analysis of alternatives

| Mode | Direction | Question it answers | Cost model |
|---|---|---|---|
| **Individual** (current) | Artifact → self | "Is this artifact structurally + strategically correct?" | O(1) artifact + fixed context loading |
| **Dependency-aware** (new) | Artifact → dependencies → back to artifact | "Does this artifact + its dependency chain produce reliable behavior?" | O(d) where d = dependency count |
| **Guidance-first** (new) | Guidance → all consumers | "Do artifacts that load this guidance actually implement it?" | O(c) where c = consumer count |

### Why all three are needed

| Scenario | Individual catches it? | Dependency-aware catches it? | Guidance-first catches it? |
|---|---|---|---|
| Agent has malformed YAML | ✅ | ✅ | ❌ |
| Agent references a context file that contradicts another context file it also loads | ❌ | ✅ | ❌ |
| H12 defines 3 ref types but validator only checks 2 | ❌ | ✅ (if reviewing the validator) | ✅ (reviewing H12's consumers) |
| Context file A and context file B define conflicting token budgets | ❌ | ✅ (if artifact loads both) | ✅ (reviewing either file) |
| 5 of 8 validators miss a rule from the priority matrix | ❌ | ❌ (only checks the one being reviewed) | ✅ (reviews all consumers) |
| Agent declares tools it never uses in process phases | ✅ (if checklist includes it) | ✅ | ❌ |

**Individual** catches per-artifact issues. **Dependency-aware** catches chain-level issues. **Guidance-first** catches systemic adherence gaps across all consumers. No single mode covers everything.

### Why not combine all three into one review?

**Token cost.** A dependency-aware review of one agent loads the agent (~250 lines) + its context files (3-5 files × 200-500 lines) + instruction files (~150 lines each). That's 2,000-4,000 tokens just for the source material. Adding guidance-first checks (enumerate all consumers of each context file) would multiply by the consumer count. This violates R-P2 (decomposition for context window control).

**Different triggers.** Individual review is triggered by "validate this file." Dependency-aware review is triggered by "assess this subsystem's coherence." Guidance-first review is triggered by "audit this rule's implementation." Different questions, different entry points.

## Proposed vision changes (v10 → v10.1)

### Change V1 (HIGH): Define three explicit review modes in the vision

**Location**: Vision § Assess layer, new subsection "Review modes"

**Proposed addition** (after the modular invocation table):

> **Review modes.** The Assess layer supports three review modes, each answering a different question at a different scope:
>
> | Mode | Invocation | Scope | What it checks | Cost |
> |---|---|---|---|---|
> | **Individual** | `/pe-meta-review <artifact>` | Single artifact | Structural + strategic (6 criteria) + per-artifact quality | Low — one artifact + fixed context |
> | **Dependency-aware** | `/pe-meta-review --with-deps <artifact>` | Artifact + its dependency chain | Individual checks + guidance adherence + dependency-chain coherence (contradictions between loaded context files, consistency of rules across the chain) | Medium — artifact + d dependencies |
> | **Guidance-first** | `/pe-meta-review --adherence <guidance-file>` | One guidance file + all its consumers | Guidance quality (6 properties) + adherence matrix (does each consumer implement each rule?) | Medium-high — guidance file + c consumers |
>
> **Mode selection guidance:**
> - Use **individual** for routine validation of a single artifact
> - Use **dependency-aware** when investigating behavioral issues (agent produces unexpected results — is its guidance chain coherent?)
> - Use **guidance-first** when updating a context file or after discovering a rule gap (are all consumers implementing the current version of this rule?)
> - `/pe-meta-update healthcheck` remains the system-wide coherence check — it uses screening + selective deep passes, not full-depth review of every file
>
> **Dependency-aware review process:**
> 1. Read the target artifact and identify its dependencies (`📖` references, `context_dependencies:`, `handoffs:`, instruction `applyTo` matches)
> 2. Screen each dependency (Tier 1 structural check — cached if fresh)
> 3. For dependencies that pass screening: verify adherence — does the target artifact implement the rules from this dependency?
> 4. For dependencies that fail screening: escalate that dependency to a deep pass before proceeding
> 5. Check cross-dependency coherence: do the context files the artifact loads contradict each other?
> 6. Report: per-artifact findings + adherence gaps + cross-dependency conflicts
>
> **Guidance-first review process:**
> 1. Read the guidance file and extract its rules (each distinct MUST/NEVER/ALWAYS statement)
> 2. Enumerate all consumers via `grep_search` for the filename/category across PE files
> 3. For each consumer: check whether each rule is implemented (present in checklist, boundary, or process)
> 4. Produce adherence matrix:
>
> ```markdown
> ## Adherence Matrix: [guidance-file]
>
> | Rule | consumer-1 | consumer-2 | consumer-3 | ... |
> |---|---|---|---|---|
> | Rule A | ✅ | ✅ | ❌ gap | ... |
> | Rule B | ✅ | ⚠️ partial | ✅ | ... |
> ```
>
> 5. Report: guidance quality assessment + adherence matrix + recommended fixes for gaps

**Classification**: EXTENDING — adds operational detail to existing adherence and quality propagation concepts.

### Change V2 (HIGH): Connect review modes to R-S10 quality propagation

**Location**: Vision § Quality flows through dependencies principle

**Proposed addition** (append):

> **Operational realization.** R-S10's bottom-up quality propagation is realized through the review modes:
> - **Individual mode** assesses per-artifact quality without checking the dependency chain
> - **Dependency-aware mode** implements R-S10 operationally: it assesses the target artifact's quality ceiling by examining the quality of its dependencies, then checks adherence
> - **Guidance-first mode** implements R-S10 in reverse: it assesses a dependency's quality and then traces how that quality propagates (or fails to propagate) to all consumers
>
> Together, the three modes provide complete coverage: individual for per-artifact, dependency-aware for chain-level, guidance-first for system-level. No single mode needs to load the entire system — each operates within R-P2's decomposition constraints.

**Classification**: EXTENDING — connects existing principle to new operational modes.

### Change V3 (MEDIUM): Add cross-dependency coherence to guidance adherence

**Location**: Vision § Assess § Guidance adherence verification

**Current text** covers rule presence, rule accuracy, rule completeness for a single referenced guidance file.

**Proposed addition** (after "Rule completeness"):

> 4. **Cross-dependency coherence**: When an artifact loads multiple guidance files, check that the rules from those files don't contradict each other. A context file on tool alignment and a context file on token optimization may each be individually correct but define conflicting tool-count limits. The consumer artifact inherits both — and has no way to signal the conflict. Cross-dependency coherence catches this at review time rather than at execution time.
>
> **Example**: An agent loads `01.04-tool-composition-guide.md` (says "3-7 tools") and `01.06-system-parameters.md` (also says "3-7 tools" — consistent ✅). But if one file were updated to "3-5 tools" while the other stayed at "3-7," the agent would receive contradictory guidance with no explicit precedence. Cross-dependency coherence flags this.

**Classification**: EXTENDING — adds a fourth adherence check dimension.

### Change V4 (MEDIUM): Add guidance optimization as an Assess output

**Location**: Vision § Assess layer, after finding severity classification

**Proposed addition**:

> **Guidance optimization findings.** Beyond individual artifact findings, the Assess layer can produce system-level optimization recommendations:
>
> | Finding type | What it detects | Example |
> |---|---|---|
> | **Rule fragmentation** | Same logical rule defined piecewise across multiple files | Tool count limit in 01.04 + 01.06 + system-parameters — should be in one canonical location with others referencing it |
> | **Priority gap** | Rules that could conflict but no explicit precedence in priority matrix | Two context files both define boundary minimums with different thresholds — which takes precedence? |
> | **Layer misplacement** | Content in the wrong artifact type | Behavioral guidance in an instruction file (minimization violation); testable rule in a context file (could be an instruction) |
> | **Consumer burden** | Artifact loads more guidance than it needs | Agent references 6 context files but only uses rules from 2; the other 4 add token cost without behavioral impact |
>
> These findings are MEDIUM severity — they don't break individual artifacts but degrade system-level efficiency and clarity. They're reported during dependency-aware and guidance-first reviews, not individual reviews.

**Classification**: EXTENDING — adds optimization findings as a new output type.

## Why this matters

The session that produced the earlier plans demonstrated the problem:
- We ran pe-meta on 24 agents → found structural issues ✅
- We fixed them all ✅
- We discovered 174 broken slash-command references pe-meta never caught ❌
- Root cause: the rule existed (H12) but validators didn't implement all of it ❌

With the three review modes:
- **Individual** would have caught structural issues (same as today) ✅
- **Dependency-aware** would have caught that the validator doesn't implement all H12 checks (loads 01.07 but doesn't implement all rules from it) ✅
- **Guidance-first** on 01.07 would have produced an adherence matrix showing H12 partially implemented across validators ✅

The guidance-first mode is especially powerful for optimization: run it on the 5 most-consumed context files and you get a complete picture of which rules are well-implemented, which have gaps, and which are over-specified (defined but never enforced by any consumer).
