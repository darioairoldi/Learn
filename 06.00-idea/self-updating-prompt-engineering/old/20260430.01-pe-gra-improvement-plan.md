## ✅ G6: Convergence/oscillation detection in orchestrators (HIGH)

**Problem**: Design and review orchestrators have iteration budgets (max 2-3 per phase) but no convergence detection. If changes oscillate (artifact changed → reverted → changed), the system retries until budget exhausted rather than detecting the pattern and escalating.

**Impact**: Wasted iterations on unstable loops. Risk of silent quality degradation in edge cases.

### Vision: Metadata-as-Contract Strategy

Artifact YAML metadata (`goal:`, `scope:`, `boundaries:`, `rationales:`) is the **declared contract** — the stable reference frame. Body content is the **implementation**. The metadata is the RULER against which body changes are measured, not the thing being monitored.

**Why this works**: Oscillation is rarely mechanical (create file → delete file). Real oscillation is semantic — broadening scope then narrowing it, adding a criterion then removing it, making an agent more generic then specializing it again. Metadata with recorded **rationales** captures the intent behind each decision, making silent reversals detectable.

### Primary Mechanism: Three-Outcome Compatibility Gate

Before applying any change, the orchestrator classifies it against the current metadata contract:

| Outcome | Meaning | Metadata update? | Risk level |
|---|---|---|---|
| **COMPATIBLE** | Change achievable within declared `scope:`, `goal:`, `boundaries:` | No — body change only | Low |
| **EXTENDING** | Change requires adding new metadata entries (broader scope, new rationale) | Yes — additive | Medium |
| **CONTRADICTING** | Change requires removing/modifying existing metadata entries | Yes — breaking | High — halt & escalate |

**The compatibility test** (applied before every change):

1. Does the change introduce something not covered by `scope:`? → scope update needed (EXTENDING)
2. Does the change violate a `boundaries:` item? → boundary must be removed/modified (CONTRADICTING)
3. Does the change serve a different purpose than `goal:`? → goal drift (CONTRADICTING — escalate immediately)
4. If the answer to all three is "no" → COMPATIBLE, proceed with body-only change

**Contradiction resolution protocol**:

- If a `rationales:` entry explains WHY the thing being contradicted was added → **HALT** — present the conflict to the user (the prior decision was intentional and recorded)
- If no rationale exists for the contradicted entry → proceed with caution, but **require** a rationale for the new state
- Never silently drop a metadata entry that has a recorded rationale

**Metadata hygiene during EXTENDING changes**:

- Check if the new metadata entry makes an existing entry redundant (subset relationship)
- If so, synthesize into a single broader entry rather than accumulating overlapping metadata
- Check if the new entry contrasts with (not contradicts) existing entries — signal potential design tension to user

### Secondary Mechanism: In-Context Change Ledger

A lightweight within-session defense against obvious mechanical churn. At each phase transition, the orchestrator logs a structured change record:

```
Iteration 0 (baseline): scope="Handles .md files", boundaries=3, tools=5
Iteration 1: scope extended to ".md and .qmd" [EXTENDING], boundaries +1, version 1.0→1.1
Iteration 2: scope narrowed to ".md only" [CONTRADICTING prior rationale]
→ HALT: iteration 2 contradicts rationale added in iteration 1
```

Before each new iteration, check the ledger for:
- Reversal to a previous state (field values returning to an earlier iteration)
- Increasing change volume without new external triggers (diverging, not converging)

If detected → halt and escalate with the ledger as evidence.

**Role**: Catches rapid back-and-forth within a single fix loop. Does NOT persist across sessions — that's handled by the metadata contract itself.

### Detection Signals Summary

| Signal | Mechanism | Action |
|---|---|---|
| Body change within contract bounds | Compatibility gate → COMPATIBLE | Continue — healthy convergence |
| New capability added with rationale | Compatibility gate → EXTENDING | Proceed, record rationale |
| Change contradicts recorded rationale | Compatibility gate → CONTRADICTING | **Halt** — present conflict to user |
| Metadata itself keeps changing | Frequency check at startup | **Escalate** — contract unstable, freeze for human review |
| Body reverts to previous iteration state | In-context ledger | **Halt** — stalemate, needs human input |
| `version:` shows rapid recent bumps | Startup metadata check | Proceed with caution, warn user |

### What to Add to Each Orchestrator

A **"Change Stability Protocol"** section containing:

1. **Pre-change compatibility gate** — classify every proposed change as COMPATIBLE / EXTENDING / CONTRADICTING before applying it
2. **Rationale enforcement** — CONTRADICTING changes that conflict with recorded rationales are blocked; EXTENDING changes must add a rationale
3. **In-context change ledger** — structured record per iteration for within-session oscillation detection
4. **Startup metadata check** — at Phase 1, read target artifact's current metadata and check for instability signals (rapid version bumps, existing contradictions between body and metadata)
5. **Escalation protocol** — halt + present conflict evidence (ledger or rationale contradiction) to user

### Scope

All 6 design orchestrators + all 6 review orchestrators (12 prompts total).

## ✅ G8: Change-driven validation depth in validators (MEDIUM)

**Problem**: Validators apply uniform validation depth regardless of what actually changed. A typo fix gets the same consumer-compatibility sweep as a rule rename. Conversely, a vocabulary change in a low-risk file can silently break 12 consumers that the validator never checks.

**Impact**: Wasted tokens on irrelevant checks (over-validation); missed breakage from cross-cutting changes (under-validation). Neither quality nor efficiency is optimized.

### Approach: Change-Driven Impact Analysis

Instead of deciding validation depth from file type alone, the validator **analyzes the change itself** to determine which consumers to check.

### Primary Mechanism: Change Impact Classification

After reading the change, the validator classifies it into one of four impact categories:

| Category | What changed | Consumer checks needed |
|---|---|---|
| **COSMETIC** | Formatting, typos, whitespace, comments, non-functional rewording | None — no consumer can break |
| **STRUCTURAL** | Sections added/removed/renamed, YAML fields modified | Check consumers that reference the modified section or field |
| **VOCABULARY** | Terms renamed, concepts redefined, key phrases replaced | Grep for old/new terms across all PE artifacts — check all users |
| **BEHAVIORAL** | Rules added/removed/modified, boundaries changed, tool lists altered | Check all consumers that depend on the affected rules |

**Classification procedure** (validator performs before consumer checks):

1. Read the change (before/after or description from builder handoff)
2. Identify WHAT moved: formatting only? section structure? specific terms? rule semantics?
3. Assign impact category
4. Derive targeted consumer list from the category (see below)

### Consumer Discovery: Layered Hybrid Protocol

No single discovery method covers all cases. The validator uses a **layered pipeline** — fast checks first, broader search only when needed.

**Three discovery mechanisms** (each with distinct strengths):

| Mechanism | Strengths | Limitations |
|---|---|---|
| **Declared deps** ("Referenced by" metadata) | Instant, section-level granularity, curated | Stale if not maintained; only exists in context files; misses auto-injection consumers |
| **Runtime search** (grep for filename, terms, `#file:` refs) | Always current; works for ALL artifact types; discovers undeclared deps | False positives (comments, examples); slower; cannot discover auto-injection consumers |
| **Pattern expansion** (applyTo glob for instruction files) | Definitive answer for "who receives this instruction"; no false positives | Only works for instruction files; overbroad (lists all matched, not just affected) |

**Layered pipeline by impact category:**

| Impact Category | Layer 1 (fast) | Layer 2 (verify/extend) | Layer 3 (safety net) |
|---|---|---|---|
| **COSMETIC** | — | — | — |
| **STRUCTURAL** | "Referenced by" → filter to modified section | grep section heading (confirm not stale) | — |
| **VOCABULARY** | grep OLD term across `.github/` + `.copilot/` | (layer 1 is already exhaustive) | For instructions: glob expansion + spot-check |
| **BEHAVIORAL** | "Referenced by" + grep filename | For instructions: glob expansion | For copilot-instructions: critical-rules matrix scan |

**Execution rules:**

1. Start at Layer 1 — cheapest, section-level granularity
2. Layer 2 validates Layer 1 output (catches stale "Referenced by") and discovers undeclared consumers
3. Layer 3 only activates for auto-injected files where grep cannot discover implicit consumers
4. Stop early when consumer list is sufficient and confidence is high

### Safety Net for High-Risk Files

The change-driven approach works well for most files, but **auto-injected files have hidden consumers** that can't be discovered by analyzing the change alone:

- `copilot-instructions.md` is injected into EVERY conversation — consumers are implicit (all agents, all prompts, all file-editing sessions)
- Instruction files are injected into every file matched by their `applyTo` glob — consumers include files that don't explicitly reference the instruction

**Rule**: For files with automatic injection (copilot-instructions and instruction files), the validator MUST apply an additional safety net:

| File type | Safety net (applied UNLESS change is COSMETIC) |
|---|---|
| `copilot-instructions.md` | After targeted consumer checks, verify no NEW contradiction exists between this file and the critical rules priority matrix (`01.07`). Check that no instruction file or context file's rules now conflict with the change. |
| Instruction files (`.instructions.md`) | After targeted consumer checks, enumerate all files matched by the `applyTo` glob. For VOCABULARY and BEHAVIORAL changes, spot-check 3–5 representative matched files for compatibility. |
| Context files with ≥10 consumers | After targeted consumer checks, verify the "Referenced by" list is still current (`grep_search` for the filename). If stale, expand consumer checks to discovered references. |

**Why the safety net is needed**: Change-driven analysis catches KNOWN dependencies. But auto-injected files create IMPLICIT dependencies — you can't grep for "who uses copilot-instructions" because the answer is "everyone, automatically." The safety net ensures these implicit consumers aren't silently broken.

### What Changes in Each Validator

Add a **"Change Impact Analysis"** step between reading the file and running consumer checks:

1. **Classify the change** → COSMETIC / STRUCTURAL / VOCABULARY / BEHAVIORAL
2. **Derive consumer list** from the classification (not from file type)
3. **Apply safety net** if the file is auto-injected (copilot-instructions, instruction files) or has ≥10 consumers
4. **Run consumer compatibility checks** only against the derived list
5. **Report** which consumers were checked and why (traceability)

### Terminology Note

"Risk Level" as a concept is retained only for the safety net trigger — it classifies the file's **injection mechanism** (automatic vs. on-demand) to decide whether the safety net applies. It is NOT used to determine validation depth — that comes from the change classification.

| Risk Level | File type | Injection | Safety net? |
|---|---|---|---|
| 0 (CRITICAL) | `copilot-instructions.md` | Always-injected | Yes — always (unless COSMETIC) |
| 1 (HIGH) | Instruction files | Auto-injected by glob | Yes — always (unless COSMETIC) |
| 2 (MEDIUM) | Context files | Semantic search | Only if ≥10 consumers |
| 3 (STANDARD) | Agents, prompts, skills, templates, hooks, snippets | Explicit invocation | No |

This is distinct from the existing "Tier" architecture (Tiers 0–5) which classifies **context file knowledge layers**. See `01.05-glossary.md` → "Tier Architecture."

- Classification: COSMETIC
- Consumer checks: None

- Result: Fast validation, no wasted checks

**Example 2: Rule rename in instruction file ("sentence case" → "sentence-style capitalization")**
- Classification: VOCABULARY
- Consumer checks: grep for "sentence case" → find 4 context files and 2 agents using the old term

**Example 3: New boundary added to an agent file**
- Classification: BEHAVIORAL
- Consumer checks: Check orchestrator prompts that hand off to this agent
- Safety net: No (Risk Level 3, explicit invocation)

- Classification: STRUCTURAL
- Consumer checks: Check "Referenced by" entries that reference the removed section heading
- Safety net: Yes (≥10 consumers) → verify Referenced-by is current via grep
- Result: Discovers 3 additional consumers not in Referenced-by, checks all

### Integration Design (Resolving Practical Gaps)

#### Gap 1: Mode Gate — Change-Driven vs Full Validation

Validators are invoked in three contexts, only one of which has change data:

| Invocation mode | Change data available? | Behavior |
|---|---|---|
| **Post-change** (builder handoff after edit) | Yes — builder provides change description | Run Change Impact Analysis → targeted consumer checks |
| **Direct validation** (user asks "validate X") | No | Skip classification → run full consumer checks (current behavior) |
| **Layer audit** (validate all files of a type) | No | Skip classification → run full checklist per file |

**Rule**: If no change data is present in the handoff, the validator MUST fall back to full validation mode. Change Impact Analysis is an **optimization** of consumer discovery, not a replacement for standalone validation.

#### Gap 2: Builder→Validator Handoff Contract (Change Data)

For G8 to work, builders (after their G7 pre-change guard) must pass structured change data when invoking the validator:

```
change_description:
  category: VOCABULARY | STRUCTURAL | BEHAVIORAL | COSMETIC
  summary: "Renamed 'sentence case' to 'sentence-style capitalization'"
  affected_sections: ["## Naming Conventions"]   # for STRUCTURAL
  old_terms: ["sentence case"]                    # for VOCABULARY
  new_terms: ["sentence-style capitalization"]    # for VOCABULARY
  rules_modified: ["Heading capitalization rule"] # for BEHAVIORAL
```

If the builder provides this, the validator uses it. If absent (direct invocation), the validator falls back to full mode. This extends G7's post-change reconciliation — builders already invoke the validator; they just need to include what they changed.

#### Gap 3: Per-Artifact Layer 1 Strategy

"Referenced by" only exists in context files. Each artifact type has a different fastest-path for consumer discovery:

| Artifact type | Layer 1 (fast) | Layer 2 (verify/extend) |
|---|---|---|
| **Context files** | "Referenced by" section → filter to affected section | grep filename across `.github/` + `.copilot/` |
| **Instruction files** | `applyTo` glob expansion → list matched files | grep filename for explicit `📖` references |
| **Agent files** | grep agent name in prompts and other agents (handoff targets) | grep filename |
| **Prompt files** | grep prompt name in agents (handoff sources) | grep filename |
| **Skill files** | grep skill name across agents and prompts | grep filename |
| **Template files** | grep `📖 template-name` + `#file:` references | grep filename |
| **Hook files** | grep hook filename in other hooks (chaining) | List all hooks to check event overlap |
| **Snippet files** | grep `#file:snippet-name` across prompts and agents | grep filename |

Each validator uses its artifact-specific Layer 1. The generic "Referenced by" entry in the hybrid pipeline is the **context-validator specialization**, not a universal step.

#### Gap 4: `copilot-instructions.md` Ownership

No dedicated validator exists for this file. The **instruction-validator** is the natural owner because:
- Same artifact family (instructional content injected automatically)
- Already checks for rule contradictions across instruction + context files
- Already has the `applyTo` overlap detection capability (extendable to copilot-instructions)

**Decision**: Extend instruction-validator's scope to include `.github/copilot-instructions.md` as a special case. Add a check: "If target is copilot-instructions.md, apply Risk Level 0 safety net instead of Risk Level 1."

#### Gap 5: Relationship to Existing Validation Checklist

G8 does NOT replace the validator's full checklist. It modifies **how one specific check executes**:

| What G8 changes | What stays the same |
|---|---|
| **Consumer compatibility check** (#28 in context-validator, #22-24 in instruction-validator, etc.) — made smarter via change classification | All other checks (metadata, structure, content quality, integrity) — always run in full |
| **Consumer discovery method** — layered hybrid instead of brute-force grep-everything | Check severity classification — unchanged |
| **Scope of consumer sweep** — targeted by impact category | Validation report format — unchanged |

**In post-change mode**: Run full checklist BUT replace the consumer compatibility check with the change-driven version (skip if COSMETIC, target if STRUCTURAL/VOCABULARY/BEHAVIORAL).

**In full-validation mode**: Run full checklist including full consumer compatibility sweep (current behavior — no change).
