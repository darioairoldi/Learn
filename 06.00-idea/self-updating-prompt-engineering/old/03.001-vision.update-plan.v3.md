## 🎯 Purpose of this document

This plan addresses two follow-up concerns raised after publishing [03.000-vision.v3.md](03.000-vision.v3.md):

1. The vision emphasizes rationales conceptually, but `rationales` is not listed in the normative metadata contract in the `### Metadata-driven` section.
2. The default state path `.copilot/self-update/` can collide when multiple self-update systems coexist in one repository (prompt engineering, documentation, validation, and others).

The goal is to clarify intent and define a conflict-safe, portable update.

---

Short answer: **it was not conceptually dropped, but it was de-scoped from the normative metadata contract in v2/v3 and is currently underspecified in the vision body.**

Evolution trace:

|---|---|---|

## 📊 Gap 1: `rationales` metadata in v3

Current v3 behavior:

- `### Rules, rationales, and examples are separable` defines rationale as content structure.
- `### Metadata-driven` defines machine-readable metadata without `rationales`.

Risk:

- The system can preserve goal/scope/boundaries while still silently losing design intent across cycles.
- Constraint preservation depends on prose interpretation instead of explicit metadata.

Required correction:

- Add `rationales:` to the metadata contract as **phase-gated** metadata:
  - Phase 1: recommended, optional
  - Phase 2+: required for artifacts eligible for autonomous semantic edits

Recommended shape:

```yaml
rationales:
  - Keep deterministic checks before LLM judgment for predictable operations.
  - Preserve coarse-grained ruleset references unless precision is load-bearing.
```

Scope:

- Keep it lightweight (list of strings).
- Defer structured rationale objects (id, constraint, impact) to Phase 3+.

## 🧭 Role of `rationales` metadata

`rationales` metadata captures **why** a design decision exists, not only **what** the artifact covers.

In the current metadata contract:

- `goal`, `scope`, and `boundaries` define objective and limits.
- `version` tracks change semantics.
- `rationales` adds decision-memory: the constraints and trade-offs that must survive future updates.

Without `rationales`, an artifact can remain formally in-scope while silently losing hard-won design intent.

### Why it is important

1. **Prevents design-decision erosion**
Goal/scope checks can pass while a change removes an important safety or quality constraint.

2. **Improves autonomy safety**
Autonomous updates need machine-readable anchors for intent preservation. `rationales` provides those anchors.

3. **Reduces oscillation risk**
When intent is explicit, the system is less likely to alternate between adding/removing the same guidance across cycles.

4. **Strengthens review quality**
Reviewers can evaluate change impact against declared reasons, not only against textual diffs.

5. **Supports portability**
In heterogeneous repositories, `rationales` makes local design choices explicit and transferable across engine implementations.

### Operational role in classification

`rationales` should be treated as **constraint metadata**:

- If a change preserves `goal/scope` but violates a declared rationale, route above baseline autonomy (notify or escalate).
- If a rationale is replaced, require explicit replacement rationale in the same change.
- If no rationale exists yet, allow update but emit a metadata improvement recommendation.

### Lightweight first, structured later

Phase model:

- **Phase 1**: `rationales` recommended, optional.
- **Phase 2**: `rationales` required for artifacts eligible for autonomous semantic edits.
- **Phase 3+**: optional upgrade to structured rationale objects where scale justifies it.

## 📊 Gap 2: state-folder collision risk

Current v3 behavior:

- Uses `.copilot/self-update/` as default repo-local state path.

Risk:

- Cross-domain collision in poly-system repositories.
- Tooling ambiguity when multiple engines write similarly named files.
- Weak portability due to non-unique state identity.

Required correction:

- Replace single shared default with a namespaced contract.
- Treat path identity as part of integration configuration, not a fixed convention.

---

## ✅ Proposed updates to the vision

Proposed text-level updates for the next vision revision:

1. In `### Metadata-driven`, add `rationales:` to the metadata list.
2. Add phase note under metadata:
   - `rationales:` optional in early adoption, required for autonomous semantic updates.
3. In `### Metadata-driven` audit/state section, replace fixed `.copilot/self-update/` with configurable `state_path` and `state_namespace`.
4. In `### Per-repo integration model`, add two config keys:
   - `state.namespace`
   - `state.path`
5. In portability principles, add explicit rule:
   - No engine may use an unscoped shared state directory by default.

## 🏗️ Conflict-safe naming contract

### State identity

Every self-update engine instance defines:

- `engine_id` — stable identifier for the engine implementation
- `domain_id` — maintained domain (`pe`, `docs`, `validation`, and so on)
- `instance_id` — optional override for multiple instances per domain

### Default repo-local path

Use:

```text
.copilot/state/<engine_id>/<domain_id>/<instance_id>/
```

Minimum required uniqueness:

- `<engine_id>` + `<domain_id>`

### Example

```text
.copilot/state/ghcp-pe-self-update/pe/default/
.copilot/state/ghcp-docs-self-update/docs/default/
.copilot/state/ghcp-validation-self-update/validation/default/
```

### Backward compatibility

- Legacy `.copilot/self-update/` is read-only fallback during migration.
- New writes go to namespaced path only.

1. **Clarify decision in vision text**
Add explicit statement that `rationales:` was deferred from schema listing and is now reinstated as phase-gated metadata.

2. **Update metadata contract**
Revise `### Metadata-driven` in [03.000-vision.v3.md](03.000-vision.v3.md) to include `rationales:` with lightweight format guidance.
