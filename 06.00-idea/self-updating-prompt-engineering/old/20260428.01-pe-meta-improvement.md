## 🎯 The problem

### Current tier responsibilities

| Tier | What it does | What it doesn't do |
|---|---|---|
| **pe-meta** (4 agents, 3 prompts) | Ecosystem health: fullcheck, healthcheck, performancecheck, release-monitor, scheduled-review | Individual artifact creation, design, or review |
| **pe-con** (3 agents, 3 prompts) | Generic PE artifact creation/review/design for ANY domain | PE-domain strategic alignment, vision compliance |
| **pe-gra** (24 agents, 17 prompts) | Deep per-type creation with researcher→builder→validator cycle for ANY domain | PE-domain strategic alignment |
| **pe-sim** (0 agents, 3 prompts) | Quick single-step artifact creation | Everything else |

### The gap

When you want to create or improve a PE artifact **for the PE domain itself** (e.g., improve pe-meta-validator, create a new context file about category patterns, update STRUCTURE-README), you currently use pe-con or pe-gra. These builders know **how to build artifacts structurally correct** — YAML fields, boundary tiers, tool alignment. But they don't know:

1. **The vision** — which rollout phase applies, which rationales constrain this artifact
2. **Strategic alignment** — does this artifact serve the self-update architecture's goals?
3. **Category contracts** — should this artifact use Level 1.5 references? Does it need to update STRUCTURE-README?
4. **Self-referential coherence** — PE artifacts that build PE artifacts must be the *best* artifacts in the system; they set the standard everything else follows
5. **Ecosystem impact** — changes to PE-for-PE artifacts cascade to all other PE artifacts

pe-meta has this strategic knowledge (it loads the vision, checks the dependency map, runs coherence checks) but can't **create or update** individual artifacts — it only audits, designs specs, and delegates to builders.

## 🔍 Analysis: two modes of PE artifact work

### PE-for-PE (self-referential)

**When:** Creating/improving artifacts in `.github/agents/00.0x-pe-*`, `.github/prompts/00.0x-pe-*`, `.copilot/context/00.00-prompt-engineering/`, `.github/instructions/pe-*`, `.github/skills/pe-*`

**What's different:**
- **No domain switch needed** — PE expertise IS the domain expertise. The agent doesn't need to "learn about article-writing" or "understand Azure deployments" — it already knows its own domain
- **Vision alignment is load-bearing** — every PE-for-PE artifact must serve the vision's goals (R-L1 through R-G3). A structurally correct but strategically misaligned PE artifact is actively harmful
- **Self-update compliance** — PE artifacts are the artifacts the self-update system maintains. They must be self-update-ready: proper metadata, N-1 separation, category-based references
- **Higher quality bar** — these artifacts set the standard. If pe-meta-validator has weak boundaries, every validator it builds will copy that weakness
- **Ecosystem coherence** — changes to one PE artifact affect all PE workflows. The builder must check the dependency map and understand blast radius

### PE-for-domain (cross-domain)

**When:** Creating/improving artifacts for article-writing (`01.00-article-writing`), documentation, devops, etc.

**What's different:**
- **Domain switch IS needed** — the builder must load domain-specific context (article-writing conventions, Diátaxis framework, etc.) that has nothing to do with PE infrastructure
- **Vision alignment is optional** — domain artifacts don't need to serve the self-update architecture. They just need to be structurally correct PE artifacts
- **Domain expertise activation** — the builder activates domain expertise patterns from `02.05-agent-workflow-patterns.md`, fetching domain sources, consulting domain experts
- **Lower coupling** — a change to a documentation-builder doesn't cascade through the PE system

### Why this distinction matters

| Dimension | PE-for-PE | PE-for-domain |
|---|---|---|
| Context loaded | PE vision + 30 context files + categories | Domain context + PE structural rules |
| Quality standard | Highest (sets the standard) | Standard (follows the standard) |
| Impact of errors | Cascading (breaks the system that builds everything) | Isolated (breaks one domain) |
| Domain expertise | Already present (self-referential) | Must be loaded/switched |
| Strategic alignment | Mandatory (vision rationales) | Structural only |

## 📋 Architecture options

### Option A: Status quo — pe-con/pe-gra handle everything

**How it works:** Keep using `/pe-con-design`, `/pe-con-review`, `/pe-con-create-update` for PE-for-PE work. No changes.

| Dimension | Assessment |
|---|---|
| **Effectiveness** | LOW — pe-con doesn't load the vision, doesn't check strategic alignment, doesn't enforce category contracts |
| **Reliability** | MEDIUM — structurally correct but strategically blind |
| **Efficiency** | HIGH — no new artifacts to maintain |
| **Complexity** | LOWEST — nothing changes |

**Verdict:** Acceptable for low-stakes PE artifacts. Insufficient for strategic PE infrastructure (pe-meta agents, context files, STRUCTURE-README).

---

### Option B: Add PE-domain prompts to pe-meta (RECOMMENDED)

**How it works:** Add 3 new prompts to `00.09-pe-meta/` that wrap the existing pe-con agents with PE-domain strategic context. No new agents needed — the prompts orchestrate existing pe-con-researcher, pe-con-builder, pe-con-validator with added PE vision loading.

**New prompts:**
- `pe-meta-pe-artifact-design.prompt.md` — design a new PE artifact with vision alignment
- `pe-meta-pe-artifact-create-update.prompt.md` — create/update a PE artifact with pre-change guards and category compliance
- `pe-meta-pe-artifact-review.prompt.md` — review a PE artifact against vision, categories, and ecosystem coherence

**How they differ from pe-con:**

| Step | pe-con prompt | pe-meta-pe prompt (proposed) |
|---|---|---|
| Phase 0 | Load dispatch table, load type instruction file | Load dispatch table + load vision document + load STRUCTURE-README categories + load dependency map |
| Research | Search for patterns in existing artifacts | Search + check vision rationale alignment + verify category coverage |
| Build | Dispatch to pe-con-builder | Dispatch to pe-con-builder WITH additional constraints: category-based references required, metadata contracts enforced, N-1 separation for rule-bearing sections |
| Validate | Dispatch to pe-con-validator | Dispatch to pe-con-validator + ecosystem coherence check (blast radius, dependency cascade, category regression) |

| Dimension | Assessment |
|---|---|
| **Effectiveness** | HIGH — vision alignment, category enforcement, ecosystem awareness |
| **Reliability** | HIGH — uses proven pe-con agents for structural work, adds strategic layer |
| **Efficiency** | HIGH — only 3 new prompts, no new agents, reuses pe-con infrastructure |
| **Complexity** | LOW — prompts are thin wrappers that add context, not logic |

**Verdict: RECOMMENDED.** Maximum strategic value, minimum new infrastructure. The prompts add the PE-domain intelligence that pe-con lacks without duplicating the structural building capability pe-con already has.

---

### Option C: Full granular pe-meta tier (24 new agents)

**How it works:** Create pe-meta-researcher, pe-meta-builder, pe-meta-validator for each PE artifact type (context, agent, prompt, instruction, skill, template, hook, snippet).

| Dimension | Assessment |
|---|---|
| **Effectiveness** | HIGHEST — per-type specialization with vision alignment |
| **Reliability** | HIGH — deep per-type expertise |
| **Efficiency** | VERY LOW — 24 new agents, massive maintenance burden, exacerbates the 41-agent discovery problem |
| **Complexity** | VERY HIGH — doubles the agent count |

**Verdict:** Over-engineered. The per-type specialization is already handled by pe-gra agents. Adding vision alignment doesn't justify 24 new agents.

---

### Option D: New consolidated agents in pe-meta (3 new agents + 3 prompts)

**How it works:** Create pe-meta-pe-researcher, pe-meta-pe-builder, pe-meta-pe-validator as new agents that specialize in PE-for-PE work.

| Dimension | Assessment |
|---|---|
| **Effectiveness** | HIGH — dedicated agents with PE vision expertise |
| **Reliability** | HIGH — clean separation of concerns |
| **Efficiency** | MEDIUM — 3 new agents + 3 new prompts. Agent discovery already has 41 agents |
| **Complexity** | MEDIUM — more agents to maintain, but clear responsibility |

**Verdict:** Good but unnecessary. The pe-con agents already do the structural work correctly. The strategic gap is in the **orchestration** (what context to load, what checks to run), not in the **execution** (how to build a file). Prompts are the right layer for orchestration; agents are the right layer for execution.

---

### Option E: Extend pe-con with a PE-domain mode flag

**How it works:** Add a `--pe-domain` flag to pe-con prompts that triggers vision loading and category enforcement.

| Dimension | Assessment |
|---|---|
| **Effectiveness** | MEDIUM — adds capability but clutters pe-con with domain-specific logic |
| **Reliability** | MEDIUM — mode flags increase complexity and testing surface |
| **Efficiency** | HIGH — no new files |
| **Complexity** | MEDIUM — pe-con prompts grow larger with conditional logic |

**Verdict:** Violates single responsibility. pe-con should remain domain-agnostic. Domain-specific orchestration belongs in domain-specific prompts.

## 💡 Recommendation

### Option B: Add 3 PE-domain prompts to pe-meta

**Why this wins:**

1. **Right layer for the job** — the gap is in orchestration (what context to load), not execution (how to build files). Prompts orchestrate; agents execute. Adding prompts is the correct architectural response.

2. **Reuses proven infrastructure** — pe-con-researcher, pe-con-builder, pe-con-validator already handle structural correctness. The new prompts add a strategic layer on top.

3. **Minimal new infrastructure** — 3 new prompts, 0 new agents. The total agent count stays at 31 pe-prefixed.

4. **Clear mental model** — "pe-meta is everything PE-for-PE" becomes true. pe-con/pe-gra remain "PE-for-any-domain".

5. **Aligns with vision** — the vision says pe-meta should be the self-improvement layer. Creating PE artifacts for PE is self-improvement.

6. **Solves the user's expectation** — "I want to use pe-meta to design and review PE artifacts" becomes possible.

|---|---|---|
| `pe-meta-prompt-engineering-update` | `/meta-prompt-engineering-update` | Ecosystem health: fullcheck/healthcheck/performancecheck |

| `pe-meta-pe-artifact-review` (NEW) | `/meta-pe-review` | Review PE artifact against vision + ecosystem |

| Agent | Role | New? |
|---|---|---|
| `pe-meta-researcher` | Research external sources, PE improvement opportunities | Existing |
| `pe-meta-designer` | Transform findings into change specifications | Existing |
| `pe-meta-validator` | Validate ecosystem health, design safety, implementation quality | Existing |
| `pe-meta-optimizer` | Apply deduplication, token optimization | Existing |

**No new agents.** The new prompts orchestrate existing pe-meta + pe-con agents.

## 📌 Implementation plan

### Phase 0: Create PE-for-PE strategic review context file (prerequisite) — ✅ DONE

Before building the prompts, create a context file that captures the **PE-strategic review guidance** that doesn't exist elsewhere. This file goes in `.copilot/context/00.00-prompt-engineering/` and provides the PE-domain expertise the new prompts need.

**File:** `05.06-pe-strategic-review-criteria.md` (Tier 5: Meta-Ops)

**What it covers (not duplicated elsewhere):**

| Topic | Why needed | Current gap |
|---|---|---|
| **Vision alignment checklist** | Which vision rationales (R-L1 through R-G3) apply to which artifact types | No formal mapping exists — pe-meta-researcher loads the full vision but doesn't have a structured checklist per artifact type |
| **PE-for-PE quality bar** | What makes a PE artifact "exemplary" vs "acceptable" — the standard-setting role | Nowhere documented. PE agents should be the best-structured artifacts because every other artifact copies their patterns |
| **Category reference enforcement** | When and how to require Level 1.5 references vs allowing Level 2 | STRUCTURE-README has the categories and rules, but no guidance on "how to audit an artifact's references for category compliance" |
| **N-1 adoption per type** | Which artifact types should use N-1 separation and how | Vision defines N-1 abstractly; no per-type "here's how to apply it to an agent body vs a context file" |
| **Self-update readiness checklist** | What makes an artifact "self-update ready" (Phases 1-4 preconditions) | Vision defines phases but no checklist for "is this artifact ready for Phase N autonomous updates?" |

**What it does NOT cover (already exists):**
- Structural rules per type → instruction files (`pe-agents.instructions.md`, etc.)
- Tool alignment → `01.04-tool-composition-guide.md`
- Token budgets → `01.06-system-parameters.md`
- Handoff patterns → `02.01-handoffs-pattern.md`
- Metadata contracts → `00.03-metadata-contracts.md`

**Category assignment:** Add to a new `pe-strategic-review` functional category in STRUCTURE-README.

**Why context file, not hardcoded in prompts:**
- Context files are discoverable, maintainable, version-controlled
- The guidance evolves with the vision — hardcoded prompt text becomes stale
- Other artifacts (pe-meta-validator ecosystem audit, pe-meta-researcher) can also load this context
- Follows the vision's R-S1 (metadata-driven) and single-source-of-truth principles

### Phase 1: Create 3 new pe-meta prompts — ✅ DONE

**Priority order:**

| # | Prompt | Why first |
|---|---|---|
| 1 | `pe-meta-pe-artifact-review.prompt.md` | Most immediately useful — review existing PE artifacts against vision |
| 2 | `pe-meta-pe-artifact-design.prompt.md` | Design new PE artifacts with strategic alignment |
| 3 | `pe-meta-pe-artifact-create-update.prompt.md` | Create/update with full PE-domain guards |

### Each prompt's structure

```
Phase 0: Load PE-domain strategic context
  - Load vision document (folder-based, not versioned filename)
  - Load STRUCTURE-README.md with Functional Categories
  - Load dependency map via `dependency-tracking` category
  - Determine artifact type via dispatch table

Phase 1: Delegate to pe-con-researcher (structural research)
  - Pass PE-domain constraints as additional requirements
  - Researcher discovers patterns, challenges use cases

Phase 2: Strategic alignment check (PE-meta specific)
  - Does this artifact serve the vision's goals?
  - Does it use category-based references (Level 1.5)?
  - Does it have proper metadata contracts?
  - Does it follow N-1 structural separation for rules?
  - What's the blast radius via dependency map?

Phase 3: Delegate to pe-con-builder (structural construction)
  - Pass strategic requirements as additional constraints
  - Builder creates/updates the artifact

Phase 4: Double validation
  - pe-con-validator: structural correctness
  - pe-meta-validator (ecosystem coherence): dependency impact, category coverage, vision alignment
```
