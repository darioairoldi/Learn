```markdown

| Direction | Partner | Template | Max Tokens |
|---|---|---|---|
| **Receives from** | `pe-gra-{type}-builder` | `output-builder-handoff.template.md` | 1500 |
```

## Priority 4 — Robustness: Iteration caps and tool-failure recovery

### Fix 4.1: Add builder↔validator loop cap

**Problem**: Builders have `max 3 retries` for pre-save validation, but the builder→validator→builder fix cycle has no documented maximum. This could loop indefinitely if fixes introduce new issues.

**Change**: Add a loop cap rule. Two options:

- **Option A (per-builder)**: Add to each builder's process section: "Max 2 builder↔validator round-trips; if still failing after 2 cycles, escalate to user with full issue list."
- **Option B (centralized)**: Add to `agent-patterns` escalation protocol so all agents inherit it.

**Decision**: Option A — keeps agents self-contained. Add one line to each builder's handoff phase.

**Affected files**: All 8 builders.

**Effort**: Low — 1 line per file.

**Status**: ✅ DONE (2026-05-01). Added "Max 2 builder↔validator round-trips" loop cap to all 8 builders' handoff phases.

---

### Fix 4.2: Add standard tool-failure recovery pattern

**Problem**: Response Management sections cover domain-level failures (missing input, constraint violations) but not infrastructure-level failures (`read_file` error, `grep_search` timeout, `replace_string_in_file` context mismatch).

**Change**: Two options:

- **Option A (per-agent)**: Add one bullet to each agent's Response Management: "On tool failure: retry once with adjusted parameters, then report `Tool failure: [tool] on [file] — [error]` and stop."
- **Option B (centralized)**: Add to `04.03-production-readiness-patterns.md` so agents pick it up via `📖` reference.

**Decision**: Option B — this is a cross-cutting concern. Add to the `production-readiness` context file rather than duplicating across 24 agents. Agents already load this file via `📖` reference.

**Affected files**:

- [x] `.copilot/context/00.00-prompt-engineering/04.03-production-readiness-patterns.md`

**Note**: This crosses into "context file content changes" (excluded from Round 1 scope). Acceptable here because it's a new section addition, not a modification of existing rules.

**Effort**: Low — add one section (~5 lines) to one context file.

**Status**: ✅ DONE (2026-05-01). Added "Standard tool-failure recovery protocol" table to Section 2 (error recovery) of `04.03-production-readiness-patterns.md`. Covers `read_file`/`grep_search` errors, `replace_string_in_file` context mismatch, `create_file` failures, and tool timeouts. Version bumped to 1.1.0.

## Priority 5 — Efficiency: Boilerplate externalization

### Fix 5.1: Extract builder change-management protocol to prompt-snippet

**Problem**: Pre-change compatibility gate + Reversibility + Post-change reconciliation = ~27 identical lines in every builder. 216 lines of pure duplication (8 × 27).

**Change**: Extract to `.github/prompt-snippets/pe-builder-change-management-protocol.md` and replace inline content with `#file:pe-builder-change-management-protocol.md` reference.

**Risk assessment**:
- **Pro**: Reduces duplication by 200+ lines, single source of truth for change-management rules
- **Con**: Introduces runtime dependency — if snippet file is deleted/renamed, all 8 builders lose their change-management protocol
- **Mitigation**: Snippet is in `.github/prompt-snippets/` (standard location), referenced via `#file:` (VS Code resolves at injection time, not runtime)

**Decision**: DEFER — same conclusion as Round 1 Fix 5.2. The operational risk of broken `#file:` references outweighs the maintenance benefit at current scale (8 files). Revisit when builder count exceeds 12 or when a change-management rule update is needed.

---

### Fix 5.2: Trim oversized agents

**Problem**: Two agents exceed the 300-line guideline:
- `pe-gra-instruction-builder.agent.md` — 355 lines (+91 above role avg)
- `pe-gra-agent-validator.agent.md` — 319 lines (+71 above role avg)

**Action**: Identify what can be trimmed or externalized.

**Decision**: DEFER — these agents are functional and their extra size comes from domain-specific content (instruction-builder has `applyTo` conflict logic; agent-validator has comprehensive validation checklist). Trimming would remove useful content. Revisit during next template audit.

---

### Fix 5.3: Standardize Phase 0.5 across validators

**Problem**: Phase 0.5 (Change Impact Analysis) appears in all 8 validators with minor variations. The core structure is identical (~20 lines each = ~160 lines total duplication).

**Decision**: DEFER — same reasoning as Fix 5.1. The variation between validators (different risk levels, different consumer discovery strategies) means a shared snippet would need parameterization, adding complexity. Current duplication is manageable.
