# Vision Update Plan: System-Level Quality and Guidance-First Architecture

> Close the gap between per-artifact validation (current) and system-level quality assurance (needed).

---

## Deep analysis: What the vision says vs. what's needed

### The vision's current focus

The vision excels at **per-artifact lifecycle management**:

| Vision component | What it ensures | Level |
|---|---|---|
| Metadata contracts | Each artifact declares goal, scope, boundaries | Per-artifact |
| Breaking/non-breaking classification | Each change is classified against one artifact's metadata | Per-artifact |
| Pre-change guard + post-change reconciliation | Each artifact's metadata stays synchronized with content | Per-artifact |
| N-1 structural separation | Each rule-bearing section enables deterministic diff classification | Per-section |
| Tier + dependency analysis | Each artifact's blast radius is assessed | Per-artifact with neighbors |

This handles **structural staleness** well — broken references, stale metadata, YAML compliance. The machinery works.

### What's missing: System-level quality

The vision's goal says "peak reliability, effectiveness, and efficiency" but the mechanisms primarily check **individual artifacts in isolation**. Three critical system-level properties are under-addressed:

#### Gap 1: Guidance quality is a gate but not a construction constraint

**The problem**: The vision now defines 6 guidance quality properties (V1) and gates autonomy phases on them (V5). R-S9 establishes quality as a prerequisite. But the vision is silent on **how guidance is produced to meet those properties**. The 9-step workflow starts at Search (detecting external changes) and assumes guidance already exists. When the system detects a platform change and needs to translate it into updated guidance — the actual rules, rationales, and examples — the vision doesn't describe the quality constraints that creation/update process must satisfy.

This creates an asymmetry: the vision has a mature framework for **validating** guidance quality (6 properties, R-S9, Phase 2/3 gates) but no framework for **producing** quality guidance. An implementation could produce guidance that passes structural checks (metadata, N-1, references) yet fails on clarity, completeness, or actionability — because the vision describes what quality looks like but not how to construct it.

**What the vision says**: R-S9 lists 6 quality properties. V1 makes them a prerequisite. V5 gates Phase 2/3 on thresholds. The Assess layer checks them during validation.

**What's missing**:
- **No construction constraint**: The 6 quality properties are defined as assessment criteria but not as construction invariants. The vision doesn't say "when producing guidance, ensure clarity/completeness/actionability" — it only says "assess quality before increasing autonomy."
- **No assessment timing protocol**: R-S9 says "assess guidance quality" but the vision doesn't specify *when* the 6 properties are checked: on every guidance change? Only during scheduled reviews? Only at phase transition? The assessment cadence determines how quickly quality issues are caught.
- **No guidance lifecycle acknowledgment**: The Execute layer describes applying changes to artifacts but doesn't distinguish guidance creation/update as a specific execution type with its own quality constraints. Guidance changes are treated identically to structural changes (reference updates, metadata fixes) despite having fundamentally different quality requirements.

#### Gap 2: Guidance adherence is unchecked

**The problem**: A context file can define a perfect rule, and an agent can completely ignore it. The current system validates that agents **reference** context files (H12 cross-reference integrity) but not that they **follow** the rules those files contain.

**Example from this session**: Context file `01.07` defines H12 as "all `📖` references and markdown links resolve." Prompt-validator's Phase 3 checks convention compliance — but it never verifies that H12's checks are actually implemented in the validator's checklist. The slash-command gap persisted because the rule existed but no one checked whether validators actually performed all three reference-type checks.

**What the vision says**: Validators check structural compliance. The Assess layer checks metadata. The coherence-check skill checks for contradictions.

**What's missing**: No concept of **adherence verification** — systematically checking that artifacts implement the guidance they claim to follow. This is the gap between "this agent references `01.04-tool-composition-guide.md`" (reference integrity ✅) and "this agent actually follows the tool alignment rules defined in that file" (adherence ❓).

#### Gap 3: System-level optimization is absent

**The problem**: Individual artifacts can each be excellent yet produce a suboptimal system. Twenty context files can all pass validation individually while the total token cost of loading them exceeds any reasonable budget. Eight validators can each be correct while duplicating 22% of their content.

**What the vision says**: The "Scoped over exhaustive" principle focuses validation effort. The efficiency dimension in the multi-dimensional impact assessment mentions token consumption.

**What's missing**: No explicit treatment of **system-level optimization properties**:
- Is guidance organized optimally for consumers? (right rules in right files)
- Is the right content in the right layer? (context vs. instruction vs. agent body)
- Is the total token cost of loading guidance sets reasonable?
- Are the right artifacts in the right tier?

---

## How these gaps affect the stated goals

### Peak reliability

**Current**: Individual artifacts are structurally correct and reference-sound.
**Missing**: System-wide rule consistency is checked by coherence skill but not systematically invoked. Guidance adherence isn't verified — an agent can structurally pass all checks while silently ignoring critical rules.
**Impact**: Reliability ceiling is determined by the weakest adherence link.

### Peak effectiveness

**Current**: The vision acknowledges effectiveness is a "hard limit" — only measurable through actual use.
**Missing**: Guidance quality directly impacts effectiveness. Ambiguous rules cause ambiguous behavior. Incomplete guidance causes unpredictable fallbacks. These are **measurable guidance properties** (clarity, completeness) that correlate with effectiveness even though effectiveness itself can't be directly measured.
**Impact**: Improving measurable guidance properties is the best available proxy for improving effectiveness.

### Peak efficiency

**Current**: Token budgets per artifact. Validation caching to skip redundant work.
**Missing**: System-level token analysis — what's the total cost when an agent loads its context chain? Is guidance fragmented across too many small files (high reference overhead) or consolidated into too-large files (context rot risk)?
**Impact**: Efficiency improvements require system-level analysis, not per-file budgets.

### Maximum autonomy

**Current**: N-1 separation enables deterministic classification. Metadata enables breaking/non-breaking assessment.
**Missing**: **Autonomy quality is bounded by guidance quality.** If the rules the system enforces are ambiguous or incomplete, autonomous corrections based on those rules will be unreliable. The vision describes the classification machinery but not the precondition that makes it trustworthy — clear, complete, non-contradictory guidance.
**Impact**: Before increasing autonomy (Phase 2→3→4), guidance quality should be assessed and improved. Otherwise higher autonomy means higher-confidence application of potentially ambiguous rules.

### Change V1 (✅ Done): Add guidance quality as a foundational concept

**Location**: New subsection after "Two dimensions" in the Goal section

**Proposed addition**:

> ### Guidance quality as prerequisite
>
> The self-update system's ability to maintain artifacts "at peak reliability, effectiveness, and efficiency" is bounded by the quality of the guidance those artifacts are built from. Six measurable properties determine guidance quality:
>
> | Property | Definition | Measurement |
> |---|---|---|
> | **Clarity** | A rule can be understood unambiguously by both humans and LLMs | LLM agreement test: two independent passes produce the same interpretation |
> | **Non-redundancy** | Each rule is defined in exactly one canonical location | Automated: scan for duplicate rule text across context files |
> | **Non-contradiction** | Rules across files do not conflict | Automated + LLM: coherence-check Workflow 2 |
> | **Completeness** | No gaps exist where guidance is needed but absent | LLM-assisted: scan agent behaviors that lack backing rules |
> | **Prioritization** | When rules could conflict, precedence is explicit | Automated: check critical-rules-priority-matrix coverage |
> | **Actionability** | An agent can act on the rule without additional interpretation | LLM test: can the rule be translated into a boolean check? |
>
> Guidance quality is a **precondition for autonomy**, not a parallel concern. Before increasing autonomy phase (Phase 2→3→4), the guidance that the system enforces should meet minimum quality thresholds on all six properties. Autonomous application of ambiguous or contradictory guidance produces unreliable results regardless of how accurate the classification machinery is.

### Change V2 (✅ Done): Add guidance adherence verification to the Assess layer

**Location**: Vision § Assess layer, new paragraph after chain alignment checks

**Proposed addition**:

> **Guidance adherence verification.** Beyond checking that references resolve (structural integrity), assessment includes spot-checking that artifacts **implement** the rules they reference. For each `📖` or category-based reference in an artifact, the Assess layer can verify:
>
> 1. **Rule presence**: Does the artifact's behavior or checklist include checks for each rule in the referenced guidance?
> 2. **Rule accuracy**: Does the artifact's implementation of the rule match the canonical definition?
> 3. **Rule completeness**: Does the artifact cover all rules in the referenced guidance, or only a subset?
>
> Adherence verification is **expensive** (requires reading both the artifact and its referenced guidance) and is therefore scoped: full adherence checks run during scheduled reviews and ecosystem audits, not on every change. For individual changes, adherence is checked only for the specific guidance affected by the change.
>
> **Example of adherence gap**: H12 defines "cross-reference integrity" as checking `📖` references, markdown links, and slash-command references. A validator that only checks `📖` references and markdown links has a reference to H12 (structural integrity ✅) but doesn't fully implement it (adherence gap ❌). This gap allowed 174 broken slash-command references to persist undetected.

### Change V9 (✅ Done): Add two-phase reasoning model to the Reason step

**Location**: Vision § The rationale, 9-step workflow table (Reason row) + Vision § Two operating levels

**Proposed changes**:

**9-step workflow table** — split the Reason row into two sub-phases:

> | **Reason (organizational)** | Assess whether the trigger implies structural changes to artifact organization | Evaluate role clarity, dependency structure, layer correctness, coverage gaps, redundancy |
> | **Reason (per-artifact)** | Assess whether the trigger implies content changes to specific artifacts | Evaluate quality, adherence, drift for individual artifacts within the validated structure |

**Two operating levels** — replace the current two levels with three that reflect the actual reasoning hierarchy:

> Assessment and proposal can run at three levels, in strict order for broad triggers:
>
> **Organizational level** — The artifact set as a system is the reference point. The system evaluates whether artifacts are organized correctly: roles are clear and non-overlapping, dependencies are well-structured, content lives in the right artifact type (instruction vs context vs agent body), and no redundancies or gaps exist. This level runs first on broad triggers (platform releases, ecosystem changes) because per-artifact improvements within a suboptimal organization are wasted effort. For narrow triggers (single file edit), this level is skipped unless the change implies organizational impact (e.g., adding a new artifact type, splitting an existing artifact).
>
> **Architecture level** — The full vision is the reference point. The system evaluates whether a major external change (a new platform capability, a fundamentally better approach from a trusted source) warrants structural rethinking.
>
> **Quality dimension level** — Individual measurable properties serve as the reference: non-contradiction, non-redundancy, consistency, prioritization, coverage, reliability, effectiveness, efficiency. These operate at both per-artifact scale (within a file) and system scale (across the artifact set). Some quality dimensions are independent (non-redundancy and non-contradiction can be assessed in isolation). Others are coupled (reliability and effectiveness are coupled with efficiency — a change improving one may degrade the other). Independent dimensions run in parallel; coupled dimensions run together.

**Rationale**: The vision's current Reason step treats reasoning as a single flat pass. In practice, reasoning has two phases with an ordering constraint: organizational reasoning (are artifacts structured correctly as a set?) must precede per-artifact reasoning (is each artifact individually excellent?) because per-artifact quality optimization within a suboptimal organizational structure is wasted work. Every trigger — platform update, ecosystem change, internal edit — should first assess structural implications, then content implications.

**Classification**: EXTENDING — refines existing workflow step and assessment levels.

### Change V10 (✅ Done): Add quality dependency chain concept (R-S10)

**Proposed additions**:

**New rationale in System infrastructure table:**

> | **R-S10-quality-propagation** | **Quality propagation through dependency chains.** An artifact's achievable quality is bounded by the quality of its dependencies. A prompt's reliability depends on the agent's quality, which depends on the context files' quality. Quality propagates downward through the dependency graph — improving a low-level artifact improves all its dependents; degrading it degrades all its dependents. This is distinct from R-S2 (structural dependency graph for blast radius): R-S2 answers "what breaks when X changes?" **New paragraph in Assess layer** (after finding severity classification):

> **Quality chain assessment.** When assessing per-artifact quality, the system traces the artifact's dependency chain and evaluates whether dependent quality is constrained by dependency quality. An artifact that references low-quality guidance can't produce high-quality output regardless of its own content quality. Quality chain assessment determines the most effective intervention point: improving a dependency may raise the quality ceiling for multiple dependents simultaneously, while improving only the dependent artifact hits a quality ceiling set by its dependencies.
>
> Quality chain assessment uses the existing coupling types from R-S2 but inverts the direction: R-S2 propagates *impact* upward (change flows from dependency to dependents); R-S10 propagates *quality constraints* downward (quality ceiling flows from dependency to dependents). The same dependency graph serves both analyses.

**New design principle:**

> ### Quality flows through dependencies
>
> *Relies on: R-S2-dependency-graph, R-S10-quality-propagation*
>
> An artifact's achievable quality is bounded by the quality of the artifacts it depends on. A perfectly written prompt that loads ambiguous context will produce ambiguous output. A well-structured agent that enforces unclear rules will enforce them inconsistently.
>
> Quality improvement is most effective bottom-up: fix the context file first, then the agent that loads it, then the prompt that invokes the agent. Top-down improvement hits a ceiling set by the weakest dependency.
>
> When a dependency improves, reassess its dependents — they may now achieve higher quality without any changes of their own. When a dependency degrades, all dependents are potentially affected — regardless of their individual quality.

**New conflict resolution row:**

> | R-S10-quality-propagation vs R-P2-decomposition | Compatible — quality chain traversal decomposes naturally by tier. Assess the lowest tier first, then propagate quality findings upward. When the chain is too long for a single context window, decompose by tier boundary. |

**Process-to-rationale mapping updates:**

> - Analyze row: add R-S10-quality-propagation
> - Reason / evaluate row: add R-S10-quality-propagation
> - Design row: add R-S10-quality-propagation (bottom-up improvement ordering)

**Rationale**: The vision's dependency model (R-S2) maps structural dependencies for blast radius but doesn't model how quality propagates through those same dependencies. A prompt's reliability depends on agent quality, which depends on context quality. Without this concept, the system optimizes each artifact in isolation and misses systemic quality constraints. The quality chain uses the same dependency graph as R-S2 but inverts the direction — R-S2 pushes impact upward, R-S10 constrains quality downward.

### Change V11 (✅ Done): Add organizational assessment ordering to the Assess layer

**Location**: Vision § Assess layer, new subsection before current per-artifact risk classification content

**Proposed addition**:

> **Assessment ordering.** For broad triggers (platform releases, ecosystem changes, scheduled reviews), assessment follows a two-pass sequence:
>
> **Organizational pass.** Before evaluating individual artifact quality, evaluate the artifact set as a system:
> - **Role clarity**: Are artifact roles well-defined and non-overlapping? Does each artifact have a single, distinct responsibility?
> - **Dependency structure**: Is the dependency graph clean? No circular dependencies, minimal coupling, clear hierarchical layers?
> - **Layer correctness**: Is content in the right artifact type? Testable rules in instruction files, behavioral guidance in context files, type-specific logic in agent bodies?
> - **Coverage and redundancy**: Are there gaps where guidance is needed but absent? Are there redundancies where the same concept is defined in multiple places?
>
> Organizational findings are addressed before the per-artifact pass begins — optimizing individual artifacts within a suboptimal organization is wasted effort. Organizational changes are HIGH impact by definition (restructuring) and route through the standard autonomy gradient.
>
> **Per-artifact pass.** With organizational structure validated or accepted, assess individual artifact quality following quality dependency chains (R-S10-quality-propagation): start with the lowest-tier dependencies and work upward, so each artifact is assessed against already-validated dependencies. Per-artifact assessment uses the existing mechanisms: tier + dependency analysis for risk, N-1 for diff classification, metadata for scope preservation.
>
> For narrow triggers (single file change, specific internal event), the organizational pass is skipped unless the change implies organizational impact (e.g., adding a new artifact type, splitting an existing artifact, creating a new dependency relationship).

**Rationale**: The Assess layer currently jumps to per-artifact risk classification (tier → dependency → chain alignment). The user's model establishes that system-level organizational reasoning must precede per-artifact quality reasoning because per-artifact quality is constrained by organizational quality. This ensures the system doesn't waste effort optimizing artifacts that should be restructured, merged, or moved to a different layer.

**Classification**: EXTENDING — adds ordering constraint to existing assessment process.

### Change V12 (✅ Done): Add artifact-type-aware quality routing to the Assess and Execute layers

**Location**: Vision § Assess layer (extend per-artifact pass), § Execute layer (new paragraph), § Design principles (new subsection after "Quality flows through dependencies")

**Supersedes**: Previous V12 ("Guidance quality by construction") and V13 ("Distinguish guidance changes in Execute") are replaced by this single change. The construction invariants and guidance-bearing distinction are preserved but reframed as natural consequences of type-aware routing rather than cross-cutting checks.

**Proposed additions**:

**New design principle — "Guidance quality by construction":**

> ### Guidance quality by construction
>
> *Relies on: R-S9-guidance-quality, R-S10-quality-propagation, R-L2-self-correction*
>
> Guidance artifacts (context files, and guidance-bearing sections of instruction files and agent bodies) set the quality ceiling for every artifact that depends on them. Their update path carries the 6 guidance quality properties as **construction constraints** — satisfied during creation/update, not only verified after the fact.
>
> **Construction invariants** (checked as part of every guidance change, not after it):
> - **Clarity**: Every rule MUST be testable for unambiguous interpretation. Ambiguous rules are rewritten before commit.
> - **Non-redundancy**: Before adding a rule, check whether it's already expressed elsewhere. Reference the canonical location — don't duplicate.
> - **Non-contradiction**: Before adding or modifying a rule, check for conflicts with existing rules in the dependency chain. Resolve before commit.
> - **Completeness**: When a new capability is added, assess whether existing guidance covers it. Gaps are flagged as findings.
> - **Prioritization**: When a new rule could conflict with an existing rule, explicit priority is declared at construction time.
> - **Actionability**: Every rule MUST be translatable to a concrete check (boolean pass/fail, measurable threshold, or enumerable options). Subjective rules are rewritten or moved to rationale blocks.
>
> **Assessment cadence** — the 6 properties are assessed at three tiers of cost:
> - **On every guidance change** (cheap): non-redundancy scan, non-contradiction check against direct dependencies, actionability self-test. These are the construction invariants.
> - **On scheduled reviews** (medium): clarity agreement test (dual-pass), completeness gap scan across the artifact set, prioritization matrix coverage.
> - **At phase transitions** (full): all 6 properties at full depth, qualifying the artifact set for the next autonomy phase.
>
> **Why construction, not just validation**: A clarity issue caught during construction costs one rewrite; the same issue caught during validation costs a finding + a fix cycle + a re-validation. For the system's own guidance (meta-infrastructure), construction constraints are especially critical — a quality issue in meta-guidance degrades every subsequent decision.
>
> Consumer artifacts (prompts, agents that consume guidance) don't carry construction invariants. They carry **adherence requirements** — verified against already-validated guidance. This separation keeps guidance update paths focused and consumer update paths lightweight.

**Extend Assess layer per-artifact pass** — add artifact-type-aware routing after the existing per-artifact pass description:

> **Artifact-type-aware routing within the per-artifact pass.** Per-artifact assessment follows R-S10's bottom-up ordering, but different artifact types carry different quality requirements reflecting their role in the system:
>
> | Artifact type | Role | Update-path quality requirements |
> |---|---|---|
> | **Context files** | Define guidance (rules, patterns, constraints) | Quality-by-construction: 6 properties as construction invariants (see Guidance quality by construction principle) |
> | **Instruction files** | Auto-inject testable rules | Minimization check (R-S8) + non-contradiction with context layer |
> | **Agent bodies** | Consume guidance + implement logic | Adherence check against loaded guidance + behavioral correctness |
> | **Prompts** | Orchestrate agents | Adherence check + workflow correctness |
> | **Templates, snippets, hooks** | Shape output / trigger wiring | Structural correctness (schema, references) |
>
> When a broad trigger fires (platform release, ecosystem change):
> 1. Assess guidance artifacts first — apply construction invariants if updates are needed
> 2. Then assess consumer artifacts — check adherence against the *already-validated* guidance
> 3. Each artifact type follows its own quality requirements — no cross-cutting classification needed
>
> When a narrow trigger fires (single artifact change):
> - If the changed artifact is a guidance artifact → apply construction invariants to the change
> - If the changed artifact is a consumer → check adherence against current guidance; don't trigger guidance quality investigation
> - If the change alters a dependency relationship → re-assess the affected quality chain (R-S10)
>
> This routing is a natural extension of R-S10's bottom-up ordering — guidance artifacts are the lowest tier in every quality chain, so they're assessed first. Consumer artifacts inherit a quality ceiling from their guidance, so they're assessed second with adherence checks instead of construction invariants.

**Extend Execute layer** — add after the risk-reduction techniques table:

> **Artifact-type requirements in Execute.** The risk-reduction techniques above apply to all changes. Additionally:
>
> - **Guidance artifact changes**: Construction invariants from the Guidance quality by construction principle run as part of the change. When a rule is modified, all artifacts with behavioral coupling (R-S2) to that rule are flagged for re-assessment — their quality ceiling may have shifted.
> - **Consumer artifact changes**: Adherence to loaded guidance is verified. If adherence gaps are found, they're fixed as part of the change — not deferred to a separate cycle.
> - **N-1 prerequisite**: A guidance change to a rule-bearing section without N-1 labels is automatically classified as HIGH risk. N-1 adoption is recommended as a prerequisite for autonomous guidance changes.

**Rationale**: The previous V12/V13 approach added construction invariants and guidance-bearing classification as cross-cutting checks on the general flow. This meant every Execute step had to classify "is this guidance-bearing?" — adding overhead even for narrow consumer changes. The user's insight: guidance lives in specific artifacts (primarily context files), so guidance quality should be handled by the guidance artifact's own update path, not as a cross-cutting check on every change.

This aligns with R-S10 (bottom-up quality propagation) and V11 (organizational → per-artifact ordering): guidance artifacts are the lowest tier in every quality chain, so they're naturally assessed and updated first. Consumer artifacts inherit quality from guidance and are assessed second with lighter requirements (adherence checks instead of construction invariants).

The result: when guidance is outdated, only the guidance update path runs with full quality-by-construction. When a consumer is outdated, it's checked against existing guidance without triggering guidance investigation. Each artifact type has the right quality requirements for its role — no cross-cutting overhead.

**Classification**: EXTENDING — adds type-aware routing within existing Assess/Execute structure. Preserves all V12/V13 concepts but reframes them as routing consequences.

## Alternative strategies considered

### Strategy A: Assessment-only (V1/V4/V5) — necessary but insufficient

V1/V4/V5 (already applied) define quality properties, create R-S9, and gate autonomy phases. This tells the system *what to check* but not *how to produce quality output*. Assessment without construction constraints is diagnostic without being prescriptive. An implementation could report "clarity: 40%, completeness: 60%" and have no guidance on how to improve those scores.

**Verdict**: Foundation layer. Required, but doesn't close the gap alone.

### Strategy B: Cross-cutting checks on all guidance-bearing changes (previous V12/V13)

The initial V12 added construction invariants as a design principle applied to any guidance-bearing change. V13 classified every Execute step as guidance-bearing or structural. This added overhead to every change — even narrow consumer fixes had to be classified and potentially trigger guidance checks.

**Verdict**: Correct invariants, wrong delivery mechanism. Every change pays classification cost; narrow consumer updates trigger unnecessary guidance investigation.

### Strategy C: Full guidance lifecycle as workflow phases — rejected as scope creep

An earlier analysis considered adding creation → validation → maintenance → deprecation as explicit workflow phases. The vision's scope is *self-updating*, and guidance creation details belong in implementation artifacts. What the vision needs is invariants, not process.

**Verdict**: Over-prescriptive. Rejected.

### Strategy D: Artifact-type-aware routing (adopted V12) — correct level of abstraction

The user's insight: guidance lives in specific artifacts (primarily context files). The self-update workflow already has bottom-up ordering (R-S10) and organizational assessment (V11). The missing piece is recognizing that different artifact types have different quality requirements reflecting their role — guidance artifacts carry construction constraints, consumer artifacts carry adherence requirements, and the bottom-up ordering naturally routes triggers to guidance first.

This preserves V12's construction invariants and V13's guidance-specific requirements but delivers them through routing instead of cross-cutting classification. When guidance is outdated, only the guidance path runs. When a consumer is outdated, it checks against existing guidance without triggering guidance investigation.

**Verdict**: Adopted. Correct invariants, efficient routing, aligned with R-S10.
