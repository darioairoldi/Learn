# Plan B (Revised): pe-meta Strategic Layer with Delegation

> pe-meta adds PE-for-PE strategic validation ON TOP of pe-gra/pe-con structural validation. pe-meta is NOT redundant — it handles checks that pe-gra/pe-con cannot perform because they lack PE-specific strategic context.

---

## The core question: Do pe-meta agents overlap with pe-gra/pe-con?

**No.** They operate at different layers with different knowledge:

| Dimension | pe-gra / pe-con | pe-meta |
|---|---|---|
| **Domain scope** | ANY PE artifact (domain or PE-for-PE) | ONLY PE-for-PE artifacts |
| **What they load** | Type-specific rules (instructions, patterns) | Vision document, strategic criteria, dependency map |
| **What they check** | Structural correctness (YAML, tools, boundaries, refs) | Vision alignment, quality bar, ecosystem coherence, adherence |
| **Quality standard** | Standard (≥3/1/2 boundaries, 3-7 tools) | Exemplary (≥5/2/3 boundaries, handoff contracts, category refs) |
| **Cross-artifact?** | No — one artifact at a time | Yes — compares artifacts, checks coherence across the system |
| **Knows the vision?** | No | Yes — loads and checks against it |
| **Knows the priority matrix?** | No | Yes — checks prioritization across rules |

### This already works today

`pe-meta-review` already implements this layering:
- **Phase 1**: Delegates to `@pe-con-validator` → structural checks (what pe-gra/pe-con do)
- **Phase 2**: pe-meta's own strategic checks → vision alignment, quality bar, N-1, category refs, self-update readiness (what ONLY pe-meta can do)
- **Phase 3**: Optionally delegates to `@pe-meta-validator` → ecosystem coherence (what ONLY pe-meta can do)

The delegation model is NOT new — it's the current architecture. The question is how to EXTEND it to support dependency-aware and guidance-first reviews.

## Per-agent role clarity: No overlap

### pe-meta-researcher vs pe-gra researchers

| pe-gra-{type}-researcher | pe-meta-researcher |
|---|---|
| "What should this agent look like?" | "Is the PE system healthy?" |
| Discovers patterns for ONE artifact | Discovers improvement opportunities across ALL PE artifacts |
| Reads 3-5 similar agents | Reads platform release notes, ecosystem changes, vision |
| Output: specification for one builder | Output: system-wide research report for meta-designer |

**No overlap**: pe-gra researchers research individual artifact requirements. pe-meta-researcher researches system-level health.

### pe-meta-designer vs pe-gra builders

| pe-gra-{type}-builder | pe-meta-designer |
|---|---|
| "Create/update this one file" | "Plan changes across 5-10 files" |
| Writes one artifact | Produces change specifications (plan mode — doesn't write) |
| Follows specification from researcher | Transforms system-wide findings into ordered change plans |
| No dependency ordering needed | Orders changes by dependency layer (L1→L2→L3→L4) |

**No overlap**: pe-gra builders write individual files. pe-meta-designer plans multi-artifact changes.

### pe-meta-optimizer vs pe-gra builders

| pe-gra-{type}-builder | pe-meta-optimizer |
|---|---|
| Creates/updates one artifact per specification | Applies cross-artifact optimizations (deduplication across files) |
| Follows a per-file spec | Follows a system-wide optimization report |
| Doesn't compare files to each other | Compares files to find redundancy, consolidation opportunities |

**Partial overlap in mechanics** (both write files), but **no overlap in scope** (individual vs system-wide).

### pe-meta-validator vs pe-gra validators

| pe-gra-{type}-validator | pe-meta-validator |
|---|---|
| "Is this one agent file structurally correct?" | "Do these PE artifacts work together as a coherent system?" |
| Checks YAML, tools, boundaries, refs for ONE file | Checks contradictions, handoff chains, scope overlaps across ALL files |
| Loads type-specific rules | Loads vision, strategic criteria, dependency map |
| Standard quality bar | Exemplary quality bar (PE-for-PE specific) |
| Already handles individual verification | Already delegates TO pe-gra validators for per-file checks |

**No overlap**: pe-gra validates individual correctness. pe-meta validates system coherence and PE-for-PE strategic alignment.

## The PE-for-PE focus: Strength, not limitation

### Concern: pe-gra/pe-con work on any domain — isn't that a problem for pe-meta?

**No — it's exactly right.** pe-gra and pe-con are domain-agnostic by design. They check structural rules that apply everywhere:
- Does this agent have 3-7 tools? (valid for any agent)
- Does this instruction file have valid `applyTo`? (valid for any instruction)
- Does this context file stay under 2,500 tokens? (valid for any context file)

pe-meta adds the PE-for-PE layer that domain artifacts don't need:
- Does this PE agent align with vision rationale R-S5? (only PE agents need this)
- Does this PE context file meet the exemplary quality bar? (domain context files use standard bar)
- Is this PE artifact ready for Phase 3 autonomous execution? (only relevant for PE system)

**When pe-meta delegates structural checks to pe-gra**, pe-gra runs the SAME structural checks it would run for any domain artifact. pe-meta then adds the strategic layer. This is correct — PE-for-PE artifacts must pass BOTH standard structural checks AND exemplary strategic checks.

### Could pe-gra accidentally apply domain-specific knowledge to PE artifacts?

No. pe-gra validators load type-specific rules (e.g., `pe-agents.instructions.md`), not domain-specific rules. These instruction files apply to ALL agents regardless of domain. A PE agent and a documentation agent both need valid YAML, 3-7 tools, and 3-tier boundaries. pe-gra checks this correctly.

## How dependency-aware review works with delegation

```
User: /pe-meta-review --with-deps pe-gra-agent-builder.agent.md

pe-meta-review (orchestrator):
  Phase 1: Delegate structural check → @pe-con-validator
           Returns: structural findings for the agent
  
  Phase 1.5: Identify dependencies
           Read agent's 📖 refs, context_dependencies, handoffs
           → discovers: 01.04, agent-patterns, pe-agents.instructions.md
  
  Phase 1.6: Delegate dependency structural checks
           For each dependency:
             → @pe-gra-{type}-validator: "validate this file structurally"
             ← structural findings per dependency
  
  Phase 2: pe-meta strategic checks (ONLY pe-meta can do these)
           - Vision alignment for the agent
           - PE quality bar (exemplary standard)
           - Category reference compliance
           
  Phase 2.5: Cross-dependency coherence (ONLY pe-meta can do this)
           - Do context files 01.04 and 01.06 contradict each other?
           - Does the agent implement ALL rules from its loaded context files?
           - Are priorities explicit where loaded guidance could conflict?
           
  Phase 3: Consolidated report
```

**Who does what:**

| Check | Who does it | Why |
|---|---|---|
| Is context file X structurally correct? | pe-gra-context-validator | Type-specific knowledge |
| Does the agent have valid YAML? | pe-gra-agent-validator | Type-specific knowledge |
| Do X and Y contradict each other? | **pe-meta** | Cross-artifact — no single validator sees both |
| Does the agent implement rules from X? | **pe-meta** | Adherence — requires reading both artifact and guidance |

## How guidance-first review works with delegation

```
User: /pe-meta-review --adherence 01.07-critical-rules-priority-matrix.md

pe-meta-review (orchestrator):
  Phase 1: Assess guidance quality (pe-meta does this itself)
           Apply 6 properties: clarity, non-redundancy, non-contradiction,
           completeness, prioritization, actionability
  
  Phase 2: Enumerate consumers via grep_search
           → discovers: 8 validators, 6 prompts, 4 builders
  
  Phase 3: Per-consumer adherence check (BATCHED)
           For each consumer:
             → @pe-gra-{type}-validator: "does this artifact implement rule H12?"
             ← {rule_id, status: implemented|partial|missing, evidence}
  
  Phase 4: Aggregate adherence matrix (ONLY pe-meta)
           
           | Rule | prompt-val | agent-val | context-val | ... |
           |------|-----------|-----------|-------------|-----|
           | H12  | ⚠️ 2/3     | ⚠️ 2/3     | ✅           | ... |
```

**Why pe-meta must aggregate**: No individual pe-gra validator sees more than one consumer. Only pe-meta sees ALL consumers and can identify systemic patterns.

## Recommended vision changes

### V1: Explicitly define the two-layer validation model

PE-for-PE artifacts receive TWO validation layers:
- **Layer 1 (structural)**: pe-gra/pe-con validates correctness (same checks as domain artifacts)
- **Layer 2 (strategic)**: pe-meta validates PE-specific alignment (vision, quality bar, coherence, adherence)

### V2: Define pe-meta's exclusive responsibilities

pe-meta handles ONLY what requires strategic context (vision, priority matrix, system-wide view) or cross-artifact comparison. Everything else delegates to pe-gra/pe-con.

### V3: Define three review modes with delegation contracts

- Individual: pe-con-validator (structural) + pe-meta (strategic) — **exists today**
- Dependency-aware: pe-gra validators (per-dep structural) + pe-meta (cross-dep coherence + adherence) — **new**
- Guidance-first: pe-meta (guidance quality) + pe-gra validators (per-consumer adherence) + pe-meta (aggregation) — **new**

### V4: Clarify pe-gra/pe-con domain-agnosticism is by design

pe-gra/pe-con are deliberately domain-agnostic. pe-meta's PE-for-PE focus is the strategic layer that gives PE infrastructure artifacts the higher quality bar they need as reference implementations.
