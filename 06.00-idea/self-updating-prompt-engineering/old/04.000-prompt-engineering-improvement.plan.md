# Prompt engineering improvement plan

> **Goal**: Bring the existing PE infrastructure into alignment with [05.000-vision.v5.md](05.000-vision.v5.md) so that every artifact type can be reliably created, reviewed, updated, and self-maintained ‚Äî and so that the same infrastructure can produce and maintain artifacts for any domain (starting with `01.00-article-writing`).
>
> **Related**: [05.000-prompt-engineering-implementation.plan.md](05.000-prompt-engineering-implementation.plan.md) ‚Äî addresses agent architecture (Option A vs Option D comparison via folder separation)

---

## üéØ Vision summary and ultimate goal

The vision describes a **self-updating PE system** that:

1. **Continuously detects drift** (platform, internal, ecosystem)
2. **Assesses impact** using tier + dependency analysis
3. **Proposes corrections** with evidence and risk classification
4. **Executes validated corrections** autonomously for low-risk changes, with human approval for high-risk ones
5. **Learns from outcomes** to tune autonomy thresholds over time

The **ultimate goal** is a PE system that behaves like a responsible maintainer: it watches, assesses, acts, and improves ‚Äî with calibrated autonomy. The system never silently degrades because it detects staleness before it causes failures.

## üìõ Naming conventions and architectural clarity

### Two layers, not two systems

The current artifacts appear to contain "two PE systems" ‚Äî but they're actually **two complementary layers** of one system:

| Layer | Role | Example | Why it looks "monolithic" |
|---|---|---|---|
| **Prompts** (orchestrators) | Workflow entry points that sequence multi-step work | `pe-agent-create-update.prompt.md` | They orchestrate full workflows ‚Äî this is intentional |
| **Agents** (workers) | Specialized workers called by prompts | `pe-agent-builder.agent.md` | Each does one thing ‚Äî this is intentional |

A prompt hands off to agents via `handoffs:`. The prompt IS the workflow; agents ARE the steps. This is the correct architecture ‚Äî not a duplication.

### The actual competing architectures

The real choice is **how granular** the agents and prompts should be:

| Architecture | Agents | Prompts | Naming pattern |
|---|---|---|---|
| **Granular** (current) | 27 type-specific triads: `pe-{type}-{role}` | 17 type-specific: `pe-{type}-{action}` | Long names ‚Äî type is in the name |
| **Consolidated** (Option D) | 3 role-based: `pe-{role}` | 3 action-based: `pe-{action}` | Short names ‚Äî no type qualifier |
| **Meta** (shared, unchanged) | 4 meta-agents: `pe-meta-{role}` | 3 meta-prompts: `pe-meta-{action}` | `pe-meta-` prefix |

### Self-documenting naming convention (replaces pe-/pe1-/pe2-)

Instead of cryptic numeric prefixes, **name length itself tells you the architecture**:

| What you type | What it means | Architecture |
|---|---|---|
| `@pe-builder` | Builds ANY artifact type (loads type rules dynamically) | Consolidated |
| `@pe-context-builder` | Builds ONLY context files (type-specific expertise) | Granular |
| `@pe-meta-researcher` | Researches the PE SYSTEM itself (meta-level) | Meta |
| `/pe-create-update` | Create/update ANY artifact type | Consolidated |
| `/pe-agent-create-update` | Create/update ONLY agent files | Granular |
| `/pe-meta-prompt-engineering-update` | Update the PE system itself | Meta |

**Why this is better than pe-/pe1-/pe2-:**
- ‚úÖ **Self-documenting** ‚Äî the name tells you the scope
- ‚úÖ **No numeric suffixes** ‚Äî nothing to memorize
- ‚úÖ **Shorter = broader** ‚Äî natural mental model (less specific name = handles more types)
- ‚úÖ **Both coexist** ‚Äî `@pe-builder` and `@pe-context-builder` are distinct, non-clashing names
- ‚úÖ **Alphabetical grouping** ‚Äî consolidated sorts before type-specific (`pe-b...` before `pe-context-b...`)
- ‚úÖ **Winner keeps its names** ‚Äî after comparison, losing architecture's files get deleted, no renames needed

### Folder strategy

Folders organize by architecture for clean comparison, but VS Code resolves by filename (subfolders don't isolate):

| Folder | Content | Architecture |
|---|---|---|
| `.github/agents/00.00-prompt-engineering/` | 28 granular agents (`pe-{type}-{role}`) + 4 meta agents | Granular + Meta |
| `.github/agents/00.01-prompt-engineering/` | 3 consolidated agents (`pe-{role}`) | Consolidated (Option D) |
| `.github/prompts/00.00-prompt-engineering/` | 17+ granular prompts (`pe-{type}-{action}`) + 3 meta prompts | Granular + Meta |
| `.github/prompts/00.01-prompt-engineering/` | 3 consolidated prompts (`pe-{action}`) | Consolidated (Option D) |

**Shared across all architectures** (zero duplication):
- Context files (`.copilot/context/00.00-prompt-engineering/`) ‚Äî subfolder = namespace
- Instruction files (`.github/instructions/pe-*.instructions.md`) ‚Äî `applyTo` globs match all architectures
- Skills (`.github/skills/pe-*/`) ‚Äî folder name = namespace
- Templates (`.github/templates/00.00-prompt-engineering/`) ‚Äî subfolder = namespace
- Hooks (`.github/hooks/pe-*`) ‚Äî already `pe-` prefixed

### Portability rule

All PE artifacts MUST use the `pe-` namespace prefix on their **identifier**:
- **Agents/prompts**: `pe-` prefix on filename stem (identifier = filename in VS Code's flat namespace)
- **Skills**: `pe-` prefix on folder name
- **Instructions**: `pe-` prefix on filename
- **Context files/templates**: subfolder (`00.00-prompt-engineering/`) serves as namespace

Consolidated agents use `pe-{role}` (shorter). Granular agents use `pe-{type}-{role}` (longer, more specific). Both are `pe-` namespaced and portable.

### Phase 1 (‚úÖ Done): Validate metadata + N-1 classification pipeline

**Goal**: Prove that the metadata and N-1 labels from Phase 0 actually improve artifact change quality. Test the deterministic checks that work, acknowledge the LLM-dependent checks that improve but don't eliminate judgment, and establish the baseline for the pipeline that Phases 2‚Äì8 build on.

**Analysis ‚Äî what's deterministic vs. what's judgment:**

| Check | Deterministic? | Value |
|---|---|---|
| Required YAML fields present (`goal:`, `scope:`, `version:`) | ‚úÖ Yes | Catches incomplete artifacts before they're used |
| `version:` bumped after change | ‚úÖ Yes | Tracks change history, enables caching |
| `scope.covers:` topic removed from content | ‚öÝÔ∏è Semi ‚Äî can detect if a topic string disappears | Catches coverage regression during refactoring |
| Diff touches a `**Rule**:` block (N-1) | ‚úÖ Yes | **Deterministic breaking classification** ‚Äî the single mechanism that enables autonomy |
| Diff touches only `**Rationale**:` or `**Example**:` blocks | ‚úÖ Yes | **Deterministic non-breaking classification** ‚Äî eligible for autonomous execution |
| "Does this change contradict `goal:`?" | ‚ùå No ‚Äî LLM judgment | Improves over no-metadata baseline but NOT deterministic |
| "Does this change violate `boundaries:`?" | ‚ùå No ‚Äî LLM judgment on prose boundaries | Same ‚Äî better than without, but probabilistic |

**Key insight**: Metadata enables *structural* deterministic checks (fields exist, version bumped, scope topics present). N-1 enables *content* deterministic checks (which block type changed). Together they cover two layers. But goal/boundary alignment remains LLM judgment ‚Äî and that's OK, because it's still better than pure LLM judgment without any metadata reference point.

| Risk | Mitigation |
|---|---|
| Metadata additions increase token consumption | Keep metadata fields minimal (5 fields, ~50 tokens per artifact). Monitor total token impact after Phase 0 |
| N-1 refactoring breaks existing context file consumers | Refactor one file at a time, validate all consumers after each change. Start with lowest-blast-radius files (Tier 5 ‚Üí Tier 0) |
| Agent consolidation removes specialization | Action 4.4 is assessment only. Any consolidation requires separate human-approved plan |
| Template deduplication breaks references | Audit all consumers before removing any template (Action 6.5). Update references first |
| Structured logs add repo clutter | State folder is gitignore-able if preferred. Portable deployments can keep state external |
