# Vision v11 Update Plan: Context File Creation/Review/Improvement Parity

> Context files are the foundation layer — every other artifact's quality is bounded by the quality of the guidance it loads. The vision says this (R-S10, R-S9). But the operational mechanisms don't fully enforce it. This plan closes the gaps.

## 1. The principle

The vision v11 establishes a clear hierarchy:

```
Context files (guidance layer)     — define rules
  ↓ quality ceiling (R-S10)
Instruction files                  — auto-inject testable rules
  ↓ quality ceiling (R-S10)
Agent bodies                       — consume guidance + implement logic
  ↓ quality ceiling (R-S10)
Prompts                            — orchestrate agents
```

**R-S10 says**: An artifact's achievable quality is bounded by the quality of its dependencies. Fix the context file first, then the agent, then the prompt.

**R-S9 says**: Guidance quality is a precondition for autonomy. The 6 properties (clarity, non-redundancy, non-contradiction, completeness, prioritization, actionability) must be verified on context files BEFORE trusting consumers that load them.

**The Assess layer says**: "Assess guidance artifacts first — apply construction invariants if updates are needed. Then assess consumer artifacts — check adherence against the already-validated guidance."

This means every dimension applicable to context files must be:
1. **Invocable** — via `--dim` on `/pe-meta-context-review`
2. **Enforced during creation** — via construction invariants in `/pe-meta-context-create-update` and `/pe-meta-context-design`
3. **Assessed first** — in system-wide reviews, before any consumer artifacts
4. **Applied to the set** — not just individual files but the context file set as a system

## 3. Identified gaps

### Gap 1: No `--dim context-full` group

**Problem**: There's no dimension group that covers ALL 15 context-applicable dimensions. A user who wants "full review of this context file" must use `--dim full` which wastes processing on 12 non-applicable dimensions (D4-D5, D16-D18, D20-D21, D23-D27) — or manually specify dimensions.

`--dim context-health` covers 8 of 15 applicable dimensions (D6-D12, D22) but misses:
- D1 (metadata), D2 (references), D3 (token-budget) — structural basics
- D13 (source-verification) — cited sources support claims
- D14 (craftsmanship) — N-1 separation, naming
- D15 (vision-alignment) — rationale compliance
- D19 (artifact-structure) — goal/scope sizing

**Impact**: Users must manually combine groups (`--dim structural --dim context-health --dim strategic`) or use `--dim full` with wasted work.

**Severity**: MEDIUM — usability gap, not a capability gap.

### Gap 2: D17 (cross-coherence) scoped too narrowly

**Problem**: D17 is currently marked "Agents, prompts" — it checks that "loaded dependencies don't contradict each other." But the underlying concern applies equally when the TARGET is a context file: "do the rules in this context file contradict rules in peer context files that consumers load alongside it?"

Example: `01.04-tool-composition-guide.md` defines tool count limits. `01.06-system-parameters.md` also defines tool count limits. A consumer agent loads both. If they contradict, the consumer has conflicting guidance. D17 today catches this only when reviewing the CONSUMER. It should also be catchable when reviewing EITHER context file.

**Impact**: Cross-context contradictions can only be detected from the consumer's perspective, not from the guidance file's perspective. This is the exact gap that allowed the slash-command H12 issue — a guidance file had a rule but the detection only ran at the consumer level.

**Severity**: HIGH — violates the principle that context files should be assessed FIRST (R-S10). If cross-context coherence can only be checked from consumers, the bottom-up ordering is broken.

### Gap 3: No explicit dimension applicability matrix

**Problem**: The dimension catalog has an "Applicable to" column per dimension, but there's no consolidated matrix showing "for artifact type X, these dimensions apply." The pe-meta-validator has a type-dispatch phase that loads type-specific checklists, but the dimension catalog doesn't explicitly say "when reviewing a context file, skip D4, D5, D16, D18, D20-D21, D23-D27."

**Impact**: The validator may attempt to run non-applicable dimensions, producing confusing "N/A" results, or implementers may not know which dimensions to include in context-specific review groups.

**Severity**: MEDIUM — operational clarity gap.

### Gap 4: Bottom-up ordering not explicit in `/pe-meta-update`

**Problem**: The vision says "assess guidance artifacts first, then consumers" (Assess § Artifact-type-aware routing). But the pe-meta update plan's P4 (`/pe-meta-update` UPDATE) says only "Map audit phases to dimensions D1-D27" without specifying artifact type ordering.

A full system review (`/pe-meta-update fullcheck`) should process artifacts in this order:
1. Context files (guidance layer) — construction invariants
2. Instruction files — minimization + non-contradiction with context
3. Agent bodies — adherence against validated guidance
4. Prompts — adherence + workflow correctness
5. Templates, snippets, hooks — structural correctness

This ordering is in the vision but NOT in the pe-meta update plan's `/pe-meta-update` spec.

**Impact**: Without explicit ordering, a full review might assess agents before their context files are validated — defeating R-S10's purpose.

**Severity**: HIGH — undermines the foundational principle.

### Gap 5: `--dim quality` group missing D8 (prioritization)

**Problem**: The `--dim quality` group includes D6, D7, D9, D10, D11, D27 but NOT D8 (prioritization). However, prioritization is one of the 6 guidance quality prerequisite properties. The vision explicitly lists it alongside the other five.

`--dim context-health` does include D8, but `--dim quality` doesn't. This means `/pe-meta-context-review --dim quality` doesn't check all 6 prerequisite properties.

**Impact**: The quality group — which is supposed to be the "guidance quality prerequisite check" — is incomplete. Missing prioritization means the autonomy gate (R-S9) has a hole.

**Severity**: HIGH — directly affects the autonomy prerequisite assessment.

### Gap 6: No guidance-first mode for context-to-context peer review

**Problem**: `/pe-meta-adherence` checks "which consumers implement this guidance file's rules?" — direction: guidance → consumers. But there's no equivalent for "which peer context files that get loaded alongside this one by consumers might contradict it?" — direction: guidance → peers.

This is related to Gap 2 but distinct: Gap 2 is about D17's scope; Gap 6 is about the absence of a review mode that starts FROM a context file and examines its peer ecosystem.

**Impact**: Context file authors can't easily answer "is my new rule safe to add, or does it conflict with existing rules in files that consumers load alongside mine?"

**Severity**: MEDIUM — the information is accessible via `/pe-meta-adherence` on the context file + manual analysis, but there's no dedicated workflow.

### Gap 7: Context file autonomy gradient not explicit

**Problem**: The pe-meta update plan doesn't specify how the autonomy gradient applies to context file changes specifically. Context files are the foundation layer (lowest tier in every dependency chain). Per R-S6, low-tier artifacts have the highest blast radius. But the plan doesn't explicitly state:
- What autonomy level context file changes route to
- Whether the tier-blast-radius classification for context files differs from other types
- That context file changes should trigger re-assessment of all consumers (per R-S10)

**Impact**: The plan treats context files with the same autonomy gradient as all artifacts, but their position as the foundation layer should impose stricter defaults.

**Severity**: MEDIUM — the vision already says Tier 0 → always escalate, and context files sit at Tier 00. But the pe-meta plan doesn't make this explicit for context-specific workflows.

## 4. Proposed changes

### V1 (HIGH): Add D8 to `--dim quality` group

**Location**: Plan A § Dimension groups + pe-meta update plan § C2 (dimension catalog)

**Change**: Add D8 (prioritization) to the `--dim quality` group.

**Current**: `--dim quality` = D6, D7, D9, D10, D11, D27

**Target**: `--dim quality` = D6, D7, **D8**, D9, D10, D11, D27

**Rationale**: The 6 guidance quality properties are clarity (D9), non-redundancy (D7), non-contradiction (D6), completeness (D10), prioritization (D8), actionability (D11). The `--dim quality` group MUST cover all 6 to serve as the autonomy prerequisite gate.

### V2 (HIGH): Extend D17 scope to include context files

**Location**: Plan A § Dimension catalog + vision v11 § Cross-dependency coherence

**Change**: Extend D17 (cross-coherence) from "Agents, prompts" to "Agents, prompts, **context files (peer review)**".

For context files, D17 answers a different question: "Do the rules in THIS context file contradict rules in peer context files that consumers load alongside it?" The mechanism:
1. From STRUCTURE-README, identify which consumers load this context file (`context_dependencies:`, `📖` references)
2. For each consumer, identify what OTHER context files it loads
3. Check for contradictions between rules in THIS file and rules in PEER files

This is the reverse direction of D17 for consumers (which checks "do MY loaded deps contradict each other?"). For context files, it checks "does MY content contradict content in files that get loaded alongside me?"

**Rationale**: Cross-context contradictions are a foundation-layer concern. Detecting them only from the consumer perspective violates R-S10's bottom-up ordering — the contradiction exists in the guidance layer regardless of which consumer reveals it.

- [ ] Apply to Plan A (D17 row in dimension catalog)
- [ ] Apply to pe-meta update plan (C2 dimension catalog + A4 validator Phase 3)

### V3 (HIGH): Add explicit bottom-up ordering to `/pe-meta-update`

**Location**: pe-meta update plan § P4 (pe-meta-update)

**Change**: Add explicit artifact-type ordering to the `/pe-meta-update` fullcheck workflow:

```
Phase A: Context files — all applicable dimensions with construction invariant enforcement
Phase B: Instruction files — D1-D3, D6-D8, D11-D14, D19 + minimization check (R-S8)
Phase C: Agent bodies — all applicable dimensions + adherence (D16) against Phase A/B results
Phase D: Prompts — all applicable dimensions + adherence (D16) against Phase A/B/C results
Phase E: Templates, snippets, hooks — structural dimensions (D1-D3, D14)
Phase F: Skills — structural + strategic dimensions
```

Each phase validates artifacts of that type. If Phase A finds issues in context files, those must be addressed (or accepted as known issues) before Phase C/D assess adherence — because consumer adherence is only meaningful against validated guidance.

**Rationale**: The vision's Assess layer defines this ordering (§ Artifact-type-aware routing) but the pe-meta update plan's P4 doesn't operationalize it.

- [ ] Apply to pe-meta update plan (P4 changes)

### V6 (MEDIUM): Make context file autonomy implications explicit

**Location**: pe-meta update plan or vision v11 § Assess

**Change**: Add a note connecting context file tier to autonomy implications:

> **Context file change routing.** Context files in `.copilot/context/00.00-prompt-engineering/` carry Tier 00 prefix (the lowest tier = highest blast radius per R-S6). Combined with their position as the foundation layer (R-S10), context file changes follow stricter default routing:
>
> | Change type | Autonomy level | Rationale |
> |---|---|---|
> | Metadata fixes (version, last_updated) | Autonomous | Non-breaking, deterministic |
> | Reference fixes (broken links, stale paths) | Autonomous with notification | Non-breaking, deterministic but user should know |
> | Wording improvements within scope | Human approval required | May affect clarity (D9) which propagates to consumers |
> | Rule addition/modification | Human approval required | Breaking for consumers — adherence checks needed |
> | Scope expansion (new topics) | Human approval required | Changes what consumers can rely on |
> | Scope contraction (removing coverage) | Human approval required | May break consumer adherence |
> | File split/merge/reorganization | Human approval required | Affects category mapping, consumer references |
>
> After ANY context file change, the system SHOULD flag affected consumers for re-assessment. This is the operational realization of R-S10: when the foundation changes, the ceiling shifts for everything above.
