#### The injection mechanism problem (audited)

Instruction files and context files have fundamentally different delivery mechanisms:

| Mechanism | Instruction files (`applyTo`) | Context files (`.copilot/context/`) |
|---|---|---|
| **Injection** | **Automatic** — injected into every file matching the `applyTo` glob | **Semantic search** — loaded when VS Code determines relevance to the conversation |
| **Guarantee** | **100%** — always present in LLM context when editing matching files | **Probabilistic** — may or may not load depending on query relevance |
| **Explicit load** | Not needed — glob handles it | Requires `📖` reference in agent body or explicit `read_file` |
| **Token cost** | Always paid — even when not relevant | Paid only when loaded |

**The risk**: Moving a rule from `pe-common.instructions.md` (guaranteed injection) to a context file (probabilistic loading) can cause the rule to **silently disappear** from the LLM's context — even though the content exists in the destination file.

#### Rule-by-rule risk analysis

The plan identifies 4 behavioral rules to extract. For each, the analysis checks: (1) does the destination already contain the content? (2) do agents explicitly reference the destination via `📖`?

### Step 5: Create artifact-type dispatch reference ✅ DONE

**Status**: Completed 2026-04-27

**Risk**: LOW — purely additive.

**Results**:

**Problem**: Consolidated agents need a lookup table to know which instruction file, template, and validation rules to load for each artifact type.

#### Who needs this and who doesn't (audited)

| PE model | Needs dispatch? | Why |
|---|---|---|
| **Consolidated agents** (`pe-con-researcher`, `pe-con-builder`, `pe-con-validator`) | ✅ YES | They handle ALL artifact types — must determine which instruction file, template, and rules to load per request |
| **Granular agents** (`pe-gra-agent-builder`, `pe-gra-hook-researcher`, etc.) | ❌ NO | Artifact type is hardcoded in the agent name and body — `pe-gra-agent-builder` already references `pe-agents.instructions.md` directly |
| **Meta agents** (`pe-meta-validator`, etc.) | ❌ NO | Operate at system level, not per-artifact-type |

#### Why NOT a context file

The original plan placed this in `.copilot/context/00.00-prompt-engineering/01.08-artifact-type-dispatch.md`. This is wrong for two reasons:

1. **Unwanted injection into granular agents**: Context files load via semantic search. During PE work, VS Code would likely inject the dispatch table into granular agents (e.g., `pe-gra-agent-builder`), adding ~500 tokens of irrelevant "determine artifact type" instructions to an agent that already knows its type. This wastes tokens and could confuse the agent with dispatch logic it shouldn't execute.

2. **Unreliable injection into consolidated agents**: The dispatch table is a CRITICAL dependency for consolidated agents — without it, they can't function. Relying on semantic search (probabilistic) for a critical dependency is unsound. The consolidated agents must `read_file` it explicitly in their Phase 1 regardless, making the context file placement redundant.

#### Chosen approach: reference file with explicit loading

**Location**: `.github/templates/00.00-prompt-engineering/artifact-type-dispatch.template.md`

**Why this location**:
- Templates are NOT indexed by semantic search — they're loaded only via explicit `read_file`
- Zero risk of polluting granular or meta agents
- Consolidated agents load it explicitly in Phase 1 as a mandatory first step
- Follows the existing convention: templates are referenced by agents, not auto-injected

**Alternatively**: the dispatch table could be inlined directly into each consolidated agent's body (~30 lines). This avoids an external file dependency but adds ~30 lines × 3 agents = 90 lines of duplication. Given the 200-line agent limit, the external reference is preferred.

**Content** (≤30 lines of table + header):

```markdown
---
title: "Artifact-type dispatch reference"
description: "Lookup table for consolidated PE agents — maps each artifact type to its instruction file, templates, validation rules, and context files"
version: "1.0.0"
audience: "pe-researcher, pe-builder, pe-validator ONLY — granular agents do NOT use this"
---

# Artifact-type dispatch reference

## Dispatch table

| Artifact type | Instruction file | Primary template | Key context files | Type-specific validation |
|---|---|---|---|---|
| **Agent** | `pe-agents.instructions.md` | `agent.template.md` | `02.04-agent-shared-patterns.md`, `01.04-tool-composition-guide.md` | Tool count 3–7, mode alignment, handoff resolution, boundary completeness ≥3/1/2 |
| **Prompt** | `pe-prompts.instructions.md` | `prompt.template.md` | `02.03-orchestrator-design-patterns.md`, `01.04-tool-composition-guide.md` | Gate checks, tool scope, agent references resolve |
| **Context file** | `pe-context-files.instructions.md` | — | `01.01-context-engineering-principles.md` | ≤2,500 tokens, single source of truth, no circular deps |
| **Instruction file** | `pe-instruction-files.instructions.md` | `instructions.template.md` | `01.07-critical-rules-priority-matrix.md` | applyTo conflict-free, ≤1,500 tokens, only testable rules |
| **Skill** | `pe-skills.instructions.md` | `skill.template.md` | `03.01-progressive-disclosure-pattern.md` | Description ≤1024 chars, Level 1 <100 tokens, required sections |
| **Template** | `pe-templates.instructions.md` | — | `03.07-template-authoring-patterns.md` | ≤100 lines, .template.md extension, category prefix |
| **Hook** | `pe-hooks.instructions.md` | — | `03.03-agent-hooks-reference.md` | Valid JSON, supported lifecycle event, companion script exists |
| **Prompt snippet** | `pe-prompt-snippets.instructions.md` | — | — | ≤500 words, no YAML frontmatter, self-contained |

## How consolidated agents use this table

1. **Determine artifact type** from user request or handoff context
2. **`read_file`** this dispatch table (MANDATORY Phase 1 step)
3. **Load the instruction file** for that type (`read_file`)
4. **Load the primary template** if creating/updating (`read_file`)
5. **Load key context files** if the artifact interacts with that domain
6. **Apply type-specific validation** rules during pre-save checks
```

#### Impact on other PE models

| PE model | Impact | Why |
|---|---|---|
| Granular agents | **ZERO** | File is a template, not a context file — not auto-injected. Granular agents never reference it. |
| Meta agents | **ZERO** | Same — no auto-injection, no references. |
| Instruction files | **ZERO** | `applyTo` globs don't match templates. |
| Context files | **ZERO** | No cross-reference needed. |
| Existing prompts | **ZERO** | No reference to this file. |
