# PE artifact compliance plan: closing vision v7 gaps

## 🎯 Problem summary

Comparison of 6 PE prompts across pe-meta and pe-gra tiers reveals **8 gaps** against vision v7 standards:

- **pe-gra prompts** excel at operational patterns (handoff contracts, summarization, error recovery, test scenarios) but lack YAML metadata and use Level 2 file references
- **pe-meta prompts** excel at strategic alignment (vision loading, double validation, metadata completeness) but lack operational detail (handoff contracts, test scenarios, structured boundaries)
- **Both tiers share test scenario gaps** — systemic across the ecosystem

The ideal artifact combines both: pe-meta's metadata + vision alignment + pe-gra's operational depth.

---

## 📋 Gap inventory

| # | Gap | Severity | Affected | Fix approach |
|---|---|---|---|---|
| **G1** | Missing YAML metadata (scope, boundaries, version, last_updated) | HIGH | All pe-gra prompts (~17 files) | Batch add structured YAML |
| **G2** | Level 2 file refs instead of Level 1.5 categories | HIGH | pe-gra prompts with context file references | Convert to category pattern (already done for instruction files — same mechanical change) |
| **G3** | No embedded test scenarios | MEDIUM | pe-meta-pe-artifact-design, pe-meta-pe-artifact-review, pe-gra-prompt-design | Add ≥3 scenarios per PE quality bar |
| **G4** | pe-meta prompts lack handoff contracts | MEDIUM | pe-meta-pe-artifact-design, pe-meta-pe-artifact-review, pe-meta-pe-artifact-create-update | Add Include/Exclude/Max-tokens table |
| **G5** | pe-meta-pe-artifact-review has no structured boundaries | MEDIUM | pe-meta-pe-artifact-review | Add Always (≥5) / Ask (≥2) / Never (≥3) |
| **G6** | pe-gra prompts don't reference vision | MEDIUM | All pe-gra prompts | Intentional — pe-gra is structural-only. Document as design decision, not gap |
| **G7** | pe-meta prompts lack summarization protocol | LOW | pe-meta-pe-artifact-design, pe-meta-pe-artifact-review | Add per-phase budget or declare N/A |
| **G8** | snippet-create-update lacks operational patterns | LOW | pe-gra-prompt-snippet-create-update | Add error recovery + response management |

---

## 💡 Recommended approach

### Priority ordering (risk × blast radius)

| Priority | Gaps | Why first |
|---|---|---|
| **P1** | G1 + G2 (pe-gra metadata + category refs) | HIGH severity, 17 files — same mechanical batch change as the instruction file migration we already completed |
| **P2** | G4 + G5 (pe-meta handoff contracts + boundaries) | MEDIUM — pe-meta prompts are strategic, fixing their operational gaps makes them exemplary |
| **P3** | G3 (test scenarios) | MEDIUM — systemic gap across both tiers, but lower blast radius |
| **P4** | G7 + G8 (summarization + snippet operational) | LOW — polish items |
| **—** | G6 | Not a gap — document as intentional design decision |

### Execution model

Use `/meta-pe-create-update` for pe-meta prompt fixes (they're PE-for-PE artifacts).
Use `/pe-con-create-update` for pe-gra prompt fixes (batch mechanical changes).

### Change 1: Add YAML metadata to all pe-gra prompts (G1) — ✅ DONE

**Files:** ~17 pe-gra prompts in `.github/prompts/00.02-pe-granular/`
**Operation:** For each prompt, add missing YAML fields:

```yaml
scope:
  covers:
    - "[infer from prompt content]"
  excludes:

  - "[extract from Always Do / Never Do sections]"
```

**Approach:** Use a subagent to batch-read each prompt, extract scope/boundaries from content, and add structured YAML. Validate each with `pe-con-validator`.

**Risk:** LOW — additive metadata doesn't change prompt behavior.

### Change 4: Add structured boundaries to pe-meta-pe-artifact-review (G5) — ✅ DONE

**File:** `.github/prompts/00.09-pe-meta/pe-meta-pe-artifact-review.prompt.md`
**Operation:** Expand the YAML `boundaries:` into a structured Always/Ask/Never section in the body:

```markdown
## CRITICAL BOUNDARIES

### ✅ Always Do (≥5)
- Verify target is PE-for-PE before proceeding (redirect domain artifacts)
- Load vision document, STRUCTURE-README, strategic review criteria, dependency map
- Run BOTH structural AND strategic validation
- Report blast radius for high-dependency artifacts
- Produce combined verdict (structural + strategic)

### ⚠️ Ask First (≥2)
- When structural PASS but strategic FAIL — ask user before recommending fixes
- When artifact has 6+ dependents — confirm scope of ecosystem coherence check

### 🚫 Never Do (≥3)
- NEVER modify the file being reviewed (plan mode = read-only)
- NEVER skip strategic validation even if structural validation passes
- NEVER approve artifacts that contradict vision rationales without explicit override
```

**Risk:** LOW — strengthens an existing prompt, doesn't change behavior.

### Change 6: Clarify vision scope includes all-domain PE artifacts (G6) — INFO

**Finding re-assessed:** G6 was originally "pe-gra prompts don't reference vision" — framed as a tier-level design decision. On reflection, this is a **vision clarity issue**, not an implementation gap.

The vision defines self-updating for PE artifacts but doesn't explicitly state that it covers PE artifacts **for any domain** (article-writing, documentation, devops, etc.), not just PE-for-PE. The self-update architecture applies universally — maintaining an article-writing agent is the same 9-step workflow as maintaining a PE context file. The PE-for-PE case is strategically special (higher quality bar, vision alignment) but architecturally identical.

**What the vision should clarify** (in a future v8 update):
- Self-updating covers PE artifacts across all domains, not just the PE domain
- The quality bar and validation depth may vary by domain criticality, but the mechanism (metadata-guarded changes, category references, dependency tracking) is universal
- Implementation tiers (pe-meta/pe-con/pe-gra/pe-sim) are NOT part of the vision — they're one possible implementation

**Action for now:** No file changes. Record as a vision clarification candidate for the next vision revision. The current vision text doesn't contradict this — it just doesn't make it explicit.

**Risk:** NONE
