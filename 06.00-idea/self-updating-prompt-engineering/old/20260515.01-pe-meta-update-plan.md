### 2.5 Target instruction files

**No instruction file changes.** The four proposed pe-meta-specific rules (type dispatch, no pe-gra delegation, exemplary quality bar, dimension mapping) are **NOT** added to `pe-agents.instructions.md` because:

1. **R-S8 violation**: `pe-agents.instructions.md` has `applyTo: '.github/agents/**/*.agent.md'` — it auto-injects into ALL agents (pe-gra, pe-meta, domain). pe-meta-specific rules injected there would waste tokens on 24+ non-pe-meta agents and create contradictions (e.g., "exemplary bar" vs pe-gra's "standard bar").
2. **Collision risk**: "MUST NOT delegate to pe-gra" would inject into pe-gra agents themselves — nonsensical and confusing.
3. **Not universally testable**: These rules are behavioral/strategic, not mechanical/boolean — they belong in context files or agent bodies per R-S8.

**Where these rules live instead:**

| Rule | Location | Mechanism |
|---|---|---|
| Type dispatch | Each pe-meta agent's boundaries (A1-A5) | Agent body — consumer controls loading |
| No pe-gra delegation | pe-meta-validator boundaries (A4) | Agent body — already specified |
| Exemplary quality bar | pe-meta-builder boundaries (A3) + `05.06` context file | Agent body + explicit context loading |
| Dimension mapping (D1-D27) | Each pe-meta agent's boundaries (A1-A5) + `05.07` dimension catalog | Agent body + explicit context loading |

#### A3: pe-meta-builder (NEW)

**Current state**: Does not exist.

**vision v12 requirement**: Self-contained builder that creates/modifies ALL 8 PE artifact types with exemplary quality bar, construction invariants, and pre/post-change metadata guards.

**Specification:**

```yaml
description: "PE artifact builder — creates and modifies all PE-for-PE artifact types with exemplary quality bar, construction invariants, and metadata-guarded changes"
agent: agent
tools:
  - read_file
  - grep_search
  - file_search
  - list_dir
  - create_file
  - replace_string_in_file
  - multi_replace_string_in_file
handoffs:
  - label: "Validate Changes"
    agent: pe-meta-validator
    send: true
version: "1.0.0"
goal: "Create or modify PE artifacts that meet the exemplary quality bar with construction invariants enforced during creation, not just validated after"
scope:
  covers:
    - "Create/modify all 8 PE artifact types (context, agent, prompt, instruction, skill, template, hook, snippet)"
    - "Exemplary quality bar enforcement (≥5/2/3 boundaries, full metadata, category refs)"
    - "Guidance quality construction invariants (6 properties — clarity, non-redundancy, non-contradiction, completeness, prioritization, actionability)"
    - "Pre-change compatibility gate (metadata guard against goal/scope/boundary/rationale violations)"
    - "Post-change reconciliation (mandatory metadata update after every change)"
    - "Type dispatch — select appropriate build patterns per artifact type"
  excludes:
    - "Research (pe-meta-researcher handles this)"
    - "Design (pe-meta-designer handles this)"
    - "Validation (pe-meta-validator handles this — builder hands off after build)"
boundaries:
  - "MUST enforce exemplary quality bar on every PE artifact created/modified"
  - "MUST run pre-change guard before ANY modification (check goal/scope/boundary/rationale preservation)"
  - "MUST run post-change reconciliation after EVERY modification (update version, metadata, timestamps)"
  - "MUST apply construction invariants during creation of guidance artifacts (context files)"
  - "MUST hand off to pe-meta-validator after every build cycle"
  - "MUST NOT exceed 3 build iterations per artifact — escalate after 3 failures"
  - "MUST read the COMPLETE target file before modifying"
  - "MUST include 3-5 lines of context in all replace operations"
  - "MUST use type dispatch to apply appropriate build patterns"
rationales:
  - "Self-contained builder eliminates dependency on pe-gra builders for PE artifacts"
  - "Construction invariants catch quality issues during creation — cheaper than fix cycles"
  - "Pre/post-change guards prevent content drift and metadata staleness (closed feedback loop)"
  - "Type dispatch enables single agent to handle all artifact types with type-appropriate patterns"
```

**Key capabilities:**
- Type dispatch based on file path/extension (same pattern as enhanced validator)
- Construction invariants for guidance artifacts (6 properties enforced during build)
- Pre-change gate: check proposed changes against declared metadata
- Post-change reconciliation: update version, scope, rationales after every change
- Exemplary quality bar: ≥5/2/3 boundaries, full metadata, category references
- **Risk-calibrated context file autonomy**: for context files, classify changes using the general autonomy gradient — non-breaking validated changes (pre-change guard + construction invariants + peer coherence pass) proceed autonomously with notification. Decompose large changes into smaller non-breaking steps to maximize autonomous execution. Flag affected consumers for re-assessment after any rule-affecting change.

**Structure of the rewritten validator:**

```
Phase 0: Input parsing
  - Parse --dim parameter → selected dimensions (default: all applicable per type)
  - Parse --with-deps flag → dependency-aware mode
  - Detect artifact type from file path → load type-specific checklist
  - Apply dimension applicability matrix → skip non-applicable dimensions

Phase 1: Structural dimensions (D1-D5, D14) — deterministic where possible
  - D1 metadata: YAML field presence and completeness
  - D2 references: 📖, markdown links, slash-commands resolve
  - D3 token-budget: line/token count within type limit
  - D4 tool-alignment: mode/tool rules (agents, prompts only)
  - D5 boundaries: three-tier completeness (exemplary bar)
  - D14 craftmanship: N-1, naming, progressive disclosure

Phase 2: Quality dimensions (D6-D11) — reasoning required
  - D6 consistency: non-contradiction within file + loaded deps
  - D7 non-redundancy: each rule in one canonical location
  - D8 prioritization: conflict precedence explicit
  - D9 clarity: LLM agreement test (2-pass)
  - D10 completeness: gaps where guidance needed but absent
  - D11 actionability: rules translatable to boolean checks

Phase 3: Strategic dimensions (D15-D19) — reasoning required
  - D15 vision-alignment: applicable rationales from v12
  - D16 adherence: artifact implements loaded rules
  - D17 cross-coherence: loaded deps don't contradict each other
       For context files: PEER MODE — check against peer context files loaded by same consumers
  - D18 coverage: use cases, unhappy paths, error recovery
  - D19 artifact-structure: goal/scope correctly sized

Phase 4: Efficiency dimensions (D20-D27) — mixed
  - D20 token-chain: consumer guidance set within budget
  - D21 deterministic-first: process phases optimally split
  - D22 context-optimization: system-level context file set review
  - D23 reference-efficiency: reference count, load necessity
  - D24 handoff-efficiency: max tokens, summarization, structured data
  - D25 processing-efficiency: targeted review, model routing per step
  - D26 model-routing: task-model alignment per step
  - D27 model-adherence: patterns follow target model best practices

Phase 5: External dimensions (D12-D13) — requires web access
  - D12 staleness: source references current
  - D13 source-verification: cited sources support claims

Phase 6: Dependency-aware (if --with-deps)
  - Identify all dependencies
  - Run Phase 1-4 on each dependency (recursive, max depth 2)
  - Cross-dependency coherence (D17)
  - Adherence verification (D16)
  - Consumer burden analysis

Phase 7: System-wide ordering (if invoked from /pe-meta-update fullcheck)
  - Execute Phase A-F ordering from v12:
    Phase A: Context files (all applicable dims + construction invariants + peer cross-coherence)
    Phase B: Instruction files (structural + minimization + non-contradiction with context)
    Phase C: Agent bodies (all applicable dims + adherence against Phase A/B)
    Phase D: Prompts (all applicable dims + adherence against Phase A/B/C)
    Phase E: Templates, snippets, hooks (structural dims)
    Phase F: Skills (structural + strategic dims)

Phase 8: Report generation
  - Severity-ranked findings per dimension
  - Combined structural + strategic + efficiency report
  - Adherence matrix (if guidance-first mode)
  - Phase ordering report (if system-wide)
```
