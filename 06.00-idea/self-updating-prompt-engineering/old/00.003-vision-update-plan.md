## 🧭 Per-artifact-type autonomous-update readiness

For each artifact type, the vision implicitly assumes autonomous update is possible. Reality check — the table below evaluates each type against four prerequisites:

- **Parseable structure** — can a deterministic tool diff the artifact and distinguish a rule change from a rationale change?
- **Semantic metadata** — does the artifact declare goal/scope/boundaries in a machine-readable way (B-5)?
- **Declared consumers** — can the system compute blast radius from the artifact alone?
- **Validation signal** — does the artifact type have a validator agent and a defined pass/fail contract?

Scoring: ✅ ready · 🟡 partial · ❌ missing.

| Artifact type | Parseable rules/rationales | Semantic metadata | Declared consumers | Validation signal | Autonomous-update readiness |
|---|---|---|---|---|---|
| Context files | 🟡 (priority matrix is structured; other files mix rules + prose) | ❌ (no goal/scope/boundary keys) | ✅ (dependency map + `STRUCTURE-README`) | ✅ (`context-validator`) | Medium — safe for metadata/reference updates; risky for rule edits |
| Instruction files | 🟡 (severity index is structured; rest is prose) | ❌ | 🟡 (`applyTo` glob; no explicit consumer list) | ✅ (`instruction-validator`) | Medium — same as context |
| Agent files | 🟡 (YAML + boundaries are structured; body is prose) | 🟡 (role declared; no scope/boundary keys) | ✅ (handoffs + referenced-by in dependency map) | ✅ (`agent-validator`) | Medium-high — tool/handoff edits are safe; role edits are breaking |
| Prompt files | 🟡 | 🟡 (purpose in description) | 🟡 | ✅ (`prompt-validator`) | Medium |
| Skill files | ✅ (SKILL.md has strict schema) | 🟡 (description is trigger; no structured scope) | ❌ (no declared consumers) | ✅ (`skill-validator`) | High for content, low for discovery metadata |
| Template files | ✅ (placeholder markers + categories) | 🟡 | 🟡 (template_metadata.consumers in newer templates) | ✅ (`template-validator`) | High — lowest blast radius; safest autonomous updates |
| Prompt snippets | ✅ (tiny, focused) | N/A | 🟡 | ✅ (`prompt-snippet-validator`) | High |
| Hooks | ✅ (JSON schema) | N/A | N/A (scripts) | ✅ (`hook-validator`) | High for JSON edits; medium for script edits |

**Implication for the vision.** The vision currently treats all artifact types uniformly ("detect → assess → propose → execute" applied to any artifact). But readiness is **not uniform** — templates, snippets, and hooks are ready today; context and instruction files need semantic metadata (B-5) and rule separation (N-1) before rule-level autonomous updates are safe. The vision should name this explicitly and sequence the rollout: start autonomous execution on high-readiness types, add it to medium-readiness types once the missing infrastructure lands.

## 🆕 New findings (not in plan1)

These emerge from the per-artifact-type analysis and the user's explicit quality properties (rule/rationale separation, tiered structuring for autoupdate, goal/role alignment across artifacts).

### N-1: Rules, rationales, and examples are not machine-separable inside artifacts (HIGH)

Plan1's B-1 introduces a breaking/non-breaking classification but assumes the system can tell *what kind* of content changed. In practice, most context and instruction files mix three content types in the same section:

- **Rules** (REQUIRED) — enforceable statements (MUST/NEVER/ALWAYS). Changing these is breaking for consumers.
- **Rationales** (OPTIONAL, recommended when not obvious) — *why* the rule exists. Should be made explicit **unless the reason is trivially deducible from the rule itself**. Changing these is non-breaking; consumers don't reference rationales.
- **Examples** (OPTIONAL, add only when they materially improve robustness or effectiveness) — illustrative code/prose. Include only when the rule is ambiguous without an example, when misapplication is a known failure mode, or when the example is itself referenced as canonical. Changing an example is non-breaking unless the example *is* the rule.

Today a deterministic diff can only tell that bytes changed. It can't tell whether those bytes were a MUST clause (breaking) or an example paragraph (non-breaking). The system has to fall back to LLM judgment for every diff — losing the efficiency and verifiability that justify autonomous execution.

**Impact.** The B-1 risk-assessment framework is limited by this. A fully autonomous loop needs to classify changes cheaply and verifiably. Without rule/rationale separation, every rule-adjacent change routes to LLM judgment or to human approval.

**Fix option.** Adopt a **structural convention** inside every rule-bearing section of context and instruction files, with the rule block mandatory and rationale/example blocks optional:

```markdown
### [Rule name]

**Rule** (REQUIRED — imperative, enforceable; diff here = breaking):
- MUST/NEVER/ALWAYS ...

**Rationale** (OPTIONAL — include unless obvious from the rule; diff here = non-breaking):
- ...

**Example** (OPTIONAL — include only when it improves robustness/effectiveness; diff here = non-breaking unless example is referenced as canonical):
```

Guidance for authors and validators:

- Always write the rule block. A section without a rule block is a rationale or example for some other rule — relocate it.
- Add a rationale when a reader could plausibly disagree with the rule, when the rule is counterintuitive, or when a downstream artifact needs to reference *why* (not just *what*). Skip it for rules whose justification is self-evident (e.g., "YAML frontmatter MUST be valid").
- Add an example when the rule's correct application is non-obvious, when examples are the fastest way to communicate the boundary, or when a known failure mode needs a concrete counter-example. Skip examples that only restate the rule in longer form.
- **Economy matters.** Optional blocks consume tokens in every consumer load. Every added rationale/example should pay for its token cost by reducing misapplication risk or eliminating LLM judgment on diffs.

Rules become parseable (deterministic diff classifies the change), rationales become editable without cascading revalidation, and examples can be refreshed routinely. The priority matrix already implements a version of this (the `Check` column is the rule, the `Canonical source` column is the rationale — and no example column exists, which is correct for a pure lookup table). Generalize the pattern without forcing rationale/example blocks where they add no value.

**Connects to.** B-1 (classification precision), B-5 (semantic metadata — rules fit naturally under `boundaries`), H3 (single source of truth — rationales can reference canonical context files instead of restating rules), the user's explicit request for "clear separation of rules vs rationales and examples."

### N-2: Goal/role alignment is never verified across artifact chains (HIGH)

The vision assumes that "goal and role alignment" holds across the artifact chain — a rule declared in a context file is enforced by the matching validator agent, which is invoked by the matching review prompt. Today nothing verifies this alignment. Rule C2 in the priority matrix (no capability removal) addresses it only at the governance level.

Concrete failure modes observed in the current system:

- A new rule added to `pe-common.instructions.md` without a matching check in the relevant validator agent → the rule is documented but never enforced.
- An agent's stated purpose drifts from the prompt that invokes it → the prompt expects behaviors the agent no longer provides.
- A template adds a new required field → no agent or validator knows about it.
- A rule is renamed or split into two rules → every downstream `enforces: [Cn]` reference becomes stale or dangling.

**The core problem: staleness and dangling references under change.** Fine-grained chain coupling (e.g., each agent declares `enforces: [C1, H2, H5]`) is precise but fragile — every rule add/remove/rename breaks every downstream reference. The maintenance cost of keeping fine-grained references fresh can exceed the cost the self-update system is trying to save. The strategy must **maximize enforcement coverage while minimizing reference-maintenance complexity**.

**Fix option — coarse-grained, resilient linkage by default; fine-grained only where it pays.** Introduce a two-level reference convention so the common case is stale-proof and the exceptional case is still expressible:

**Level 1 — folder/domain reference (default).** Artifacts reference a correlated, prioritized *set* of rules by folder or domain, not by individual rule ID. A rule set lives in a single canonical location (e.g., `01.07-critical-rules-priority-matrix.md` for cross-cutting validation rules, `02.01-handoffs-pattern.md` for handoff rules). An agent declares:

```yaml
enforces_rulesets:
  - .copilot/context/00.00-prompt-engineering/01.07-critical-rules-priority-matrix.md  # all CRITICAL + HIGH
  - .copilot/context/00.00-prompt-engineering/02.01-handoffs-pattern.md
```

Adding, removing, or renaming a rule inside a referenced file does NOT break any consumer reference. The consumer commits to enforcing "whatever this ruleset currently contains, at its declared severity." The canonical file is the single source of truth for rule identity and priority (H3); consumers never duplicate rule text.

**Level 2 — specific rule reference (exception).** When an artifact enforces only a subset of rules from a ruleset — or when a specific rule's semantics are load-bearing for that artifact — use explicit rule IDs:

```yaml
enforces_rules:
  - file: .copilot/context/00.00-prompt-engineering/01.07-critical-rules-priority-matrix.md
    ids: [C1, C3]   # only tool alignment + token budget, not the full CRITICAL set
```

Specific rule references MUST be validated — a renamed or removed ID becomes a dangling reference and is flagged as a CRITICAL finding.

**What this buys:**

| Property | Coarse (ruleset) | Fine (rule IDs) |
|---|---|---|
| Add rule to file | No consumer updates | No updates (new rule not yet referenced) |
| Remove rule from file | No consumer updates unless rule was specifically referenced | Dangling ID → flagged |
| Rename rule | No consumer updates unless specifically referenced | Dangling ID → flagged + proposed auto-fix via rename log |
| Reorder priorities | No updates — consumer commits to "current priority order" | No updates |
| Token cost of references | O(ruleset count) — small | O(rule count) — large |

**Prefer Level 1. Use Level 2 only when precision is load-bearing.** The default is coarse linkage; authors must justify fine-grained references. This matches the priority matrix's existing structure (rules are already grouped and prioritized in one file) and matches the user's stated goal: "a set of rules that are correlated and prioritized can be identified in a set of context files then artifacts can refer to the context files by folder so that adding and removing rules would not break dependent artifact."

**Rename/move support.** When a file is renamed or a rule ID changes, the change is recorded in a **rename log** (part of the outcome log schema — N-6) with `old → new` mapping. The self-update system consults the rename log to:

1. Rewrite coarse references automatically (path change is always non-breaking for consumer meaning).
2. Propose auto-fix for fine-grained references (old ID → new ID); auto-applied if the semantic meaning is preserved, escalated otherwise.
3. Detect orphaned references that point to files/IDs not in the rename log — these are true dangling references requiring human review.

**Chain validation checks (hosted in `pe-artifact-coherence-check`):**

- Every `enforces_rulesets` path resolves to an existing file.
- Every `enforces_rules.file` + `ids` resolves to existing IDs (consulting rename log).
- Every rule in a canonical ruleset is referenced by at least one validator agent (orphan-rule detection — rule exists but nothing enforces it).
- Every validator agent's declared ruleset matches its actual checks (declared-vs-implemented gap).

**Connects to.** B-5 (structured metadata), C-3 (audit trail + rename log), H3 (single source of truth for rule text), N-3 (tier determines how strict the chain-alignment check is — Tier 0/1 rules MUST have full chain coverage; Tier 4/5 rules may not), the user's request for "goal and role alignment within artifacts and across artifacts" **with minimum maintenance complexity**.

### N-2b: Staleness avoidance is a top-level goal, not a downstream concern (CRITICAL)

The self-update system exists to prevent PE artifacts from going stale. But the system itself is built from PE artifacts — its own context files, agents, prompts, skills, hooks, and templates. **If the self-update infrastructure becomes stale, it doesn't just lose some effectiveness; it loses all value.** A self-update system that doesn't update itself becomes a generator of wrong decisions.

This is the L-2 risk ("self-updates carry amplified risk") flipped into a goal: staleness in the self-update infrastructure is the highest-consequence failure mode in the entire vision, and the vision currently treats it as a downstream concern rather than a top-level principle.

#### Two kinds of staleness

Staleness is not one failure mode — it's two, and the vision must address both:

**Type A — Structural staleness** (references, metadata, chain alignment). Dangling references, stale validation timestamps, broken chain alignment, mismatched metadata. Deterministic to detect (B-3 integrity pre-step, N-2 chain checks). Fail-closed is the right response.

**Type B — Capability/logic staleness** (obsolescence relative to external change). The managed and managing artifacts still parse, still reference-resolve, still chain-align — but the **logic they encode is obsolete** with respect to the current state of the world:

- A new model family is released with meaningfully different strengths (longer context, better reasoning, different caching behavior) — existing model-specific guidance is now outdated, not broken.
- A platform feature moves from preview to GA — existing guidance still references "preview limitations" that no longer exist.
- A new tool, API, or capability becomes available that could simplify an agent's workflow — the agent still works, but less effectively than it could.
- A best practice evolves in the broader ecosystem (new prompting pattern, new evaluation approach) — internal artifacts still reflect the old practice.

Type B staleness is **silent**: nothing breaks, integrity checks pass, chains align. The artifact continues to function but produces less-than-optimal outputs, makes decisions on outdated assumptions, or fails to leverage new capabilities. Type B staleness is what causes the "silent degradation" described in the original problem statement.

**Type B is harder than Type A.** Type A is detected deterministically. Type B requires the system to notice *what is possible now that wasn't before* — which is a reasoning task (R1) informed by external knowledge (R7) against structured current state (R5, B-5).

#### Staleness avoidance must be automatic and event-driven

Periodic scheduled reviews catch staleness eventually, but "eventually" is the failure mode. **Any external signal of potential obsolescence MUST auto-trigger the self-update process** — no human has to remember to run it. Triggers include, at minimum:

| Signal | Source | What it indicates |
|---|---|---|
| New model release | Provider release notes, model catalogs | Potential capability shift — re-evaluate model-specific guidance, cost/capability tradeoffs, caching strategies |
| VS Code / Copilot release | Monthly release notes, changelog feeds | New YAML fields, new features, deprecations, GA transitions |
| Agent framework / SDK update | Provider docs, SDK release notes | New patterns, deprecated patterns |
| Ecosystem artifact published | Trusted sources (Microsoft, OpenAI, Anthropic, Google, Meta, recognized specialists) | Potential adoption/incorporation candidate |
| Platform best-practice update | Official prompting guides from providers | Potential pattern update |
| Internal artifact edit | File system watcher / git hook | Scope the dependent check |

Trigger coverage is the R7 capability. Each trigger MUST be wired to a specific pipeline scope (which tiers and artifact types to examine). Missing triggers are the perpetual gap identified as risk in the vision — they should be tracked as a first-class concern.

#### Small-step, risk-reduced integration unlocks autonomy

When a large external change arrives (a new major model, a new platform version), the naive reaction is "review everything at once" — which is high-risk, expensive, and almost always routes to human approval. The alternative is **decomposition into small, individually-verifiable steps**. Each step changes one thing, can be validated in isolation, and — because its blast radius and uncertainty are both small — can often proceed autonomously.

**Worked example: a new major model is released.**

| Step | What changes | Risk level | Autonomy |
|---|---|---|---|
| 1 | Add model-specific guidance to `03.02-model-specific-optimization.md` (new model as additional entry; existing models unchanged) | Low — additive, non-breaking, no existing consumer uses the new model yet | Autonomous |
| 2 | Update trusted-sources / capability registry to include the new model | Low — metadata only | Autonomous |
| 3 | Add the new model as an optional target for one specific low-risk agent (e.g., a validator that benefits from the new model's better reasoning); keep the default model unchanged | Low — opt-in, A/B path preserved | Autonomous with notification |
| 4 | Observe outcomes (R14) on Step 3 over N invocations; collect regression/quality signal | Low — observation only | Autonomous |
| 5 | If Step 4 outcomes are positive, switch that agent's default to the new model | Low-medium — default change is reversible via B-2 rollback | Autonomous with notification |
| 6 | Propagate model selection to a broader set of agents (batched, one tier at a time) | Medium — many defaults changing | Autonomous with notification per tier |
| … | | | |
| n-1 | Use the new model to propose updates to other artifacts (e.g., re-review prompts with the new model's better judgment) | Medium — the model is now participating in the loop | Autonomous with notification |
| n | Use the new model to review the self-update infrastructure itself | Elevated (L-2) — meta-level change | Human approval, unless validation confidence is high enough via redundant processing |

Each step is individually non-breaking under N-1's classification, individually small under B-1's impact dimension, and individually reversible under B-2's rollback mechanism. The sequence is what delivers the full capability integration; the small-step decomposition is what keeps each step autonomous.

**This converts "large change → human approval marathon" into "large change → many small autonomous steps + a few human escalations at natural decision points."** It is the primary mechanism by which the vision achieves *maximum autonomy at acceptable risk*.

#### Risk-reduction techniques that unlock autonomy

For any individual step, these techniques reduce residual risk enough to justify autonomous execution:

| Technique | What it does | Example |
|---|---|---|
| **Static validation** | Deterministic checks before execution: schema validity, reference resolution, boundary preservation (B-5), chain alignment (N-2), token budgets, goal/scope preservation | Before adding a new model entry: YAML valid, no duplicate keys, token budget unchanged |
| **Runtime validation** | Actually execute the change in a sandbox or shadow path; compare outputs to pre-change baseline on a test set | Run the updated agent on N canonical test inputs; diff outputs against the pre-change agent |
| **Modular feature integration** | Add capability as opt-in/optional first; default unchanged until observation confirms value | Step 3 above — new model is available but not default |
| **Redundant processing and validation** | Two independent LLM passes (R2), or LLM + deterministic cross-check, or multi-model ensemble — agreement required before commit | Updater proposes change; independent validator reviews; only commit on agreement |
| **Shadow execution with outcome tracking** | Run new and old paths in parallel on real invocations for a window; compare outcomes; only promote on parity or improvement | Steps 3–4 above |
| **Bounded rollout with canary** | Apply change to one low-risk consumer first; expand only after success signal | Step 3 → Step 6 cascade |
| **Reversibility guarantee** | Every autonomous step has a pre-recorded snapshot (B-2); rollback is one deterministic operation | Group snapshot before each step |

**The rule.** Autonomy is earned per-step, not per-change-set. A step that passes enough risk-reduction techniques to drive assessed risk below the autonomy threshold proceeds without human intervention. A step that doesn't — perhaps because no runtime validation is available, or because the change touches Tier 0 — escalates. The self-update system's job on any given trigger is to **decompose the integration into as many autonomous steps as possible and escalate only the irreducible minimum**.

#### Design principle (add to vision)

Add a new top-level design principle (placed alongside or before "Human governance, autonomous execution"):

> **Staleness avoidance first — automatic, event-driven, small-step.** The self-update infrastructure MUST be maintained at higher freshness than the artifacts it manages, against both structural and capability staleness. Any external signal of potential obsolescence (platform release, model release, ecosystem publication) auto-triggers the self-update cycle. Large changes are decomposed into small, individually-verifiable steps; each step is subject to risk-reduction techniques (static validation, runtime validation, modular integration, redundant processing, shadow execution, reversibility) that allow autonomous execution when assessed risk drops below threshold. Failure of any risk-reduction check — or any staleness detected in the self-update infrastructure itself — fails closed to human review.
>
> **Why.** (1) Stale managed artifacts degrade specific capabilities; stale self-update infrastructure silently degrades every subsequent decision and its ability to detect its own degradation. (2) Capability/logic staleness (Type B) is silent — passes all integrity checks yet produces obsolete outputs — and must be actively hunted, not passively noticed. (3) Monolithic "review everything on release day" is unaffordable and routes to human approval by default; small-step decomposition is what makes autonomous keep-up actually possible.
>
> **How.**
>
> 1. **Self-update infrastructure has the shortest validation cadence.** Meta artifacts (Tier 0 governance, Tier 5 meta-ops, all meta-agents, all hooks, all skills used by the update loop) are validated on every self-update cycle — never cached, never skipped. R10 caching does NOT apply to meta components.
> 2. **Event-driven triggers.** Every defined external signal (model releases, platform releases, ecosystem publications, internal edits) wires to a scoped pipeline entry point. Missing or broken trigger wiring is itself a CRITICAL finding.
> 3. **Decompose-first.** When a trigger fires, the first action is to produce a decomposition plan: the smallest sequence of individually-verifiable steps that integrates the capability end-to-end. The decomposition plan is itself validated before execution begins.
> 4. **Earn autonomy per step.** Each step is classified (B-1) using the best-available risk-reduction techniques. Steps below the autonomy threshold execute; steps above it escalate — but only the individual step, not the whole integration.
> 5. **Infrastructure integrity pre-step is a hard gate.** Before any managed-artifact work begins, run the deterministic integrity check against the self-update infrastructure. Any failure aborts the cycle and escalates.
> 6. **No silent fallback.** Suspect meta-state never proceeds under degraded operation. "Missed a review opportunity" beats "applied a wrong change based on stale self-knowledge."
> 7. **Dangling references in self-update infrastructure are CRITICAL** (vs. HIGH in managed artifacts).
> 8. **Rename log is append-only and load-bearing.** Corruption or missing entries is staleness and fails closed.

This principle subsumes and strengthens L-2 (amplified risk for self-updates), B-3 (integrity pre-step), N-2 (chain alignment), and the original R11 autonomy gradient (small-step decomposition is the mechanism by which more work lands in the "autonomous" band). It reframes the vision around a single overriding concern: **the updater must keep up with the world, automatically, in small enough steps that keeping up is safe.**

### N-3: Tiered structure is an autoupdate enabler but not named as one (MEDIUM)

#### What the tier system already is

The PE context folder (`.copilot/context/00.00-prompt-engineering/`) is organized into a **6-tier layered hierarchy** documented in `STRUCTURE-README.md`:

```
Tier 5: Meta-Ops (05.01–05.05)       ← System self-improvement
Tier 4: Conventions (04.01–04.04)     ← Validation & production readiness
Tier 3: Specialized (03.01–03.07)     ← Platform-specific patterns
Tier 2: Multi-Agent (02.01–02.05)     ← Orchestration & workflows
Tier 1: Foundations (01.01–01.07)     ← Core principles
Tier 0: Governance (00.01–00.02)      ← North star
```

Two invariants hold by construction:

1. **Dependencies flow upward only.** A Tier 1 file may reference only Tier 0 or Tier 1. A Tier 3 file may reference Tiers 0–3 but never Tier 4 or 5. Rule **C5 (no circular dependencies)** in the priority matrix enforces this.
2. **Stability decreases with tier number.** Tier 0 is the "north star" — rarely changes, human-only. Tier 5 is the fastest-moving layer (logs, runtime artifacts). Tiers 1–4 fall in between.

The same layering pattern exists beyond context files, even if less formally: instruction files → agents → prompts → templates form a 4-layer dependency chain documented in `05.01-artifact-dependency-map.md`.

#### What makes this an autoupdate enabler

Tier is **already a deterministic predictor of blast radius**. It answers the question "if I change this file, how much of the system can I possibly affect?" without traversing the dependency graph:

| Tier of changed artifact | Maximum blast radius | Implication for autonomy |
|---|---|---|
| Tier 0 (governance) | Every artifact in every layer | Always escalate — changing "what better means" invalidates every downstream validation |
| Tier 1 (foundations) | Everything above Tier 1 — most of the system | Rule changes escalate; rationale/example edits (N-1) can be autonomous |
| Tier 2 (multi-agent patterns) | Orchestrators, multi-agent prompts, downstream specialized files that reference them | Usually autonomous with notification; rule-level changes need validation |
| Tier 3 (specialized patterns) | Files referencing the specific pattern — usually a bounded set | Mostly autonomous |
| Tier 4 (conventions, validation) | Validator agents, review prompts, CI-like checks | Autonomous for convention tuning; rule changes need validation |
| Tier 5 (meta-ops, logs) | Nothing depends on these by the upward-only rule | Maximally autonomous — safe to rewrite routinely |

**This is a free, deterministic input to risk assessment (B-1)** that the vision currently ignores. It's free because every context file already has a tier number in its filename prefix. It's deterministic because the tier-to-blast-radius mapping is a lookup, not a reasoning task.

#### Why plan1 missed it

Plan1 and the current vision frame blast radius as "count dependents" (R9). Dependent count is useful but less informative than tier for two reasons:

- **Dependent count is local**; tier is systemic. A Tier 1 file with 3 dependents has far higher blast radius than a Tier 4 file with 3 dependents, because the Tier 1 dependents themselves have dependents — the transitive set is much larger.
- **Dependent count requires a full dependency-graph query**; tier is a O(1) filename lookup. For a first-pass classification, tier is cheaper *and* more accurate.

Tier and dependent count are **complementary**, not redundant. Tier gives "how far up the stack is this?" Dependent count gives "how connected is this specific file within its tier?" Both feed risk assessment.

#### Fix option — add rationale R-S6 and wire tier into the pipeline

Add a rationale **R-S6 — Tier-based blast radius**:

> **R-S6 — Tier-based blast radius.** Tier encodes how many layers of the system can depend on an artifact. A change at Tier N may affect Tiers N+1 through 5 but never Tiers 0 through N-1, by the upward-only dependency rule (C5). The self-update system uses tier as a primary, O(1) input to risk classification — alongside dependent count (R9), change type (B-1), and meta-membership (N-2b).

Wire tier into each layer of the vision:

| Vision layer | How tier is used |
|---|---|
| **Detect** | Trigger source → affected tiers. A model release triggers Tiers 1 (foundations — context engineering principles, model-specific optimization), 3 (specialized — model-specific patterns), 4 (validation — if validators call out models), and 5 (meta-ops — if self-update infrastructure uses the model). A template edit triggers only the downstream tiers that consume that template. |
| **Assess** | Risk classification uses tier as the first sort key. Tier 0 → always escalate. Tier 5 → presumptively autonomous. Tiers 1–4 → route by B-1 (breaking vs. non-breaking) and N-2 (chain coverage). |
| **Propose** | Decomposition plans (N-2b small-step integration) order steps bottom-up: first changes land in high-numbered tiers (lowest blast radius), then propagate downward only after validation. The worked example in N-2b already does this implicitly — Step 1 touches Tier 3, Step n touches Tier 5/meta. |
| **Execute** | Group snapshots (B-2) are tier-scoped. A Tier 3 change snapshots the Tier 3 folder + affected Tier 4 dependents, not the entire context tree. Rollback unit = tier slice. |

#### Tier extended to other artifact sets

The context folder's tier system is the clearest example, but the pattern generalizes:

| Artifact set | "Tier" equivalent | Stability ranking |
|---|---|---|
| Instruction files | `pe-copilot-instructions-file.instructions.md` → `pe-common.instructions.md` → per-type instructions | Root = highest, per-type = lowest |
| Agents | Meta-agents (`meta-*`) → type-specific researchers/builders/validators | Meta = highest impact, per-type = lowest |
| Prompts | Meta-prompts (`meta-prompt-engineering-*`) → creation/design/review prompts | Same pattern |
| Templates | `promptengineering-*-structure.template.md` (structural) → `guidance-*.template.md` (content) → `output-*.template.md` (ephemeral) | Structural = most stable |

The vision should declare that **every artifact set has a stability gradient**, and where the gradient is explicit (numeric prefixes, `meta-*` naming), the self-update system MUST use it as a first-pass blast-radius input. Where it's implicit, the set's `STRUCTURE-README` or equivalent index makes it explicit.

#### Strictness of chain-alignment and validation scales with tier

A secondary benefit: N-2's chain-alignment check and B-3's integrity pre-step should **scale their strictness with tier**.

- **Tier 0/1 rules** — MUST have complete chain coverage (every rule is enforced by at least one validator agent; every agent references the rulesets it enforces). A missing chain is CRITICAL.
- **Tier 2/3 patterns** — SHOULD have chain coverage. Missing chains are HIGH.
- **Tier 4 conventions** — chain coverage is recommended but tier-specific (some are author-facing docs, not runtime enforced).
- **Tier 5 meta-ops** — different rules apply (meta-infrastructure staleness is CRITICAL per N-2b regardless of tier-number ordering).

This gives validators a **concrete, file-based severity policy** that is cheaper to maintain than per-rule severity tags.

#### Summary of what adding R-S6 buys

1. O(1) blast-radius classification for any change — no graph traversal needed for the first-pass decision.
2. Deterministic input to B-1 (risk assessment) complementing N-2b's small-step decomposition.
3. Principled ordering for N-2b decomposition plans (bottom-up integration).
4. Tier-scoped group snapshots for B-2 rollback — smaller, cheaper snapshots than whole-folder.
5. Tier-scaled strictness for N-2 chain alignment and B-3 integrity checks.
6. Free consistency check — any file whose reference list violates upward-only flow (C5) is immediately detectable from its tier + its outbound references.

**Connects to.** R9 (dependency graph — tier is a coarser, cheaper version that is always available), R6 (decomposition — tier defines the natural decomposition boundary), B-1 (risk inputs — tier is a primary impact dimension), B-2 (rollback — tier defines snapshot scope), N-2 (chain strictness scales with tier), N-2b (decomposition plans order steps by tier), C5 (upward-only dependencies — tier enforces this structurally), the user's explicit request for "tiered information structuring for autoupdate and validation."

### N-4: Per-artifact-type readiness is not sequenced (MEDIUM)

The vision treats autonomous execution as a single switch ("the system executes low-risk changes autonomously"). The readiness matrix in §[Per-artifact-type autonomous-update readiness](#-per-artifact-type-autonomous-update-readiness) shows readiness is **heterogeneous** — some artifact types can go autonomous today, others need new infrastructure first. The vision should sequence the rollout instead of pretending it's binary.

#### The underlying question: what makes a change-type safe for autonomy?

A change is safely autonomous when the system can answer three questions deterministically:

1. **What kind of content just changed?** (Rule text vs. rationale vs. example vs. structural field vs. narrative prose.) Without this, every diff routes to LLM judgment. This is what N-1 (rule/rationale/example separation) enables, and it is only needed for artifacts that actually **encode rules with rationales and examples**.
2. **What is this artifact's purpose and boundary?** (Goal, scope, consumers, non-goals.) Without this, "did the change preserve intent?" cannot be mechanically checked. This is what B-5 (semantic metadata) enables, and it is most valuable for artifacts whose **purpose is not obvious from structure alone**.
3. **Can we reconstruct what happened and undo it?** (Audit trail, versioning, rollback.) Without this, regressions can't be isolated or reverted. This is what C-3 (audit trail) + B-2 (versioning) enable, and it is needed for **every** artifact type — there's no autonomy without reversibility.

Different artifact types need different subsets of these prerequisites because they encode different kinds of content. A template is a structural scaffold with placeholders — it has no rules to classify (N-1 N/A) and its purpose is obvious from the filename (B-5 optional), but it still needs audit trail (C-3). An instruction file is almost entirely rules with rationales — it needs all three.

#### Cumulative preconditions (important clarification)

**Preconditions are cumulative across phases.** Phase N requires its own listed preconditions PLUS everything that was required to enable any earlier phase. The table shows only the *newly-added* prerequisite per phase so the progression is readable. Phase 1 has no prerequisites beyond what exists today. Phase 2 adds C-3. Phase 3 adds B-5 + N-1 (and still benefits from C-3 being in place). Phase 4 adds N-2 chain validation on top of everything else.

This also means: infrastructure built for a later phase can "back-fill" earlier phases. For example, once B-5 lands for Phase 3, templates (Phase 1) benefit from richer metadata too — it just wasn't on the critical path to put them in Phase 1.

#### Fix option — phased rollout with explicit scope and rationale

Add a **rollout sequence** to the vision (new subsection under "The vision" or new principle):

| Phase | Autonomous execution enabled for | Newly-added precondition | Why this scope matches this prereq set |
|---|---|---|---|
| **1** | Templates (structural edits), prompt snippets, hooks JSON (field-level edits) | None beyond today | These artifacts have **no rules with rationales** (N-1 N/A), **obvious purpose from filename/schema** (B-5 optional), and **existing validators** (`template-validator`, `prompt-snippet-validator`, `hook-validator`). Change = a schema diff the existing validators already understand. |
| **2** | Skills content (resource files, examples), agent tool lists, agent handoff lists | Audit trail (C-3) in place | This is **structured metadata, not rule-bearing prose**. Skills are procedural workflows with resource files — they don't encode classifiable rules, so N-1 does not apply. Agent tool/handoff lists are YAML arrays — the diff is trivially deterministic (added X, removed Y), so N-1 does not apply. What's missing today is reversibility: without C-3's audit trail, a bad tool addition or handoff rewrite cannot be isolated to its change record. Once C-3 lands, these changes become safely autonomous. **B-5 is not required here** because the artifact's purpose is already declared in its top-level frontmatter (`description:` for skills, agent headers for agents) — goal/scope/boundaries would add polish but aren't load-bearing for tool/handoff diffs. |
| **3** | Agent bodies (narrative instructions), context files, instruction files | Semantic metadata (B-5) + rule separation (N-1) in place | These artifacts **encode rules with rationales and examples in prose**. Without N-1, a diff cannot be classified as breaking (rule change) vs. non-breaking (rationale edit) — every diff escalates to LLM judgment. Without B-5, "did the change preserve the artifact's declared scope and boundaries?" is a prose-comprehension task, not a structured check. Both infrastructure pieces are load-bearing. |
| **4** | Cross-tier refactors, rule renames, file moves affecting multiple artifacts | Chain alignment (N-2) verified + rename log operational | These changes **ripple across the artifact graph**. Without the two-level reference convention (N-2) and the rename log, a rule rename in a Tier 1 file silently breaks every downstream `enforces_rules: [Cn]` reference. Phase 4 unlocks autonomous refactoring only after the linkage itself is resilient. |

#### Why the phases are ordered this way

The ordering is not arbitrary — it follows **what infrastructure is cheapest to build and what value it unlocks first**:

- **Phase 1 is free today.** The validators exist; the schemas are closed. No new infrastructure to build. The system can start earning autonomy immediately on low-stakes changes — which also provides the operational experience and outcome data needed to tune the risk thresholds for later phases.
- **Phase 2 needs only an audit log.** C-3 is the smallest new piece of infrastructure (one structured append-only file per the N-6 schema). It unlocks a large, useful class of changes — tool additions, handoff rewrites, skill resource updates — that happen frequently when the ecosystem evolves.
- **Phase 3 is the biggest investment.** B-5 and N-1 require rewriting significant portions of existing context and instruction files. That's high-value but high-cost. Deferring it to Phase 3 means the benefits of Phases 1 and 2 are realized while Phase 3 infrastructure is being built — the system isn't waiting for the hardest piece before delivering any value.
- **Phase 4 is last because it depends on Phase 3.** Without N-1 (Phase 3 prereq), rule renames can't be classified. Without B-5 (Phase 3 prereq), a cross-tier refactor can't be verified against declared scope. N-2 chain alignment is the integration test that proves Phases 1–3 are working correctly.

#### What this reframes

This reframes the vision from aspirational ("autonomous execution") to implementable ("autonomous execution for these classes of changes, today; expanding to these classes as each prerequisite lands"). It also **forces honesty** about the current state: Phase 1 is achievable now; everything else is conditional on named work. The vision should declare the current phase and the outstanding preconditions for the next phase as first-class status information, not background detail.

**Connects to.** B-5 (Phase 3 prereq), N-1 (Phase 3 prereq), C-3 (Phase 2 prereq), N-2 (Phase 4 prereq), N-3 (tier also interacts with readiness — a Tier 5 edit in a Phase 3 artifact may still be autonomous via tier-based blast-radius override), the Per-artifact-type readiness matrix earlier in this plan.

### N-6: Outcome log schema is unspecified (LOW)

#### What the outcome log is

The **outcome log** is the self-update system's append-only record of every change it has considered, executed, or rejected. Each entry describes **one change event** — one artifact edit proposed by the loop — with enough structured information that:

- A human reviewer can reconstruct *what happened, why, and with what result* without re-running the pipeline.
- The loop itself can **learn from its own history** (R14 — progressive learning) by querying past entries: which trigger types produce regressions, which artifact tiers have the highest rollback rate, which validation checks are predictive of success.
- **Rollback (B-2)** can isolate the exact change to undo: the log entry points to the pre-change hash and version, which point to the snapshot.
- **Audit (C-3)** has a machine-queryable source of truth: "show me every autonomous edit to Tier 1 files in the last 30 days that had confidence < 0.9."

It is the self-update system's **equivalent of a git log**, but enriched with the metadata the loop needs to reason about its own behavior: classification, confidence, autonomy level, trigger, validation results, outcome.

Without a schema, every consumer invents its own fields, and the log silently drifts into the next staleness source (which is ironic given its purpose is to prevent drift).

#### Where it is written

**Location.** One append-only file per repository, under the self-update infrastructure folder:

```
.copilot/self-update/outcome-log.jsonl
```

Rationale for location and format:

- **`.copilot/self-update/`** — Separate from `.copilot/context/` (rules) and `.github/` (consumers). The log is **infrastructure artifact**, not rules and not consumers. It belongs in a third space dedicated to self-update operational state (alongside the rename log from N-2 and future state files).
- **JSON Lines (`.jsonl`)** rather than a single YAML array — **append-only without parsing the whole file**. Each entry is one line, one JSON object. Writers append; readers stream. The log can grow large without becoming expensive to write to.
- **Version-controlled in git** — the log IS the audit trail. Losing it is losing the audit trail. Git is the natural fit (append-only, signed commits, immutable history once pushed).
- **Per-repository**, not per-artifact or global — the log is scoped to one self-update system's decisions. A monorepo has one log; separate repositories have separate logs.

**Related operational files in the same folder** (defined by other findings, grouped here so the infrastructure is coherent):

| File | Defined by | Purpose |
|---|---|---|
| `outcome-log.jsonl` | N-6 | One line per change event (this file) |
| `rename-log.jsonl` | N-2 | One line per rule/file rename with `old → new` mapping |
| `snapshots/<change_id>/` | B-2 | Pre-change content snapshots referenced by log entries |
| `triggers/` | N-2b | Trigger definitions and last-fired timestamps |

Each file has a single writer (the self-update loop) and many readers (validators, review prompts, human reviewers).

#### How it is used

**Write path** — every change event, regardless of autonomy level, produces exactly one log entry:

1. Trigger fires (N-2b) → loop enters decomposition → produces N candidate change events.
2. For each event, before execution: loop writes a **draft entry** with `trigger`, `artifact`, `classification`, `confidence`, `autonomy_level`, `validations_passed` (list of checks planned to run).
3. If the event executes (autonomous or after approval): loop runs the validations, takes the snapshot (B-2), applies the change, updates the entry with `pre_change_hash` / `post_change_hash` / `pre_change_version` / `post_change_version` and `outcome: null` (pending observation).
4. If the event is rejected (validation failure or human rejection): loop writes `outcome: rolled_back` or leaves `outcome: null` with `rejection_reason` set.
5. After an observation window (shadow execution from N-2b, or post-deployment monitoring), loop updates the entry's `outcome` to `success` or `regression`.

**Entries are append-only updates.** Corrections happen by appending a new entry that references the original `change_id` (e.g., a rollback is its own entry with `outcome: rolled_back` and a link to the reverted `change_id`).

**Read paths** — the log serves five distinct consumers:

| Consumer | Query pattern | What it does with the answer |
|---|---|---|
| **Human reviewer (audit)** | "Show me all autonomous Tier 1 edits in the last 7 days" | Spot-check autonomy decisions; catch creeping false confidence (L-1) |
| **Rollback (B-2)** | "Get the pre-change snapshot for `change_id=X`" | Deterministic one-operation restore |
| **Progressive learning (R14)** | "Regression rate by `classification` × `confidence` bucket × `artifact_tier`" | Tune autonomy thresholds; re-calibrate confidence scoring |
| **Self-update loop (next cycle)** | "Has this artifact been edited in the last N cycles? What was the outcome?" | Avoid thrashing the same file; back off if recent edits regressed |
| **Trigger analysis (N-2b)** | "Which trigger types produce the most `regression` outcomes?" | Identify unreliable triggers; strengthen validation for high-risk trigger classes |

Without the structured schema, each of these consumers would need to parse free-form prose — which is exactly the kind of ad-hoc judgment the self-update system is trying to eliminate from its own operation.

#### Minimal required schema

Every entry MUST include at least these fields:

```yaml
# One JSON object per line in outcome-log.jsonl; YAML shown here for readability.
change_id: <uuid>                     # Stable ID for cross-references (rollback, follow-up entries)
timestamp: <iso8601>                  # When the entry was first written (draft time)
artifact: <path>                      # Workspace-relative path of the edited artifact
artifact_tier: <0–5 | "unversioned">  # From filename prefix (N-3) or "unversioned" for non-tiered artifacts
pre_change_hash: <sha256>             # Content hash before the edit; points to snapshot
post_change_hash: <sha256>            # Content hash after the edit
pre_change_version: <semver>          # From artifact's own version field
post_change_version: <semver>         # Bumped per B-2 rules based on classification
classification: <breaking | non-breaking-rule | non-breaking-rationale | non-breaking-example | structural>
                                      # From N-1 block-type diff; drives autonomy gate
confidence: <0.0–1.0>                 # System's self-assessed confidence in the classification
autonomy_level: <autonomous | notify | approved | escalated>
                                      # Which autonomy band was used (R11)
trigger: <platform | model | ecosystem | internal | scheduled | manual>
                                      # What initiated the change (N-2b trigger registry)
trigger_source: <url | path | event_id | null>
                                      # Specific source that fired (e.g., release-notes URL, file path)
validations_passed: [<check_id>, ...] # Names of static/runtime checks that ran and passed
validations_failed: [<check_id>, ...] # Names of checks that ran and failed (empty if all passed)
rejection_reason: <string | null>     # Present when the change was blocked or reverted
outcome: <success | regression | rolled_back | null>
                                      # null until observation window closes; then resolved
outcome_signal: <string | null>       # Optional pointer to the measurement that set outcome (test IDs, review refs)
rolls_back: <change_id | null>        # Set when this entry is a rollback of another entry
```

**Field rationale (only where not obvious):**

- **`classification` values** align with N-1 block types — so the log directly reflects the rule/rationale/example distinction. `structural` covers schema-field edits (templates, hooks JSON, YAML arrays from N-4 Phase 1–2).
- **`confidence`** is the system's own assessment, not a human's. Progressive learning (R14) calibrates the threshold at which `confidence` is high enough to justify `autonomy_level: autonomous`.
- **`trigger_source`** is the specific artifact that caused the trigger to fire, so the loop can query "which release caused this regression?" without guessing.
- **`rolls_back`** makes rollback events first-class queryable entries, not special-cased prose.

**Validation rules for the log itself:**

1. **Append-only.** Existing entries MUST NOT be edited — corrections are new entries referencing prior `change_id`.
2. **Schema-validated on write.** Each entry is validated against the schema before being appended; a malformed entry is a CRITICAL finding (N-2b: self-update infrastructure staleness).
3. **No PII / no secrets.** The log is committed to the repository; it MUST NOT contain credentials, user emails, or other sensitive data. Trigger sources that include URLs MUST be sanitized.
4. **Size management.** The log can grow unbounded; after N months, entries older than a threshold may be compacted into a summary entry (with the raw entries archived). Compaction is itself an audited operation.

#### Connection to other findings

| Finding | What N-6 provides to it |
|---|---|
| **R14** (progressive learning) | The data source — without structured entries, "learn from outcomes" is aspirational |
| **C-3** (audit trail) | The concrete implementation — C-3 says "keep an audit trail"; N-6 says "here is the file, format, and fields" |
| **B-2** (rollback) | Snapshot pointer + change identity — log entries reference the snapshot folder by `change_id` |
| **N-1** (rule/rationale/example) | Provides the `classification` vocabulary used in every log entry |
| **N-2** (chain alignment) | Rename log (same folder, same JSONL pattern) is a companion file using the same discipline |
| **N-2b** (staleness avoidance) | Every auto-triggered cycle writes entries; missing entries for a fired trigger is itself a CRITICAL staleness signal |
| **N-3** (tier) | `artifact_tier` field; enables tier-sliced queries for rollback and learning |
| **N-4** (phased rollout) | `autonomy_level` field shows which phase the change came from; Phase 2 autonomy is a hard precondition for C-3 = this log existing |

**Priority.** LOW because the system can start operating (Phases 1–2 of N-4) with a simpler ad-hoc log and formalize the schema later. But the schema SHOULD be adopted early; retrofitting a structured schema onto months of free-form entries is much more expensive than adopting it up front.
