## 🎯 Purpose of this document

The [original update plan](01.001-vision.update-plan.md) identifies 45 gaps (G1–G45) across 14 analysis sections, totaling ~2,000 lines of analysis for a ~500-line vision document.

This document answers three questions:

1. **Which changes genuinely make the vision more reliable, effective, and efficient?**
2. **Which changes are redundant, premature, or over-engineered?**
3. **Do the recommended changes themselves maintain robustness, manageability, and cost control — or do they risk destabilizing the vision?**

The result is a focused action plan: fewer changes, each stress-tested against the same properties the vision aims to preserve.

## 📊 Assessment of the original plan

### What the plan gets right

The **core insight is solid**: the vision's breaking/non-breaking classification uses a single dimension (goal/scope/boundary preservation). A change can be structurally non-breaking while introducing contradictions (consistency) or exceeding token budgets (efficiency). This creates both over-blocking (safe changes escalated) and under-blocking (risky changes approved). The staged integration concept — validate safety first, optimize later — is a sound pattern.

The **loop-level stability concern is valid**: a feedback system without convergence criteria, oscillation detection, or iteration bounds can churn indefinitely. Per-change safeguards aren't enough.

### What the plan gets wrong

**It's 4× longer than the document it's improving.** The analysis iterates on its own findings (G7 is superseded by G40, G8 by G43, G6 by G39), expanding simple concepts into multi-page analyses with framing sections, matrices, and sub-gap findings. This makes it very difficult to extract the actionable changes.

**It optimizes for automation that doesn't run yet.** The PE system exists and is operational — the Learning Hub already has context files, instruction files, agents, prompts, skills, templates, and hooks. What's at Phase 1 is the *self-update automation* (manual invocation, templates and hooks only). The vision is being defined precisely so that automation has a stable, convergent reference point. But many proposed changes — partitioned audit trail storage with three-tier lifecycle (G21), seven-dimension classification model (G29), cost-ordered execution tiers within the non-regression gate (G36) — solve scaling problems for Phase 3–4 automation. They add design complexity to the vision now without delivering value until the automation is far more mature.

**It expands the dimension model beyond what's useful — but gets one thing right.** The expansion to seven dimensions is overkill: referential integrity is a structural sub-check and dependency propagation is a routing modifier. But behavioral compatibility is genuinely independent. A change can pass structural, consistency, and efficiency checks while breaking consumers (YAML field rename, section restructuring that breaks deterministic parsers). The original plan's analysis proves this orthogonality. The correct model is **four + one**: structural, consistency, efficiency, behavioral compatibility, + effectiveness-deferred. Not three + one, not seven.

**It proposes metadata fields faster than they can be populated — but the metadata categories aren't equal.** The plan adds `rationales:` (G17), `implements:` (G38), plus a per-dimension severity scale — for artifacts that don't have `goal:` and `scope:` yet. But three distinct metadata categories are conflated:

| Category | Fields | Purpose | When needed |
|---|---|---|---|
| **Anchoring** | `goal:`, `scope:`, `boundaries:` | Declare *what the artifact is for* — the fixed reference point for convergence | **Phase 1** — already defined in the vision's metadata-driven principle; populating them is the investment that enables convergence measurement |
| **Constraint** | `rationales:` | Declare *why specific design decisions were made* — prevents learned lessons from being silently un-learned | **Phase 2** — lightweight format (string list, not structured objects); populated incrementally as changes are reviewed |
| **Linking** | `implements:` | Declare *which vision rationales an artifact operationalizes* — enables targeted impact analysis | **Phase 3–4** — O(k) vs. O(N) analysis benefit matters at scale, not with a few dozen artifacts |

Anchoring metadata (`goal:`, `scope:`) isn't "additional overhead" — it's the prerequisite for convergence measurement. Without it, the system can't answer "is this artifact drifting from its purpose?" The vision already requires it; the gap is that nothing enforces population.

Constraint metadata (`rationales:`) is the stability mechanism the previous analysis undervalued. A change can pass goal/scope alignment while removing a learned lesson — one cycle adds grammar checks, the next removes them, and neither violates the artifact's goal. Rationales make design decisions explicit so they can't be silently dropped. In lightweight format (a list of one-line strings, not structured objects with `constraint:` + `impact_if_violated:`), the population burden is low: add a rationale when a change is reviewed, not all at once up front.

Linking metadata (`implements:`) is genuinely deferrable. Vision rationale changes are rare and human-governed; an O(N) LLM scan of ~50 artifacts is acceptable at Phase 1–2.

## 🔬 Robustness review

The initial distillation (above) filtered 45 gaps to 11 changes. This section applies an additional filter: **does each proposed change maintain the vision's robustness, manageability, and cost control?**

An improvement that's conceptually correct can still harm the vision if it:
- **Introduces a mechanism that requires infrastructure that doesn't exist** — you're designing what you can't test
- **Adds a binary concept to a gradient-based system** — forcing exception logic almost immediately
- **Solves Phase 2–3 problems with Phase 1 design** — building on assumptions you haven't validated
- **Requires maintenance effort disproportionate to benefit** — each field, mechanism, or process is a future maintenance obligation

Changes that survive this filter share three traits:
1. They cost nothing to maintain — a note, a table, a rule, a number
2. They don't require new infrastructure — no tracking systems, no log fields, no multi-stage processes
3. They improve classification or bounding — making existing mechanisms more precise, not adding new ones

### Change-by-change verdict

| Change | Original # | Verdict | Reason |
|---|---|---|---|
| Multi-dimensional impact note | 1 | ✅ Keep (as awareness note) | Fixes real blind spot; now covers 4+1 dimensions (structural, consistency, efficiency, behavioral compatibility, + effectiveness-deferred); zero-cost if descriptive, not prescriptive |
| Non-regression gate | 2 | ⚠️ Reframe as checks | "Gate" introduces a binary concept into a gradient system; add consistency, efficiency, and behavioral compatibility (schema-level) to the existing validation row instead |
| Staged integration | 3 | ⚠️ Reframe as risk-ordering principle | The Stage 1/Stage 2 *mechanism* (tracking, scheduling, coherence verification) is over-engineered for Phase 1. But the *principle* — risk-ordered execution within decomposed plans — is the mechanism that makes autonomous self-update operational. Without it, one escalated step blocks the entire sequence. Apply as a principle, not a pipeline. |
| Convergent iteration | 4 | ⚠️ Split | Keep design intent ("the loop must converge"); defer the measurement mechanism (change-volume slopes require multi-cycle history that doesn't exist) |
| Oscillation detection | 5 | ⚠️ Keep rule, defer mechanism | Important safety wisdom; but at Phase 1 (manual invocation) there are no "consecutive cycles" to compare — the tracking infrastructure doesn't exist |
| Iteration budget | 6 | ✅ Keep as-is | Simple, effective, zero maintenance cost — works even at Phase 1 |
| Severity classification | 7 | ✅ Keep as-is | Operationalizes an existing but undefined concept; it's a table, not a mechanism |
| Extract strategy | 8 | ❌ Reduce to one sentence | The situation doesn't arise at Phase 1; one sentence in Trust-based adoption captures the wisdom without adding a mechanism |

**Result:** 11 changes narrow to **7 lean additions** totaling ~640 words (~14% vision growth) instead of ~1,200 words (~25%).

## ✅ Changes that genuinely improve the vision

These changes address real gaps, are proportional to the problem, and make the vision more reliable, effective, or efficient without adding unnecessary complexity. They're organized by theme and ordered by priority within each theme.

### Theme 1: Multi-dimensional impact classification

**The problem it solves:** the vision classifies changes on a single axis (structural). This misses consistency and efficiency impacts, causing misrouted autonomy decisions.

| # | Source | Change | Why it's valuable |
|---|---|---|---|
| 1 | G1, G29 | Add an awareness note after the breaking/non-breaking table: impact is multi-dimensional (structural, consistency, efficiency, behavioral compatibility, + effectiveness-deferred) and the highest-risk dimension determines routing | Corrects the classification blind spot; behavioral compatibility catches format-breaking changes that other dimensions miss |
| 2 | G2, G29 | Add consistency, efficiency, and behavioral compatibility (schema-level) as named check types within the existing "static validation" row of the risk-reduction table | Extends an existing mechanism; schema-level behavioral checks are deterministic and near-zero cost |
| 3 | G33–G34 | Add a one-sentence refinement to the consistency check: "priority shifts that serve the artifact's declared goal are non-breaking; shifts that oppose the goal or violate boundaries are breaking" | Prevents the consistency check from over-blocking goal-aligned changes |

**Scope note:** keep the dimension model at **four + one** (structural, consistency, efficiency, behavioral compatibility, + effectiveness-deferred). Of the original plan's three additional dimensions, behavioral compatibility is genuinely independent — a change can pass structural, consistency, and efficiency checks while breaking consumers (YAML field rename, section restructuring that defeats deterministic parsers). Referential integrity remains a structural sub-check; dependency propagation remains a routing modifier. Don't expand to seven.

**Behavioral compatibility has two cost layers:** Schema-level checks (YAML structure, section headings, metadata field names) are deterministic and near-zero cost — same as referential checks. Consumer-behavioral checks (does restructured content change LLM behavior?) require extra LLM passes and belong in Phase 2–3. Only the schema-level layer is applied now.

**Robustness note:** change 1 must stay *descriptive* ("when assessing impact, also consider consistency, efficiency, and behavioral compatibility"), not *prescriptive* ("every assessment MUST evaluate all five dimensions independently"). The latter inflates every assessment's cost by 4–5×. Change 2 deliberately avoids introducing a "gate" — a binary pass/fail concept conflicts with the vision's autonomy gradient, which routes findings to appropriate autonomy levels rather than blocking them.

### Theme 2: Loop-level stability

**The problem it solves:** the vision defines per-change safeguards but doesn't govern the update loop as a whole. Without iteration bounds, the loop can churn or amplify errors.

| # | Source | Change | Why it's valuable |
|---|---|---|---|
| 4 | G9 | Add an iteration budget: max autonomous changes per cycle (configurable, default: 10), excess queued for the next cycle | Bounds cost and blast radius per cycle — works even at Phase 1 |
| 5 | G7 + G40, G8 + G43 | Add a "loop stability" design principle stating: (a) the loop's goal is convergence toward a fixed point (no proposed changes, absent new triggers), and (b) oscillation between states (same artifact changed and reverted across cycles) is a signal to freeze and escalate | Captures the essential safety wisdom without requiring tracking infrastructure |
| 6 | G3 | Add a "risk-ordered execution" principle to decomposition-first: when decomposition produces steps with different risk profiles, execute in risk order — non-regressive steps first (autonomous), potentially-regressive steps next (escalated as needed), optimization steps last (separate cycle). A step's execution is never blocked by a higher-risk step it doesn't depend on. | Makes the autonomy gradient operational for compound triggers (model releases, platform updates). Without it, one escalated step blocks the entire decomposed sequence. Zero infrastructure cost — it's an ordering rule, not a tracking system. |

**Robustness note:** the original plan proposed measuring change-volume slopes across cycles and building oscillation detection infrastructure. Both require multi-cycle history that doesn't exist at Phase 1 (manual invocation). The *rules* are valuable now; the *mechanisms* belong in Phase 2–3 when the loop runs automatically and has history to analyze.

**Convergence strategy (layered by phase):**

| Phase | Convergence mechanism | What it catches |
|---|---|---|
| **Phase 1** | Activity bounding (iteration budget, oscillation rule) + goal/scope anchoring (metadata-driven principle) | Gross instability, purpose drift |
| **Phase 2** | + Rationale-based constraint preservation (lightweight `rationales:` metadata) | Subtle drift — learned lessons silently un-learned, design decisions dropped without justification |
| **Phase 3+** | + Structured rationales (id + constraint + impact) + trajectory analysis (multi-cycle intent-preservation scoring) | Quantitative convergence measurement, automated severity classification |

Phase 1 catches runaway loops and purpose changes. Phase 2 catches the harder problem: changes that respect goal/scope but erode quality by dropping undeclared design decisions. Phase 3+ adds quantitative measurement to what Phase 2 makes qualitatively checkable.

### Theme 2b: Risk-ordered execution

**The problem it solves:** the vision's decomposition-first principle says *decompose* and the per-step autonomy says *classify each step* — but nothing says "execute the safe steps immediately while the risky ones await review." Without risk ordering, one escalated step blocks the entire decomposed sequence, eliminating the throughput benefit of decomposition.

| # | Source | Change | Why it's valuable |
|---|---|---|---|
| 6 | G3 | Add a risk-ordered execution principle: when decomposition produces steps with different risk profiles, execute in risk order — non-regressive steps first (autonomous), potentially-regressive steps next (escalated as needed), optimization steps last (separate cycle). A step's execution is never blocked by a higher-risk step it doesn't depend on. | Makes the autonomy gradient operational for compound triggers. Without it, a model release that decomposes into "add guidance" + "update agents" + "deprecate old models" waits entirely for the human to approve the deprecation, even though the additive step is autonomous. |

**Why this isn't the original Stage 1/Stage 2:** The original plan proposed named stages with tracking, scheduling, and coherence verification between stages. That's infrastructure. This is an *ordering rule* — it says "sort by risk, don't serialize unnecessarily." It works with the existing decomposition-first + per-step autonomy mechanisms. No new tracking, no scheduling, no coherence checks between stages.

**Example (LLM model release):**

| Step | Risk profile | Execution order |
|---|---|---|
| Add model guidance to context files | Additive, non-regressive → autonomous | Immediate |
| Update agents to use new model where effective | Behavioral change → autonomous + notification or human approval | After step 1 (depends on new guidance) |
| Deprecate superseded models | Removal, breaking → human approval | Independent of steps 1–2 timeline |

Without risk ordering, all three steps queue behind the human approval for step 3. With it, step 1 executes immediately, step 2 follows when step 1 lands, and step 3 awaits review — the system stays current faster while the human only reviews the genuinely risky part.

### Theme 3: Severity classification

**The problem it solves:** the vision mentions "severity thresholds" for over-correction control but never defines them. Without a scale, every finding competes equally for attention.

| # | Source | Change | Why it's valuable |
|---|---|---|---|
| 7 | G25 | Add a finding severity table to the Assess layer: CRITICAL (blocks operation), HIGH (breaks contract), MEDIUM (affects scope/quality dimension), LOW (cosmetic/optimization) | Operationalizes severity; connects to metadata — findings that affect `goal:` are HIGH, findings affecting neither goal nor scope are LOW |

### Summary: 7 changes, ~640 words

| # | Theme | Target section | Size |
|---|---|---|---|
| 1 | Multi-dimensional impact awareness note (4+1 dimensions) | The goal → after breaking/non-breaking table | ~120 words |
| 2 | Consistency, efficiency, + behavioral compatibility (schema-level) as validation check types | Execute → within static validation row | ~80 words |
| 3 | Consistency refinement rule | Execute → within validation checks | ~50 words |
| 4 | Iteration budget (max 10 per cycle) | Design principles | ~80 words |
| 5 | Severity classification table | Assess | ~100 words |
| 6 | Risk-ordered execution principle (non-regressive first, escalated next, optimization last) | Propose → decomposition-first (or Design principles) | ~60 words |
| 7 | Loop stability design principle (convergence + oscillation rules) | Design principles | ~150 words |

**Total addition:** ~640 words to a ~4,500-word document (~14% increase). Each addition is a note, table, or rule — none requires new infrastructure, log fields, or tracking systems.

### Minimal additions to existing text

These aren't numbered changes — they're one-sentence insertions into existing prose:

- **Trust-based adoption principle** — add to the "Unknown sources" bullet: "When valuable ideas come from unvetted sources, extract the core concept into fully-owned content rather than creating a dependency."
- **Deterministic rigidity risk** — add: "When a deterministic check encounters unexpected structure, fall back to LLM processing and log the fallback."

### Defer to Phase 2–3

These ideas are logged here for future reference. Don't add them to the vision now.

- **Rationale metadata in lightweight format** (`rationales:` as string list per artifact) — add in Phase 2 when autonomous changes begin; populate incrementally as changes are reviewed, not all at once. This is the primary convergence mechanism for catching design-decision erosion that goal/scope checks miss. Upgrade to structured format (id + constraint + impact_if_violated) in Phase 3+ when volume justifies it.
- **Implements metadata** (`implements:` per artifact) — genuinely deferrable to Phase 3–4; add when vision-rationale impact analysis becomes a bottleneck at scale
- **Partitioned audit trail storage** — add when the outcome log exceeds ~5,000 entries
- **Consumer-behavioral compatibility checks** (LLM-assessed impact on downstream model behavior) — add when output comparison infrastructure exists; schema-level checks are applied now
- **Seven-dimension model** (referential integrity as separate dimension, dependency propagation as separate dimension) — add only if the current model produces classification errors in practice
- **Three-stage integration** (coherence verification between stages, formal Stage 1/Stage 2 tracking) — the risk-ordering *principle* is applied now (change 6); the *mechanism* (named stages with tracking, scheduling, and inter-stage coherence checks) belongs in Phase 2+ when compound triggers are common enough to justify infrastructure
- **Distance-based convergence measurement** — the automated mechanism (intent-preservation scoring across cycles) requires multi-cycle history. Defer to Phase 3+. But the groundwork — populating `goal:` and `scope:` metadata — is a Phase 1 activity, not a Phase 3 prerequisite. Without anchors, there's nothing to measure distance from.
- **Complexity governor** — avoid; use editorial discipline instead

### Don't apply

- YAML `description:` updates (cosmetic)
- Metadata defense citations in risks section (documentation polish)
- Duplicate rationale entries (R-G5 as both principle and rationale)
- Self-correcting gaps (G42/G44 fixing earlier proposals)
- Trust graduation mechanism (speculative)
