## 🎯 The problem

> **Scope note.** This vision is designed for prompt engineering systems built on GitHub Copilot's customization framework (instructions, agents, prompts, context files, skills, templates, hooks) in VS Code. The system is architected for portability: its core engine is repository-agnostic and installable via MCP servers, GHCP SDK components, or VS Code extensions. The principles and rationales transfer to any repository with PE artifacts.

Prompt engineering artifacts form a layered system where every layer depends on the ones below it. A context file defines a rule. An instruction file references that rule. An agent enforces it. A prompt orchestrates the agent. A template shapes the output.

This architecture works well when it's fresh. But it decays through three forces.

**Platform drift** is the primary cause. VS Code ships monthly. GitHub Copilot evolves weekly. New YAML fields appear, tool behaviors change, features move from preview to GA or get deprecated. A context file written for VS Code 1.100 may be actively wrong by 1.110.

**Internal drift** is the secondary cause. As you add artifacts, you introduce subtle inconsistencies: a rule tightened in one context file but not propagated to the agents that enforce it, a template referenced by a prompt that was renamed, a token budget defined in two places with different numbers. Beyond reference integrity, internal drift includes **guidance quality degradation**: rules that were clear when first written become ambiguous as the system grows, complete guidance that develops gaps as new artifact types are added, and priorities that become unclear as the rule set expands.

**Ecosystem evolution** is the third force. Microsoft ships official agent skills. OpenAI, Anthropic, Google, and Meta publish best practices and tooling. Community specialists produce high-quality, narrowly scoped artifacts. External work may overlap with, improve upon, or entirely replace something you've built. The system needs a way to evaluate what's available, decide whether to adopt or learn from it, and integrate either choice without breaking internal consistency.

All three forces share the same outcome: **the system silently degrades**. Silent degradation takes two forms that require different responses.

**Structural staleness** — dangling references, stale validation timestamps, broken chain alignment, mismatched metadata. Deterministic to detect. Fail-closed is the right response.

**Capability/logic staleness (Type B)** — the artifacts still parse, still reference-resolve, still chain-align, but *the logic they encode is obsolete*. A new model family is released and existing model-specific guidance is outdated. A feature moves from preview to GA and the "preview limitations" note remains. Type B staleness passes all integrity checks yet produces progressively worse outputs. This is the harder variant — and the primary motivating concern. Manual audits catch it eventually, but "eventually" is the failure mode.

**The cost challenge.** Detecting drift — especially Type B — requires comparing artifacts against external reality. For a system with dozens of artifacts across multiple types, full-depth assessment of every artifact on every review cycle is prohibitively expensive. The cost grows with the number of artifacts × the number of external change sources × the depth of the dependency chain. A sustainable self-update system must bound this cost without sacrificing detection quality — assessing deeply where it matters and lightly where it doesn't, guided by what's actually changed in the world.

## 💡 The goal

Build a self-update system that maintains prompt engineering artifacts at peak reliability, effectiveness, and efficiency — with the maximum degree of autonomy that assessed risk allows — by continuously detecting drift, evaluating improvement opportunities, and executing validated corrections.

### Priority: staleness resolution through source-grounded autonomy

Resolving staleness is not a secondary optimization. It is a top-priority objective for the PE system.

The system should:

1. Investigate authoritative sources autonomously as a default behavior.
2. Investigate user-provided sources autonomously when they are supplied.
3. Validate source trust, relevance, and claim consistency before integration.
4. Integrate validated findings autonomously when risk is below threshold.
5. Escalate to human approval for high-impact, low-confidence, or contradictory evidence.

This priority is how reliability, effectiveness, and efficiency are preserved over time under Type B staleness pressure.

### Two dimensions

**The self-update process itself** must be reliable (detecting the same drift consistently when run twice), effective (catching real drift and proposing correct fixes), and efficient (assessing deeply where changes warrant it and lightly where they don't — bounded by a progressive cost gradient from zero-cost cache hits through lightweight screening to full-depth review, never applying more assessment than the evidence requires).

**The artifacts the process maintains** must reliably, effectively, and efficiently serve their defined purpose — producing predictable outputs, achieving their stated objectives, consuming minimal tokens, attention, and effort.

### Design and review parity contract

Design and review share one destination: the same quality objective and the same scope intent, filtered only by artifact applicability.

Process differences are role-specific, not quality-specific:

1. **Design** performs requirements synthesis and construction to reach the quality target.
2. **Review** performs analysis, verification, and improvement to keep artifacts at that same target.
3. **Governance and cost controls** can change execution depth or approval posture, but they MUST NOT lower quality ambition for either role.

### Guidance quality as prerequisite

The self-update system's ability to maintain artifacts "at peak reliability, effectiveness, and efficiency" is bounded by the quality of the guidance those artifacts are built from. Six measurable properties determine guidance quality:

| Property | Definition | Measurement |
|---|---|---|
| **Clarity** | A rule can be understood unambiguously by both humans and LLMs | LLM agreement test: two independent passes produce the same interpretation |
| **Non-redundancy** | Each rule is defined in exactly one canonical location | Automated: scan for duplicate rule text across context files |
| **Non-contradiction** | Rules across files do not conflict | Automated + LLM: coherence-check Workflow 2 |
| **Completeness** | No gaps exist where guidance is needed but absent | LLM-assisted: scan agent behaviors that lack backing rules |
| **Prioritization** | When rules could conflict, precedence is explicit | Automated: check critical-rules-priority-matrix coverage |
| **Actionability** | An agent can act on the rule without additional interpretation — AND can enforce its own metadata as runtime constraints | LLM test: can the rule be translated into a boolean check? Can the model verify "am I within scope?" |

Guidance quality is a **precondition for autonomy**, not a parallel concern. Before increasing autonomy phase (Phase 2→3→4), the guidance that the system enforces should meet minimum quality thresholds on all six properties. Autonomous application of ambiguous or contradictory guidance produces unreliable results regardless of how accurate the classification machinery is.

### Prerequisites

The self-update workflow requires two things that can't be substituted:

1. **A clear vision with explicit rationale.** If the system knows *what* it's trying to achieve and *why*, it has a reference point for evaluating any change — internal or external, platform-driven or ecosystem-driven. Without a well-defined vision, there's nothing to validate against.
2. **Access to the world.** If the system can observe what's happening — platform releases, ecosystem developments, its own internal state — it has the raw material to reason about. Without awareness of change, there's nothing to respond to.
3. **Modular, type-aware invocation.** The system can be invoked for specific artifact types — "review my agents", "update context files" — not just "review everything." Each invocation assesses the target type deeply while screening its dependencies lightly, escalating to deep assessment only when screening detects issues. This keeps the cost of any single invocation proportional to the work that actually needs doing, not the size of the entire artifact set.

### The autonomy gradient

The degree of autonomy isn't binary — it's a gradient calibrated to assessed risk:

| Autonomy level | Scope | Examples |
|---|---|---|
| **Autonomous** | Low impact, high confidence | Reference updates, stale timestamp fixes, formatting corrections |
| **Autonomous with notification** | Low-medium impact, validated | Redundancy removal after validation passes, adding new metadata fields |
| **Human approval required** | High impact or medium confidence | Restructuring artifacts, changing rules, adopting external artifacts |
| **Human-only** | Architectural or strategic | Vision changes, principle modifications, risk threshold adjustments |

**Low-risk autonomy rule:** Changes classified as non-breaking (goal, scope, and boundaries preserved) with high confidence (machine-verifiable or multi-pass validated) MUST proceed autonomously regardless of which command family triggered them. A command's default mode (`plan` vs `apply`) is a UX convenience for the user's workflow preference — it MUST NOT be interpreted as a permission ceiling that overrides risk classification. The only legitimate gate is assessed risk, never command identity.

### Change classification

The autonomy level for any change is determined by **assessed impact × confidence**. A **non-breaking** change preserves the artifact's declared goal, scope, and boundaries. A **breaking** change alters them. Impact is multi-dimensional: consistency, efficiency, behavioral compatibility, and effectiveness are each assessed independently — the highest-risk dimension governs.

Confidence increases when classification is machine-verifiable: metadata comparison (R-S1), structural separation labels (R-P4), and historical success rates (R-G3). Without metadata and N-1 labels, all classification falls to LLM judgment — the most expensive and least confident path.

**Full classification mechanics**: See The rationale § Change classification mechanics.

### Key definitions

| Term | Definition |
|---|---|
| **Reliable** | Produces consistent, predictable outcomes across invocations |
| **Effective** | Achieves its stated purpose |
| **Efficient** | Achieves its purpose with minimal resource consumption |
| **Autonomous** | Operates without human intervention for decisions at or below a validated risk threshold |
| **Risk-calibrated** | Autonomy level is determined by assessed impact × confidence |

### Goal-trio↔use-case mapping

The three goal poles (reliability, effectiveness, efficiency) are the canonical organizing axis for the use-case catalog. Every use case MUST serve at least one pole. The catalog folder structure mirrors this mapping:

| Goal pole | Use-case folder | Canonical `--dim` group |
|---|---|---|

## ⚙️ The rationale (system design justifications)

> **Terminology note.** This section describes **system-level design rationales** — the architectural reasons WHY the self-update system is built the way it is (R-L1, R-P1, R-S1, etc.). These are distinct from **artifact `rationales:` metadata** — the per-artifact YAML field that captures design decisions specific to each individual artifact. See the Metadata-driven principle for artifact-level rationales.

Maintaining a PE system against drift is familiar engineering work: read release notes, scan documentation, review external work, assess relevance, apply changes aligned with your vision. LLMs can perform every step of this process. The self-update workflow has nine steps — each within reach of current LLM capabilities, often augmented by deterministic tooling.

| Step | What happens | LLM role |
|---|---|---|
| **Search** | Gather change digest: monitor platforms, ecosystem, model updates, tooling, vision changes, outcome history, and internal modifications since last review | Browse documentation, release notes, changelogs; analyze outcome log; compare file hashes — producing a structured change digest that informs all subsequent assessment |
| **Analyze** | Screen dependencies against change digest; deep-assess target artifacts | Tier 1: deterministic structural checks (references, hashes, metadata). Tier 2: relevance screening against change digest (~800 tokens). Deep pass: full quality assessment for targets and escalated dependencies |
| **Reason (organizational)** | Assess whether the trigger implies structural changes to artifact organization | Evaluate role clarity, dependency structure, layer correctness, coverage gaps, redundancy |
| **Reason (per-artifact)** | Assess whether the trigger implies content changes to specific artifacts | Evaluate quality, adherence, drift for individual artifacts within the validated structure |
| **Design** | Propose what the implementation should look like | Restructure artifacts, rethink relationships |
| **Plan** | Sequence changes for minimal disruption | Break design into concrete steps, prioritize by effectiveness per effort |
| **Validate** | Confirm proposed changes don't regress existing behavior | Compare before and after, check coherence |
| **Implement** | Execute the plan | Apply changes through tool use |
| **Verify** | Confirm actual changes match the plan's intent | Run coherence checks, validate outcomes |
| **Iterate** | Cycle back if verification reveals gaps | Refine design, adjust plan, escalate |

### LLM capabilities

| ID | Rationale | Design implication |
|---|---|---|
| **R-L1-llm-reasoning** | **LLM reasoning and judgment.** LLMs can analyze artifacts, compare implementations against specifications, identify inconsistencies, and make judgment calls about whether a change improves adherence to the vision. | Delegate analysis, comparison, and decision-making to LLMs. Reserve human oversight for decisions above the risk threshold. |
| **R-L2-self-correction** | **Self-correction through multi-pass validation.** A single LLM pass can introduce errors. A second pass — by the same or different model — detects and corrects those errors before commit. | No change is committed without at least one independent validation pass. Critical changes get multiple passes. |
| **R-L3-model-specialization** | **Model specialization.** Reasoning models excel at analysis and planning. Standard models excel at execution. Small models suffice for deterministic-like tasks. | Assign models by task type: research/analysis → reasoning models; implementation → standard models. See R-P6-model-routing for the operational application. |
| **R-L4-external-knowledge** | **External knowledge access.** LLMs can consume platform documentation, release notes, changelogs, and community repositories — giving the system awareness of changes without requiring a human to read and summarize them. | Monitor defined sources (VS Code release notes, Copilot documentation, trusted ecosystem projects). |

### Processing strategies

| ID | Rationale | Design implication |
|---|---|---|
| **R-P1-deterministic-processing** | **Deterministic processing for predictable operations.** Parsing YAML, checking file existence, comparing timestamps, computing content hashes don't need LLM judgment. Deterministic tools are cheaper and more reliable. | Use deterministic tools for all predictable operations. Reserve LLM processing for tasks requiring reasoning or natural language understanding. |
| **R-P2-decomposition** | **Decomposition for context window control.** LLM accuracy degrades as context grows (context rot — accuracy drops from 88% to 30% at 32K tokens). Self-update tasks spanning dozens of artifacts must be decomposed into focused units. | Self-update workflows decompose into phases. Each phase has a token budget. Handoffs carry only essential state. |
| **R-P3-structured-contracts** | **Structured contracts between phases.** When LLM output feeds a downstream phase, structured formats (JSON, YAML, tables) bridge probabilistic LLM output with deterministic downstream processing. | Inter-phase communication uses defined schemas. LLM output is parsed deterministically into structured records. |
| **R-P4-structural-separation** | **Structural separation for deterministic classification.** Labeled blocks (Rule/Rationale/Example) make breaking/non-breaking classification deterministic — without them, every diff requires LLM judgment. See *Rules, rationales, and examples are separable* principle. | Classification logic checks block labels first, content second. |
| **R-P5-progressive-depth** | **Progressive validation depth for cost-bounded assessment.** Full-depth validation of all dependencies doesn't scale. Three phases (research → screening → deep) bound cost while preserving quality chain guarantees. See Assess § Assessment depth for the three-phase model and cost gradient. | Apply three phases: research (once) → screening (per dependency) → deep pass (targets + escalated). Maximum cascade depth: 2 levels. |
| **R-P6-model-routing** | **Model routing for cost-optimal quality.** Different assessment steps have different computational requirements. Routing reasoning tasks to reasoning models, execution tasks to standard models, and predictable operations to deterministic tools optimizes both cost AND quality — each step gets the best-suited processing without over-provisioning. | Assign models by task type within each assessment step. Research/analysis → reasoning models. Implementation/formatting → standard models. YAML parsing, reference checking, hash comparison → deterministic tools. Model routing is a dimension of review quality (see Assess § Selective review dimensions) — adherence to routing recommendations is verifiable. |
| **R-P7-capability-applicability** | **Capability-based option applicability.** Commands accept options based on implemented capabilities, not family membership. When an unsupported option is provided, the system MUST: (1) reject deterministically, (2) name the missing capability, (3) suggest the canonical command that provides it. This enables consistent UX across command families while allowing capability-gated extension. | Each command declares its supported option classes. The option applicability matrix is the single source of truth for which options each command accepts. Unsupported options produce deterministic corrective guidance, not silent ignoring or ambiguous errors. |

### System infrastructure

| ID | Rationale | Design implication |
|---|---|---|
| **R-S1-metadata-driven** | **Metadata-driven automation.** Machine-readable metadata transforms implicit knowledge into actionable state, enabling deterministic classification, scoped validation, and runtime self-constraint enforcement. See *Metadata-driven* principle. | Every artifact carries metadata declaring dependencies, freshness, validation history, goal, scope, boundaries, and rationales. Executed artifacts (agents, prompts) ground their runtime behavior on this metadata. |
| **R-S2-dependency-graph** | **Dependency graph for scoped impact analysis.** Given a change to artifact X, the dependency map identifies all artifacts that reference X, assesses the blast radius, and scopes validation to only what's affected. | Every self-update action starts with a dependency query. High-dependency artifacts get broader validation; leaf artifacts get minimal validation. |
| **R-S3-validation-caching** | **Validation caching to avoid redundant work.** If an artifact was validated against the current platform version and hasn't changed since, skip it. | Self-update cycles check cached validation state before initiating processing. (Carve-out: meta-infrastructure is never cached — see Staleness avoidance principle.) |
| **R-S4-role-declaration** | **Artifact role and purpose declaration.** Each artifact declares its role (what it does), its purpose (why it exists), and its success criteria (how to tell if it's working). | The self-update system evaluates artifacts against their declared purpose, not just structural rules. |
| **R-S5-chain-alignment** | **Chain alignment for stable cross-artifact references.** Artifacts should reference capabilities at the coarsest grain that preserves precision. Three levels are available, in order of preference: **Level 1 (folder/domain)** — reference by folder path, stale-proof but imprecise. **Level 1.5 (category)** — reference by functional category ID defined in a structure index (e.g., STRUCTURE-README.md), precise AND stale-proof because category IDs are stable semantic identifiers that survive file renames. **Level 2 (file-specific)** — reference by filename, most precise but fragile. Default to Level 1.5 for cross-artifact references. Use Level 1 when the entire folder's content is relevant. Use Level 2 only when precision is load-bearing (singleton files like dispatch tables, templates, handoff targets) and the filename is descriptive and non-numeric. | Declare `enforces_rulesets` by canonical file path (Level 1, default for YAML metadata). For prose references in agent bodies and prompts, use category IDs from the structure index (Level 1.5, preferred). Use `enforces_rules` with specific IDs only by exception (Level 2). Structure indexes maintain a Functional Categories section mapping stable category IDs to current filenames. A rename log at `<state.path>/rename-log.jsonl` records every rule/file rename with `old → new` mapping for automated reference rewrite. |
| **R-S6-tier-blast-radius** | **Tier-based blast radius.** Tier encodes how many layers can depend on an artifact. A change at Tier N may affect Tiers N+1 through 5 only (upward-only dependency rule). Tier is an O(1) input to risk classification, cheaper and broader than dependency-count alone. | Use tier as the primary sort key in risk classification. Tier 0 → always escalate. Tier 5 → presumptively autonomous. Tiers 1–4 → route by breaking/non-breaking classification and chain coverage. Group snapshots for rollback are tier-scoped. Risk classification matures with the rollout phases. At Phase 1, tier (from filename prefix) and dependent count (from grep) provide a coarse but cheap estimate. At Phase 2, actual dependency topology (from `context_dependencies:`, `handoffs:`, `📖` references, and `applyTo` patterns) replaces the grep count with graph-based reachability. At Phase 3, each dependency is classified by coupling strength: behavioral (enforces rules from), structural (parses structure of), informational (mentions), or cosmetic (documents). The blast radius estimate improves at each phase without requiring the prior phase's method to be discarded — each phase adds a refinement layer. |
| **R-S7-portable-packaging** | **Portable packaging through platform mechanisms.** Engine logic delivered via standard mechanisms (MCP, SDK, extensions) without imposing framework structure. See *Portable by design* principle. | Package for standard delivery vehicles. Discover rather than prescribe artifact structure. |
| **R-S8-instruction-minimization** | **Instruction minimization for collision prevention.** Auto-inject with no precedence means conflicting behavioral rules collide silently. Testable/mechanical rules only. See *Instruction minimization* principle. | Instruction files contain only testable/mechanical rules. Behavioral rules delegated to context files and agent bodies. |
| **R-S9-guidance-quality** | **Guidance quality as autonomy prerequisite.** Autonomy reliability is bounded by guidance quality — ambiguous, incomplete, or contradictory guidance produces unreliable autonomous actions. See Goal § Guidance quality as prerequisite and *Guidance quality by construction* principle. | Phase 2 requires non-redundancy + non-contradiction + prioritization. Phase 3 requires all 6 properties above threshold. |
| **R-S10-quality-propagation** | **Quality propagation through dependency chains.** Achievable quality is bounded by dependency quality. Distinct from R-S2: R-S2 answers "what breaks?" — R-S10 answers "what can't be reliable?" See *Quality flows through dependencies* principle. | Assess bottom-up through dependency chains. Full traversal during reviews; scoped to affected path for individual changes. |

### Governance model

| ID | Rationale | Design implication |
|---|---|---|
| **R-G1-autonomy-gradient** | **Risk-calibrated autonomy gradient.** Autonomy isn't binary. The system operates at different levels depending on assessed risk — from fully autonomous through notification-only to human-required to human-only. | Each proposed change is classified on the gradient (canonical table: The goal → The autonomy gradient). The gradient is configurable — humans can tighten or loosen based on observed performance. |
| **R-G2-trust-calibrated-adoption** | **Trust-calibrated ecosystem adoption.** The system evaluates external work against internal standards and either adopts it (replacing owned artifacts) or incorporates its ideas (improving owned artifacts) — calibrated to source trustworthiness. | Maintain a trust registry. Adoption decisions surface to humans; incorporation can proceed autonomously if validated. |
| **R-G3-progressive-learning** | **Progressive learning from outcomes.** The system tracks which changes improved quality and which introduced regressions. Human rejections, including the stated reason, feed back as negative signal for threshold tuning. | Log all autonomous actions and their outcomes in the structured outcome log (`<state.path>/outcome-log.jsonl`). Periodic meta-analysis informs threshold tuning. |

### Conflict resolution among rationales

When rationales pull in opposite directions, the meta-principle is: **safety takes precedence over efficiency; reliability takes precedence over autonomy.** Specific resolution rules for known conflict pairs:

| Conflict | Resolution |
|---|---|
| R-L1-llm-reasoning vs R-P1-deterministic-processing | R-P1-deterministic-processing wins when both can perform the operation. LLM only when judgment is required. |
| R-S3-validation-caching vs R-L4-external-knowledge | R-L4-external-knowledge wins when there's evidence of platform change. Re-validate even if cached. |
| R-L2-self-correction vs R-P2-decomposition | R-L2-self-correction wins up to the budget, then escalate. Never skip validation to save tokens. |
| R-G1-autonomy-gradient vs R-L2-self-correction | R-L2-self-correction is a precondition for R-G1-autonomy-gradient. The gradient classifies only after validation has run. |
| R-G2-trust-calibrated-adoption vs R-G1-autonomy-gradient | R-G1-autonomy-gradient governs the final decision. Trust reduces assessment cost, not the autonomy threshold. |
| R-P6-model-routing vs R-P1-deterministic-processing | R-P1-deterministic-processing wins when the operation is fully deterministic. R-P6-model-routing governs model selection only for operations that require LLM processing. |
| Process optimization vs design/review parity | Design/review parity wins on quality target and scope intent. Process optimization may reduce cost or change execution posture, but MUST NOT define a lower quality objective for either role. |
| R-S9-guidance-quality vs R-G1-autonomy-gradient | R-S9-guidance-quality is a precondition for R-G1-autonomy-gradient. The gradient classifies only after guidance quality is confirmed for the affected rules. |
| Terminology overload (robustness alias) | "Robustness" is a reliability property in standard SE usage. The legacy `--dim robustness` alias bound to adherence dimensions is renamed to `--dim adherence`. The name "robustness" is freed and kept synonymous with `--dim reliability` (system-level process reliability — reproducibility, loop stability, rollback readiness, boundary actionability). Back-compat: `--dim robustness` accepted for one release with deprecation notice; resolves to `--dim adherence`. |

Most rationale pairs are orthogonal and never compete. The conflict set is small and bounded.

### Change classification mechanics

The core impact dimension is whether a change is breaking or non-breaking. A **non-breaking** change preserves the artifact's declared goal, scope, and boundaries while improving its reliability, effectiveness, or efficiency. A **breaking** change alters what the artifact is for, what it covers, or what it constrains.

| Change category | Goal/scope/boundary impact | Impact level |
|---|---|---|
| Content fix, better wording, example update | Preserved | Low |
| Deeper coverage within declared scope | Preserved | Low-medium |
| Scope extension (new topics) | Scope expands | Medium-high |
| Scope narrowing (removes coverage) | Scope contracts | High |
| Goal or boundary change | Goal or boundaries changed | High |

**Impact is multi-dimensional.** Goal/scope/boundary preservation describes the structural dimension only. A change can be non-breaking structurally while having impact on other dimensions:

- **Consistency** — does it contradict or reprioritize existing rules?
- **Efficiency** — does it increase token consumption beyond declared budgets?
- **Behavioral compatibility** — does it break consumers that parse the artifact's structure (YAML fields, section headings, metadata format)?
- **Effectiveness** — does it degrade output quality? (Only measurable through actual use — see effectiveness hard limit in Risks.)

Each dimension is assessed independently. The autonomy level for a change is determined by the *highest-risk* dimension. When assessing impact, consider all applicable dimensions — but don't mandate a formal multi-dimensional evaluation for every change. The cost of assessment must be proportionate to the change's scope.

Behavioral compatibility has two cost layers. Schema-level checks (YAML structure, section headings, metadata field names) are deterministic and near-zero cost. Consumer-behavioral checks (does restructured content change LLM behavior?) require extra LLM passes and belong in Phase 2–3. Only the schema-level layer is applied now.

**Confidence** increases when the determination is machine-verifiable: deterministic comparison of goal/scope/boundary metadata (R-S1-metadata-driven), multi-pass LLM validation (R-L2-self-correction), deterministic structural checks (R-P1-deterministic-processing), and historical success rate for the change type (R-G3-progressive-learning). Without structured metadata declarations, "is this change non-breaking?" requires LLM judgment and is therefore lower-confidence.

**How classification works in practice.** The breaking/non-breaking distinction is only useful if it can be assessed cheaply and confidently. The mechanism that enables this is **structural separation (R-P4-structural-separation)**: the N-1 convention where rule-bearing sections label their content as Rule, Rationale, or Example.

With N-1 in place, classification follows a deterministic path — see *Rules, rationales, and examples are separable* principle for the block-label mechanism. Without N-1, every diff requires LLM judgment — collapsing the autonomy gradient to a binary: trivial structural changes (autonomous) vs. everything else (human-required).

## 🏗️ The vision

A self-updating prompt engineering system **continuously monitors its own health**, **detects when artifacts may have drifted from reality**, **proposes corrections before the drift causes failures**, and **executes validated corrections autonomously when risk is low enough**.

It isn't a system that rewrites itself without oversight. It's a system that behaves like a responsible maintainer: it watches for change signals, assesses impact, runs targeted checks, and acts on findings — autonomously for low-risk changes, with human approval for high-risk ones.

The architecture separates **engine** from **integration**. The engine contains detection, assessment, proposal, validation, and execution logic. The integration is the thin per-repo layer that points the engine at local artifacts and optional runtime hooks. This separation keeps the logic portable while preserving repository-specific control.

The system operates **modularly and progressively**. It can be invoked for a specific artifact type — reviewing agents without full-depth analysis of every context file they depend on. Assessment proceeds in three phases: research gathers what's changed in the world (once per invocation), informed screening checks dependencies cheaply (deterministic + lightweight relevance), and deep pass applies full quality requirements only to targets and dependencies that screening flags. This progressive model keeps each invocation cost-bounded while preserving detection quality — the system never applies more assessment than the evidence requires.

The vision has four operational layers.

### Per-repo integration model

The integration footprint stays intentionally small.

| File/component | Purpose | Required? |
|---|---|---|
| `pe-self-update.config.json` | Declares artifact paths, autonomy thresholds, iteration budget, monitored sources, and state namespacing | Yes |
| Optional `.github/copilot-instructions.md` addition | Makes self-update capability visible in conversational workflows | No |
| Optional `.github/agents/pe-self-update.agent.md` | Provides `@mention` wrapper over engine tools | No |
| Optional namespaced state folder | Stores repo-local logs and snapshots at the configured `state.path` | No |

Default target is **1 required file + optional wrappers**. The engine can also keep state outside the repo when configured.

Required state config keys in `pe-self-update.config.json`:

- `state.namespace` — unique namespace for engine/domain instance
- `state.path` — resolved state directory path (repo-local or external)

Guardrail: no engine may use an unscoped shared state directory by default.

### Delivery vehicles

The same engine can be delivered through different interfaces.

| Vehicle | How it works | Best for |
|---|---|---|
| **MCP server** (primary) | Exposes engine capabilities as tools callable from chat, prompts, and agents | Composable automation and direct tool control |
| **GHCP SDK component** | Runs as a hosted service with managed execution | Always-on and centrally managed workflows |

### Three operating levels

Assessment and proposal can run at three levels, in strict order for broad triggers:

**Organizational level** — The artifact set as a system is the reference point. Evaluates role clarity, dependency structure, layer correctness, and coverage/redundancy. Runs first on broad triggers because per-artifact improvements within a suboptimal organization are wasted effort. See Assess § Assessment ordering for the full checklist. For narrow triggers, skipped unless the change implies organizational impact.

**Architecture level** — The full vision is the reference point. The system evaluates whether a major external change (a new platform capability, a fundamentally better approach from a trusted source) warrants structural rethinking.

**Quality dimension level** — Individual measurable properties serve as the reference. These operate at two scales:

**Per-artifact quality**: non-contradiction (within the file), consistency (with declared metadata), reliability (produces predictable output), effectiveness (achieves stated purpose), efficiency (within token budget).

**System-level quality**: non-contradiction (across files), non-redundancy (each concept in one canonical location), completeness (no guidance gaps for active artifact behaviors), prioritization (conflict resolution is explicit), layer correctness (right content in right artifact type — testable rules in instructions, behavioral guidance in context files, type-specific logic in agent bodies), and token-chain efficiency (total loading cost for a consumer's guidance set stays within context window bounds).

Per-artifact quality is assessed on every validation. System-level quality is assessed during ecosystem audits (`/pe-meta-update --mode plan --skip research`) and scheduled reviews.

Some quality dimensions are independent (non-redundancy and non-contradiction can be assessed in isolation). Others are coupled (reliability and effectiveness are coupled with efficiency — a change improving one may degrade the other). Independent dimensions run in parallel; coupled dimensions run together.

### Detect

*Relies on: R-S1-metadata-driven, R-L4-external-knowledge, R-S3-validation-caching*

The system knows when to look. Five trigger sources initiate scoped health checks:

| Signal | Source | What it indicates |
|---|---|---|
| **Platform event** | VS Code/Copilot release notes, changelog feeds | New YAML fields, new features, deprecations, GA transitions |
| **Model event** | Provider release notes, model catalogs | Capability shifts — re-evaluate model-specific guidance |
| **Ecosystem event** | Trusted sources (Microsoft, OpenAI, Anthropic, Google, Meta, recognized specialists) | Potential adoption or incorporation candidate |
| **Internal event** | File system watcher / git hook (existing `pe-file-modification-tracker`) | Scope the dependent check |
| **Scoped invocation** | User or automation requests review of a specific artifact type | "Review my agents", "Update context files", "Validate prompts" |
| **Scheduled** | Time-based interval | Catch anything that event triggers missed |

Trigger wiring is integration-dependent. Existing hooks (`pe-healthcheck`, `pe-staleness-check`, `pe-file-modification-tracker`) remain a valid deterministic host, but MCP/SDK/extension schedulers are equally valid trigger hosts in portable deployments. Validation caching (R-S3-validation-caching) skips artifacts validated recently against the current platform version — but never meta-infrastructure components (see Staleness avoidance principle). Missing or broken trigger wiring is a CRITICAL finding.

### Assess

*Relies on: R-L1-llm-reasoning, R-S2-dependency-graph, R-S6-tier-blast-radius, R-G2-trust-calibrated-adoption, R-S9-guidance-quality, R-S10-quality-propagation*

The system knows what to check.

**Assessment ordering.** For broad triggers (platform releases, ecosystem changes, scheduled reviews), assessment follows a two-pass sequence:

**Organizational pass.** Before evaluating individual artifact quality, evaluate the artifact set as a system:
- **Role clarity**: Are artifact roles well-defined and non-overlapping? Does each artifact have a single, distinct responsibility?
- **Dependency structure**: Is the dependency graph clean? No circular dependencies, minimal coupling, clear hierarchical layers?
- **Layer correctness**: Is content in the right artifact type? Testable rules in instruction files, behavioral guidance in context files, type-specific logic in agent bodies?
- **Coverage and redundancy**: Are there gaps where guidance is needed but absent? Are there redundancies where the same concept is defined in multiple places?

Organizational findings are addressed before the per-artifact pass begins — optimizing individual artifacts within a suboptimal organization is wasted effort. Organizational changes are HIGH impact by definition (restructuring) and route through the standard autonomy gradient.

**Per-artifact pass.** With organizational structure validated or accepted, assess individual artifact quality following quality dependency chains (R-S10-quality-propagation): start with the lowest-tier dependencies and work upward, so each artifact is assessed against already-validated dependencies. Per-artifact assessment uses the existing mechanisms: tier + dependency analysis for risk, N-1 for diff classification, metadata for scope preservation.

**Artifact-type-aware routing within the per-artifact pass.** Per-artifact assessment follows R-S10's bottom-up ordering, but different artifact types carry different quality requirements reflecting their role in the system:

| Artifact type | Role | Update-path quality requirements |
|---|---|---|
| **Context files** | Define guidance (rules, patterns, constraints) | Quality-by-construction: 6 properties as construction invariants (see Guidance quality by construction principle) |
| **Instruction files** | Auto-inject testable rules | Minimization check (R-S8) + non-contradiction with context layer |
| **Agent bodies** | Consume guidance + implement logic | Adherence check against loaded guidance + behavioral correctness |
| **Prompts** | Orchestrate agents | Adherence check + workflow correctness |
| **Templates, snippets, hooks** | Shape output / trigger wiring | Structural correctness (schema, references) |

When a broad trigger fires (platform release, ecosystem change):
1. Assess guidance artifacts first — apply construction invariants if updates are needed
2. Then assess consumer artifacts — check adherence against the *already-validated* guidance
3. Each artifact type follows its own quality requirements — no cross-cutting classification needed

**Explicit phase sequence for system-wide assessment.** The bottom-up ordering above is operationalized as a strict phase sequence:

| Phase | Artifact type | Assessment | Why this order |
|---|---|---|---|
| **Phase A** | Context files | All applicable dimensions + construction invariants + peer cross-coherence | Foundation layer — sets the quality ceiling for everything above |
| **Phase B** | Instruction files | Structural + minimization (R-S8) + non-contradiction with context layer | Must be validated against already-validated context files |
| **Phase C** | Agent bodies | All applicable dimensions + adherence (D16) against Phase A/B results | Consumes guidance — adherence only meaningful against validated guidance |
| **Phase D** | Prompts | All applicable dimensions + adherence (D16) against Phase A/B/C results | Orchestrates agents — depends on validated agents and guidance |
| **Phase E** | Templates, snippets, hooks | Structural dimensions (D1-D3, D14) | Leaf artifacts — minimal dependencies |
| **Phase F** | Skills | Structural + strategic dimensions | May reference context files and agents |

If Phase A finds issues in context files, those MUST be resolved (or accepted as known issues with documented rationale) before Phase C/D assess consumer adherence. Adherence against unvalidated guidance produces unreliable results.

When a narrow trigger fires (single artifact change):
- If the changed artifact is a guidance artifact → apply construction invariants to the change
- If the changed artifact is a consumer → check adherence against current guidance; don't trigger guidance quality investigation
- If the change alters a dependency relationship → re-assess the affected quality chain (R-S10)

This routing is a natural extension of R-S10's bottom-up ordering — guidance artifacts are the lowest tier in every quality chain, so they're assessed first. Consumer artifacts inherit a quality ceiling from their guidance, so they're assessed second with adherence checks instead of construction invariants.

**Assessment depth: research → informed screening → deep pass.** Assessment operates in three phases. The research phase runs once per invocation. Screening runs per-dependency using research output. Deep pass runs on targets and escalated dependencies.

**Research phase** (once per invocation) — Gathers what's changed since the last review: platform releases, ecosystem events, model updates, tooling/infrastructure changes, vision/governance changes, outcome history (R-G3), and internal modifications. Produces a structured **change digest** (~500 tokens) that all subsequent screening checks reference. Cost: one research operation per invocation, amortized across all screened artifacts. When no external changes are detected, the digest is empty and Tier 2 screening is skipped entirely.

`last_review_timestamp` records when the research phase last ran. It is updated in the state folder (`<state.path>/triggers/last_review_timestamp`) at the end of every successful research phase — not at invocation start, but after the digest is successfully produced. When absent or older than the configurable maximum staleness window (default: 90 days), research gathers changes from the lookback window instead.

**Informed screening** (per dependency) — Two tiers within the same pass:

*Tier 1 — Structural screening (deterministic)*: metadata currency, reference integrity, content hash stability, structural completeness (YAML fields, N-1 labels), and validation cache freshness (R-S3). Catches internal changes. If Tier 1 passes and the change digest is empty → screening passes.

*Tier 2 — Relevance screening (context-aware, runs only when change digest is non-empty)*: compares the artifact's declared scope and authoritative sources against the change digest. If the artifact's scope intersects with a changed platform area, overlaps with a new ecosystem artifact, or references capabilities affected by a model update → escalate to deep pass. Cost: lightweight LLM relevance judgment (~800 tokens per artifact). Catches semantic staleness (Type B) — the primary motivating concern.

**Deep pass** (per target + escalated dependencies) — Full quality requirements from the artifact-type routing table: construction invariants for guidance artifacts, adherence checks for consumers, minimization for instructions. Informed by the change digest so the reviewer assesses against specific external changes rather than doing open-ended staleness detection.

**Escalation**: When screening fails on a dependency (Tier 1 structural failure or Tier 2 relevance match), that dependency receives a deep pass before the target's review proceeds. Escalation is scoped to the specific failing dependency. Maximum cascade depth: 2 levels. Second-level escalation is reported as a finding rather than automatically pursued.

**Cost gradient**:

| Condition | Action | Cost |
|---|---|---|
| Cache fresh, no external changes | Skip (R-S3) | Zero |
| Cache stale, Tier 1 passes, no external changes | Pass at Tier 1 | Near-zero |
| Cache stale, Tier 1 passes, external changes exist | Tier 2 relevance screening | Low (~800 tokens) |
| Screening fails (Tier 1 or Tier 2) | Deep pass | Full |

The research phase extends the Detect layer's trigger sources into the Assess layer — triggers identify *that* something changed; research identifies *what* changed. Screening extends R-S3 (validation caching) from a binary skip/deep model to a four-level cost gradient. Tier 2 directly addresses Type B staleness — the vision's primary motivating concern — by checking whether the world has changed in ways that affect the artifact's scope, even when the artifact's content is unchanged.

**Modular invocation.** The self-update pipeline can be invoked for a specific artifact type rather than the full artifact set. Scoped invocation determines which artifacts receive the deep pass and which receive screening:

| Invocation scope | Deep pass target | Screening dependencies | Cascade on failure |
|---|---|---|---|
| Context files | Context files | None (base layer) | N/A |
| Instruction files | Instruction files | Context files | → deep context review |
| Agents | Agent files | Context + instruction files | → deep instruction/context |
| Prompts | Prompt files | Agents + context + instructions | → deep agent (may cascade) |
| Skills | Skill files | Context + instructions | → deep instruction/context |
| Templates, hooks | Templates, hooks | None (leaf artifacts) | N/A |
| Full system | All types | N/A (all deep) | N/A |

Scoped invocation operates within the existing Assess pipeline — it's a trigger type that constrains which artifacts are targets and which are screened. The organizational pass is skipped for scoped invocations unless the invocation implies organizational impact.

**Cross-invocation caching**: Screening results are cached within the invocation. If context files are screened during instruction review and then needed again during agent review (in a cascade), the cached screening result is reused.

**Cascade depth limit**: Maximum 2 levels. If a screening-triggered deep pass itself triggers further screening escalation, the second-level failure is reported as a finding for human review rather than automatically pursued. This bounds the cost of any single invocation.

For narrow triggers (single file change, specific internal event), the organizational pass is skipped unless the change implies organizational impact (e.g., adding a new artifact type, splitting an existing artifact, creating a new dependency relationship).

Risk classification uses two inputs in order:

1. **Tier (R-S6-tier-blast-radius) — first sort key.** O(1) lookup from the artifact's filename prefix. Tier 0 → always escalate. Tier 5 → presumptively autonomous. Tiers 1–4 → route by breaking/non-breaking classification (see The goal). Tier is a **heuristic proxy** for dependency depth — it approximates blast radius from naming convention rather than measuring actual topology. This is acceptable at Phase 1 (manual invocation) where humans validate every decision. At Phase 2+, tier should be cross-validated against the actual dependency graph — a tier mismatch (filename says Tier 3 but graph shows Tier 1 reachability) is a finding.
2. **Dependent analysis (R-S2-dependency-graph) — secondary.** Within a tier, dependency characteristics determine blast radius scope. At Phase 1, raw dependent count (grep-based) provides a coarse signal — more connections mean wider blast radius. At Phase 2+, dependencies are classified by coupling type:

   | Coupling type | Weight | Detection | Example |
   |---|---|---|---|
   | **Behavioral** | High | Agent/prompt loads and enforces rules from this artifact | Agent's `📖` reference + rule citation in boundaries |
   | **Structural** | Medium | Consumer parses specific sections or fields | Template expects specific headings; `applyTo` matches |
   | **Informational** | Low | Consumer mentions for context but doesn't depend on content | "See X for details" in References section |
   | **Cosmetic** | None | Consumer mentions in comments or version history | "Based on findings from X" |

   Weighted coupling replaces raw count for blast radius estimation. A single behavioral dependent carries more risk than five informational mentions. Transitive dependencies (A→B→C) are discovered through graph traversal of the dependency map, with impact attenuating by depth (direct = full weight, one-hop = half, two-hop = quarter).

The `pe-prompt-engineering-validation` and `pe-artifact-coherence-check` skills are the validation pattern library. Chain alignment checks (R-S5-chain-alignment) run here: every `enforces_rulesets` path resolves to an existing file; every specific `enforces_rules` ID is non-dangling (consulting the rename log). Category-based references (Level 1.5) are validated by checking that the referenced category ID exists in the target folder's structure index and maps to at least one existing file.

For artifacts with N-1 structural separation, diff classification is deterministic: parse labeled blocks, check which block type changed, route accordingly. For artifacts without N-1, classification falls back to LLM judgment — flagging a capability gap and recommending N-1 adoption.

For instruction files, assessment includes an instruction minimization check (R-S8-instruction-minimization): every rule is classified as testable/mechanical (appropriate) or behavioral/strategic (violation — must be moved to context file or agent body). Minimization violations are HIGH severity findings because they create silent collision risk when the PE engine is installed in repositories with their own instruction files.

For ecosystem changes, assessment includes a trust-and-fit evaluation (R-G2-trust-calibrated-adoption): does the external artifact come from a trustworthy source? Does it overlap with or improve upon an owned artifact? The answer determines the response strategy.

**Finding severity classification.** Each finding is classified by its impact on the artifact's declared purpose:

| Severity | Criteria | Examples |
|---|---|---|
| **CRITICAL** | Blocks artifact operation or self-update infrastructure integrity | Dangling reference in meta-infrastructure, broken hook wiring, missing required metadata, required category with no mapped files in structure index |
| **HIGH** | Breaks the artifact's declared contract (goal, scope, or boundaries) | Goal drift, scope violation, boundary breach, consumer-breaking schema change, instruction file contains behavioral/strategic rules (minimization violation), category-based reference resolves to non-existent category ID |
| **MEDIUM** | Affects a quality dimension without breaking the contract | Consistency issue, efficiency regression, coverage gap within declared scope |
| **LOW** | Cosmetic or optimization opportunity | Wording improvement, example update, token savings within budget |

Severity determines processing priority and autonomy routing. CRITICAL findings abort the cycle and escalate. HIGH findings require human approval. MEDIUM and LOW findings are routed through the standard autonomy gradient.

**Guidance adherence verification.** Beyond checking that references resolve (structural integrity), assessment includes spot-checking that artifacts **implement** the rules they reference. For each `📖` or category-based reference in an artifact, the Assess layer can verify:

1. **Rule presence**: Does the artifact's behavior or checklist include checks for each rule in the referenced guidance?
2. **Rule accuracy**: Does the artifact's implementation of the rule match the canonical definition?
3. **Rule completeness**: Does the artifact cover all rules in the referenced guidance, or only a subset?

Adherence verification is **expensive** (requires reading both the artifact and its referenced guidance) and is therefore scoped: full adherence checks run during scheduled reviews and ecosystem audits, not on every change. For individual changes, adherence is checked only for the specific guidance affected by the change.

**Example of adherence gap**: H12 defines "cross-reference integrity" as checking `📖` references, markdown links, and slash-command references. A validator that only checks `📖` references and markdown links has a reference to H12 (structural integrity ✅) but doesn't fully implement it (adherence gap ❌). This gap allowed 174 broken slash-command references to persist undetected.

**Quality chain assessment.** Assessment traces quality chains per the *Quality flows through dependencies* principle. The key operational detail: quality chain assessment uses R-S2's coupling types but inverts the direction — R-S2 propagates *impact* upward; R-S10 propagates *quality constraints* downward. The same dependency graph serves both analyses.

**Three review modes.** Assessment can operate in three modes, each answering a different question at a different scope and cost:

| Mode | Direction | Question it answers | Cost model |
|---|---|---|---|
| **Individual** | Artifact → self | "Is this artifact structurally and strategically correct?" | O(1) artifact + fixed context loading |
| **Dependency-aware** | Artifact → dependencies → back to artifact | "Does this artifact + its dependency chain produce reliable behavior?" | O(d) where d = dependency count |
| **Guidance-first** | Guidance → all consumers | "Do artifacts that load this guidance actually implement it?" | O(c) where c = consumer count |

No single mode catches everything. Each covers a distinct failure class:

| Scenario | Individual | Dependency-aware | Guidance-first |
|---|---|---|---|
| Agent has malformed YAML | ✅ | ✅ | ❌ |
| Agent loads context files that contradict each other | ❌ | ✅ | ❌ |
| Guidance defines 3 ref types but validator only checks 2 | ❌ | ✅ (reviewing validator) | ✅ (reviewing guidance's consumers) |
| Two context files define conflicting token budgets | ❌ | ✅ (if artifact loads both) | ✅ (reviewing either file) |
| 5 of 8 validators miss a rule from the priority matrix | ❌ | ❌ (only checks one) | ✅ (reviews all consumers) |
| Agent declares tools it never uses in process phases | ✅ | ✅ | ❌ |

**Why three modes, not one combined review?** Token cost. A dependency-aware review loads the target (~250 lines) + context files (3-5 × 200-500 lines) + instruction files (~150 lines). That's 2,000-4,000 tokens of source material. Adding guidance-first checks (enumerate all consumers of each context file) multiplies by the consumer count. This violates R-P2-decomposition. Different triggers also imply different modes: individual review for "validate this file," dependency-aware for "assess this subsystem's coherence," guidance-first for "audit this rule's implementation."

Review modes extend the modular invocation model: modular invocation determines *which artifacts* are targets and screened; review modes determine *what kind of assessment* runs during the deep pass. The screening mechanism is unchanged — review modes affect the deep pass only.

**Connection to R-S10-quality-propagation.** The three review modes are the operational realization of R-S10:
- Individual assessment verifies per-artifact quality
- Dependency-aware mode implements R-S10's bottom-up direction: assesses dependency quality first, then the target, ensuring the target's quality ceiling is known
- Guidance-first mode implements R-S10 in reverse: traces quality from guidance to consumers, producing an adherence matrix showing which consumers implement which rules
- Together, the three modes cover R-S10's full quality propagation path — bottom-up quality constraints AND top-down adherence verification

**Selective review dimensions.** Within each mode, assessment can target specific quality dimensions per invocation rather than applying every check. Each dimension is a named, independently invocable quality concern:

| Dimension group | What it covers | Example dimensions |
|---|---|---|
| **Structural** | Per-artifact structural correctness | YAML validity, reference integrity, metadata completeness, token budget compliance |
| **Strategic** | Vision alignment and design quality | Goal alignment, scope coherence, boundary enforcement, rationale currency |
| **Adherence** | Guidance implementation verification | Rule presence in consumers, rule accuracy, rule completeness |
| **Quality** | Guidance quality properties (all 6 prerequisite properties) | Clarity, non-redundancy, non-contradiction, completeness, **prioritization**, actionability |
| **Efficiency** | Resource optimization | Reference efficiency, handoff efficiency, processing efficiency, model routing adherence |
| **System** | Cross-artifact system properties | Cross-dependency coherence, layer correctness, coverage/redundancy analysis |

**Type-scoped dimension groups.** Not all dimensions apply to all artifact types. To avoid wasting processing on non-applicable dimensions, type-scoped groups select only the dimensions relevant to the target type:

| Group | Applicable to | Dimensions | Use when |
|---|---|---|---|
| `--dim context-full` | Context files | D1-D3, D6-D15, D17, D19, D22 (16 dims) | Full context file review — every applicable dimension |
| `--dim context-health` | Context files | D6-D12, D22 (8 dims) | Quick quality + staleness + organization check |

The `--dim context-full` group is the correct way to run a complete review of a context file. Using `--dim full` (all 27) on a context file wastes processing on 11 non-applicable dimensions (D4-D5, D16, D18, D20-D21, D23-D27).

Dimensions operate within the progressive depth model: lightweight dimensions (structural) cost near-zero; heavyweight dimensions (adherence, quality) cost proportionally more. The cost of any invocation is bounded by the dimensions selected, not the full dimension catalog.

**Cost gradient for dimensions:**
- Zero cost: cached results for unchanged artifacts (R-S3)
- Near-zero: deterministic structural dimensions (YAML parsing, reference resolution, hash comparison)
- Low: single-artifact strategic dimensions (goal/scope check against metadata)
- Medium: dependency-aware dimensions (load and compare guidance chains)
- High: guidance-first adherence (enumerate consumers, check rule implementation)

Selective dimensions implement R-P5-progressive-depth at the per-artifact level — R-P5 controls assessment depth across the artifact set (research → screening → deep); selective dimensions control assessment breadth within the deep pass. Together they bound cost in two orthogonal dimensions.

**Model routing within dimensions.** Different dimensions have different computational requirements. Applying R-P6-model-routing to dimensions:
- Structural dimensions → deterministic tools (YAML parsers, reference checkers, hash functions)
- Strategic dimensions → reasoning models (vision alignment requires judgment)
- Adherence dimensions → standard models for rule extraction, reasoning models for gap analysis
- Efficiency dimensions → deterministic tools for measurement, standard models for recommendations

Routing is a verifiable property: each dimension's processing should use the appropriate model tier. Misrouting (using a reasoning model for YAML parsing, or a deterministic tool for strategic analysis) is a finding.

**Cross-dependency coherence.** When an artifact loads multiple guidance files, assessment checks that the rules from those files don't contradict each other. Cross-dependency coherence extends adherence verification (which checks artifact → guidance) with guidance → guidance checks within the consumer's loaded set. This catches conflicts at review time rather than execution time.

**Cross-dependency coherence also applies to context files (peer review).** When the target artifact IS a context file, cross-coherence checks whether the rules in THIS context file contradict rules in peer context files that consumers load alongside it. The mechanism:
1. Identify which consumers load this context file (from `context_dependencies:`, `📖` references, category references)
2. For each consumer, identify what OTHER context files it loads
3. Check for contradictions between rules in THIS file and rules in those PEER files

This is the reverse direction of consumer cross-coherence (which asks "do MY loaded deps contradict each other?"). For context files, it asks "does MY content contradict content in files that get loaded alongside me?" This enables bottom-up detection of cross-context contradictions — detecting them from the guidance layer rather than waiting for a consumer review to reveal them.

**Guidance optimization findings.** Beyond individual artifact findings, assessment can produce system-level optimization recommendations:

| Finding type | What it detects | Example |
|---|---|---|
| **Rule fragmentation** | Same logical rule defined piecewise across multiple files | Tool count limit in three separate files — should be in one canonical location |
| **Priority gap** | Rules that could conflict but no explicit precedence in priority matrix | Two context files define boundary minimums with different thresholds — which takes precedence? |
| **Layer misplacement** | Content in the wrong artifact type | Behavioral guidance in an instruction file (minimization violation); testable rule in a context file |
| **Consumer burden** | Artifact loads more guidance than it needs | Agent references 6 context files but only uses rules from 2; the other 4 add token cost without behavioral impact |

Guidance optimization findings are MEDIUM severity — they don't break individual artifacts but degrade system-level efficiency and clarity. They are produced during dependency-aware and guidance-first reviews where the full guidance chain is visible, not during individual reviews.

### Propose

*Relies on: R-L2-self-correction, R-P3-structured-contracts, R-G1-autonomy-gradient*

The system knows what to suggest. When assessment reveals drift, the system generates a scoped correction plan using structured contracts (R-P3-structured-contracts). When assessment reveals a valuable ecosystem development, the system proposes:

- **Adopt.** When the external artifact has a well-defined scope, credible roadmap, and comes from a trustworthy source. The system proposes relying on it directly — replacing or deferring to the external artifact.
- **Incorporate.** When the external work contains valuable ideas but doesn't meet the adoption bar. The system proposes updating owned artifacts to absorb the useful parts.

Each proposed change carries an assessed risk level (R-G1-autonomy-gradient), which determines whether it proceeds autonomously or escalates to a human.

**Decomposition-first.** When a trigger fires involving a large change, the first action is to produce a decomposition plan: the smallest sequence of individually-verifiable steps. Steps are ordered bottom-up by tier — first changes land in high-numbered tiers (lowest blast radius), then propagate downward only after validation. This converts "large change → human approval marathon" into "large change → many small autonomous steps + a few human escalations at natural decision points."

**Risk-ordered execution.** When decomposition produces steps with different risk profiles, execute in risk order: non-regressive steps first (autonomous), potentially-regressive steps next (escalated as needed), optimization steps last (separate cycle). A step's execution is never blocked by a higher-risk step it doesn't depend on. This prevents a single human-approval escalation from blocking the entire decomposed sequence.

### Execute

*Relies on: R-L2-self-correction, R-P1-deterministic-processing, R-P2-decomposition, R-G1-autonomy-gradient*

The system knows how to act. Implementation is decomposed into focused units (R-P2-decomposition) to avoid context rot. Before committing any change, at least one independent validation pass runs (R-L2-self-correction).

Autonomy is earned per step, not per change-set. Each step applies risk-reduction techniques:

| Technique | What it does |
|---|---|
| **Static validation** | Schema validity, reference resolution (including category ID resolution via structure indexes), boundary preservation, token budgets, goal/scope preservation, N-1 block-level classification (deterministic when labels present; LLM fallback when absent). Also includes named check types: **consistency check** (no contradictions introduced, no unintended priority shifts), **efficiency check** (token delta within budget), and **behavioral compatibility check** (schema-level: YAML structure, section headings, and metadata field names unchanged or backward-compatible — deterministic and near-zero cost) |
| **Runtime validation** | Execute in a sandbox; compare outputs to pre-change baseline on a test set |
| **Modular integration** | Add capability as opt-in/optional first; default unchanged until observation confirms value |
| **Redundant processing** | Two independent LLM passes (R-L2-self-correction), or LLM + deterministic cross-check — agreement required before commit |
| **Reversibility guarantee** | Pre-change tier-scoped snapshot; rollback is one deterministic operation |

**Consistency refinement.** Priority shifts that serve the artifact's declared goal are non-breaking; shifts that oppose the goal or violate boundaries are breaking. This prevents the consistency check from over-blocking goal-aligned changes.

A step that passes enough risk-reduction techniques to drop below the autonomy threshold proceeds without human intervention. A step that doesn't escalates — only the individual step, not the whole change-set.

**Artifact-type requirements in Execute.** The risk-reduction techniques above apply to all changes. Additionally:

- **Guidance artifact changes**: Construction invariants from the Guidance quality by construction principle run as part of the change. When a rule is modified, all artifacts with behavioral coupling (R-S2) to that rule are flagged for re-assessment — their quality ceiling may have shifted.
- **Consumer artifact changes**: Adherence to loaded guidance is verified. If adherence gaps are found, they're fixed as part of the change — not deferred to a separate cycle.
- **N-1 prerequisite**: A guidance change to a rule-bearing section without N-1 labels is automatically classified as HIGH risk. N-1 adoption is recommended as a prerequisite for autonomous guidance changes.

**Rollback.** Before Execute: the system records the current group snapshot (version format: `YYYYMMDD.NN`, scoped to the affected tier). During Execute: individual artifacts receive SemVer bumps — patch = non-breaking fix; minor = additive; major = breaking. After Execute: the system validates the tier and creates a new snapshot. If regression is detected: the system reverts the tier slice automatically. Snapshot files live at `<state.path>/snapshots/<change_id>/`.

### Rollout phases

Autonomous execution is not a single switch — readiness is heterogeneous across artifact types. The rollout is phased.

| Phase | Autonomous execution enabled for | Precondition (new) | Why |
|---|---|---|---|
| **Phase 0 (new)** | Engine extraction and packaging: detect + assess in installable form | Engine/integration separation complete, configuration schema defined | Proves portability outside the Learning Hub before higher-autonomy execution |
| **Phase 1 (current)** | Templates, prompt snippets, hooks JSON | None — validators already exist | No rules-with-rationales to classify; purpose obvious from filename/schema |
| **Phase 2** | Skills content, agent tool/handoff lists | Outcome log (C-3) in place + guidance non-redundancy, non-contradiction, and prioritization verified for affected rules | Structured metadata only — diffs are trivially deterministic. Reversibility was the missing piece. |
| **Phase 3** | Agent bodies, context files, instruction files | Semantic metadata (R-S1-metadata-driven) + rule/rationale/example separation (N-1) in place + all 6 guidance quality properties above threshold for affected rules | These encode rules in prose. N-1 structural separation (R-P4) is the mechanism that enables deterministic diff classification — without it, every content diff requires LLM judgment and escalates by default. Structured metadata (R-S1) enables mechanical scope preservation. Together they unlock autonomous execution for rule-bearing artifacts. |
| **Phase 4** | Cross-tier refactors, rule renames, file moves | Chain alignment (R-S5-chain-alignment) + rename log operational | Changes ripple across the artifact graph. Resilient references are load-bearing. |

Preconditions are cumulative — Phase 3 also requires Phase 2's outcome log. *Current phase (mutable status):* **Phase 0** — portability extraction. Phases 1–4 each have a named precondition; the vision doesn't claim autonomy it hasn't yet earned.

Together, the four layers create a **continuous quality signal** rather than periodic snapshots. You don't wait for a manual audit to discover that half your agents reference a deprecated tool behavior.

## 📋 Design principles

Two principles are **foundational** — they enable the mechanisms that all other principles depend on:

1. **Staleness avoidance first** — ensures the system's own infrastructure is trustworthy before it acts on managed artifacts.
2. **Rules, rationales, and examples are separable** — enables deterministic classification that makes the autonomy gradient practical rather than theoretical.

The remaining principles operate within the foundation these two establish.

### Staleness avoidance first — automatic, event-driven, small-step

*Relies on: R-L4-external-knowledge, R-S1-metadata-driven, R-G1-autonomy-gradient, R-L2-self-correction, R-P2-decomposition, R-G3-progressive-learning*

Stale managed artifacts degrade specific capabilities. Stale self-update infrastructure silently degrades every subsequent decision and the system's ability to detect its own degradation. The self-update infrastructure MUST be maintained at higher freshness than the artifacts it manages.

**Rules:**

1. **Meta-infrastructure has the shortest validation cadence.** Tier 0/governance artifacts, all meta-agents, all hooks, and all skills used by the update loop are validated on every self-update cycle. R-S3-validation-caching does NOT apply to these components — never skip, never cache.
2. **Event-driven triggers.** Every defined external signal (model releases, platform releases, ecosystem publications, internal edits) wires to a scoped pipeline entry point. Missing trigger wiring is a CRITICAL finding.
3. **Decompose-first.** When a trigger fires, the first action is a decomposition plan: the smallest sequence of individually-verifiable steps. The plan itself is validated before execution begins.
4. **Earn autonomy per step.** Each step is classified using the best-available risk-reduction techniques. Steps below the threshold execute; steps above escalate — only the individual step, not the whole integration.
5. **Infrastructure integrity is a hard gate.** Before any managed-artifact work begins, run deterministic integrity checks against the self-update infrastructure. Any failure aborts the cycle and escalates.
6. **No silent fallback.** Suspect meta-state never proceeds under degraded operation. "Missed a review opportunity" beats "applied a wrong change based on stale self-knowledge."
7. **Dangling references in self-update infrastructure are CRITICAL** (vs. HIGH in managed artifacts). Self-update infrastructure refers to the hooks, meta-agents, outcome log, rename log, and snapshot files.

*Subsumes: L-2 (amplified risk for self-updates). Extends: B-3 (integrity pre-step as hard gate). Governs: R-S3-validation-caching carve-out for meta components.*

### Rules, rationales, and examples are separable

*Relies on: R-S4-role-declaration, R-P1-deterministic-processing*

Most context and instruction files mix three content types in the same section:

- **Rules** (REQUIRED) — enforceable statements (MUST/NEVER/ALWAYS). A diff here is breaking for consumers.
- **Rationales** (OPTIONAL, recommended unless the reason is trivially self-evident) — *why* the rule exists. A diff here is non-breaking.
- **Examples** (OPTIONAL — include only when the rule is non-obvious or misapplication is a known failure mode) — illustrative prose or code. A diff here is non-breaking unless the example is referenced as canonical.

Every rule-bearing section uses this structural convention:

```markdown
### [Rule name]

**Rule** (REQUIRED):
- MUST/NEVER/ALWAYS ...

**Rationale** (OPTIONAL):
- ...

**Example** (OPTIONAL):
- ...
```

Without this separation, every diff requires LLM judgment to classify (breaking vs. non-breaking). With it, classification is deterministic and cheap.

**Economy matters.** Optional blocks consume tokens on every consumer load. Every added rationale or example should pay for its token cost by reducing misapplication risk or eliminating LLM judgment on diffs.

*Applies to: context files, instruction files, agent bodies. Connects to: breaking/non-breaking classification in The goal; outcome log `classification` field.*

This separation is the structural counterpart of the metadata-driven principle — metadata makes *artifact-level* classification deterministic; N-1 makes *content-level* classification deterministic. Together, they enable the earned-autonomy-per-step model.

### Human governance, autonomous execution

*Relies on: R-L2-self-correction, R-G1-autonomy-gradient*

Humans define the vision, the goals, and the governing rules. The system never changes its own purpose or principles without explicit human direction.

Implementation is a different matter. Once the vision and rules are established, the system executes improvements autonomously — provided it can validate that risk is low enough. Autonomy level follows the gradient defined in The goal (canonical table is there; this section cross-references it).

The human's role shifts to: defining and refining the vision; setting risk thresholds; reviewing escalations when the system can't confidently validate; and auditing outcomes periodically.

### Minimal disruption

*Relies on: R-G1-autonomy-gradient*

A self-updating system that constantly demands attention is worse than manual audits. The system should be quiet when everything is healthy and specific when something isn't. Notifications should be actionable, not informational.

### Scoped over exhaustive

*Relies on: R-S2-dependency-graph, R-S6-tier-blast-radius, R-S1-metadata-driven*

A full system audit is expensive. Most changes affect a small subset of artifacts. The system identifies the blast radius of a change — using tier (R-S6-tier-blast-radius) as the first sort key and dependency count (R-S2-dependency-graph) as secondary — and checks only what's affected.

### Evidence-based

*Relies on: R-L1-llm-reasoning, R-P3-structured-contracts*

Every proposed change must cite what triggered it, what assessment found, and why the proposed correction is appropriate. "File X is outdated" isn't enough. "File X references Copilot Spaces as 'public preview' but the April 2026 release notes announce GA availability" is.

### Trust-based adoption

*Relies on: R-G2-trust-calibrated-adoption*

Not all external work deserves equal trust:

- **Trustworthy sources** (Microsoft, OpenAI, Anthropic, Google, Meta, recognized domain specialists) — adopt directly when scope and quality align.
- **Valuable but unvetted sources** — incorporate ideas into owned artifacts after evaluation and adaptation.
- **Unknown sources** — don't trigger adoption or incorporation proposals autonomously. When valuable ideas come from unvetted sources, extract the core concept into fully-owned content rather than creating a dependency.

Trust is earned through consistency: clear scope, maintained roadmap, demonstrated expertise.

### Graceful degradation

*Relies on: R-S3-validation-caching, R-P1-deterministic-processing*

If the system can't determine whether an artifact is affected by a change, it flags uncertainty rather than guessing. If a trigger source becomes unavailable, the system falls back to time-based checks. The worst outcome should be "missed a review opportunity," never "applied an incorrect change."

For self-update infrastructure specifically: suspect meta-state fails closed — no degraded operation, direct escalation.

### Deterministic where possible

*Relies on: R-P1-deterministic-processing, R-P3-structured-contracts*

When an operation has a predictable outcome — parsing YAML, checking file existence, comparing timestamps, counting dependencies, computing content hashes — use deterministic tools instead of LLM processing. This principle applies across all four vision layers.

### Portable by design

*Relies on: R-P1-deterministic-processing, R-S1-metadata-driven, R-S7-portable-packaging*

Separate **engine** from **integration**. The engine holds reusable logic; integration connects that logic to repository-local artifacts and optional execution hosts. The engine discovers repository conventions and avoids imposing Learning Hub-specific structure. Integration should remain thin, incremental, and optional in depth.

**Artifact namespacing (REQUIRED).** All PE artifacts (agents, prompts, skills) MUST use a namespace prefix (e.g., `pe-`) that distinguishes them from domain-specific artifacts. Without namespacing, generic names like `pe-prompt-researcher` or `pe-agent-builder` clash when the PE system is ported to a repository that has its own agents with those names. Namespacing is a prerequisite for portability — it ensures PE artifacts can coexist with any domain's artifacts without naming conflicts.

**Instruction minimization (REQUIRED for portability).** See *Instruction minimization* principle (R-S8) — required to prevent silent collisions when PE instruction files auto-inject alongside a target repository's own instruction files.

**Category-based references (RECOMMENDED for cross-artifact references).** When artifacts reference specific context files, they should use functional category IDs defined in the folder's structure index rather than filenames. Category IDs are stable semantic identifiers that describe the capability needed (e.g., `validation-rules`, `token-optimization`) and map to specific files in the structure index. This decouples consumer artifacts from the physical file layout — renames, renumbering, and file splits only require updating the structure index, not every consumer.

### Discovery over prescription

*Relies on: R-S1-metadata-driven, R-P1-deterministic-processing, R-S7-portable-packaging*

Discover PE structure from what already exists. Parse file types, frontmatter fields, references, and conventions. When rich metadata exists, perform precise impact analysis. When metadata is incomplete, run in structural-only mode and report capability-level uncertainty explicitly, with concrete recommendations to improve metadata quality.

### Quality flows through dependencies

*Relies on: R-S2-dependency-graph, R-S10-quality-propagation*

An artifact's achievable quality is bounded by the quality of the artifacts it depends on. A perfectly written prompt that loads ambiguous context will produce ambiguous output. A well-structured agent that enforces unclear rules will enforce them inconsistently.

Quality improvement is most effective bottom-up: fix the context file first, then the agent that loads it, then the prompt that invokes the agent. Top-down improvement hits a ceiling set by the weakest dependency.

When a dependency improves, reassess its dependents — they may now achieve higher quality without any changes of their own. When a dependency degrades, all dependents are potentially affected — regardless of their individual quality.

**Operational realization through review modes.** This principle is implemented through the three review modes in the Assess layer: individual assessment checks per-artifact quality, dependency-aware mode traces quality bottom-up through dependency chains, and guidance-first mode traces adherence top-down from guidance to consumers. Together the three modes cover the full quality propagation path that this principle requires.

### Guidance quality by construction

*Relies on: R-S9-guidance-quality, R-S10-quality-propagation, R-L2-self-correction*

Guidance artifacts (context files, and guidance-bearing sections of instruction files and agent bodies) set the quality ceiling for every artifact that depends on them. Their update path carries the 6 guidance quality properties as **construction constraints** — satisfied during creation/update, not only verified after the fact.

**Construction invariants** — the 6 guidance quality properties (see Goal § Guidance quality as prerequisite for definitions) enforced during creation/update: rewrite-before-commit (clarity, actionability), reference-canonical (non-redundancy), resolve-before-commit (non-contradiction), flag-gaps (completeness), declare-priority (prioritization).

**Construction invariant → dimension mapping.** Each construction invariant maps to a review dimension, enabling the create/update workflow to report which invariant was enforced and which dimension it satisfies:

| Construction invariant | Dimension | Verification method |
|---|---|---|
| rewrite-before-commit | D9 (clarity) + D11 (actionability) | LLM agreement test + boolean-check translation |
| reference-canonical | D7 (non-redundancy) | Grep for duplicate rule text across context files |
| resolve-before-commit | D6 (consistency) | Check against direct dependencies |
| resolve-peer-coherence | D17 (cross-coherence, peer mode) | Check against peer context files loaded by same consumers |
| flag-gaps | D10 (completeness) | Scan for uncovered agent/prompt behaviors |
| declare-priority | D8 (prioritization) | Check priority matrix coverage |

The 4th invariant (resolve-peer-coherence) extends resolve-before-commit to include peer context files — checking that new or modified rules don't contradict rules in files that consumers load alongside this one.

**Assessment cadence** — the 6 properties are assessed at three tiers of cost:
- **On every guidance change** (cheap): non-redundancy scan, non-contradiction check against direct dependencies, actionability self-test. These are the construction invariants.
- **On scheduled reviews** (medium): clarity agreement test (dual-pass), completeness gap scan across the artifact set, prioritization matrix coverage.
- **At phase transitions** (full): all 6 properties at full depth, qualifying the artifact set for the next autonomy phase.

**Why construction, not just validation**: A clarity issue caught during construction costs one rewrite; the same issue caught during validation costs a finding + a fix cycle + a re-validation. For the system's own guidance (meta-infrastructure), construction constraints are especially critical — a quality issue in meta-guidance degrades every subsequent decision.

Consumer artifacts (prompts, agents that consume guidance) don't carry construction invariants. They carry **adherence requirements** — verified against already-validated guidance. This separation keeps guidance update paths focused and consumer update paths lightweight.

**Context file change routing.** Context files carry the lowest tier prefix (Tier 00) — the highest blast radius per R-S6. Combined with their position as the foundation layer (R-S10), context file changes require **deeper validation** than other artifact types. However, the autonomy level follows the same risk-calibrated gradient as all artifacts: non-breaking changes that pass validation proceed autonomously; breaking changes escalate. The foundation-layer position increases validation depth, not the autonomy threshold for validated non-breaking changes.

**Non-breaking classification for context files.** A context file change is non-breaking when it preserves the artifact's declared goal, scope, and boundaries AND the new content is consistent with existing guidance and priorities. The pre-change guard (§ Metadata-guarded changes) is the primary mechanism — if the guard passes, the change doesn't contradict the artifact's declared contract. Additional validation through construction invariants and peer cross-coherence (D17) further reduces risk. When both pass, the breaking risk is LOW regardless of tier.

**Decomposition enables autonomy.** Large context file changes that would be HIGH risk as a single change can be decomposed into smaller independently-verifiable steps, each of which may classify as non-breaking. This is a direct application of the Decompose-first principle — the system should split changes to maximize the autonomous portion.

| Change type | Autonomy level | Pre-conditions for autonomous execution |
|---|---|---|
| Metadata fixes (version, last_updated) | Autonomous | Deterministic, no content change |
| Reference fixes (broken links, stale paths) | Autonomous | Deterministic, no guidance change |
| Wording improvements within scope that preserve the same guidance and priorities | Autonomous with notification | Pre-change guard passes + construction invariants pass + peer coherence (D17) passes |
| Non-redundancy refactoring (consolidating duplicate rules to canonical location) | Autonomous with notification | Pre-change guard passes + no rule lost (D7 non-redundancy improves) + all consumers still have access to the rule |
| Adding new coverage within declared scope (e.g., new model patterns when scope covers model optimization) | Autonomous with notification | Pre-change guard passes + new content doesn't contradict existing rules (D6) + peer coherence passes |
| Rule modification that changes behavioral guidance | Human approval required | Breaking for consumers — even if metadata preserved, behavioral impact requires human judgment |
| Scope expansion (new topics not in scope.covers) | Human approval required | Changes what consumers can rely on |
| Scope contraction (removing coverage) | Human approval required | May break consumer adherence |
| File split, merge, or reorganization | Human approval required | Affects category mapping, consumer references |

**Validation-reduced escalation.** When a change initially classifies as potentially breaking, successful validation can reduce the autonomy threshold. If the pre-change guard passes, construction invariants pass, peer coherence passes, AND the change can be validated against consumer test scenarios (embedded test cases in agents/prompts that load this guidance), the breaking risk is reduced. A change that passes all four validation layers can be reclassified from "Human approval required" to "Autonomous with notification" — the validation evidence substitutes for human judgment on the breaking question.

After ANY context file change that modifies rules (not just metadata), the system SHOULD flag affected consumers for re-assessment. This is the operational realization of R-S10: when the foundation changes, the quality ceiling shifts for everything above. This consumer flagging is the foundation-layer safeguard — it operates regardless of whether the context file change itself was autonomous or human-approved.

### Instruction minimization

*Relies on: R-S7-portable-packaging, R-P1-deterministic-processing*

Instruction files auto-inject via `applyTo` patterns with no precedence mechanism. When multiple PE systems coexist, overlapping patterns cause silent rule collisions. To prevent this:

**Rule:** Instruction files MUST contain only testable, mechanical rules — rules evaluable with a boolean pass/fail without LLM judgment. Behavioral, strategic, and interpretation-dependent rules MUST be placed in context files or agent bodies where consumers explicitly choose what to load.

**Rationale:** Testable rules either agree (identical checks), contradict on a measurable dimension (resolvable by choosing the stricter value), or are additive (non-overlapping checks). Behavioral rules are inherently irreconcilable when doubled — "warm tone" plus "professional tone" has no deterministic resolution.

**Example:** ✅ "YAML MUST have `goal:` field" (testable). ❌ "Write in a warm, conversational tone" (behavioral — belongs in context file).

### Metadata-driven

*Relies on: R-S1-metadata-driven, R-S4-role-declaration*

The system's ability to scope its work, skip redundant checks, and evaluate artifact effectiveness depends on machine-readable metadata.

Metadata serves **two audiences**:

1. **Validation tooling** (pe-meta): deterministic change classification, drift detection, scope preservation — consumed during modification workflows (pre-change guard, post-change reconciliation).
2. **Executing model** (runtime): self-constraint enforcement — the model grounds its behavior on declared goal, scope, and boundaries during execution. This is consumed during execution workflows (the model running the prompt or agent).

Both audiences consume the same YAML fields. The difference is timing: validation happens before/after changes; runtime grounding happens during execution. Without runtime grounding, metadata is passive documentation that only pe-meta tooling reads — the executing model can silently drift outside its declared scope.

Every artifact should declare:

- **`goal:`** — single sentence: the one outcome the artifact exists to achieve
- **`scope: {covers: [...], excludes: [...]}`** — what topics the artifact covers and explicitly excludes
- **`boundaries:`** — constraints the artifact operates within (max tokens, single-source-of-truth for, etc.)
- **`rationales:`** — lightweight list of **artifact-specific** design constraints that explain why key decisions in THIS artifact must be preserved across updates. NOT the same as the vision's system-level rationales (R-L1, R-P1, etc.) — those justify the architecture; these justify individual artifact choices
- **`version:`** — SemVer for individual artifact versioning (patch = non-breaking fix, minor = additive, major = breaking)
- **`required_categories:`** (structure indexes only) — list of functional category IDs that MUST remain mapped to at least one existing file. Refactoring operations that would leave a required category unmapped are blocked by the pre-change guard. This extends R-S5 chain alignment to make category contracts enforceable.

Placement: top YAML for context/instruction files; `agent_metadata:` for agents; `template_metadata:` for templates.

`rationales:` is **artifact-level constraint metadata**: it preserves decision memory for THAT specific artifact, not the system architecture. Without it, an artifact can remain in-scope while silently losing important safeguards.

**How artifact `rationales:` are used:**

1. **Decision memory** — When someone modifies this artifact later, the rationales explain WHY specific choices were made. Without them, a reasonable-looking change can silently undo a deliberate decision.
2. **Pre-change guard input** — The metadata-guarded change protocol checks: "does this proposed change invalidate any declared rationale?" If yes, a replacement rationale is required — you can't delete a design constraint without explaining why it's no longer valid.
3. **Scope-preserving safety net** — A change that preserves `goal:` and `scope:` but violates a `rationale:` is routed above baseline autonomy. This catches harmful changes that pass structural checks but undo load-bearing decisions.

Phase gating for `rationales:`:

- Phase 1: recommended, optional
- Phase 2+: required for artifacts eligible for autonomous semantic edits
- Phase 3+: optional upgrade to structured rationale objects where scale justifies it

Routing rule: if a change preserves `goal/scope` but violates a declared rationale, route above baseline autonomy (notification or escalation). If a rationale is replaced, the same change must include replacement rationale text.

Metadata makes artifact-level classification deterministic; N-1 structural separation makes content-level classification deterministic. Together they form the two-layer mechanism the autonomy gradient depends on. See The rationale § Change classification mechanics for the three classification tiers and confidence model.

### Metadata-guarded changes

*Relies on: R-S1-metadata-driven, R-S4-role-declaration, R-L2-self-correction*

Every artifact change — whether autonomous or human-initiated — is validated against the artifact's declared metadata **before** and **after** the change.

**Pre-change guard (default: block).** Before applying a change, compare its effect against the artifact's `goal`, `scope`, `boundaries`, and `rationales`. If the change contradicts any of these declarations, it is **blocked by default**. The change may proceed only with explicit human override.

**Post-change reconciliation (mandatory).** After every change is applied, the artifact's metadata MUST be reviewed and updated to accurately reflect the new state. This includes:
- `goal:` — still accurate? If the change expanded or shifted purpose, update.
- `scope.covers:` — still complete? If the change added coverage, add the topic. If it removed coverage, flag as potential regression.
- `scope.excludes:` — still accurate? Exclusions may need updating.
- `boundaries:` — still enforced? New constraints may apply; old ones may be relaxed.
- `rationales:` — still valid? If the change invalidated a rationale, it MUST be replaced with the new rationale.
- `version:` — bump per SemVer (patch/minor/major based on breaking classification).

**Why this matters.** Without pre-change guards, drift accumulates silently — an artifact's content diverges from its declared purpose while the metadata still claims the old purpose. Without post-change reconciliation, metadata becomes stale — a secondary form of drift that degrades the system's ability to classify future changes. Together, these two guards create a **closed feedback loop** between content and metadata that prevents both content drift and metadata staleness.

**Refactoring protection.** During refactoring (restructuring, splitting, merging artifacts), pre-change guards detect two types of coverage regression:

1. **Scope regression** — if a refactored artifact covers less than its predecessor's declared scope, the gap is flagged.
2. **Category regression** — if a refactored artifact is listed in a structure index's functional categories, the pre-change guard verifies that every affected category will still map to at least one valid file after the change. Missing category coverage is flagged as a regression.

Both types prevent silent capability loss — scope regression protects individual artifact coverage; category regression protects the stable reference contracts that consumers depend on.

| Guard | When | Default action | Override |
|---|---|---|---|
| Pre-change: contradicts goal | Before apply | **BLOCK** | Explicit human approval |
| Pre-change: contradicts scope | Before apply | **BLOCK** | Explicit human approval |
| Pre-change: contradicts boundary | Before apply | **BLOCK** | Explicit human approval |
| Pre-change: contradicts rationale | Before apply | **ESCALATE** (notification or approval) | Replacement rationale required |
| Post-change: metadata reconciliation | After apply | **MANDATORY** — always runs | Cannot be skipped |

**Audit trail.** All autonomous changes produce a structured log entry at `<state.path>/outcome-log.jsonl`. Every entry records: change ID, artifact path, artifact tier, pre/post content hashes, pre/post SemVer versions, breaking/non-breaking classification, confidence score, autonomy level, trigger type and source, validations passed/failed, and outcome (updated after observation window). A companion rename log at `<state.path>/rename-log.jsonl` records every rule/file rename with `old → new` mapping for automated reference rewrite.

Human rejections, including the stated reason, feed back into the outcome log as negative signal for threshold tuning (R-G3-progressive-learning).

### Runtime grounding

*Relies on: R-S1-metadata-driven, R-S4-role-declaration, R-P1-deterministic-processing*

YAML metadata is not passive documentation — it's an enforceable execution contract. While *Metadata-guarded changes* protects artifacts during modification, **runtime grounding** protects artifacts during execution.

**The problem it solves.** When a model executes a prompt or agent, it receives the full file (YAML + body). The model CAN read the YAML, but without explicit instruction to enforce those fields, it may drift outside its declared scope — silently processing requests it should reject, ignoring boundaries it should enforce. This is a distinct failure mode from modification drift: the artifact doesn't change, but the model's behavior diverges from the artifact's intent.

**The requirement.** Every artifact that is independently executed (agents, prompts) MUST include a grounding mechanism in its body that bridges declarative YAML metadata to imperative runtime behavior:

| Artifact type | Grounding mechanism | Notes |
|---|---|---|
| **Agent** | Three-tier boundaries (Always Do / Ask First / Never Do) MUST reflect all YAML `boundaries:` entries | Existing pattern — ensure alignment |
| **Prompt** | Workflow MUST include scope enforcement: verify request falls within `scope.covers` before proceeding; reject requests matching `scope.excludes` | Lightweight — one process step |
| **Context/Instruction/Skill/Template** | N/A — not independently executed | No grounding needed |

**Alignment rule.** Body-level enforcement content MUST be consistent with YAML metadata. A boundary that exists in YAML but has no corresponding enforcement in the body workflow is a **runtime grounding violation** — the metadata claims a constraint that the executing model has no instruction to enforce.

**This does NOT mean duplicating metadata in the body.** The body references metadata constraints operationally (e.g., "verify request falls within declared scope") — it doesn't restate the full scope list. YAML remains the single source of truth; the body is the enforcement mechanism.

**Infrastructure folder layout (default repo-local mode).** The self-update operational state uses conflict-safe namespacing:

```
.copilot/state/<engine_id>/<domain_id>/<instance_id>/
  outcome-log.jsonl        # Append-only change event log
  rename-log.jsonl         # Append-only rename map for chain alignment
  snapshots/<change_id>/   # Pre-change content snapshots for rollback
  triggers/                # Trigger definitions and last-fired timestamps
```

Minimum uniqueness requirement: `<engine_id>` + `<domain_id>`.

Backward compatibility: legacy `.copilot/self-update/` may be read during migration, but new writes target the namespaced `state.path` only.

Portable deployments can keep this state in engine-managed storage outside the repository and project only audit artifacts into the repo when needed.

### Iteration budget

*Relies on: R-P2-decomposition, R-G1-autonomy-gradient*

Each self-update cycle has a configurable maximum number of autonomous changes (default: 10). When the budget is reached, remaining proposed changes are queued for the next cycle. This bounds both cost and blast radius per cycle — a runaway loop can't consume unbounded tokens or make unbounded changes before a human reviews the outcomes. The budget applies to autonomous changes only; human-approved changes don't count against it.

### Loop stability

*Relies on: R-L2-self-correction, R-G1-autonomy-gradient, R-G3-progressive-learning*

The self-update loop's goal is **convergence toward a fixed point**: a state where, absent new external triggers, the system proposes no further changes. Every cycle should bring the artifact set closer to this fixed point, not farther from it.

**Convergence rule.** If successive cycles produce increasing numbers of proposed changes without new triggers, the loop is diverging. Divergence is a signal to freeze autonomous execution and escalate for human review.

**Oscillation rule.** If the same artifact is changed and then reverted (or changed in alternating directions) across consecutive cycles, the loop is oscillating. Oscillation is a signal to freeze the affected artifact and escalate — the system lacks sufficient information to make a stable decision.

**Phase note.** The specific mechanisms for detecting convergence and oscillation (change-volume slopes, per-artifact history tracking, trajectory analysis) are Phase 2–3 concerns. At Phase 1 (manual invocation), these rules serve as design intent — they inform how the loop is built, even before automated detection exists.

### Execution depth preserves quality objective

*Relies on: R-P7-capability-applicability, R-G1-autonomy-gradient*

Process optimization (pipeline skipping via `--skip`, dimension scoping via `--dim`, target narrowing via `--scope`) can change **execution depth** — how much processing occurs — but MUST NOT change the **quality objective** or **scope intent** of the invocation.

**Rule:** A `--skip` flag or a narrowed `--dim` selection reduces processing effort while targeting the same quality standard for the dimensions that remain in scope. If a process change would lower quality expectations or narrow scope beyond what the user specified, it must be expressed as explicit `--dim` or `--scope` options — never hidden inside a convenience abstraction.

**Example:** Invoking `/pe-meta-update --mode apply --dim optimize --skip research,structure,consistency` skips three pipeline stages (execution depth reduction) but still applies the full quality standard to all efficiency-relevant dimensions across the selected scope. It does NOT lower the quality bar — `--dim optimize` narrows WHICH dimensions are assessed; `--skip` narrows HOW MUCH processing runs. The two controls compose; neither weakens the quality target.

### Scope and depth compose orthogonally

*Relies on: R-P7-capability-applicability, R-S2-dependency-graph*

The `--scope` and `--deps` options address two independent concerns:

- **`--scope`** = "what artifacts to target" — a filter that selects the initial set of review targets by artifact type or file path.
- **`--deps`** = "how deep to traverse" — a traversal depth control that determines whether dependencies of the target set are included in assessment.

These compose orthogonally: `--scope context --deps direct` means "target context files, and include their direct dependents in assessment." Neither option implies or overrides the other. Both default independently when omitted (scope defaults to all applicable types; deps defaults to the command's natural traversal depth).

**Why orthogonality matters:** Without separation, scope and depth conflate into complex combined options (e.g., `--mode dep-aware-full` which mixed "assess dependencies" with "full traversal depth"). Orthogonal composition enables fine-grained control through combination rather than proliferation of compound modes.

## 🎛️ Command families and option model

*This section defines the canonical command families, their shared option model, and the normalization principles that govern option applicability across the PE-meta command set.*

### Command families

The PE-meta system organizes commands into families based on their mutation posture and operational purpose:

| Family | Mutation posture | Purpose | Example commands |
|---|---|---|---|
| **Creation** | Write (new artifact) | Research from scratch, create artifacts at best quality (may reuse existing as foundations) | `pe-meta-create-*` |
| **Review** | Assessment-first (write via `--mode apply`) | Understand existing artifacts, improve structure and quality to reach the same target as Creation | `pe-meta-review` |
| **Guidance-first** | Assessment-first (write via `--mode apply`) | Trace quality from guidance to consumers, improve adherence | `pe-meta-adherence` |
| **Scheduled** | Autonomous low-risk apply | Periodic detection + autonomous fix for low-risk findings | `pe-meta-scheduled-review` |
| **Update** | Write (modify existing) | Apply assessed improvements to existing artifacts | `pe-meta-update` |
| **Release-diff** | Write (modify existing) | Reconcile artifacts with external platform changes | `pe-meta-release-diff` |

**Create/Review parity principle:** Create and Review commands achieve the same quality goal. They differ in starting point and process:

- **Create** commands focus on research and investigation from scratch, producing artifacts at best quality (possibly reusing existing artifacts as foundations)
- **Review** commands focus on understanding existing artifacts and improving their structure and quality to reach the same target

The confirmation gate is **risk-proportional, not command-proportional**. A Review command that identifies a low-risk improvement applies it autonomously (or with lightweight notification), exactly as an Update command would. The command family determines the *default posture* (assess-first vs. write-first), not the *permission ceiling*.

### Option taxonomy

Five canonical option classes provide consistent control across all command families:

| Option | Purpose | Value space | Default when omitted |
|---|---|---|---|
| **`--dim <group\|D#>`** | Dimension scoping — which quality dimensions to assess or enforce | `full`, `structural`, `quality`, `strategic`, `freshness`, `efficiency`, `adherence`, `reliability`, `context-full`, `context-health`, `model`, `optimize`, or specific `D1`–`D35` (legacy alias `robustness` accepted for one release with deprecation notice; resolves to `adherence`) | `full` (all applicable to target type) |
| **`--deps none\|direct\|full\|<N>`** | Dependency traversal depth — whether to include dependencies in assessment | `none` (target only), `direct` (one hop), `full` (transitive), or numeric depth `<N>` | Command-specific natural depth |
| **`--scope <type\|path>`** | Target scoping — which artifacts to assess or modify | Artifact type (`context`, `agents`, `prompts`, `instructions`, `skills`, `hooks`, `snippets`, `templates`) or file path | All applicable artifact types |
| **`--mode plan\|apply`** | Execution depth — assess only or assess and implement | `plan` (findings only), `apply` (findings + changes) | `apply` for all families where `--mode` is applicable — risk classification governs what is applied autonomously vs. proposed; use `--mode plan` to explicitly request assessment-only output |
| **`--skip <stage>[,<stage>...]`** | Pipeline stage skipping — omit specific processing phases | `research`, `external`, `structure`, `consistency`, `content` | None skipped |

#### Option applicability matrix

Not every option applies to every command family. Applicability is governed by the capability-based applicability principle (R-P7):

| Option | Creation | Review | Guidance-first | Scheduled | Update | Release-diff |
|---|---|---|---|---|---|---|
| `--dim` | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| `--deps` | ❌ | ✅ | ✅ | ✅ | ✅ | ❌ |
| `--scope` | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| `--mode` | ❌ | ✅ | ✅ | ❌ | ❌ | ❌ |
| `--skip` | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |

**Key design decisions:**

- **`--mode` is NOT applicable to Scheduled**: Scheduled review uses `--deps` for traversal depth control and an internal rotation mechanism for mode cycling. See § Mode elimination from scheduled review.
- **`--mode` is NOT applicable to Update/Release-diff**: These are inherently write operations — `apply` is their natural and only mode.
- **`--deps` is NOT applicable to Release-diff**: External platform diffs determine scope; dependency traversal is not meaningful.

#### Mode elimination from scheduled review

The scheduled review command historically used a `--mode` parameter with values `individual|dep-aware-full|dep-aware-shallow|guidance-first`. This created semantic overloading — `--mode` meant "execution depth" in other commands but "traversal depth + rotation behavior" in scheduled review.

**Resolution:** Scheduled review's modes map directly to `--deps` values plus an internal rotation mechanism:

| Legacy mode | Replacement | Behavior |
|---|---|---|
| `--mode individual` | `--deps none` | Assess each target in isolation |
| `--mode dep-aware-full` | `--deps full` | Include transitive dependencies |
| `--mode dep-aware-shallow` | `--deps direct` | Include direct dependencies only |
| `--mode guidance-first` | Internal rotation | Auto-delegates to `/pe-meta-adherence` |

**Auto-rotation cycle:** When `--deps` is omitted, scheduled review automatically cycles through depth levels:

```
--deps none → --deps full → --deps direct → adherence-delegation → repeat
```

This cycle ensures comprehensive coverage across invocations without requiring the user to manually specify traversal depth each time. The adherence-delegation step invokes the guidance-first family (`/pe-meta-adherence`) to cover the reverse-direction quality trace.

### Explicit option discipline

Commands accept only the canonical option classes (`--dim`, `--deps`, `--scope`, `--mode`, `--skip`). Convenience presets are not provided: every invocation MUST state its execution depth (`--mode`), pipeline stages (`--skip`), and target dimensions/scope explicitly. The default values defined in the [option taxonomy](#option-taxonomy) apply when an option is omitted.

**Rationale.** A self-documenting command line is more valuable than a memorable shortcut. Named presets force the reader to look up an expansion table to predict behavior; explicit options encode the behavior on the command line itself. With only five orthogonal option classes, the taxonomy is small enough that explicit invocation is the better default.

**Worked equivalences** (for migration from earlier vision drafts that defined `healthcheck` / `performancecheck` / `fullcheck` preset aliases):

| Legacy preset | Canonical replacement |
|---|---|
| `healthcheck` | `--mode plan --skip research` |
| `performancecheck` | `--mode apply --dim optimize --skip research,structure,consistency` |
| `fullcheck` | `--mode apply` (or omit `--mode` entirely — `apply` is the default) |

**Composition rule.** Explicit user flags always win. When a `--skip` value is supplied on top of one of the legacy expansions during migration, lists are merged with deduplication (e.g., legacy `healthcheck --skip research` resolves to `--mode plan --skip research` with `research` appearing once).

**UX safety net.** Natural-language intent words (`review`, `audit`, `optimize`, `dry-run`) MAY be resolved by the orchestrator prompt into canonical option combinations, but the resolved combination MUST be echoed back to the user as a canonical invocation string before execution. No hidden expansions, no named presets in the parser's accepted vocabulary.

## 🔍 What "self-updating" means (and what it doesn't)

**Maintenance scope.** Self-update maintains and evolves *existing* artifacts — updating, restructuring, merging, splitting, or deprecating them. Creation of entirely new artifact categories (those that don't replace or extend an existing one) is out of scope and flagged as a gap for human action.

**It means:**

- Watching for platform, internal, and ecosystem signals that artifacts may need attention
- Mapping signals to specific artifacts via tier + dependency analysis
- Running targeted validation checks on affected artifacts
- Evaluating external artifacts for adoption or incorporation based on trust and fit
- Generating human-readable correction proposals with evidence
- Executing low-risk changes autonomously with post-execution notification
- Escalating high-risk or uncertain changes to a human with full context
- Tracking outcomes to tune autonomy thresholds

**It doesn't mean:**

- Changing the system's vision, goals, or governing rules without human direction
- Proceeding when risk assessment indicates high uncertainty
- Skipping validation to save time or tokens
- Generating entirely new artifact categories from scratch
- Replacing human judgment for architectural or strategic decisions
- Automatically adopting every external artifact that overlaps with an owned one
- Guaranteeing zero drift (some lag between change and detection is expected and acceptable)
- Executing changes that haven't passed at least one independent validation pass

## ⚠️ Risks and tensions

### False confidence (includes effectiveness hard limit)

A passing health check doesn't mean artifacts are optimal — just internally consistent and reference-sound. More critically: validation can check structural integrity and internal consistency, but **it cannot verify that a prompt still produces good output after a change**. Effectiveness is only measurable through actual use. This is a permanent limitation. Autonomous operation can maintain reliability and efficiency; effectiveness can only be validated indirectly or through human judgment. *Mitigation: R-L2-self-correction (multi-pass) + R-G3-progressive-learning (outcome tracking) catch regressions that post-deployment behavior reveals. Residual: silent effectiveness drift remains a hard boundary.*

### Self-update infrastructure staleness (amplified risk)

A bug in a regular context file affects the prompts that reference it. A bug in the system's own meta-guidance affects every decision the system makes — including its ability to detect the bug. The system CAN update its operational components through the standard pipeline, but these updates carry amplified blast radius and receive elevated risk classification. Updates to the vision, meta-principles, and risk threshold boundaries are human-only. *Mitigation: Staleness avoidance first principle — hard gate before every cycle, no caching on meta-components, CRITICAL findings for dangling references in self-update infrastructure.*

### Over-correction

A system tuned to detect every minor inconsistency generates noise. A typo in a version history entry isn't worth a correction proposal. Severity thresholds limit findings to those affecting artifact reliability or correctness.

### Trigger completeness

The system can only react to changes it knows about. Missing triggers — in monitored platform channels, ecosystem sources, or internal events — let drift accumulate until a time-based check catches it. Trigger coverage is a perpetual gap to manage, not a problem to solve once.

### Adoption judgment

Deciding whether to adopt or incorporate external work requires judgment the system can only partially automate. Trust signals are heuristic, not definitive. The adopt-vs-incorporate decision remains a human call.

### Cost of vigilance

Every health check consumes tokens. Tier-based priority cadence bounds the cost: Tier 0/1 artifacts are reviewed every cycle; lower tiers are checked only on relevant stale triggers. The dependency graph (R-S2-dependency-graph) minimizes redundant checks; caching (R-S3-validation-caching) skips recently-validated artifacts; meta-components are always checked.

### Concurrency (future concern)

Concurrent modification — two self-update cycles running simultaneously, or a human editing an artifact while the system proposes changes — requires file-level locking or sequential processing. Acceptable at current manual-invocation scale; must be addressed before event-driven automation.

### Complexity ceiling

A self-updating system that becomes too complex to understand is self-defeating. Simplicity in the update mechanism is a design goal, not a nice-to-have.

### Deterministic rigidity

Deterministic parsers break when formats change (new YAML field, new metadata structure). When a deterministic check encounters unexpected structure, the system falls back to LLM processing and logs the fallback for investigation. Accumulated fallback logs signal that the deterministic check needs updating.

### Instruction file collision

Instruction files auto-inject by glob pattern with no precedence mechanism. Overlapping `applyTo` patterns cause silent rule collisions. *Mitigation: Instruction minimization principle (R-S8). Residual: testable rules can still contradict on measurable dimensions, but these contradictions are detectable and resolvable.*

### Insufficient outcome data

Progressive learning (R-G3) requires sufficient volume of autonomous actions and outcomes to produce meaningful signal. Start with conservative thresholds and only loosen as outcome data accumulates.

## ✅ Success criteria

The self-updating prompt engineering system succeeds when:

1. **Time to detect drift drops from weeks to days.** *Query: `outcome-log.jsonl` entries where `trigger ≠ scheduled` have `timestamp` within N days of the triggering event.*

2. **Manual audit frequency decreases.** Full system reviews become confirmation checks rather than discovery sessions. *Query: count of `autonomy_level: autonomous | notify` entries over the period.*

3. **Correction proposals are actionable.** When the system surfaces a finding, it's specific enough to approve or reject without additional investigation. *Query: human-approved entries in the log without additional follow-up actions recorded.*

4. **False positive rate stays low.** *Query: ratio of `outcome: success` to `outcome: rolled_back` for `autonomy_level: autonomous` entries over rolling 30 days.*

5. **The system stays simple enough to maintain.** Update machinery is understandable, debuggable, and modifiable. *Query: percentage of meta-infrastructure changes that required human escalation.*

6. **Artifact quality improves measurably.** Review scores trend upward. CRITICAL and HIGH findings per audit cycle decrease. *Query: per-cycle average `validations_failed` count over time.*

7. **Self-update process cost stays bounded.** Token consumption per review cycle doesn't grow unbounded as the artifact set grows. Scoped invocations (per-type review) cost significantly less than full-system reviews. *Query: estimated token cost per `trigger_source` type, tracked over releases. Ratio of screening-only completions to screening-triggered-deep-pass escalations, tracked per invocation type.*

8. **Autonomous execution success rate exceeds threshold.** The percentage of autonomous changes that pass post-implementation verification stays above 95%. When it drops, the system tightens autonomy thresholds. *Query: `outcome: success` / total `autonomy_level: autonomous` entries over rolling 30 days.*

9. **Guidance quality improves measurably.** The six guidance quality properties (clarity, non-redundancy, non-contradiction, completeness, prioritization, actionability) score above threshold across all active context files. *Query: per-context-file scores on each property, tracked across review cycles.*

10. **Design/review parity stays explicit and stable.** Goal, rationale, and process sections consistently state one shared quality objective and one shared scope intent for design and review. *Query: deterministic parity check confirms required parity statements are present and no section claims lower quality targets for either role.*

11. **Reliability coverage is explicit.** Every reliability anchor in `scope.covers` and in the rationales (loop stability, multi-pass validation invariant, rollback readiness, metadata-guarded protocol enforcement, fail-closed boundaries, boundary actionability, autonomy calibration, conflict-resolution precedence, portability boundary, mode-vs-risk decoupling) is addressed by at least one use case in `05-reliability/`. *Query: deterministic coverage check confirms each anchor maps to ≥1 UC ID in `usecase-index.json` with `group: "05-reliability"`.*

### Parity acceptance checks for future vision updates

Use these deterministic checks in future vision updates:

1. **P-1 shared quality objective present** — Goal section and rationale text include explicit parity wording for design and review.
2. **P-2 shared scope intent present** — Scope/goal language states that applicability filtering is by artifact type, not by role quality target.
3. **P-3 process-role separation present** — Design is described as synthesis/construction and review as analysis/verification/improvement.
4. **P-4 conflict guardrail present** — Conflict-resolution section states that process optimization cannot downgrade parity on quality objective or scope intent.
