## v15.4.0 — 2026-06-04

**v15.4.0 always plan-then-execute + orthogonal `--plan-file` (8th parameter) + fresh/reconcile/trust execution modes.** Unifies `--mode plan` and `--mode apply` onto one execution engine and makes the plan file the single pivot artifact every mutating run passes through. Companion plan: [01-vision-pe-meta-improvement-plan.md](../../src/docs/90.%20Issues/202606/20260603.02-pe-meta-update/01-mode-apply-repeated-processing/01-vision-pe-meta-improvement-plan.md).

- **§ Plan-mode output contract → § Plan output contract.** A plan file is now materialized in **both** modes; `apply` additionally executes it through the same execution engine shared with later plan consumption (`apply = plan + execute`, `plan = apply minus execute`). Plan emission is non-skippable on every mutating run. Added an **execution-ready precision** clause: every actionable row MUST carry a literal `old → new` edit or an unambiguous anchor so a cheaper executor applies edits without re-deriving intent.

- **New eighth canonical parameter `--plan-file <path>`.** Governs the plan artifact's **location/identity only** — orthogonal to every scoping parameter, never deciding regenerate-vs-trust. Default auto-name `<run-folder>/<NN>-<kebab-name>.plan.md`; fallback `.copilot/temp/pe-meta-state/plans/YYYYMMDD-HHMMSS-<kebab-name>.plan.md`. Surface grows **7 → 8** — touches P0 `minimal-canonical-surface`; justified because no existing parameter can express plan-artifact location and the new axis carries exactly one meaning (vision-owner consent recorded in the companion plan). Updated every canonical-surface enumeration (description, scope.covers `eight-parameter-canonical-surface`, boundary list, `minimal-canonical-surface` principle, Option taxonomy table + applicability matrix + default-full table, pre-parser phase text, explicit-option-discipline section, success criterion #12).

- **New § Plan execution modes.** Two orthogonal booleans — **(baseline available?) × (research runs?)** — resolve to **fresh** (no baseline, research runs), **reconcile** (baseline + research), **trust** (baseline, research skipped), and a rejected **invalid** combination. A same-conversation just-generated plan counts as an available baseline. `--skip research` (not file existence) decides regenerate-vs-trust; at `breadth=full` it is legal ONLY when a validated baseline substitutes for research — a principled exception to `default-full-investigation`, since the plan IS the prior research product.

- **New § Reconcile semantics.** A baseline is a living input: load → merge fresh evidence → **preserve human-authored decisions** (park-lot rulings, consents, scope tags, rationales) — contradicting evidence escalates rather than overwriting → re-verify goal coverage + actionability → overwrite → execute. Applies the `metadata-guarded-changes` guard/reconcile pair to the plan artifact itself; converges the loop (`loop-stability`).

- **New § Model-routing seam.** Pins the reasoning→standard model boundary to the plan→execute boundary (Phases 1–4 reasoning model, Phases 5–7 standard model), realized via each delegated meta-agent's own `model:` field — not mid-prompt switching. Makes the declared `model-routing` / `model-specialization` scope items concrete and cuts cost on every mutating run.

- **New § Execution-ready precision and drift-guard scope.** Drift guard (target-section hashing) is **skipped** in same-run fresh/reconcile (plan and execute are back-to-back) and **REQUIRED** in cross-run trust mode; a hash mismatch degrades gracefully by escalating the drifted row rather than applying a stale edit.

## v15.3.0 — 2026-06-04

**v15.3.0 value-shape `--start`/`--end` + Processing-state model + two P1 principles.** Makes incremental re-processing explicit and re-baselineable without adding a parameter. Companion plan: [02-pe-meta-update-plan.md](../../src/docs/90.%20Issues/202606/20260602.01-pe-meta-update/02-pe-meta-update-plan.md).

- **`--start` / `--end` extended to a value-shape parameter (date OR source-version).** A bound may now be an ISO-8601 date / `now` / relative offset OR a source-version token (e.g. `1.099`) resolved to a publish timestamp via the scoped source's `version_scheme`. Phase 0a performs the version→timestamp resolution; phases 1–8 consume only a resolved time window. Mirrors the existing two-shape `--scope` parser. **No new parameter** — the canonical surface stays seven, so `minimal-canonical-surface` (P0) is preserved and breadth remains a derived attribute. Two guard rails: a version-shaped bound REQUIRES a singleton `--source`; a source with `version_scheme: none` rejects version tokens. Updated: § Option taxonomy parameter row + new value-shape parser paragraph, "What full means operationally" table, Default-full semantics by parameter row, Option applicability key-decision bullet, and the § Assess manual/trigger/bounded-delta breadth list and research-phase contract table.

- **New `#### Processing-state model` subsection (under § Assess).** Defines the two-axis state model: an **input axis** (per-source ledger `<state.path>/triggers/<source-id>.json` with `source_id`, `version_scheme`, `last_seen_version`, `last_seen_timestamp`, `last_digest_hash`) and a **processing axis** (per-artifact, per-dimension coverage with `source_versions`, `depth`, `status`, `last_run`). Specifies the **processing unit (PU = artifact × applicable-dimension)** staleness rule and three watermark origins (full = beginning of time, coverage ignored; incremental = recorded `source_versions`, coverage trusted; bounded-delta = explicit `--start`, coverage OVERRIDDEN inside the window). The bounded-delta override is the **re-baseline / distrust-recovery path** — re-run a window whose prior processing is no longer trusted even though it was already covered.

- **Two new P1 principles.** `processing-state-is-multidimensional` (*relies on:* `metadata-driven`, `deterministic-processing`) — state is recorded on two independent axes, never collapsed to a per-source scalar. `coverage-completeness-guarantee` (*relies on:* `validation-caching`, `deterministic-processing`) — a never-covered PU is always in the work set; a PU is skipped only when provably current; a narrowed pass never leaves a permanent coverage gap. Matching body subsections added under § Design principles after *Active dimensions follow assessment capability*. P0 count unchanged (5, within 3–7); per `vision-frontmatter.instructions.md` a P1 addition requires a changelog entry (this one) — the version bump to 15.3.0 also reflects the new behavioral contract.

- **New P2 `scope.covers` item `value-shape-start-end-parser`.** Mirrors the existing `value-shape-scope-parser` entry.
