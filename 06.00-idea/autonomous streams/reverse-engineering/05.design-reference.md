## 1. Purpose and Audience

The Reference section answers: **"What exactly does this system expose, accept, and store?"**

It is the lookup layer — readers already understand the architecture; they come here with a specific question: "what parameters does this endpoint accept?", "what does this table's schema look like?", "what does this configuration key do?"

| Audience | What they need |
|----------|---------------|
| Developer integrating with the API | Endpoint routes, request body shapes, response codes, auth requirements |
| Developer extending the system | Domain model properties, service method signatures, repository interfaces |
| Operations | Configuration key meanings, feature flag effects, connection string format |
| QA | HTTP test files ready to run; error response shapes and expected codes |

## 2. Sub-areas Overview

| Sub-area | Files produced | Source artifacts |
|----------|---------------|-----------------|
| API controllers | `api/overview.md`, `api/controllers/{Name}.md`, `testing/http-tests/{Name}.http` | `*Controller.cs` files, auth handlers |
| Database schema | `data/overview.md`, `data/schema.md`, `data/tables/{Name}.md`, `data/repositories.md`, `data/migrations.md`, `data/stored-procedures.md` (if any), `data/views.md` (if any) | DbContext, SQL scripts, migration files, repository interfaces |
| Domain models | `reference/domain-models/{Name}.md`, `reference/dtos/{Name}.md`, `reference/enums.md` | Model classes, DTO classes, enum types |

| Services (optional) | `reference/services/{Interface}.md` | Service interfaces + implementations |

The API controller sub-area applies to **🔴 Principal components**. Supporting components that also expose HTTP endpoints document them within their own component page (see [06.design-components.md](06.design-components.md)).

## 3. API Controllers

### 3.1 Source artifacts to read

For each controller:

| Artifact | What to extract |
|----------|----------------|
| `*Controller.cs` | Class-level route prefix, `[Authorize]` / `[AllowAnonymous]` attributes, all action methods with routes and HTTP verbs |
| Per-action method | Method signature, route parameters, query parameters (`[FromQuery]`), request body (`[FromBody]`), return type |
| Auth handler classes | What the handler validates (JWT, HMAC, API key, etc.); which header it reads |
| Validation attributes | `[Required]`, `[MaxLength]`, `[Range]`, FluentValidation rules |
| XML comments (`/// <summary>`) | Use as description if present |
| Swagger / OpenAPI attributes (`[ProducesResponseType]`) | Documented response codes |

### 3.2 Controller overview page (`api/overview.md`)

Criteria:
1. **Base URL** — show format with environment placeholder, e.g., `https://{host}/api`
2. **Authentication summary** — table of all schemes found, one row per scheme:
   - Scheme name
   - Header name
   - Token format (e.g., "HMAC `amx {signature}`", "Bearer JWT")
   - Which endpoints use it
3. **At a glance** — full endpoint inventory table across all controllers:
   | Method | Route | Controller | Description | Auth |
4. **Common patterns** — note any global filters, model binding conventions, or response envelope shapes

### 3.3 Per-controller page (`api/controllers/{ControllerName}.md`)

Criteria — required sections:

1. **Swagger screenshot embed** — at the top of the page, immediately below the summary paragraph. Freshness rules apply (see Section 3.4). Use provenance comment.

   ```markdown
   ![{ControllerName} endpoints]({relative-path-to-screenshot})
   <!-- swagger-screenshot: source=swagger-ui date={YYYYMMDD} controller={ControllerName} -->
   ```

2. **Base route** — the class-level `[Route]` attribute value

3. **Authentication** — table of auth requirements per endpoint group. If different endpoints use different auth schemes, show each group explicitly:

   | Endpoint group | Auth scheme | Required header |
   |----------------|-------------|-----------------|
   | All except `/health` | `amx` HMAC | `Authorization: amx {signature}` |
   | `/health` | None | — |

4. **At a glance** — summary table of all endpoints:

   | Method | Route | Description | Auth |
   |--------|-------|-------------|------|

5. **Per-endpoint sections** — one `###` section per action method:
   - Full route (class prefix + method route)
   - Parameters table (all `[FromRoute]`, `[FromQuery]`, `[FromBody]` params)
   - Request body (JSON schema or example)
   - Response table (all documented status codes + response body shape)
   - At least one request/response example
   - Error responses table (status code, meaning, resolution)
   - Source anchor comment

### 3.4 Swagger screenshot freshness

Screenshots are captured using the SwaggerScreenshot Procedure defined in the v4 agent (`documentation-manager.v4.agent.md`). Apply these freshness rules before deciding whether to recapture:

| Trigger | Action |
|---------|--------|
| Screenshot file does not exist | Capture now |
| Screenshot file exists, date in filename is current year, no breaking changes in controller since | Skip capture — reuse existing |
| Screenshot file is older than 12 months | Recapture |
| Controller has had breaking changes (new endpoints, removed endpoints, route changes) since screenshot date | Recapture |
| User explicitly requests screenshot refresh | Recapture |

Screenshot naming: `{YYYYMMDD}-{controller-kebab}-swagger.png`  
Storage location: `docs/assets/screenshots/api/`

## 4. Database Schema

### 4.1 Source artifacts to read

| Artifact | What to extract |
|----------|----------------|
| `DbContext` classes | `DbSet<T>` entity registrations, `OnModelCreating` configurations (table names, column mappings, indexes, constraints, relationships) |
| EF migration files | Migration history (names + dates); flag non-trivial migrations (schema changes, data transformations) |
| SQL script files (`.sql`) | Table definitions (columns, types, nullability, defaults, constraints, PKs, FKs, indexes); stored procedures; views |
| Repository interface files | Method signatures, return types; which entities each method works with |
| Repository implementation files | SQL queries used (LINQ, raw SQL, Dapper); transaction scope |

### 4.2 Schema overview page (`data/schema.md`)

Criteria:
1. **ER diagram** — Mermaid `erDiagram` showing all entities and their relationships. Follow complexity rules: >15 entities → split into bounded context sub-diagrams.
2. **Table inventory table** — one row per table: name, purpose, row count estimate (if discoverable from migrations or comments), primary key type
3. **Key relationships** — brief prose description of the most important FK relationships

### 4.3 Per-table page (`data/tables/{Name}.md`)

Criteria:
1. **Columns table** — every column: name, data type, nullable, default, description
2. **Constraints** — PK, unique constraints, check constraints
3. **Indexes** — all indexes with included columns and purpose
4. **Foreign keys** — referenced table, referencing columns, ON DELETE behavior
5. **Source anchor** — `<!-- Source: {TableName} in {sql-file-path} -->`

### 4.4 Repositories page (`data/repositories.md`)

Criteria:
- Table of all `I*Repository` interfaces
- For each interface: a sub-table of methods with signature, purpose, tables accessed
- Notes on transaction scope or unit-of-work pattern

### 4.5 Migrations page (`data/migrations.md`)

Criteria:
- Chronological table of migrations (name, date, summary of changes)
- Flag any data transformation migrations with `⚠️` and describe what they do
- Note the migration tool and framework version

## 5. Domain Models and DTOs

### 5.1 Domain model pages (`reference/domain-models/{Name}.md`)

Criteria (simple pages — no progressive disclosure needed):
1. **Properties table** — every public property: name, type, nullable, description, validation rules
2. **Relationships table** — related entities, relationship type (1:1, 1:N, M:N), FK property
3. **Validation rules** — any `[Required]`, `[MaxLength]`, data annotations, or FluentValidation rules
4. **Consumers** — which controllers / services / use cases reference this entity
5. Source anchor

Document only **domain entities** and **value objects** in this sub-area. DTOs have their own sub-area below.

### 5.2 DTO pages (`reference/dtos/{Name}.md`)

Criteria:
1. **Properties table** — same format as domain models
2. **Mapping source** — which domain entity this DTO maps from/to (and whether it is partial or full)
3. **Consuming endpoints** — which controller actions use this DTO as request body or response
4. Source anchor

Only document DTOs that are request bodies or response types for **public endpoints**. Internal DTOs used within service layers do not need individual pages.

### 5.3 Enums page (`reference/enums.md`)

Criteria:
- One `###` section per enum type
- Values table: numeric value, name, description
- Usage: which fields / properties use this enum
- Source anchor

## 6. Configuration and Feature Flags

### 6.1 Source artifacts to read

| Artifact | What to extract |
|----------|----------------|
| `appsettings.json` | Base configuration keys and default values |
| `appsettings.{environment}.json` (all envs) | Per-environment overrides |
| `IOptions<T>` classes | Typed option class properties → map to appsettings key paths |
| Feature flag evaluator / provider | Feature flag names, default states, what feature each flag gates |
| Startup / DI registration | Which option classes are bound to which config paths (`services.Configure<T>`) |

### 6.2 Settings reference page (`configuration/settings-reference.md`)

Criteria:
- One section per **configuration section** (e.g., `ConnectionStrings`, `Authentication`, `Features`)
- Per-section: a table of all keys under that section:

  | Key | Type | Required | Default | Description | Secret? |
  |-----|------|----------|---------|-------------|---------|

- The **Secret?** column marks keys whose values come from Key Vault or environment variables (never show the actual values)
- Keys that map to an `IOptions<T>` class: note the class name and namespace
- Keys that differ between environments: note that overrides exist (point to environment-overrides page)

### 6.3 Feature flags page (`configuration/feature-flags.md`)

If the project has feature flags, document each one:

| Flag name | Default | What it enables | How to activate | Risk if enabled |
|-----------|---------|-----------------|-----------------|-----------------|
| `{FlagName}` | `{true/false}` | {what it turns on or off} | {config key / env var / API call} | {what could break} |

If no feature flags exist, this page is omitted.

### 6.4 Environment overrides page (`configuration/environment-overrides.md`)

Criteria:
- A diff table showing which keys have different values per environment:

  | Config key | Dev | Test | Stage | Prod |
  |-----------|-----|------|-------|------|
  | `ConnectionStrings:Default` | `localdb` format | Key Vault | Key Vault | Key Vault |
  | `Authentication:Schemes` | All schemes | All schemes | All schemes | Legacy removed |

- Note any keys that exist only in specific environments

## 7. HTTP Test Files

**Files:** `testing/http-tests/{ControllerName}.http`

One `.http` file per controller, usable in VS Code REST Client without manual editing. See the HTTP test file template in the documentation standards.

### Criteria

1. **Variable declarations at the top** — all environment-specific values as `@variable = value` or `{{VARIABLE_NAME}}` placeholders
2. **A `.env` comment block** showing which variables are needed and their format
3. **One request block per action method** — minimum one success case per endpoint
4. **Both success and error cases** for endpoints with validation or auth checks
5. **Variable naming matches the project's auth scheme** — e.g., `{{HMAC_KEY}}` for HMAC-signed endpoints, `{{BEARER_TOKEN}}` for JWT
6. Request names (for referencing responses) set with `# @name {requestName}`

## 8. Templates

### 8.1 `api/controllers/{ControllerName}.md` template

```markdown
# {ControllerName}

<!-- Source: {ControllerName}.{ActionName}() in {file-path} -->

{One-paragraph description of what this controller manages.}

## Base route

`/api/{route}`

## Authentication

| Endpoint group | Auth scheme | Required header |
|----------------|-------------|-----------------|
| {group or "All"} | {scheme name} | `{header: value-format}` |

## At a glance

| Method | Route | Description | Auth |
|--------|-------|-------------|------|
| GET | `/api/{route}` | {description} | {scheme} |
| POST | `/api/{route}` | {description} | {scheme} |

## Swagger UI

![{ControllerName} endpoints](../../assets/screenshots/api/{YYYYMMDD}-{controller-kebab}-swagger.png)
<!-- swagger-screenshot: source=swagger-ui date={YYYYMMDD} controller={ControllerName} -->

---

## GET `/api/{route}`

<!-- Source: {ControllerName}.{GetAll}() in {file-path} -->

{Description.}

### Parameters

| Name | In | Type | Required | Description |
|------|----|------|----------|-------------|
| `{param}` | query / path | `{type}` | Yes / No | {description} |

### Response 200 OK

```json
[
  {
    "{field}": "{value}"
  }
]
```

### Error responses

| Status | Meaning | Resolution |
|--------|---------|------------|
| 401 | Unauthorized | Provide valid {scheme} token |
| 404 | Not found | Verify `{param}` exists |

---

## POST `/api/{route}`

<!-- Source: {ControllerName}.{Create}() in {file-path} -->

{Description.}

### Request body

```json
{
  "{field}": "{value}"
}
```

### Response 201 Created

```json
{
  "id": "{generated-id}",
  "{field}": "{value}"
}
```

### Error responses

| Status | Meaning | Resolution |
|--------|---------|------------|
| 400 | Validation error | Check request body against schema |
| 409 | Conflict | Resource already exists |
```

### 8.2 `data/tables/{TableName}.md` template

```markdown
# Table: `{TableName}`

<!-- Source: {TableName} in {sql-file-path} -->

{One-sentence description of what this table stores.}

## Columns

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `Id` | `int` | No | — | Primary key |
| `{Column}` | `{type}` | Yes/No | `{default}` | {description} |

## Constraints

| Name | Type | Columns |
|------|------|---------|
| `PK_{TableName}` | PRIMARY KEY | `Id` |
| `UQ_{TableName}_{Column}` | UNIQUE | `{column}` |

## Indexes

| Name | Columns | Included | Purpose |
|------|---------|----------|---------|
| `IX_{TableName}_{Column}` | `{col}` | — | {purpose} |

## Foreign keys

| FK name | Column | References | ON DELETE |
|---------|--------|-----------|-----------|
| `FK_{TableName}_{RefTable}` | `{col}` | `{RefTable}.Id` | CASCADE / SET NULL |
```

### 8.4 `testing/http-tests/{ControllerName}.http` template

```http
# {ControllerName} — HTTP test file
# 
# Required variables:
# @BASE_URL = https://your-host/api
# @TENANT_ID = {your-tenant-id}
# @AUTH_HEADER = amx {computed-hmac-signature}  # or: Bearer {jwt-token}

@BASE_URL = https://localhost:44300/api
@TENANT_ID = test-tenant

### List all {resources}
# @name list{Resources}

GET {{BASE_URL}}/{route}?tenantId={{TENANT_ID}}
Authorization: {{AUTH_HEADER}}

### Expected: 200 OK - Returns array of {resource} objects

###

### Create {resource}
# @name create{Resource}

POST {{BASE_URL}}/{route}
Content-Type: application/json
Authorization: {{AUTH_HEADER}}

{
  "{field}": "{value}"
}

### Expected: 201 Created

###

### Unauthorized request (missing auth header)

GET {{BASE_URL}}/{route}?tenantId={{TENANT_ID}}

### Expected: 401 Unauthorized
```

## 9. Quality Gates

Before the Reference section is considered complete, verify:

| Check | Pass condition |
|-------|---------------|
| All public endpoints documented | Every `[HttpGet]`, `[HttpPost]`, etc. action method has a section in its controller page |
| Swagger screenshot present | Every controller page has a `![...]` screenshot with a provenance comment |
| Screenshot freshness | No screenshot is older than 12 months (check filename date) |
| Parameters complete | Every action method's parameters are in a table with type, required, and description |
| Response codes documented | Every action has at least HTTP 200/201 + error codes documented |
| HTTP test files present | One `.http` file per controller in `testing/http-tests/` |
| ER diagram present | `data/schema.md` has at least one `erDiagram` block |
| All tables documented | Every table in the schema has a page in `data/tables/` |
| Configuration keys complete | Every key in `appsettings.json` is in `settings-reference.md` |
| Feature flags page exists if flags found | Skipped only if zero feature flags discovered |
| Source anchors present | Every controller section, table page, and config key has `<!-- Source: ... -->` |
| No real secrets | Zero passwords, connection strings with credentials, or API key values in any page |
