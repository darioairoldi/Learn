## 1. Purpose and Audience

`docs/index.md` is not a tutorial, not a reference page, and not a nav mirror. It is an **orientation hub** — its sole job is to answer four questions for any reader who lands on it, within 30 seconds:

1. **What is this system?** — plain-language identity
2. **Am I in the right place?** — is this the system the reader is looking for?
3. **What does it do?** — business-language capabilities
4. **Where do I go next?** — curated reading path for my role

Every section on the home page serves one of these four goals. If a section serves none of them, it does not belong on the home page.

### The three reader types the home page must serve

| Reader | Their question | What sends them deeper |
|--------|---------------|----------------------|
| Developer onboarding | "How do I get this running?" | A clearly numbered path starting with Prerequisites |
| Architect / tech lead | "How is this system designed?" | A path that leads to architecture overview and decisions |
| Operations / DevOps | "How do I deploy and monitor this?" | A path that leads to deployment, configuration, runbooks |

A reader who cannot find their path within 30 seconds will navigate away. The home page fails if any of these three types is not served.

## 2. Source Artifacts to Read

| Artifact | What to extract |
|----------|----------------|
| Root `README.md` | Project name, one-line description, repository purpose |
| All `.sln` / `.slnx` files | Solution name → used in "At a glance" table |
| All `*.csproj` files — `<TargetFramework>` | Framework version → used in "At a glance" table |
| All `package.json` files — `"name"` field | Frontend technology → used in tech stack table |
| `mkdocs.yml` — `site_name` | Canonical documentation site name |
| `appsettings*.json` — environment names | Environments → used in "At a glance" table |
| Pipeline YAML files | Environments deployed to (cross-check) |
| Component Registry (from Phase 1, if run) | Component tier table |
| Auth handler files | Auth schemes → one-line security note |
| Any `CHANGELOG.md` or ADR files | Key capabilities context |

If Phase 1 discovery has not been run, read `README.md` plus one representative `.csproj` and `appsettings.json` — enough to fill the identity block accurately.

## 3. The 30-Second Test

Before finalizing the home page, mentally simulate a reader who has never seen this project landing on it. They should, within 30 seconds, be able to answer:

- **What is this?** The first paragraph must state: the product name, what kind of system it is, what it does (in business terms), and who uses it.
- **Where do I start?** At least one reading path is immediately visible without scrolling.
- **What's in here?** The component table and capabilities list tell the full scope.

**The first paragraph is the pitch.** It must be:
- **Concrete, not generic.** "Manages firmware distribution and device provisioning for ABB Ekip COM Hub devices" is concrete. "A modern REST API built with Clean Architecture" is generic and could describe any project.
- **One or two sentences maximum.** No preamble, no hedging.
- **No unexplained acronyms.** If an acronym is unavoidable, expand it on first use.

## 4. Required Content Blocks

The home page must contain **exactly these five blocks in this order**. No block may be omitted. Additional content may be added after block 5.

### Block 1 — Identity paragraph

One or two sentences. Subject = the product name. Verb = what it does. Object = for whom.

```markdown
{ProductName} is a {system type} that {primary function} for {primary user/client}.
```

**Criteria:**
- Names the specific domain (not just "a REST API")
- Names the specific users or devices it serves
- No marketing language ("powerful", "robust", "next-generation")
- No version numbers in the identity paragraph

### Block 2 — At a glance table

A compact factual table — the "business card" of the system. Serves readers who want to quickly assess whether this system is relevant to them.

Required rows:

| Property | Value |
|----------|-------|
| Type | {what kind of system: REST API, SPA, background service, CLI, etc.} |
| Framework | {primary technology + version} |
| Deployment | {where it runs: Azure App Service / Kubernetes / on-premise / etc.} |
| Environments | {list: Development, Test, Stage, Production} |
| Solution | `{solution-file-name}` |

**Criteria:**
- Values are factual — derived from `.csproj` target frameworks, pipeline YAML, and IaC files
- No empty or "N/A" rows — omit a row if the information is genuinely not applicable
- Solution file is the canonical entry point for developers who want to open the project

### Block 3 — Audience reading paths

Minimum **three** curated reading paths for the three primary audiences. Each path is a numbered list of 4–6 linked pages that form a complete reading journey for that audience.

**Path design rules:**
- Each path tells a **narrative** — "here is how you go from zero to expert for your role"
- Each link in the path has a **one-line description** explaining why that page is next in the sequence
- The paths are **not** mirrors of the nav tree — they are curated subsets
- Paths may share pages (a page can appear in more than one path)
- The Developer path always starts with the Prerequisites page

**Required paths:**
1. **Developer onboarding** (👨‍💻) — starts at Prerequisites, ends at API overview or a domain-specific next step
2. **Architecture review** (🏗️) — starts at Architecture overview, includes logical architecture, key decisions
3. **Operations guide** (⚙️) — starts at Deployment overview, includes configuration, runbooks

**Optional paths** (add if the system has corresponding content):
- Security review (🔒) — starts at Security overview
- QA / Testing — starts at Testing overview

### Block 4 — Component summary table

A table listing every component in the system with its tier symbol, type, and purpose. This gives any reader an instant picture of the system's scope.

Required columns:

| Component | Type | Priority | Purpose |
|-----------|------|----------|---------|
| `{Name}` (linked to its docs page) | {API / Library / SPA / Function / CLI} | 🔴 / 🟠 / 🟡 / ⚪ | {one-line purpose} |

**Criteria:**
- All components discovered in Phase 1 (or during home page analysis) appear here
- Tier symbols are consistent with the Component Classification in `00.documentationmanager.design.md`
- Purpose column uses business language, not technical ("manages device firmware" not "handles HTTP requests")
- Link targets are valid doc pages — if a component page does not exist yet, link to the nearest existing page (e.g., the architecture overview)

### Block 5 — Key capabilities

A bulleted list of 4–8 capabilities the system provides. Each capability is **bold + linked** to the relevant documentation section, followed by a one-sentence explanation.

```markdown
- **[Capability name](link)** — one-sentence explanation of what this does for users/devices.
```

**Criteria:**
- Business language: what the system does for its users or clients
- Not a technology list ("Uses Entity Framework Core" does not belong here)
- Each capability links to the deepest relevant page — not to the home page or overview
- Maximum 8 capabilities — if more exist, group or select the most prominent

### Optional Block 6 — Technology stack table

A concise table of the technologies used. This is supplementary context for developers who want to quickly assess the technology landscape before onboarding.

| Layer | Technology |
|-------|-----------|
| Framework | .NET 10 |
| ORM | Entity Framework Core 10 |
| Database | SQL Server / PostgreSQL |
| Auth | HMAC (`amx`) + JWT Bearer |
| Observability | OpenTelemetry + Application Insights |

**Criteria:**
- List only primary technologies — no transitive dependencies, no test-only packages
- This table is optional. Include it if the codebase uses non-obvious technology choices.
- Never put version numbers in the home page unless they are significant (e.g., "migrated to .NET 10")

## 5. Audience Reading Paths

### How to construct a reading path

1. **Identify the reader's goal** — what does this reader need to accomplish?
2. **Map the minimal viable journey** — what is the shortest sequence of pages that takes them from zero to goal?
3. **Write the link text as an action or answer** — "Prerequisites — tools and accounts you need" is better than just "Prerequisites"
4. **Order by dependency** — never link to a page before the reader has the context to understand it

### Developer onboarding path — required structure

The developer path must cover this sequence in order:

1. Prerequisites (tools, accounts)
2. Installation (clone, restore, configure secrets)
3. Build and run (verify the application starts)
4. Project structure (understand the solution layout)
5. First domain concept (the most important domain-specific page for this project)
6. API overview or integration start point

Items 5 and 6 vary per project — they are the "now go deeper" links that transition the reader from onboarding into productive work.

### Architecture review path — required structure

1. Architecture overview (high-level diagram)
2. Logical architecture (layers, patterns)
3. Data layer (if the project has a significant data model)
4. Key architectural decision (link to most relevant ADR, if any)
5. Security overview (how auth is designed)

### Operations path — required structure

1. Deployment overview (pipelines, promotion flow)
2. Configuration reference (settings and their meanings)
3. Infrastructure overview (environments, resources)
4. Production runbook or operational procedure (if it exists)

## 6. Tone and Writing Rules

| Rule | Rationale |
|------|-----------|
| **First person is forbidden** — the page speaks about the system, not the author | Documentation outlives its author; impersonal is timeless |
| **Avoid "our", "we", "I"** | Same reason |
| **Present tense for facts** | "The API manages firmware distribution" not "The API will manage" |
| **No marketing language** | "Powerful", "robust", "seamless", "next-generation" add no information |
| **No implementation detail in capabilities** | "Manages firmware distribution" not "Stores firmware blobs in Azure Blob Storage and serves download URLs via a signed SAS token endpoint" — link to the detail page instead |
| **Acronyms expanded on first use** | Define `HMAC`, `SAS`, `BFF`, `CLI` when first mentioned |

## 7. What NOT to Put on the Home Page

| Prohibited content | Where it belongs |
|-------------------|-----------------|
| Full API endpoint list | `api/overview.md` or controller pages |
| Full configuration key list | `configuration/settings-reference.md` |
| Step-by-step setup instructions | `getting-started/` pages |
| Code snippets | Architecture or API pages |
| Changelog / release notes | A separate `CHANGELOG.md` or appendix page |
| Pipeline execution details | `deployment/pipelines/` pages |
| Security posture findings | `security/posture.md` |
| Troubleshooting guides | Per-component or per-operation pages |
| The full nav tree reproduced as text | The nav bar already provides this |

## 9. Quality Gates

Before the home page is considered complete, verify:

| Check | Pass condition |
|-------|---------------|
| 30-second test passes | Identity paragraph is concrete, specific, two sentences maximum |
| No generic language | No "powerful", "robust", "modern", "seamless" in identity paragraph |
| At-a-glance table populated | All rows have factual values derived from source files |
| Three reading paths present | Developer, Architecture, Operations paths each have 4+ links |
| Developer path starts with Prerequisites | Link 1 in Developer path goes to `getting-started/prerequisites.md` |
| Each reading path link has a description | No bare links — every link has a one-line explanation after `—` |
| Component table present | All discovered components appear with tier symbol and business-language purpose |
| All component links valid | Every link in the component table resolves to an existing doc page |
| Key capabilities use business language | No implementation details in capabilities bullets |
| All capability links valid | Every bold capability link resolves |
| No prohibited content | No full API lists, config key lists, code snippets, or troubleshooting on this page |
| Source not fabricated | Every factual claim (framework, environments, solution file) is traceable to a source artifact |
