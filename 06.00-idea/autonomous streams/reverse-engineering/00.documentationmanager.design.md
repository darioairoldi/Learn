## 1. Vision and Problem Statement

### The problem

Technical documentation in software repositories tends to suffer from one of two failure modes:

- **Absent** — engineers rely entirely on reading source code, onboarding is slow, knowledge is siloed
- **Stale** — documentation was written once and never updated, leading to incorrect or misleading content that erodes reader trust

A documentation manager agent attacks both problems simultaneously: it generates documentation from live repository sources (making it accurate on day one) and provides a repeatable, structured update workflow (keeping it accurate thereafter).

### The vision

The agent is an **evidence-based documentation specialist**. It reads every layer of the repository — source code, configuration files, Infrastructure as Code, CI/CD pipelines, database projects, Docker files — and produces a structured MkDocs documentation site that:

- Answers the questions of **four distinct audiences** without requiring them to read code:
  - **Developers** onboarding to the project
  - **Architects** reviewing system design
  - **Operations teams** deploying and monitoring
  - **Security reviewers** assessing posture
- Stays **synchronized with the repository** — content is derived from evidence, not speculation
- Is **organized by progressive disclosure** — from broad orientation to granular reference
- Is **complete by default** — every component, every endpoint, every configuration key is documented

### Key design commitment

> The agent documents **what the repository shows**, never what was intended. Source code, configuration, IaC, pipelines, database schemas, and deployment artifacts are all valid evidence. If a behavior is not observable in any repository artifact, it is not documented as a confirmed fact.

## 2. Role Definition and Audience

### What the agent does

| Task | Description |
|------|-------------|
| Repository discovery | Scans the entire project for solutions, projects, frontends, functions, and database projects |
| Source analysis | Reads controllers, services, models, configuration, IaC, pipelines, and SQL schemas |
| Documentation generation | Creates structured Markdown files under `docs/` optimized for MkDocs |
| Diagram generation | Produces Mermaid diagrams for architecture, data flows, sequences, and deployments |
| Screenshot capture | Uses a browser tool to capture live Swagger UI and application screenshots |
| Nav tree management | Maintains `mkdocs.yml` so every page is reachable via the site navigation |
| Iterative updates | Assesses existing docs, extends or corrects what has changed, preserves valid content |
| Validation | Checks internal links, nav coverage, configuration key accuracy, and content quality |

### What the agent does NOT do

| Prohibited action | Rationale |
|-------------------|-----------|
| Modify source code (`.cs`, `.csproj`, config files) | Documentation is a read-only consumer of source artifacts |
| Create files outside `docs/` | All output is contained; agent cannot alter project behavior |
| Invent API behavior not visible in code | Evidence-only rule; invented content would mislead readers |
| Embed secrets, tokens, or connection strings | Security boundary; docs are often publicly published |
| Delete existing documentation without approval | Preservation rule; existing content has value until contradicted |
| Disclose internal security findings in public pages | Information disclosure control; see Section 7 |

### Companion capabilities

The agent is designed to use a browser tool for live screenshot capture and may delegate large reads to a read-and-summarize subagent. It should NOT duplicate or overlap with a general code-documenter skill — it IS the code documenter.

## 3. Core Principles

These principles govern every decision the agent makes. They are ordered from most fundamental to most operational.

### 3.1 Evidence-only documentation

Documentation claims must be traceable to a repository artifact. The agent never writes "the system supports X" without citing which file proves it. This applies even to negative assertions: if a security control is not visible in code, the agent writes "requires external verification" rather than "absent."

**Rationale:** Documentation that may be wrong is worse than no documentation — it misleads readers and erodes the credibility of everything else the doc says. An evidence-only rule makes every sentence auditable.

### 3.2 Multi-source analysis

A repository contains many knowledge sources, each authoritative for a different documentation area:

| Knowledge source | Documentation area it feeds |
|-----------------|------------------------------|
| Source code (`.cs`, `.tsx`, `.ts`) | Architecture, API surface, domain models, services |
| Configuration files (`appsettings*.json`, `.env`) | Configuration section, environment differences |
| IaC files (Bicep, ARM, Terraform) | Infrastructure section — provisioning intent |
| Pipeline definitions (YAML, GitHub Actions) | Deployment section — CI/CD behavior |
| Database projects (`.sqlproj`, migrations) | Data section — schema, stored procedures, views |
| Docker files | Infrastructure and deployment containerization |
| Auth config, certificate files | Security section |

The agent must read **all applicable sources**, not just source code, before writing any section. Skipping a source produces documentation that is accurate in one dimension and blind to others.

### 3.3 Multi-component awareness

Before writing a single doc page, the agent must know everything the repository contains. It should never assume a repository is a single project. Repositories commonly contain:

- A primary API or application
- Admin panels or BFF layers
- CLI tools for maintenance
- Azure Functions or background services
- Database projects
- Legacy code pending migration

Every component is discovered, classified, and documented — at a depth appropriate to its priority tier (see Section 6).

#### The principal component concept

The **principal component** is the deliverable that a documentation section (Architecture, Infrastructure, Reference, Getting Started, etc.) is organized around. Identifying principal components is the most important classification decision in Phase 1.

**A component is principal when it:**
- Is an independently deployable unit (API host, web application, background service)
- Has its own CI/CD pipeline
- Has its own configuration, secrets, and runtime identity
- Is a first-class product deliverable — not a supporting layer that only exists to serve another component

**Non-principal components** are those that serve, extend, or support a principal component without being independently meaningful to documentation readers: CLI tools used only by developers, Azure Functions invoked only by the primary API, legacy modules pending migration, and BFF layers that are co-deployed with the component they serve and have no independent deployment pipeline.

**BFF edge case:** A BFF that has its own CI/CD pipeline, its own configuration scope, and its own runtime identity is Principal even though it exists to serve another Principal component. For example, an Admin tool Bff that deploys independently to its own App Service and has its own pipeline qualifies as Principal — it is documented at the same structural level as the Admin tool and the API, but it is ordered after the component it primarily serves (see principal ordering below).

#### Single principal component (standard case)

When the repository has exactly one principal component, the documentation is structured around it directly. Every major section — Getting Started, Architecture, Infrastructure, Reference, Security, DevOps — documents that one component without qualification or nesting.

Non-principal components (admin panel, CLI, functions) are documented in the **Components** section. Reference detail for non-principal components (controllers, data models) may overflow into the **Reference** section if the Components section would otherwise become cluttered.

```
Architecture/          ← documents the one principal component
Getting Started/       ← onboards developers to the one principal component
Reference/             ← API surface of the one principal component
Components/            ← admin panel, CLI, functions, background workers
```

#### Multiple principal components (multi-principal case)

When the repository contains more than one principal component, every major section is restructured so that each component receives full coverage at the same structural depth. All principals are documented with the same thoroughness — but they are **ordered by documentation priority**, and that order is consistent everywhere: sub-folder order, nav entry order, at-a-glance table row order.

**Detection trigger:** The agent must propose multi-principal structure when it finds two or more components that each have: an independent deployment pipeline, independent configuration, and independent runtime identity. If uncertain, present the proposed classification to the user before proceeding (see Section 6 — User confirmation required).

**Principal ordering:** Principals are ordered from the primary system deliverable outward. The first principal is the component that is the core reason the repository exists. Subsequent principals are ordered by their dependency relationship — a component that exists to serve another principal is listed after it.

Example ordering for a repository containing MicroserviceA, Admin tool, and Admin tool Bff:

| Order | Component | Rationale |
|-------|-----------|----------|
| 1 | MicroserviceA | The core system deliverable — the primary external-facing API |
| 2 | Admin tool | A first-class product, but exists to manage MicroserviceA |
| 3 | Admin tool Bff | Purpose-built to serve the Admin tool; documented after the panel it backs |

This ordering is confirmed with the user at the Phase 1 checkpoint and applied consistently to all doc sections.

**Structural change:** Each section that documents a principal component gains a per-component sub-folder, in priority order:

```
architecture/
├── overview.md              ← cross-component system diagram
├── api-device/              ← principal 1 (first in nav)
│   ├── overview.md
│   ├── logical-architecture.md
│   ├── project-structure.md
│   ├── data-layer.md
│   └── services.md
├── admin-panel/             ← principal 2
│   ├── overview.md
│   ├── logical-architecture.md
│   └── project-structure.md
└── admin-panel-bff/         ← principal 3 (depends on admin-panel)
    ├── overview.md
    ├── logical-architecture.md
    └── services.md
```

The same sub-folder pattern and ordering applies to: `getting-started/`, `infrastructure/`, `reference/`, `security/`, and `deployment/`.

**Home page:** Introduces all principal components in priority order. Each has its own entry in the at-a-glance table and its own audience reading path row. The order conveys relative importance without demoting any component — each is still fully documented.

**Components section:** In the multi-principal case, the Components section documents only the non-principal components (supporting services, tools, functions). Principal components do not appear there.

**Reference section overflow rule:** When a non-principal component has enough API surface or data model detail that documenting it in the Components section would make that section cluttered, its reference detail (controllers, data models) may be documented in the Reference section under `reference/{component-name}/`. This rule applies regardless of whether the repository is single-principal or multi-principal.

### 3.4 Priority-ordered processing

Components are not equally important. Processing order: 🔴 Principal → 🟠 Supporting → 🟡 Tooling → ⚪ Peripheral.

This order is applied at every phase: discovery reads Principal projects first, documentation writes Principal pages first, validation checks Principal docs first.

**Rationale:** If context runs low or a session ends early, the most important documentation is already complete. Low-priority peripheral components are documented last and may be deferred to a follow-up run without leaving critical gaps.

### 3.5 Progressive disclosure

Both the documentation structure and individual pages are organized so readers receive broad orientation before narrow detail.

At the **structural level:**
- `docs/index.md` — project identity, audience-based reading paths
- Area overviews (`architecture/overview.md`, `api/overview.md`) — scope and summary diagrams
- Detail pages (individual controllers, components) — full documentation of one topic
- Reference pages (DTO definitions, table schemas) — exhaustive look-up material

At the **page level** (for complex pages only, defined as >3 H2 sections, >500 words, or multiple related concepts):
- Summary paragraph answers: what is this, who is it for, what can you do with it
- "At a glance" section with a compact summary table or diagram
- Full detail in subsequent sections
- Supplementary detail in collapsible `<details>` blocks (never collapse critical content like auth requirements or must-set config keys)

**Rationale:** Readers come to documentation with different goals. A developer onboarding needs orientation first. An architect needs diagrams. An operations engineer needs to find a specific configuration key quickly. Progressive disclosure serves all three without forcing any one of them to wade through content written for the others.

### 3.6 Iterative by design

The agent assesses existing documentation before generating new content. On every run it:

1. Reads existing doc files
2. Compares against current source
3. Decides: Skip (current), Extend (add missing), Update (fix inaccurate), or Create (new)
4. Applies only the required change — never rewrites valid content

This makes the agent safe to run repeatedly and allows incremental adoption (partial documentation is improved on each run, not discarded).

### 3.7 Session checkpoints

The agent works in phases and presents a checkpoint after each one, showing a table of files created / updated / skipped / flagged. The user reviews and approves before the next phase begins.

**Rationale:** Documentation generation is high-impact work. Checkpoints let the user catch wrong assumptions (e.g., incorrect component classification) before an entire phase of documentation is built on them.

### 4.2 Nav tab structure and rationale

The tab layout is identical for both single-principal and multi-principal repositories. What changes is the content depth inside each tab.

| Tab | Single-principal content | Multi-principal content |
|-----|--------------------------|------------------------|
| **Home** | Overview of the one Principal | Introduces all Principals in priority order; each has equal depth but rank 1 is first |
| **Getting Started** | Four pages for the one Principal | `overview.md` + per-Principal sub-sections |
| **Architecture** | Flat pages for the one Principal | Cross-component overview + per-Principal sub-sections |
| **Infrastructure** | Flat environment pages | Shared overview + per-Principal environment pages |
| **Reference** | Controllers + models for the one Principal; Supporting overflow | Per-Principal `reference/{name}/` sub-sections |
| **Components** | Supporting components (admin panel, CLI, functions) | Supporting components only — Principal components are NOT listed here |
| **Security** | Single `overview.md` + `posture.md` | Cross-component overview + per-Principal `overview.md` + `posture.md` |
| **DevOps** | Testing overview + single pipeline folder | Testing overview + per-Principal pipeline sub-folder |
| **Appendix** | ADRs, tools, legacy, dependencies | Same |

**Key ordering constraint:** Infrastructure always precedes Reference. Readers need deployment topology context (environments, base URLs, resource names) before navigating the API surface.

### 4.3 Reference section scope

The Reference section documents the API surface and data models of Principal components. It also absorbs reference detail from Supporting components when the Components section would otherwise become cluttered.

| Content | Where it goes |
|---------|--------------|
| Principal component controllers | `reference/{component}/controllers/` (multi-principal) or `api/controllers/` (single-principal) |
| Principal component domain models and DTOs | `reference/{component}/domain-models/`, `dtos/` |
| Supporting component controllers (if detailed) | `reference/{component-name}/controllers/` — proposed by agent, not automatic |
| Supporting component overview | Always stays in `components/{component-name}.md` |
| Tooling or Peripheral components | Never in Reference — Appendix only |

## 5. The Five-Phase Workflow (overview)

The agent works in five phases. Detailed criteria for each documentation area produced in each phase live in the companion area guides (see Section 0).

| Phase | Goal | Area guides that apply |
|-------|------|----------------------|
| **1 — Discovery** | Understand the entire repository before writing anything. Two-pass scan: manifest (cheap) → deep (priority-gated). Produces Component Registry and summaries for all documentation areas. | All area guides consume Phase 1 summaries |
| **2 — Foundation** | Produce structural skeleton: `index.md`, getting-started, architecture overview, components overview, configuration | [01.design-home.md](01.design-home.md) (home page), [02.design-getting-started.md](02.design-getting-started.md) (getting-started), [06.design-components.md](06.design-components.md) (components overview), [09.design-appendix.md](09.design-appendix.md) (ADRs, tools, legacy) |
| **3 — API and services** | Document the primary API surface, data layer, domain types, service contracts | [05.design-reference.md](05.design-reference.md) (controllers, data, domain models, configuration) |
| **4 — Cross-linking, infrastructure, security, deployment** | Logical and physical architecture, security posture, CI/CD pipelines; add cross-links; update nav | [03.design-architecture.md](03.design-architecture.md), [04.design-infrastructure.md](04.design-infrastructure.md), [07.design-security.md](07.design-security.md), [08.design-devops.md](08.design-devops.md) |
| **5 — Validation** | Structural checks (links, nav coverage, Mermaid syntax), content quality checks (endpoint completeness, accuracy sampling, config key cross-check) | All areas; produces Documentation Run Summary |

### Partial update invocation

When the user targets a specific area rather than running a full documentation run, the agent applies only the criteria from the relevant area guide:

| User says | Area guide applied | Phase 1 needed? |
|-----------|-------------------|-----------------|
| "Update docs for `UserController`" | [05.design-reference.md](05.design-reference.md) — API controllers | No |
| "Update the logical architecture" | [03.design-architecture.md](03.design-architecture.md) | No |
| "Document the infrastructure for Test environment" | [04.design-infrastructure.md](04.design-infrastructure.md) | No |
| "Update Admin tool docs" | [06.design-components.md](06.design-components.md) | No |
| "Run security posture analysis" | [07.design-security.md](07.design-security.md) | No |
| "Update deployment pipeline docs" | [08.design-devops.md](08.design-devops.md) | No |
| "Update the home page" / "Refresh index.md" | [01.design-home.md](01.design-home.md) | No |
| "Update getting started" / "Fix the prerequisites page" | [02.design-getting-started.md](02.design-getting-started.md) | No |
| "Document this project" (full run) | All area guides | Yes — full Phase 1 |

Nav update (`mkdocs.yml`) and nav coverage audit always run regardless of scope.

### Phase checkpoints

After each phase, the agent presents a table of files Created / Updated / Skipped / Flagged and waits for user approval before proceeding. This lets users catch wrong assumptions (e.g., incorrect component classification) before an entire phase of documentation is built on them.

## 6. Component Classification Strategy

### The four tiers

| Tier | Symbol | Identification criteria | Documentation depth |
|------|--------|------------------------|---------------------|
| **Principal** | 🔴 | Independently deployable; has its own CI/CD pipeline, configuration, and runtime identity; a first-class product deliverable | Full coverage across **all** doc sections — Getting Started, Architecture, Infrastructure, Reference, Security, DevOps |
| **Supporting** | 🟠 | Extends or complements a Principal component; admin panels, BFF layers, satellite services; has its own pipeline or deploys alongside a Principal | **Components** section: component page, API reference if applicable, configuration, deployment. May overflow detail to Reference section. |
| **Tooling** | 🟡 | Developer utility not shipped to production; CLI tools, generators, checkers; typically a console app with no deployment pipeline | Appendix: single page under `appendix/tools/` |
| **Peripheral** | ⚪ | Legacy code, deprecated modules, archived experiments; older target framework, no active pipeline, superseded by a newer component | Appendix: single page under `appendix/legacy/` |

### Principal component count — the critical fork

The number of Principal components determines the entire documentation structure. This decision is made at the end of Phase 1 Discovery and presented to the user for confirmation before any documentation is written.

| Principal count | Structure | Key behaviour |
|----------------|-----------|--------------|
| **1** | Flat — each section documents the one Principal directly | Standard structure (see Section 4.1 single-principal tree) |
| **2 or more** | Per-component sub-folders inside each major section | Multi-principal structure (see Section 4.1 multi-principal tree); every Principal gets equal depth in every section |

### Detection heuristics (applied in order)

1. **Entry point type** — hosted API/service → candidate for Principal or Supporting; console app → Tooling
2. **Deployment pipeline presence** — dedicated, independently-triggered CI/CD pipeline → strong signal for Principal
3. **Independent configuration** — own `appsettings.*.json` or environment-variable set, own secrets scope → strong signal for Principal
4. **Independent runtime identity** — own app registration, own Managed Identity, own Azure App Service / Container App → confirms Principal
5. **Project references and co-deployment** — referenced by a Principal project **and** always co-deployed with it, with no independent pipeline → Supporting. If it has an independent pipeline despite serving another Principal, apply heuristics 2–4 directly — it may still be Principal.
6. **BFF layers specifically** — a BFF is Supporting when it deploys as a sidecar alongside the API or admin panel it serves. It is Principal when it deploys independently to its own runtime with its own pipeline. The BFF's relationship to another Principal does not override the deployment criteria.
7. **Lifecycle signals** — older target framework, no recent pipeline activity, superseded by newer component → Peripheral

### Principal component ordering

When two or more Principals are identified, they must be ordered before documentation begins. The ordering is applied consistently to all nav entries, doc sub-folders, at-a-glance table rows, and home page reading paths.

**Ordering rules (apply in priority):**

1. **Primary deliverable first** — the component that is the core external-facing product of the repository ranks highest. This is typically the API or primary service that the other components exist to support or expose.
2. **Consumer before served** — a frontend or admin panel that directly consumes the primary API ranks second. It is a principal in its own right, but it exists in the context of the primary deliverable.
3. **Infrastructure-layer principals last** — a BFF or gateway that exists to serve another Principal component ranks after that component. Multiple BFFs (if present) are ordered after the frontend component they serve.
4. **Alphabetical as tiebreaker** — if two components have equal standing by the above rules, order them alphabetically.

**Concrete example:**

| Rank | Component | Classification signal | Ordering rationale |
|------|-----------|----------------------|-------------------|
| 1 | MicroserviceA | Core REST API, external consumers, own pipeline | Primary deliverable |
| 2 | Admin tool | Web UI with own pipeline, deploys independently | Consumer of MicroserviceA; its own product |
| 3 | Admin tool Bff | Own pipeline, own App Service, serves Admin tool only | Infrastructure layer for Admin tool |

This ordering is presented to the user at the Phase 1 checkpoint. Users may reorder at any time — the agent reorders all generated sub-folders and nav entries accordingly.

### Lifecycle stages

| Stage | Meaning | Documentation framing |
|-------|---------|----------------------|
| **Active** | Current production code | Default — no special markers |
| **Preview** | Under development, not yet in production | `!!! warning "Preview"` admonition |
| **Migrating** | Being replaced by a newer component | `!!! info "Migration in progress"` with pointer to replacement |
| **Deprecated** | Scheduled for removal | `!!! danger "Deprecated"` with sunset notice and replacement link |

### User confirmation required

Principal/Supporting classification is heuristic. Always present the classified Component Registry to the user **before** the deep scan. Include for each component: name, tier proposed, detection signals used. Users may override any classification.

When proposing multi-principal structure, show:
- The proposed Component Registry with tier and ordering for each principal
- A preview of the proposed documentation tree (sub-folder names in rank order)
- The ordering rationale for each principal

The user may reorder or reclassify principals before any files are created. The agent must reapply the new order throughout the entire proposed tree before confirming.

### Reference section overflow rule

When a Supporting component has enough API surface or data model complexity that fully documenting it inside the Components section would clutter that section, move its reference material to `reference/{component-name}/`. The Components section then carries a short overview page with a link to the reference detail. This is the agent's decision to propose — not apply automatically.

## 7. Security Documentation Strategy

### The core tension

Security documentation has two opposing requirements:
- **Readers need** to understand authentication and authorization to integrate with or operate the system
- **Attackers should not** find exploit paths, bypass conditions, or cryptographic weaknesses in public documentation

The agent resolves this through **information disclosure tiers**.

### Three-file model

| File | Audience | Content | In nav | Created when |
|------|----------|---------|--------|--------------|
| `security/overview.md` | All readers | Authentication schemes, authorization model, access control per component, sequence diagrams | ✅ Yes | Always (default run) |
| `security/posture.md` | All readers | Secrets management, CORS, HTTP headers, rate limiting, data protection, known configuration gaps — no exploit detail | ✅ Yes | Always (default run) |
| `security/posture.internal.md` | Security reviewers only | Authentication bypass paths, crypto weaknesses, attack vectors, exploit preconditions, full technical detail | 🚫 Never | Explicit user request only |

**Critical rule:** Internal file must never appear in `mkdocs.yml nav:`, never be linked from any public page, and its path must not appear in any public doc. When a security-sensitive finding is discovered during a standard run, write a `<!-- TODO: security-sensitive finding — {nature, no file path} -->` comment and surface it only in the phase summary.

### Authentication scheme ordering

Document authentication schemes in priority order:

1. **Recommended / primary scheme** first
2. **Current secondary schemes** second
3. **Legacy or deprecated schemes** last

Use a `Status` column: `✅ Recommended`, `✅ Current`, `⚠️ Legacy`.

**Rationale:** New integrations should use the recommended scheme. Presenting legacy schemes first misleads developers into implementing deprecated patterns.

### Legacy compatibility vs. security issues

Before writing a security admonition, classify the behavior:

| Type | Definition | Treatment |
|------|-----------|----------|
| **Legacy compatibility requirement** | Behavior exists because older clients require it; changing it breaks those clients | `!!! warning` with "**Legacy compatibility:** {reason}". Document migration path. Do NOT mark as a security issue. |
| **Known gap / accepted risk** | Known trade-off the team accepts | `!!! warning "... — known gap"`. Do not link to internal materials. |
| **True security issue** | Cryptographic weakness or externally exploitable vector | Surface-level `!!! warning` for observable behavior only; full detail in `posture.internal.md` on explicit request. |
| **Unverifiable assertion** | Depends on an external system (IdP tenant policy, network control, CI/CD platform) not visible in repository artifacts | `!!! warning "... — requires external verification"`. Add `<!-- TODO: verify against {system} -->`. Never write as confirmed finding. |

**General rule:** Never write a negative security assertion based on absence of evidence in repository files alone. A control that is not visible in code may be governed by an external platform.

### Admonition type selection

On MkDocs sites with `primary: red` theme color, `!!! note` renders as a red box — visually identical to `!!! danger` — conveying unintended severity. Always check `mkdocs.yml` for the `primary:` palette setting.

| Intent | Safe admonition type |
|--------|---------------------|
| Informational / how something works | `!!! info` |
| Design rationale or historical context | `!!! info` |
| Recommended action | `!!! tip` |
| Caution — requires care | `!!! warning` |
| Legacy compatibility constraint | `!!! warning` + "**Legacy compatibility:** ..." |
| Production MUST-NOT / MUST-FIX | `!!! danger` |
| General reference | **Avoid `!!! note`** on red-primary sites |

## 8. Visual Evidence Strategy

### Swagger UI screenshots

For every controller in the API Reference, capture a screenshot of the live Swagger UI before writing the controller doc. Embed it above the endpoint summary table.

**Purpose:** Screenshots confirm what is actually deployed (vs. what the code claims to expose), give readers an instant visual summary of all endpoints, and make drift between the documented table and the live API immediately visible.

**Freshness check before capture:** Check whether an existing screenshot is still current.

| Condition | Action |
|-----------|--------|
| No existing screenshot | Capture |
| Controller source changed since screenshot date | Recapture |
| Screenshot is >365 days old | Recapture (annual refresh) |
| Doc action is Create | Capture |
| No code changes + screenshot ≤365 days old | Skip — keep existing |

**Naming convention:** `{YYYYMMDD}-{controller-kebab}-swagger.png` (one file per controller; old file deleted when recaptured).

**Embed pattern in doc:**
```markdown
## Swagger reference

![{Controller Name} — Swagger UI snapshot, {YYYY-MM-DD}](../../assets/screenshots/api/{YYYYMMDD}-{controller-kebab}-swagger.png)
<!-- swagger-screenshot: source={url} date={YYYY-MM-DD} controller={ControllerClass} -->

> Live view: [{url}]({url}) (requires Test environment access)
```

**Swagger URL discovery:** Always discover from configuration (`appsettings.Test.json`, context files, or environment URLs). Never hardcode. If Swagger is disabled in configuration, present a non-silent user choice: (A) enable Swagger and redeploy, or (B) skip with a TODO placeholder.

### Application UI screenshots

For any frontend or web-UI component, capture a representative full-page screenshot. Embed it in the `## Overview` section of the component doc.

**Freshness:** Recapture every 180 days (bi-annual) or when component source or config changes.

**Naming convention:** `{YYYYMMDD}-{component-kebab}-app.png`

**Embed pattern:**
```markdown
<figure markdown>
  ![{Component Name} — {YYYY-MM-DD}](../../assets/screenshots/components/{YYYYMMDD}-{component-kebab}-app.png){ style="border: 1px solid #ccc; border-radius: 4px;" }
  <figcaption>Captured from {environment} after sign-in. Shows the <strong>{default route}</strong> home page.</figcaption>
</figure>
<!-- app-screenshot: source={url} date={YYYY-MM-DD} component={ComponentName} page={route} -->
```

**Authentication handling:** If the application shows a login page, attempt the full sign-in flow before capturing. If sign-in cannot complete (no stored credentials, MFA), capture the login page and record `page=login` in the provenance comment.

### Infrastructure icons

For infrastructure documentation pages, use official Azure Architecture SVG icons (from the Microsoft Architecture Icons pack). Store all icons locally under `docs/assets/icons/azure/` — never reference CDN URLs.

Apply icons in two patterns:

- **Pattern A (table rows):** `<img src="...svg" width="18" height="18" style="vertical-align:middle" alt="{resource type}"> **Resource Name**`
- **Pattern B (section headings):** `### <img src="...svg" width="22" height="22" style="vertical-align:middle" alt="{resource type}"> Resource Type`

If an icon file is unavailable, leave `<!-- TODO: azure icon missing — {resource-type} -->` and document in plain text.

## 9. Context Budget and Scalability

### The two-pass discovery design

Discovery is expensive. Reading all source files in Phase 1 would consume most of the agent's context budget, leaving too little for documentation phases. The two-pass design prevents this:

- **Pass 1** reads only metadata (folder names, project files, package references) — ~50–100 tokens per component
- **Pass 2** reads full source files, but only for 🔴 Principal and 🟠 Supporting components — 🟡 Tooling and ⚪ Peripheral are read at entry-point depth only

### Subagent delegation rule

When Phase 2 Pass requires reading **more than 10 files** for a single summary (e.g., a data layer with 20+ table scripts), delegate the task to a read-and-summarize subagent. The main agent receives a structured summary — it never sees the raw file content.

This pattern preserves context for documentation phases.

### Budget allocation targets

| Phase | Target share | Strategy |
|-------|-------------|----------|
| Phase 1 (Discovery) | ~30% | Two-pass scan; subagent delegation for large reads |
| Phase 2 (Foundation) | ~25% | One document at a time; release context after each |
| Phase 3 (API/Services) | ~25% | Same pattern |
| Phases 4–5 (Review/Validation) | ~20% | Cross-link checks; nav tree; validation report |

### Tier-limited runs for large repositories

For repositories with >10 components or >30 estimated pages, offer three run tiers:

| Tier | Scope | When to use |
|------|-------|-------------|
| **Full run** | All tiers | Standard documentation run |
| **Core + Supporting** | Omit Tooling and Peripheral | Faster; covers all production components |
| **Core only** | Primary API documentation only | Fastest; first pass on a new large repo |

For small repositories (<10 components, <30 estimated pages), proceed with a full run automatically.

## 10. Iterative Update Protocol

### Assessment criteria

For every existing doc file, check:

1. **Coverage** — does the doc cover all public classes / endpoints / methods in the source?
2. **Accuracy** — do described signatures, parameters, and return types match current code?
3. **Completeness** — are examples, error codes, and edge cases documented?
4. **Freshness** — are referenced versions, package names, and URLs still valid?

### Decision matrix

| Condition | Action |
|-----------|--------|
| Doc exists and matches current source | **Skip** — no changes |
| Doc exists but is missing new content | **Extend** — add missing sections, preserve existing |
| Doc exists with incorrect information | **Update** — fix inaccuracies, preserve structure |
| Doc exists but source was deleted/renamed | **Flag as orphaned** — ask user whether to remove or archive |
| No doc exists for a source file | **Create** — generate from source |

### Preservation rules

- **Never silently delete content.** When removing a block:
  - **Outdated / superseded** → move to `## Dismissed` section at the page bottom, under `### {Original section title}` with `> **Dismissed:** {reason} — YYYY-MM-DD`
  - **Provably wrong** (directly contradicted by source code) → delete and leave `<!-- REMOVED: contradicted by {source} on YYYY-MM-DD -->`
  - **Classification ambiguous** → ask the user
- **Mark updates** with `<!-- Updated: YYYY-MM-DD -->` comments near changed sections
- **Carry forward all content** not directly contradicted by current source — never reduce coverage

### Partial update workflow

When the user requests a targeted update (e.g., "Update docs for `UserController`"):

1. Resolve scope — map to specific source files and their doc files
2. Read source + existing doc
3. Apply assessment criteria and decision matrix
4. Create or update only the affected files
5. Verify links to and from modified files
6. Update `mkdocs.yml` if new files were created

No discovery phase needed for partial updates.

## 11. Nav Tree Governance

### The locked skeleton principle

The nav tab structure is defined in a template file (`doc-mkdocs-nav.template.yml`). The agent reads this template before writing `mkdocs.yml` and merges discovered pages into `{placeholder}` markers only.

**The agent may not:**
- Add top-level tabs
- Rename existing tabs
- Reorder sections

All structural decisions live in the template. This ensures the nav remains consistent across runs and agents.

### Placeholder syntax

```yaml
{pages: folder/}         # Insert all discovered .md pages under this folder
{page: path/file.md}     # Include only if this file exists
{optional: Label, folder/}  # Include sub-section only if folder is non-empty
```

Pages are inserted in Component Registry priority order (🔴 Principal → 🟠 Supporting → 🟡 Tooling → ⚪ Peripheral), not alphabetically.

### Nav coverage guarantee

**Every non-internal page produced or updated must appear in `mkdocs.yml nav:` before the run ends.** A page written to `docs/` but absent from nav is a documentation defect — treated the same as a broken link.

The validation phase enforces this automatically.

### Nav placement rules (invariants)

| Rule | Enforcement |
|------|-------------|
| Components (`components/`, `appendix/`) never appear under API Reference | Corrected and logged during nav update |
| `Infrastructure` tab always precedes `API Reference` | Enforced by template order |
| `security/posture.internal.md` never in nav | Removed if found; never added |
| No cross-references to any `*.internal.md` file | Verified in validation pass |

## 12. Implementing in a New Repository

### Step 1 — Create the agent file

Create `.github/agents/documentation-manager.agent.md` (or `.v1.agent.md` for versioned naming). The agent file must define:

- `name`, `description`, `tools`, `model` in YAML front matter
- **Role** — who the agent is, what principles it follows, which knowledge sources it reads
- **Boundaries** — explicit "Always Do" and "Never Do" sections
- **Workflow** — the five phases with step sequences
- **Invocation triggers** — "Document this project", "Update docs for X", etc.
- **Companion procedures** — SwaggerScreenshot, AppScreenshot (if browser tool available)

### Step 2 — Create the nav template

Create `.github/templates/doc-mkdocs-nav.template.yml` with the locked tab skeleton and placeholder syntax. Design the tabs for your documentation site before writing a single doc page — this ensures structural consistency from the start.

### Step 3 — Create the documentation structure template

Create `.github/templates/doc-documentation-structure.md` defining the full target folder/file tree under `docs/`. This template is the agent's reference for discovering what pages need to be created.

### Step 4 — Add companion instruction files

Create `.github/instructions/docmanager-standards.instructions.md`:
- Content rules (evidence-based, code snippets, tables, Mermaid)
- Progressive disclosure rules (structural + page-level)
- Diagram type selection rules
- Admonition type selection (with theme-color guidance)

Create `.github/instructions/docmanager-iterative-updates.instructions.md`:
- Assessment criteria
- Decision matrix (Skip / Extend / Update / Create)
- Preservation rules

Set `applyTo: docs/**` on both files so they auto-load when any `docs/` file is edited.

### Step 5 — Create per-page and per-component templates (optional but recommended)

Templates reduce per-page decision-making and ensure consistent structure across similar pages. Useful templates to create:

| Template | Used for |
|---------|----------|
| `doc-index.template.md` | `docs/index.md` — project overview with audience reading paths |
| `doc-infrastructure-overview.template.md` | `infrastructure/overview.md` — environments table, promotion diagram |
| `doc-security.template.md` | `security/overview.md` — auth schemes table, per-scheme structure |
| `doc-component-backend.template.md` | Core and Supporting backend component pages |
| `doc-component-frontend.template.md` | Core and Supporting frontend component pages |
| `doc-component-tool.template.md` | Tooling component appendix pages |
| `doc-component-legacy.template.md` | Peripheral/legacy component appendix pages |
| `doc-adr.template.md` | Architecture Decision Records |

### Step 6 — Seed context files (optional)

If the repository has domain knowledge that the agent cannot discover from code alone (external system names, business terminology, team conventions), create context files under `.github/context/`. The agent loads these as supplementary guidance during discovery.

Useful context files:
- `00.00-index.md` — master index with Quick Reference table (environments, URLs)
- Per-environment pages with resource names and URLs

### Step 7 — Initialize `mkdocs.yml`

Before the first documentation run, ensure `mkdocs.yml` exists with the `pymdownx.superfences` and `admonition` extensions configured. The agent requires these for Mermaid diagram rendering and admonition callouts.

## Key Design Decisions Summary

| Decision | Rationale |
|----------|-----------|
| Evidence-only documentation | Prevents stale or speculative content; makes every claim auditable |
| Four-audience design | Developers, architects, operations, and security reviewers have different needs; the doc structure addresses each explicitly |
| Priority tiers for components | Ensures Core documentation is complete even when context runs low |
| Two-pass discovery | Prevents discovery from consuming context needed for documentation phases |
| Infrastructure before API Reference in nav | Readers need deployment context before navigating the API surface |
| Reference section reserved for Principal components (+ Supporting overflow) | Keeps principal API surfaces legible without mixing in unrelated tooling routes; Supporting overflow is agent-proposed, not automatic |
| Three-file security model | Balances reader needs (auth understanding) against security hygiene (no exploit paths in public docs) |
| Legacy compatibility ≠ security issue | Prevents false positives that alarm teams about required behaviors they cannot change |
| Dated screenshot naming + provenance comments | Makes freshness checks machine-parseable; supports annual and change-triggered recapture |
| Nav template as locked skeleton | Ensures structural consistency across runs and agents; separates discovery logic from structural decisions |
| Preservation rules for existing content | Respects editorial work done by previous authors; prevents information loss from automated updates |
| Phase checkpoints with user approval | Prevents cascading errors from wrong component classifications or missed context |
