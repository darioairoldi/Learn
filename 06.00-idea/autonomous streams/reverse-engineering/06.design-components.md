## 1. Purpose and Audience

The Components section answers: **"What are all the pieces of this system, and how do I set up, use, or deploy each one?"**

It documents every component that is **not** the primary REST API host. This includes: admin panels, CLIs, Azure Functions, background services, BFF layers, worker services, and legacy/peripheral components.

| Audience | What they need |
|----------|---------------|
| Developer onboarding | How to run each component locally; what it does; how it relates to the API |
| Operations | How to configure and deploy each component; what environment variables it needs |
| QA | What endpoints the component exposes; how to test it |
| Architect | How components are connected; which components own which responsibilities |

## 2. Component Classification Recap

Components are classified into four priority tiers. The tier determines where the component is documented and how much detail is produced.

| Tier | Symbol | Description | Documentation location | Detail level |
|------|--------|-------------|----------------------|-------------|
| Core | 🔴 | The primary API host — the system's main external-facing interface | `api/` (API Reference tab) | Full, exhaustive |
| Supporting | 🟠 | Components that are part of the active product but are not the primary API | `components/{name}.md` | Full page with screenshot (if UI) |
| Tooling | 🟡 | Developer tools, CLIs, scripts, admin utilities | `appendix/tools/{name}.md` | Simplified summary |
| Peripheral | ⚪ | Legacy, deprecated, or rarely-used components | `appendix/legacy/{name}.md` | Minimal — redirect to replacement |

**The Components section documents only 🟠 Supporting components.** 🟡 Tooling and ⚪ Peripheral go to Appendix (see [07.design-appendix.md](07.design-appendix.md)).

### Tier detection heuristics

Use these signals to assign a tier during discovery:

| Signal | Likely tier |
|--------|-------------|
| Primary REST API with external consumers | 🔴 Principal |
| Frontend web application used by end users | 🟠 Supporting |
| BFF / backend-for-frontend serving the frontend | 🟠 Supporting |
| Admin panel with active use | 🟠 Supporting |
| Azure Function processing production events | 🟠 Supporting |
| CLI used by operators / developers only | 🟡 Tooling |
| Maintenance script / migration tool | 🟡 Tooling |
| Component marked `Deprecated` in README or comments | ⚪ Peripheral |
| `.NET Framework` project with no recent changes and a `Migrating` note | ⚪ Peripheral |

Present the tier assignments to the user at the Phase 1 checkpoint and wait for corrections.

### Lifecycle stages

Each component also has a lifecycle stage:

| Stage | Description | Documentation behavior |
|-------|-------------|----------------------|
| Active | Currently in production use | Full documentation |
| Preview | In development or beta | Document with `!!! warning "Preview"` admonition |
| Migrating | Being replaced by a newer component | Document the migration path; link to successor |
| Deprecated | Scheduled for removal | Minimal docs; `!!! danger "Deprecated"` + removal timeline |

## 3. Components Overview Page

**File:** `docs/components/overview.md`

### Criteria

1. **Component map diagram** — Mermaid `graph TD` or `flowchart` showing all 🟠 Supporting components and their relationships. Include 🔴 Principal components as nodes to show integration points, but do not re-document them.

2. **Component summary table** — one row per 🟠 Supporting component:

   | Component | Type | Stack | Status | Purpose | Detail |
   |-----------|------|-------|--------|---------|--------|
   | Admin Panel | Frontend | React + C# BFF | Active | Web UI for administrators | [→ admin-panel](admin-panel.md) |
   | CRL Checker | Azure Function | .NET 6 | Active | Validates CRL expiry | [→ crl-checker](crl-checker.md) |

3. **Cross-references** — link to `architecture/overview.md` for the full system view; link to `api/overview.md` for the primary API reference

4. **No documentation** for 🟡 Tooling and ⚪ Peripheral in this table — those appear in the Appendix

## 4. Per-Component Page Criteria

**File:** `docs/components/{component-name}.md`

### Source artifacts to read (per component)

| Artifact | What to extract |
|----------|----------------|
| `README.md` (if present) | Setup instructions, purpose description |
| Entry point (`Program.cs`, `index.tsx`, `App.tsx`, `Program.cs` of BFF) | Service registrations, startup behavior, port/URL |
| Package manifest (`package.json`, `.csproj`) | Tech stack, key dependencies, target framework |
| Route definitions (React Router, controller routes, Function triggers) | URL paths / trigger types exposed |
| `appsettings*.json` / `.env.example` | Configuration keys the component reads |
| Pipeline YAML for this component | How it is built and deployed |
| Docker / `docker-compose.yml` (if present) | Container image, port mappings, environment variables |

### 4.1 One-paragraph summary

Required. Answers: what is this component, who uses it, what does it do.

### 4.2 "At a glance" section (required for complex pages)

Include this for any component page with >3 H2 sections or >500 words. Use a summary table:

| Property | Value |
|----------|-------|
| Type | Frontend / Backend API / Azure Function / CLI |
| Tech stack | `{framework + version}` |
| Status | Active / Preview / Migrating / Deprecated |
| Port / URL (local) | `http://localhost:{port}` |
| Auth required | Yes — {scheme} / No |
| Deployment | {App Service / Container App / Azure Function / etc.} |

### 4.3 Tech stack table

| Technology | Version | Purpose |
|------------|---------|---------|
| React | 18 | Frontend UI framework |
| TypeScript | 5 | Type-safe JS |
| `{package}` | `{ver}` | {purpose} |

### 4.4 Setup and access

Instructions to run the component locally. Minimum:
- Prerequisites (Node.js version, .NET version, Docker, etc.)
- How to install dependencies (`npm install`, `dotnet restore`)
- How to start (`npm run dev`, `dotnet run`)
- Local URL and any required environment variables or secrets
- How to access / authenticate (if UI: how to log in)

Use code blocks with language identifiers for all commands.

### 4.5 Screenshot (UI components only)

If the component has a web UI, embed an application screenshot immediately after the purpose paragraph (or at the top of the "Setup and access" section):

```markdown
![{Component name} UI]({relative-path-to-screenshot})
<!-- app-screenshot: source=running-app date={YYYYMMDD} component={component-kebab} -->
```

See Section 5 for AppScreenshot Procedure and freshness rules.

**Components without a UI** (CLIs, Azure Functions, background services, BFFs): skip the screenshot silently. Do not add a placeholder or TODO.

### 4.6 Endpoints (BFF components only)

If the component is a BFF or backend service exposing HTTP endpoints, add an endpoint summary table:

| Method | Route | Description | Auth |
|--------|-------|-------------|------|
| GET | `/api/{route}` | {description} | {scheme} |

These endpoints are **NOT** duplicated into `reference/` — they stay in the component page only (the Reference section is reserved for 🔴 Principal components and approved Supporting overflow).

### 4.7 Configuration summary

A table of the component's configuration keys (from `appsettings*.json`, `.env.example`, or environment variables):

| Key | Type | Required | Default | Description |
|-----|------|----------|---------|-------------|

Do not show actual secret values — show Key Vault secret name or environment variable name.

### 4.8 Dependencies

Which other components or external services this component depends on:

| Dependency | Type | Purpose |
|-----------|------|---------|
| `{ApiDevice API}` | Internal component | Proxied via BFF |
| `{Azure Service Bus}` | Azure service | Event ingestion |
| `{External API}` | External HTTP | {purpose} |

Link to the dependency's own page if it is documented in this repository.

### 4.9 Deployment

Brief note on how the component is deployed:
- Which pipeline deploys it
- Target Azure resource type and name
- Environment differences (if significant)
- Link to `infrastructure/physical-architecture.md` for resource detail

## 5. Screenshot Rules

### AppScreenshot Procedure

Screenshots of running UI components are taken using the AppScreenshot Procedure defined in the v4 agent (`.github/agents/documentation-manager.v4.agent.md`). The procedure requires:

1. **A1** — Confirm the component is running (check port / URL from config)
2. **A2** — Navigate to the component's primary URL using the browser tool
3. **A3** — Wait for the page to fully load (no spinners / skeleton screens)
4. **A4** — Take a screenshot using `browser_take_screenshot`
5. **A5** — Save to `docs/assets/screenshots/components/{YYYYMMDD}-{component-kebab}-app.png`

### Freshness rules

| Trigger | Action |
|---------|--------|
| Screenshot file does not exist | Capture now |
| Screenshot is older than 24 months | Recapture |
| Component has had a major UI change since screenshot date | Recapture |
| User explicitly requests screenshot refresh | Recapture |
| Screenshot is less than 24 months old and no major UI changes | Skip — reuse existing |

Screenshots are bi-annual by default (recapture every 2 years). This is less frequent than API screenshots because UI components evolve more slowly and are more expensive to capture (requires a running instance).

### Screenshot naming

`{YYYYMMDD}-{component-kebab}-app.png` — e.g., `20250115-admin-panel-app.png`

Storage: `docs/assets/screenshots/components/`

## 6. Routing Between Components and API Reference

This is a frequently confused boundary. Apply these rules explicitly:

| Rule | Example |
|------|---------|
| 🔴 Principal component endpoints → `reference/{component}/controllers/` | `PublisherController.md` |
| 🟠 Supporting component endpoints → component page only | `components/admin-panel.md` (BFF routes in that page) |
| Component also has a UI → component page | Admin panel React app → `components/admin-panel.md` |
| Component is a CLI → Appendix tools | `appendix/tools/apidevice-cli.md` |
| Component is a legacy .NET Framework app → Appendix legacy | `appendix/legacy/apidevice-legacy.md` |

If a Supporting component exposes its own REST API that third parties also consume (unusual), add a `!!! tip` note in the component page pointing readers to the full endpoint list, but keep the endpoints in the component page — not in the `api/` folder.

## 8. Quality Gates

Before the Components section is considered complete, verify:

| Check | Pass condition |
|-------|---------------|
| All 🟠 Supporting components documented | One page per Supporting component in `components/` |
| Component map present | `components/overview.md` has a Mermaid diagram |
| Screenshots for all UI components | Every frontend component has a screenshot with provenance comment |
| No screenshots for non-UI components | CLIs, Functions, workers have no screenshot placeholder |
| Endpoint tables for BFFs | Every BFF component has an endpoint summary table |
| Config tables present | Every component has a configuration table |
| No primary API endpoints in component pages | PublisherController etc. are in `api/controllers/`, not `components/` |
| Tier assignments reviewed | User has confirmed component tier assignments at Phase 1 checkpoint |
| Source anchors present | Each page has `<!-- Source: ... -->` comments |
| Internal links valid | Links to `architecture/overview.md`, `infrastructure/physical-architecture.md`, `api/overview.md` resolve |
