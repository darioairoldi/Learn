## 1. Purpose and Audience

Getting Started serves **one primary goal**: take a developer who has never seen this project and get them to a running local instance in the shortest path possible — then direct them to the part of the documentation that is most relevant to what they do next.

The section is a **progressive learning sequence**, not a reference collection. Each page builds on the previous one. A reader who completes all pages has:

1. All required tools installed
2. The repository cloned, restored, and with secrets configured
3. The application running locally and responding to requests
4. A clear sense of where to go next for their specific goal

**Who this section is for:**
- Developers joining the project for the first time
- Contractors or external integrators onboarding to this API
- QA engineers setting up a test environment

**Who this section is NOT for:**
- Operations engineers deploying to production → `deployment/` pages
- Architects reviewing design → `architecture/` pages
- Security reviewers → `security/` pages

Getting Started provides enough context to start working. It does not replace the full reference documentation — it links to it at the right moment.

## 2. Source Artifacts to Read

| Artifact | What to extract |
|----------|----------------|
| Root `README.md` and per-component `README.md` files | Existing setup instructions, prerequisites, quick-start commands |
| All `*.csproj` files — `<TargetFramework>` | Minimum .NET SDK version required |
| `package.json` (frontend projects) — `engines` field | Minimum Node.js version |
| `docker-compose.yml` | Available Docker Compose services, whether it covers the full stack |
| All `Dockerfile` files | Whether containerized setup is possible |
| `appsettings.json` and `appsettings.*.json` | Which settings must be overridden locally |
| `.vscode/extensions.json` | Recommended VS Code extensions |
| `*.slnx` / `*.sln` | Solution name and which projects are included |
| `NuGet.Config` | Whether a private NuGet feed requires authentication |
| Any `.env.template`, `appsettings.template.json` | Template for secrets / local configuration |
| `launchSettings.json` | Application URLs and launch profiles for local development |
| Test project `*.csproj` files | Test runner and framework for the "run tests" step |
| `*.RunSettings` | Test run configuration parameters |

## 3. The Learning Sequence

Getting Started is always four pages in this order. No page may be omitted. Pages may be extended (additional content) but not reordered.

| Page | File | Answers | Reader state after |
|------|------|---------|-------------------|
| 1 — Prerequisites | `prerequisites.md` | What do I need to install and access? | Tools ready; accounts ready |
| 2 — Installation | `installation.md` | How do I get the code and configure secrets? | Code cloned; secrets configured; NuGet restored |
| 3 — Build and run | `build-and-run.md` | How do I start the application? | App running locally; verified working |
| 4 — Configuration | `configuration.md` | What can I change and how? | Knows how to adjust settings; knows where the reference lives |

**Why this order:**
- Prerequisites before Installation — you cannot restore NuGet packages without the SDK
- Installation before Build and Run — you cannot build without secrets
- Build and Run before Configuration — you need to see the app working before you can meaningfully change its behavior
- Configuration last — it is the "now deepen" page that transitions readers to the full `configuration/` reference

### Navigation between pages

Each page must end with a **"Next" link** pointing to the next page in the sequence. The last page (Configuration) ends with "Lift-Off Links" (see Section 7).

```markdown
---

**Next:** [Installation →](installation.md)
```

Each page (except Prerequisites) also has a **"Previous" link** at the top:

```markdown
← [Prerequisites](prerequisites.md)
```

## 4. Setup Track Detection

A repository may support more than one way to run locally. Detect available tracks from repository artifacts and document only the tracks that have enough support:

| Track | Detection criteria | Document? |
|-------|--------------------|-----------|
| **Local (full)** | `appsettings.Development.json` or equivalent exists; `launchSettings.json` present | Always — this is the baseline track |
| **Docker Compose** | `docker-compose.yml` exists and covers the full required stack (app + dependencies) | Yes, if `docker-compose.yml` is present |
| **Local with cloud services** | `appsettings.json` references Azure Key Vault, Azure SQL, or similar | Yes, with a note that Azure access is required |
| **Minimal (mocked)** | Project has in-memory database or fake services for testing | Yes, if the project explicitly supports this mode |

If multiple tracks are available, document the **simplest first** (usually Docker Compose if present, otherwise Local with mocked dependencies if available, then full Local).

### Presenting multiple tracks

Use tabs (MkDocs `pymdownx.tabbed`) to separate track-specific commands:

```markdown
=== "Docker Compose"
    ```bash
    docker compose up --build
    ```

=== "Local (.NET CLI)"
    ```bash
    cd src/ApiDevice.Api
    dotnet run
    ```
```

Only use tabs if there are genuinely two different command sequences. If both tracks share all steps except one, use a `!!! tip` admonition instead.

## 5. Per-Page Criteria

### 5.1 Prerequisites (`getting-started/prerequisites.md`)

**Opening line:** Must state what the reader will be able to do after this page: "After completing this page, you will have all the tools and access you need to install and run {ProductName} locally."

**Required sections:**

1. **Software requirements** — table of every required tool with minimum version and install link:

   | Tool | Minimum version | Install |
   |------|----------------|---------|
   | .NET SDK | `{version}` (from `<TargetFramework>` in `.csproj`) | [dot.net](https://dot.net) |
   | Node.js | `{version}` (from `engines` in `package.json` or from Docker base image) | [nodejs.org](https://nodejs.org) |
   | Docker Desktop | Any recent | [docker.com](https://docker.com) |
   | Git | Any recent | [git-scm.com](https://git-scm.com) |

   Add rows only for tools that are actually required by the repository. Do not list "nice to have" tools on this page — those go in a `!!! tip` admonition.

2. **Access requirements** — what accounts, permissions, or credentials are needed:
   - Azure subscription / resource group access (if the app connects to Azure at dev time)
   - App registration / Entra ID access (if OAuth is used locally)
   - Private NuGet feed credentials (if `NuGet.Config` references a non-public feed)
   - Database access (local SQL Server, Docker, or Azure SQL)

   For each requirement: describe *what it is* and *who to contact to get it* — never include actual credentials.

3. **Recommended VS Code extensions** (if `.vscode/extensions.json` exists):

   ```markdown
   If you use VS Code, install the recommended extensions when prompted, or run:
   ```bash
   code --install-extension {extension-id}
   ```
   ```

4. **Success indicator** — how the reader knows they are ready: "You are ready to proceed when you can run `dotnet --version` and see `{min-version}` or higher."

### 5.2 Installation (`getting-started/installation.md`)

**Opening line:** "After completing this page, you will have the repository cloned, dependencies restored, and local secrets configured."

**Required sections:**

1. **Clone the repository:**

   ```bash
   git clone {repo-url}
   cd {repo-folder}
   ```

2. **Restore dependencies:**

   ```bash
   dotnet restore
   ```

   If there are frontend projects:

   ```bash
   cd {frontend-folder}
   npm install
   ```

   If `NuGet.Config` has a private feed: add a note about authentication before the restore command.

3. **Configure local secrets** — the most critical and most commonly forgotten step. Cover:
   - How local secrets are managed (`.NET User Secrets`, `.env` files, `appsettings.Development.json`)
   - The minimal set of secrets that must be configured before the app starts
   - Where to find the actual values (Key Vault, team contact, `.env.template`) — never the values themselves
   - Command to initialize user secrets if the project uses them:
     ```bash
     cd src/{ApiProject}
     dotnet user-secrets set "{Key}" "{value}"
     ```

4. **Database setup** (if applicable):
   - How to run migrations locally
   - Whether a Docker container is needed for the database
   - Success indicator: "You'll know this worked when…"

5. **Success indicator** — what the reader should see after this step is complete.

### 5.3 Build and run (`getting-started/build-and-run.md`)

**Opening line:** "After completing this page, the application will be running locally and you will have verified it responds to requests."

**Required sections:**

1. **Build:**

   ```bash
   dotnet build
   ```

   Note any expected warnings (do not let the reader be confused by benign build warnings).

2. **Run** — per available track (see Section 4):

   For each 🔴 Principal component and each 🟠 Supporting component that must run for the system to be functional, one sub-section with:
   - The run command
   - The URL it serves at (from `launchSettings.json`)
   - Any startup log message that confirms it is running

   Example:
   ```markdown
   ### API Host

   ```bash
   cd src/ABB.Ability.ApiDevice.Api
   dotnet run
   ```
   
   Application starts at `https://localhost:44300`. You'll see:
   ```
   info: Microsoft.Hosting.Lifetime[14]
         Now listening on: https://localhost:44300
   ```

3. **Verify** — the most important step. Provide a concrete verification command or URL:

   ```bash
   curl https://localhost:44300/health
   # Expected: HTTP 200
   ```

   Or: "Open `https://localhost:44300/swagger` in your browser. You should see the Swagger UI."

4. **Running tests:**

   ```bash
   dotnet test
   ```

   Note the expected pass/fail count if known (do not invent it — only note it if the test count is stable and verifiable from the repository).

5. **Common issues** — minimum 3 entries. Source these from `README.md` troubleshooting sections and common failure modes visible in the code (e.g., port conflicts, database connection string not set, certificate trust):

   | Symptom | Cause | Fix |
   |---------|-------|-----|
   | `Failed to bind to address https://localhost:44300` | Port in use | Change port in `launchSettings.json` or stop the conflicting process |
   | `KeyVaultException: certificate not found` | User secrets not configured | Complete [Installation](installation.md) step 3 |

6. **Success indicator** — the definitive proof that everything is working.

### 5.4 Configuration (`getting-started/configuration.md`)

**Opening line:** "This page explains how to change application settings and where to find the complete configuration reference."

This page is intentionally **brief** — it is the on-ramp to the full `configuration/settings-reference.md`, not a replacement for it.

**Required sections:**

1. **Configuration files** — which files are used per environment:

   | Environment | File | How to activate |
   |-------------|------|----------------|
   | Development | `appsettings.Development.json` | Set `ASPNETCORE_ENVIRONMENT=Development` |
   | Test | `appsettings.Test.json` | Set `ASPNETCORE_ENVIRONMENT=Test` |

2. **Override hierarchy** — briefly explain which setting source wins (e.g., environment variables override `appsettings.json`). One paragraph or a simple priority list.

3. **Minimum required keys** — the 3–5 settings that must be set before the app starts. Link to the full reference for the complete list:

   | Key | Purpose | How to set locally |
   |-----|---------|-------------------|
   | `{Section:Key}` | {purpose} | User secrets or `appsettings.Development.json` |

   "For the complete list of all settings and their meanings, see [Configuration reference](../configuration/settings-reference.md)."

4. **Feature flags** (if the project has them) — one sentence + link: "This project uses feature flags to control {what}. See [Feature flags](../configuration/feature-flags.md)."

5. **Lift-Off Links** (see Section 7).

## 6. Success Indicators

Every page in Getting Started must end the main instruction sequence with a **success indicator** — a concrete, verifiable statement that tells the reader they completed the page correctly.

**Criteria for a good success indicator:**
- It is **observable** — the reader can verify it without interpretation
- It is **specific** — not "the app works" but "you see HTTP 200 when calling `/health`"
- It uses a **command or URL** where possible

**Patterns:**

```markdown
<!-- Command output -->
You'll know this worked when `dotnet --version` returns `{version}` or higher.

<!-- HTTP response -->
You'll know the application is running when `https://localhost:{port}/health` returns HTTP 200.

<!-- Log message -->
You'll know the database connection is working when you see:
info: Microsoft.EntityFrameworkCore.Database.Command[20101]
      Executed DbCommand (1ms) ...

<!-- File existence -->
You'll know secrets are configured when the application starts without throwing `KeyNotFoundException`.
```

## 7. Lift-Off Links — Exiting to Deeper Documentation

The last page of Getting Started (Configuration) must close with a **Lift-Off Links** section — a curated set of "where to go next" links that match the three primary reader paths from the home page.

These links do not repeat the home page reading paths — they are the continuation of where the developer *specifically* needs to go after the setup is complete.

```markdown
## Where to go next

Now that the application is running locally, continue to the section most relevant to your goal:

| Goal | Start here |
|------|-----------|
| Understand the API endpoints | [API overview](../api/overview.md) |
| Understand the system design | [Architecture overview](../architecture/overview.md) |
| Set up for deployment | [Deployment overview](../deployment/overview.md) |
| Review all settings | [Configuration reference](../configuration/settings-reference.md) |
| Understand the admin panel | [Admin Panel overview](../components/admin-panel/overview.md) |
```

**Rule:** Every link in the Lift-Off table must resolve to an existing doc page. If a target page does not exist yet, omit that row rather than creating a broken link.

## 9. Quality Gates

Before the Getting Started section is considered complete, verify:

| Check | Pass condition |
|-------|---------------|
| All four pages present | `prerequisites.md`, `installation.md`, `build-and-run.md`, `configuration.md` exist |
| Page order preserved | Pages are ordered 1→4 in nav; each page links to the next |
| Opening line on each page | Each page starts with "After completing this page, you will…" |
| Software versions factual | All tool versions are sourced from `.csproj`, `package.json`, or Docker files — not guessed |
| No actual secret values | No passwords, tokens, or connection strings in any command example |
| Success indicator on each page | Every page ends with a `!!! success "You are ready when…"` block |
| Build command is copy-pasteable | Every command block can be run exactly as written from the project root |
| Docker Compose track documented | If `docker-compose.yml` exists, it is documented as an alternative track |
| Common issues populated | Build-and-run page has at least 2 common issues with solutions |
| Lift-off links on last page | Configuration page ends with a "Where to go next" table |
| All lift-off links resolve | No broken links in the Lift-Off table |
| Setup tracks match repo capabilities | Only tracks documented are tracks the repository actually supports |
