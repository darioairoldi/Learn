## 1. Purpose and Audience

The Appendix section is a **reference area** — it contains information that does not fit into the main documentation tree but must be preserved and accessible.

| Sub-area | Answers | Audience |
|---------|---------|---------|
| ADRs | "Why was this decision made?" | Architects, new developers |
| Tools | "How do I use this tool?" | Developers, operations |
| Legacy | "What happened to {component}?" | Developers inheriting old integrations |
| Dependencies | "What libraries does this project use?" | Security reviewers, architects |

The Appendix is **not** a dumping ground — only include content that has no other logical home.

| Artifact | What to extract |

## 3. Architecture Decision Records (ADRs)

### ADR index page (`appendix/adr/index.md`)

Criteria:
1. **ADR inventory table** — every ADR found, sorted by date:

   | ID | Title | Date | Status | Supercedes |
   |----|-------|------|--------|-----------|
   | `ADR-001` | [Use HMAC authentication for device API](adr-001.md) | 2023-01-15 | Accepted | — |
   | `ADR-002` | [Migrate to Clean Architecture](adr-002.md) | 2024-03-01 | Accepted | — |

   Status values: **Proposed** / **Accepted** / **Deprecated** / **Superseded**

2. **Date sort** — newest first in the table, but ADR IDs are sequential and immutable
3. **Deprecated/Superseded ADRs** stay in the index — never remove them

### Per-ADR page (`appendix/adr/{id}-{slug}.md`)

If an ADR file already exists in the repository, copy it to `appendix/adr/` without changing its content — ADRs are immutable records. If the file is not already in Markdown format, convert to Markdown preserving all content.

If no ADRs exist, create the index page with an empty table and a note:

```markdown
!!! info "No ADRs found"
    No Architecture Decision Records were found in this repository.
    To add one, create a file following the template in [ADR template](adr-template.md).
```

### ADR template (for reference only — do not create unless no ADRs exist)

```markdown
# ADR-{NNN}: {Title}

**Date:** YYYY-MM-DD  
**Status:** Proposed | Accepted | Deprecated | Superseded by ADR-{NNN}  
**Deciders:** {Names or roles}

## Context

{Describe the problem or the forces at play.}

## Decision

{Describe the decision made.}

## Rationale

{Why this decision was made over alternatives.}

## Alternatives considered

| Alternative | Why not chosen |
|-------------|---------------|
| {alt-1} | {reason} |

## Consequences

**Positive:** {outcomes}  
**Negative / Trade-offs:** {trade-offs}

## References

- {Link to relevant PR, issue, or discussion}
```

## 4. Tools (Tooling Tier)

🟡 Tooling-tier components are stand-alone developer tools, scripts, or CLI utilities that support the development or operations workflow. They are not deployed as part of the production system.

See `04.design-components.md` for the full tier classification. Examples: `MicroserviceA.Cli`, `ApiDocGen`, migration scripts.

### Per-tool page (`appendix/tools/{tool-name}.md`)

Each tooling-tier component gets a **single page** (not a subfolder). Required sections:

1. **What it does** — one paragraph: purpose, when to use it, what problem it solves
2. **Install / setup** — commands to build or install

   ```bash
   cd {tool-folder}
   dotnet build
   ```

3. **Commands reference** — table of available commands/subcommands:

   | Command | Description | Required args | Optional args |
   |---------|-------------|--------------|--------------|
   | `{tool} {command}` | {what it does} | `--arg1 <value>` | `--arg2` |

4. **Configuration** — which config file or environment variables are needed; link to a sample (`appsettings.template.json` if it exists)
5. **Example output** — a representative invocation with sample output (sanitized — no real connection strings or keys)

### Tools overview page (`appendix/tools/index.md`)

If there are 2 or more tooling-tier components, create an index:

| Tool | Purpose | Quick command |
|------|---------|--------------|
| [MicroserviceA.Cli](microservicea-cli.md) | Manage device registrations from the command line | `microservicea-cli devices list` |
| [ApiDocGen](ApiDocGen.md) | Generate device documentation from API metadata | `ApiDocGen --config config.json` |

## 5. Legacy and Deprecated Components

⚪ Peripheral-tier components are deployed but not actively developed, or have been deprecated and replaced. They still need documentation so that developers maintaining old integrations know what they are.

### Per-legacy-component page (`appendix/legacy/{component-name}.md`)

Required sections:

1. **What it was** — one paragraph describing what this component did and who used it
2. **Current status** — one of:
   - **Deprecated** — still running but no longer actively developed; do not build new integrations against it
   - **Decommissioned** — shut down; no longer accessible
   - **Superseded** — replaced by `{new-component}` on `{date}`
3. **Migration path** — if superseded, how to migrate:

   | Old | New | Notes |
   |-----|-----|-------|
   | `{old-endpoint}` | `{new-endpoint}` | {breaking changes if any} |

4. **Replacement pointer** — link to the replacement component's page
5. **Known callers** — list known systems or components that call this legacy component (from code references or docs). This helps track who still depends on it.

### Legacy index page (`appendix/legacy/index.md`)

| Component | Status | Superseded by | Date |
|-----------|--------|--------------|------|
| `{component}` | Deprecated | [{new}]({link}) | YYYY-MM-DD |

## 6. Dependency Inventory

### `appendix/dependencies.md`

This page gives a snapshot of third-party dependencies across the solution — primarily for security auditing and license compliance.

Criteria:
1. **Scope** — document production dependencies only. Exclude test-only packages from the primary table (list them in a separate collapsible section).

2. **Per-project dependency tables** — one table per project:

   ```markdown
   ## {ProjectName}

   | Package | Version | License | Purpose |
   |---------|---------|---------|---------|
   | `{PackageName}` | `{version}` | {MIT / Apache-2.0 / etc.} | {one-line purpose} |
   ```

3. **License column** — determine from the package's NuGet/npm page. If unknown, write "Unknown — verify" and flag with `<!-- TODO: verify license -->`.

4. **Purpose column** — explain what the package is used for in this project specifically (not a generic description of the library).

5. **Collapsible test dependencies** — wrap test project packages in a `<details>` block:

   ```markdown
   <details>
   <summary>Test dependencies</summary>
   ...
   </details>
   ```

6. **Snapshot date** — add a note at the top:

   ```markdown
   !!! info "Snapshot"
       Dependency list generated on YYYY-MM-DD. Run `dotnet list package` or `npm list` to get the current state.
   ```

### What NOT to document

- Internal project-to-project references (those are covered in architecture documentation)
- The .NET framework itself (only third-party packages)
- Transitive dependencies (only direct / top-level packages)

```markdown
# {Tool name}

<!-- Source: {tool-folder}/README.md and {tool-folder}/*.cs entry point -->

{One-paragraph description.}

```bash
cd {tool-folder}
dotnet build
```

## Commands

| Command | Description | Required | Optional |
|---------|-------------|---------|---------|
| `{command}` | {description} | `--arg` | `--flag` |

## Configuration

Copy `appsettings.template.json` to `appsettings.json` and fill in:

| Key | Description | Example |
|-----|-------------|---------|
| `{Section:Key}` | {description} | `{example-value}` |

## Example

```bash
{tool-name} {command} --arg {value}
# Output:
# {expected output}
```
```

## 8. Quality Gates

Before the Appendix section is considered complete, verify:

| Check | Pass condition |
|-------|---------------|
| ADR index exists | `appendix/adr/index.md` exists with a table (may be empty if no ADRs found) |
| Existing ADRs preserved | No ADR was modified during the documentation run |
| Tools index exists (if > 1 tool) | `appendix/tools/index.md` exists listing all tooling-tier components |
| Per-tool pages present | One page per 🟡 tooling-tier component |
| Legacy index exists (if any) | `appendix/legacy/index.md` lists all ⚪ peripheral-tier components |
| Legacy pages have migration paths | Every superseded component links to its replacement |
| Dependencies page present | `appendix/dependencies.md` exists with at least one table |
| No hardcoded secrets in tool examples | All example commands use placeholder values |
| License column populated | All packages have a license value or a `<!-- TODO -->` marker |
| Source anchors present | All pages have `<!-- Source: ... -->` comments |
