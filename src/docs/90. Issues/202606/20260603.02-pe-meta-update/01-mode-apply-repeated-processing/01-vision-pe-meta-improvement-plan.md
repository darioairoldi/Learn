# Vision amendment — always plan-then-execute + orthogonal `--plan-file` and reconcile/trust/fresh modes

> Source analysis: [01-overview.md](01-overview.md) § "Refinement: make `--mode apply` ALWAYS plan-then-execute", extended by the in-conversation design of the orthogonal `--plan-file` parameter and the baseline-availability execution model.
> This plan amends a **vision document**; per the vision boundary *"MUST NOT be modified by autonomous processes — vision changes are human-only"*, the vision edits in this plan are **proposals requiring human consent before execution** (see § Human consent).

## Goal

**Verbatim trigger (this turn):** *"please review and update `01-vision-pe-meta-improvement-plan.md` to integrate whatever discussed up to now so that pe-meta artifacts support effective, reliable and efficient `--mode plan`/`apply` behavior."*

**Verbatim trigger (originating analysis):** *"in case of `/pe-meta-update --mode apply` (no plan) we could anyway generate the plan and then run it immediately — this would bring more consistency between the two paths, comparable costs, and the generated plan would always be available for analysis."*

**Verbatim trigger (surface refinement):** *"provide support for `--plan-file` option; in case not present, the workflow can use a default readable and understandable filename … Of course in this case the vision document could be changed explaining eight parameters are supported instead of 7."*

**Verbatim trigger (reconcile semantics):** *"the existing previously created plan should be the base for the assessment. The plan can be improved of course but existing decisions should be analyzed and considered for actionability, and goal coverage … a just-generated plan (same conversation) must be considered as in case of an existing plan."*

The change has two orthogonal halves that together make `--mode plan`/`apply` effective, reliable, and efficient:

1. **Control-flow unification (output side).** `--mode apply` becomes **materialize plan → execute plan** (one execution engine, shared with plan consumption). The plan file is the single pivot artifact every mutating run passes through. `apply = plan + execute`; `plan = apply minus execute`.
2. **Plan-artifact surface (input side).** Add an **eighth canonical parameter `--plan-file`** that governs *only* the plan artifact's location/identity. Whether the plan is **regenerated or trusted** is governed by the *existing* `--skip research` parameter — never by file existence. The two booleans **(baseline available?) × (research runs?)** resolve to three named modes: **fresh**, **reconcile**, **trust**. A plan generated earlier *in the same conversation* counts as an available baseline, so it reconciles rather than regenerates.

The reasoning→standard model seam is pinned to the plan→execute boundary; the drift guard is required **only** in the cross-run trust mode (skip research with no same-run re-validation).

## Why this change (one-paragraph rationale)

Today `plan` and `apply` are two independent pipelines that both start at Phase 1 research; `apply` has no plan-consumption path, so the two routes can diverge in result and cost, and there is no way to point at a plan or to reuse one. Making `apply` *always* materialize a plan and then execute it collapses the control flow to one rule (`apply = plan + execute`), and adding the orthogonal `--plan-file` parameter — governed for regenerate-vs-trust by the *existing* `--skip research` — gives a clean, predictable two-boolean execution model: **fresh** (no baseline, research runs), **reconcile** (baseline available, research runs, decisions preserved), **trust** (baseline available, research skipped, drift-guarded). This is **effective** (one engine, no path divergence), **reliable** (a reviewable plan lands on every mutating run; human decisions are never silently overwritten; the loop converges instead of thrashing), and **efficient** (reconcile reuses prior valid rows; trust skips research entirely; the model seam runs execution on a cheaper model every run). The hard preconditions are execution-ready plan precision (item 6) and the cross-run drift guard (item 6) — without them a cheaper executor applying a plan cold is the one real quality risk.
