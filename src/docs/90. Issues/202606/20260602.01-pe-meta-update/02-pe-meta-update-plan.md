# Plan — pe-meta artifact conformance to vision v15.2

> **Authority:** [20260531.01-vision.v15.md](../../../../../06.00-idea/self-updating-prompt-engineering/20260531.01-vision.v15.md) (v15.2.0, 2026-06-03).
> **Source analysis:** [overview.md](overview.md) (Findings A–E).
> **Reference implementation:** [pe-meta-update.prompt.md](../../../../../.github/prompts/00.09-pe-meta/pe-meta-update.prompt.md) already conforms — every step below replicates its established pattern.

## 🎯 Goal

Bring the pe-meta artifact set (prompts, parser tests, applicability matrix, agents) into **full conformance with vision v15.2**, so that the v15.2 cross-family contracts are honored uniformly per the P0 `command-family-agnostic` principle, and so that no artifact contradicts the vision's own option-applicability model.

A second objective is added (Group 5): make **incremental processing correct and complete** by tracking processing state along the two axes that actually vary between passes — the **input axis** (which source version/timestamp was observed) and the **processing axis** (which dimension was assessed, against which source version, at what depth). A single per-source watermark conflates these two and cannot promise at-least-once coverage when a prior pass narrowed `--dim` or `--scope`. Group 5 therefore (a) defines the principles and metadata that make the state model unambiguous, (b) specifies the **full**, **incremental**, and **bounded-delta** invocation models against that state, and (c) extends `--start`/`--end` to a **value-shape window** (date **or** source-version) so incremental re-processing is expressible across both time and versions through the existing window parameter — no new flag. Manual default stays **full** per the P0 `default-full-investigation` principle; a `--start`/`--end` window is a *subtractive user opt-in* (the user signals the narrow window — the system never narrows on the user's behalf), and within that window it **overrides recorded coverage** (reprocesses even already-`pass` PUs) so a distrusted past pass can be re-baselined. This objective drives a companion vision amendment (v15.3) whose items are sequenced as prerequisites inside Group 5, and a paired update to the affected use-cases.

**In scope:** pe-meta prompts, `pe-meta-option-parser-tests.md`, `pe-meta-option-applicability-matrix.md`, pe-meta agents, dismissal of the redundant `pe-meta-release-monitor` prompt; the incremental-processing capability (value-shape `--start`/`--end` accepting a date or a source-version bound; the two-axis processing-state model — source ledger + per-artifact per-dimension coverage metadata; the full/incremental/bounded-delta invocation models; time- and version-keyed delta detection across platform / model / ecosystem sources) with the companion vision v15.3 amendment; and the **use-cases affected by the new state model** (freshness + trigger-driven use-cases plus their `usecase-index.json` entries).
**Out of scope (parked):** full use-case catalog regeneration unrelated to the state model, and the `00-overview.md` use-case-list omission fix — broad catalog hygiene, not state-model work.

### Group 5 — Incremental processing: two-axis coverage state, value-shape `--start`/`--end` (time- and version-bounded), full/incremental/bounded-delta invocation models

Today the incremental filter keys off a single per-source watermark (`last_review_timestamp`). That watermark conflates two independent variables: **which source version was observed** (input axis) and **what processing was applied to it** (processing axis — which `--dim`, at what depth). When a prior pass narrowed `--dim` or `--scope`, the watermark reads "done" while real `(artifact × dimension)` coverage gaps remain — so it cannot promise the "every processing performed at least once" guarantee, and a pure time window also mis-detects (a docs typo bumps the page hash and re-triggers; a mid-cycle run brackets awkwardly).

This group replaces the scalar watermark with a **two-store, two-axis processing-state model**, defines the **principles and metadata** that make it unambiguous, specifies the **full / incremental / bounded-delta** invocation models against it, extends `--start`/`--end` to a **value-shape window** (date **or** source-version) so re-processing can be bounded across time *and* versions without a new flag, and unifies the two parallel state stores (machine `triggers/<source-id>.json` vs the hand-maintained *Last Processed Versions* table). Manual `full` re-baselines every PU; a `--start`/`--end` window re-baselines every PU **inside the window**, overriding recorded coverage so a distrusted past pass can be re-run.

**Design reference — the two-axis state model (the spec items 5.4–5.10 implement):**

| Store | Location | Axis it records | Keyed by | Fields |
|---|---|---|---|---|
| **Source ledger** | `<state.path>/triggers/<source-id>.json` | Input (what the world published, what we last observed) | `source-id` | `source_id`, `version_scheme`, `last_seen_version`, `last_seen_timestamp`, `last_digest_hash` |
| **Artifact coverage record** | each artifact's bottom validation-metadata block | Processing (what assessment was applied) | `artifact × dimension` | per dim: `source_versions: {<source-id>: <ver\|ts>}`, `depth: research\|screening\|deep`, `status: pass\|fail\|partial\|never`, `last_run` |

A processing unit (**PU**) is one `(artifact × applicable-dimension)`. A PU is **stale** when `status == never` (coverage gap — never assessed) **OR** `status != pass` (prior fail/partial) **OR** any source dependency's ledger `last_seen_version`/`last_seen_timestamp` is newer than the PU's recorded `source_versions[dep]` (input advanced). The intersection of the two stores — not a single scalar — is the coverage matrix.

**Two new vision principles (P1) this group adds (see § Principles to add):**

- `coverage-completeness-guarantee` — incremental assessment guarantees every applicable PU is assessed at least once; it skips only PUs already current, and a narrowed prior pass MUST NOT create a permanent coverage gap. `relies_on: [validation-caching, deterministic-processing]`.
- `processing-state-is-multidimensional` — processing state is recorded along the input axis (source ledger) and the processing axis (per-artifact per-dimension coverage), never collapsed into a per-source scalar; the source ledger is a rebuildable cache, the per-artifact coverage metadata is the durable, self-describing record. `relies_on: [metadata-driven, deterministic-processing]`.

> **`relies_on` discipline ([vision-frontmatter.instructions.md](../../../../../.github/instructions/vision-frontmatter.instructions.md) v1.3.0).** `relies_on` lists **rationale IDs only** (entries resolving to a body `## ⚙️ The rationale` block) — never principle IDs. Principle-to-principle links (`coverage-completeness-guarantee` ↔ `active-dimensions-follow-evidence`; `processing-state-is-multidimensional` ↔ `single-source-of-truth`) are therefore expressed in prose via inline code inside the body principle subsections, NOT in `relies_on`.

**Invocation models this group specifies (see § Invocation models to add to the vision § Assess research-phase table).** All three process PUs whose source-input is newer than a **watermark**; they differ only in where the watermark comes from and whether recorded coverage is trusted:

- **Full** — manual, no params. Watermark = beginning-of-time; recorded coverage **ignored**. Work set = every in-scope artifact × every applicable dimension. On completion: write each processed PU's coverage cell (`source_versions` = current latest, `depth` = deep, `status`) and advance the source ledger. Full is the at-least-once re-baseline. (The cost gradient — `validation-caching` + Tier-1 screening — keeps already-current PUs near-zero cost, so full is the cheap manual catch-up path too.)
- **Incremental** — trigger-fired. Watermark = each PU's **recorded** `source_versions` (the implicit per-PU coverage state); recorded coverage **trusted**. Work set = only **stale** PUs (per the staleness rule above). On completion: update only processed cells and advance the ledger for sources whose deltas were consumed. Guarantees at-least-once (never-covered PUs always included) and no-redundant (current PUs skipped). New artifacts and newly-introduced dimensions surface automatically (their cells are `never`).
- **Bounded-delta** — any caller with `--start`/`--end` (date **or** source-version, resolved to a timestamp window at Phase 0a via the source's `version_scheme`). Watermark = the **explicit** `--start` bound; recorded coverage is **overridden inside the window** — every PU whose source-input falls in `[--start, --end]` is reprocessed *regardless of* `status == pass`. This is the distrust-recovery / re-baseline path ("re-run release 1.099→now even though it was already processed"). A version-shaped bound requires a single resolvable `--source`; `version_scheme: none` sources reject version bounds. Writes coverage cells the same way, so it composes with full and incremental; the at-least-once `never` rule still applies.
