# PE-meta improvement plan for full vision and use-case compliance

## 🎯 Objective

Align PE-meta command contracts and prompt wording so design and review share the same quality objective and scope intent, while process differences remain explicit, deterministic, and role-correct, and command options are normalized for consistency and flexibility.

## 📌 Context baseline

The issue analysis confirms directional alignment with the vision and use cases. Remaining work is contractual normalization across command families and type-specific prompt definitions, especially where option ownership currently overlaps or is too rigid.

## 🏗️ Workstream A: design and review parity normalization (✅ done)

- Normalize goal wording across all `pe-meta-{type}-design` and `pe-meta-{type}-review` prompts. (✅ done)
  Done (2026-05-22): All 8 design/review type pairs now use shared quality-objective and scope-intent goal wording.

- Normalize scope intent wording across all type pairs. (✅ done)
  Done (2026-05-22): Added a shared scope-intent cover item across all type-specific design/review prompt frontmatter blocks.

- Keep process-role differentiation in process and boundaries sections only. (✅ done)
  Done (2026-05-22): Added explicit parity boundary lines in every pair while preserving existing role-specific process sections (design = construction-capable, review = analysis-first/read-only).

## ⚙️ Workstream B: capability-based option applicability model (✅ done)

- Build a canonical option matrix for all PE-meta commands. (✅ done)
  Done (2026-05-22): Added `.github/prompts/00.09-pe-meta/pe-meta-option-applicability-matrix.md` with global, capability-gated, command-specific, and alias routing classifications.

- Normalize globally meaningful options across command families. (✅ done)
  Done (2026-05-22): Normalized shared semantics and deterministic rejection wording in top-level PE-meta prompts for `--dim`, dependency flags, mode controls, and research/preview/scope controls.

- Replace hard ownership rules with capability ownership rules. (✅ done)
  Done (2026-05-22): Updated core command contracts to capability-based applicability and added alias routing for guidance-first review behavior.

## 🧱 Workstream C: command overlap and deprecation normalization (✅ done)

- Identify overlapped commands and overlapping modes. (✅ done)
  Done (2026-05-22): Evaluated and normalized overlap for guidance-first behavior (`/pe-meta-adherence` canonical, review-mode alias route) and release-diff behavior (`/pe-meta-release-monitor` canonical, update workflow compatibility route).

- Dismiss or alias redundant surfaces. (✅ done)
  Done (2026-05-22): Added canonical ownership policy, removed redundant `--mode` surface from core review, and kept non-overlapping controls per command capability.

- Keep orchestration comments where they add value. (✅ done)
  Done (2026-05-22): Preserved scheduling/cadence/lifecycle narrative in scheduled-review and release-diff narrative in release-monitor while normalizing command ownership and alias routing.

## 🧪 Workstream D: deterministic combinations and evidence (✅ done)

- Add parser tests for accepted option combinations across command families. (✅ done)
  Done (2026-05-22): Added deterministic acceptance checks for core, orchestration, and capability-gated combinations in `02.39-validation-status-option-contract/20260522-option-contract-evidence.json` (D-AC-01..D-AC-05).

- Add parser tests for deterministic rejection paths. (✅ done)
  Done (2026-05-22): Added deterministic rejection checks for redundant `--mode` in core review, mutually exclusive dependency flags, context-only dim misuse, guidance-first routing, and release-diff routing (D-RJ-01..D-RJ-05).

- Add validation evidence artifacts proving contract support. (✅ done)
  Done (2026-05-22): Stored rerunnable evidence in `06.00-idea/self-updating-prompt-engineering/20260521.01-pe-meta-implementation/02.39-validation-status-option-contract/`.

## 🚀 Workstream E: rollout and documentation updates (✅ done)

- Update user-facing command guidance with concrete examples by capability class. (✅ done)
  Done (2026-05-22): Added capability-class command examples and deterministic rejection examples in `.github/prompts/00.09-pe-meta/pe-meta-option-applicability-matrix.md`.

- Reconcile issue and plan documents with final normalized contract. (✅ done)
  Done (2026-05-22): Reconciled issue and plan wording to enforce non-overlapping option ownership, canonical routing, and explicit rejection of redundant controls.

- Add traceability rows to validation-status files for parity compliance checks. (✅ done)
  Done (2026-05-22): Added parity and option-contract traceability rows in `02.00-validation-status.md` plus evidence-linked execution entries in prompt validation status files.

## 📋 Exit criteria (✅ done)

- Every design/review type pair has parity in quality objective wording. (✅ done)
- Every design/review type pair has parity in scope-intent wording. (✅ done)
- A canonical option applicability matrix exists and is referenced by all PE-meta command prompts. (✅ done)
- Overlapping commands or modes are dismissed, aliased, or explicitly justified. (✅ done)
- Accepted and rejected combinations are test-covered and evidenced with deterministic corrective outputs. (✅ done)
