# pe-meta update plan — reliability dimensions and `--dim` alias rename

## 🎯 Objective

Update the pe-meta infrastructure so that:

1. The dimension catalog [`05.07-pe-meta-dimension-catalog.md`](../../../../.copilot/context/00.00-prompt-engineering/05.07-pe-meta-dimension-catalog.md) defines new **reliability dimensions** addressing R1–R12 from the [issue overview](overview.md).
2. A new `--dim reliability` group bundles those dimensions.
3. The legacy `--dim robustness` alias is renamed to `--dim adherence` (matching the canonical `/pe-meta-adherence` command), with one release of back-compat.
4. Affected pe-meta prompts in [`.github/prompts/00.09-pe-meta/`](../../../../.github/prompts/00.09-pe-meta/) recognize the new group and the renamed alias.
5. The applicability matrix ([`pe-meta-option-applicability-matrix.md`](../../../../.github/prompts/00.09-pe-meta/pe-meta-option-applicability-matrix.md)) reflects which commands accept `--dim reliability`.

## 📋 Prerequisites

- Workstream 1 ([vision update](01.01-visionupdate-plan.md)) — terminology rename recorded in v13 changelog
- Workstream 2 ([use-cases update plan](01.02-use-cases-update-plan.md)) — defines what each new dimension MUST measure (per-UC `Dimensions covered` blocks specify the boolean checks)

## 🔎 Current state analysis (✅ done)

### Dimension catalog

`05.07-pe-meta-dimension-catalog.md` defines **D1–D27** and groups. The `--dim robustness` group binds to:

```text
--dim robustness | D5, D6, D16, D18 | Boundaries + consistency + adherence + coverage
```

These dimensions are **adherence-flavoured**, not reliability-flavoured. None of D1–D27 covers process reproducibility, oscillation detection, multi-pass validation invariant, rollback drill, metadata-guard protocol enforcement, fail-closed boundaries, runtime boundary actionability, autonomy calibration, conflict-resolution coverage, portability scan, or mode-vs-risk decoupling.

## ⚙️ Deliverables

### Change 1: Add reliability dimensions to `05.07-pe-meta-dimension-catalog.md` (✅ done)

**Proposed new dimension rows** (final IDs TBD; the suggested D28–D35 numbering is a starting point — if any of these overlap conceptually with existing dimensions, prefer extending the existing dimension instead of adding a new one):

| New D | Name | Subject | Anchor | Handler tier |
|---|---|---|---|---|
| D28 | `reproducibility` | Same command + unchanged inputs → identical diff across runs | R3 | Deterministic (rerun + diff) |
| D29 | `loop-stability` | No artifact oscillates across recent runs (configurable window) | R1 | Deterministic (outcome-log scan) |
| D30 | `metadata-guard-enforcement` | Every change in the outcome log shows pre-change block + post-change reconciliation entries | R5 + R6 | Deterministic (log validation) |
| D31 | `multipass-validation-invariant` | Every committed change records ≥1 independent validation pass | R2 | Deterministic (log validation) |
| D32 | `rollback-readiness` | Tier-scoped snapshot exists for every tier-N change; restore drill succeeds | R8 | Mixed (deterministic snapshot check + drill execution) |
| D33 | `boundary-actionability` | For every artifact with `scope.excludes` or `boundaries:`, the body contains an enforceable check the model can run at runtime | R7 | Reasoning model (red-team prompts) |
| D34 | `autonomy-calibration` | Autonomy-gradient thresholds match observed outcome-log success rates within tolerance | R9 | Reasoning model + analytics |
| D35 | `portability-boundary` | All agent/prompt names use the `pe-` prefix (or declared namespace); no generic names that clash on port | R11 | Deterministic (regex over filenames + YAML `name:`) |

**Also add coverage rows** for R10 (conflict-resolution) and R12 (mode-vs-risk) — decide whether they extend existing dimensions or warrant their own. R10 can extend D17 (cross-coherence); R12 can extend D5 (boundaries) or land as a sub-check inside D33.

**Per-type applicability matrix updates:** add rows for D28–D35 in the existing applicability table. Most reliability dimensions apply to the **system** (state path + outcome log) rather than to a single artifact type — mark with a new column "System" if needed, or use the existing convention for system-level dims (D22 precedent).

**Proposed group row:**

```markdown
| `--dim reliability` | D28, D29, D30, D31, D32, D33, D34, D35 | System-reliability audit: reproducibility, loop stability, validation invariants, rollback, boundary actionability, autonomy calibration, portability |
```

**Cost-gradient row:**

```markdown
| `--dim reliability` | 8 dimensions | Medium-high (mostly deterministic checks + 2 reasoning) |
```

### Change 3: Rename `--dim robustness` → `--dim adherence` (✅ done)

| Step | Action |
|---|---|
| 3a | Rename the row in the dimension-groups table. The binding (D5, D6, D16, D18) stays the same. |
| 3b | Add a deprecation note: *"Legacy alias `--dim robustness` accepted for one release with deterministic warning; resolves to `--dim adherence`."* |
| 3c | Update [`pe-meta-option-parser-tests.md`](../../../../.github/prompts/00.09-pe-meta/pe-meta-option-parser-tests.md): add cases asserting (a) the alias resolves correctly, (b) a deprecation notice is emitted, (c) the canonical form is recommended in the notice. |

### Change 4: Update applicability matrix (✅ done)

Edit [`pe-meta-option-applicability-matrix.md`](../../../../.github/prompts/00.09-pe-meta/pe-meta-option-applicability-matrix.md) so every command that accepts `--dim` also lists `reliability` as a valid value, and `adherence` replaces `robustness`. Per R-P7-capability-applicability in the vision, unsupported combinations MUST be rejected deterministically with a corrective message.

### Change 5: Update prompt files that enumerate `--dim` options (✅ done)

Grep [`.github/prompts/00.09-pe-meta/`](../../../../.github/prompts/00.09-pe-meta/) for `--dim robustness` and for hard-coded `--dim` value lists. For each match:

- Replace `robustness` with `adherence` (canonical) in user-visible help text and examples.
- Add `reliability` to the enumerated options.
- Where prompts include preset aliases (`healthcheck`, `performancecheck`, `fullcheck`), confirm whether `healthcheck` should now include `--dim reliability` (recommended: yes — at least D28, D29, D30 belong in a basic health check).

### Change 6: Update `pe-meta-scheduled-review.prompt.md` rotation (✅ done)

The scheduled-review rotation currently cycles adherence/consistency/quality. Add reliability to the rotation:

- Every Nth run (configurable; suggest 5th) → `--dim reliability` deep pass.
- Document the new rotation in the prompt's process steps.
