# PE-meta improvement: vision update plan — dual-purpose metadata

## 🎯 Objective

Update `20260523.01-vision.v13.md` to:

1. Explicitly state that YAML metadata serves **two audiences**: validation tooling (pe-meta) AND the executing model (runtime self-constraint)
2. Add a design principle for **runtime grounding** — metadata is not just declarative documentation, it's an enforceable contract
3. Ensure the "Metadata-driven" principle in the design principles section covers both purposes

## 📋 Prerequisites

None. This is the first workstream. Metadata contracts and instruction changes depend on the vision being explicit about dual-purpose.

## 🔎 Current state analysis (✅ done)

### What the vision already says

| Section | Content | Covers runtime? |
|---|---|---|
| Design principles — "Metadata-driven" | "Every artifact change MUST be validated against declared metadata (goal, scope, boundaries, rationales) before apply" | ❌ Only covers changes, not execution |
| Goal § Guidance quality | Defines clarity, actionability, etc. as prerequisites for autonomy | Partially — actionability is about execution, but metadata isn't linked to it |
| Rationale for metadata-contracts | "Without structured metadata, breaking/non-breaking classification requires LLM judgment" | ❌ Only covers classification |

### What's missing

A statement that metadata also serves **runtime enforcement** — the executing model uses its own declared goal/scope/boundaries to self-constrain during execution.

## ⚙️ Deliverables

### Change 1: Design principles — extend "Metadata-driven" principle (✅ done)

**Location:** Design principles section, "Metadata-driven" principle

**Current (paraphrased):** Metadata enables deterministic change classification.

**Proposed addition:**

```markdown
Metadata serves two audiences:
1. **Validation tooling** (pe-meta): deterministic change classification, drift detection, scope preservation
2. **Executing model** (runtime): self-constraint enforcement — the model grounds its behavior on declared goal, scope, and boundaries during execution

Both audiences consume the same YAML fields. The difference is timing: validation happens before/after changes; runtime grounding happens during execution.
```

### Change 2: Add "Runtime grounding" to the design principles list (✅ done)

**Location:** After the existing metadata-driven principle

**Proposed:**

```markdown
**Runtime grounding.** YAML metadata is not passive documentation — it's an enforceable execution contract. Every artifact's body MUST include a grounding mechanism that instructs the executing model to verify requests against its declared `scope.covers`, reject requests matching `scope.excludes`, and enforce all `boundaries:` constraints throughout execution. This bridges declarative metadata to imperative behavior.
```

### Change 3: Link to guidance quality properties (✅ done)

**Location:** Guidance quality table

**Proposed addition to "Actionability" row:**

```markdown
| **Actionability** | An agent can act on the rule without additional interpretation — AND can enforce its own metadata as runtime constraints | LLM test: can the rule be translated into a boolean check? Can the model verify "am I within scope?" |
```
