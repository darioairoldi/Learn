## ⚙️ Deliverables (✅ done)

### Change 1: Command families table — fix mutation posture labels

**Before:**

| Family | Mutation posture |
|---|---|
| Creation | Write (new artifact) |
| Review | Read-only |
| Guidance-first | Read-only |
| Scheduled | Read-only |
| Update | Write (modify existing) |
| Release-diff | Write (modify existing) |

**After:**

| Family | Mutation posture |
|---|---|
| Creation | Write (new artifact) — research from scratch, create with best quality |
| Review | Assessment-first (write via `--mode apply`) — understand existing, improve to same quality target |
| Guidance-first | Assessment-first (write via `--mode apply`) — trace quality, improve adherence |
| Scheduled | Autonomous low-risk apply — periodic detection + autonomous fix for low-risk findings |
| Update | Write (modify existing) — apply assessed improvements |
| Release-diff | Write (modify existing) — reconcile with platform changes |

**Rationale:** The mutation posture should describe what the command *can do*, not just its default mode. The parity contract says Create and Review reach the same destination — the table should reflect that.

### Change 2: Add "Create/Review parity statement" below the table

Add a clarification paragraph that replaces the current "read-only families" sentence:

**Before:**
> Design/review parity applies within each family: read-only families assess against the same quality standard that write families enforce during construction.

**After:**
> **Create/Review parity principle:** Create and Review commands achieve the same quality goal. They differ in starting point and process:
> - **Create** commands focus on research and investigation from scratch, producing artifacts at best quality (possibly reusing existing artifacts as foundations)
> - **Review** commands focus on understanding existing artifacts and improving their structure and quality to reach the same target
>
> The confirmation gate is **risk-proportional, not command-proportional**. A Review command that identifies a low-risk improvement applies it autonomously (or with lightweight notification), exactly as an Update command would. The command family determines the *default posture* (assess-first vs. write-first), not the *permission ceiling*.

### Change 3: Strengthen autonomy principle — add "low-risk autonomy" rule

In the "Autonomy gradient" section (after the current table), add an explicit rule:

> **Low-risk autonomy rule:** Changes classified as non-breaking (goal, scope, and boundaries preserved) with high confidence (machine-verifiable or multi-pass validated) MUST proceed autonomously regardless of which command family triggered them. A command's default mode (`plan` vs `apply`) is a UX convenience for the user's workflow preference — it MUST NOT be interpreted as a permission ceiling that overrides risk classification. The only legitimate gate is assessed risk, never command identity.

### Change 4: Update `--mode` option definition note

In the option taxonomy table, update the "Default when omitted" column for `--mode`:

**Before:**
> `plan` for Review/Guidance-first; `apply` for Update/Release-diff

**After:**
> `apply` for all families where `--mode` is applicable — risk classification governs what is applied autonomously vs. proposed; use `--mode plan` to explicitly request assessment-only output

### Change 5: Resolve the Scheduled review inconsistency

The command families table says `Scheduled | Read-only` but the actual `pe-meta-scheduled-review.prompt.md` already uses `agent: agent` with `replace_string_in_file`. The vision should acknowledge that scheduled review applies low-risk fixes autonomously.

Already addressed in Change 1's updated table.
