### Change 3: Add "Post-assessment behavior" section to affected use cases

After the "Workflow" section in each Review/Guidance-first use case, add a section documenting what happens with findings:

```markdown
## Post-assessment behavior

By default (`--mode apply`), the command evaluates each finding against the vision's change classification:

| Finding risk | Action | Confirmation |
|---|---|---|
| **Non-breaking** (goal, scope, boundaries preserved) + high confidence | Apply autonomously | Post-execution notification only |
| **Non-breaking** + medium confidence | Apply with lightweight pre-notification | User can abort within notification window |
| **Breaking** (changes goal, scope, or boundaries) | Propose as plan | Requires explicit human confirmation |
| **Uncertain** (cannot classify risk) | Propose as plan | Requires explicit human confirmation |

The risk assessment integrates with the autonomy gradient defined in the vision:
- **Structural fixes** (formatting, reference links, metadata) → Always non-breaking → Autonomous
- **Content additions** (missing dimensions, expanded coverage) → Usually non-breaking → Autonomous with notification
- **Content removals or rewrites** → May be breaking → Propose
- **Scope changes** (new artifact types, new boundary rules) → Breaking → Always propose
```

### Change 4: Add new use case — UC-NEW: Autonomous improvement workflow

Create a new P0 use case in `03-consumer-correctness/` that documents the full autonomous improvement flow:

**File:** `03-consumer-correctness/p0-02-autonomous-improvement-workflow.md`

**Key content:**
- **Command:** `/pe-meta-review <path> --deps direct` (default `--mode apply`)
- **Purpose:** Demonstrate how Review achieves the same quality goal as Create — by investigating existing artifacts, identifying improvements, classifying risk, and applying low-risk changes autonomously (this is the default behavior, no explicit `--mode` needed)
- **Workflow:**
  1. Full dimensional assessment of target + direct dependencies
  2. Generate findings with per-finding risk classification
  3. For each non-breaking finding: apply fix → validate fix → record change
  4. For each breaking finding: add to proposal report
  5. Output: combined report (applied changes + proposed changes requiring confirmation)
- **Relationship to Create:** Show that if an artifact were created from scratch, it would reach the same quality target that `--mode apply` now brings the existing artifact to — just via a different path (understand + improve vs. research + build)
