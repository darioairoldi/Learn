## 📋 Things to do (✅ done)

### Step 1 — YAML frontmatter update (✅ done)

To do: rewrite the YAML header to drop preset vocabulary.

- `description:` — replace `Modes: fullcheck, healthcheck, performancecheck. Every phase independently disableable.` with `Pipeline phases (structure / consistency / content) are independently disableable via --skip. Execution depth controlled by --mode.`
- `argument-hint:` — replace `[preset] [--mode plan|apply] [--dim <group>] [--scope <type>] [--skip <stage>,...] [--incremental]` with `[--mode plan|apply] [--dim <group>] [--scope <type>] [--skip <stage>,...] [--incremental]`.
- `scope.covers:` — remove the three lines that describe `fullcheck`, `healthcheck`, `performancecheck` as modes. Replace with a single line: `End-to-end pipeline with --mode (plan|apply), --dim, --scope, --skip controls per the v13 option taxonomy`.
- `boundaries:` — change `MUST parse mode + scope + flags before executing any phase` to `MUST parse --mode, --dim, --scope, --skip before executing any phase` (drop the implicit "mode" reference to preset names).
- `rationales:` — append: `Removing preset aliases eliminates ambiguity between named shortcuts and explicit flags; one source of truth (option taxonomy) replaces two (presets + flags).`
- Bump `version:` (e.g., `1.0.0` → `2.0.0` to signal a breaking parser change).

### Step 2 — Replace the "Modes" section (✅ done)

To do: delete the `## Modes` heading + its table, the `### Execution mode (--mode)` subsection's preset reference, and the `### Preset aliases` subsection. Replace them with a single `## Invocation options` section that documents the canonical taxonomy only.

Proposed replacement structure:

```markdown
## Invocation options

The orchestrator accepts five canonical options. There are no shortcut presets.

### --mode plan|apply (default: apply)
- `apply` — full Research → Build → Validate cycle with low-risk autonomous apply + propose for higher-risk findings (default).
- `plan` — Research-only execution (no Build/Validate substeps); produces health score and findings report; no file writes.

### --dim <group|D#> (default: full)
Dimension selection per the v13 option taxonomy: `full`, `freshness`, `quality`, `adherence`, `reliability`, `optimize`, `model`, `structural`, `strategic`, `context-full`, `context-health`, or specific `D1`–`D35`.

### --scope <type|path> (default: all)
`all`, `context`, `instructions`, `agents`, `prompts`, `skills`, `hooks`, `snippets`, `templates`, or a specific file path.

### --skip <stage>[,<stage>...] (default: none)
| Stage | Skips | Use when |
|---|---|---|
| `research` | Phase 1 source research | No external input needed |
| `external` | Internet/URL fetching in all phases | Local-only check |
| `structure` | Phase 2 structure audit | Structure known-good |
| `consistency` | Phase 3 consistency audit | Cross-artifact consistency verified |
| `content` | Phase 4 content audit | Individual artifacts already reviewed |

### --incremental (default: false)
Process only artifacts modified since the last apply-mode run (uses `last_updated` + manifest in `05.04-meta-review-log.md`).
```

### Step 3 — Update intent-resolution and defaults paragraph (✅ done)

To do: rewrite the paragraph currently at line ~61.

Replace:

> **Default**: URL/file/description = fullcheck. "review"/"audit"/"health" = healthcheck. "optimize"/"dedup"/"tokens" = performancecheck. No arguments = fullcheck (full analysis of current state). "plan"/"preview"/"what-if" = fullcheck `--mode plan`.

With:

> **Intent resolution from natural-language input.** When the user provides natural-language hints alongside or instead of flags, resolve them to canonical options as follows:
>
> | User intent / keyword | Resolves to |
> |---|---|
> | URL or file path provided | `--mode apply` (default), Phase 1 research enabled |
> | "review", "audit", "health", "diagnose" | `--mode plan --skip research` |
> | "optimize", "dedup", "tokens", "efficiency" | `--mode apply --dim optimize --skip research,structure,consistency` |
> | "plan", "preview", "what-if", "dry run" | `--mode plan` |
> | No arguments | `--mode apply` with full pipeline |
>
> Resolved options are echoed back to the user as a single canonical invocation string before execution begins. Explicit `--` flags always override intent-derived defaults.

This preserves the natural-language UX without exposing the preset names as a flag layer.

### Step 5 — Update backward-compatibility table (✅ done)

To do: extend the existing "Backward-compatible aliases" table inside the prompt to handle preset deprecation if the deprecation path is selected (see [§ Backward-compat decision](#-backward-compat-decision)). Two options:

- **Option A (hard removal, recommended).** Delete preset recognition entirely. If the parser sees a positional token equal to `healthcheck` / `performancecheck` / `fullcheck`, it MUST refuse with a deterministic error message:
  > `"healthcheck" is no longer a supported preset. Use: --mode plan --skip research [other flags]. See pe-meta-update.prompt.md § Invocation options.`
  Add this as a rule in the prompt's "Argument parsing" section.

- **Option B (one-release deprecation).** Add three rows to the back-compat table:
  | Legacy | Canonical |
  |---|---|
  | `healthcheck` (positional) | `--mode plan --skip research` |
  | `performancecheck` (positional) | `--mode apply --dim optimize --skip research,structure,consistency` |
  | `fullcheck` (positional) | `--mode apply` |
  …and emit a deprecation warning on each match. Remove next release.

This plan defaults to **Option A** unless the user requests Option B explicitly.

## 🔄 Backward-compat decision

Two options were defined in Step 5. Choose one before Step 5 executes:

| | Option A — Hard removal | Option B — One-release deprecation |
|---|---|---|
| Effort | Minimal — one rejection rule | Moderate — three new alias rows + warning emit |
| User impact | Immediate breakage for users on old commands | Soft landing for one release |
| Code surface | Smaller (less parser logic) | Larger (deprecation handling) |
| Doc surface | Clean | Carries a "deprecated" callout |
| Recommended when | Internal-only PE system, low user count, vision change already announced | Wider user base or external consumers |

This plan defaults to **Option A** based on the system's current internal use. The decision is captured here, not in the prompt file itself.
