## 📝 Description

### Brief description

The `/pe-meta-update` orchestrator accepts three positional **preset alias** tokens — `healthcheck`, `performancecheck`, `fullcheck` — that expand to fixed combinations of canonical options (`--mode`, `--dim`, `--skip`). When a user types:

```text
/pe-meta-update healthcheck --dim freshness --scope context
```

they cannot tell from the command line alone what is actually being executed:

- Is research enabled or skipped?
- What is the execution mode (plan vs. apply)?
- Are structure/consistency/content audits running?

To answer those questions the user must look up the preset's expansion table in `pe-meta-update.prompt.md` or in vision v13. The alias hides the very options the v13 vision designed to be explicit and composable.

### Trigger that surfaced this

A direct user question during the current conversation:

> "What does it mean healthcheck? `/pe-meta-update healthcheck --dim freshness --scope context`. I thought only `--dim freshness --scope context` were enough. Please investigate the keys understand the meaning of health check full check in addition to the meaning `--dim` option."

The fact that the user — the author of the vision document and the prompt file — needed to re-derive what `healthcheck` does from first principles is itself the bug: the abstraction is not paying for itself.

### Impact points

| # | Impact | Severity contribution |
|---|---|---|
| 1 | Users cannot read a command and predict its behavior without consulting an expansion table | Medium |
| 2 | Vision v13, the use-case catalog, and the prompt file each restate the preset expansions — three sources of truth that can drift | Medium |
| 3 | The parser has to run a preset-resolution step before the canonical option pipeline, adding one normalization branch | Low |
| 4 | Composition rules (e.g., what happens when `healthcheck --skip research` is invoked since `research` is already skipped) are implicit and re-derived per use | Medium |
| 5 | New options added later (e.g., a future `--dim reliability` group) must be retro-fitted into preset definitions, doubling the maintenance surface | Medium |

## 🔬 Analysis

### Root cause

The preset alias mechanism was introduced (vision v12 → v13) as a **usability convenience** to give common invocations a memorable name. But:

1. **The convenience is single-direction.** Memorizing `healthcheck` is easier than typing `--mode plan --skip research`. Reading `healthcheck` in a script, a use case, or a log entry is **harder** than reading the expanded options, because expansion requires lookup.
2. **The vision v13 already committed to a small, orthogonal option taxonomy** (`--mode`, `--dim`, `--scope`, `--skip`, `--deps`) designed to be self-documenting. The preset layer is therefore **redundant with** rather than **complementary to** the v13 design.
3. **Composition is ambiguous.** When a preset coexists with explicit flags, the merge rule (preset defaults + user flags, with user flags winning, with `--skip` lists deduplicated) is not stated in the prompt file. Each user re-derives it.
4. **Three sources of truth drift.** The vision states the expansion in one form, the prompt file in another, and the use-case catalog in a third. A change to any one of them requires synchronizing the other two — which has not been happening reliably (e.g., `--dim robustness` deprecation note appears in only one place).

### Why this was not caught earlier

- The aliases predated the canonical option taxonomy. They were "grandfathered in" when v13 formalized the taxonomy rather than being re-evaluated against it.
- The presets cover the most-common case ("am I healthy?"), so most invocations only use the preset and never expose the merge ambiguity.
- The aliases appear in the prompt file's argument-hint, so they look like a first-class part of the contract rather than a convenience layer.

### Impact assessment

| Dimension | Before fix | After fix |
|---|---|---|
| Readability of invocations | Requires lookup | Self-documenting |
| Parser complexity | preset-resolution step + canonical parsing | canonical parsing only |
| Sources of truth | 3 (vision, prompt, use cases) | 1 (vision option taxonomy) |
| Composition rules | Implicit | Not needed (no aliases to merge) |
| Onboarding friction | "Learn 3 presets + 5 options" | "Learn 5 options" |
| User keystrokes for common cases | Lower | Higher (mitigated by intent-resolution from natural-language keywords — see plan 01.03 Step 3) |

### Affected workflows

- **Ad-hoc health check** — replace `healthcheck` with `--mode plan --skip research`. One extra clause but the behavior is now visible.
- **Scheduled review** — uses `--deps` + auto-rotation, does not invoke presets; unaffected.
- **Release-driven update** — uses `release-diff` / `release-monitor` families; unaffected.
- **Tooling that reads `usecase-index.json`** — every `primary_entry_point` string changes shape but remains parseable as `/pe-meta-update <flags>`.

## ✅ Solution implemented

**Implementation status:** Not yet executed. Three coordinated plans have been drafted instead of a direct fix because the change touches three distinct artifact tiers (vision, catalog, prompt) that must be updated in lockstep.

### Plan inventory

| Plan | File | Scope |
|---|---|---|
| 1 | [01.01-visionupdate-plan.md](01.01-visionupdate-plan.md) | Remove preset section from `20260523.01-vision.v13.md`; replace with `### Explicit option discipline`; rewrite execution-depth example |
| 2 | [01.02-use-cases-update-plan.md](01.02-use-cases-update-plan.md) | Rewrite ~25 use-case files + 5 group READMEs + catalog README + `usecase-index.json` to use expanded options |
| 3 | [01.03-pe-meta-update-plan.md](01.03-pe-meta-update-plan.md) | Refactor `pe-meta-update.prompt.md`: drop `[preset]` from argument-hint, replace `## Modes` and `### Preset aliases` with unified `## Invocation options`, rewrite all examples, decide backward-compat strategy (Option A hard removal vs. Option B one-release deprecation) |

### Canonical expansion (shared by all plans)

| Removed preset | Canonical replacement |
|---|---|
| `healthcheck` | `--mode plan --skip research` |
| `performancecheck` | `--mode apply --dim optimize --skip research,structure,consistency` |
| `fullcheck` | `--mode apply` |

**Composition rule** (when a preset previously appeared with explicit flags): explicit user flags win; `--skip` stage lists merge with deduplication.

### Worked migration examples

| Before | After |
|---|---|
| `/pe-meta-update healthcheck --dim freshness --scope context` | `/pe-meta-update --mode plan --skip research --dim freshness --scope context` |
| `/pe-meta-update healthcheck --dim quality --skip structure,consistency` | `/pe-meta-update --mode plan --dim quality --skip research,structure,consistency` |
| `/pe-meta-update performancecheck --scope agents,prompts` | `/pe-meta-update --mode apply --dim optimize --skip research,structure,consistency --scope agents,prompts` |
| `/pe-meta-update fullcheck --scope context` | `/pe-meta-update --mode apply --scope context` |

### UX preservation

To avoid forcing users to memorize five flags for the common case, plan 01.03 Step 3 retains **intent resolution from natural-language input**: keywords like `review`, `audit`, `optimize`, `dry-run` still resolve to canonical option combinations, but the resolved combination is echoed back to the user as a single canonical invocation string before execution. The aliases disappear; the convenience does not.

## 🎓 Lessons learned

### What went wrong

- **Convenience abstractions added on top of an already-orthogonal taxonomy create rather than reduce friction.** The v13 option taxonomy was designed to be self-documenting; the preset layer worked against that design.
- **Three sources of truth without a generator drift.** Vision, prompt, and use-cases each restated the preset expansion in slightly different wording. No tooling enforced consistency.
- **The argument-hint placed `[preset]` first**, signaling that the preset was the primary way to invoke the command, when in fact the canonical options were the primary contract.

### What went right

- **Vision v13's option taxonomy is sound.** Removing the preset layer leaves a complete, expressive surface — no behavior is lost.
- **The use-case catalog has a machine-readable index** (`usecase-index.json`), which makes the rewrite mechanically verifiable: every entry's `primary_entry_point` is a parseable string.
- **The user identified the comprehension problem from first principles** rather than waiting for a downstream failure.

### Improvements for future

| Lesson | Mitigation |
|---|---|
| Convenience abstractions need a "delete test" before adoption | When proposing a new alias/preset/shortcut, draft the canonical form first and ask: "if I delete the shortcut, do I lose meaningful expressivity, or only keystrokes?" If only keystrokes, do not add the shortcut. |
| Three documentation sources drift | Add a CI check (or pe-meta hook) that asserts the canonical option list in the vision matches the parser's accepted set in `pe-meta-update.prompt.md`. |
| Argument-hint signals command identity | Argument-hints should list canonical options only; convenience inputs go in a separate "Intent resolution" section. |

### D. Example: anatomy of the comprehension problem

The original command:

```text
/pe-meta-update healthcheck --dim freshness --scope context
```

Hidden facts the user must look up:

- `--mode` is implicitly `plan` (no apply, no file writes).
- `--skip research` is implicitly applied (Phase 1 source research is skipped).
- Build and Validate substeps inside Phases 2/3/4 are skipped (research-only).
- Output is a health score and findings report, not a change list.

The same command after this issue's fix:

```text
/pe-meta-update --mode plan --skip research --dim freshness --scope context
```

Every fact is visible on the command line. Nothing is hidden; nothing is lost.
